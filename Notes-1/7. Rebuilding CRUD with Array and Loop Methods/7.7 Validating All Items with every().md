# 1. What `every()` Does

## What You Will Learn

`every()` checks whether all elements in an array satisfy a condition.

It returns:

```js
true
```

when every element passes.

It returns:

```js
false
```

when at least one element fails.

Examples of questions `every()` can answer:

* Does every product have a valid name?
* Are all prices non-negative?
* Are all quantities whole numbers?
* Is every task completed?
* Does every employee have an email?
* Are all imported records valid?
* Does every cart item have a quantity greater than zero?

---

## Concept Explanation

Suppose:

```js
const numbers = [
  2,
  4,
  6,
];
```

Check whether every number is even:

```js
const allEven =
  numbers.every(
    (number) =>
      number % 2 === 0
  );
```

Results:

```text
2 is even → true
4 is even → true
6 is even → true
```

Final result:

```js
true
```

Now consider:

```js
const numbers = [
  2,
  5,
  6,
];
```

Results:

```text
2 is even → true
5 is even → false
```

`every()` returns:

```js
false
```

It does not need to check `6`.

---

## Why It Matters for CRUD

CRUD applications often validate one form submission at a time.

However, applications may also need to validate the entire state.

Examples:

* Before saving data to `localStorage`.
* Before sending a batch of products to an API.
* After importing records from a file.
* Before calculating inventory totals.
* Before allowing checkout.
* Before exporting a report.
* During debugging.

`every()` is useful because it answers:

```text
Is the entire array valid?
```

---

## Common Beginner Mistakes

* Using `every()` when only one match is needed.
* Confusing it with `some()`.
* Forgetting to return the condition.
* Assuming it returns the invalid object.
* Mutating state during validation.
* Using `||` when all validation rules require `&&`.
* Forgetting that an empty array returns `true`.
* Replacing individual form validation with only whole-array validation.

# 4. Plain Loop Versus `every()`

## Using a Plain Loop

```js
let allPricesAreValid =
  true;

for (
  let index = 0;
  index < products.length;
  index++
) {
  const product =
    products[index];

  if (
    !Number.isFinite(
      product.price
    ) ||
    product.price < 0
  ) {
    allPricesAreValid =
      false;

    break;
  }
}
```

## Using `every()`

```js
const allPricesAreValid =
  products.every(
    (product) =>
      Number.isFinite(
        product.price
      ) &&
      product.price >= 0
  );
```

---

## What Both Versions Do

Both:

* Check products from the beginning.
* Validate one product at a time.
* Stop after the first failure.
* Produce a boolean.
* Leave state unchanged.
* Confirm whether all products pass.

---

## Which Is Easier to Read?

For a simple whole-array rule, `every()` is often clearer.

```js
const allProductsAreValid =
  products.every(
    isValidProduct
  );
```

This reads naturally:

```text
Does every product satisfy isValidProduct?
```

---

## Which Gives More Control?

A plain loop may be better when:

* You need the exact failing index immediately.
* You need several error messages.
* You need detailed logging.
* You need to collect all invalid items.
* You need unusual iteration rules.
* You need custom recovery behavior.

---

# 5. `every()` Performs Internal Iteration

This:

```js
const allValid =
  products.every(
    (product) =>
      product.quantity >= 0
  );
```

can be understood as:

```js
let allValid = true;

for (
  let index = 0;
  index < products.length;
  index++
) {
  if (
    products[index].quantity <
    0
  ) {
    allValid = false;
    break;
  }
}
```

`every()` packages this pattern into an array method.

---

# 6. `every()` Stops After the First Failure

State:

```js
const products = [
  {
    id: 1,
    quantity: 2,
  },
  {
    id: 2,
    quantity: -1,
  },
  {
    id: 3,
    quantity: 5,
  },
];
```

Validation:

```js
products.every(
  (product) =>
    product.quantity >= 0
);
```

Process:

```text
Product 1 → true
Product 2 → false
```

Final result:

```js
false
```

Product 3 is not checked.

# 7. `every()` and Empty Arrays

```js
const products = [];

const result =
  products.every(
    (product) =>
      product.price >= 0
  );

console.log(result);
```

Expected output:

```text
true
```

This may surprise beginners.

---

## Why Does an Empty Array Return `true`?

The statement is:

```text
Every product in this empty array has a valid price.
```

There is no product that violates the rule.

Therefore, the statement is considered true.

This is called **vacuous truth**.

---

## Practical Empty-State Check

If your business rule requires at least one product, combine conditions:

```js
const inventoryIsValid =
  products.length > 0 &&
  products.every(
    isValidProduct
  );
```

Now an empty inventory returns:

```js
false
```

# 8. `some()` Versus `every()`

## `some()`

At least one item must pass.

```js
const hasOutOfStockProduct =
  products.some(
    (product) =>
      product.quantity === 0
  );
```

---

## `every()`

All items must pass.

```js
const allProductsInStock =
  products.every(
    (product) =>
      product.quantity > 0
  );
```

---

## Comparison

| Method    | Question                     | Returns `true` when    |
| --------- | ---------------------------- | ---------------------- |
| `some()`  | Does at least one item pass? | One or more items pass |
| `every()` | Do all items pass?           | No item fails          |

---

# 9. Logical Relationship Between `some()` and `every()`

These two conditions are related:

```js
const allPricesAreValid =
  products.every(
    (product) =>
      product.price >= 0
  );
```

and:

```js
const hasInvalidPrice =
  products.some(
    (product) =>
      product.price < 0
  );
```

For non-empty arrays:

```js
allPricesAreValid ===
  !hasInvalidPrice
```

This is a useful mental model.

# 14. Complete Product Validator

```js
function isValidProduct(
  product
) {
  return (
    Number.isInteger(
      product.id
    ) &&
    product.id > 0 &&
    typeof product.name ===
      "string" &&
    product.name.trim() !==
      "" &&
    Number.isFinite(
      product.price
    ) &&
    product.price >= 0 &&
    Number.isInteger(
      product.quantity
    ) &&
    product.quantity >= 0
  );
}
```

Use it:

```js
const allProductsAreValid =
  products.every(
    isValidProduct
  );
```

---

# 15. Why Use a Named Validation Function?

Instead of writing a large callback repeatedly:

```js
products.every(
  (product) => {
    // Many conditions.
  }
);
```

create:

```js
function isValidProduct(
  product
) {
  // Return boolean.
}
```

Benefits:

* Easier to read.
* Easier to test.
* Reusable.
* Separates rules from iteration.
* Simplifies `every()`.

# 16. Full Inventory Validation Function

```js
function validateInventory(
  products
) {
  if (!Array.isArray(products)) {
    return {
      isValid: false,
      message:
        "Inventory must be an array.",
    };
  }

  if (products.length === 0) {
    return {
      isValid: false,
      message:
        "Inventory must contain at least one product.",
    };
  }

  const allProductsAreValid =
    products.every(
      isValidProduct
    );

  if (!allProductsAreValid) {
    return {
      isValid: false,
      message:
        "One or more products contain invalid data.",
    };
  }

  return {
    isValid: true,
    message:
      "All products are valid.",
  };
}
```

# 17. Finding the Invalid Product

`every()` returns only a boolean.

It does not return the failing object.

To retrieve the first invalid product, use:

```js
const invalidProduct =
  products.find(
    (product) =>
      !isValidProduct(product)
  );
```

This returns:

* The first invalid product.
* `undefined` when all products are valid.

---

# 18. Getting All Invalid Products

Use `filter()`:

```js
const invalidProducts =
  products.filter(
    (product) =>
      !isValidProduct(product)
  );
```

This returns an array of all invalid products.

---

# 19. Choosing the Correct Method

| Need                                | Method        |
| ----------------------------------- | ------------- |
| Are all products valid?             | `every()`     |
| Is at least one product invalid?    | `some()`      |
| What is the first invalid product?  | `find()`      |
| Which products are invalid?         | `filter()`    |
| What is the first invalid position? | `findIndex()` |

# 20. Whole-State Validation Versus Form Validation

Whole-state validation:

```js
products.every(
  isValidProduct
);
```

checks existing state.

Form validation checks one submitted object before it enters state.

```js
validateProductData(
  formData
);
```

Both are useful.

---

## Form Validation Prevents Invalid State

```text
Read form
→ validate submitted values
→ update state only if valid
```

## Whole-State Validation Detects Invalid State

```text
Read complete state
→ verify all objects
→ block save/export if invalid
```

Do not rely on `every()` as the only form-validation method.

# 23. Validating All Tasks

Task structure:

```js
{
  id,
  title,
  completed,
  priority
}
```

Validator:

```js
function isValidTask(task) {
  const validPriorities = [
    "low",
    "medium",
    "high",
  ];

  return (
    Number.isInteger(task.id) &&
    task.id > 0 &&
    typeof task.title ===
      "string" &&
    task.title.trim() !== "" &&
    typeof task.completed ===
      "boolean" &&
    validPriorities.includes(
      task.priority

}
```

```js
const allTasksAreValid =
  tasks.every(

```

# 26. Avoid Mutating State During Validation

Incorrect:

```js
const allProductsAreValid =
  products.every(
    (product) => {
      product.name =
        product.name.trim();

      return (
        product.name !== ""
      );
    }
  );
```

This changes product state during validation.

Correct:

```js
const allProductsAreValid =
  products.every(
    (product) =>
      product.name.trim() !==
      ""
  );
```

Validation should normally be read-only.

---

# 27. Missing Return with Braces

Incorrect:

```js
const allValid =
  products.every(
    (product) => {
      product.price >= 0;
    }
  );
```

The callback returns:

```js
undefined
```

`undefined` is falsy.

Result:

```js
false
```

for any non-empty array.

---

## Correct with Explicit Return

```js
const allValid =
  products.every(
    (product) => {
      return (
        product.price >= 0
      );
    }
  );
```

---

## Correct with Concise Arrow Syntax

```js
const allValid =
  products.every(
    (product) =>
      product.price >= 0
  );
```

---

# 28. Incorrect Use of `||`

Suppose every product must have:

* A non-empty name.
* A non-negative price.

Incorrect:

```js
products.every(
  (product) =>
    product.name.trim() !==
      "" ||
    product.price >= 0
);
```

A product passes if only one condition is true.

Correct:

```js
products.every(
  (product) =>
    product.name.trim() !==
      "" &&
    product.price >= 0
);
```

Both rules must pass.

```html
      productList.innerHTML = "";

      products.forEach(
        (product, index) => {
          const card =
            document.createElement(
              "li"
            );

          const name =
            document.createElement(
              "h2"
            );

          const price =
            document.createElement(
              "p"
            );

          const quantity =
            document.createElement(
              "p"
            );

          card.classList.add(
            "product-card"
          );

          if (
            !isValidProduct(
              product
            )
          ) {
            card.classList.add(
              "invalid-product"
            );
          }

          name.textContent =
            `${index + 1}. ${product.name}`;

          price.textContent =
            `Price: KSh ${product.price.toLocaleString()}`;

          quantity.textContent =
            `Quantity: ${product.quantity}`;

          card.append(
            name,
            price,
            quantity
          );

          productList.appendChild(
            card
          );
        }
      );
    }

    validateButton.addEventListener(
      "click",
      function () {
        const allProductsAreValid =
          products.every(
            isValidProduct
          );

        if (
          allProductsAreValid
        ) {
          validationMessage.textContent =
            "All products are valid.";

          return;
        }

        const invalidProduct =
          products.find(
            (product) =>
              !isValidProduct(
                product
              )
          );

        validationMessage.textContent =
          `Invalid product found: ${invalidProduct.name}.`;
      }
    );

    renderProducts();
  </script>
</body>
</html>
```

# 30. Common Mistakes

## Mistake 1: Expecting an Object

```js
const invalidProduct =
  products.every(
    isValidProduct
  );
```

The result is a boolean.

---

## Mistake 2: Confusing `some()` and `every()`

```js
products.some(
  isValidProduct
);
```

This checks whether at least one product is valid, not all products.

---

## Mistake 3: Missing Return

```js
products.every(
  (product) => {
    product.price >= 0;
  }
);
```

---

## Mistake 4: Mutating During Validation

```js
product.quantity =
  Number(
    product.quantity
  );
```

Validation should not silently rewrite state.

---

## Mistake 5: Using `||` for Required Rules

When all rules must pass, use `&&`.

---

## Mistake 6: Forgetting Empty-Array Behavior

An empty array returns `true`.

Add a separate length rule when needed.

---

## Mistake 7: Rejecting Valid Zero Values

A price or quantity of zero may be valid depending on business rules.

---

## Mistake 8: Checking Only Values, Not Types

```js
"5000" >= 0
```

is true because of coercion.

Use strict numeric validation.

---

## Mistake 9: Replacing Form Validation

Whole-array validation does not replace validating submitted form values before insertion.

---

## Mistake 10: Updating the DOM Only

Highlighting an invalid card does not correct invalid state.

---

# 31. Best Practices

* Use `every()` when all array elements must satisfy a condition.
* Keep validation callbacks pure and read-only.
* Use named validators for complex object rules.
* Validate types as well as values.
* Combine required rules with `&&`.
* Handle empty arrays according to business requirements.
* Use `find()` when the first failing item is required.
* Use `filter()` when all failing items are required.
* Use `some()` when you need to know whether any invalid item exists.
* Validate form submissions before state mutation.
* Validate complete state before save, export, or batch processing.
* Keep the array as the source of truth.

## Key Takeaways

* `every()` returns `true` only when all array elements satisfy a condition.
* It returns `false` as soon as one element fails.
* It performs internal iteration.
* It returns `true` for an empty array.
* Add a separate length check when the array must contain items.
* `some()` asks whether at least one item passes.
* `every()` asks whether all items pass.
* Named validation functions improve readability and reuse.
* Whole-state validation should check types and business rules.
* Validation callbacks should not mutate state.
* Use `find()` for the first invalid item.
* Use `filter()` for all invalid items.
* Whole-state validation complements rather than replaces form validation.
