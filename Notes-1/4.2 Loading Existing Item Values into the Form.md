# 13. Loading Existing Item Values into the Form

## 13.1 What You Will Learn

An Update operation usually has two user-facing stages:

1. Load the existing values.
2. Save the changed values.

This lesson focuses on the first stage.

Suppose the user wants to edit:

```js
{
  id: 205,
  name: "Phone",
  price: 25000,
  quantity: 4,
  category: "phones",
  isActive: true
}
```

The form should be populated with those current values so the user can modify them.

The user should not need to re-enter every field from memory.

---

# 13.2 Update Flow So Far

You have already learned how to find the selected item.

The flow is now:

```text
User clicks Edit
        ↓
Read product ID from the button
        ↓
Convert the ID to a number
        ↓
Find the product in state
        ↓
Handle a missing product
        ↓
Load the product values into the form
        ↓
Remember which product is being edited
```

The next lessons will continue:

```text
User changes values
        ↓
Form is submitted
        ↓
Validate new values
        ↓
Update the matching object
        ↓
Render updated state
```

# 13.3 The Editing ID Variable

Create a variable outside event handlers:

```js
let editingProductId = null;
```

Meaning:

```text
null → no product is being edited
```

During editing:

```js
editingProductId = 205;
```

Meaning:

```text
The product with ID 205 is being edited
```

This variable is UI state.

It controls how the form should behave.

---

# 13.4 Why Store the Editing ID?

The form contains editable values such as:

```text
Phone
25000
4
```

But those values do not reliably identify the original product.

The user may change the name from:

```text
Phone
```

to:

```text
Smartphone
```

If the application used the name as identity, it might lose track of the original object.

The stable ID solves this.

```js
editingProductId = 205;
```

Even when the name changes, the application still knows which object to update.

```js
productNameInput.value =
```

# 13.6 Loading Number Inputs

HTML:

```html
<input
  id="product-price"
  type="number"
/>
```

JavaScript:

```js
productPriceInput.value =
  selectedProduct.price;
```

Even though `.value` is a string property, assigning a number works because the browser converts it for display.

You may also be explicit:

```js
productPriceInput.value =
  String(selectedProduct.price);
```

Both are acceptable.

# 13.9 Loading a Checkbox

HTML:

```html
<input
  id="product-active"
  type="checkbox"
/>
```

JavaScript:

```js
productActiveInput.checked =
  selectedProduct.isActive;
```

If:

```js
selectedProduct.isActive === true;
```

the checkbox becomes checked.

If:

```js
selectedProduct.isActive === false;
```

the checkbox becomes unchecked.

Use `.checked`, not `.value`.

# 13.10 Loading All Fields

```js
function loadProductIntoForm(product) {
  productNameInput.value =
    product.name;

  productPriceInput.value =
    product.price;

  productQuantityInput.value =
    product.quantity;

  productCategorySelect.value =
    product.category;

  productActiveInput.checked =
    product.isActive;
}
```

This function changes the form UI.

It does not change the product object.

---

# 13.11 Complete Edit-Start Function

```js
function startEditingProduct(
  productId
) {
  const product =
    findProductById(productId);

  if (product === null) {
    showError(
      "The selected product could not be found."
    );

    return;
  }

  editingProductId =
    product.id;

  loadProductIntoForm(product);

  productNameInput.focus();
  productNameInput.select();
}
```

This function:

1. Finds the product.
2. Handles missing state.
3. Stores the editing ID.
4. Loads form values.
5. Focuses the name input.

---

# 13.12 Loading Does Not Update State

This code:

```js
productNameInput.value =
  product.name;
```

changes the form.

It does not change:

```js
product.name;
```

The product object remains unchanged until the user submits valid changes.

That distinction is important.

```text
Loading form values → DOM update
Saving form values → State update
```

---

# 13.13 Form State Versus Application State

Application state:

```js
{
  id: 205,
  name: "Phone",
}
```

After loading:

```text
Input value: Phone
```

The user changes the input to:

```text
Smartphone
```

At this point:

```js
productNameInput.value
```

contains:

```text
Smartphone
```

But application state still contains:

```js
{
  id: 205,
  name: "Phone",
}
```

The change is temporary form state until submission succeeds.

---

# 13.14 Why This Separation Is Useful

The user may:

* Change their mind.
* Cancel the edit.
* Enter invalid data.
* Close the form.
* Click another product.

State should not change merely because the input value changed.

A controlled Update flow waits for:

1. Submission.
2. Validation.
3. Confirmation of the matching ID.
4. State mutation.

# 13.19 Reading the ID from an Edit Button

Rendered button:

```html
<button
  type="button"
  data-action="edit"
  data-id="205"
>
  Edit
</button>
```

Event handler:

```js
const selectedId =
  Number(
    clickedButton.dataset.id
  );
```

Validate:

```js
if (
  !Number.isInteger(selectedId) ||
  selectedId <= 0
) {
  showError(
    "Invalid product ID."
  );

  return;
}
```

Then:

```js
startEditingProduct(selectedId);
```

---

# 13.20 Event Delegation for Edit Buttons

```js
productList.addEventListener(
  "click",
  function (event) {
    const editButton =
      event.target.closest(
        '[data-action="edit"]'
      );

    if (editButton === null) {
      return;
    }

    const selectedId =
      Number(
        editButton.dataset.id
      );

    if (
      !Number.isInteger(
        selectedId
      ) ||
      selectedId <= 0
    ) {
      showError(
        "Invalid product ID."
      );

      return;
    }

    startEditingProduct(
      selectedId
    );
  }
);
```

This uses one parent listener for all Edit buttons.

---

# 13.21 Why Use `closest()`?

The user might click:

* The button itself.
* An icon inside the button.
* A text span inside the button.

Example:

```html
<button data-action="edit">
  <span>Edit</span>
</button>
```

If the user clicks the `<span>`, then:

```js
event.target
```

is the span.

Using:

```js
event.target.closest(
  '[data-action="edit"]'
)
```

finds the surrounding button.

---

# 13.22 Clearing Old Validation Messages

Before loading an item:

```js
clearError();
```

Otherwise, an unrelated old error may remain visible while the form displays valid existing data.

Example:

```js
function startEditingProduct(
  productId
) {
  clearError();

  const product =
    findProductById(productId);

  // Continue.
}
```

# 13.24 Focus and Selection

Focus the most useful input:

```js
productNameInput.focus();
```

Select its text:

```js

```

This makes the name ready to replace.

You may choose not to select the text if accidental replacement would be undesirable.

# 13.25 Handle a Missing Select Option

Suppose state contains:

```js
category: "tablets"
```

but the form has no:

```html
<option value="tablets">
```

Then:

```js
productCategorySelect.value =
  product.category;
```

may result in no valid selection.

This suggests the state and form options are inconsistent.

A safe check:

```js
productCategorySelect.value =
  product.category;

if (
  productCategorySelect.value === ""
) {
  showError(
    "The product category is no longer available."
  );
}
```

Ideally, valid state should use supported category values.

---

# 13.26 Do Not Load the ID into an Editable Input

The stable ID usually should not be editable.

Avoid:

```html
<input id="product-id" />
```

for normal user editing.

Instead, store it in JavaScript:

```js
editingProductId =
  product.id;
```

You may display the ID as read-only information:

```html
<p id="editing-id-message"></p>
```

```js
editingIdMessage.textContent =
  `Editing product ID: ${product.id}`;
```

---

# 13.27 Do Not Store the Editing Index

Less reliable:

```js
let editingProductIndex = 1;
```

If the array is sorted or filtered while editing, the item at index `1` may change.

Prefer:

```js
let editingProductId = 205;
```

Before updating, find the item again by ID.

This protects the operation from changes in array position.

---

# 13.28 Loading a Copy Versus Loading Direct Values

This function reads values from the product:

```js
function loadProductIntoForm(product) {
  productNameInput.value =
    product.name;
}
```

The input receives a primitive string value.

Changing the input does not mutate the object.

You do not need to copy the entire product merely to populate primitive form controls.

However, more complex nested data may require careful copying.

---

# 13.29 Nested Object Warning

Suppose:

```js
{
  id: 1,
  name: "Laptop",
  supplier: {
    name: "Tech Supplies"
  }
}
```

If you assign the nested object directly to another state object, references may be shared.

For normal HTML input fields:

```js
supplierNameInput.value =
  product.supplier.name;
```

only the string is loaded, so the product remains unchanged.

# 13.30 Three Complete Examples

## Example 1: Load an Object into Plain Variables

### Problem

Find a product and copy its current values into variables representing form state.

### Input Data

```js
const products = [
  {
    id: 101,
    name: "Laptop",
    price: 50000,
  },
  {
    id: 205,
    name: "Phone",
    price: 25000,
  },
];
```

### Complete Code

```js
const products = [
  {
    id: 101,
    name: "Laptop",
    price: 50000,
  },
  {
    id: 205,
    name: "Phone",
    price: 25000,
  },
];

let editingProductId = null;

let formName = "";
let formPrice = 0;

function findProductById(
  productId
) {
  for (
    let index = 0;
    index < products.length;
    index++
  ) {
    if (
      products[index].id ===
      productId
    ) {
      return products[index];
    }
  }

  return null;
}

function startEditingProduct(
  productId
) {
  const product =
    findProductById(productId);

  if (product === null) {
    console.log(
      "Product not found."
    );

    return;
  }

  editingProductId =
    product.id;

  formName =
    product.name;

  formPrice =
    product.price;

  console.log(
    editingProductId
  );

  console.log(formName);
  console.log(formPrice);
}

startEditingProduct(205);
```

### Expected Console Output

```text
205
Phone
25000
```

### Final Application State

The products array is unchanged.

### Final UI-Like State

```js
editingProductId = 205;
formName = "Phone";
formPrice = 25000;
```

### Common Mistakes

* Changing the product while trying only to load values.
* Storing the array index instead of ID.
* Not handling a missing item.

### Small Modification Challenge

Add quantity and category variables.

## Example 3: Complete Edit-Selection Interface

### Problem

Build a safer edit-selection interface that:

* Uses event delegation.
* Handles stale IDs.
* Shows edit mode.
* Loads every field.
* Shows a Cancel button.
* Scrolls to the form.

### Complete Code

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />

  <meta
    name="viewport"
    content="width=device-width, initial-scale=1.0"
  />

  <title>Edit Product Selection</title>

  <style>
    * {
      box-sizing: border-box;
```

# 13.31 Loading Values Versus Saving Values

| Action                | State Changes? |         DOM Changes? |
| --------------------- | -------------: | -------------------: |
| Find product          |             No |      Not necessarily |
| Load form values      |             No |                  Yes |
| Change input manually |             No |                  Yes |
| Validate form         |             No | Error DOM may change |
| Submit valid update   |            Yes |     Yes after render |
| Cancel edit           |             No |                  Yes |

This distinction prevents accidental state mutation.

# 13.32 Common Beginner Mistakes

## Mistake 1: Updating State on Edit Click

Incorrect:

```js
product.name =
  productNameInput.value;
```

when Edit is first clicked.

The user has not submitted changes yet.

---

## Mistake 2: Forgetting the Editing ID

The application cannot reliably determine which object to save later.

---

## Mistake 3: Storing the Index

Array order may change.

Store the stable ID.

---

## Mistake 4: Using `.value` for Checkbox State

Incorrect:

```js
productActiveInput.value =
  product.isActive;
```

Correct:

```js
productActiveInput.checked =
  product.isActive;
```

---

## Mistake 5: Using Display Text for Select Values

The option value must match state.

```html
<option value="phones">
  Phones
</option>
```

Use:

```js
select.value = "phones";
```

not:

```js
select.value = "Phones";
```

---

## Mistake 6: Not Handling Missing Products

An old or invalid button may reference an item no longer in state.

---

## Mistake 7: Regenerating an ID

Editing should preserve the existing ID.

---

## Mistake 8: Resetting the Form Immediately After Loading

The selected values disappear.

---

## Mistake 9: Updating State on Every Input Event

This makes cancellation harder and may store invalid temporary data.

---

## Mistake 10: Leaving Create-Mode Labels Visible

The user may think clicking Submit will add a new product instead of updating one.

## Key Takeaways

* Update begins by loading the selected item into the form.
* The product should be found using its stable ID.
* `editingProductId` records which item is being edited.
* Form values are temporary UI state until submission.
* Loading values does not mutate the product.
* Text and number inputs use `.value`.
* Checkboxes use `.checked`.
* Select values must match option values.
* IDs should not be editable.
* Editing IDs are safer than editing indexes.
