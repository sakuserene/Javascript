* `logic.js` → CRUD functions
* `ui.js` → DOM rendering
* `main.js` → app entry point

# 1. The Core Problem

## What You Will Learn

Why putting everything in one file breaks real applications.

---

## Bad Example (Single File)

```js
// 1000+ lines in one file ❌

let items = [];

function createItem() {}
function updateItem() {}
function deleteItem() {}

function render() {}
function handleClick() {}
function validate() {}
```

---

## Problems

```text
❌ Hard to read
❌ Hard to debug
❌ Hard to scale
❌ Hard to reuse
❌ Everything is mixed
```

---

# 2. What is Modular JavaScript?

## Concept Explanation

Modular JavaScript means:

```text
Split your code into smaller files (modules)
```

Each file has **one responsibility**.

---

## Real-World Analogy

```text
Restaurant:

Chef → cooks
Waiter → serves
Cashier → handles money

NOT one person doing everything ❌
```

---

# 3. Separation by Responsibility

## Good Structure

```text
Project/

state.js      → stores data
logic.js      → CRUD operations
ui.js         → rendering
main.js       → connects everything
```

---

## Why This Works

```text
✔ Clean
✔ Scalable
✔ Maintainable
✔ Testable
```

---

# 4. Designing a Real Project Structure

## Example Folder

```text
project/
│
├── index.html
├── main.js
├── state.js
├── logic.js
├── ui.js
└── utils.js
```

---

## Role of Each File

| File     | Responsibility |
| -------- | -------------- |
| state.js | holds state    |
| logic.js | CRUD functions |
| ui.js    | DOM rendering  |
| main.js  | event wiring   |
| utils.js | helpers        |

# 5. Example Breakdown

---

## state.js

```js
export let items = [];

export function setItems(newItems) {
  items = newItems;
}
```

---

## logic.js

```js
import { items, setItems } from "./state.js";

export function createItem(data) {
  const newItem = {
    id: Date.now(),
    ...data
  };

  setItems([...items, newItem]);
}
```

---

## ui.js

```js
import { items } from "./state.js";

export function render(listElement) {
  listElement.innerHTML = "";

  items.forEach(item => {
    const li = document.createElement("li");
    li.textContent = item.name;
    listElement.appendChild(li);
  });
}
```

---

## main.js

```js
import { createItem } from "./logic.js";
import { render } from "./ui.js";

const form = document.querySelector("form");
const list = document.getElementById("list");

form.addEventListener("submit", (e) => {
  e.preventDefault();

  createItem({ name: "Item" });
  render(list);
});
```

# 6. Three Complete Examples

---

## Example 1: Basic (No Modules)

```js
let numbers = [];

function addNumber(n) {
  numbers.push(n);
}
```

---

## Example 2: Split Logic

```js
// logic.js
export function addNumber(arr, n) {
  return [...arr, n];
}
```

---

## Example 3: Full Modular App

```text
User submits form
→ main.js handles event
→ logic.js updates state
→ ui.js renders
```

---

# 7. Flow of a Modular App

```text
User Action
→ main.js (event handler)
→ logic.js (state update)
→ state.js (store data)
→ ui.js (render)
→ DOM updates
```

# 8. Why Modular Design Matters

## Benefits

### 1. Maintainability

```text
Change one file without breaking others
```

---

### 2. Reusability

```text
Use logic.js in another project
```

---

### 3. Scalability

```text
Small app → Large system
```

---

### 4. Debugging

```text
Bug in UI? → check ui.js only
```

---

# 9. Common Mistakes

❌ Putting everything in one file
❌ Mixing DOM + logic
❌ Not organizing folders
❌ Tight coupling between files

---

# 10. Best Practices

* One responsibility per file
* Use clear naming
* Keep files small
* Separate state, logic, UI
* Use modules (import/export)
