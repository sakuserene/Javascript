# 4. State and DOM Synchronization

## 4.1 What You Will Learn

This subtopic connects the previous two lessons.

You already learned:

* JavaScript state stores application data.
* DOM state represents what the user sees.

Now you will learn how to keep them synchronized.

### Why This Matters

A CRUD application has two important realities:

1. The data stored in JavaScript.
2. The information displayed in the browser.

If those two realities disagree, the application becomes unreliable.

Example:

```js
let products = [
  {
    id: 1,
    name: "Laptop",
  },
];
```

But the DOM displays:

```text
1. Phone
```

The application now has conflicting information.

The correct solution is to treat the JavaScript state as the source of truth and render the DOM from it.

---

# 4.2 What Does Synchronization Mean?

Synchronization means:

> Keeping JavaScript state and the displayed DOM consistent with each other.

If state contains:

```js
[
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
]
```

Then the DOM should display:

```text
1. Laptop
2. Phone
```

If the phone is deleted from state, the DOM should become:

```text
1. Laptop
```

If a keyboard is added to state, the DOM should become:

```text
1. Laptop
2. Keyboard
```

The DOM should always reflect the current state.

---

# 4.3 The Correct Data Flow

A reliable CRUD application usually follows this flow:

```text
User action
    ↓
Event handler runs
    ↓
Validate the action
    ↓
Update JavaScript state
    ↓
Call the render function
    ↓
DOM is rebuilt from state
    ↓
User sees the updated result
```

Example:

```text
User submits "Keyboard"
    ↓
Submit event handler runs
    ↓
Input is validated
    ↓
Keyboard object is added to products
    ↓
renderProducts() runs
    ↓
The product list is rebuilt
    ↓
Keyboard appears in the browser
```

This flow should become a habit.

---

# 4.4 State First, DOM Second

The recommended order is:

```text
1. Update state
2. Render state into the DOM
```

Example:

```js
products.push({
  id: 3,
  name: "Keyboard",
});

renderProducts();
```

Do not manually add a new `<li>` as the only update while leaving the state unchanged.

Incorrect approach:

```js
const listItem =
  document.createElement("li");

listItem.textContent = "Keyboard";

productList.appendChild(listItem);
```

This changes the DOM only.

The state still does not contain the keyboard.

Correct approach:

```js
products.push({
  id: 3,
  name: "Keyboard",
});

renderProducts();
```

---

# 4.5 Why State Should Be the Source of Truth

The source of truth is the place the application trusts as the correct data.

For this course, the source of truth will usually be an array of objects.

```js
let products = [
  {
    id: 1,
    name: "Laptop",
  },
];
```

The DOM should be generated from the array.

This creates a simple rule:

> If you want to know what products exist, inspect the state, not the rendered HTML.

This is useful because state can contain more information than the DOM displays.

```js
{
  id: 1,
  name: "Laptop",
  price: 50000,
  quantity: 2,
  category: "Computers",
  isActive: true
}
```

The DOM may display only:

```text
Laptop — KSh 50,000
```

The full object remains available in state.

---

# 4.6 What Is Rendering?

Rendering means:

> Reading application state and creating or updating the DOM so the user can see that state.

Example state:

```js
const products = [
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
];
```

Rendering produces DOM output such as:

```text
Products

1. Laptop
2. Phone
```

A render function usually:

1. Selects a container.
2. Clears old displayed content.
3. Checks whether the state is empty.
4. Loops through the current state.
5. Creates DOM elements.
6. inserts safe text.
7. Appends elements to the container.

---

# 4.7 What Is Re-Rendering?

Re-rendering means:

> Running the render function again after the state changes.

Initial state:

```js
let products = [
  {
    id: 1,
    name: "Laptop",
  },
];
```

Initial render:

```text
1. Laptop
```

State changes:

```js
products.push({
  id: 2,
  name: "Phone",
});
```

Re-render:

```js
renderProducts();
```

Updated DOM:

```text
1. Laptop
2. Phone
```

The same render function is reused.

---

# 4.8 A Basic Render Function

HTML:

```html
<ul id="product-list"></ul>
```

JavaScript:

```js
let products = [
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
];

const productList =
  document.getElementById("product-list");

function renderProducts() {
  productList.innerHTML = "";

  for (
    let index = 0;
    index < products.length;
    index++
  ) {
    const product = products[index];

    const listItem =
      document.createElement("li");

    listItem.textContent =
      `${product.id}. ${product.name}`;

    productList.appendChild(listItem);
  }
}

renderProducts();
```

Expected DOM output:

```text
1. Laptop
2. Phone
```

---

# 4.9 Why the Container Is Cleared

The render function begins with:

```js
productList.innerHTML = "";
```

This removes the old rendered elements.

Then the function rebuilds the list from the current state.

Without clearing, repeated renders cause duplicates.

Incorrect:

```js
function renderProducts() {
  for (
    let index = 0;
    index < products.length;
    index++
  ) {
    const listItem =
      document.createElement("li");

    listItem.textContent =
      products[index].name;

    productList.appendChild(listItem);
  }
}
```

First render:

```text
Laptop
Phone
```

Second render:

```text
Laptop
Phone
Laptop
Phone
```

Third render:

```text
Laptop
Phone
Laptop
Phone
Laptop
Phone
```

This is called duplicate rendering.

---

# 4.10 Is Clearing with `innerHTML` Safe?

This line is safe in this specific use:

```js
productList.innerHTML = "";
```

It inserts no user content.

It simply clears the container.

Unsafe use would be:

```js
productList.innerHTML = userInput;
```

Safer user-content handling:

```js
listItem.textContent = userInput;
```

Therefore:

* Using `innerHTML = ""` to clear a container is acceptable.
* Using `innerHTML` to insert untrusted user input is unsafe.

---

# 4.11 Rendering an Empty State

If the array is empty:

```js
let products = [];
```

The page should not display a confusing blank area.

Instead, display an empty-state message.

```js
function renderProducts() {
  productList.innerHTML = "";

  if (products.length === 0) {
    const emptyMessage =
      document.createElement("li");

    emptyMessage.textContent =
      "No products available.";

    productList.appendChild(emptyMessage);

    return;
  }

  for (
    let index = 0;
    index < products.length;
    index++
  ) {
    const product = products[index];

    const listItem =
      document.createElement("li");

    listItem.textContent =
      `${product.id}. ${product.name}`;

    productList.appendChild(listItem);
  }
}
```

Expected DOM:

```text
No products available.
```

---

# 4.12 Why `return` Is Used in Empty-State Rendering

Inside the empty-state check:

```js
if (products.length === 0) {
  const emptyMessage =
    document.createElement("li");

  emptyMessage.textContent =
    "No products available.";

  productList.appendChild(emptyMessage);

  return;
}
```

The `return` stops the function.

Without it, the loop would still be reached.

In this case, the loop would not run because the array is empty, but the early return makes the intention clear:

```text
The empty state has been handled.
Stop rendering.
```

---

# 4.13 Updating State Without Rendering

Example:

```js
products.push({
  id: 3,
  name: "Keyboard",
});
```

Final state:

```js
[
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
  {
    id: 3,
    name: "Keyboard",
  },
]
```

If `renderProducts()` is not called, the DOM may still display:

```text
1. Laptop
2. Phone
```

The state is correct.

The DOM is outdated.

---

# 4.14 Updating the DOM Without State

Example:

```js
const listItem =
  document.createElement("li");

listItem.textContent = "Keyboard";

productList.appendChild(listItem);
```

The DOM displays:

```text
Laptop
Phone
Keyboard
```

But state still contains:

```js
[
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
]
```

A later re-render will remove the keyboard because the state does not contain it.

```js
renderProducts();
```

DOM after re-render:

```text
Laptop
Phone
```

This proves why DOM-only changes are unreliable.

---

# 4.15 The Render Function Should Not Invent State

A render function should display current state.

It should not create new state records.

Incorrect:

```js
function renderProducts() {
  products.push({
    id: 3,
    name: "Keyboard",
  });

  // Render products.
}
```

Every render adds another keyboard.

The render function has two responsibilities:

1. Mutating state.
2. Displaying state.

This creates bugs.

Better:

```js
function addProduct() {
  products.push({
    id: 3,
    name: "Keyboard",
  });
}

function renderProducts() {
  // Display products only.
}
```

Use them in this order:

```js
addProduct();
renderProducts();
```

---

# 4.16 Separate State Logic from DOM Logic

A useful design is:

```js
function addProduct(product) {
  products.push(product);
}

function renderProducts() {
  productList.innerHTML = "";

  // Render current products.
}
```

The first function changes state.

The second function changes the DOM.

This separation makes code easier to:

* Read.
* test.
* debug.
* reuse.
* refactor.

---

# 4.17 Preserving State Across Event Handlers

State should normally be declared outside event handlers.

Correct:

```js
let products = [];

productForm.addEventListener(
  "submit",
  function (event) {
    event.preventDefault();

    products.push({
      id: 1,
      name: "Laptop",
    });

    renderProducts();
  }
);
```

Incorrect:

```js
productForm.addEventListener(
  "submit",
  function (event) {
    event.preventDefault();

    let products = [];

    products.push({
      id: 1,
      name: "Laptop",
    });

    renderProducts();
  }
);
```

In the incorrect example, every submission creates a new empty array.

Previous products are lost.

---

# 4.18 Derived DOM State

Some displayed values can be calculated from the main state.

Example state:

```js
const products = [
  {
    id: 1,
    name: "Laptop",
  },
  {
    id: 2,
    name: "Phone",
  },
];
```

Product count:

```js
products.length
```

Expected value:

```text
2
```

DOM:

```html
<p id="product-count"></p>
```

Rendering:

```js
productCount.textContent =
  `Total products: ${products.length}`;
```

The count is derived from the state.

You usually do not need a separate variable such as:

```js
let productCount = 2;
```

That creates another value that must be synchronized.

Prefer:

```js
products.length
```

---

# 4.19 Temporary UI State

Not all state represents application records.

Some state controls the user interface.

Example:

```js
let editingProductId = null;
```

This means:

```text
No product is currently being edited.
```

During editing:

```js
editingProductId = 2;
```

This means:

```text
The product with ID 2 is being edited.
```

This type of state will later help distinguish between:

* Create mode.
* Edit mode.

It should also remain synchronized with the DOM.

For example, when editing starts:

```text
Form title: Edit Product
Submit button: Update Product
Cancel button: Visible
```

#### Defining but not calling the function


Incorrect:

Add a product to state when a button is clicked, then re-render the list.

```js
let products = [
```

## Example 3: Empty State, Count, and Synchronization

### Problem

Create a small product interface that:

* Displays products.
* Shows the product count.
* Shows an empty-state message.
* Deletes all products.
* Resets the list.

### Complete Code

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />

  <meta
    name="viewport"
    content="width=device-width, initial-scale=1.0"
  />

  <title>State Synchronization</title>

  <style>
    body {
      max-width: 650px;
      margin: 0 auto;
      padding: 24px;
      font-family: Arial, sans-serif;
    }

    .actions {
      display: flex;
      gap: 8px;
      margin-bottom: 16px;
    }

    .product-item {
      margin-bottom: 8px;
      padding: 10px;
      border: 1px solid #cccccc;
      border-radius: 4px;
    }

    .empty-message {
      padding: 12px;
      border: 1px dashed #999999;
    }
  </style>
</head>

<body>
  <h1>Product Inventory</h1>

  <p id="product-count"></p>

  <div class="actions">
    <button id="clear-button" type="button">
      Delete All
    </button>

    <button id="reset-button" type="button">
      Reset Products
    </button>
  </div>

  <ul id="product-list"></ul>

  <script>
    const initialProducts = [
      {
        id: 1,
        name: "Laptop",
      },
      {
        id: 2,
        name: "Phone",
      },
      {
        id: 3,
        name: "Keyboard",
      },
    ];

    let products = [];

    const productCount =
      document.getElementById("product-count");

    const productList =
      document.getElementById("product-list");

    const clearButton =
      document.getElementById("clear-button");

    const resetButton =
      document.getElementById("reset-button");

    function copyInitialProducts() {
      products = [];

      for (
        let index = 0;
        index < initialProducts.length;
        index++
      ) {
        products.push({
          id: initialProducts[index].id,
          name: initialProducts[index].name,
        });
      }
    }

    function renderProducts() {
      productList.innerHTML = "";

      productCount.textContent =
        `Total products: ${products.length}`;

      if (products.length === 0) {
        const emptyMessage =
          document.createElement("li");

        emptyMessage.classList.add(
          "empty-message"
        );

        emptyMessage.textContent =
          "No products available.";

        productList.appendChild(emptyMessage);

        clearButton.disabled = true;

        return;
      }

      clearButton.disabled = false;

      for (
        let index = 0;
        index < products.length;
        index++
      ) {
        const product = products[index];

        const listItem =
          document.createElement("li");

        listItem.classList.add(
          "product-item"
        );

        listItem.textContent =
          `${product.id}. ${product.name}`;

        productList.appendChild(listItem);
      }
    }

    clearButton.addEventListener(
      "click",
      function () {
        products = [];

        renderProducts();
      }
    );

    resetButton.addEventListener(
      "click",
      function () {
        copyInitialProducts();

        renderProducts();
      }
    );

    copyInitialProducts();
    renderProducts();
  </script>
</body>
</html>
```

### Step-by-Step Explanation

#### Step 1: Store initial reference data

```js
const initialProducts = [
  {
    id: 1,
    name: "Laptop",
  },
];
```

#### Step 2: Store active application state

```js
let products = [];
```

This array will change.

#### Step 3: Copy the initial products

```js
function copyInitialProducts() {
  products = [];

  for (...) {
    products.push({
      id: initialProducts[index].id,
      name: initialProducts[index].name,
    });
  }
}
```

This avoids making both variables point to the exact same array.

#### Step 4: Render the count

```js
productCount.textContent =
  `Total products: ${products.length}`;
```

The count is derived from state.

#### Step 5: Handle empty state

```js
if (products.length === 0) {
  // Show empty message.
}
```

#### Step 6: Synchronize button state

```js
clearButton.disabled = true;
```

The Delete All button is disabled when nothing can be deleted.

#### Step 7: Delete all products

```js
products = [];
renderProducts();
```

#### Step 8: Reset products

```js
copyInitialProducts();
renderProducts();
```

# 4.21 Plain Rendering Flow

The reusable rendering pattern is:

```js
function renderItems() {
  itemList.innerHTML = "";

  if (items.length === 0) {
    const emptyMessage =
      document.createElement("li");

    emptyMessage.textContent =
      "No items available.";

    itemList.appendChild(emptyMessage);

    return;
  }

  for (
    let index = 0;
    index < items.length;
    index++
  ) {
    const item = items[index];

    const listItem =
      document.createElement("li");

    listItem.textContent = item.name;

    itemList.appendChild(listItem);
  }
}
```

After a state change:

```js
items.push(newItem);

renderItems();
```

Or:

```js
items.splice(indexToDelete, 1);

renderItems();
```

Or:

```js
items[indexToUpdate].name =
  "Updated Item";

renderItems();
```

# 4.22 Common Synchronization Mistakes

## Mistake 1: State Change Without Render

```js
products.push(newProduct);
```

The state changes, but the user sees old DOM content.

---

## Mistake 2: DOM Change Without State Change

```js
productList.appendChild(listItem);
```

The user sees the new element, but the state does not contain it.

---

## Mistake 3: Not Clearing Before Rendering

```js
function renderProducts() {
  for (...) {
    productList.appendChild(listItem);
  }
}
```

Repeated rendering creates duplicates.

---

## Mistake 4: Clearing Inside the Loop

```js
for (...) {
  productList.innerHTML = "";
}
```

Earlier rendered items are repeatedly removed.

---

## Mistake 5: Recreating State Inside an Event Handler

```js
button.addEventListener("click", function () {
  let products = [];
});
```

Previous state is lost.

---

## Mistake 6: Render Function Mutates State

```js
function renderProducts() {
  products.push(newProduct);
}
```

Rendering unexpectedly changes data.

---

## Mistake 7: Maintaining Duplicate Derived State

```js
let products = [];
let productCount = 0;
```

The count can become incorrect.

Prefer:

```js
products.length
```

---

## Mistake 8: Using DOM Indexes as IDs

The DOM position may change after filtering, sorting, or deletion.

Use stable object IDs.

---

## Mistake 9: Repeated Event Listeners

Attaching listeners during every render may cause one click to run several handlers.

This will later be solved using event delegation.

---

## Mistake 10: Losing Edit State During Re-Rendering

A full render may clear form fields or replace elements while the user is editing.

Later lessons will use an `editingItemId` variable and controlled form updates to preserve edit mode.

## Key Takeaways

* JavaScript state and DOM state are separate.
* Synchronization keeps them consistent.
* The JavaScript array should normally be the source of truth.
* User actions should update state first.
* The DOM should be rendered after state changes.
* Rendering means creating DOM output from state.
* Re-rendering means rendering again after a change.
* A reusable render function avoids duplicated DOM logic.
* The display container should be cleared before rebuilding.
* Failing to clear the container causes duplicate rendering.
* An empty state should display a helpful message.
* Derived values such as counts should be calculated from state.
* Render functions should display state, not mutate it.
* State should be declared outside event handlers so it persists.
* Business logic and rendering logic should be separated where possible.
* Future CRUD operations will follow the same pattern:

```text
User action
→ validate
→ update state
→ render
→ update DOM
```
