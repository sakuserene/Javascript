# 9. Validating Item Form Values

## 9.1 What You Will Learn

Validation means checking whether data follows the application's rules before accepting it.

Suppose a user submits:

```js
{
  name: "",
  price: -500,
  quantity: -2
}
```

This object should not be added to an inventory.

Without validation, state could contain invalid data:

```js
[
  {
    id: 1,
    name: "",
    price: -500,
    quantity: -2
  }
]
```

Once invalid data enters state, every later operation becomes harder.

For example:

* Rendering may display blank product names.
* Inventory totals may become incorrect.
* Search may fail.
* Updates may preserve invalid values.
* Reports may become unreliable.

Validation protects the application state.

---

# 9.2 Validation Must Happen Before State Mutation

Incorrect order:

```js
products.push(newProduct);

if (newProduct.name === "") {
  showError("Name is required.");
}
```

The invalid item has already entered state.

Correct order:

```js
if (newProduct.name === "") {
  showError("Name is required.");
  return;
}

products.push(newProduct);
```

The rule is:

```text
Read values
→ normalize values
→ validate values
→ create or accept object
→ update state
→ render
```

Do not mutate state before validation succeeds.

---

# 9.3 Browser Validation Versus JavaScript Validation

HTML can provide basic validation.

```html
<input
  id="product-name"
  type="text"
  required
/>
```

Number constraints:

```html
<input
  id="product-price"
  type="number"
  min="0"
  required
/>
```

These browser features are useful, but JavaScript validation is still important.

JavaScript validation allows you to:

* Apply custom rules.
* Display custom messages.
* Combine multiple conditions.
* Protect state even when HTML attributes are changed.
* Validate values from sources other than the form.
* Keep business rules inside JavaScript.

Use both when appropriate.

---

# 9.4 Required Text Validation

Read and trim the value:

```js
const name =
  productNameInput.value.trim();
```

Validate it:

```js
if (name === "") {
  showError("Product name is required.");
  return;
}
```

This rejects:

```text
""
```

and:

```text
"      "
```

because trimming whitespace produces an empty string.

---

# 9.5 Why `.trim()` Matters

Without trimming:

```js
const name =
  productNameInput.value;
```

A whitespace-only value may pass this check:

```js
if (name === "") {
  // This may not run for "   ".
}
```

With trimming:

```js
const name =
  productNameInput.value.trim();
```

Then:

```js
"   ".trim();
```

returns:

```text
""
```

Now validation works correctly.

---

# 9.6 Minimum Name Length

You may require at least two characters.

```js
if (name.length < 2) {
  showError(
    "Product name must contain at least 2 characters."
  );

  return;
}
```

Examples:

| Name       | Valid? |
| ---------- | ------ |
| `""`       | No     |
| `"A"`      | No     |
| `"TV"`     | Yes    |
| `"Laptop"` | Yes    |

Application rules determine the correct minimum.

---

# 9.7 Maximum Name Length

Long values may make the interface difficult to use.

```js
if (name.length > 80) {
  showError(
    "Product name cannot exceed 80 characters."
  );

  return;
}
```

HTML can also include:

```html
<input
  id="product-name"
  maxlength="80"
/>
```

JavaScript should still enforce important business rules.

---

# 9.8 Reading Numeric Inputs Safely

A common mistake is:

```js
const price =
  Number(productPriceInput.value);
```

If the input is empty:

```js
Number("");
```

returns:

```text
0
```

This may incorrectly treat an empty field as a valid zero.

Therefore, inspect the raw value before conversion.

```js
const priceText =
  productPriceInput.value.trim();

if (priceText === "") {
  showError("Price is required.");
  return;
}

const price = Number(priceText);
```

This distinguishes:

* Empty input.
* An actual value of `0`.

---

# 9.9 Detecting `NaN`

`NaN` means:

```text
Not a Number
```

Example:

```js
const price = Number("hello");
```

Result:

```text
NaN
```

Check with:

```js
Number.isNaN(price);
```

Example:

```js
if (Number.isNaN(price)) {
  showError("Price must be a valid number.");
  return;
}
```

Do not use:

```js
price === NaN
```

That comparison is always false.

---

# 9.10 Why `NaN === NaN` Is False

JavaScript treats `NaN` as a special invalid numeric result.

```js
console.log(NaN === NaN);
```

Expected output:

```text
false
```

Use:

```js
Number.isNaN(value)
```

Example:

```js
console.log(
  Number.isNaN(Number("hello"))
);
```

Expected output:

```text
true
```

---

# 9.11 Validating Price

A price may need these rules:

* Required.
* Numeric.
* Zero or greater.
* Finite.
* At most two decimal places, depending on the application.

Basic validation:

```js
const priceText =
  productPriceInput.value.trim();

if (priceText === "") {
  showError("Price is required.");
  return;
}

const price = Number(priceText);

if (Number.isNaN(price)) {
  showError("Price must be a number.");
  return;
}

if (price < 0) {
  showError("Price cannot be negative.");
  return;
}
```

---

# 9.12 Is Zero a Valid Price?

This depends on the application.

A free product may allow:

```js
price: 0
```

Then use:

```js
if (price < 0)
```

If every product must have a positive price, use:

```js
if (price <= 0) {
  showError(
    "Price must be greater than zero."
  );

  return;
}
```

Validation rules must reflect real business requirements.

---

# 9.13 Validating Quantity

Quantity normally should be:

* Required.
* Numeric.
* An integer.
* Zero or greater.

```js
const quantityText =
  productQuantityInput.value.trim();

if (quantityText === "") {
  showError("Quantity is required.");
  return;
}

const quantity =
  Number(quantityText);

if (Number.isNaN(quantity)) {
  showError(
    "Quantity must be a valid number."
  );

  return;
}

if (!Number.isInteger(quantity)) {
  showError(
    "Quantity must be a whole number."
  );

  return;
}

if (quantity < 0) {
  showError(
    "Quantity cannot be negative."
  );

  return;
}
```

---

# 9.14 Why Quantity Should Usually Be an Integer

Inventory quantities normally represent whole units.

Valid:

```text
0
1
2
15
```

Usually invalid:

```text
1.5
2.75
```

Check with:

```js
Number.isInteger(quantity)
```

Example:

```js
Number.isInteger(2);
```

returns:

```text
true
```

Example:

```js
Number.isInteger(2.5);
```

returns:

```text
false
```

Some applications track weights or measurements, where decimals may be valid. Again, rules depend on the domain.

# 9.16 Checkbox Validation

A checkbox may simply provide a boolean:

```js
const isActive =
  productActiveInput.checked;
```

No validation may be needed.

However, agreement checkboxes may be required.

```js
if (!termsCheckbox.checked) {
  showError(
    "You must accept the terms."
  );

  return;
}
```

For a product's active status, both `true` and `false` may be valid.

# 9.17 Showing Validation Errors

HTML:

```html
<p
  id="error-message"
  role="alert"
></p>
```

JavaScript:

```js
const errorMessage =
  document.getElementById(
    "error-message"
  );

function showError(message) {
  errorMessage.textContent = message;
}
```

Usage:

```js
showError("Product name is required.");
```

Expected DOM:

```text
Product name is required.
```

Using `textContent` safely displays the error.

---

# 9.18 Clearing Validation Errors

Old errors should disappear before a new validation attempt.

```js
function clearError() {
  errorMessage.textContent = "";
}
```

At the beginning of submission:

```js
productForm.addEventListener(
  "submit",
  function (event) {
    event.preventDefault();

    clearError();

    // Read and validate.
  }
);
```

You can also clear errors after successful submission.

# 9.19 Focusing the Invalid Input

When validation fails, focus the relevant field.

```js
if (name === "") {
  showError("Product name is required.");

  productNameInput.focus();

  return;
}
```

For price:

```js
if (price < 0) {
  showError("Price cannot be negative.");

  productPriceInput.focus();

  return;
}
```

This helps the user correct the problem quickly.

---

# 9.20 Selecting Invalid Text

For text inputs, you can select existing content:

```js
productNameInput.select();
```

Example:

```js
if (name.length > 80) {
  showError(
    "Product name is too long."
  );

  productNameInput.focus();
  productNameInput.select();

  return;
}
```

The user can immediately replace the invalid value.

---

# 9.21 Returning Early

Validation often uses early returns.

```js
if (name === "") {
  showError("Name is required.");
  return;
}

if (price < 0) {
  showError("Price cannot be negative.");
  return;
}

products.push(newProduct);
```

When `return` runs:

* The event handler stops.
* No object is added.
* No ID is consumed.
* The form is not reset.
* The existing state remains unchanged.

This is called a guard clause.

---

# 9.22 Guard Clauses

A guard clause rejects invalid conditions early.

```js
if (conditionIsInvalid) {
  showError("Validation failed.");
  return;
}
```

This keeps valid creation logic at the end.

```js
const newProduct = {
  id: nextProductId,
  name,
  price,
  quantity,
};

products.push(newProduct);
```

Without guard clauses, code may become deeply nested.

Less readable:

```js
if (name !== "") {
  if (price >= 0) {
    if (quantity >= 0) {
      products.push(newProduct);
    }
  }
}
```

More readable:

```js
if (name === "") {
  return;
}

if (price < 0) {
  return;
}

if (quantity < 0) {
  return;
}

products.push(newProduct);
```

---

# 9.23 Validate Before Creating the Final Object

You may first read normalized values:

```js
const name =
  productNameInput.value.trim();

const priceText =
  productPriceInput.value.trim();

const quantityText =
  productQuantityInput.value.trim();

const category =
  productCategorySelect.value;
```

Then validate.

Only after all values pass should you create:

```js
const newProduct = {
  id: nextProductId,
  name,
  price,
  quantity,
  category,
  isActive,
};
```

This avoids creating an object that should never enter state.

Creating a temporary object before validation is not always wrong, but creating after validation makes the flow clearer for beginners.

---

# 9.24 Validate Before Incrementing the ID

Incorrect:

```js
nextProductId++;

if (name === "") {
  return;
}
```

An invalid submission consumes an ID.

Correct:

```js
if (name === "") {
  return;
}

const newProduct = {
  id: nextProductId,
  name,
};

products.push(newProduct);

nextProductId++;
```

Only successful creation advances the ID.

---

# 9.25 Validate Before Resetting the Form

Incorrect:

```js
productForm.reset();

if (name === "") {
  showError("Name is required.");
  return;
}
```

The user loses the values they need to correct.

Correct:

```js
if (name === "") {
  showError("Name is required.");
  return;
}

products.push(newProduct);

productForm.reset();
```

Only successful submissions reset the form.

# 9.26 Validating All Product Fields

```js
function handleProductSubmit(event) {
  event.preventDefault();

  clearError();

  const name =
    productNameInput.value.trim();

  const priceText =
    productPriceInput.value.trim();

  const quantityText =
    productQuantityInput.value.trim();

  const category =
    productCategorySelect.value;

  const isActive =
    productActiveInput.checked;

  if (name === "") {
    showError(
      "Product name is required."
    );

    productNameInput.focus();
    return;
  }

  if (name.length < 2) {
    showError(
      "Product name must contain at least 2 characters."
    );

    productNameInput.focus();
    return;
  }

  if (priceText === "") {
    showError("Price is required.");

    productPriceInput.focus();
    return;
  }

  const price = Number(priceText);

  if (Number.isNaN(price)) {
    showError(
      "Price must be a valid number."
    );

    productPriceInput.focus();
    return;
  }

  if (price < 0) {
    showError(
      "Price cannot be negative."
    );

    productPriceInput.focus();
    return;
  }

  if (quantityText === "") {
    showError(
      "Quantity is required."
    );

    productQuantityInput.focus();
    return;
  }

  const quantity =
    Number(quantityText);

  if (Number.isNaN(quantity)) {
    showError(
      "Quantity must be a valid number."
    );

    productQuantityInput.focus();
    return;
  }

  if (!Number.isInteger(quantity)) {
    showError(
      "Quantity must be a whole number."
    );

    productQuantityInput.focus();
    return;
  }

  if (quantity < 0) {
    showError(
      "Quantity cannot be negative."
    );

    productQuantityInput.focus();
    return;
  }

  if (category === "") {
    showError(
      "Please select a category."
    );

    productCategorySelect.focus();
    return;
  }

  const newProduct = {
    id: nextProductId,
    name,
    price,
    quantity,
    category,
    isActive,
  };

  products.push(newProduct);

  nextProductId++;

  productForm.reset();
  productActiveInput.checked = true;
  productNameInput.focus();

  renderProducts();
}
```

# 9.27 Reusable Validation Functions

Validation can be separated into small functions.

```js
function isBlank(value) {
  return value.trim() === "";
}
```

Usage:

```js
if (isBlank(name)) {
  showError("Name is required.");
  return;
}
```

Number validation:

```js
function isValidNonNegativeNumber(value) {
  return (
    typeof value === "number" &&
    !Number.isNaN(value) &&
    value >= 0
  );
}
```

Integer validation:

```js
function isValidQuantity(value) {
  return (
    Number.isInteger(value) &&
    value >= 0
  );
}
```

Small functions make business rules easier to test.

---

# 9.28 Returning Validation Results

A validation function can return an object.

```js
function validateProductData(data) {
  if (data.name === "") {
    return {
      isValid: false,
      message: "Name is required.",
      field: "name",
    };
  }

  if (data.price < 0) {
    return {
      isValid: false,
      message:
        "Price cannot be negative.",
      field: "price",
    };
  }

  return {
    isValid: true,
    message: "",
    field: null,
  };
}
```

Usage:

```js
const result =
  validateProductData(productData);

if (!result.isValid) {
  showError(result.message);
  return;
}
```

This is a more scalable design, but direct guard clauses are easier when first learning validation.

# 9.29 Per-Field Validation Messages

Instead of one general error paragraph, each field can have its own message.

HTML:

```html
<input id="product-name" />

<p id="product-name-error"></p>
```

JavaScript:

```js
const nameError =
  document.getElementById(
    "product-name-error"
  );

nameError.textContent =
  "Product name is required.";
```

Benefits:

* Error appears near the field.
* Multiple errors can be shown at once.
* Users can correct several issues together.

A single-message approach is simpler for early lessons.

# 9.30 Showing One Error Versus All Errors

## First-error approach

```js
if (name === "") {
  showError("Name is required.");
  return;
}
```

Benefits:

* Simple.
* Easy to understand.
* Focuses the first invalid field.

## All-errors approach

```js
const errors = [];

if (name === "") {
  errors.push("Name is required.");
}

if (price < 0) {
  errors.push(
    "Price cannot be negative."
  );
}
```

Benefits:

* Displays all problems together.
* Reduces repeated submissions.

The first-error approach will be used initially.

# 9.31 Validation and the Source of Truth

The state should contain valid application data.

Good state:

```js
[
  {
    id: 1,
    name: "Laptop",
    price: 50000,
    quantity: 2,
  },
]
```

Invalid state:

```js
[
  {
    id: 1,
    name: "",
    price: -500,
    quantity: 2.75,
  },
]
```

Rendering should not be responsible for repairing invalid state.

Incorrect responsibility:

```js
function renderProducts() {
  if (product.price < 0) {
    product.price = 0;
  }
}
```

Validation should prevent the invalid value before state changes.

# 9.32 Three Complete Examples

## Example 1: Validating Values Without the DOM

### Problem

Validate a product name, price, and quantity before creating an object.

### Input Data

```js
const nameInput = "  Laptop  ";
const priceInput = "50000";
const quantityInput = "2";
```

### Complete Code

```js
const nameInput = "  Laptop  ";
const priceInput = "50000";
const quantityInput = "2";

const name = nameInput.trim();

if (name === "") {
  console.log(
    "Product name is required."
  );
} else if (priceInput.trim() === "") {
  console.log("Price is required.");
} else if (quantityInput.trim() === "") {
  console.log("Quantity is required.");
} else {
  const price = Number(priceInput);
  const quantity = Number(quantityInput);

  if (Number.isNaN(price)) {
    console.log(
      "Price must be a number."
    );
  } else if (price < 0) {
    console.log(
      "Price cannot be negative."
    );
  } else if (
    Number.isNaN(quantity)
  ) {
    console.log(
      "Quantity must be a number."
    );
  } else if (
    !Number.isInteger(quantity)
  ) {
    console.log(
      "Quantity must be a whole number."
    );
  } else if (quantity < 0) {
    console.log(
      "Quantity cannot be negative."
    );
  } else {
    const product = {
      name,
      price,
      quantity,
    };

    console.log(product);
  }
}
```

### Step-by-Step Explanation

#### Step 1: Normalize the name

```js
const name = nameInput.trim();
```

#### Step 2: Check required fields

Empty strings are rejected before conversion.

#### Step 3: Convert numeric fields

```js
const price = Number(priceInput);
```

#### Step 4: Validate numeric results

```js
Number.isNaN(price)
```

#### Step 5: Validate range and integer rules

```js
price >= 0
quantity >= 0
Number.isInteger(quantity)
```

#### Step 6: Create the object only after success

```js
const product = {
  name,
  price,
  quantity,
};
```

### Expected Console Output

```text
{
  name: "Laptop",
  price: 50000,
  quantity: 2
}
```

### Final State

No state array is used in this example.

The valid product object is:

```js
{
  name: "Laptop",
  price: 50000,
  quantity: 2
}
```

### Common Mistakes

#### Converting empty input before checking it

```js
Number("");
```

returns `0`.

#### Checking `price === NaN`

This does not work.

#### Creating the object before validating

This makes the logical order less clear.

### Small Modification Challenge

Change:

```js
const quantityInput = "2.5";
```

Confirm that the code displays:

```text
Quantity must be a whole number.
```

# 9.34 Common Beginner Mistakes

## Mistake 1: Checking Numeric Input After `Number("")`

```js
const price =
  Number(priceInput.value);
```

An empty input becomes `0`.

Check the raw string first.

---

## Mistake 2: Comparing with `NaN`

Incorrect:

```js
if (price === NaN)
```

Correct:

```js
if (Number.isNaN(price))
```

---

## Mistake 3: Validating After `push()`

Invalid state has already been created.

---

## Mistake 4: Incrementing ID Before Validation

Failed submissions create gaps.

---

## Mistake 5: Resetting on Validation Failure

The user loses values that need correction.

---

## Mistake 6: Forgetting `return`

```js
if (name === "") {
  showError("Name is required.");
}

products.push(newProduct);
```

The invalid item is still added.

---

## Mistake 7: Rejecting Zero Accidentally

```js
if (!price)
```

This treats `0` as false.

Use explicit rules:

```js
if (priceText === "")
```

and:

```js
if (price < 0)
```

---

## Mistake 8: Allowing Decimal Quantity

```js
quantity >= 0
```

alone does not reject `2.5`.

Use:

```js
Number.isInteger(quantity)
```

---

## Mistake 9: Leaving Old Error Messages Visible

Clear old errors at the beginning and after success.

---

## Mistake 10: Mutating State to Fix Invalid Values

Do not silently change:

```js
price = 0;
```

unless the application explicitly defines that behavior.

Usually, reject the invalid value and ask the user to correct it.

## Key Takeaways

* Validation checks whether input follows application rules.
* Invalid data should never enter the main state.
* Validation must happen before `push()`.
* Raw numeric input should be checked for emptiness before conversion.
* `Number("")` returns `0`.
* `Number.isNaN()` checks failed numeric conversion.
* `Number.isInteger()` checks whole-number quantities.
* Explicit comparisons are safer than broad truthy or falsy checks.
* Guard clauses stop invalid submissions early.
* Failed validation should not reset the form.
* Failed validation should not increment the ID.
* The invalid field should receive focus.
* Errors should be displayed safely with `textContent`.
* Rendering should assume that state has already been validated.
* Validation rules created here will later be reused for Update operations.
