Now it's time to understand another important topic that every frontend and backend developer works with every day:
HTTP Status Codes
Every time your application sends a request to a server, the server sends back two things:
The response data (if any)
A status code
The status code tells your application whether the request was successful or if something went wrong.

What is an HTTP Status Code?
An HTTP status code is a 3-digit number sent by the server.
It tells the client (browser or JavaScript):
"This is the result of your request."
Imagine you order food online.
After placing your order, the restaurant responds.
Possible responses:
✅ Order accepted.

❌ Item not available.

❌ Payment failed.

✅ Order delivered.
Status codes work the same way.
Instead of words, servers use numbers.

Without status codes, your application wouldn't know what happened.
Suppose you request a student record.

The status code tells your application which situation occurred.

You don't need to memorize every status code.
Instead, focus on the ones used most often.
1xx – Informational
These codes mean:
"The request has been received. Continue."
Examples:
100 Continue
101 Switching Protocols
As a frontend developer, you'll rarely use these directly.
2xx – Success
These indicate that everything worked correctly.
The most common success codes are:
200 OK
201 Created
204 No Content
Let's learn each one.
200 OK
Meaning:
The request succeeded.
Example:
GET /students
Response:
200 OK
Student data is returned.
Example using Fetch:
fetch("https://jsonplaceholder.typicode.com/users")
    .then(function(response) {
        console.log(response.status);
    });
Output:
200
When is 200 Used?
Examples:
Get all users
Get products
Get blog posts
Search students
Whenever the request succeeds and data is returned.
201 Created
Meaning:
A new resource was successfully created.
Example:
POST /students
Response:
201 Created
A new student now exists in the database.
Real-World Example
You register a new account.
Fill Registration Form

↓

POST Request

↓

Server Saves User

↓

201 Created
204 No Content
Meaning:
The request succeeded.
But there is nothing to return.
Example:
DELETE /students/5
The student is deleted.
The server doesn't need to send data back.
Response:
204 No Content
3xx – Redirection
These tell the browser:
"The resource has moved."
Examples:
301 Moved Permanently
302 Found
Example:
Old Website

↓

Redirect

↓

New Website
Modern browsers usually handle redirects automatically.

These errors mean:
The problem is on the client's side.
Usually:
Wrong URL
Invalid request
Missing login

The most common ones are:

400 Bad Request
Meaning:
The request was invalid.
Example:
You submit this JSON:
{
    "name":
}
The JSON is incomplete.
The server responds:
400 Bad Request
Another example:
A required field is missing.
401 Unauthorized
Meaning:
You are not authenticated.
Example:
User tries to access dashboard.

↓

No login token.

↓

401 Unauthorized
The server is saying:
"Please log in first."
Real-World Example
Imagine entering a cinema.
You haven't bought a ticket.
Security says:
"Please buy a ticket first."
That's 401.
403 Forbidden
Meaning:
You are logged in.
But you don't have permission.
Example:
A normal user tries to delete another user's account.
Logged In

↓

No Permission

↓

403 Forbidden
Think of it as:
"You may enter the building...
But not this room."
404 Not Found
One of the most famous status codes.
Meaning:
The requested resource does not exist.
Example:
GET /students/9999
If student 9999 doesn't exist:
404 Not Found
Another example:
Typing a wrong website address.
Real-World Analogy
You ask a librarian:
"Can I have Book #999?"
The librarian replies:
"We don't have that book."
That's a 404.
405 Method Not Allowed
Meaning:
The endpoint exists.
But the HTTP method is not allowed.
Example:
Suppose an API only supports:
GET /students
You send:
DELETE /students
Response:
405 Method Not Allowed
5xx – Server Errors
These mean:
The request was correct.
The problem is on the server.
Common examples:
500 Internal Server Error
502 Bad Gateway
503 Service Unavailable
500 Internal Server Error
Meaning:
Something unexpected happened on the server.
Example:
Database crashes.
Server code throws an exception.
The user did nothing wrong.
Real-World Analogy
You order food.
The restaurant kitchen catches fire.
The customer didn't cause the problem.
The restaurant failed.
That's like a 500 error.
Checking Status Codes in JavaScript
You can inspect the status code.
fetch("https://jsonplaceholder.typicode.com/users")
    .then(function(response) {

        console.log(response.status);

    });
Output:
200
Checking if a Request Was Successful
Instead of checking only the number, you can use:
fetch("https://jsonplaceholder.typicode.com/users")
    .then(function(response) {

        console.log(response.ok);

    });
Output:
true
response.ok is true for status codes between 200 and 299.
Otherwise:
false
Using async/await
async function getUsers() {

    const response = await fetch("https://jsonplaceholder.typicode.com/users");

    console.log(response.status);

    console.log(response.ok);

}

getUsers();
Handling Errors Properly
Example:
async function getUsers() {

    try {

        const response = await fetch("https://jsonplaceholder.typicode.com/users");

        if (!response.ok) {

            throw new Error("Request Failed");

        }

        const users = await response.json();

        console.log(users);

    } catch (error) {

        console.log(error.message);

    }

}

getUsers();
Notice something important.
fetch() only rejects its Promise for network errors (such as no internet connection).
If the server responds with 404 or 500, the Promise is still fulfilled.
That's why we manually check:
response.ok
or
response.status

Common Beginner Mistakes
Mistake 1
Assuming every successful request returns 200.
Not true.
Creating a resource usually returns:
201 Created
Deleting a resource often returns:
204 No Content
Mistake 2
Thinking fetch() throws an error for 404.
It doesn't.
You must check:
response.ok
or
response.status
Mistake 3
Confusing 401 and 403.
Remember:
401
You are not logged in.
403
You are logged in, but not allowed.
