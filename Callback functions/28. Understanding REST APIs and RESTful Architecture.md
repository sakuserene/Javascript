What is REST?
REST stands for:
Representational State Transfer
That name sounds difficult.
Don't worry.
You do not need to memorize it.
Instead, understand the idea behind it.
REST is simply a set of rules or guidelines for designing APIs.
Think of REST as traffic rules.
Traffic rules tell drivers:
Drive on the correct side.
Stop at a red light.
Give way where required.
REST tells developers:
How to organize URLs.
Which HTTP methods to use.
How to structure requests.
How to structure responses.
Why Was REST Created?
Imagine every website created APIs differently.
One website might use:
/getUsers
Another:
/all_students
Another:
/show_everything
Another:
/fetch_people_now
It would be confusing.
REST provides a common standard.
Now most APIs look similar.
This makes them easier to learn and use.
What is a REST API?
A REST API is simply an API that follows REST principles.
Instead of creating random URLs,
we organize them logically.
Example:
/users
Instead of:
/getAllUsersNowPlease
Much cleaner.
Much easier to understand.

What is a Resource?
A Resource is anything your application manages.
Examples:
Users
Students
Books
Products
Orders
Comments
Employees
Courses
Each of these is a resource.
Examples:
/users

/books

/products

/orders

/students
Notice something.
Resources are usually written as plural nouns.
What is an Endpoint?
An endpoint is simply a URL that points to a resource.
Examples:
/users
/products
/orders
/students
Each endpoint performs an action depending on the HTTP method used.
Understanding Endpoints
Suppose your website is:
https://school.com
Endpoints might be:
https://school.com/students
https://school.com/teachers
https://school.com/courses
Each endpoint represents a resource.
URI vs URL
These terms confuse many beginners.
Let's simplify them.
URL
A URL tells the browser exactly where something is located.
Example:
https://example.com/users
URI
A URI identifies a resource.
Every URL is a URI.
Not every URI is necessarily a URL.
As a frontend developer, you'll mostly work with URLs.
REST Uses HTTP Methods
Remember CRUD?
CRUD

REST combines resources with HTTP methods.
Example:
GET /students
Meaning:
Read all students.
POST /students
Meaning:
Create a student.
PATCH /students/5
Meaning:
Update student number 5.
DELETE /students/5
Meaning:
Delete student number 5.
Notice that the URL stays almost the same.
Only the HTTP method changes.

Why Don't We Use Different URLs?
Imagine this design:
/getStudents

/createStudent

/updateStudent

/deleteStudent
This works...
But REST recommends:
GET /students

POST /students

PATCH /students/5

DELETE /students/5
This keeps APIs clean and consistent.

How Django REST Framework Uses REST
Suppose you have this Django model:
class Student(models.Model):
    name = models.CharField(max_length=100)
    age = models.IntegerField()
Django REST Framework can automatically create endpoints like:
GET /students/
POST /students/
GET /students/1/
PUT /students/1/
PATCH /students/1/
DELETE /students/1/
Notice how DRF follows REST principles automatically.
REST URL Design Best Practices
Good URLs:
/users
/products
/orders
/books
Avoid URLs like:
/getUsers
/createBook
/deleteOrder
Let the HTTP method describe the action.
Common Beginner Mistakes
Mistake 1
Putting actions inside URLs.
Wrong:
/createStudent
Correct:
POST /students
Mistake 2
Using GET to delete data.
Wrong:
GET /deleteStudent
Correct:
DELETE /students/10
Mistake 3
Creating too many endpoints.
Wrong:
/getBooks

/addBooks

/updateBooks

/removeBooks
Correct:
/books
Use different HTTP methods.
REST Architecture Principles (Simplified)
A RESTful API generally follows these principles:
Client-Server
The frontend and backend are separate.
Example:
Browser

↓

API

↓

Database
Stateless
Each request contains all the information the server needs.
The server does not remember previous requests.
Example:
Request 1

↓

Response

Request 2

↓

Response
Each request is independent.
Uniform Interface
Resources use consistent URLs.
Example:
/users

/products

/orders
Not random names.
Summary
