# Module 17 — Callback Hell: Why JavaScript Introduced Promises
Congratulations! You have mastered callback functions.
Now it's time to learn the biggest problem callbacks can create.
Understanding this problem will help you understand why JavaScript introduced Promises.
What is Callback Hell?
Callback Hell happens when you have many asynchronous tasks that depend on each other, and each task is placed inside another callback.
Imagine this real-world process:
Create User
      ↓
Verify Email
      ↓
Login User
      ↓
Load Dashboard
      ↓
Show Welcome Message
Each step must wait until the previous step finishes.
Using callbacks, the code might look like this:
createUser(function () {

    verifyEmail(function () {

        login(function () {

            loadDashboard(function () {

                showWelcome();

            });

        });

    });

});
Why is it Called Callback Hell?
Look carefully at the code.
Every callback is inside another callback.
createUser()
    ↓
    verifyEmail()
        ↓
        login()
            ↓
            loadDashboard()
                ↓
                showWelcome()
Every new task pushes the code further to the right.
Developers nicknamed this the Pyramid of Doom.
createUser(
    verifyEmail(
        login(
            loadDashboard(
                showWelcome()
            )
        )
    )
)
As your application grows, the nesting becomes harder to follow.
Real-World Example
Imagine building a house.
You cannot paint the walls before building them.
You cannot build the walls before laying the foundation.
Everything depends on something else.
Lay Foundation
      ↓
Build Walls
      ↓
Install Roof
      ↓
Paint House
      ↓
Move In
Callbacks can represent these dependencies.
The problem appears when too many callbacks become deeply nested.
Example — Downloading a File
Suppose your application must:
Download a file.
Validate it.
Save it.
Notify the user.
Using callbacks:
downloadFile(function () {

    validateFile(function () {

        saveFile(function () {

            notifyUser(function () {

                console.log("Done!");

            });

        });

    });

});
This works.
But imagine doing this for 15 or 20 steps.
The code quickly becomes difficult to read.
Problems with Callback Hell
Problem 1 — Difficult to Read
Every callback adds another level of indentation.
Task 1
    Task 2
        Task 3
            Task 4
                Task 5
Finding where one callback ends and another begins becomes difficult.
Problem 2 — Difficult to Debug
Suppose something fails here:
saveFile(function () {
Now you have to trace through several nested callbacks to find the problem.
Problem 3 — Difficult to Reuse
Imagine you want to reuse only the login() function.
If it is deeply nested inside other callbacks, extracting it becomes harder.
Problem 4 — Error Handling Becomes Messy
Suppose downloading fails.
downloadFile(function (error) {

    if (error) {
        console.log(error);
    }

});
Now imagine writing error handling for every callback.
Your code becomes cluttered.
Visual Comparison
Small callback chain:
Task 1
   ↓
Task 2
   ↓
Task 3
Easy to understand.
Deep callback chain:
Task 1
   ↓
    Task 2
       ↓
        Task 3
           ↓
            Task 4
               ↓
                Task 5
                   ↓
                    Task 6
This becomes difficult to maintain.
Are Callbacks Bad?
No.
Callbacks are still used everywhere.
Examples include:
forEach()
map()
filter()
find()
some()
every()
reduce()
addEventListener()
setTimeout()
setInterval()
Callbacks themselves are not the problem.
The problem is deeply nested callbacks.
The Solution — Promises
Instead of nesting callbacks, Promises let you write code like this:
createUser()
    .then(verifyEmail)
    .then(login)
    .then(loadDashboard)
    .then(showWelcome)
    .catch(handleError);
Notice how much cleaner this looks.
Instead of moving to the right, the code flows downward.
Create User
      ↓
Verify Email
      ↓
Login
      ↓
Load Dashboard
      ↓
Show Welcome
This is easier to read and maintain.
Another Real-World Analogy
Ordering food.
Without Promises:
Restaurant
      ↓
Delivery Driver
      ↓
Security Guard
      ↓
Reception
      ↓
You
Many people pass the message along.
With Promises:
Restaurant
      ↓
Delivery
      ↓
You
The flow is cleaner and easier to follow.
Very Important
Promises did not replace callbacks.
Promises still use callbacks.
Example:
promise.then(function (result) {
    console.log(result);
});
The function passed to .then() is still a callback.
That is why learning callbacks first is so important.
Summary
Callbacks are:
Simple
Flexible
Powerful
But deeply nested callbacks become:
Hard to read
Hard to debug
Hard to maintain
Promises solve these problems by organizing asynchronous code in a cleaner way.
