## Previous Knowledge

Previously we learned that a JavaScript program is made from statements and expressions.

A **statement** tells JavaScript to perform an action.

```javascript
const price = 500;
console.log(price);
```

An **expression** produces a value.

```javascript
500 * 2
```

Those building blocks are enough for very small programs. However, as a program grows, the same groups of statements often need to run many times. Repeating those statements manually creates duplication, inconsistency, and maintenance problems.

This lesson introduces the tool JavaScript provides for grouping and reusing instructions: the **function**.

## Key Terms

| Term | Meaning |
|---|---|
| Function | A named or stored group of instructions that can be executed when needed. |
| Declaration | A statement that introduces something to JavaScript, such as a variable or function. |
| Function body | The statements placed between a function's `{` and `}`. |
| Function call | An instruction that tells JavaScript to execute a function. |
| Reusability | The ability to use the same logic in multiple places. |

## Why This Topic Matters

Functions are one of the most important ideas in programming.

Without functions, a large application would become a long sequence of repeated instructions. Developers would have to copy and modify code every time they wanted to perform a familiar task.

Functions make it possible to:

- Reuse logic.
- Give meaningful names to operations.
- Separate a large problem into smaller problems.
- Test one unit of behavior at a time.
- Fix logic in one place.
- Build modules, components, APIs, event handlers, and application services.
- Pass behavior into other functions.
- Create closures and private state.
- Organize asynchronous workflows.
- Build frameworks and libraries.

React components, Express route handlers, browser event listeners, validation rules, API clients, and test cases all depend heavily on functions.

## Mental Model

Think of a function as a machine.

```text
             Function Machine
        ┌─────────────────────────┐
Input → │ Perform named steps     │ → Output
        └─────────────────────────┘
```

A machine may:

- Receive input.
- Perform work.
- Produce output.
- Change something outside itself.
- Be reused whenever the same job is needed.

A function gives a sequence of instructions a reusable identity.

Instead of remembering every internal step, other code can say:

```text
calculate transaction total
```

The name communicates **what** should happen. The function body contains **how** it happens.

## Real-world Motivation

Imagine a Point of Sale application that calculates the total price of a transaction.

Without a function, the program might repeat the same calculation:

```javascript
const firstTotal = 3 * 150;
const secondTotal = 2 * 300;
const thirdTotal = 5 * 100;
```

This is manageable now, but a production POS system may perform thousands of calculations across:

- Sales screens.
- Receipt generation.
- Reports.
- Refund processing.
- Daily summaries.
- API requests.
- Validation.

If the pricing rule changes, developers must locate and update every copy. Missing one copy creates inconsistent results.

A function centralizes that rule:

```javascript
function calculateTotal(quantity, unitPrice) {
  return quantity * unitPrice;
}
```

Now every part of the application can use the same rule.

## Concept Explanation

### What is a function?

A function is a reusable unit of behavior.

It groups one or more instructions and allows them to be executed through a function call.

### Why were functions created?

Programs naturally contain repeated operations.

Early programmers discovered that copying instructions caused several problems:

- Programs became longer.
- Bugs were copied.
- Updates had to be repeated.
- Code became difficult to understand.
- Different copies of the same logic could become inconsistent.

Functions were created to let programmers define an operation once and invoke it many times.

### What problem does a function solve?

A function solves several related problems:

1. **Duplication** — the same logic does not need to be copied.
2. **Poor readability** — a meaningful name can explain an operation.
3. **Large complexity** — a large problem can be divided into smaller responsibilities.
4. **Difficult maintenance** — shared logic can be changed in one location.
5. **Difficult testing** — isolated logic can be tested separately.

### How was the problem solved before reusable functions?

The simplest approach was copying instructions.

```javascript
const firstSubtotal = 2 * 100;
const firstTax = firstSubtotal * 0.16;
const firstTotal = firstSubtotal + firstTax;

const secondSubtotal = 5 * 100;
const secondTax = secondSubtotal * 0.16;
const secondTotal = secondSubtotal + secondTax;
```

This works, but duplicates the tax rule.

Another historical approach was jumping to labeled sections of machine instructions. Such control flow was hard to reason about and easy to misuse. Structured procedures and functions provided a safer, clearer model.

### Why is the function approach better?

A function:

- Has a meaningful name.
- Can receive different data.
- Can return a result.
- Can be called from multiple places.
- Creates a boundary around a responsibility.
- Can be tested independently.
- Can be composed with other functions.

### When should functions be used?

Use a function when:

- A meaningful operation needs a name.
- Logic is repeated.
- Logic may change independently.
- A task contains several related steps.
- You need to test behavior separately.
- You need a callback or event handler.
- You need to transform input into output.
- You want to hide implementation details.

### When should functions not be used?

A function may be unnecessary when:

- The operation is a single obvious expression used once.
- Extracting it would make the code harder to follow.
- The proposed function has no meaningful responsibility.
- The function would merely rename a language feature without adding clarity.
- Excessive fragmentation would force readers to jump between many tiny files or functions.

Bad abstraction:

```javascript
function addOne(number) {
  return number + 1;
}

const nextPage = addOne(currentPage);
```

This function may be justified in a specialized domain, but in ordinary code it may add no useful meaning.

### What are the limitations of functions?

Functions do not automatically guarantee good design.

A function can still:

- Be too long.
- Have too many responsibilities.
- Depend on hidden global state.
- Mutate data unexpectedly.
- Use unclear names.
- Accept too many parameters.
- Produce side effects.
- Be difficult to test.
- Retain memory through closures.
- Create performance overhead when called excessively in a critical path.

The goal is not simply to create functions. The goal is to create functions with clear responsibilities.

### How do functions work internally?

At a high level:

1. JavaScript reads and prepares the function definition.
2. The function becomes available through its name or variable.
3. The function body does not execute merely because it was defined.
4. A function call is encountered.
5. JavaScript creates a function execution context.
6. A stack frame is placed on the call stack.
7. Parameters receive argument values.
8. Statements in the function body execute.
9. A result may be returned.
10. The stack frame is removed.
11. Execution resumes after the function call.

### Where are functions used in real applications?

Functions are used for:

- Validating login forms.
- Calculating shopping cart totals.
- Formatting dates and currency.
- Handling button clicks.
- Sending API requests.
- Rendering data in the DOM.
- Processing payments.
- Filtering reports.
- Creating users.
- Authenticating requests.
- Retrying failed network operations.
- Writing automated tests.
- Creating reusable modules.
- Building React components.
- Defining Express middleware.

## Internal Working

Consider this function:

```javascript
function greetCustomer() {
  console.log("Welcome");
}

greetCustomer();
```

The runtime process is:

```text
1. Global execution begins.
2. JavaScript prepares greetCustomer.
3. JavaScript reaches greetCustomer().
4. A function execution context is created.
5. greetCustomer is pushed onto the call stack.
6. console.log("Welcome") executes.
7. greetCustomer finishes.
8. greetCustomer is removed from the call stack.
9. Global execution continues.
```

Call stack visualization:

```text
Before the call

┌──────────────────────┐
│ Global execution     │
└──────────────────────┘

During the call

┌──────────────────────┐
│ greetCustomer()      │
├──────────────────────┤
│ Global execution     │
└──────────────────────┘

After the call

┌──────────────────────┐
│ Global execution     │
└──────────────────────┘
```

### Memory behavior

The function object is stored in memory.

The function's name refers to that function object.

When the function is called, JavaScript creates temporary runtime information for that call, including:

- Parameters.
- Local variables.
- The current instruction position.
- The connection to its lexical environment.
- The return location.

Most call-specific data becomes eligible for cleanup when the call finishes, unless another reachable object or closure still references it.

## Syntax

The simplest modern introduction uses a function declaration:

```javascript
function functionName() {
  // Instructions
}
```

To execute it:

```javascript
functionName();
```

Parts:

| Part | Purpose |
|---|---|
| `function` | Keyword that begins a traditional function declaration. |
| `functionName` | The identifier used to refer to the function. |
| `()` | Holds parameters when the function needs input. |
| `{}` | Contains the function body. |
| `functionName()` | Calls the function. |
| `;` | Ends the function-call statement. |

## Rules

1. A function declaration uses the `function` keyword.
2. A function name should describe an action or calculation.
3. The function body appears between curly braces.
4. Defining a function does not execute its body.
5. Parentheses after the name perform the call.
6. A function may be called more than once.
7. A function may receive inputs called arguments.
8. A function may produce an output using `return`.
9. Local variables belong to the function call that created them.
10. A function should normally have one clear responsibility.

## Example 1 — A Reusable Greeting

### Problem

A program must print the same welcome message three times.

### Requirements

- Create one function.
- Put the message inside the function.
- Call the function three times.
- Use the console only.

### Initial State

No function exists and no message has been printed.

### Why this code exists

The code demonstrates the smallest useful reason for a function: defining an instruction once and executing it repeatedly.

### Expected Output

```text
Welcome to JavaScript
Welcome to JavaScript
Welcome to JavaScript
```

### Code

```javascript
function displayWelcomeMessage() {
  console.log("Welcome to JavaScript");
}

displayWelcomeMessage();
displayWelcomeMessage();
displayWelcomeMessage();
```

### Line-by-line explanation

#### `function displayWelcomeMessage() {`

- `function` tells JavaScript that a function declaration begins.
- `displayWelcomeMessage` is the function's name.
- The name begins with a verb because the function performs an action.
- `()` means the function currently accepts no input.
- `{` begins the function body.

#### `console.log("Welcome to JavaScript");`

- `console` is an object supplied by the runtime environment.
- `.` accesses a property of that object.
- `log` is a function stored on the console object.
- `()` calls `console.log`.
- `"Welcome to JavaScript"` is the argument passed to `console.log`.
- `;` ends the statement.

#### `}`

The closing brace ends the function body.

At this point, the message has not been printed. The instructions have only been defined.

#### `displayWelcomeMessage();`

- `displayWelcomeMessage` finds the function.
- `()` calls it.
- JavaScript enters its body.
- `console.log()` runs.
- The function finishes.
- Execution returns to the next line.

The next two calls repeat the same process.

### Step-by-step execution

```text
Define displayWelcomeMessage
        ↓
Call 1
        ↓
Print message
        ↓
Call 2
        ↓
Print message
        ↓
Call 3
        ↓
Print message
```

### Console Output

```text
Welcome to JavaScript
Welcome to JavaScript
Welcome to JavaScript
```

### Final State

- One function definition exists.
- The function has been executed three times.
- No application data was changed.
- Three console messages were produced.

### Common mistakes

Incorrect:

```javascript
function displayWelcomeMessage() {
  console.log("Welcome to JavaScript");
}
```

The function is defined but never called.

Incorrect:

```javascript
displayWelcomeMessage;
```

This refers to the function but does not call it.

Correct:

```javascript
displayWelcomeMessage();
```

### Small Challenge

Change the function so it prints:

```text
Welcome to the JavaScript Mastery Course
```

Then call it five times.

## Example 2 — POS Transaction Total

### Problem

A POS system must calculate transaction totals for different services.

### Requirements

- Store services as objects in an array.
- Create a reusable function.
- Calculate `quantity × unitPrice`.
- Add the calculated total to each transaction.
- Display each result.

### Initial Data

```javascript
const transactions = [
  { service: "Printing", quantity: 3, unitPrice: 50 },
  { service: "Photocopying", quantity: 10, unitPrice: 5 },
  { service: "Binding", quantity: 2, unitPrice: 150 },
];
```

### Initial State

Each transaction contains a service, quantity, and unit price. No total has been calculated.

### Why this solution was chosen

The pricing rule belongs in one function because the same calculation is needed for every transaction. The loop controls repetition over the data; the function owns the business rule.

### Expected Output

```text
Printing: KES 150
Photocopying: KES 50
Binding: KES 300
```

### Code

```javascript
const transactions = [
  { service: "Printing", quantity: 3, unitPrice: 50 },
  { service: "Photocopying", quantity: 10, unitPrice: 5 },
  { service: "Binding", quantity: 2, unitPrice: 150 },
];

function calculateTransactionTotal(quantity, unitPrice) {
  return quantity * unitPrice;
}

for (const transaction of transactions) {
  const total = calculateTransactionTotal(
    transaction.quantity,
    transaction.unitPrice
  );

  console.log(`${transaction.service}: KES ${total}`);
}
```

### What every variable stores

| Variable | Stored value |
|---|---|
| `transactions` | An array containing three transaction objects. |
| `transaction` | The current object during each loop iteration. |
| `quantity` | The quantity passed into the function for the current call. |
| `unitPrice` | The unit price passed into the function for the current call. |
| `total` | The number returned by `calculateTransactionTotal`. |

### What every function does

| Function | Responsibility |
|---|---|
| `calculateTransactionTotal` | Multiplies quantity by unit price and returns the result. |
| `console.log` | Writes a formatted result to the console. |

### Execution walkthrough

First iteration:

```text
transaction = Printing object
        ↓
calculateTransactionTotal(3, 50)
        ↓
quantity = 3
unitPrice = 50
        ↓
3 × 50 = 150
        ↓
return 150
        ↓
total = 150
        ↓
Print "Printing: KES 150"
```

The same function is reused for the remaining objects.

### Memory behavior

- The array and its objects are reference values stored in heap memory.
- `transactions` holds a reference to the array.
- Each function call receives primitive number values.
- The multiplication produces a new number.
- The function returns that number.
- The original transaction objects are not modified.

### Console Output

```text
Printing: KES 150
Photocopying: KES 50
Binding: KES 300
```

### Final State

The array remains unchanged:

```javascript
[
  { service: "Printing", quantity: 3, unitPrice: 50 },
  { service: "Photocopying", quantity: 10, unitPrice: 5 },
  { service: "Binding", quantity: 2, unitPrice: 150 },
]
```

The totals were calculated and displayed but were not stored back in the objects.

### Common mistakes

Incorrect:

```javascript
function calculateTransactionTotal(quantity, unitPrice) {
  quantity * unitPrice;
}
```

The multiplication runs, but no value is returned. The caller receives `undefined`.

Correct:

```javascript
function calculateTransactionTotal(quantity, unitPrice) {
  return quantity * unitPrice;
}
```

### Small Challenge

Add a `discountRate` property to each transaction and create another function that calculates the final amount after discount.

## Example 3 — Browser Discount Calculator

### Problem

Build a small browser application that allows a user to enter a price and discount percentage, then calculate the final price.

### Requirements

- Use semantic HTML.
- Accept a price.
- Accept a discount percentage.
- Validate both values.
- Use a reusable calculation function.
- Respond to a button click.
- Display success or error feedback.
- Avoid `innerHTML` for user-controlled output.

### Initial Interface

```text
Discount Calculator

Price: [          ]
Discount (%): [          ]

[Calculate]

Enter values and select Calculate.
```

### Initial State

```javascript
const state = {
  price: 0,
  discountPercentage: 0,
  finalPrice: null,
  error: "",
};
```

### Why this solution was chosen

The application separates responsibilities:

- `calculateDiscountedPrice` performs business logic.
- `readNumberInput` reads and validates input.
- `renderResult` updates the interface.
- The event listener coordinates user interaction.

This is the beginning of application architecture.

### HTML

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <title>Discount Calculator</title>
    <link rel="stylesheet" href="styles.css" />
  </head>
  <body>
    <main class="calculator" aria-labelledby="page-title">
      <h1 id="page-title">Discount Calculator</h1>

      <form id="discount-form" novalidate>
        <div class="field">
          <label for="price">Price</label>
          <input
            id="price"
            name="price"
            type="number"
            min="0"
            step="0.01"
            required
          />
        </div>

        <div class="field">
          <label for="discount">Discount (%)</label>
          <input
            id="discount"
            name="discount"
            type="number"
            min="0"
            max="100"
            step="0.01"
            required
          />
        </div>

        <button type="submit">Calculate</button>
      </form>

      <p id="result" role="status" aria-live="polite">
        Enter values and select Calculate.
      </p>
    </main>

    <script src="app.js"></script>
  </body>
</html>
```

### What the HTML does

- `main` identifies the primary content.
- `aria-labelledby` connects the region to the heading.
- `form` groups the user inputs.
- Each `label` is explicitly connected to an input through `for` and `id`.
- `type="number"` provides numeric input behavior.
- `min`, `max`, and `required` provide browser-level constraints.
- `novalidate` lets the lesson demonstrate custom validation.
- `role="status"` and `aria-live="polite"` allow assistive technology to announce result changes.
- The script is loaded after the interface, so the elements exist before JavaScript selects them.

### JavaScript

```javascript
const form = document.querySelector("#discount-form");
const priceInput = document.querySelector("#price");
const discountInput = document.querySelector("#discount");
const resultOutput = document.querySelector("#result");

const state = {
  price: 0,
  discountPercentage: 0,
  finalPrice: null,
  error: "",
};

function calculateDiscountedPrice(price, discountPercentage) {
  const discountAmount = price * (discountPercentage / 100);
  return price - discountAmount;
}

function readNumberInput(input, fieldName) {
  const value = Number(input.value);

  if (!Number.isFinite(value)) {
    throw new Error(`${fieldName} must be a valid number.`);
  }

  return value;
}

function validateValues(price, discountPercentage) {
  if (price < 0) {
    throw new Error("Price cannot be negative.");
  }

  if (discountPercentage < 0 || discountPercentage > 100) {
    throw new Error("Discount must be between 0 and 100.");
  }
}

function renderResult() {
  if (state.error) {
    resultOutput.textContent = state.error;
    return;
  }

  resultOutput.textContent = `Final price: KES ${state.finalPrice.toFixed(2)}`;
}

function handleFormSubmit(event) {
  event.preventDefault();

  try {
    const price = readNumberInput(priceInput, "Price");
    const discountPercentage = readNumberInput(
      discountInput,
      "Discount"
    );

    validateValues(price, discountPercentage);

    state.price = price;
    state.discountPercentage = discountPercentage;
    state.finalPrice = calculateDiscountedPrice(
      price,
      discountPercentage
    );
    state.error = "";
  } catch (error) {
    state.finalPrice = null;
    state.error = error.message;
  }

  renderResult();
}

form.addEventListener("submit", handleFormSubmit);
```

### What every major variable stores

| Variable | Stored value |
|---|---|
| `form` | Reference to the form element. |
| `priceInput` | Reference to the price input. |
| `discountInput` | Reference to the discount input. |
| `resultOutput` | Reference to the live result paragraph. |
| `state` | Current application data and error status. |

### What every function does

| Function | Responsibility |
|---|---|
| `calculateDiscountedPrice` | Calculates and returns the discounted price. |
| `readNumberInput` | Converts an input value to a number and rejects invalid numeric values. |
| `validateValues` | Enforces business rules for price and discount. |
| `renderResult` | Synchronizes the result paragraph with application state. |
| `handleFormSubmit` | Coordinates the complete submission workflow. |

### Execution order after the user submits

```text
User selects Calculate
        ↓
Browser creates a submit event
        ↓
handleFormSubmit(event) runs
        ↓
preventDefault() stops page reload
        ↓
Read price and discount
        ↓
Validate values
        ↓
Calculate final price
        ↓
Update state
        ↓
renderResult()
        ↓
Update textContent
        ↓
Browser displays and announces result
```

### Example interaction

Input:

## Common Mistakes

### 1. Defining but not calling the function

Incorrect:

```javascript
function printReceipt() {
  console.log("Receipt printed");
}
```

Why beginners write it:

They expect declaration to perform execution.

Why it fails:

A declaration defines behavior. It does not request that behavior.

Correct:

```javascript
function printReceipt() {
  console.log("Receipt printed");
}

printReceipt();
```

Best practice:

Look for `()` at the place where execution should happen.

### 2. Referring to a function instead of calling it

Incorrect:

```javascript
printReceipt;
```

Why it fails:

This expression evaluates to the function object but does not invoke it.

Correct:

```javascript
printReceipt();
```

### 3. Calling a misspelled name

Incorrect:

```javascript
function calculateTotal() {
  return 100;
}

calulateTotal();
```

Why it fails:

`calulateTotal` and `calculateTotal` are different identifiers.

Expected error:

```text
ReferenceError: calulateTotal is not defined
```

Correct:

```javascript
calculateTotal();
```

### 4. Forgetting `return`

Incorrect:

```javascript
function calculateTotal() {
  100 + 50;
}

const total = calculateTotal();
console.log(total);
```

Output:

```text
undefined
```

Correct:

```javascript
function calculateTotal() {
  return 100 + 50;
}
```

### 5. Writing unrelated responsibilities in one function

Incorrect:

```javascript
function processSale() {
  // Validate form
  // Calculate total
  // Save transaction
  // Render receipt
  // Send email
  // Update dashboard
}
```

Why it becomes a problem:

The function is difficult to test, reuse, modify, and debug.

Improved:

```javascript
function validateSale() {}
function calculateSaleTotal() {}
function saveSale() {}
function renderReceipt() {}
function sendReceiptEmail() {}
function updateDashboard() {}
```

### 6. Using unclear names

Incorrect:

```javascript
function doIt() {}
```

Correct:

```javascript
function calculateInvoiceTotal() {}
```

### 7. Depending on hidden global variables

Incorrect:

```javascript
let quantity = 3;
let price = 100;

function calculateTotal() {
  return quantity * price;
}
```

Problem:

The function's requirements are hidden.

Improved:

```javascript
function calculateTotal(quantity, price) {
  return quantity * price;
}
```

### 8. Mutating input unexpectedly

Incorrect:

```javascript
function applyDiscount(product) {
  product.price = product.price * 0.9;
}
```

Why it is risky:

The caller's object changes even though the function name does not clearly communicate mutation.

Safer approach:

```javascript
function createDiscountedProduct(product) {
  return {
    ...product,
    price: product.price * 0.9,
  };
}
```

### 9. Creating meaningless tiny wrappers

Potentially unnecessary:

```javascript
function print(value) {
  console.log(value);
}
```

This may be useful if logging policy is centralized, but otherwise it may add no clarity.

### 10. Copying a function instead of reusing it

Incorrect architecture:

```javascript
function calculateAdminTotal() {
  return quantity * price;
}

function calculateCashierTotal() {
  return quantity * price;
}
```

Improved:

```javascript
function calculateTransactionTotal(quantity, price) {
  return quantity * price;
}
```

## Best Practices

1. Name functions using clear verbs.
2. Give each function one main responsibility.
3. Prefer explicit inputs over hidden global dependencies.
4. Return useful values when the caller needs a result.
5. Keep business logic separate from DOM manipulation.
6. Keep validation separate from calculation.
7. Avoid surprising mutation.
8. Extract repeated knowledge, not merely repeated text.
9. Keep functions small enough to understand, but not artificially tiny.
10. Test important calculations independently.
11. Use modern declarations and modules as the course progresses.
12. Document unusual business rules.
13. Handle errors at an appropriate application boundary.
14. Prefer deterministic functions for core calculations.
15. Choose names based on domain language.

## Performance Notes

Function calls have a small runtime cost because JavaScript creates call-related execution information.

In normal application code, this cost is negligible compared with the benefits of clarity and reuse.

Performance concerns become relevant when:

- A tiny function runs millions of times in a critical loop.
- Recursion creates a very deep call stack.
- Closures unintentionally retain large objects.
- Rendering functions perform excessive DOM work.
- A function repeatedly queries layout information.
- Functions allocate many unnecessary objects.

Professional recommendation:

> Write clear, correct code first. Measure actual performance before removing useful abstractions.

Do not avoid functions because of imagined micro-optimization.

## Security Notes

Functions improve security when they centralize sensitive rules.

Examples:

- Input validation.
- Output encoding.
- Authorization checks.
- Token handling.
- Request construction.
- Safe DOM rendering.

However, a function is not automatically secure.

Dangerous:

```javascript
function renderUserMessage(message) {
  output.innerHTML = message;
}
```

If `message` contains untrusted HTML, this may create an XSS vulnerability.

Safer:

```javascript
function renderUserMessage(message) {
  output.textContent = message;
}
```

Never store real application secrets in frontend JavaScript functions. Browser code is delivered to users and can be inspected.

## Accessibility Notes

Functions often control dynamic user interfaces.

When a function updates the DOM, it must preserve accessibility.

Professional checks:

- Is the updated content announced when necessary?
- Does keyboard focus move appropriately?
- Can the operation be performed without a mouse?
- Is the function changing `disabled`, `hidden`, or ARIA attributes correctly?
- Are errors connected to their inputs?
- Does the interface avoid relying only on color?
- Is a modal's focus managed correctly?
- Are event handlers attached to semantic controls?

A technically working function may still create an unusable experience for keyboard or screen-reader users.

        ↓

Useful techniques:

Remove temporary logs after debugging.

## Summary

A function is a reusable unit of behavior.

Functions exist because repeated, unnamed groups of statements make programs difficult to understand and maintain.

A function allows developers to:

- Give an operation a meaningful name.
- Define logic once.
- Execute it many times.
- Receive inputs.
- Return outputs.
- Divide large applications into smaller responsibilities.
- Test and debug isolated behavior.

When JavaScript calls a function, it creates a function execution context, pushes a stack frame, executes the body, returns a value if requested, and removes the frame.

Functions are powerful, but they must be designed carefully. Poorly named, highly coupled, side-effect-heavy functions can still create fragile code.

## Key Takeaways

- A function is behavior with an identity.
- Function definitions and function calls are different.
- Reusability is only one benefit; naming and decomposition are equally important.
- Functions create runtime execution contexts when called.
- Core calculations should usually be independent from DOM code.
- Explicit inputs make dependencies easier to understand.
- `return` sends a result back to the caller.
- Functions should have clear responsibilities.
- Do not create abstractions that make simple code harder to read.
- Functions form the foundation for callbacks, closures, modules, events, promises, and application architecture.
