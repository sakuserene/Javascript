# Introduction

When beginners see this:

```js
for (let i = 0; i < 5; i++) {
    console.log(i);
}
```

they usually think it's one big statement.

The JavaScript engine does **not**.

It separates it into **four independent parts**.

Understanding these four parts means you can understand **every loop in JavaScript**, not just `for`.

---

# Mental Model

Imagine a security guard checking people entering a building.

```text
Start

↓

Register Visitor

↓

Can they enter?

↓

YES

↓

Do the work

↓

Update visitor count

↓

Check again

↓

NO

↓

Stop
```

Every loop works exactly like this.

---

# Theory

Every loop, regardless of syntax, consists of these four phases:

```text
1. Initialization

↓

2. Condition

↓

3. Body

↓

4. Update

↓

Repeat
```

If you remove **any one** of these, the loop changes behavior.

---

# The Four Parts of a Loop

Consider:

```js
for (let i = 0; i < 3; i++) {
    console.log(i);
}
```

Let's break it apart.

---

## Part 1 — Initialization

```js
let i = 0;
```

This runs **once**.

Only one time.

Never again.

Memory becomes:

```text
+---------+
| i | 0   |
+---------+
```

---

## Part 2 — Condition

```js
i < 3
```

The engine asks:

```text
Is this true?
```

If yes:

Continue.

If no:

Stop immediately.

---

## Part 3 — Body

```js
console.log(i);
```

This is the actual work.

Anything inside `{}` belongs to the body.

---

## Part 4 — Update

```js
i++
```

This changes memory.

Before:

```text
i = 0
```

After:

```text
i = 1
```

Then the engine jumps back to the condition.

---

# Internal Mechanics

The JavaScript engine behaves roughly like this:

```text
Run Initialization

↓

Evaluate Condition

↓

True?

↓

Execute Body

↓

Run Update

↓

Jump Back

↓

Evaluate Condition Again

↓

Repeat

↓

Condition False

↓

Exit
```

Notice:

The **condition** is checked **before every iteration**.

Not once.

Not twice.

Every single iteration.

---

# Syntax

A `for` loop places all four parts together.

```js
for (initialization; condition; update) {
    // body
}
```

Think of it like:

```text
for (
    start;
    shouldContinue;
    changeSomething
) {
    work();
}
```

## Line-by-Line Explanation

### Initialization

```js
let i = 0;
```

Memory:

```text
i → 0
```

Runs once.

---

### Condition

```js
i < 3
```

Checks:

```text
0 < 3

↓

true
```

---

### Body

```js
console.log(i);
```

Outputs:

```text
0
```

---

### Update

```js
i++;
```

Memory:

```text
Before

i → 0

↓

After

i → 1
```

Now the engine jumps back.

Second iteration:

```text
1 < 3

↓

true
```

Repeat.

Third iteration:

```text
2 < 3

↓

true
```

Repeat.

Fourth check:

```text
3 < 3

↓

false
```

Exit.

Output:

```text
0
1
2
```

# Example 2

Changing only the update.

```js
for (let i = 0; i < 10; i += 2) {
    console.log(i);
}
```

Notice:

Instead of:

```js
i++
```

we use:

```js
i += 2;
```

Execution:

```text
0

↓

2

↓

4

↓

6

↓

8

↓

10

↓

Condition fails
```

Output:

```text
0
2
4
6
8
```

Nothing else changed.

Only the update.

# Common Mistakes

| Mistake | Why it happens | Fix |
|----------|----------------|-----|
| Thinking initialization runs every iteration | It only runs once | Remember: only the condition repeats first |
| Forgetting the update | The condition never changes | Always update the loop variable |
| Updating the wrong variable | The controlling variable stays the same | Update the variable used in the condition |
| Assuming the body runs before the first condition | The condition is checked first | The body only executes if the condition is `true` |

---

# Performance

For one iteration:

| Operation | Time |
|-----------|------|
| Initialization | O(1) *(runs once)* |
| Condition | O(1) |
| Body | Depends on your code |
| Update | O(1) |

If the loop runs **n** times:

| Part | Total Cost |
|------|------------|
| Initialization | O(1) |
| Condition | O(n + 1) *(one extra final check)* |
| Update | O(n) |
| Body | Depends on its contents |

> **Important**
>
> The condition is evaluated **one more time** than the body executes because the final failed check is what tells the engine to stop.

---

# Best Practices

- Read a loop in this order:
  1. Initialization
  2. Condition
  3. Body
  4. Update
- Trace the value of the loop variable after every update.
- Ask yourself: "What will eventually make the condition become false?"
- Keep the update simple and predictable.

# Summary

A loop is not a single action. It is a cycle composed of four repeating phases:

1. Initialize.
2. Check the condition.
3. Execute the body.
4. Update the controlling variable.

Once you understand this cycle, you can mentally simulate any loop, regardless of whether it is a `for`, `while`, or `do...while` loop.

---

# Key Takeaways

- Every loop has four essential parts.
- Initialization runs exactly once.
- The condition controls whether the loop continues.
- The body contains the work performed each iteration.
- The update changes the state so the condition can eventually become `false`.
- The condition is checked one extra time before the loop exits.

---

# Glossary

| Term | Definition |
|------|------------|
| Initialization | The one-time setup before the loop starts. |
| Condition | The boolean expression checked before each iteration. |
| Body | The statements executed while the condition is true. |
| Update | The operation that changes the loop variable. |
| Iteration | One complete execution of the loop body. |
