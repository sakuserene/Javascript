# Introduction

Most beginners think a loop is simply:

```js
for (...) {
    ...
}
```

That is **not** what JavaScript thinks.

The JavaScript engine does **not** see a "loop."

Instead, it sees something more like this:

```text
Evaluate condition.

Is it true?

YES
↓

Execute body

↓

Update variable

↓

Evaluate condition again

↓

Still true?

Repeat.

False?

Exit.
```

A loop is nothing more than repeated execution.

Understanding this idea is the foundation of every algorithm you'll ever write.

---

# Mental Model

Imagine a robot.

The robot knows only four instructions:

```text
1. Read memory

2. Compare values

3. Execute code

4. Jump backwards
```

That's it.

Everything else—including loops—is built from those four operations.

---

# Theory

## What Is an Execution Context?

Every time JavaScript starts running code, it creates an environment called an **Execution Context**.

Think of it as a workspace.

```text
+--------------------------------+
| Execution Context              |
|--------------------------------|
| Variables                      |
| Functions                      |
| this                           |
| Scope                          |
+--------------------------------+
```

Whenever JavaScript needs to execute code, it creates one.

---

## Variables Live in Memory

Consider this:

```js
let x = 5;
```

JavaScript creates a place in memory.

```text
Memory

+-----------+
| x |   5   |
+-----------+
```

Now:

```js
x = 10;
```

Memory changes.

```text
Before

+-----------+
| x |   5   |
+-----------+

After

+-----------+
| x |  10   |
+-----------+
```

Notice something important.

JavaScript did **not** create another variable.

It simply updated the existing memory location.

---

## The Call Stack

JavaScript executes code using a stack.

Think of stacked books.

```text
+------------+
| function C |
+------------+
| function B |
+------------+
| function A |
+------------+
```

Only the top book is active.

When a function finishes, it is removed.

Example:

```js
function greet() {
    console.log("Hello");
}

greet();
```

Execution:

```text
Before

Stack

(empty)

↓

Call greet()

+---------+
| greet() |
+---------+

↓

console.log()

+-------------+
| console.log |
+-------------+
| greet()     |
+-------------+

↓

console.log finishes

+---------+
| greet() |
+---------+

↓

greet finishes

(empty)
```

# Internal Mechanics

Suppose JavaScript sees:

```js
let count = 0;
```

Internally it does something similar to:

```text
Allocate memory

Create variable "count"

Store value 0
```

Memory:

```text
+---------------+
| count | 0     |
+---------------+
```

Nothing magical happened.

Just memory allocation.

# Example 1

```js
let count = 0;

count = count + 1;

console.log(count);
```

## Line-by-Line Explanation

### Line 1

```js
let count = 0;
```

Memory becomes

```text
+---------------+
| count | 0     |
+---------------+
```

---

### Line 2

```js
count = count + 1;
```

JavaScript does **not** instantly know the answer.

It performs steps.

Step 1

Read current value.

```text
count

↓

0
```

Step 2

Add 1.

```text
0 + 1

↓

1
```

Step 3

Replace memory.

```text
Before

count → 0

After

count → 1
```

---

### Line 3

```js
console.log(count);
```

Reads memory.

Outputs

```text
1
```

```text
| apples | 3  |
+-------------+

↓

Increment

↓

+-------------+
| apples | 4  |
+-------------+
```

# Visual Walkthrough

Suppose we eventually write:

```js
for (let i = 0; i < 3; i++) {
    console.log(i);
}
```

Don't worry about the syntax yet.

Let's see what the engine does.

---

## Step 1

Create variable.

```text
Memory

+---------+
| i | 0   |
+---------+
```

---

## Step 2

Evaluate condition.

```text
0 < 3

↓

true
```

---

## Step 3

Run body.

```text
console.log(0)
```

Output

```text
0
```

---

## Step 4

Increment.

```text
i++

↓

1
```

Memory

```text
+---------+
| i | 1   |
+---------+
```

---

## Step 5

Check condition again.

```text
1 < 3

↓

true
```

Repeat.

Eventually

```text
i

↓

3
```

Condition becomes

```text
3 < 3

↓

false
```

The loop exits.

# How a Loop Executes

The JavaScript engine repeatedly performs the following sequence:

```text
┌────────────────────┐
│ Initialize variable│
└─────────┬──────────┘
          │
          ▼
┌────────────────────┐
│ Evaluate condition │
└─────────┬──────────┘
          │
   True   │   False
          ▼
┌────────────────────┐
│ Execute loop body  │
└─────────┬──────────┘
          │
          ▼
┌────────────────────┐
│ Update variable    │
└─────────┬──────────┘
          │
          └───────────────► Back to condition
```

Notice something.

The engine never says:

> "I'm looping."

Instead, it keeps **jumping back** to evaluate the condition again.

That repeated jump is what creates a loop.

---

# Why Loops Stop

A loop stops only when its condition becomes `false`.

Example:

```js
let i = 0;

while (i < 5) {
    i++;
}
```

Timeline:

```text
i = 0

↓

0 < 5

true

↓

i++

↓

1

↓

1 < 5

true

↓

2

↓

3

↓

4

↓

5

↓

5 < 5

false

↓

Exit
```

The **condition** controls the loop.

The **update** changes the condition.

Without the update, the condition may never change.

---

# Infinite Loops

Example:

```js
let i = 0;

while (i < 5) {
    console.log(i);
}
```

Why is this infinite?

Because `i` never changes.

The condition is always:

```text
0 < 5

↓

true
```

The engine repeats forever:

```text
Check

↓

Run

↓

Check

↓

Run

↓

Check

↓

Run

↓

...
```

> **Warning**
>
> Every loop needs a path that eventually makes its condition become `false`.

# Common Mistakes

| Mistake | Why it happens | Fix |
|----------|----------------|-----|
| Forgetting to update the loop variable | Condition never changes | Increment or decrement the variable |
| Using the wrong comparison operator | Loop runs too many or too few times | Carefully define the stopping condition |
| Modifying the wrong variable | The controlling variable stays unchanged | Update the correct variable |
| Assuming `i++` creates a new variable | It updates the existing variable in memory | Remember variables are mutable when declared with `let` |

---

# Performance

| Operation | Time | Space |
|-----------|------|-------|
| Read variable | **O(1)** | O(1) |
| Update variable | **O(1)** | O(1) |
| Compare values | **O(1)** | O(1) |
| Increment | **O(1)** | O(1) |

At this stage, every operation we've discussed takes constant time.

We'll later combine these operations into algorithms whose total cost depends on how many times they repeat.

# Best Practices

- Think of a loop as **condition → body → update**.
- Always ask: *What changes each iteration?*
- Make sure the condition can eventually become `false`.
- Trace the loop on paper before running it.
- Visualize memory updates step by step.

> **Tip**
>
> If you can explain how memory changes after each line of code, you truly understand what the JavaScript engine is doing.

# Summary

In this lesson, you learned that loops are not magical language features. From the engine's point of view, a loop is simply a sequence of repeated actions:

1. Initialize state.
2. Evaluate a condition.
3. Execute statements.
4. Update state.
5. Repeat until the condition becomes `false`.

Understanding these mechanics is the key to writing correct, efficient, and debuggable algorithms.

---

# Key Takeaways

- A variable is a named location in memory.
- Updating a variable changes its stored value rather than creating a new one.
- JavaScript executes code within an Execution Context.
- Function calls are managed using the Call Stack.
- Loops repeat by repeatedly checking a condition.
- The loop variable must change for the condition to eventually become `false`.
- Infinite loops occur when the condition never changes or never becomes `false`.

---

# Glossary

| Term | Definition |
|------|------------|
| Execution Context | The environment where JavaScript executes code. |
| Call Stack | A stack that tracks active function calls. |
| Memory | Storage used to hold variables and values. |
| Variable | A named reference to a value in memory. |
| Condition | An expression that evaluates to `true` or `false`. |
| Increment | Increasing a value, commonly by one. |
| Infinite Loop | A loop that never terminates because its condition never becomes `false`. |
