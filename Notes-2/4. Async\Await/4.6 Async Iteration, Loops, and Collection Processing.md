# 1. Introduction

Real applications frequently need to perform asynchronous work for every item in a collection.

Examples include:

* Uploading multiple files.
* Synchronizing pending transactions.
* Fetching details for several users.
* Sending several emails.
* Validating records with a server.
* Processing a queue of orders.
* Downloading images.
* Reading multiple files.
* Saving several form entries.
* Updating inventory items.

A beginner may attempt this:

```javascript
const files = [
  "report.pdf",
  "photo.png",
  "invoice.docx",
];

files.forEach(async file => {
  await uploadFile(file);

  console.log(`${file} uploaded`);
});

console.log("All uploads complete");
```

The expected output might be:

```text
report.pdf uploaded
photo.png uploaded
invoice.docx uploaded
All uploads complete
```

But the actual output is likely to begin with:

```text
All uploads complete
```

The individual upload messages appear later.

Why?

Because `forEach()` does not understand or await the Promises returned by its callback.

This is one of the most common asynchronous JavaScript mistakes.

To process collections correctly, you must understand three separate questions:

1. Should items be processed sequentially?
2. Should items be processed concurrently?
3. Should concurrency be limited?

The answer determines which loop or array method you should use.

# 2. Why Async Collection Processing Is Difficult

Synchronous collection processing is straightforward.

```javascript
for (const number of [1, 2, 3]) {
  console.log(number);
}
```

Output:

```text
1
2
3
```

Each iteration completes before the next one begins.

Asynchronous operations introduce waiting.

```javascript
for (const file of files) {
  await uploadFile(file);
}
```

Now every iteration may pause the async function.

Other approaches create multiple Promises:

```javascript
const uploadPromises = files.map(file => uploadFile(file));
```

At this point, the code has not produced upload results. It has produced an array of Promises.

```text
files
  ↓ map(async callback)
[
  Promise,
  Promise,
  Promise
]
```

You must decide how those Promises should be coordinated.

Common possibilities include:

```javascript
await Promise.all(uploadPromises);
```

or:

```javascript
await Promise.allSettled(uploadPromises);
```

or processing them in smaller groups.

The difficulty comes from separating:

* Iteration.
* Promise creation.
* Promise waiting.
* Failure behavior.
* Result ordering.
* Concurrency control.

# 3. Mental Model

Imagine processing customers at a service desk.

## Sequential model

One worker handles one customer completely before moving to the next.

```text
Customer 1
    ↓ finished
Customer 2
    ↓ finished
Customer 3
```

This resembles:

```javascript
for (const customer of customers) {
  await serveCustomer(customer);
}
```

## Unlimited concurrent model

Every customer is sent to a separate available worker immediately.

```text
Customer 1 → Worker 1
Customer 2 → Worker 2
Customer 3 → Worker 3
Customer 4 → Worker 4
```

This resembles:

```javascript
await Promise.all(
  customers.map(customer => serveCustomer(customer))
);
```

## Controlled-concurrency model

Only three service desks are available.

```text
Desk 1 → Customer 1 → Customer 4 → Customer 7
Desk 2 → Customer 2 → Customer 5 → Customer 8
Desk 3 → Customer 3 → Customer 6 → Customer 9
```

This resembles a concurrency pool with a limit of three.

Each model is useful in different situations.

# 4. Synchronous Loops Revisited

Before adding asynchronous operations, review how ordinary loops work.

## Traditional `for` loop

```javascript
const numbers = [10, 20, 30];

for (let index = 0; index < numbers.length; index += 1) {
  console.log(numbers[index]);
}
```

Benefits:

* Access to the index.
* Full control over iteration.
* Supports `break`.
* Supports `continue`.
* Supports reverse iteration.
* Supports custom increments.

## `for...of`

```javascript
for (const number of numbers) {
  console.log(number);
}
```

Benefits:

* Clear syntax.
* Direct access to values.
* Supports `break`.
* Supports `continue`.
* Works naturally with `await`.

## `forEach()`

```javascript
numbers.forEach(number => {
  console.log(number);
});
```

Benefits:

* Concise.
* Useful for simple synchronous side effects.

Limitations:

* Cannot be stopped using `break`.
* Does not await async callbacks.
* Does not return transformed values.
* Always returns `undefined`.

## `map()`

```javascript
const doubled = numbers.map(number => number * 2);
```

Benefits:

* Produces a new array.
* Appropriate for transformations.
* When given an async callback, produces an array of Promises.

# 5. `await` Inside a Traditional `for` Loop

A traditional `for` loop works correctly with `await` inside an async function.

```javascript
async function uploadFiles(files) {
  for (let index = 0; index < files.length; index += 1) {
    const file = files[index];

    await uploadFile(file);

    console.log(
      `Uploaded ${index + 1} of ${files.length}: ${file.name}`
    );
  }
}
```

Execution:

```text
Upload file 1
    ↓ wait
Log completion
    ↓
Upload file 2
    ↓ wait
Log completion
    ↓
Upload file 3
```

## When this is useful

Use a sequential `for` loop when:

* Order matters.
* Each operation depends on the previous result.
* The server allows only one active request.
* You need to stop after a certain result.
* You need index-based logic.
* You need controlled delays between requests.
* Operations modify the same shared resource.

## Internal execution

```javascript
for (let index = 0; index < files.length; index += 1) {
  await uploadFile(files[index]);
}
```

Conceptually:

```text
Iteration 0 begins
    ↓
Call uploadFile(files[0])
    ↓
Receive Promise
    ↓
Pause current async function
    ↓
Promise fulfills
    ↓
Resume function
    ↓
Increment index
    ↓
Iteration 1 begins
```

The next iteration does not begin while the current `await` is pending.

# 6. `await` Inside `for...of`

`for...of` is often the clearest syntax for sequential asynchronous collection processing.

```javascript
async function processTransactions(transactions) {
  for (const transaction of transactions) {
    await uploadTransaction(transaction);
  }
}
```

## Step-by-step execution

Given:

```javascript
const transactions = [
  {
    id: 1,
  },
  {
    id: 2,
  },
  {
    id: 3,
  },
];
```

Execution:

```text
transaction = { id: 1 }
    ↓ upload and wait
transaction = { id: 2 }
    ↓ upload and wait
transaction = { id: 3 }
    ↓ upload and wait
Function completes
```

## Complete example

```javascript
function delay(milliseconds) {
  return new Promise(resolve => {
    setTimeout(resolve, milliseconds);
  });
}

async function processTransactions(transactions) {
  for (const transaction of transactions) {
    console.log(`Starting transaction ${transaction.id}`);

    await delay(500);

    console.log(`Finished transaction ${transaction.id}`);
  }

  console.log("All transactions processed");
}

processTransactions([
  {
    id: 1,
  },
  {
    id: 2,
  },
  {
    id: 3,
  },
]);
```

Output:

```text
Starting transaction 1
Finished transaction 1
Starting transaction 2
Finished transaction 2
Starting transaction 3
Finished transaction 3
All transactions processed
```

## Why professionals often prefer `for...of`

It clearly communicates:

> Process these values one at a time, waiting for each operation.

This is easier to understand than forcing sequential behavior through Promise chains or `reduce()`.

# 7. Why `forEach()` Does Not Await Async Callbacks

Consider:

```javascript
async function uploadFiles(files) {
  files.forEach(async file => {
    await uploadFile(file);

    console.log(`${file.name} uploaded`);
  });

  console.log("All files uploaded");
}
```

This function does not wait for all uploads.

## Why?

`forEach()` is a synchronous array method.

Its conceptual implementation resembles:

```javascript
function forEach(array, callback) {
  for (let index = 0; index < array.length; index += 1) {
    callback(
      array[index],
      index,
      array
    );
  }
}
```

Notice that it calls:

```javascript
callback(...)
```

It does not do:

```javascript
await callback(...)
```

When the callback is async, it returns a Promise.

`forEach()` ignores that return value.

## Internal flow

```javascript
files.forEach(async file => {
  await uploadFile(file);
});
```

Conceptually:

```text
Iteration 1
    ↓ call async callback
    ↓ callback returns Promise immediately
    ↓ forEach ignores Promise

Iteration 2
    ↓ call async callback
    ↓ callback returns Promise immediately
    ↓ forEach ignores Promise

Iteration 3
    ↓ call async callback
    ↓ callback returns Promise immediately
    ↓ forEach ignores Promise

forEach returns undefined
```

The uploads continue, but the outer function has no collection of Promises to await.

## Demonstration

```javascript
function delay(milliseconds, value) {
  return new Promise(resolve => {
    setTimeout(() => {
      resolve(value);
    }, milliseconds);
  });
}

async function demo() {
  const numbers = [1, 2, 3];

  numbers.forEach(async number => {
    await delay(500);

    console.log(number);
  });

  console.log("Finished");
}

demo();
```

Likely output:

```text
Finished
1
2
3
```

## Another important problem: error handling

```javascript
async function processItems(items) {
  try {
    items.forEach(async item => {
      await processItem(item);
    });
  } catch (error) {
    console.error("Processing failed:", error);
  }
}
```

The outer `try...catch` may not catch later callback rejections because `forEach()` does not await the returned Promises.

> [!WARNING]
> Do not use `forEach()` when the surrounding code must wait for async callbacks or handle their rejections as one coordinated workflow.

# 8. What `async` Callbacks Return

Every async function returns a Promise.

Therefore:

```javascript
const callback = async number => {
  return number * 2;
};
```

Calling:

```javascript
callback(5);
```

returns:

```text
Promise fulfilled with 10
```

This affects array methods.

## Async `map()`

```javascript
const numbers = [1, 2, 3];

const results = numbers.map(async number => {
  return number * 2;
});

console.log(results);
```

Output resembles:

```text
[
  Promise,
  Promise,
  Promise
]
```

It does not produce:

```text
[2, 4, 6]
```

because each async callback returns a Promise.

## Async `filter()` problem

Consider:

```javascript
const numbers = [1, 2, 3, 4];

const evenNumbers = numbers.filter(async number => {
  return number % 2 === 0;
});
```

This does not asynchronously filter the values.

Each callback returns a Promise object, and Promise objects are truthy.

The result may incorrectly contain every number.

## Async `some()` and `every()`

These methods also do not await async predicates.

```javascript
numbers.some(async number => {
  return number > 3;
});
```

The returned Promise is truthy, so the method may return an incorrect result immediately.

> [!IMPORTANT]
> Array methods such as `filter()`, `some()`, `every()`, and `find()` are designed for synchronous predicates. They do not automatically await async callbacks.

# 9. Using `map()` with Async Functions

`map()` can still be useful with async callbacks.

It transforms each item into a Promise.

```javascript
const promises = files.map(async file => {
  const result = await uploadFile(file);

  return {
    file,
    result,
  };
});
```

Result:

```text
[
  Promise for file 1,
  Promise for file 2,
  Promise for file 3
]
```

This is useful when combined with a Promise combinator.

```javascript
const results = await Promise.all(promises);
```

Full version:

```javascript
async function uploadFiles(files) {
  const uploadPromises = files.map(async file => {
    const uploadResult = await uploadFile(file);

    return {
      fileName: file.name,
      uploadResult,
    };
  });

  return Promise.all(uploadPromises);
}
```

Because all callbacks begin during the synchronous `map()` pass, uploads start concurrently.

# 10. Combining `map()` with `Promise.all()`

This is one of the most common professional async collection patterns.

```javascript
const results = await Promise.all(
  items.map(async item => {
    return processItem(item);
  })
);
```

It can often be simplified:

```javascript
const results = await Promise.all(
  items.map(item => processItem(item))
);
```

Or:

```javascript
const results = await Promise.all(
  items.map(processItem)
);
```

The final version is appropriate only when `processItem` accepts the same arguments that `map()` supplies or ignores the extra arguments safely.

## Execution model

```text
map begins
    ↓
Call processItem(item 1) → Promise 1
Call processItem(item 2) → Promise 2
Call processItem(item 3) → Promise 3
    ↓
map returns [Promise 1, Promise 2, Promise 3]
    ↓
Promise.all waits
    ↓
Return ordered result array
```

## Example

```javascript
async function fetchUsers(userIds) {
  return Promise.all(
    userIds.map(userId => fetchUser(userId))
  );
}
```

Usage:

```javascript
const users = await fetchUsers([
  1,
  2,
  3,
]);
```

Even if user 3 finishes first, the results remain ordered by input:

```javascript
[
  userForId1,
  userForId2,
  userForId3,
]
```

# 11. Sequential Collection Processing

Sequential processing waits for each item.

```javascript
async function processSequentially(items, processItem) {
  const results = [];

  for (const item of items) {
    const result = await processItem(item);

    results.push(result);
  }

  return results;
}
```

## Benefits

* Predictable order.
* Easy error handling.
* Supports `break` and `continue`.
* Low resource usage.
* Appropriate for dependent operations.
* Appropriate for strict rate limits.

## Cost

Total duration is approximately the sum of all operation durations.

```text
Item 1: 1 second
Item 2: 1 second
Item 3: 1 second

Total: approximately 3 seconds
```

## Example: ordered receipt printing

```javascript
async function printReceipts(receipts) {
  for (const receipt of receipts) {
    await printReceipt(receipt);
  }
}
```

This may be required because a physical printer should receive one job at a time.

---

# 12. Concurrent Collection Processing

Concurrent processing starts every item operation immediately.

```javascript
async function processConcurrently(items, processItem) {
  return Promise.all(
    items.map(item => processItem(item))
  );
}
```

## Benefits

* Often much faster.
* Concise.
* Appropriate for independent operations.
* Results preserve input order.

## Risks

* Unlimited resource usage.
* One rejection rejects `Promise.all()`.
* May overwhelm APIs or servers.
* May cause rate-limit failures.
* May consume large amounts of memory.

## Example: independent profile requests

```javascript
async function fetchProfiles(userIds) {
  return Promise.all(
    userIds.map(userId => fetchProfile(userId))
  );
}
```

# 13. Controlled-Concurrency Processing

Controlled concurrency allows only a fixed number of operations to remain active.

## Why it exists

Suppose there are 1,000 transactions to upload.

Sequential execution may be too slow.

Unlimited concurrency may overload the server.

A concurrency limit of five provides a compromise.

```text
Maximum active uploads: 5
```

## Basic worker-pool implementation

```javascript
async function mapWithConcurrency(
  items,
  concurrencyLimit,
  asyncMapper
) {
  if (!Array.isArray(items)) {
    throw new TypeError("items must be an array.");
  }

  if (
    !Number.isInteger(concurrencyLimit) ||
    concurrencyLimit < 1
  ) {
    throw new RangeError(
      "concurrencyLimit must be a positive integer."
    );
  }

  if (typeof asyncMapper !== "function") {
    throw new TypeError(
      "asyncMapper must be a function."
    );
  }

  const results = new Array(items.length);
  let nextIndex = 0;

  async function worker() {
    while (true) {
      const currentIndex = nextIndex;
      nextIndex += 1;

      if (currentIndex >= items.length) {
        return;
      }

      results[currentIndex] = await asyncMapper(
        items[currentIndex],
        currentIndex,
        items
      );
    }
  }

  const workerCount = Math.min(
    concurrencyLimit,
    items.length
  );

  const workers = Array.from(
    {
      length: workerCount,
    },
    () => worker()
  );

  await Promise.all(workers);

  return results;
}
```

## Usage

```javascript
const uploadedTransactions = await mapWithConcurrency(
  transactions,
  3,
  uploadTransaction
);
```

## Execution

```text
Worker 1 → Item 1 → Item 4 → Item 7
Worker 2 → Item 2 → Item 5 → Item 8
Worker 3 → Item 3 → Item 6 → Item 9
```

At most three operations are active.

## Result order

Even though tasks finish in unpredictable order, this implementation writes each result into its original index.

```javascript
results[currentIndex] = ...
```

Therefore, output order matches input order.

# 14. Error Handling Strategies

Collection processing needs a clear error policy.

Possible policies include:

1. Stop at the first error.
2. Continue after errors.
3. Retry failed items.
4. Return partial results.
5. Collect every result and error.
6. Stop only for specific error types.

## 14.1 Stop at the first error

Sequential:

```javascript
async function processSequentially(items) {
  for (const item of items) {
    await processItem(item);
  }
}
```

If one operation rejects, the loop stops unless the error is caught.

Concurrent:

```javascript
await Promise.all(
  items.map(processItem)
);
```

The combined Promise rejects after the first observed rejection.

## 14.2 Continue after individual errors

```javascript
async function processWithIndividualErrors(items) {
  const results = [];

  for (const item of items) {
    try {
      const value = await processItem(item);

      results.push({
        status: "fulfilled",
        value,
      });
    } catch (error) {
      results.push({
        status: "rejected",
        reason: error,
      });
    }
  }

  return results;
}
```

This version is sequential but preserves every outcome.

## 14.3 Concurrent partial success

```javascript
const results = await Promise.allSettled(
  items.map(processItem)
);
```

## 14.4 Catch errors inside `map()`

```javascript
const results = await Promise.all(
  items.map(async item => {
    try {
      const value = await processItem(item);

      return {
        status: "fulfilled",
        value,
      };
    } catch (error) {
      return {
        status: "rejected",
        reason: error,
      };
    }
  })
);
```

Because each callback catches its own error, the outer `Promise.all()` fulfills with status objects.

## 14.5 Error context

Add information about the failed item.

```javascript
async function processItemWithContext(item) {
  try {
    return await processItem(item);
  } catch (error) {
    throw new Error(
      `Failed to process item ${item.id}`,
      {
        cause: error,
      }
    );
  }
}
```

This produces more useful diagnostics.

# 15. Result Ordering

Concurrent completion order is not guaranteed.

Given:

```javascript
const tasks = [
  delay(1000, "A"),
  delay(200, "B"),
  delay(500, "C"),
];
```

Completion order:

```text
B
C
A
```

But:

```javascript
const results = await Promise.all(tasks);
```

returns:

```javascript
[
  "A",
  "B",
  "C",
]
```

The results follow input order.

## Completion-order processing

Sometimes you need results as soon as each one completes.

That requires a different pattern, such as:

* Individual `.then()` handlers.
* An async queue.
* Async iterators.
* A completion-channel abstraction.
* Stream processing.

This will be explored further in the next lesson.

# 16. Breaking, Continuing, and Returning

One advantage of ordinary loops is control flow.

## `continue`

Skip the current item:

```javascript
async function processValidTransactions(transactions) {
  for (const transaction of transactions) {
    if (!transaction.isValid) {
      continue;
    }

    await uploadTransaction(transaction);
  }
}
```

## `break`

Stop processing:

```javascript
async function processUntilFailure(items) {
  for (const item of items) {
    const result = await processItem(item);

    if (result.shouldStop) {
      break;
    }
  }
}
```

## `return`

Exit the entire function:

```javascript
async function findFirstAvailableServer(servers) {
  for (const server of servers) {
    try {
      const response = await checkServer(server);

      return response;
    } catch (error) {
      console.warn(`${server} unavailable`);
    }
  }

  throw new Error("No servers are available.");
}
```

`forEach()` does not support equivalent control-flow behavior.

A `return` inside its callback returns only from that callback, not from the outer function.

# 17. Using `reduce()` with Promises

Promises can be chained sequentially using `reduce()`.

```javascript
async function processSequentiallyWithReduce(
  items,
  processItem
) {
  return items.reduce(
    async (previousPromise, item) => {
      const results = await previousPromise;
      const result = await processItem(item);

      results.push(result);

      return results;
    },
    Promise.resolve([])
  );
}
```

This works, but it is usually less readable than `for...of`.

A simpler loop:

```javascript
async function processSequentially(items, processItem) {
  const results = [];

  for (const item of items) {
    results.push(
      await processItem(item)
    );
  }

  return results;
}
```

## When Promise `reduce()` may appear

You may encounter it in:

* Functional-programming codebases.
* Older Promise-heavy code.
* Pipelines that intentionally avoid mutable local arrays.
* Libraries emphasizing expression-based composition.

## Professional recommendation

Prefer `for...of` when it makes the sequential workflow clearer.

Use `reduce()` only when the team understands the pattern and it genuinely improves composition.

# 18. Choosing the Correct Pattern

Use this decision guide.

## Does each item depend on the previous result?

Yes:

```javascript
for (const item of items) {
  await processItem(item);
}
```

## Must order be preserved during execution?

Yes:

```javascript
for (const item of items) {
  await processItem(item);
}
```

## Are all operations independent and the collection small?

Yes:

```javascript
await Promise.all(
  items.map(processItem)
);
```

## Should every result be preserved after failures?

Yes:

```javascript
await Promise.allSettled(
  items.map(processItem)
);
```

## Is the collection large or the external service limited?

Yes:

```javascript
await mapWithConcurrency(
  items,
  concurrencyLimit,
  processItem
);
```

## Do you need `break`, `continue`, or early `return`?

Use:

```javascript
for
```

or:

```javascript
for...of
```

## Do you only need synchronous side effects?

`forEach()` may be appropriate.

## Example 3: Real-World Application — Transaction Synchronization

### Requirements

* Upload pending transactions.
* Allow at most three concurrent uploads.
* Preserve every result.
* Retry temporary failures.
* Keep permanent failures for later review.
* Return a complete synchronization report.

### Temporary error class

```javascript
class TemporaryNetworkError extends Error {
  constructor(message, options) {
    super(message, options);

    this.name = "TemporaryNetworkError";
  }
}
```

### Delay utility

```javascript
function delay(milliseconds) {
  return new Promise(resolve => {
    setTimeout(resolve, milliseconds);
  });
}
```

### Simulated upload

```javascript
async function uploadTransaction(transaction) {
  await delay(300);

  const randomValue = Math.random();

  if (randomValue < 0.2) {
    throw new TemporaryNetworkError(
      `Temporary failure for transaction ${transaction.id}`
    );
  }

  if (randomValue < 0.3) {
    throw new Error(
      `Permanent failure for transaction ${transaction.id}`
    );
  }

  return {
    transactionId: transaction.id,
    status: "uploaded",
  };
}
```

### Retry helper

```javascript
async function retry(
  operation,
  {
    retries = 3,
    delayMilliseconds = 500,
  } = {}
) {
  let lastError;

  for (let attempt = 1; attempt <= retries; attempt += 1) {
    try {
      return await operation();
    } catch (error) {
      lastError = error;

      const canRetry =
        error instanceof TemporaryNetworkError &&
        attempt < retries;

      if (!canRetry) {
        throw error;
      }

      await delay(delayMilliseconds);
    }
  }

  throw lastError;
}
```

### Synchronize one transaction

```javascript
async function synchronizeTransaction(transaction) {
  try {
    const value = await retry(
      () => uploadTransaction(transaction),
      {
        retries: 3,
        delayMilliseconds: 400,
      }
    );

    return {
      transaction,
      status: "fulfilled",
      value,
    };
  } catch (error) {
    return {
      transaction,
      status: "rejected",
      reason: error,
    };
  }
}
```

### Controlled synchronization

```javascript
async function synchronizeTransactions(transactions) {
  const results = await mapWithConcurrency(
    transactions,
    3,
    synchronizeTransaction
  );

  const successful = [];
  const failed = [];

  for (const result of results) {
    if (result.status === "fulfilled") {
      successful.push(result);
    } else {
      failed.push(result);
    }
  }

  return {
    total: results.length,
    successfulCount: successful.length,
    failedCount: failed.length,
    successful,
    failed,
  };
}
```

### Usage

```javascript
const pendingTransactions = [
  {
    id: 1,
    amount: 500,
  },
  {
    id: 2,
    amount: 800,
  },
  {
    id: 3,
    amount: 250,
  },
  {
    id: 4,
    amount: 950,
  },
  {
    id: 5,
    amount: 300,
  },

  pendingTransactions

  console.log(report);
});
```

### Professional design decisions

* `mapWithConcurrency()` prevents server overload.
* Each item catches its own error.
* Temporary errors are retried.
* Permanent errors are reported.
* Result order matches transaction order.
* The entire batch finishes even after individual failures.
* The final report is suitable for a UI or audit log.

# 20. Common Mistakes

## Mistake 1: Using `forEach()` with `await`

Incorrect:

```javascript
items.forEach(async item => {
  await processItem(item);
});
```

Correct sequential form:

```javascript
for (const item of items) {
  await processItem(item);
}
```

Correct concurrent form:

```javascript
await Promise.all(
  items.map(processItem)
);
```

---

## Mistake 2: Forgetting that async `map()` returns Promises

Incorrect:

```javascript
const results = items.map(async item => {
  return processItem(item);
});

console.log(results);
```

`results` is an array of Promises.

Correct:

```javascript
const results = await Promise.all(
  items.map(async item => {
    return processItem(item);
  })
);
```

---

## Mistake 3: Using async callbacks with `filter()`

Incorrect:

```javascript
const activeUsers = users.filter(async user => {
  return await isUserActive(user);
});
```

Correct two-stage approach:

```javascript
const activityResults = await Promise.all(
  users.map(async user => {
    return {
      user,
      isActive: await isUserActive(user),
    };
  })
);

const activeUsers = activityResults
  .filter(result => result.isActive)
  .map(result => result.user);
```

---

## Mistake 4: Unlimited concurrency

Risky:

```javascript
await Promise.all(
  fiftyThousandItems.map(processItem)
);
```

Use controlled concurrency.

---

## Mistake 5: Sequential processing when independence exists

Slow:

```javascript
for (const userId of userIds) {
  users.push(
    await fetchUser(userId)
  );
}
```

Potentially better:

```javascript
const users = await Promise.all(
  userIds.map(fetchUser)
);
```

---

## Mistake 6: Concurrent processing when order is required

Incorrect for a single printer:

```javascript
await Promise.all(
  receipts.map(printReceipt)
);
```

Better:

```javascript
for (const receipt of receipts) {
  await printReceipt(receipt);
}
```

---

## Mistake 7: Losing error context

Weak:

```javascript
throw error;
```

Better:

```javascript
throw new Error(
  `Failed to process transaction ${transaction.id}`,
  {
    cause: error,
  }
);
```

---

## Mistake 8: Expecting `break` to work in `forEach()`

Incorrect:

```javascript
items.forEach(item => {
  if (item.shouldStop) {
    break;
  }
});
```

This is a syntax error.

Use `for...of`.

# 21. Best Practices

## 21.1 Choose the execution policy explicitly

Before writing code, decide:

* Sequential.
* Concurrent.
* Concurrent with a limit.
* Stop on error.
* Continue on error.
* Preserve ordering.
* Process completion order.

## 21.2 Use `for...of` for readable sequential workflows

```javascript
for (const item of items) {
  await processItem(item);
}
```

## 21.3 Use `Promise.all()` with `map()` for small independent sets

```javascript
const results = await Promise.all(
  items.map(processItem)
);
```

## 21.4 Use `Promise.allSettled()` for complete batch reports

```javascript
const results = await Promise.allSettled(
  items.map(processItem)
);
```

## 21.5 Limit concurrency for large collections

A safe concurrency limit depends on:

* Server capacity.
* API limits.
* File sizes.
* Network quality.
* Device resources.
* Database connection limits.
* Business requirements.

## 21.6 Include item context in failures

Every failure report should identify the affected item.

## 21.7 Preserve original order unless completion order is required

Stable ordering makes UI rendering and auditing easier.

## 21.8 Avoid unnecessary async callbacks

This is redundant:

```javascript
items.map(async item => {
  return processItem(item);
});
```

When no extra `await` or transformation is needed:

```javascript
items.map(processItem);
```

## 21.9 Measure realistic workloads

A pattern that works for ten items may fail for ten thousand.

# 22. Performance Considerations

## Sequential duration

```text
Total ≈ duration1 + duration2 + duration3 + ...
```

## Unlimited concurrent duration

```text
Total ≈ duration of slowest operation
```

But only until resource contention becomes significant.

## Controlled-concurrency duration

```text
Total ≈ workload distributed among N workers
```

This often provides the best balance.

## Main performance factors

* Number of items.
* Average task duration.
* Network bandwidth.
* Request latency.
* Server rate limits.
* File size.
* Memory usage.
* CPU work before and after awaits.
* Error and retry rates.
* Concurrency limit.

## Promise creation

Each async callback creates a Promise.

For ordinary application sizes, this overhead is small.

For extremely large collections, allocating thousands or millions of Promises simultaneously can consume significant memory.

## Rendering

Do not update the DOM after every item if hundreds of items complete rapidly.

Consider batching UI updates.

```text
100 network completions
    ↓
100 separate DOM updates
    ↓
Potential rendering overhead
```

A better approach may collect results and render periodically.

# 24. Interview Questions and Answers

## Beginner Questions

### 1. Can `await` be used inside a `for...of` loop?

Yes, provided the loop is inside an async function or another context where `await` is valid.

### 2. Why does `forEach()` not wait for async callbacks?

Because `forEach()` calls each callback synchronously and ignores the Promise returned by an async callback.

### 3. What does `map()` return when its callback is async?

An array of Promises.

### 4. How do you obtain the resolved values from an async `map()`?

Use `Promise.all()`:

```javascript
const results = await Promise.all(
  items.map(async item => {
    return processItem(item);
  })
);
```

### 5. Which loop is commonly used for sequential async processing?

`for...of`.

---

## Intermediate Questions

### 1. What is the difference between these patterns?

```javascript
for (const item of items) {
  await processItem(item);
}
```

and:

```javascript
await Promise.all(
  items.map(processItem)
);
```

The first processes items sequentially. The second starts all item operations concurrently and waits for all of them.

### 2. Why is async `filter()` usually incorrect?

Because `filter()` expects a synchronous Boolean result. An async callback returns a Promise, and Promise objects are truthy.

### 3. How can asynchronous filtering be implemented?

First asynchronously calculate predicate results, then synchronously filter.

### 4. Why might `Promise.all(items.map(...))` be dangerous?

It can start too many operations simultaneously and overwhelm resources.

### 5. How can batch processing preserve every failure?

Use `Promise.allSettled()` or catch each item error and return structured status objects.

### 6. Does `Promise.all()` return results in completion order?

No. It returns results in input order.

---

## Advanced Questions

### 1. How would you implement controlled concurrency without a library?

Create a fixed number of async worker functions that pull the next unprocessed index from shared state until all items are processed.

### 2. Why does a worker-pool implementation preserve input order?

Because each result is written into an array at the original item index rather than appended in completion order.

### 3. What are the trade-offs between sequential, unlimited concurrent, and limited-concurrency processing?

Sequential processing is predictable and resource-safe but slow. Unlimited concurrency is fast for small independent sets but can exhaust resources. Limited concurrency balances throughput and resource usage.

### 4. Why might an outer `try...catch` fail to catch errors from async `forEach()` callbacks?

Because `forEach()` does not await or return the callback Promises. The outer async flow may complete before those Promises reject.

### 5. How would you asynchronously find the first matching item?

Possible strategies include:

* Sequentially await each predicate and return on the first match.
* Evaluate predicates concurrently and select the earliest input match after all finish.
* Evaluate concurrently and return the first completed successful match, depending on the required semantics.

The correct strategy depends on whether "first" means input order or completion order.

### 6. Why may retries interact badly with high concurrency?

Retries increase the number of active operations and may amplify server overload. Retry systems should use limits, delays, and backoff.

# 28. Summary

In this lesson, you learned how asynchronous operations interact with JavaScript loops and collection methods.

You learned that:

* `await` works naturally inside `for` and `for...of`.
* Sequential loops wait for each item before continuing.
* `forEach()` does not await async callbacks.
* Async callbacks always return Promises.
* Async `map()` creates an array of Promises.
* `Promise.all()` converts that Promise array into resolved values.
* `filter()`, `some()`, `every()`, and `find()` do not await async predicates.
* Async filtering requires separate asynchronous evaluation and synchronous selection.
* `for...of` supports `break`, `continue`, and early `return`.
* Small independent collections can be processed concurrently.
* Large collections should often use controlled concurrency.
* `Promise.allSettled()` preserves every success and failure.
* Result order and completion order are different concepts.
* Professional collection processing requires an explicit execution and error policy.

The central rule is:

```text
Sequential dependency or strict order
        ↓
Use for...of with await

Small independent collection
        ↓
Use map() + Promise.all()

Large independent collection
        ↓
Use controlled concurrency

Need every result after failures
        ↓
Use Promise.allSettled()
```
