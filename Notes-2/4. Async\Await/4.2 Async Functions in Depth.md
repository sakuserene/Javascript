# What Makes a Function Async?

Consider this function.

```javascript
function add(a, b) {

    return a + b;

}
```

It executes immediately.

It returns immediately.

Execution

```text
Call Function

↓

Run Code

↓

Return Value

↓

Finished
```

Nothing unusual happens.

Now compare it with

```javascript
async function add(a, b) {

    return a + b;

}
```

Only one keyword changed.

But internally,

JavaScript behaves very differently.

---

# The async Keyword

The keyword

```javascript
async
```

changes the **contract** of the function.

A normal function promises:

> "I'll return a value."

An async function promises:

> "I'll return a Promise."

This distinction is extremely important.

---

# Internal Transformation

Suppose we write

```javascript
async function greet() {

    return "Hello";

}
```

JavaScript conceptually transforms it into something like

```javascript
function greet() {

    return Promise.resolve(

        "Hello"

    );

}
```

Notice

The returned value

```text
Hello
```

never leaves the function directly.

Instead,

it becomes

```text
Promise

↓

Fulfilled

↓

Hello
```

---

# Execution Timeline

Calling

```javascript
greet();
```

produces

```text
Call Function

↓

Create Promise

↓

Execute Body

↓

Return Value

↓

Resolve Promise

↓

Return Promise
```

The caller immediately receives

```
Promise
```

not

```
Hello
```

---

# Example

```javascript
async function greet() {

    return "Hello";

}

const result = greet();

console.log(result);
```

Output

```text
Promise {<fulfilled>: "Hello"}
```

Notice

The function did not return

```text
Hello
```

It returned

```text
Promise
```

---

# Getting the Value

To access the value

we must either

```javascript
greet()

.then(console.log);
```

or

```javascript
const message = await greet();
```

Output

```text
Hello
```

# Returning Normal Values

Suppose

```javascript

    return number * number;

```

Execution

```
25

↓

Promise.resolve(25)

↓

Fulfilled Promise
```

JavaScript automatically wraps the value.

# Returning Objects

```javascript
async function getUser() {

    return {

        name: "Alice",

        age: 30

    };

}
```

Internally

```text
Object

↓

Promise.resolve(object)

↓

Fulfilled Promise
```

# Returning Arrays

```javascript
async function numbers() {

    return [1,2,3];

}
```

Internally

```
Array

↓

Promise.resolve(Array)
```

Everything returned becomes a fulfilled Promise.

---

# Returning Another Promise

Suppose

```javascript
async function loadUser() {

    return fetch("/user");

}
```

Question

Does JavaScript produce

```
Promise

↓

Promise

↓

User
```

No.

Instead,

JavaScript automatically adopts the returned Promise.

Conceptually

```
fetch()

↓

Promise

↓

Return Same State

↓

Caller Receives Promise
```

JavaScript avoids creating nested Promises.

---

# Promise Adoption

This behavior is called

> **Promise Adoption**

If an async function returns

```javascript
Promise.resolve(100);
```

the caller simply receives

```
Promise

↓

100
```

instead of

```
Promise

↓

Promise

↓

100
```

This automatic flattening is similar to how `.then()` unwraps returned Promises.

---

# Throwing Errors

Suppose

```javascript
async function login() {

    throw new Error(

        "Invalid Password"

    );

}
```

Question

Does the application crash?

No.

Internally

JavaScript transforms

```text
throw Error
```

into

```text
Promise.reject(

Error

)
```

Conceptually

```javascript
async function login() {

    throw new Error(

        "Invalid Password"

    );

}
```

becomes

```javascript
function login() {

    return Promise.reject(

        new Error(

            "Invalid Password"

        )

    );

}
```

---

# Example

```javascript
async function login() {

    throw new Error(

        "Access Denied"

    );

}

login()

.catch(error => {

    console.log(

        error.message

    );

});
```

Output

```text
Access Denied
```

# Event Loop Interaction

Suppose

```javascript
async function demo() {

    return 100;

}

demo()

.then(console.log);

console.log("Finished");
```

Output

```text
Finished

100
```

Why?

Because

```
demo()

↓

Fulfilled Promise

↓

.then()

↓

Microtask Queue

↓

console.log(100)
```

The Promise reaction waits until synchronous code finishes.

# Common Mistakes

## Mistake 1

Expecting async functions to return ordinary values.

Incorrect

```javascript
const result = greet();
```

Correct

```javascript
const result = await greet();
```

or

```javascript
greet()

.then(...);
```

---

## Mistake 2

Returning nested Promises manually.

JavaScript already adopts returned Promises automatically.

---

## Mistake 3

Thinking `throw` crashes async functions.

Instead,

`throw` creates a rejected Promise.

---

## Mistake 4

Forgetting that async functions execute immediately until the first `await`.

Only when `await` is encountered can the function pause.

---

# Best Practices

- Declare functions `async` only when they naturally perform asynchronous work.
- Return meaningful values rather than manually wrapping them with `Promise.resolve()` inside async functions.
- Throw `Error` objects instead of strings.
- Remember that async functions are Promise factories.
- Use descriptive names that communicate asynchronous intent, such as `loadUser`, `fetchOrders`, or `saveTransaction`.

---

# Performance Considerations

Async functions themselves are lightweight.

However:

- Every async function creates a Promise.
- Avoid marking purely synchronous utility functions as `async` unless necessary.
- Excessive unnecessary Promises can slightly increase scheduling overhead.

Use async functions where asynchronous behavior genuinely exists.

# Interview Questions

### Beginner

**Q:** What does every async function return?

**Answer:**

A Promise.

---

### Intermediate

**Q:** What happens if an async function returns a normal value?

**Answer:**

JavaScript automatically wraps the value in a fulfilled Promise using behavior equivalent to `Promise.resolve(value)`.

---

### Advanced

**Q:** What happens if an async function throws an error?

**Answer:**

The returned Promise becomes rejected with that error, behaving similarly to `Promise.reject(error)`.
