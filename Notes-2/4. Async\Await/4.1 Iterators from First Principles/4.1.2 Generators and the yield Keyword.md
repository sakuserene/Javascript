# 1. Introduction

In the previous lesson, you built iterators manually.

A manual iterator looked similar to this:

```javascript
function createNumberIterator() {
  let current = 1;

  return {
    next() {
      if (current <= 3) {
        return {
          value: current++,
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },
  };
}
```

This works, but it requires significant machinery:

* A closure for the current position.
* A `next()` method.
* Iterator result objects.
* Manual state updates.
* Manual completion checks.
* Optional cleanup methods.

For complex sequences, manual iterators become difficult to read.

JavaScript introduced **generator functions** to make iterator creation easier.

The same sequence can be written as:

```javascript
function* generateNumbers() {
  yield 1;
  yield 2;
  yield 3;
}
```

JavaScript automatically creates the iterator machinery.

The generator still follows the iterator protocol, but the code is much closer to the order in which values should be produced.

---

# 2. Why Generators Were Created

Ordinary functions have a simple lifecycle.

```text
Call function
    ↓
Run from beginning
    ↓
Return once
    ↓
Function finishes
```

Example:

```javascript
function getNumber() {
  return 10;
}
```

Once `return` executes, the function is complete.

It cannot pause and continue later.

However, many programming problems involve producing a sequence gradually.

Examples include:

* Reading records one at a time.
* Traversing a tree.
* Generating numeric ranges.
* Processing pages of data.
* Building lazy transformation pipelines.
* Producing an infinite sequence.
* Coordinating staged workflows.
* Representing resumable computation.

Before generators, developers created closures and state machines manually.

Generator functions were designed to provide language-level support for:

```text
Start
    ↓
Produce one value
    ↓
Pause
    ↓
Resume later
    ↓
Produce another value
    ↓
Pause again
```

The key feature is that a generator can suspend its execution while preserving its local state.

---

# 3. What Is a Generator Function?

A **generator function** is a special function that can pause and resume its execution.

It is declared using:

```javascript
function*
```

and usually produces values using:

```javascript
yield
```

Example:

```javascript
function* generateColors() {
  yield "red";
  yield "green";
  yield "blue";
}
```

A generator function differs from an ordinary function in an important way:

> Calling it does not immediately execute its body.

Instead, calling it creates and returns a **generator object**.

```javascript
const generator =
  generateColors();
```

The generator object controls execution through methods such as:

```javascript
generator.next();
generator.return();
generator.throw();
```

# 4. Generator Function Syntax

## Function declaration

```javascript
function* generateValues() {
  yield 1;
  yield 2;
}
```

The asterisk may appear next to `function`:

```javascript
function* generateValues() {
}
```

Some code styles allow spacing:

```javascript
function *generateValues() {
}
```

The most common style is:

```javascript
function* generateValues() {
}
```

## Generator function expression

```javascript
const generateValues =
  function* () {
    yield 1;
    yield 2;
  };
```

## Named generator expression

```javascript
const generateValues =
  function* valuesGenerator() {
    yield 1;
    yield 2;
  };
```

## Generator method in an object

```javascript
const collection = {
  *values() {
    yield 1;
    yield 2;
  },
};
```

## Generator method in a class

```javascript
class NumberCollection {
  *values() {
    yield 1;
    yield 2;
  }
}
```

## No arrow generator functions

JavaScript does not support arrow generator syntax.

This is invalid:

```javascript
const generate = *() => {
};
```

There is no generator equivalent of an arrow function.

Use a generator declaration, expression, object method, or class method.

# 5. What Happens When a Generator Is Called?

Consider:

```javascript
function* demo() {
  console.log("Generator started");

  yield 10;

  console.log("Generator resumed");

  yield 20;
}
```

Call it:

```javascript
const generator = demo();
```

At this point, nothing inside the generator body has executed.

You will not yet see:

```text
Generator started
```

Instead, JavaScript creates a generator object in a suspended state.

```text
Call demo()
    ↓
Create generator object
    ↓
Body not executed
    ↓
Return generator object
```

Execution begins only when:

```javascript
generator.next();
```

is called.

This is one of the most important generator rules.

---

# 6. The Generator Object

Calling a generator function returns a generator object.

```javascript
function* values() {
  yield 1;
}

const generator = values();
```

The generator object:

* Is an iterator.
* Is iterable.
* Preserves execution state.
* Has a `next()` method.
* Has a `return()` method.
* Has a `throw()` method.
* Can be consumed by `for...of`.
* Can be consumed by spread syntax.
* Can be consumed by destructuring.

Check its iterator behavior:

```javascript
console.log(
  generator[Symbol.iterator]() === generator
);
```

Output:

```text
true
```

The generator object returns itself from `[Symbol.iterator]()`.

This means it is generally a single-use iterator.

Once exhausted, it does not restart.

To restart, call the generator function again:

```javascript
const firstRun = values();
const secondRun = values();
```

---

# 7. The `yield` Keyword

The `yield` keyword does two main things:

1. Produces a value for the consumer.
2. Pauses the generator function.

Example:

```javascript
function* numbers() {
  yield 10;
  yield 20;
}
```

First call:

```javascript
const generator = numbers();

console.log(
  generator.next()
);
```

Output:

```javascript
{
  value: 10,
  done: false,
}
```

Execution pauses immediately after producing `10`.

Second call:

```javascript
console.log(
  generator.next()
);
```

Output:

```javascript
{
  value: 20,
  done: false,
}
```

Third call:

```javascript
console.log(
  generator.next()
);
```

Output:

```javascript
{
  value: undefined,
  done: true,
}
```

The generator reached the end of its body.

# 8. Pause and Resume Behavior

Consider:

```javascript
function* workflow() {
  console.log("Step A");

  yield 1;

  console.log("Step B");

  yield 2;

  console.log("Step C");
}
```

Create the generator:

```javascript
const generator = workflow();
```

No output appears.

First `next()`:

```javascript
console.log(
  generator.next()
);
```

Output:

```text
Step A
```

Then:

```javascript
{
  value: 1,
  done: false,
}
```

The generator pauses at the first `yield`.

Second `next()`:

```javascript
console.log(
  generator.next()
);
```

Output:

```text
Step B
```

Then:

```javascript
{
  value: 2,
  done: false,
}
```

Third `next()`:

```javascript
console.log(
  generator.next()
);
```

Output:

```text
Step C
```

Then:

```javascript
{
  value: undefined,
  done: true,
}
```

Execution timeline:

```text
Create generator
    ↓
Suspended before start
    ↓ next()
Run Step A
    ↓
yield 1
    ↓
Suspended
    ↓ next()
Run Step B
    ↓
yield 2
    ↓
Suspended
    ↓ next()
Run Step C
    ↓
Reach end
    ↓
Completed
```

# 9. Generator State

A generator typically moves through these states:

```text
Suspended Start
    ↓
Executing
    ↓
Suspended Yield
    ↓
Executing
    ↓
Completed
```

## Suspended start

The generator object exists, but its body has not started.

## Executing

The generator body is currently running.

## Suspended yield

The generator paused at a `yield` expression.

Its local variables and execution position are preserved.

## Completed

The function reached the end, returned, threw an unhandled error, or was terminated with `return()`.

Once completed, later `next()` calls continue returning:

```javascript
{
  value: undefined,
  done: true,
}
```

---

# 10. `next()` and Generator Results

Each `next()` call returns an iterator result object.

```javascript
{
  value,
  done,
}
```

## Yielded result

```javascript
{
  value: "A",
  done: false,
}
```

## Completed result

```javascript
{
  value: undefined,
  done: true,
}
```

## Returned result

If the generator explicitly returns:

```javascript
function* demo() {
  yield 1;

  return 99;
}
```

Then:

```javascript
const generator = demo();

console.log(generator.next());
console.log(generator.next());
```

Output:

```javascript
{
  value: 1,
  done: false,
}
```

```javascript
{
  value: 99,
  done: true,
}
```

The returned value is attached to the completed result.

However, `for...of` ignores a generator's final returned value.

# 11. How Generators Implement the Iterator Protocol

A generator object already has:

```javascript
next()
```

and:

```javascript
[Symbol.iterator]()
```

Therefore it follows both:

* The iterator protocol.
* The iterable protocol.

Example:

```javascript
function* values() {
  yield "A";
  yield "B";
}
```

The generator object behaves conceptually like a manually written iterator:

```javascript
function createValuesIterator() {
  let state = 0;

  return {
    next() {
      if (state === 0) {
        state = 1;

        return {
          value: "A",
          done: false,
        };
      }

      if (state === 1) {
        state = 2;

        return {
          value: "B",
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },

    [Symbol.iterator]() {
      return this;
    },
  };
}
```

The generator syntax hides this manual state-machine implementation.

# 12. Using Generators with `for...of`

Because generators are iterable, they work naturally with `for...of`.

```javascript
function* colors() {
  yield "red";
  yield "green";
  yield "blue";
}

for (const color of colors()) {
  console.log(color);
}
```

Output:

```text
red
green
blue
```

`for...of` repeatedly calls `next()` until `done` is `true`.

## Spread syntax

```javascript
const colorArray = [
  ...colors(),
];

console.log(colorArray);
```

Output:

```text
["red", "green", "blue"]
```

## Destructuring

```javascript
const [
  first,
  second,
] = colors();

console.log(
  first,
  second
);
```

Output:

```text
red green
```

## `Array.from()`

```javascript
const values =
  Array.from(colors());
```

# 13. Returning from a Generator

A generator may use `return`.

```javascript
function* demo() {
  yield 1;
  return 2;
  yield 3;
}
```

The final `yield` never executes.

Manual consumption:

```javascript
const generator = demo();

console.log(generator.next());
console.log(generator.next());
console.log(generator.next());
```

Output:

```javascript
{
  value: 1,
  done: false,
}
```

```javascript
{
  value: 2,
  done: true,
}
```

```javascript
{
  value: undefined,
  done: true,
}
```

## `for...of` behavior

```javascript
for (const value of demo()) {
  console.log(value);
}
```

Output:

```text
1
```

The returned value `2` is not included because `for...of` consumes only results where:

```javascript
done === false
```

> [!NOTE]
> Use `yield` for sequence values. Use `return` for completion, not for an ordinary value you expect `for...of` to consume.

# 14. Passing Values into a Generator

Generators support two-way communication.

A consumer can pass a value into a paused generator through:

```javascript
generator.next(value);
```

Consider:

```javascript
function* conversation() {
  const name = yield "What is your name?";

  yield `Hello ${name}`;
}
```

Create generator:

```javascript
const generator =
  conversation();
```

First call:

```javascript
console.log(
  generator.next()
);
```

Output:

```javascript
{
  value: "What is your name?",
  done: false,
}
```

Now send a value back:

```javascript
console.log(
  generator.next("Amina")
);
```

Output:

```javascript
{
  value: "Hello Amina",
  done: false,
}
```

## How it works

This expression:

```javascript
const name =
  yield "What is your name?";
```

does two separate things.

First:

```javascript
yield "What is your name?"
```

produces a value and pauses.

Later:

```javascript
generator.next("Amina")
```

causes the paused `yield` expression to evaluate to:

```javascript
"Amina"
```

Then:

```javascript
const name = "Amina";
```

## Important first-call rule

The argument passed to the very first `next()` call is ignored.

Why?

Because the generator has not yet paused at a `yield` expression that can receive it.

```javascript
generator.next("ignored");
```

starts execution but does not assign `"ignored"` anywhere.

# 15. Throwing Errors into a Generator

The consumer can inject an error into a paused generator using:

```javascript
generator.throw(error);
```

Example:

```javascript
function* process() {
  try {
    const value =
      yield "Waiting for value";

    yield `Received ${value}`;
  } catch (error) {
    yield `Handled: ${error.message}`;
  }
}
```

Usage:

```javascript
const generator = process();

console.log(
  generator.next()
);
```

Output:

```javascript
{
  value: "Waiting for value",
  done: false,
}
```

Inject error:

```javascript
console.log(
  generator.throw(
    new Error("Invalid input")
  )
);
```

Output:

```javascript
{
  value: "Handled: Invalid input",
  done: false,
}
```

Conceptually, the paused `yield` behaves as though it threw:

```javascript
throw new Error("Invalid input");
```

inside the generator.

If the generator does not catch the injected error, it escapes to the caller and completes the generator.

---

# 16. Stopping a Generator with `return()`

A consumer can terminate a generator early.

```javascript
generator.return(value);
```

Example:

```javascript
function* numbers() {
  yield 1;
  yield 2;
  yield 3;
}
```

Usage:

```javascript
const generator = numbers();

console.log(generator.next());

console.log(
  generator.return("Stopped")
);

console.log(generator.next());
```

Output:

```javascript
{
  value: 1,
  done: false,
}
```

```javascript
{
  value: "Stopped",
  done: true,
}
```

```javascript
{
  value: undefined,
  done: true,
}
```

The generator does not continue to `yield 2` or `yield 3`.

# 17. Generator Cleanup with `finally`

A generator can release resources in a `finally` block.

```javascript
function* resourceSequence() {
  try {
    yield "Resource value 1";
    yield "Resource value 2";
  } finally {
    console.log("Cleaning up resource");
  }
}
```

Manual termination:

```javascript
const generator =
  resourceSequence();

console.log(generator.next());

console.log(
  generator.return()
);
```

Output:

```text
Cleaning up resource
```

## `for...of` early termination

```javascript
for (
  const value
  of resourceSequence()
) {
  console.log(value);

  break;
}
```

The loop performs iterator closing, which causes generator cleanup.

Possible output:

```text
Resource value 1
Cleaning up resource
```

This pattern matters when a generator controls:

* File handles.
* Database cursors.
* Locks.
* subscriptions.
* stream readers.
* temporary resources.

# 18. Delegating with `yield*`

The `yield*` expression delegates iteration to another iterable.

Without delegation:

```javascript
function* combined() {
  yield 1;
  yield 2;
  yield 3;
  yield 4;
}
```

With delegation:

```javascript
function* firstGroup() {
  yield 1;
  yield 2;
}

function* secondGroup() {
  yield 3;
  yield 4;
}

function* combined() {
  yield* firstGroup();
  yield* secondGroup();
}
```

Usage:

```javascript
console.log([
  ...combined(),
]);
```

Output:

```text
[1, 2, 3, 4]
```

## Delegate to arrays

```javascript
function* values() {
  yield* [
    "A",
    "B",
    "C",
  ];
}
```

## Delegate to strings

```javascript
function* characters() {
  yield* "Hi";
}
```

## Return value from delegated generator

`yield*` can receive the delegated generator's return value.

```javascript
function* inner() {
  yield 1;
  return 99;
}

function* outer() {
  const result =
    yield* inner();

  yield result;
}
```

Output:

```javascript
console.log([
  ...outer(),
]);
```

```text
[1, 99]
```

The delegated generator's yielded values flow outward, while its returned value becomes the value of the `yield*` expression.

# 19. Generators and Lazy Evaluation

Generators calculate values only when requested.

Example:

```javascript
function* squares(limit) {
  for (
    let number = 1;
    number <= limit;
    number += 1
  ) {
    console.log(
      `Calculating square for ${number}`
    );

    yield number * number;
  }
}
```

Create generator:

```javascript
const generator =
  squares(1_000_000);
```

No squares are calculated yet.

First request:

```javascript
console.log(
  generator.next().value
);
```

Only the first square is calculated.

This is lazy evaluation.

## Benefits

* Lower memory use.
* Reduced unnecessary work.
* Support for huge sequences.
* Support for infinite sequences.
* Efficient early termination.
* Composable pipelines.

---

# 20. Infinite Generator Sequences

Generators are ideal for infinite sequences.

## Infinite counter

```javascript
function* counter(
  start = 0,
  step = 1
) {
  let current = start;

  while (true) {
    yield current;

    current += step;
  }
}
```

Consume first five values:

```javascript
const numbers =
  counter(10, 2);

for (
  let count = 0;
  count < 5;
  count += 1
) {
  console.log(
    numbers.next().value
  );
}
```

Output:

```text
10
12
14
16
18
```

## Fibonacci generator

```javascript
function* fibonacci() {
  let previous = 0;
  let current = 1;

  while (true) {
    yield previous;

    [
      previous,
      current,
    ] = [
      current,
      previous + current,
    ];
  }
}
```

Consume first ten:

```javascript
const sequence =
  fibonacci();

for (
  let count = 0;
  count < 10;
  count += 1
) {
  console.log(
    sequence.next().value
  );
}
```

> [!WARNING]
> Never spread an infinite generator into an array.

Unsafe:

```javascript
const allNumbers = [
  ...counter(),
];
```

# 21. Generator Pipelines

Generators can transform values lazily.

## Lazy map

```javascript
function* mapIterable(
  iterable,
  transform
) {
  for (const value of iterable) {
    yield transform(value);
  }
}
```

## Lazy filter

```javascript
function* filterIterable(
  iterable,
  predicate
) {
  for (const value of iterable) {
    if (predicate(value)) {
      yield value;
    }
  }
}
```

## Take a limited number

```javascript
function* take(
  iterable,
  limit
) {
  if (limit <= 0) {
    return;
  }

  let count = 0;

  for (const value of iterable) {
    yield value;

    count += 1;

    if (count >= limit) {
      return;
    }
  }
}
```

## Compose pipeline

```javascript
const numbers =
  counter(1);

const evenNumbers =
  filterIterable(
    numbers,
    number => number % 2 === 0
  );

const squares =
  mapIterable(
    evenNumbers,
    number => number * number
  );

const firstFiveSquares =
  take(
    squares,
    5
  );

console.log([
  ...firstFiveSquares,
]);
```

Output:

```text
[4, 16, 36, 64, 100]
```

Only the values required to produce the first five results are calculated.

# 22. Generator Expressions and Methods

## Generator expression

```javascript
const createValues =
  function* () {
    yield 1;
    yield 2;
  };
```

## Object generator method

```javascript
const transactionCollection = {
  transactions: [
    {
      id: 1,
    },
    {
      id: 2,
    },
  ],

  *[Symbol.iterator]() {
    for (
      const transaction
      of this.transactions
    ) {
      yield transaction;
    }
  },
};
```

Now the object itself is iterable:

```javascript
for (
  const transaction
  of transactionCollection
) {
  console.log(transaction);
}
```

## Class generator method

```javascript
class Range {
  constructor(
    start,
    end,
    step = 1
  ) {
    this.start = start;
    this.end = end;
    this.step = step;
  }

  *[Symbol.iterator]() {
    for (
      let current = this.start;
      current <= this.end;
      current += this.step
    ) {
      yield current;
    }
  }
}
```

Usage:

```javascript
console.log([
  ...new Range(1, 5),
]);
```

# 23. Generators vs Ordinary Functions

| Ordinary Function                  | Generator Function                            |
| ---------------------------------- | --------------------------------------------- |
| Declared with `function`           | Declared with `function*`                     |
| Runs immediately when called       | Returns generator object without running body |
| Can return once                    | Can yield many times                          |
| Cannot pause normally              | Pauses at `yield`                             |
| State is discarded after return    | State is preserved between yields             |
| Returns ordinary value             | Returns generator object                      |
| Cannot be resumed after completion | Resumed with `next()` while suspended         |

Example ordinary function:

```javascript
function values() {
  return [
    1,
    2,
    3,
  ];
}
```

Example generator:

```javascript
function* values() {
  yield 1;
  yield 2;
  yield 3;
}
```

The ordinary function creates the complete array eagerly.

The generator produces values lazily.

# 24. Generators vs Manual Iterators

## Manual iterator

```javascript
function createRange(
  start,
  end
) {
  let current = start;

  return {
    next() {
      if (current <= end) {
        return {
          value: current++,
          done: false,
        };
      }

      return {
        done: true,
      };
    },

    [Symbol.iterator]() {
      return this;
    },
  };
}
```

## Generator version

```javascript
function* createRange(
  start,
  end
) {
  for (
    let current = start;
    current <= end;
    current += 1
  ) {
    yield current;
  }
}
```

The generator version:

* Is shorter.
* Is easier to read.
* Makes control flow visible.
* Automatically creates iterator result objects.
* Automatically preserves state.
* Supports cleanup with `try...finally`.
* Supports two-way communication.

Manual iterators remain useful when you need precise control over object shape, state sharing, or unusual iterator behavior.

# 25. Three Progressive Examples

## Example 1: Beginner — Numeric Range Generator

### Goal

Generate numbers in a range lazily.

```javascript
function* range(
  start,
  end,
  step = 1
) {
  if (
    !Number.isFinite(start) ||
    !Number.isFinite(end) ||
    !Number.isFinite(step)
  ) {
    throw new TypeError(
      "start, end, and step must be finite numbers."
    );
  }

  if (step === 0) {
    throw new RangeError(
      "step cannot be zero."
    );
  }

  const movingForward =
    step > 0;

  for (
    let current = start;
    movingForward
      ? current <= end
      : current >= end;
    current += step
  ) {
    yield current;
  }
}
```

Usage:

```javascript
console.log([
  ...range(1, 5),
]);
```

Output:

```text
[1, 2, 3, 4, 5]
```

Descending:

```javascript
console.log([
  ...range(10, 2, -2),
]);
```

Output:

```text
[10, 8, 6, 4, 2]
```

### Why professionals write it this way

* Invalid input fails early.
* One implementation supports ascending and descending ranges.
* Values are lazy.
* The code is significantly clearer than a manual iterator.
* It integrates with all iterable consumers.

## Example 2: Intermediate — Lazy Transaction Pipeline

### Goal

Process transactions without creating multiple intermediate arrays.

Data:

```javascript
const transactions = [
  {
    id: 1,
    amount: 500,
    isActive: true,
    isPaid: false,
  },
  {
    id: 2,
    amount: 800,
    isActive: true,
    isPaid: true,
  },
  {
    id: 3,
    amount: 250,
    isActive: false,
    isPaid: false,
  },
  {
    id: 4,
    amount: 900,
    isActive: true,
    isPaid: false,
  },
];
```

Filter generator:

```javascript
function* filterIterable(
  iterable,
  predicate
) {
  for (const value of iterable) {
    if (predicate(value)) {
      yield value;
    }
  }
}
```

Map generator:

```javascript
function* mapIterable(
  iterable,
  transform
) {
  for (const value of iterable) {
    yield transform(value);
  }
}
```

Build pipeline:

```javascript
const pendingTransactions =
  filterIterable(
    transactions,
    transaction =>
      transaction.isActive &&
      !transaction.isPaid
  );

const receiptSummaries =
  mapIterable(
    pendingTransactions,
    transaction => ({
      id: transaction.id,
      displayAmount:
        `KES ${transaction.amount}`,
    })
  );
```

Consume:

```javascript
console.log([
  ...receiptSummaries,
]);
```

Output:

```javascript
[
  {
    id: 1,
    displayAmount: "KES 500",
  },
  {
    id: 4,
    displayAmount: "KES 900",
  },
]
```

### Internal flow

```text
Consumer requests next result
    ↓
map requests next filtered value
    ↓
filter requests next transaction
    ↓
Skip or yield transaction
    ↓
map transforms it
    ↓
Consumer receives one result
```

No complete filtered array is created.

## Example 3: Real-World Application — Tree Traversal Generator

### Goal

Traverse a nested service-category tree.

Data:

```javascript
const serviceTree = {
  id: "root",
  name: "Services",
  children: [
    {
      id: "printing",
      name: "Printing",
      children: [
        {
          id: "black-white",
          name: "Black and White",
          children: [],
        },
        {
          id: "color",
          name: "Color Printing",
          children: [],
        },
      ],
    },
    {
      id: "finishing",
      name: "Finishing",
      children: [
        {
          id: "binding",
          name: "Binding",
          children: [],
        },
        {
          id: "lamination",
          name: "Lamination",
          children: [],
        },
      ],
    },
  ],
};
```

Depth-first traversal:

```javascript
function* walkDepthFirst(node) {
  yield node;

  for (const child of node.children) {
    yield* walkDepthFirst(child);
  }
}
```

Usage:

```javascript
for (
  const node
  of walkDepthFirst(serviceTree)
) {
  console.log(
    node.id,
    node.name
  );
}
```

```text

printing Printing
black-white Black and White
color Color Printing
finishing Finishing
binding Binding
lamination Lamination
```

### Why `yield*` is ideal

Each recursive call returns another generator.

```javascript
yield* walkDepthFirst(child);
```

delegates all child values to the outer consumer.

### Early stopping

```javascript
function findServiceById(
  root,
  targetId
) {
  for (
    const node
    of walkDepthFirst(root)
  ) {
    if (node.id === targetId) {
      return node;
    }
  }

  return null;
}
```

Only nodes required to find the target are traversed.

# 26. Common Mistakes

## Mistake 1: Expecting the body to run immediately

```javascript
function* demo() {
  console.log("Started");
}

demo();
```

No output appears because the returned generator was not advanced.

Correct:

```javascript
const generator = demo();

generator.next();
```

---

## Mistake 2: Forgetting the asterisk

Incorrect ordinary function:

```javascript
function values() {
  yield 1;
}
```

`yield` is not valid inside an ordinary function.

Correct:

```javascript
function* values() {
  yield 1;
}
```

---

## Mistake 3: Using `return` for values expected by `for...of`

```javascript
function* values() {
  yield 1;
  return 2;
}
```

`for...of` produces only `1`.

Use:

```javascript
function* values() {
  yield 1;
  yield 2;
}
```

---

## Mistake 4: Reusing an exhausted generator

```javascript
const generator = values();

console.log([
  ...generator,
]);

console.log([
  ...generator,
]);
```

The second array is empty.

Call the generator function again for a fresh sequence.

---

## Mistake 5: Passing a value to the first `next()` call

```javascript
generator.next("Amina");
```

The value is ignored if the generator has not yet paused at a `yield`.

---

## Mistake 6: Spreading an infinite generator

```javascript
const values = [
  ...counter(),
];
```

This never completes.

Use a `take()` helper or a bounded loop.

---

## Mistake 7: Forgetting cleanup

A generator controlling a resource should use `try...finally`.

---

## Mistake 8: Assuming generators are asynchronous

Generator functions are synchronous unless they are specifically async generators.

A normal generator's `next()` returns an iterator result immediately, not a Promise.

# 27. Best Practices

## 27.1 Use generators for lazy sequences

Generators are especially useful when values are expensive, numerous, or potentially infinite.

## 27.2 Prefer clear `yield` points

Each `yield` should represent a meaningful stage or produced value.

## 27.3 Use `yield*` for delegation

Avoid manually looping only to re-yield every value when delegation is clearer.

## 27.4 Create fresh generator objects for repeated iteration

Store the generator function or iterable factory, not one exhausted generator.

## 27.5 Use `try...finally` for resource cleanup

This ensures cleanup during normal completion and early termination.

## 27.6 Avoid generators for simple arrays unless laziness helps

A plain array or ordinary function may be clearer for small, fixed data.

## 27.7 Document two-way communication

Code using `next(value)`, `throw()`, or `return()` is powerful but less obvious. Explain its protocol.

# 28. Performance Considerations

## Memory

Generators can reduce memory usage by yielding values one at a time.

Eager:

```javascript
const millionNumbers =
  Array.from(
    {
      length: 1_000_000,
    },
    (_, index) => index
  );
```

Lazy:

```javascript
function* millionNumbers() {
  for (
    let index = 0;
    index < 1_000_000;
    index += 1
  ) {
    yield index;
  }
}
```

The lazy version does not allocate one million array elements immediately.

## Computation

Generators avoid computing values that are never requested.

## Overhead

Each generator step involves:

* Resuming execution.
* Creating or returning an iterator result.
* Suspending again.

For tiny fixed collections, a direct loop may be faster and simpler.

## Early termination

Generators can save substantial work when consumers stop early.

## Infinite sequences

Generators make them possible without infinite memory allocation.

# 30. Interview Questions and Answers

## Beginner Questions

### 1. What is a generator function?

A special function declared with `function*` that can pause at `yield` and resume later.

### 2. What does calling a generator function return?

A generator object.

### 3. Does the generator body run immediately when called?

No. It begins when `next()` is called.

### 4. What does `yield` do?

It produces a value and pauses generator execution.

### 5. Is a generator object iterable?

Yes. It implements both the iterator and iterable protocols.

---

## Intermediate Questions

### 1. What is the difference between `yield` and `return`?

`yield` produces a value and pauses while keeping the generator resumable. `return` completes the generator.

### 2. What does `generator.next(value)` do?

It resumes the generator and makes the paused `yield` expression evaluate to the supplied value.

### 3. Why is the first `next(value)` argument ignored?

Because the generator has not yet paused at a `yield` expression that can receive it.

### 4. What does `yield*` do?

It delegates iteration to another iterable.

### 5. Can a completed generator restart?

No. Call the generator function again to obtain a fresh generator object.

### 6. Are generators lazy?

Yes. They produce values only when requested.

---

## Advanced Questions

### 1. How does JavaScript implement generator pause and resume behavior conceptually?

The engine preserves the generator's instruction position, local variables, and execution state in a resumable generator context.

### 2. What happens when `generator.throw(error)` is called?

The error is injected at the generator's current suspension point. It may be caught inside the generator or escape to the caller.

### 3. What happens when `generator.return(value)` is called?

The generator begins completion with `done: true`, while executing relevant `finally` cleanup blocks.

### 4. Why does `for...of` ignore a generator's returned value?

Because `for...of` consumes only iterator results where `done` is `false`.

### 5. What is the difference between a generator function and an async generator function?

A generator yields synchronous values and its `next()` returns iterator results immediately. An async generator may await asynchronous operations, and its `next()` returns a Promise for an iterator result.

### 6. When are generators better than arrays?

When sequences are large, expensive, infinite, lazily transformed, or likely to be stopped early.

# 34. Summary

Generator functions provide a simpler way to create iterators.

They are declared with:

```javascript
function*
```

and produce values with:

```javascript
yield
```

Calling a generator function:

```javascript
const generator =
  values();
```

does not immediately run the body.

Instead, it creates a generator object in a suspended state.

The consumer resumes execution using:

```javascript
generator.next();
```

You learned that generators:

* Pause at `yield`.
* Preserve local state.
* Resume after the previous `yield`.
* Implement the iterator protocol automatically.
* Work with `for...of`, spread, destructuring, and `Array.from()`.
* Can receive values through `next(value)`.
* Can receive injected errors through `throw(error)`.
* Can be terminated using `return(value)`.
* Can clean up using `try...finally`.
* Can delegate with `yield*`.
* Support lazy and infinite sequences.
* Simplify recursive traversal and transformation pipelines.

The essential mental model is:

```text
Call generator function
    ↓
Receive generator object
    ↓
next()
    ↓
Run until yield
    ↓
Produce value and pause
    ↓
next()
    ↓
Resume after yield
```

Generators are the bridge between synchronous iterators and asynchronous generators.
