# 1. Introduction

In the previous lesson, you learned how to process arrays and other existing collections asynchronously.

You used constructs such as:

```javascript
for (const item of items) {
  await processItem(item);
}
```

You also learned patterns such as:

```javascript
await Promise.all(
  items.map(processItem)
);
```

However, one major question remains:

> How does JavaScript know how to retrieve values from an array when `for...of` runs?

Consider:

```javascript
const numbers = [10, 20, 30];

for (const number of numbers) {
  console.log(number);
}
```

The loop somehow knows:

1. How to obtain the first value.
2. How to obtain the next value.
3. When there are no more values.
4. How to stop.
5. How to work with arrays, strings, sets, maps, and custom objects using similar syntax.

This is possible because JavaScript defines formal **iteration protocols**.

These protocols allow different data structures to answer a common question:

> “How should consumers retrieve your values one at a time?”

Understanding synchronous iterators is necessary before learning:

* Generator functions.
* Async iterators.
* Async generators.
* `for await...of`.
* Streams.
* Paginated APIs.
* Incremental network responses.
* Database cursors.
* Large data pipelines.

# 2. Why Iteration Exists

Programs frequently work with groups of values.

Examples include:

```javascript
const names = [
  "Amina",
  "Brian",
  "Caroline",
];
```

```javascript
const serviceNames = new Set([
  "Printing",
  "Binding",
  "Scanning",
]);
```

```javascript
const prices = new Map([
  ["Printing", 20],
  ["Binding", 100],
]);
```

```javascript
const message = "Hello";
```

Each structure stores multiple values.

A program needs a consistent way to retrieve those values.

Without a shared iteration system, every data structure might require completely different code.

```javascript
readArrayItem();
readSetEntry();
readMapEntry();
readStringCharacter();
readCustomCollectionRecord();
```

That would make the language harder to learn and APIs harder to compose.

JavaScript instead defines standard protocols.

```text
Collection
    ↓
Provides iterator
    ↓
Iterator returns values
    ↓
Consumer reads values
```

The collection controls **how values are produced**.

The consumer controls **what to do with them**.

# 3. The Problem Before Iteration Protocols

Imagine a custom collection:

```javascript
const transactionCollection = {
  firstTransaction: {
    id: 1,
    amount: 500,
  },
  secondTransaction: {
    id: 2,
    amount: 800,
  },
  thirdTransaction: {
    id: 3,
    amount: 250,
  },
};
```

How should a loop retrieve its transaction values?

A normal object does not automatically describe:

* Which properties represent collection values.
* In what order they should be returned.
* Whether metadata should be skipped.
* When iteration is finished.

The programmer could use:

```javascript
Object.values(transactionCollection);
```

but that creates an array and assumes all enumerable values belong to the collection.

Suppose the object also contains:

```javascript
{
  collectionName: "Pending Transactions",
  totalCount: 3,
  firstTransaction: {},
  secondTransaction: {},
  thirdTransaction: {},
}
```

Now not every value is a transaction.

A custom iterator allows the object to define its own sequence precisely.

```text
collectionName → ignore
totalCount → ignore
transactions → return one at a time
```

# 4. What Is Iteration?

Iteration means retrieving values from a sequence one at a time.

```text
Sequence

Value 1
Value 2
Value 3
Value 4
```

An iteration process typically asks:

```text
Is another value available?
    ↓
Yes
    ↓
Return the value
    ↓
Repeat
```

Eventually:

```text
Is another value available?
    ↓
No
    ↓
Stop
```

In JavaScript, an iterator communicates these two pieces of information using an object:

```javascript
{
  value: someValue,
  done: false,
}
```

When iteration finishes:

```javascript
{
  value: undefined,
  done: true,
}
```

---

# 5. What Is an Iterable?

An **iterable** is an object that can provide an iterator.

More precisely:

> An iterable is an object that implements the iterable protocol by defining a method at `Symbol.iterator`.

Conceptual shape:

```javascript
const iterable = {
  [Symbol.iterator]() {
    return iterator;
  },
};
```

The iterable is usually the collection.

Examples include:

* Arrays.
* Strings.
* Sets.
* Maps.
* Typed arrays.
* Node lists in modern browsers.
* Custom iterable objects.

An iterable answers:

> “How can someone begin iterating over me?”

---

# 6. What Is an Iterator?

An **iterator** is an object that produces one result at a time.

It must provide a `next()` method.

Conceptual shape:

```javascript
const iterator = {
  next() {
    return {
      value: someValue,
      done: false,
    };
  },
};
```

After all values have been returned:

```javascript
iterator.next();
```

returns:

```javascript
{
  value: undefined,
  done: true,
}
```

The iterator answers:

> “What is the next value, and has iteration finished?”

---

# 7. Iterable vs Iterator

These two terms are related but not identical.

| Iterable                      | Iterator                             |
| ----------------------------- | ------------------------------------ |
| Represents a source of values | Produces values one at a time        |
| Has `[Symbol.iterator]()`     | Has `next()`                         |
| Starts iteration              | Advances iteration                   |
| Usually reusable              | Often stateful and single-use        |
| Example: an array             | Example: the array's iterator object |

Visual model:

```text
Array
  ↓ [Symbol.iterator]()
Iterator
  ↓ next()
10
  ↓ next()
20
  ↓ next()
30
  ↓ next()
done
```

Example:

```javascript
const numbers = [10, 20, 30];

const iterator =
  numbers[Symbol.iterator]();
```

Here:

```javascript
numbers
```

is the iterable.

```javascript
iterator
```

is the iterator.

# 8. The Iterator Protocol

An object follows the iterator protocol if it provides a `next()` method that returns an iterator result object.

```javascript
const iterator = {
  next() {
    return {
      value: "A",
      done: false,
    };
  },
};
```

The protocol requires:

1. A callable `next()` method.
2. `next()` returns an object.
3. The returned object normally has:

   * `value`
   * `done`

## `done`

`done` indicates whether iteration is complete.

```javascript
done: false
```

means:

> A value is available.

```javascript
done: true
```

means:

> Iteration is complete.

## `value`

`value` contains the item produced during that step.

```javascript
{
  value: "Printing",
  done: false,
}
```

When `done` is `true`, `value` is usually `undefined`, although a final value may technically be provided in some iterator designs.

---

# 9. The Iterable Protocol

An object follows the iterable protocol if it defines:

```javascript
object[Symbol.iterator]
```

as a method.

That method must return an iterator.

```javascript
const collection = {
  [Symbol.iterator]() {
    return {
      next() {
        return {
          value: undefined,
          done: true,
        };
      },
    };
  },
};
```

Now `collection` is iterable.

```javascript
for (const value of collection) {
  console.log(value);
}
```

The loop obtains the iterator automatically.

# 10. `Symbol.iterator`

## What is a symbol?

A Symbol is a unique primitive value frequently used as a special object-property key.

```javascript
const identifier = Symbol("identifier");
```

Even if two symbols have the same description, they are different:

```javascript
Symbol("id") === Symbol("id");
```

Output:

```text
false
```

## What is `Symbol.iterator`?

`Symbol.iterator` is a built-in well-known Symbol.

It is JavaScript's standard property key for requesting a synchronous iterator.

```javascript
const iteratorMethod =
  collection[Symbol.iterator];
```

Calling it:

```javascript
const iterator =
  collection[Symbol.iterator]();
```

returns the iterator object.

## Why use a Symbol?

If JavaScript had used a normal string property such as:

```javascript
collection.iterator
```

application code might accidentally overwrite it.

Symbols reduce naming collisions.

```javascript
collection[Symbol.iterator]
```

is a standardized internal-style hook with a unique key.

# 11. How `for...of` Works Internally

Consider:

```javascript
const numbers = [10, 20, 30];

for (const number of numbers) {
  console.log(number);
}
```

Conceptually, JavaScript performs something similar to this:

```javascript
const iterator =
  numbers[Symbol.iterator]();

while (true) {
  const result = iterator.next();

  if (result.done) {
    break;
  }

  const number = result.value;

  console.log(number);
}
```

Step-by-step:

```text
Request iterator
    ↓
Call next()
    ↓
Receive { value: 10, done: false }
    ↓
Run loop body
    ↓
Call next()
    ↓
Receive { value: 20, done: false }
    ↓
Run loop body
    ↓
Call next()
    ↓
Receive { value: 30, done: false }
    ↓
Run loop body
    ↓
Call next()
    ↓
Receive { value: undefined, done: true }
    ↓
Stop
```

The real language algorithm includes additional details such as abrupt-completion cleanup, but this is the correct foundational mental model.

# 13. The `next()` Method

The `next()` method advances the iterator.

```javascript
iterator.next();
```

Each call usually:

1. Checks the current internal position.
2. Produces the current value.
3. Advances the internal position.
4. Reports whether iteration has finished.

Conceptual implementation:

```javascript
function createIterator(values) {
  let index = 0;

  return {
    next() {
      if (index < values.length) {
        const value = values[index];

        index += 1;

        return {
          value,
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },
  };
}
```

The `index` variable is preserved through a closure.

# 14. The `{ value, done }` Result Object

An iterator result object contains two important properties.

```javascript
{
  value,
  done,
}
```

## Active result

```javascript
{
  value: 500,
  done: false,
}
```

Interpretation:

> Here is the next value. Iteration is not finished.

## Completed result

```javascript
{
  value: undefined,
  done: true,
}
```

Interpretation:

> There are no more values.

## Why an object?

An object can communicate both:

* The produced value.
* The completion status.

Returning only `undefined` would be ambiguous because `undefined` might itself be a legitimate sequence value.

Consider:

```javascript
const values = [
  10,
  undefined,
  30,
];
```

The iterator must distinguish:

```text
Value is undefined, but iteration continues
```

from:

```text
No more values
```

The `done` property solves this ambiguity.

# 16. Values That Are Not Iterable

Plain objects are not iterable by default.

```javascript
const user = {
  name: "Amina",
  role: "Admin",
};

for (const value of user) {
  console.log(value);
}
```

This produces a `TypeError`.

Why?

Because the object does not define:

```javascript
user[Symbol.iterator]
```

## Use object utilities

```javascript
for (const key of Object.keys(user)) {
  console.log(key);
}
```

```javascript
for (const value of Object.values(user)) {
  console.log(value);
}
```

```javascript
for (const [key, value] of Object.entries(user)) {
  console.log(key, value);
}
```

These utility methods return arrays, and arrays are iterable.

## Or define a custom iterator

You can make an object iterable by implementing `[Symbol.iterator]()`.

That will be demonstrated shortly.

# 17. Creating a Custom Iterator

Let's build an iterator manually.

## Goal

Produce the numbers:

```text
1
2
3
```

## Implementation

```javascript
function createNumberIterator() {
  let current = 1;

  return {
    next() {
      if (current <= 3) {
        const value = current;

        current += 1;

        return {
          value,
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },
  };
}
```

Usage:

```javascript
const iterator =
  createNumberIterator();

console.log(iterator.next());
console.log(iterator.next());
console.log(iterator.next());
console.log(iterator.next());
```

Output:

```javascript
{
  value: 1,
  done: false,
}
```

```javascript
{
  value: 2,
  done: false,
}
```

```javascript
{
  value: 3,
  done: false,
}
```

```javascript
{
  value: undefined,
  done: true,
}
```

## Internal state

The iterator remembers:

```javascript
let current = 1;
```

After each call:

```text
next() → current becomes 2
next() → current becomes 3
next() → current becomes 4
next() → done
```

# 18. Creating a Custom Iterable

The iterator above cannot yet be used directly with `for...of` unless it is also iterable.

Let's build a range object.

```javascript
function createRange(start, end) {
  return {
    start,
    end,

    [Symbol.iterator]() {
      let current = start;

      return {
        next() {
          if (current <= end) {
            const value = current;

            current += 1;

            return {
              value,
              done: false,
            };
          }

          return {
            value: undefined,
            done: true,
          };
        },
      };
    },
  };
}
```

Usage:

```javascript
const range =
  createRange(3, 7);

for (const number of range) {
  console.log(number);
}
```

Output:

```text
3
4
5
6
7
```

## Why it works

`for...of` calls:

```javascript
range[Symbol.iterator]()
```

This returns a new iterator.

The loop repeatedly calls:

```javascript
iterator.next()
```

until:

```javascript
done === true
```

# 19. Reusable vs Single-Use Iterators

A well-designed iterable often returns a new iterator every time.

```javascript
const range =
  createRange(1, 3);

console.log([
  ...range,
]);

console.log([
  ...range,
]);
```

Output:

```text
[1, 2, 3]
[1, 2, 3]
```

Why?

Each call to:

```javascript
range[Symbol.iterator]()
```

creates a fresh `current` variable.

## Single-use iterator

An iterator object may store its position directly and return itself.

```javascript
const iterator = {
  current: 1,

  next() {
    if (this.current <= 3) {
      return {
        value: this.current++,
        done: false,
      };
    }

    return {
      value: undefined,
      done: true,
    };
  },

  [Symbol.iterator]() {
    return this;
  },
};
```

Usage:

```javascript
console.log([
  ...iterator,
]);
```

Output:

```text
[1, 2, 3]
```

Use it again:

```javascript
console.log([
  ...iterator,
]);
```

Output:

```text
[]
```

The iterator was already exhausted.

## Design distinction

```text
Iterable collection
    ↓ creates fresh iterator
Reusable

Iterator that returns itself
    ↓ preserves one cursor
Usually single-use
```

# 20. Iterator State

Iterators are stateful.

They remember where iteration stopped.

```javascript
const iterator =
  [10, 20, 30][Symbol.iterator]();

console.log(iterator.next());
```

Result:

```javascript
{
  value: 10,
  done: false,
}
```

Later:

```javascript
console.log(iterator.next());
```

Result:

```javascript
{
  value: 20,
  done: false,
}
```

The iterator did not restart.

It remembered that `10` had already been consumed.

This state may include:

* Current index.
* Current numeric value.
* Current node.
* Current page.
* Current cursor.
* Whether iteration is complete.
* Cleanup information.

# 21. Lazy Evaluation

Iterators often produce values lazily.

**Lazy evaluation** means:

> Compute a value only when the consumer asks for it.

Consider an eager array:

```javascript
const values = [
  1,
  2,
  3,
  4,
  5,
];
```

Every value already exists in memory.

A lazy iterator can calculate one value at a time.

```javascript
function createSquares(limit) {
  let current = 1;

  return {
    next() {
      if (current <= limit) {
        const value =
          current * current;

        current += 1;

        return {
          value,
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },
  };
}
```

No array of squares is created.

Each square is calculated only when `next()` is called.

## Benefits

Lazy iteration can:

* Reduce memory use.
* Avoid unnecessary calculations.
* Support very large sequences.
* Support infinite sequences.
* Stop early without computing remaining values.
* Process stream-like data.

# 22. Infinite Iterators

An iterator does not need to finish.

```javascript
function createCounter(start = 0) {
  let current = start;

  return {
    next() {
      const value = current;

      current += 1;

      return {
        value,
        done: false,
      };
    },
  };
}
```

This iterator produces:

```text
0
1
2
3
4
...
```

forever.

## Safe manual consumption

```javascript
const counter =
  createCounter(10);

console.log(iterator.next());
```

To avoid an infinite loop, consume a limited number of values:

```javascript
for (let count = 0; count < 5; count += 1) {
  console.log(
    counter.next().value
  );
}
```

Output:

```text
10
11
12
13
14
```

> [!WARNING]
> Never spread an infinite iterable into an array. The program will continue requesting values until it runs out of memory or is interrupted.

Unsafe:

```javascript
const values = [
  ...infiniteIterable,
];
```

# 23. Iterator Cleanup

Iterators may optionally provide a `return()` method.

This method can be used when iteration stops early.

Examples include:

* A `break` statement.
* A thrown error.
* A `return` from the surrounding function.

Conceptual iterator:

```javascript
const iterable = {
  [Symbol.iterator]() {
    let current = 1;

    return {
      next() {
        return {
          value: current++,
          done: false,
        };
      },

      return() {
        console.log("Iterator cleanup");

        return {
          value: undefined,
          done: true,
        };
      },
    };
  },
};
```

Usage:

```javascript
for (const value of iterable) {
  console.log(value);

  if (value === 3) {
    break;
  }
}
```

Possible output:

```text
1
2
3
Iterator cleanup
```

## Why cleanup matters

An iterator may control:

* A file handle.
* A database cursor.
* A network stream.
* A subscription.
* A lock.
* A temporary resource.

When the consumer stops early, the iterator should be able to release resources.

This becomes especially important with async iterators and streams.

Spread syntax consumes an iterable.

```javascript

  ...iterable,
```

Conceptually:

```text
Request iterator
    ↓
Call next repeatedly
    ↓
Collect values
    ↓
Create array
```

```javascript
const range =
  createRange(1, 5);

  ...range,
];

console.log(numbers);
```

Output:

```text
[1, 2, 3, 4, 5]
```

Spread can also pass iterable values as arguments.

```javascript

  5,
  10,
  15,
];

  Math.max(...values)
);
```

The array iterator provides the values.

# 25. Destructuring and Iteration

Array destructuring also uses iteration.

```javascript
const values = [
  10,
  20,
  30,
];

const [
  first,
  second,
] = values;
```

Conceptually:

```text
Request iterator
    ↓
next() → 10 → first
next() → 20 → second
Stop
```

This means custom iterables can also be destructured.

```javascript
const range =
  createRange(5, 10);

const [
  first,
  second,
  third,
] = range;

console.log(
  first,
  second,
  third
);
```

Output:

```text
5 6 7
```

The remaining values are not requested.

This is a benefit of lazy evaluation.

# 28. JavaScript Engine Mental Model

Consider:

```javascript
for (const service of services) {
  console.log(service);
}
```

A useful conceptual execution model is:

```text
1. Evaluate services
2. Read services[Symbol.iterator]
3. Verify it is callable
4. Call it to obtain an iterator
5. Repeatedly call iterator.next()
6. Verify each returned result is an object
7. Read result.done
8. If done is true, finish
9. Read result.value
10. Assign value to service
11. Execute loop body
12. Repeat
13. If loop exits early, perform iterator cleanup when applicable
```

ASCII diagram:

```text
Iterable
   ↓
[Symbol.iterator]()
   ↓
Iterator object
   ↓
next()
   ↓
{ value, done }
   ↓
done === false?
   ├── Yes → use value → next()
   └── No  → finish
```

## Memory model

The iterable may store the collection:

```text
Range object
├── start: 1
└── end: 5
```

The iterator stores traversal state:

```text
Iterator
└── current: 3
```

Keeping collection data and cursor state separate allows multiple independent iterators.

```javascript
const range =
  createRange(1, 3);

const iteratorA =
  range[Symbol.iterator]();

const iteratorB =
  range[Symbol.iterator]();
```

Both begin independently at `1`.

## Example 2: Intermediate — Reusable Range Iterable

### Goal

Create a reusable numeric range.

```javascript
function createRange(
  start,
  end,
  step = 1
) {
  if (
    !Number.isFinite(start) ||
    !Number.isFinite(end) ||
    !Number.isFinite(step)
  ) {
    throw new TypeError(
      "start, end, and step must be finite numbers."
    );
  }

  if (step === 0) {
    throw new RangeError(
      "step cannot be zero."
    );
  }

  const movesForward =
    step > 0;

  return {
    start,
    end,
    step,

    [Symbol.iterator]() {
      let current = start;

      return {
        next() {
          const isWithinRange =
            movesForward
              ? current <= end
              : current >= end;

          if (!isWithinRange) {
            return {
              value: undefined,
              done: true,
            };
          }

          const value = current;

          current += step;

          return {
            value,
            done: false,
          };
        },
      };
    },
  };
}
```

Usage:

```javascript
console.log([
  ...createRange(1, 5),
]);
```

Output:

```text
[1, 2, 3, 4, 5]
```

Descending range:

```javascript
console.log([
  ...createRange(5, 1, -2),
]);
```

Output:

```text
[5, 3, 1]
```

Destructuring:

```javascript
const [
  first,
  second,
] = createRange(10, 20, 5);

console.log(
  first,
  second
);
```

Output:

```text
10 15
```

### Professional design points

* Inputs are validated.
* Zero step is rejected.
* Both ascending and descending ranges are supported.
* Each iteration creates a fresh iterator.
* Values are generated lazily.
* No full range array is created unless the consumer requests one.

## Example 3: Real-World Application — Transaction Collection

### Goal

Create a collection that iterates only over active pending transactions.

```javascript
function createPendingTransactionCollection(
  transactions
) {
  if (!Array.isArray(transactions)) {
    throw new TypeError(
      "transactions must be an array."
    );
  }

  return {
    name: "Pending Transactions",
    source: transactions,

    get totalSourceRecords() {
      return transactions.length;
    },

    [Symbol.iterator]() {
      let index = 0;

      return {
        next() {
          while (index < transactions.length) {
            const transaction =
              transactions[index];

            index += 1;

            const shouldInclude =
              transaction.isActive === true &&
              transaction.isPaid === false &&
              transaction.isDeleted === false;

            if (shouldInclude) {
              return {
                value: transaction,
                done: false,
              };
            }
          }

          return {
            value: undefined,
            done: true,
          };
        },
      };
    },
  };
}
```

Data:

```javascript
const transactions = [
  {
    id: 1,
    amount: 500,
    isActive: true,
    isPaid: false,
    isDeleted: false,
  },
  {
    id: 2,
    amount: 300,
    isActive: true,
    isPaid: true,
    isDeleted: false,
  },
  {
    id: 3,
    amount: 800,
    isActive: false,
    isPaid: false,
    isDeleted: false,
  },
  {
    id: 4,
    amount: 250,
    isActive: true,
    isPaid: false,
    isDeleted: false,
  },
];
```

Create collection:

```javascript
const pendingTransactions =
  createPendingTransactionCollection(
    transactions
  );
```

Iterate:

```javascript
for (
  const transaction
  of pendingTransactions
) {
  console.log(
    transaction.id,
    transaction.amount
  );
}
```

Output:

```text
1 500
4 250
```

Convert to an array:

```javascript
const pendingArray = [
  ...pendingTransactions,
];

console.log(pendingArray);
```

### Why this is useful

The collection encapsulates its filtering rules.

Consumers do not need to repeat:

```javascript
transaction.isActive === true &&
transaction.isPaid === false &&
transaction.isDeleted === false
```

The consumer simply asks for the collection's values.

```javascript
for (const transaction of pendingTransactions) {
  // Only valid pending transactions appear.
}
```

### Professional insight

Custom iterables can hide:

* Filtering logic.
* Traversal logic.
* Tree traversal.
* Pagination state.
* Cursor movement.
* Resource management.
* Domain-specific ordering.

# 30. Common Mistakes

## Mistake 1: Confusing iterable and iterator

Incorrect statement:

> An array is an iterator.

More accurate:

> An array is iterable and can create iterator objects.

```javascript
const iterator =
  array[Symbol.iterator]();
```

---

## Mistake 2: Returning a value instead of an iterator

Incorrect:

```javascript
const collection = {
  [Symbol.iterator]() {
    return 10;
  },
};
```

The method must return an object with `next()`.

Correct:

```javascript
const collection = {
  [Symbol.iterator]() {
    return {
      next() {
        return {
          value: undefined,
          done: true,
        };
      },
    };
  },
};
```

---

## Mistake 3: Returning a primitive from `next()`

Incorrect:

```javascript
next() {
  return 10;
}
```

Correct:

```javascript
next() {
  return {
    value: 10,
    done: false,
  };
}
```

---

## Mistake 4: Never setting `done: true`

```javascript
next() {
  return {
    value: 1,
    done: false,
  };
}
```

A loop over this iterator never finishes.

This may be intentional for an infinite iterator, but consumers must limit iteration.

---

## Mistake 5: Sharing cursor state unintentionally

```javascript
let current = 1;

const range = {
  [Symbol.iterator]() {
    return {
      next() {
        // Uses shared outer current.
      },
    };
  },
};
```

Multiple iterators now interfere with one another.

Prefer creating cursor state inside `[Symbol.iterator]()`.

---

## Mistake 6: Attempting `for...of` on a plain object

```javascript
for (const value of user) {
}
```

Plain objects are not iterable by default.

Use `Object.keys()`, `Object.values()`, `Object.entries()`, or implement a custom iterator.

---

## Mistake 7: Consuming a single-use iterator twice

```javascript
const iterator =
  createSingleUseIterator();

console.log([
  ...iterator,
]);

console.log([
  ...iterator,
]);
```

The second result may be empty because the iterator is exhausted.

---

## Mistake 8: Expanding an infinite iterable

```javascript
const values = [
  ...infiniteIterable,
];
```

This never finishes and may exhaust memory.

---

## Mistake 9: Performing all work eagerly

A custom iterable loses lazy-evaluation benefits if it precomputes the entire result array before returning its iterator.

# 31. Best Practices

## 31.1 Separate iterable data from iterator state

Place traversal state inside the iterator returned by `[Symbol.iterator]()`.

## 31.2 Return a fresh iterator for reusable collections

```javascript
[Symbol.iterator]() {
  let index = 0;

  return {
    next() {
      // ...
    },
  };
}
```

## 31.3 Validate constructor or factory inputs

Invalid boundaries or step values should fail early with meaningful errors.

## 31.4 Keep `next()` predictable

Each call should either:

* Produce one value.
* Report completion.
* Throw a meaningful error.

## 31.5 Use lazy evaluation when collections may be large

Avoid allocating large intermediate arrays unnecessarily.

## 31.6 Support cleanup when controlling resources

Implement `return()` when early termination should release a resource.

## 31.7 Prefer generator functions for complex iterators

Manual iterators are useful for learning and precise control, but generators usually produce clearer code.

You will study them next.

## 31.8 Document single-use iterators

Consumers should know whether iteration can restart.

# 32. Performance Considerations

## Memory

An iterator can generate one value at a time.

Array approach:

```text
Create 1,000,000 values
Store all in memory
Process values
```

Lazy iterator:

```text
Create iterator
Generate one value
Process it
Discard or continue
```

The lazy approach can significantly reduce peak memory.

## Time

Iterators add method-call and result-object overhead.

For small arrays, direct loops may be simpler and potentially faster.

Choose iterators primarily for:

* Abstraction.
* Laziness.
* Custom traversal.
* Large sequences.
* Composability.
* Streams.

## Early stopping

A consumer may stop after a few values.

```javascript
const [
  first,
  second,
] = hugeIterable;
```

Only the required values need to be produced.

## Infinite sequences

Only lazy iteration can represent an infinite sequence safely.

An infinite array cannot be fully created.

## Intermediate allocations

Compare:

```javascript
const filtered =
  values.filter(predicate);

const mapped =
  filtered.map(transform);
```

This creates intermediate arrays.

A lazy iterator pipeline can potentially filter and transform one value at a time.

Generators will make this pattern much easier.

# 33. Debugging Iterators

## Inspect the iterator

```javascript
const iterator =
  collection[Symbol.iterator]();

console.log(iterator);
```

Check whether it has:

```javascript
typeof iterator.next === "function"
```

## Log every `next()` result

```javascript
let result =
  iterator.next();

while (!result.done) {
  console.log(result);

  result =
    iterator.next();
}

console.log(result);
```

## Track cursor state

```javascript
next() {
  console.log({
    current,
  });

  // ...
}
```

## Check protocol compliance

Verify:

```javascript
typeof collection[Symbol.iterator] === "function"
```

and:

```javascript
const iterator =
  collection[Symbol.iterator]();

typeof iterator.next === "function"
```

and:

```javascript
typeof iterator.next() === "object"
```

## Detect shared state

Create two iterators:

```javascript
const firstIterator =
  collection[Symbol.iterator]();

const secondIterator =
  collection[Symbol.iterator]();

console.log(
  firstIterator.next()
);

console.log(
  secondIterator.next()
);
```

For a reusable collection, both should usually begin with the first value.

## Test completion behavior

After completion, repeatedly call:

```javascript
iterator.next();
```

A well-behaved iterator should continue reporting completion.

# 34. Interview Questions and Answers

## Beginner Questions

### 1. What is an iterable?

An iterable is an object that defines a method at `Symbol.iterator` which returns an iterator.

### 2. What is an iterator?

An iterator is an object with a `next()` method that returns iterator result objects.

### 3. What does `next()` return?

An object usually containing:

```javascript
{
  value,
  done,
}
```

### 4. What does `done: true` mean?

It means the iterator has no more values to produce.

### 5. Are arrays iterable?

Yes. Arrays implement the iterable protocol.

### 6. Are plain objects iterable?

Not by default.

---

## Intermediate Questions

### 1. How does `for...of` use an iterable?

It obtains an iterator through `[Symbol.iterator]()` and repeatedly calls `next()` until `done` becomes `true`.

### 2. What is the difference between an iterable and an iterator?

An iterable can create an iterator. An iterator tracks traversal state and produces values.

### 3. Why is `Symbol.iterator` a Symbol rather than a string key?

It reduces naming collisions and provides a standardized, unique protocol hook.

### 4. What is lazy evaluation?

Calculating values only when they are requested.

### 5. Can an iterator be infinite?

Yes, if it never returns `done: true`. Consumers must limit how much they request.

### 6. What is a single-use iterator?

An iterator whose traversal state is consumed and cannot automatically restart.

---

## Advanced Questions

### 1. Why should a reusable iterable create a new iterator for each `[Symbol.iterator]()` call?

So multiple consumers have independent traversal state and iteration can restart.

### 2. Why is `{ value: undefined, done: false }` different from `{ value: undefined, done: true }`?

The first produces a legitimate `undefined` sequence value. The second signals completion.

### 3. What purpose does an iterator's optional `return()` method serve?

It allows cleanup when iteration ends early due to `break`, `return`, or an exception.

### 4. How does destructuring use iterators?

It requests values through the iterable protocol until it has filled the required destructuring positions or consumed the rest element.

### 5. Why can iterators reduce memory usage?

They can calculate and return one value at a time instead of storing the entire sequence.

### 6. What is the relationship between generators and iterators?

Calling a generator function returns a generator object that implements iterator behavior automatically.

### 7. How are async iterators different from synchronous iterators?

An async iterator's `next()` returns a Promise for an iterator result rather than returning the iterator result immediately.
