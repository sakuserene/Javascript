# 1. Introduction

In the previous lessons, you learned how synchronous iterators produce values one at a time.

A synchronous iterator follows this shape:

```javascript
const iterator = {
  next() {
    return {
      value: "A",
      done: false,
    };
  },
};
```

Calling:

```javascript
iterator.next();
```

returns an iterator result immediately.

That works well when the next value is already available in memory.

Examples include:

* The next array element.
* The next character in a string.
* The next number in a range.
* The next node in an in-memory tree.
* The next value generated by a synchronous calculation.

However, many real-world values are not available immediately.

Examples include:

* The next page from an API.
* The next chunk from a network response.
* The next database record.
* The next WebSocket message.
* The next line from a large file.
* The next item from a background queue.
* The next sensor measurement.
* The next locally synchronized transaction.
* The next message from a server.

These values become available in the future.

A synchronous iterator cannot wait asynchronously before returning its next result.

That limitation led to the introduction of **async iterators**.

---

# 2. The Limitation of Synchronous Iterators

Consider a synchronous iterator:

```javascript
const iterator = {
  next() {
    return {
      value: 10,
      done: false,
    };
  },
};
```

The `next()` method must return the iterator result immediately.

```text
Call next()
    ↓
Receive result immediately
```

Now imagine that obtaining the next value requires a network request.

```text
Call next()
    ↓
Send HTTP request
    ↓
Wait for server
    ↓
Receive response later
```

The result cannot be returned immediately.

A synchronous iterator cannot safely do this:

```javascript
const iterator = {
  next() {
    const response = await fetch("/api/value");

    return {
      value: response,
      done: false,
    };
  },
};
```

This is invalid because `await` cannot be used inside a normal non-async method.

We could make `next()` async:

```javascript
const iterator = {
  async next() {
    const response =
      await fetch("/api/value");

    return {
      value: response,
      done: false,
    };
  },
};
```

Now `next()` returns a Promise.

That is the essential idea behind the async iterator protocol.

# 3. Values That Arrive Over Time

A normal collection often already exists.

```javascript
const transactions = [
  {
    id: 1,
  },
  {
    id: 2,
  },
  {
    id: 3,
  },
];
```

Every value is ready.

Iteration can be synchronous:

```text
Request next value
    ↓
Return it immediately
```

A paginated API is different.

```text
Request page 1
    ↓
Wait for network
    ↓
Return page 1 records
    ↓
Request page 2
    ↓
Wait for network
    ↓
Return page 2 records
```

The collection is not fully available at the start.

It is produced gradually.

This is sometimes described as a **time-dependent sequence**.

```text
Value 1 available now

Value 2 available later

Value 3 available later still
```

An async iterator models this naturally.

---

# 4. Why Promises Alone Are Not Always Enough

A Promise represents one future outcome.

```javascript
const userPromise =
  fetchUser();
```

Conceptually:

```text
Promise
    ↓
One future result
```

But what if there are many future values?

Examples:

```text
Page 1
Page 2
Page 3
Page 4
```

or:

```text
Message 1
Message 2
Message 3
...
```

You could return one Promise containing a complete array:

```javascript
const allMessages =
  await fetchEveryMessage();
```

But this may be inefficient or impossible when:

* The data set is very large.
* The sequence is infinite.
* The user may stop early.
* Each page must be requested separately.
* Values should be processed immediately.
* The complete collection cannot fit in memory.
* The source remains open continuously.

Promises model one future result.

Async iterators model many future results over time.

```text
Promise

One future value
```

```text
Async Iterator

Future value
    ↓
Future value
    ↓
Future value
    ↓
Completion
```

---

# 5. What Is Asynchronous Iteration?

Asynchronous iteration means requesting sequence values that may require asynchronous work before they become available.

The consumer repeatedly asks:

```text
Do you have another value?
```

The producer responds with a Promise:

```text
I will tell you later.
```

Eventually, the Promise fulfills with:

```javascript
{
  value: someValue,
  done: false,
}
```

or:

```javascript
{
  value: undefined,
  done: true,
}
```

The pattern is:

```text
Call next()
    ↓
Receive Promise
    ↓
Wait for Promise
    ↓
Receive iterator result
    ↓
Process value
    ↓
Repeat
```

---

# 6. What Is an Async Iterator?

An **async iterator** is an object whose `next()` method returns a Promise for an iterator result object.

Conceptual shape:

```javascript
const asyncIterator = {
  next() {
    return Promise.resolve({
      value: "A",
      done: false,
    });
  },
};
```

It may also be written using an async method:

```javascript
const asyncIterator = {
  async next() {
    return {
      value: "A",
      done: false,
    };
  },
};
```

Because async functions always return Promises, these are conceptually similar.

An async iterator answers:

> “What is the next value, once it becomes available?”

---

# 7. What Is an Async Iterable?

An **async iterable** is an object that provides an async iterator.

It implements a method at:

```javascript
Symbol.asyncIterator
```

Conceptual shape:

```javascript
const asyncIterable = {
  [Symbol.asyncIterator]() {
    return asyncIterator;
  },
};
```

The method must return an object following the async iterator protocol.

An async iterable answers:

> “How can asynchronous iteration over me begin?”

---

# 8. Async Iterable vs Async Iterator

| Async Iterable                       | Async Iterator                      |
| ------------------------------------ | ----------------------------------- |
| Represents an async source of values | Produces async values one at a time |
| Has `[Symbol.asyncIterator]()`       | Has `next()`                        |
| Begins asynchronous iteration        | Advances asynchronous iteration     |
| Often reusable                       | Usually stateful                    |
| May create fresh iterators           | Often consumed once                 |
| Example: paginated API collection    | Example: active page cursor         |

Visual model:

```text
Async Iterable
    ↓ [Symbol.asyncIterator]()
Async Iterator
    ↓ next()
Promise
    ↓ await
{ value, done }
```

---

# 9. The Async Iterator Protocol

An object follows the async iterator protocol when:

1. It has a `next()` method.
2. Calling `next()` returns a Promise or Promise-like value.
3. That Promise fulfills with an iterator result object.
4. The iterator result object contains:

   * `value`
   * `done`

Example:

```javascript
const iterator = {
  async next() {
    return {
      value: 10,
      done: false,
    };
  },
};
```

Calling:

```javascript
const resultPromise =
  iterator.next();
```

returns a Promise.

Use:

```javascript
const result =
  await resultPromise;
```

Result:

```javascript
{
  value: 10,
  done: false,
}
```

---

# 10. The Async Iterable Protocol

An object follows the async iterable protocol if it defines:

```javascript
object[Symbol.asyncIterator]
```

as a method returning an async iterator.

Example:

```javascript
const collection = {
  [Symbol.asyncIterator]() {
    return {
      async next() {
        return {
          value: undefined,
          done: true,
        };
      },
    };
  },
};
```

The collection can later be consumed using:

```javascript
for await (
  const value
  of collection
) {
  console.log(value);
}
```

The `for await...of` syntax will be studied deeply in a later lesson.

For now, focus on the protocol itself.

# 11. `Symbol.asyncIterator`

## What is it?

`Symbol.asyncIterator` is a built-in well-known Symbol.

JavaScript uses it to request an async iterator from an object.

```javascript
const method =
  object[Symbol.asyncIterator];
```

Calling it:

```javascript
const iterator =
  object[Symbol.asyncIterator]();
```

returns the async iterator.

## Why a separate Symbol?

Synchronous and asynchronous iteration have different contracts.

Synchronous iteration:

```javascript
object[Symbol.iterator]()
```

produces an iterator whose `next()` returns an iterator result directly.

Asynchronous iteration:

```javascript
object[Symbol.asyncIterator]()
```

produces an iterator whose `next()` returns a Promise for an iterator result.

The separate Symbols prevent ambiguity.

```text
Symbol.iterator
    ↓
Synchronous next()

Symbol.asyncIterator
    ↓
Asynchronous next()
```

# 12. Synchronous vs Asynchronous Iterator Results

## Synchronous iterator

```javascript
const result =
  iterator.next();
```

`result` is immediately:

```javascript
{
  value: 10,
  done: false,
}
```

## Asynchronous iterator

```javascript
const resultPromise =
  asyncIterator.next();
```

`resultPromise` is:

```text
Promise
```

Await it:

```javascript
const result =
  await resultPromise;
```

Then:

```javascript
{
  value: 10,
  done: false,
}
```

## Comparison table

| Feature                         | Sync Iterator          | Async Iterator              |
| ------------------------------- | ---------------------- | --------------------------- |
| Protocol key                    | `Symbol.iterator`      | `Symbol.asyncIterator`      |
| `next()` return                 | Iterator result object | Promise for iterator result |
| Can wait for network            | No                     | Yes                         |
| Consumer                        | `for...of`             | `for await...of`            |
| Good for in-memory values       | Yes                    | Sometimes unnecessary       |
| Good for streamed future values | No                     | Yes                         |

# 13. How `next()` Works in an Async Iterator

Consider:

```javascript
const asyncIterator = {
  async next() {
    await delay(500);

    return {
      value: "A",
      done: false,
    };
  },
};
```

Calling:

```javascript
const promise =
  asyncIterator.next();
```

does not immediately return the iterator result.

It returns a Promise.

Execution:

```text
Call next()
    ↓
Enter async method
    ↓
Encounter await
    ↓
Pause next()
    ↓
Return Promise to caller
    ↓
Delay completes
    ↓
Resume next()
    ↓
Return iterator result
    ↓
Promise fulfills
```

The consumer can write:

```javascript
const result =
  await asyncIterator.next();
```

This waits for the next sequence result.

# 14. Manual Consumption of an Async Iterator

Before learning `for await...of`, consume an async iterator manually.

```javascript
async function consume(
  asyncIterable
) {
  const iterator =
    asyncIterable[
      Symbol.asyncIterator
    ]();

  while (true) {
    const result =
      await iterator.next();

    if (result.done) {
      break;
    }

    console.log(result.value);
  }
}
```

This is the asynchronous equivalent of manual synchronous iteration.

Execution:

```text
Get async iterator
    ↓
Await iterator.next()
    ↓
Check done
    ↓
Use value
    ↓
Await next result
    ↓
Repeat
```

# 15. The Async Iterator Lifecycle

An async iterator typically moves through states such as:

```text
Created
    ↓
Waiting for next request
    ↓
Producing next value
    ↓
Promise pending
    ↓
Result available
    ↓
Waiting for next request
    ↓
Completed
```

More detailed model:

```text
Idle
    ↓ next()
Executing async operation
    ↓
Awaiting external result
    ↓
Promise fulfills
    ↓
Iterator result returned
    ↓
Idle again
```

Eventually:

```text
No more values
    ↓
Return {
  value: undefined,
  done: true
}
    ↓
Completed
```

# 16. Internal JavaScript Runtime Behavior

Consider:

```javascript
const result =
  await iterator.next();
```

Conceptual runtime process:

```text
1. Read iterator.next
2. Call next()
3. Receive Promise
4. Suspend current async consumer
5. Continue other JavaScript work
6. Underlying operation completes
7. Resolve next() Promise
8. Schedule consumer continuation as microtask
9. Resume consumer
10. Assign iterator result to result
11. Check result.done
12. Process result.value
```

ASCII diagram:

```text
Consumer
   ↓ next()
Async Iterator
   ↓ starts async work
Browser API / Server / Timer
   ↓ completes
Promise resolves
   ↓ microtask
Consumer resumes
   ↓
{ value, done }
```

No JavaScript thread is blocked while waiting.

# 17. Creating a Basic Manual Async Iterator

Create an iterator that produces three values with delays.

## Delay utility

```javascript
function delay(
  milliseconds
) {
  return new Promise(resolve => {
    setTimeout(
      resolve,
      milliseconds
    );
  });
}
```

## Iterator implementation

```javascript
function createAsyncNumberIterator() {
  let current = 1;

  return {
    async next() {
      await delay(500);

      if (current <= 3) {
        const value =
          current;

        current += 1;

        return {
          value,
          done: false,
        };
      }

      return {
        value: undefined,
        done: true,
      };
    },
  };
}
```

## Manual usage

```javascript
async function run() {
  const iterator =
    createAsyncNumberIterator();

  console.log(
    await iterator.next()
  );

  console.log(
    await iterator.next()
  );

  console.log(
    await iterator.next()
  );

  console.log(
    await iterator.next()
  );
}

run();
```

Output over time:

```javascript
{
  value: 1,
  done: false,
}
```

```javascript
{
  value: 2,
  done: false,
}
```

```javascript
{
  value: 3,
  done: false,
}
```

```javascript
{
  value: undefined,
  done: true,
}
```

Each result takes approximately 500 milliseconds to become available.

# 18. Creating a Reusable Async Iterable

The previous object is an async iterator but not necessarily a reusable async iterable.

Create a reusable range-like source:

```javascript
function createAsyncRange(
  start,
  end,
  delayMilliseconds = 500
) {
  return {
    [Symbol.asyncIterator]() {
      let current = start;

      return {
        async next() {
          await delay(
            delayMilliseconds
          );

          if (current <= end) {
            const value =
              current;

            current += 1;

            return {
              value,
              done: false,
            };
          }

          return {
            value: undefined,
            done: true,
          };
        },
      };
    },
  };
}
```

Manual consumption:

```javascript
async function consumeRange() {
  const range =
    createAsyncRange(
      1,
      3,
      300
    );

  const iterator =
    range[
      Symbol.asyncIterator
    ]();

  console.log(
    await iterator.next()
  );

  console.log(
    await iterator.next()
  );
}
```

Each call to:

```javascript
range[Symbol.asyncIterator]()
```

creates independent state.

That makes the source reusable.

# 19. Single-Use Async Iterators

An async iterator can return itself from `Symbol.asyncIterator`.

```javascript
const iterator = {
  current: 1,

  async next() {
    await delay(300);

    if (this.current <= 3) {
      return {
        value:
          this.current++,
        done: false,
      };
    }

    return {
      value: undefined,
      done: true,
    };
  },

  [Symbol.asyncIterator]() {
    return this;
  },
};
```

This object is both:

* An async iterator.
* An async iterable.

However, it is usually single-use because it stores one shared cursor.

After exhaustion, it does not automatically restart.

Use a factory or reusable async iterable when multiple independent iterations are required.

# 20. Async Iterator State

Async iterators must preserve state between asynchronous `next()` calls.

State may include:

* Current page number.
* Current record index.
* Continuation token.
* Whether the source is complete.
* Active stream reader.
* Pending data buffer.
* Cancellation state.
* Retry count.
* Connection details.
* Resource handles.

Example:

```javascript
function createPagedIterator() {
  let page = 1;
  let finished = false;

  return {
    async next() {
      if (finished) {
        return {
          value: undefined,
          done: true,
        };
      }

      const response =
        await fetchPage(page);

      page += 1;

      if (!response.hasMore) {
        finished = true;
      }

      return {
        value:
          response.records,
        done: false,
      };
    },
  };
}
```

The iterator remembers which page comes next.

# 21. Error Handling in Async Iterators

If `next()` fails, its returned Promise rejects.

```javascript
const iterator = {
  async next() {
    throw new Error(
      "Unable to load next value"
    );
  },
};
```

Consumer:

```javascript
try {
  const result =
    await iterator.next();

  console.log(result);
} catch (error) {
  console.error(
    error.message
  );
}
```

Output:

```text
Unable to load next value
```

## Error sources

Async iteration may fail because of:

* Network errors.
* Authentication failures.
* Invalid server responses.
* File-read errors.
* Connection closure.
* Parsing errors.
* Cancellation.
* Timeout.
* Database errors.

## Contextual errors

Wrap lower-level errors:

```javascript
async next() {
  try {
    return await loadNextPage();
  } catch (error) {
    throw new Error(
      `Failed to load page ${page}`,
      {
        cause: error,
      }
    );
  }
}
```

This preserves useful context.

# 22. Async Iterator Cleanup

Async iterators may provide an async `return()` method.

```javascript
const iterator = {
  async next() {
    // Produce values.
  },

  async return() {
    await closeConnection();

    return {
      value: undefined,
      done: true,
    };
  },
};
```

Cleanup may include:

* Closing a stream.
* Releasing a database cursor.
* Aborting a network request.
* Removing an event listener.
* Unsubscribing from a message source.
* Releasing a lock.
* Clearing timers.

An async consumer should clean up when iteration ends early.

`for await...of` performs iterator closing automatically in relevant early-exit situations.

# 23. Early Termination

A consumer may stop before the source is exhausted.

Examples:

* A target record was found.
* The user cancelled.
* A limit was reached.
* An error occurred.
* A loop used `break`.
* The surrounding function returned.

Conceptual flow:

```text
Consumer stops early
    ↓
Call iterator.return()
    ↓
Await cleanup
    ↓
Finish
```

Manual consumer with cleanup:

```javascript
async function consumeFirst(
  asyncIterable
) {
  const iterator =
    asyncIterable[
      Symbol.asyncIterator
    ]();

  try {
    const result =
      await iterator.next();

    return result.value;
  } finally {
    if (
      typeof iterator.return ===
      "function"
    ) {
      await iterator.return();
    }
  }
}
```

# 24. Backpressure Introduction

**Backpressure** describes how a consumer controls the rate at which a producer supplies values.

Consider an async iterator.

```javascript
const result =
  await iterator.next();
```

The consumer requests one value.

It does not request another until it is ready.

```text
Consumer requests value 1
    ↓
Producer creates value 1
    ↓
Consumer processes value 1
    ↓
Consumer requests value 2
```

This naturally limits production.

Compare with an event listener:

```javascript
socket.addEventListener(
  "message",
  handleMessage
);
```

Messages may arrive faster than the handler can process them.

An async iterator can provide a pull-based model:

```text
Consumer pulls when ready
```

This can help manage:

* Memory use.
* Processing rate.
* Network streams.
* Large files.
* Queues.
* Data transformations.

Backpressure will be studied more deeply in the stream-processing lesson.

# 25. Async Iterators vs Arrays of Promises

Suppose you have:

```javascript
const promises = [
  fetchA(),
  fetchB(),
  fetchC(),
];
```

All operations generally start immediately.

```text
A starts
B starts
C starts
```

An async iterator may start one operation only when the consumer requests the next value.

```text
Request A
    ↓ process A
Request B
    ↓ process B
Request C
```

## Arrays of Promises

Best when:

* The task count is known.
* All operations can start together.
* Results are independent.
* Concurrent execution is desirable.

## Async iterators

Best when:

* Values arrive gradually.
* The sequence length is unknown.
* The source may be infinite.
* The consumer may stop early.
* Each next value should be requested lazily.
* Backpressure matters.
* The source owns a cursor or stream.

---

# 26. Async Iterators vs Promise Combinators

Promise combinators coordinate a fixed set of Promises.

```javascript
await Promise.all([
  promiseA,
  promiseB,
  promiseC,
]);
```

Async iterators coordinate a sequence that may be generated over time.

```text
next Promise
    ↓
next Promise
    ↓
next Promise
```

## `Promise.all()`

```text
Known group
Start together
Wait for all
```

## Async iterator

```text
Potentially unknown group
Request values incrementally
Stop when done or no longer needed
```

Do not replace every `Promise.all()` with an async iterator.

They solve different problems.

---

# 27. Async Iterators vs Event Listeners

Event listeners use a push model.

```javascript
button.addEventListener(
  "click",
  handleClick
);
```

The producer pushes events whenever they occur.

Async iterators often use a pull model.

```text
Consumer requests next event
    ↓
Wait until event occurs
    ↓
Receive event
```

## Event listener strengths

* Natural for user interface events.
* Multiple independent listeners.
* Persistent event handling.
* Browser-native design.

## Async iterator strengths

* Linear consumption syntax.
* Easier cleanup in some workflows.
* Natural sequencing.
* Backpressure.
* Useful for streams and message queues.
* Values can be processed with `for await...of`.

Some systems wrap events as async iterables, but ordinary event listeners remain the standard choice for many DOM interactions.

---

# 28. When Async Iterators Should Be Used

Use async iterators when:

* Values arrive over time.
* The number of values is unknown.
* The source is paginated.
* The source may be infinite.
* The consumer should process values incrementally.
* The consumer may stop early.
* Backpressure matters.
* Holding all results in memory is undesirable.
* The source owns a stream, cursor, or subscription.
* Values need sequential asynchronous processing.

Real-world examples:

* Paginated REST API results.
* Streaming fetch response chunks.
* Database cursors.
* Large file chunks.
* WebSocket message streams.
* Background queue consumers.
* Sensor readings.
* Server-sent events wrapped in an async abstraction.
* Offline synchronization records.

---

# 29. When Async Iterators Should Not Be Used

Avoid async iterators when:

* There is only one future value.
* A normal Promise is sufficient.
* All operations are known and should run concurrently.
* `Promise.all()` communicates the intent better.
* The data is already fully available in memory.
* A synchronous iterator is simpler.
* DOM event listeners are more natural.
* The abstraction would add complexity without improving behavior.

Examples:

Use a Promise:

```javascript
const user =
  await fetchUser();
```

Use `Promise.all()`:

```javascript
const [
  user,
  orders,
] = await Promise.all([
  fetchUser(),
  fetchOrders(),
]);
```

Use a normal loop:

```javascript
for (const item of items) {
  console.log(item);
}
```

# 30. Three Progressive Examples

## Example 1: Beginner — Delayed Values

### Goal

Produce three values over time.

```javascript
function delay(
  milliseconds
) {
  return new Promise(resolve => {
    setTimeout(
      resolve,
      milliseconds
    );
  });
}
```

Async iterable:

```javascript
function createDelayedValues(
  values,
  delayMilliseconds
) {
  return {
    [Symbol.asyncIterator]() {
      let index = 0;

      return {
        async next() {
          if (
            index >=
            values.length
          ) {
            return {
              value: undefined,
              done: true,
            };
          }

          await delay(
            delayMilliseconds
          );

          const value =
            values[index];

          index += 1;

          return {
            value,
            done: false,
          };
        },
      };
    },
  };
}
```

Manual consumer:

```javascript
async function consumeValues() {
  const source =
    createDelayedValues(
      [
        "A",
        "B",
        "C",
      ],
      500
    );

  const iterator =
    source[
      Symbol.asyncIterator
    ]();

  while (true) {
    const result =
      await iterator.next();

    if (result.done) {
      break;
    }

    console.log(result.value);
  }
}

consumeValues();
```

Output over time:

```text
A
B
C
```

```javascript
[Symbol.asyncIterator]()
```

creates a fresh cursor.

```javascript
let index = 0;
```

stores iteration position.

```javascript

```

means every result is delivered through a Promise.

```javascript

```

simulates a future value.

```javascript
return {
  value,
  done: false,

```

fulfills the `next()` Promise with an iterator result.

## Example 3: Real-World Application — Pending Transaction Cursor

### Goal

Produce pending transactions in batches while supporting cleanup.

Data source:

```javascript
const pendingTransactions = [
  {
    id: 1,
    amount: 500,
  },
  {
    id: 2,
    amount: 800,
  },
  {
    id: 3,
    amount: 250,
  },
  {
    id: 4,
    amount: 900,
  },
  {
    id: 5,
    amount: 300,
  },
];
```

Batch reader:

```javascript
async function readTransactionBatch(
  offset,
  limit
) {
  await delay(400);

  return pendingTransactions.slice(
    offset,
    offset + limit
  );
}
```

Async iterable:

```javascript
function createTransactionCursor({
  batchSize = 2,
} = {}) {
  if (
    !Number.isInteger(batchSize) ||
    batchSize < 1
  ) {
    throw new RangeError(
      "batchSize must be a positive integer."
    );
  }

  return {
    [Symbol.asyncIterator]() {
      let offset = 0;
      let closed = false;

      return {
        async next() {
          if (closed) {
            return {
              value: undefined,
              done: true,
            };
          }

          const batch =
            await readTransactionBatch(
              offset,
              batchSize
            );

          if (
            batch.length === 0
          ) {
            closed = true;

            return {
              value: undefined,
              done: true,
            };
          }

          offset +=
            batch.length;

          return {
            value: batch,
            done: false,
          };
        },

        async return() {
          closed = true;

          await delay(100);

          console.log(
            "Transaction cursor closed"
          );

          return {
            value: undefined,
            done: true,
          };
        },
      };
    },
  };
}
```

Manual consumer that stops early:

```javascript
async function findTransaction(
  targetId
) {
  const cursor =
    createTransactionCursor({
      batchSize: 2,
    });

  const iterator =
    cursor[
      Symbol.asyncIterator
    ]();

  try {
    while (true) {
      const result =
        await iterator.next();

      if (result.done) {
        return null;
      }

      const match =
        result.value.find(
          transaction =>
            transaction.id ===
            targetId
        );

      if (match) {
        return match;
      }
    }
  } finally {
    if (
      typeof iterator.return ===
      "function"
    ) {
      await iterator.return();
    }
  }
}
```

Usage:

```javascript
findTransaction(3)
  .then(console.log);
```

Possible output:

```text
Transaction cursor closed
{ id: 3, amount: 250 }
```

### Professional design points

* Data is loaded lazily in batches.
* The cursor stores the current offset.
* The consumer can stop after finding a target.
* Cleanup runs in `finally`.
* The complete transaction list does not need to be copied into another structure.
* The same protocol can later be consumed with `for await...of`.

# 31. Common Mistakes

## Mistake 1: Using `Symbol.iterator`

Incorrect for asynchronous iteration:

```javascript
[Symbol.iterator]() {
  return asyncIterator;
}
```

Use:

```javascript
[Symbol.asyncIterator]() {
  return asyncIterator;
}
```

---

## Mistake 2: Returning the iterator result directly from an expected async `next()`

This may technically be assimilated in some consuming mechanisms, but it violates the intended async contract when future work is expected.

Prefer:

```javascript
async next() {
  return {
    value,
    done,
  };
}
```

or:

```javascript
next() {
  return Promise.resolve({
    value,
    done,
  });
}
```

---

## Mistake 3: Forgetting to await `next()`

Incorrect:

```javascript
const result =
  iterator.next();

console.log(result.value);
```

`result` is a Promise.

Correct:

```javascript
const result =
  await iterator.next();
```

---

## Mistake 4: Never returning `done: true`

The consumer waits forever or continues requesting values indefinitely.

Only do this intentionally for infinite sources.

---

## Mistake 5: Loading all data eagerly

This defeats the purpose of async iteration:

```javascript
const allPages =
  await fetchEveryPage();

return createIterable(
  allPages
);
```

When incremental loading is required, fetch the next page only when requested.

---

## Mistake 6: Missing cleanup

A cursor or stream may remain open after early termination.

Implement `return()` and ensure consumers call it or use `for await...of`.

---

## Mistake 7: Sharing state across iterations

Place cursor state inside:

```javascript
[Symbol.asyncIterator]()
```

for independent reusable iteration.

---

## Mistake 8: Starting several `next()` calls concurrently without protocol support

Risky:

```javascript
const first =
  iterator.next();

const second =
  iterator.next();
```

Some custom iterators assume only one active `next()` operation.

Unless concurrency is explicitly supported, await one `next()` before requesting another.

---

# 32. Best Practices

## 32.1 Keep iteration state private

Use closures inside `Symbol.asyncIterator`.

## 32.2 Return a fresh iterator for reusable sources

Each consumer should usually receive an independent cursor.

## 32.3 Make completion stable

After completion, later `next()` calls should continue reporting:

```javascript
{
  value: undefined,
  done: true,
}
```

## 32.4 Add cleanup for resource-backed sources

Implement `return()` for streams, cursors, subscriptions, and connections.

## 32.5 Add context to errors

Include page number, cursor position, record ID, or source name.

## 32.6 Avoid concurrent `next()` calls unless supported

Document whether the iterator permits them.

## 32.7 Keep each result meaningful

Decide whether one result represents:

* One record.
* One page.
* One chunk.
* One message.
* One batch.

## 32.8 Prefer async generators for complex sources

Manual async iterators are valuable for understanding the protocol, but async generators often produce clearer code.

# 33. Performance Considerations

## Memory

Async iterators can process values incrementally.

```text
Load one page
Process it
Discard or store required results
Load next page
```

This reduces peak memory compared with loading all pages first.

## Latency

Consumers can begin processing before the entire source is complete.

```text
Page 1 received
    ↓
Begin processing
```

instead of:

```text
Wait for every page
    ↓
Begin processing
```

## Request overhead

One-item-per-request iteration may be inefficient.

Batching may reduce network and protocol overhead.

## Sequential production

Most async iterators produce one value per `next()` request.

This supports backpressure but may be slower than parallel requests.

Choose based on source limits and workflow needs.

## Cleanup cost

Resource cleanup may itself be asynchronous and should be awaited.

# 35. Interview Questions and Answers

## Beginner Questions

### 1. What is an async iterator?

An object whose `next()` method returns a Promise for an iterator result object.

### 2. What is an async iterable?

An object with a method at `Symbol.asyncIterator` that returns an async iterator.

### 3. What does async `next()` eventually produce?

An object usually containing:

```javascript
{
  value,
  done,
}
```

### 4. Which Symbol identifies asynchronous iteration?

```javascript
Symbol.asyncIterator
```

### 5. What loop consumes async iterables?

```javascript

```

---

## Intermediate Questions

### 1. How does an async iterator differ from a synchronous iterator?

A synchronous iterator's `next()` returns an iterator result directly. An async iterator's `next()` returns a Promise for that result.

### 2. Why are async iterators useful for paginated APIs?

They can fetch and expose pages incrementally instead of loading all pages before processing begins.

### 3. Can an async iterator be infinite?

Yes. It may continue producing future values indefinitely.

### 4. What is backpressure?

A flow-control concept where the consumer controls when the next value is requested, preventing the producer from overwhelming it.

### 5. Why might an async iterator implement `return()`?

To release resources when iteration stops early.

### 6. Are async iterators always reusable?

No. Some are single-use cursors. A reusable async iterable should create a fresh iterator for each iteration.

---

## Advanced Questions

### 1. What happens internally when an async iterator's `next()` is awaited?

The consumer pauses, other JavaScript continues, the Promise settles later, and the consumer resumes through the microtask queue.

### 2. Why should concurrent `next()` calls be avoided for many custom async iterators?

The iterator may rely on sequential cursor state and may not be designed to process overlapping requests safely.

### 3. How do async iterators help with memory usage?

They allow incremental processing instead of materializing the entire asynchronous data set.

### 4. What is the difference between an async iterator and an array of Promises?

An array of Promises generally represents a known fixed group that may start concurrently. An async iterator represents values requested and produced gradually over time.

### 5. How does an async generator relate to the async iterator protocol?

Calling an async generator function returns an object that implements the async iterator protocol automatically.

### 6. How should errors from `next()` be handled?

Await `next()` inside `try...catch`, add context where appropriate, and ensure cleanup executes.
