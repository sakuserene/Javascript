# What is `await`?

The `await` keyword tells JavaScript:

> **Pause the execution of the current async function until the awaited value is ready.**

Notice the wording carefully.

It **does not** pause:

- JavaScript
- The browser
- The Event Loop
- Other async functions

It pauses **only the current async function**.

This distinction is one of the most misunderstood concepts in JavaScript.

---

# Why Was `await` Created?

Before `await`, asynchronous code looked like this:

```javascript

.then(user => {

    return fetchOrders(user.id);

})

.then(orders => {

    return calculateTotals(orders);

})

.then(totals => {

    console.log(totals);

});
```

Readable?

Yes.

Easy to follow?

Mostly.

But humans naturally think in **steps**, not in callback chains.

Developers wanted asynchronous code to look like ordinary procedural code.

Instead of:

```

‚Üì

‚Üì

.then()
```

they wanted

```

‚Üì

‚Üì

```

That is exactly what `await` provides.

# Await Is Built on Promises

Remember from previous lessons:

```javascript

```

is conceptually similar to:

```javascript

    // continue here

});
```

The difference is that JavaScript hides this transformation from you.

This is why Async/Await is often described as **syntactic sugar**.

Let's examine

```javascript

    console.log("B");

```

Pause here.

Try to predict the output before reading further.

Output

```text

```

---

## Step 2

`demo()` begins executing.

Output

```text

```

## Step 3

`await` is encountered.

The async function pauses.

Control returns to the main program.

---

## Step 4

Output


## Step 5

The awaited Promise is already fulfilled.

Its continuation enters the **Microtask Queue**.

The Event Loop processes microtasks.


This demonstrates that `await` resumes asynchronously, even when the Promise is already fulfilled.

# Common Mistakes

## Mistake 1

Thinking `await` blocks JavaScript.

It does not.

It suspends only the current async function.

---

## Mistake 2

Using sequential `await`s for independent operations.

Incorrect

```javascript
const a = await getA();
const b = await getB();
```

Better

```javascript
const [a, b] = await Promise.all([

    getA(),

    getB()

]);
```

---

## Mistake 3

Forgetting that `await` always resumes asynchronously.

Even if the Promise is already fulfilled,

the continuation is scheduled as a microtask.

---

## Mistake 4

Using `await` where ordinary synchronous code would suffice.

Example

```javascript
const number = await 5;
```

Valid,

but unnecessary.

---

# Best Practices

- Use `await` only with Promise-returning operations.
- Use sequential `await`s only when operations depend on one another.
- Use `Promise.all()` for independent operations.
- Keep async functions small and focused.
- Think in terms of workflows rather than isolated statements.

# Interview Questions

### Beginner

**Q:** What does `await` do?

**Answer:**

It pauses the current async function until the awaited Promise settles, then resumes execution with the fulfilled value or throws the rejection reason.

---

### Intermediate

**Q:** Can `await` be used with non-Promise values?

**Answer:**

Yes.

JavaScript automatically wraps non-Promise values using behavior equivalent to `Promise.resolve(value)`.

---

### Advanced

**Q:** Why does execution after `await` happen later, even if the Promise is already resolved?

**Answer:**

Because the continuation of an async function is scheduled as a **microtask**. JavaScript resumes the function only after the current synchronous execution completes and the Event Loop processes the Microtask Queue.
