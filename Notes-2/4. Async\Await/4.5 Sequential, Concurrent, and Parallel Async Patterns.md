# 1. Introduction

Writing asynchronous code that produces the correct result is only the beginning.

Professional JavaScript developers must also decide:

* Which operations should run one after another?
* Which operations can begin together?
* Which operations depend on earlier results?
* What should happen if one operation fails?
* How many operations should run at once?
* How can unnecessary waiting be removed?
* How can the application remain responsive?
* How can external systems be protected from excessive traffic?

Consider the following code:

```javascript
async function loadDashboard() {
  const user = await fetchUser();
  const notifications = await fetchNotifications();
  const services = await fetchServices();

  return {
    user,
    notifications,
    services,
  };
}
```

This code is readable and correct.

However, it may be unnecessarily slow.

If the three requests are independent, each request waits for the previous request to finish before it even begins.

```text
fetchUser()
    ↓ wait
fetchNotifications()
    ↓ wait
fetchServices()
    ↓ wait
Finished
```

A concurrent design can start all three requests together:

```javascript
async function loadDashboard() {
  const [user, notifications, services] = await Promise.all([
    fetchUser(),
    fetchNotifications(),
    fetchServices(),
  ]);

  return {
    user,
    notifications,
    services,
  };
}
```

```text
fetchUser() ──────────────┐
fetchNotifications() ────┼──→ Wait for all → Finished
fetchServices() ─────────┘
```

The goal of this lesson is not to make every operation concurrent.

The goal is to choose the correct execution strategy based on dependencies, performance, failure requirements, resource limits, and business rules.

# 2. Why Execution Strategy Matters

Suppose an application needs to perform three network requests.

Each request takes approximately two seconds.

## Sequential execution

```text
Request A: 2 seconds
Request B: 2 seconds
Request C: 2 seconds

Approximate total: 6 seconds
```

## Concurrent execution

```text
Request A: 2 seconds
Request B: 2 seconds
Request C: 2 seconds

All begin together.

Approximate total: 2 seconds
```

The difference can be substantial.

However, concurrent execution is not always correct.

Suppose Request B needs the user ID returned by Request A:

```javascript
const user = await fetchUser();
const orders = await fetchOrders(user.id);
```

`fetchOrders()` cannot begin until `fetchUser()` supplies the ID.

Trying to start both at the same time would be logically incorrect.

Therefore, performance optimization begins with understanding dependencies.

> [!IMPORTANT]
> Do not parallelize operations merely because it is possible. Parallelize only operations that are logically independent and safe to execute together.

---

# 3. Important Terminology

The words **sequential**, **concurrent**, and **parallel** are often used carelessly.

They describe different ideas.

## 3.1 Sequential execution

Tasks start and finish one after another.

```text
Task A
   ↓
Task B
   ↓
Task C
```

Task B does not begin until Task A finishes.

## 3.2 Concurrent execution

Multiple tasks are in progress during overlapping periods.

```text
Task A ──────────────┐
Task B ──────────┐   │
Task C ───────┐  │   │
              ▼  ▼   ▼
            Results become available
```

The tasks may be handled by browser APIs, network systems, databases, operating-system services, or worker threads.

## 3.3 Parallel execution

Multiple operations are physically executing at the same instant, usually on different CPU cores, threads, devices, or systems.

```text
CPU Core 1 → Task A
CPU Core 2 → Task B
CPU Core 3 → Task C
```

## 3.4 Asynchronous execution

The caller does not block while waiting for an operation to finish.

The operation may complete later, and its result is processed when available.

## 3.5 Dependency

A dependency exists when one task requires the output or completion of another task.

```text
Authenticate user
        ↓
Receive access token
        ↓
Fetch protected profile
```

## 3.6 Concurrency limit

A concurrency limit defines the maximum number of asynchronous operations allowed to remain active simultaneously.

For example:

```text
Maximum concurrent uploads: 3
```

This protects memory, bandwidth, servers, databases, and third-party APIs.

# 4. Sequential Execution

Sequential execution occurs when each operation begins only after the previous operation finishes.

```javascript
async function runSequentially() {
  const firstResult = await firstTask();
  const secondResult = await secondTask();
  const thirdResult = await thirdTask();

  return {
    firstResult,
    secondResult,
    thirdResult,
  };
}
```

Execution:

```text
firstTask()
    ↓ await completion
secondTask()
    ↓ await completion
thirdTask()
    ↓ await completion
Return results
```

## 4.1 When sequential execution is correct

Use sequential execution when:

* A later task requires an earlier result.
* Order is a business requirement.
* A task must not begin until another task succeeds.
* Operations modify the same resource and must be ordered.
* Rate limits require slow, controlled execution.
* A transaction workflow must follow strict stages.

## 4.2 Example: dependent requests

```javascript
async function loadUserOrders() {
  const user = await fetchCurrentUser();
  const orders = await fetchOrders(user.id);

  return {
    user,
    orders,
  };
}
```

`fetchOrders()` depends on `user.id`.

Parallel execution would not be appropriate.

## 4.3 Example: transaction workflow

```javascript
async function processTransaction(transaction) {
  const validatedTransaction = await validateTransaction(transaction);
  const savedTransaction = await saveTransaction(validatedTransaction);
  const receipt = await generateReceipt(savedTransaction);

  return receipt;
}
```

The workflow is naturally sequential:

```text
Validate
   ↓
Save
   ↓
Generate receipt
```

Generating the receipt before saving the transaction may create inconsistent data.

## 4.4 Internal working

Consider:

```javascript
const first = await taskOne();
const second = await taskTwo();
```

JavaScript proceeds conceptually as follows:

```text
Call taskOne()
    ↓
Receive Promise A
    ↓
Pause current async function
    ↓
Promise A fulfills
    ↓
Resume function
    ↓
Assign first
    ↓
Call taskTwo()
    ↓
Receive Promise B
    ↓
Pause current async function
    ↓
Promise B fulfills
    ↓
Resume function
    ↓
Assign second
```

`taskTwo()` is not called until after Promise A fulfills.

That is why this form is sequential.

# 5. Concurrent Execution

Concurrent execution begins multiple asynchronous operations before waiting for their results.

```javascript
async function runConcurrently() {
  const firstPromise = firstTask();
  const secondPromise = secondTask();
  const thirdPromise = thirdTask();

  const firstResult = await firstPromise;
  const secondResult = await secondPromise;
  const thirdResult = await thirdPromise;

  return {
    firstResult,
    secondResult,
    thirdResult,
  };
}
```

The important detail is that all task functions are called before the first `await`.

```text
Call firstTask()  → Promise A starts
Call secondTask() → Promise B starts
Call thirdTask()  → Promise C starts
Await Promise A
Await Promise B
Await Promise C
```

Although the results are awaited one at a time, the operations themselves are already in progress.

## 5.1 Cleaner form with `Promise.all()`

```javascript
async function runConcurrently() {
  const [firstResult, secondResult, thirdResult] = await Promise.all([
    firstTask(),
    secondTask(),
    thirdTask(),
  ]);

  return {
    firstResult,
    secondResult,
    thirdResult,
  };
}
```

This form clearly communicates:

> Start these independent tasks and continue when all of them have fulfilled.

## 5.2 When concurrent execution is appropriate

Use concurrent execution when:

* Tasks are independent.
* Their order of completion does not matter.
* All can safely begin at the same time.
* External systems can handle the load.
* The combined results are needed before continuing.

Examples:

* Loading independent dashboard widgets.
* Fetching unrelated API resources.
* Reading several independent files.
* Checking multiple backup servers.
* Uploading independent files within a safe limit.
* Running independent validation services.

# 6. Parallel Execution

JavaScript on the browser's main thread executes one JavaScript instruction at a time.

However, asynchronous operations may still execute in parallel outside that thread.

Examples include:

* Network requests handled by the browser and operating system.
* Database queries handled by a database server.
* File operations handled by the runtime and operating system.
* Work performed by Web Workers.
* Work performed by Node.js worker threads.
* Server-side requests processed by separate remote machines.

Consider:

```javascript
await Promise.all([
  fetch("/api/users"),
  fetch("/api/orders"),
  fetch("/api/services"),
]);
```

The JavaScript code is coordinating three requests.

The network operations may overlap or execute physically in parallel across browser networking systems, the operating system, and remote servers.

> [!NOTE]
> In ordinary frontend discussions, developers often say that `Promise.all()` runs tasks "in parallel." More precisely, it starts or coordinates them concurrently. Whether the underlying work is physically parallel depends on the runtime and the operation.

# 7. Concurrency vs Parallelism

A useful analogy is a chef.

## Concurrency

One chef manages several dishes:

```text
Start rice
    ↓
While rice cooks, prepare vegetables
    ↓
While vegetables cook, prepare sauce
```

The chef works on overlapping tasks but does only one manual action at a time.

## Parallelism

Three chefs each prepare a separate dish at the same time:

```text
Chef 1 → Rice
Chef 2 → Vegetables
Chef 3 → Sauce
```

## JavaScript example

The main thread resembles one chef.

Browser APIs and remote systems act like additional workers.

```text
Main JavaScript thread
    ├── Starts network request A
    ├── Starts network request B
    ├── Starts network request C
    └── Continues other work

Browser/network systems
    ├── Process request A
    ├── Process request B
    └── Process request C
```

# 8. How `await` Affects Start Time

A common performance mistake is assuming that placing `await` inside a function merely waits for a task that has already started.

That depends on where the task function is called.

Compare these two versions.

## 8.1 Sequential start

```javascript
const user = await fetchUser();
const notifications = await fetchNotifications();
```

Timeline:

```text
Call fetchUser()
    ↓ wait
Call fetchNotifications()
    ↓ wait
```

`fetchNotifications()` does not begin until `fetchUser()` finishes.

## 8.2 Concurrent start

```javascript
const userPromise = fetchUser();
const notificationsPromise = fetchNotifications();

const user = await userPromise;
const notifications = await notificationsPromise;
```

Timeline:

```text
Call fetchUser()
Call fetchNotifications()
    ↓ both are now in progress
Await userPromise
Await notificationsPromise
```

## 8.3 Concurrent start with `Promise.all()`

```javascript
const [user, notifications] = await Promise.all([
  fetchUser(),
  fetchNotifications(),
]);
```

This is usually the clearest approach.

# 9. Dependency Analysis

Before choosing an execution strategy, ask whether each task depends on another.

Consider this workflow:

```text
1. Authenticate user
2. Fetch user profile
3. Fetch public services
4. Fetch user transactions
5. Fetch exchange rates
6. Calculate transaction totals
7. Render dashboard
```

Dependencies:

```text
Authenticate user
        ↓
Fetch user profile
        ↓
Fetch user transactions
        ↓
Calculate transaction totals
```

Independent tasks:

```text
Fetch public services
Fetch exchange rates
```

A dependency graph may look like this:

```text
Authenticate
     ↓
Profile
     ↓
Transactions
     ↓
Totals
     ┐
     │
Services ─────┤
Exchange rates┤
              ↓
       Render dashboard
```

A well-designed workflow can mix sequential and concurrent execution.

# 10. Using `Promise.all()` with `await`

`Promise.all()` accepts an iterable of values and Promises and returns one Promise.

The returned Promise:

* Fulfills when all input Promises fulfill.
* Rejects when the first input Promise rejects.
* Produces results in input order.

## Syntax

```javascript
const results = await Promise.all([
  promiseOne,
  promiseTwo,
  promiseThree,
]);
```

## Destructuring results

```javascript
const [user, orders, services] = await Promise.all([
  fetchUser(),
  fetchOrders(),
  fetchServices(),
]);
```

The result order matches the input order even if completion order differs.

```text
Input position 0 → user result
Input position 1 → orders result
Input position 2 → services result
```

## Internal workflow

```text
Create input Promises
        ↓
Promise.all() subscribes to each
        ↓
All operations continue
        ↓
Store each fulfilled result by input index
        ↓
All fulfilled?
        ↓
Resolve combined Promise with array
```

## Failure workflow

```text
Any input rejects
        ↓
Combined Promise rejects
        ↓
await throws
        ↓
Nearest catch handles error
```

# 11. Starting Promises Before Awaiting Them

Sometimes `Promise.all()` is not necessary, but tasks should still begin concurrently.

```javascript
async function loadData() {
  const userPromise = fetchUser();
  const settingsPromise = fetchSettings();

  const user = await userPromise;

  const preferencesPromise = fetchPreferences(user.id);

  const settings = await settingsPromise;
  const preferences = await preferencesPromise;

  return {
    user,
    settings,
    preferences,
  };
}
```

Here:

* `fetchUser()` and `fetchSettings()` start together.
* `fetchPreferences()` must wait for `user.id`.
* `settingsPromise` may finish while the user request is still running.
* The final function waits for all required results.

Timeline:

```text
fetchUser() ───────────────┐
fetchSettings() ───────┐   │
                       │   ▼ user available
                       │ fetchPreferences() ─────┐
                       ▼ settings available       │
                                                  ▼
                                            return results
```

This technique is useful when the dependency graph is more complex than one `Promise.all()` call can clearly express.

# 12. Mixed Sequential and Concurrent Workflows

Real applications often require a mixture.

## Example

```javascript
async function loadDashboard() {
  const session = await authenticateUser();

  const [
    profile,
    transactions,
    services,
    notifications,
  ] = await Promise.all([
    fetchProfile(session.userId),
    fetchTransactions(session.userId),
    fetchServices(),
    fetchNotifications(session.userId),
  ]);

  const totals = calculateTotals(transactions);

  return {
    profile,
    transactions,
    services,
    notifications,
    totals,
  };
}
```

Execution:

```text
Authenticate user
        ↓
Receive user ID
        ↓
┌─────────────────────────────────────┐
│ Fetch profile                       │
│ Fetch transactions                  │
│ Fetch services                      │
│ Fetch notifications                 │
└─────────────────────────────────────┘
        ↓
Wait for all
        ↓
Calculate totals synchronously
        ↓
Return dashboard data
```

This is often the correct production pattern:

1. Perform required prerequisite work.
2. Start all newly independent operations together.
3. Wait for the group.
4. Process the combined result.

# 13. Error Behaviour in Concurrent Operations

Consider:

```javascript
const [user, orders, services] = await Promise.all([
  fetchUser(),
  fetchOrders(),
  fetchServices(),
]);
```

If `fetchOrders()` rejects, the `Promise.all()` Promise rejects.

```javascript
async function loadDashboard() {
  try {
    const [user, orders, services] = await Promise.all([
      fetchUser(),
      fetchOrders(),
      fetchServices(),
    ]);

    return {
      user,
      orders,
      services,
    };
  } catch (error) {
    console.error("Dashboard loading failed:", error);
    throw error;
  }
}
```

## Important detail

`Promise.all()` rejecting does not necessarily cancel the other operations.

```text
Request A → still running
Request B → rejected
Request C → still running
Promise.all() → rejected
```

Promises do not provide automatic cancellation.

Network requests may require `AbortController` for cancellation, which will be taught later.

> [!WARNING]
> `Promise.all()` is fail-fast with respect to its returned Promise. It does not automatically stop the remaining underlying operations.

# 14. Partial Success with `Promise.allSettled()`

Sometimes the application should preserve successful results even when some tasks fail.

```javascript
const results = await Promise.allSettled([
  fetchProfile(),
  fetchNotifications(),
  fetchWeather(),
]);
```

Possible result:

```javascript
[
  {
    status: "fulfilled",
    value: {
      name: "Amina",
    },
  },
  {
    status: "rejected",
    reason: new Error("Notification service unavailable"),
  },
  {
    status: "fulfilled",
    value: {
      temperature: 24,
    },
  },
]
```

## Processing results

```javascript
function separateSettledResults(results) {
  const fulfilled = [];
  const rejected = [];

  for (const result of results) {
    if (result.status === "fulfilled") {
      fulfilled.push(result.value);
    } else {
      rejected.push(result.reason);
    }
  }

  return {
    fulfilled,
    rejected,
  };
}
```

## Appropriate uses

Use `Promise.allSettled()` when:

* Dashboard widgets are independent.
* Background synchronization should continue after partial failure.
* Multiple files are uploaded and every outcome must be reported.
* Analytics operations should not break user-facing features.
* A batch process needs a complete success/failure report.

# 15. First-Result Patterns

## 15.1 `Promise.race()`

`Promise.race()` settles when the first input settles.

```javascript
const result = await Promise.race([
  fetchFromPrimaryServer(),
  createTimeoutPromise(5000),
]);
```

It is useful for:

* Timeout wrappers.
* Competing data sources.
* Whichever operation completes first.
* Cancellation-like coordination patterns.

However, the first rejection also wins.

## 15.2 `Promise.any()`

`Promise.any()` fulfills when the first input fulfills.

```javascript
const data = await Promise.any([
  fetchFromServerA(),
  fetchFromServerB(),
  fetchFromServerC(),
]);
```

Rejected inputs are ignored unless all inputs reject.

Appropriate uses include:

* Redundant servers.
* Mirror endpoints.
* Multiple cached data sources.
* First successful geolocation provider.
* Resilient fallback services.

# 16. Controlling Concurrency

Starting thousands of operations simultaneously can be harmful.

Example:

```javascript
await Promise.all(
  files.map(file => uploadFile(file))
);
```

If `files` contains 10,000 items, the application may attempt 10,000 uploads immediately.

Possible consequences:

* Excessive memory usage.
* Network congestion.
* Browser instability.
* Server overload.
* API rate-limit failures.
* Database connection exhaustion.
* Device battery drain.

## 16.1 Batching

One simple strategy is processing small batches.

```javascript
async function processInBatches(items, batchSize, processItem) {
  const results = [];

  for (let index = 0; index < items.length; index += batchSize) {
    const batch = items.slice(index, index + batchSize);

    const batchResults = await Promise.all(
      batch.map(item => processItem(item))
    );

    results.push(...batchResults);
  }

  return results;
}
```

Usage:

```javascript
const uploadedFiles = await processInBatches(
  files,
  3,
  uploadFile
);
```

Execution:

```text
Batch 1: files 1–3 concurrently
        ↓ wait
Batch 2: files 4–6 concurrently
        ↓ wait
Batch 3: remaining files
```

## 16.2 Worker-style concurrency pool

A more flexible approach uses a fixed number of workers.

```javascript
async function mapWithConcurrency(
  items,
  concurrencyLimit,
  asyncMapper
) {
  if (!Number.isInteger(concurrencyLimit) || concurrencyLimit < 1) {
    throw new RangeError(
      "concurrencyLimit must be a positive integer."
    );
  }

  const results = new Array(items.length);
  let nextIndex = 0;

  async function worker() {
    while (true) {
      const currentIndex = nextIndex;
      nextIndex += 1;

      if (currentIndex >= items.length) {
        return;
      }

      results[currentIndex] = await asyncMapper(
        items[currentIndex],
        currentIndex
      );
    }
  }

  const workerCount = Math.min(
    concurrencyLimit,
    items.length
  );

  const workers = Array.from(
    {
      length: workerCount,
    },
    () => worker()
  );

  await Promise.all(workers);

  return results;
}
```

Usage:

```javascript
const results = await mapWithConcurrency(
  files,
  3,
  uploadFile
);
```

At most three uploads remain active at once.

> [!NOTE]
> Concurrency-control helpers become especially important in production systems that process files, API requests, transactions, emails, or database operations.

# 17. Three Progressive Examples

## Example 1: Beginner — Independent Timers

### Goal

Compare sequential and concurrent waiting.

### Utility function

```javascript
function delay(milliseconds, label) {
  return new Promise(resolve => {
    setTimeout(() => {
      resolve(label);
    }, milliseconds);
  });
}
```

## Sequential version

```javascript
async function runSequentialDelays() {
  const first = await delay(1000, "First");
  const second = await delay(1000, "Second");
  const third = await delay(1000, "Third");

  return [
    first,
    second,
    third,
  ];
}

runSequentialDelays().then(console.log);
```

Output after approximately three seconds:

```text
["First", "Second", "Third"]
```

### Why it takes approximately three seconds

```text
First delay:  1 second
Second delay: 1 second
Third delay:  1 second

Total: approximately 3 seconds
```

Each delay begins after the previous delay finishes.

## Concurrent version

```javascript
async function runConcurrentDelays() {
  const results = await Promise.all([
    delay(1000, "First"),
    delay(1000, "Second"),
    delay(1000, "Third"),
  ]);

  return results;
}

runConcurrentDelays().then(console.log);
```

Output after approximately one second:

```text
["First", "Second", "Third"]
```

### Why it takes approximately one second

All timers start together.

```text
First delay  ───────────┐
Second delay ───────────┼──→ approximately 1 second
Third delay  ───────────┘
```

### Professional lesson

Operations with no dependencies should not be forced into a sequential workflow.

## Example 2: Intermediate — Dashboard Data

### Goal

Load independent dashboard resources efficiently.

```javascript
function fetchProfile() {
  return delay(1200, {
    name: "Amina",
    role: "Admin",
  });
}

function fetchTransactions() {
  return delay(1800, [
    {
      id: 1,
      amount: 500,
    },
    {
      id: 2,
      amount: 750,
    },
  ]);
}

function fetchServices() {
  return delay(900, [
    "Printing",
    "Binding",
    "Photocopying",
  ]);
}
```

### Inefficient sequential version

```javascript
async function loadDashboardSequentially() {
  const profile = await fetchProfile();
  const transactions = await fetchTransactions();
  const services = await fetchServices();

  return {
    profile,
    transactions,
    services,
  };
}
```

Approximate total:

```text
1.2 + 1.8 + 0.9 = 3.9 seconds
```

### Efficient concurrent version

```javascript
async function loadDashboardConcurrently() {
  const [
    profile,
    transactions,
    services,
  ] = await Promise.all([
    fetchProfile(),
    fetchTransactions(),
    fetchServices(),
```

## Example 3: Real-World Application — POS Startup Workflow

### Requirements

A POS application must:

1. Authenticate the user.
2. Use the returned user ID to load profile and transactions.
3. Load public services independently.
4. Load printer status independently.
5. Calculate totals after transactions arrive.
6. Continue when every required startup resource is available.

### Supporting functions

```javascript
function authenticateUser() {
  return delay(800, {
    userId: 42,
    accessToken: "token-example",
  });
}

function fetchProfile(userId) {
  return delay(900, {
    id: userId,
    firstName: "Amina",
    role: "pos",
  });
}

function fetchTransactions(userId) {
  return delay(1400, [
    {
      id: 1,
      userId,
      amount: 500,
      isPaid: true,
    },
    {
      id: 2,
      userId,
      amount: 300,
      isPaid: false,
    },
  ]);
}

function fetchServices() {
  return delay(700, [
    {
      id: 1,
      name: "Printing",
      chargeAmount: 20,
    },
    {
      id: 2,
      name: "Binding",
      chargeAmount: 100,
    },
  ]);
}

function checkPrinterStatus() {
  return delay(600, {
    connected: true,
    hasPaper: true,
  });
}

  let total = 0;

  for (const transaction of transactions) {
    total += transaction.amount;
  }

  return {
    transactionCount: transactions.length,
```

### Optimized workflow

```javascript
async function initializePOS() {
  const servicesPromise = fetchServices();
  const printerStatusPromise = checkPrinterStatus();

  const session = await authenticateUser();

  const [
    profile,
    transactions,
    services,
    printerStatus,
  ] = await Promise.all([
    fetchProfile(session.userId),
    fetchTransactions(session.userId),
    servicesPromise,
    printerStatusPromise,
  ]);

  const totals = calculateTotals(transactions);

  return {
    session,
    profile,
    transactions,
    services,
    printerStatus,
    totals,
  };
}
```

### Why this design is efficient

`fetchServices()` and `checkPrinterStatus()` do not require authentication, so they begin immediately.

Authentication runs concurrently with those operations.

After authentication succeeds:

* Profile loading begins.
* Transaction loading begins.
* Existing service and printer Promises continue or may already be fulfilled.

```text
                                  ↓

                                  ↓

```

# 18. Common Mistakes

## Mistake 1: Sequentially awaiting independent operations

Inefficient:

```javascript
const profile = await fetchProfile();
const services = await fetchServices();
const notifications = await fetchNotifications();
```

Better:

```javascript
const [
  profile,
  services,
  notifications,
] = await Promise.all([
  fetchProfile(),
  fetchServices(),
  fetchNotifications(),
]);
```

---

## Mistake 2: Parallelizing dependent operations

Incorrect:

```javascript
const [user, orders] = await Promise.all([
  fetchUser(),
  fetchOrders(user.id),
]);
```

`user` does not exist when the array is created.

Correct:

```javascript
const user = await fetchUser();
const orders = await fetchOrders(user.id);
```

Or mix sequential and concurrent execution when additional independent work exists.

---

## Mistake 3: Assuming `Promise.all()` cancels other tasks

```javascript
await Promise.all([
  requestA(),
  requestB(),
  requestC(),
]);
```

If `requestB()` rejects, `requestA()` and `requestC()` may continue.

Use explicit cancellation mechanisms where supported.

---

## Mistake 4: Starting too many operations

Risky:

```javascript
await Promise.all(
  tenThousandFiles.map(uploadFile)
);
```

Prefer batching or a concurrency pool.

---

## Mistake 5: Using `Promise.all()` when partial results are useful

If one dashboard widget can fail without breaking the entire dashboard, `Promise.allSettled()` may be more appropriate.

---

## Mistake 6: Creating Promises too late

Sequential:

```javascript
const first = await taskOne();
const second = await taskTwo();
```

Concurrent:

```javascript
const firstPromise = taskOne();
const secondPromise = taskTwo();

const first = await firstPromise;
const second = await secondPromise;
```

The start time matters.

---

## Mistake 7: Confusing concurrency with CPU parallelism

`Promise.all()` does not make CPU-heavy JavaScript calculations run simultaneously on the main thread.

```javascript
await Promise.all([
  heavyCalculationA(),
  heavyCalculationB(),
]);
```

If both functions perform long synchronous calculations before returning, they still block the main thread.

CPU-intensive work may require Web Workers or worker threads.

---

# 19. Best Practices

## 19.1 Identify dependencies first

Before writing code, create a simple dependency graph.

```text
A → B → D
C ─────→ D
```

This immediately reveals which operations can overlap.

## 19.2 Start independent work early

```javascript
const settingsPromise = fetchSettings();

const user = await fetchUser();

const [settings, orders] = await Promise.all([
  settingsPromise,
  fetchOrders(user.id),
]);
```

## 19.3 Use `Promise.all()` for required independent results

It communicates intent clearly.

```javascript
const [profile, orders] = await Promise.all([
  fetchProfile(),
  fetchOrders(),
]);
```

## 19.4 Use `Promise.allSettled()` for independent optional results

```javascript
const widgetResults = await Promise.allSettled([
  loadWeatherWidget(),
  loadNewsWidget(),
  loadAnalyticsWidget(),
]);
```

## 19.5 Limit concurrency for large collections

Do not create unbounded network, file, or database activity.

## 19.6 Preserve result order deliberately

`Promise.all()` preserves input order. Use that fact with destructuring.

## 19.7 Handle errors at the appropriate level

A low-level function may add context and rethrow. A high-level workflow may display a user message or activate fallback behavior.

## 19.8 Measure before optimizing

Use browser tools and timing APIs instead of relying only on intuition.

# 20. Performance Considerations

## 20.1 Total duration

For sequential tasks:

```text
Total ≈ A + B + C
```

For concurrent independent tasks:

```text
Total ≈ max(A, B, C)
```

This is a simplified model. Real performance also depends on:

* Connection setup.
* Browser connection limits.
* Server capacity.
* Network bandwidth.
* API throttling.
* CPU processing.
* Serialization and parsing.
* Queueing.
* Device resources.

## 20.2 Network contention

Launching many requests together can cause them to compete for bandwidth.

More concurrency does not always mean better performance.

## 20.3 Server load

A browser may tolerate many requests, but the server may enforce:

* Rate limits.
* Connection limits.
* Request quotas.
* Database pool limits.

## 20.4 Memory usage

Every active operation may retain:

* Request data.
* Response buffers.
* closures.
* progress state.
* error metadata.

Thousands of concurrent operations may consume substantial memory.

## 20.5 User experience

Prioritize work that affects the visible interface.

Critical data may load first, while optional features load later.

```text
Critical:
- User session
- Main transactions
- Core services

Optional:
- Analytics
- Recommendations
- Background backup
```

## 20.6 CPU-heavy functions

Promises do not move synchronous CPU work off the main thread.

```javascript
async function runHeavyWork() {
  const result = await heavyCalculation();
  return result;
}
```

If `heavyCalculation()` blocks for five seconds before returning, the page still freezes.

Use worker technologies for CPU-intensive workloads.

## 21.5 Log task start and finish times

```javascript
async function traceOperation(name, operation) {
  const startedAt = performance.now();

  console.log(`${name} started`);

  try {
    const result = await operation();

    const duration = performance.now() - startedAt;

    console.log(
      `${name} finished in ${duration.toFixed(2)} ms`
    );

    return result;
  } catch (error) {
    const duration = performance.now() - startedAt;

    console.error(
      `${name} failed after ${duration.toFixed(2)} ms`,
      error
    );

    throw error;
  }
}
```

Do not assume concurrency is faster. Measure both implementations in realistic conditions.

# 22. Interview Questions and Answers

## Beginner Questions

### 1. What is sequential asynchronous execution?

Sequential asynchronous execution means one asynchronous operation begins only after the previous operation has finished.

### 2. What is concurrent asynchronous execution?

Concurrent asynchronous execution means multiple asynchronous operations are active during overlapping periods.

### 3. How can independent Promise-returning operations be started together?

Call them before awaiting, often using `Promise.all()`.

```javascript
const [a, b] = await Promise.all([
  getA(),
  getB(),
]);
```

### 4. Does `Promise.all()` preserve result order?

Yes. Results are returned in the same order as the input iterable.

### 5. What happens when one input Promise rejects in `Promise.all()`?

The combined Promise rejects with that rejection reason.

---

## Intermediate Questions

### 1. What is the difference between concurrency and parallelism?

Concurrency means tasks overlap in progress. Parallelism means tasks physically execute at the same time.

### 2. Why can consecutive `await` statements be slow?

Each awaited operation may prevent the next operation from starting until it finishes.

### 3. When should consecutive `await` statements be used?

When later operations depend on earlier results or strict ordering is required.

### 4. Does `Promise.all()` cancel other operations after one rejection?

No. It rejects its combined Promise, but other underlying operations may continue.

### 5. When is `Promise.allSettled()` more appropriate than `Promise.all()`?

When every result matters and partial failure is acceptable.

### 6. Why might unlimited concurrency be dangerous?

It can exhaust memory, bandwidth, connections, API quotas, database resources, or server capacity.

### 7. How can concurrency be limited?

Use batching, a worker pool, a semaphore, or a dedicated concurrency-control utility.

---

## Advanced Questions

### 1. Does `Promise.all()` make synchronous CPU operations run in parallel?

No. Synchronous CPU-heavy work still executes on the main JavaScript thread unless moved to workers.

### 2. Explain the difference between starting Promises concurrently and awaiting them concurrently.

Promises begin when the Promise-producing operation is called. `await` only waits for their completion. Calling several functions before awaiting them allows their operations to overlap.

### 3. How would you optimize a workflow with both dependent and independent requests?

Create a dependency graph, start independent operations as early as possible, await prerequisite results, then group newly independent work with `Promise.all()`.

### 4. Why may very high concurrency reduce performance?

Tasks may compete for CPU, bandwidth, memory, browser connection slots, database connections, or server capacity.

### 5. What is head-of-line blocking in a sequential async workflow?

It occurs when later independent operations cannot start because an unrelated earlier operation is being awaited.

### 6. How would you preserve all upload outcomes in a batch?

Use `Promise.allSettled()` or catch errors per individual operation and return structured status objects.

# 26. Summary

In this lesson, you learned that asynchronous JavaScript can use different execution strategies.

Sequential execution:

```text
Task A
   ↓
Task B
   ↓
Task C
```

is appropriate when tasks depend on one another or order is required.

Concurrent execution:

```text
Task A ────┐
Task B ────┤
Task C ────┘
```

is appropriate when tasks are independent and can safely overlap.

You learned that task start time is determined by when the Promise-producing function is called—not when the result is awaited.

You learned to use:

* `Promise.all()` when every independent operation must succeed.
* `Promise.allSettled()` when partial success is acceptable.
* `Promise.race()` when the first settled result matters.
* `Promise.any()` when the first successful result matters.

You also learned that:

* `Promise.all()` does not automatically cancel remaining operations.
* Unlimited concurrency can harm performance and reliability.
* Large workloads should use batching or concurrency limits.
* CPU-heavy synchronous code does not become parallel merely because it is placed inside `Promise.all()`.
* Real applications often combine sequential and concurrent stages.

The central professional skill is not using `Promise.all()` everywhere.

It is correctly identifying dependencies and starting each operation as early as safely possible.

```text
Async Iterators
        ↓
DOM and Browser Applications
```

Understanding this connection is essential because real applications frequently process:

* Transactions.
* Files.
* API resources.
* Form submissions.
* Uploaded images.
* Database records.
* Background synchronization jobs.
