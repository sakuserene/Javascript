# Why This Lesson Matters

Let's look at some code.

```javascript
console.log("A");

setTimeout(() => {
    console.log("B");
}, 0);

console.log("C");
```

Many beginners expect

```text
A

B

C
```

Actual output

```text
A

C

B
```

Why?

The timer was **zero milliseconds**.

Shouldn't it execute immediately?

No.

Understanding why requires learning the **JavaScript Runtime**.

# Component 1 вҖ” JavaScript Engine

The engine executes JavaScript code.

Popular engines include:

| Browser | Engine |
|----------|---------|
| Chrome | V8 |
| Edge | V8 |
| Firefox | SpiderMonkey |
| Safari | JavaScriptCore |

The engine understands JavaScript syntax.

It knows nothing about:

- Timers
- Network requests
- HTML
- CSS
- Mouse clicks

Those belong elsewhere.

# Component 2 вҖ” The Call Stack

The Call Stack keeps track of functions currently executing.

Imagine stacking books.

```
Top

вҶ“

Book 3

Book 2

Book 1
```

Last placed,

first removed.

This is called **LIFO**

Last In,

First Out.

---

Example

```javascript
function one() {

    two();

}

function two() {

    three();

}

function three() {

    console.log("Done");

}

one();
```

Call Stack

```
Global

вҶ“

one()

вҶ“

two()

вҶ“

three()

вҶ“

console.log()
```

Then everything returns in reverse order.

# Component 4 вҖ” Callback Queue

When asynchronous work finishes,

the callback **does not immediately execute**.

Instead,

it waits inside the Callback Queue.

Think of it as people waiting outside a meeting room.

```
Callback Queue

вҶ“

Task 1

вҶ“

Task 2

вҶ“

Task 3
```

They wait patiently.

---

# Why Can't They Execute Immediately?

Suppose the Call Stack looks like this.

```
Global

вҶ“

calculateInvoice()

вҶ“

calculateTax()

вҶ“

printReceipt()
```

Imagine another callback suddenly interrupted everything.

That would corrupt execution.

Instead,

callbacks wait.

Only when the Call Stack becomes empty can another callback begin.

---

# Component 5 вҖ” Event Loop

The Event Loop is extremely simple.

Its job is basically:

```
Repeat Forever

вҶ“

Is Call Stack Empty?

вҶ“

YES

вҶ“

Move First Callback

вҶ“

From Queue

вҶ“

To Stack
```

That's it.

The Event Loop doesn't execute your code.

The JavaScript Engine does.

The Event Loop simply moves callbacks when it is safe.

# Why Doesn't `setTimeout(..., 0)` Execute Immediately?

This is a classic interview question.

Example

```javascript
console.log("A");

setTimeout(() => {

    console.log("B");

}, 0);

console.log("C");
```

Output

```text
A

C

B
```

Even though

```
0 milliseconds
```

elapsed,

the callback still entered the Callback Queue.

It still waited for:

```
Call Stack

вҶ“

Become Empty
```

Zero milliseconds means:

> **"Do not wait before placing the callback into the queue."**

It does **not** mean:

> **"Execute immediately."**

That distinction is critical.

# Common Mistakes

## Mistake 1

Thinking the Event Loop executes callbacks.

It doesn't.

The JavaScript Engine executes them.

The Event Loop only moves callbacks to the Call Stack.

---

## Mistake 2

Believing Browser APIs are part of JavaScript.

They are provided by the browser (or by Node.js in a server environment).

---

## Mistake 3

Thinking `setTimeout(..., 0)` skips the Callback Queue.

It never does.

---

## Mistake 4

Thinking timers interrupt currently running JavaScript.

They don't.

Callbacks must wait until the Call Stack is empty.

---

# Best Practices

- Never assume timer callbacks execute at the exact specified time.
- Remember that delays are **minimum delays**, not guarantees.
- Keep the Call Stack free of long-running synchronous work.
- Understand the runtime model before debugging asynchronous code.

---

# Performance Considerations

Long-running synchronous code blocks the Call Stack.

While the stack is busy:

- Timers cannot execute.
- User events cannot be processed.
- Network callbacks cannot run.
- The page may become unresponsive.

Keeping functions small and efficient helps the Event Loop process queued work promptly.

# Interview Questions

### Beginner

**Q:** What is the Callback Queue?

**Answer:**

A queue where completed asynchronous callbacks wait until the Call Stack becomes empty.

---

### Intermediate

**Q:** What is the Event Loop?

**Answer:**

A runtime mechanism that continuously checks whether the Call Stack is empty and, if so, moves the next callback from the Callback Queue onto the Call Stack.

---

### Advanced

**Q:** Why does `setTimeout(callback, 0)` execute after synchronous code?

**Answer:**

Because the callback must first be placed in the Callback Queue, and the Event Loop will only move it to the Call Stack after the current synchronous execution has completely finished.
