# Introduction

In the previous lesson we studied **synchronous callbacks**.

Those callbacks execute immediately.

Example

```javascript
run(callback);
```

Execution

```
run()

↓

callback()

↓

Continue
```

Everything happens immediately.

No waiting.

No delays.

No scheduling.

Now we're going to study something completely different.

Instead of executing immediately,

the callback will execute **later**.

This changes everything.

---

# What Does Asynchronous Mean?

The word

> **Asynchronous**

means

> **Tasks do not have to finish in the order they started.**

Some tasks may continue later.

Some may wait.

Some may complete while other work is happening.

---

# Real-Life Example

Imagine ordering food.

You enter a restaurant.

You order food.

Should you stand in front of the chef doing absolutely nothing until your food is ready?

That would be extremely inefficient.

Instead,

you

```
Order Food

↓

Sit Down

↓

Talk

↓

Read

↓

Use Phone

↓

Waiter Calls You

↓

Food Ready
```

Notice

You continued doing useful work while the food was being prepared.

That is asynchronous programming.

---

# Why Was Asynchronous Programming Invented?

Imagine JavaScript without asynchronous programming.

Suppose downloading a user's profile takes

```
5 seconds
```

Without asynchronous programming,

your program would freeze.

Example

```
Download User

↓

Wait

↓

Wait

↓

Wait

↓

Wait

↓

Wait

↓

Continue
```

The user couldn't

- click buttons
- scroll
- type
- interact

The browser would appear frozen.

Clearly unacceptable.

---

# Blocking Code

Consider

```javascript
function longTask() {

    // imagine 5 seconds of work

}

console.log("Start");

longTask();

console.log("End");
```

Execution

```
Start

↓

Long Task

↓

Wait

↓

End
```

Nothing else can happen until

```
longTask()
```

finishes.

This is called **blocking**.

---

# Non-Blocking Code

Now imagine

```
Start

↓

Schedule Task

↓

Continue Running

↓

Later

↓

Task Finishes

↓

Execute Callback
```

Notice

The program keeps working.

Only the callback waits.

This is called

> **non-blocking programming.**

---

# Why JavaScript Needed This

Remember

JavaScript has

```
ONE THREAD
```

Only one.

If every slow operation blocked that thread,

web pages would constantly freeze.

Instead,

JavaScript delegates slow work elsewhere.

---

# The Biggest Beginner Misconception

Many beginners believe

```
JavaScript

↓

Waits

↓

Runs Timer

↓

Returns
```

This is completely incorrect.

JavaScript itself does **not** contain timers.

JavaScript itself does **not** download web pages.

JavaScript itself does **not** read files.

JavaScript itself does **not** perform HTTP requests.

So who does?

```

↓

↓

↓

End
```

This is **wrong**.

Actual output

```text
Start

End

Hello
```

Why?

Let's investigate carefully.

# Why Is This Called an Asynchronous Callback?

Because the callback

does **not**

execute immediately.

Instead,

it waits for another system

(the browser)

to tell JavaScript

when it should execute.

---

# Example 1 — Beginner

```javascript
console.log("One");

setTimeout(function () {

    console.log("Two");

}, 1000);

console.log("Three");
```

Output

```text
One

Three

Two
```

---

# Example 2 — Intermediate

```javascript
function notify() {

    console.log("Download Finished");

}

console.log("Downloading...");

setTimeout(notify, 2000);

console.log("Please Wait...");
```

Output

```text
Downloading...

Please Wait...

Download Finished
```

Notice

The program continued.

---

# Example 3 — Real-World Application

Imagine a POS application.

```javascript
console.log("Uploading transaction...");

setTimeout(function () {

    console.log("Upload Complete");

}, 5000);

console.log("User may continue working...");
```

Output

```text
Uploading transaction...

User may continue working...

Upload Complete
```

Professional applications behave exactly like this.

Users continue interacting while background work happens.

# Common Mistakes

## Mistake 1

Believing JavaScript waits.

It doesn't.

The browser waits.

---

## Mistake 2

Thinking

```javascript
setTimeout()
```

belongs to JavaScript.

It belongs to the browser environment.

(Node.js provides its own timer APIs independently.)

---

## Mistake 3

Thinking

```
Delay

↓

Immediate Execution
```

No.

The callback executes only after:

- the timer finishes, **and**
- JavaScript is free to run it.

We'll study why in the next lesson.

# Best Practices

- Never block the JavaScript thread with long-running synchronous work if it can be avoided.
- Use asynchronous callbacks for operations that naturally take time, such as timers, network requests, and user interactions.
- Remember that scheduling a callback does not pause the rest of your code.
- Keep asynchronous callbacks focused and easy to understand.

---

# Performance Considerations

Asynchronous callbacks improve responsiveness because they allow JavaScript to continue executing other work instead of waiting for slow operations.

However, scheduling many asynchronous callbacks can make execution order harder to reason about, which is one reason Promises and `async`/`await` were later introduced.

# Interview Questions

### Beginner

**Q:** What is an asynchronous callback?

**Answer:**

A callback that executes later rather than immediately, usually after another system (such as the browser) finishes some work.

---

### Intermediate

**Q:** Does JavaScript wait during `setTimeout()`?

**Answer:**

No. JavaScript asks the browser to start the timer and immediately continues executing the remaining code.

---

### Advanced

**Q:** Is `setTimeout()` part of the JavaScript language?

**Answer:**

No. In browsers, `setTimeout()` is provided by the browser's Web APIs. In Node.js, it is provided by the Node.js runtime.
