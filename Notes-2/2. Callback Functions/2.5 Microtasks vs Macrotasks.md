# Review of the Previous Lesson

Previously, we simplified the JavaScript runtime as:

```text
JavaScript Engine

↓

Call Stack

↓

Browser APIs

↓

Callback Queue

↓

Event Loop
```

While this model is useful for beginners, it is not the whole story.

Professional JavaScript engines actually manage **multiple queues**.

---

# Why One Queue Was Not Enough

Imagine this code.

```javascript
console.log("Start");

setTimeout(() => {
    console.log("Timer");
}, 0);

Promise.resolve().then(() => {
    console.log("Promise");
});

console.log("End");
```

What will the output be?

Many beginners guess:

```text
Start

End

Timer

Promise
```

Others guess:

```text
Start

Promise

End

Timer
```

Both are incorrect.

The actual output is:

```text
Start

End

Promise

Timer
```

Why?

Because **not all asynchronous tasks have equal priority**.

---

# JavaScript Classifies Asynchronous Work

Instead of one queue, JavaScript distinguishes between:

```text
Asynchronous Tasks

│

├── Microtasks

└── Macrotasks
```

Microtasks always have higher priority.

---

# What Is a Task?

A **task** is simply a unit of work waiting to be executed.

Examples include:

- Running a callback
- Handling a mouse click
- Processing a timer
- Resolving a Promise

Tasks wait until JavaScript is ready to execute them.

---

# What Is a Macrotask?

Macrotasks are the "normal" asynchronous tasks.

Examples:

```text
setTimeout()

setInterval()

DOM Events

Mouse Clicks

Keyboard Events

Network Events

MessageChannel

postMessage()
```

These tasks are placed into the **Task Queue**, often called the **Macrotask Queue**.

---

# What Is a Microtask?

Microtasks are higher-priority tasks.

Examples include:

```text
Promise.then()

Promise.catch()

Promise.finally()

queueMicrotask()

MutationObserver
```

These tasks enter the **Microtask Queue**.

---

# Why Were Microtasks Introduced?

Imagine this code.

```javascript
fetchUser()
    .then(updateUI)
    .then(showSuccessMessage);
```

If Promise callbacks waited behind every timer, click event, and keyboard event, user interfaces would become inconsistent.

Instead, JavaScript guarantees that Promise reactions happen **as soon as possible after the current synchronous code finishes**.

This makes Promise chains predictable.

---

# The JavaScript Runtime (More Accurate)

Our runtime model now becomes:

```text
                JavaScript Runtime

                        │
                        ▼

               ┌──────────────────┐
               │    Call Stack    │
               └────────┬─────────┘
                        │
                        ▼

        ┌─────────────────────────────────┐
        │          Event Loop             │
        └───────┬───────────────┬─────────┘
                │               │
                ▼               ▼

      Microtask Queue     Macrotask Queue
```

Notice that the Event Loop must choose between two queues.

---

# Priority Rule

The Event Loop follows one extremely important rule:

> **Always empty the Microtask Queue before taking the next Macrotask.**

This rule explains nearly every interview question involving Promises.

---

# Event Loop Algorithm

Conceptually, the Event Loop behaves like this:

```text
Program Starts

↓

Execute Synchronous Code

↓

Call Stack Empty?

↓

YES

↓

Are Microtasks Waiting?

↓

YES

↓

Execute ALL Microtasks

↓

Microtask Queue Empty?

↓

YES

↓

Take ONE Macrotask

↓

Repeat
```

Notice something important.

The Event Loop does **not** alternate:

```text
Microtask

↓

Macrotask

↓

Microtask

↓

Macrotask
```

Instead:

```text
ALL Microtasks

↓

ONE Macrotask

↓

ALL New Microtasks

↓

ONE Macrotask
```

This behavior is critical.

# Step-by-Step Example

```javascript
console.log("A");

setTimeout(() => {
    console.log("B");
}, 0);

Promise.resolve().then(() => {
    console.log("C");
});

console.log("D");
```

---

## Step 1

Output

```text
A
```

---

## Step 2

The timer callback is sent to:

```text
Macrotask Queue
```

---

## Step 3

The Promise callback is sent to:

```text
Microtask Queue
```

---

## Step 4

Output

```text
D
```

Now synchronous execution finishes.

---

Current state:

```text
Microtask Queue

↓

Promise Callback

-------------------

Macrotask Queue

↓

Timer Callback
```

---

## Step 5

The Event Loop checks:

```text
Are there Microtasks?
```

Yes.

Execute:

```text
C
```

---

## Step 6

Now the Microtask Queue is empty.

Take one Macrotask.

Execute:

```text
B
```

---

Final Output

```text
A

D

C

B
```

# Example 3 — Real-World Application

Imagine a POS application.

```javascript
console.log("Saving Transaction");

Promise.resolve().then(() => {
    console.log("Update Dashboard");
});

setTimeout(() => {
    console.log("Sync Cloud Backup");
}, 0);

console.log("Continue Using POS");
```

Output

```text
Saving Transaction

Continue Using POS

Update Dashboard

Sync Cloud Backup
```

Updating the dashboard (a Promise microtask) happens before the background cloud synchronization (a macrotask).

# Common Mistakes

## Mistake 1

Thinking every asynchronous callback has the same priority.

They do not.

---

## Mistake 2

Believing timers execute before Promises.

Promises always execute first if both are ready.

---

## Mistake 3

Thinking `setTimeout(..., 0)` means "execute immediately."

It means:

> "Place this callback into the Macrotask Queue without an intentional delay."

It still waits its turn.

---

## Mistake 4

Believing the Event Loop executes one microtask and then one macrotask.

Instead, it drains the entire Microtask Queue before executing the next Macrotask.

---

# Best Practices

- Use Promises for asynchronous workflows rather than deeply nested timers.
- Avoid creating endless chains of microtasks, as they can delay macrotasks such as user interactions.
- Understand task priorities when debugging unexpected execution order.
- Keep asynchronous code readable and predictable.

---

# Performance Considerations

Because microtasks always run before macrotasks, creating a very large number of microtasks can temporarily delay:

- User input
- Rendering updates
- Timer callbacks

Professional applications should avoid generating unbounded microtask chains.

# Interview Questions

### Beginner

**Q:** What is the difference between a microtask and a macrotask?

**Answer:**

Microtasks have higher priority and execute before macrotasks after the current synchronous code finishes.

---

### Intermediate

**Q:** Name two examples of microtasks.

**Answer:**

- `Promise.then()`
- `Promise.catch()`
- `Promise.finally()`
- `queueMicrotask()`

---

### Advanced

**Q:** Why does a resolved Promise execute before `setTimeout(..., 0)`?

**Answer:**

Because Promise callbacks enter the Microtask Queue, which the Event Loop completely drains before taking the next callback from the Macrotask Queue.
