# Function Hoisting

One of the most surprising behaviors in JavaScript is that **some code appears to work before it has been written.**

Consider this example.

```javascript
sayHello();

function sayHello() {
    console.log("Hello");
}
```

Output

```text
Hello
```

Most beginners expect an error because:

```
Call Function

↓

Declare Function
```

appears to happen in that order.

Yet JavaScript executes it successfully.

Why?

To answer this, we must understand one of JavaScript's most misunderstood concepts:

> **Hoisting**

---

# What is Hoisting?

Many books define hoisting like this:

> "JavaScript moves declarations to the top."

Although this explanation is useful for beginners,

it is **not technically correct.**

JavaScript does **not** physically move your code.

Instead,

during parsing,

the JavaScript engine performs a preparation step before execution begins.

Nothing is rewritten.

Nothing changes position.

The engine simply creates identifiers in memory before executing your code.

---

# A Better Definition

> **Hoisting is the behavior where JavaScript creates bindings for declarations during the Creation Phase before executing any code.**

This definition matches what JavaScript engines actually do.

---

# Why Was Hoisting Introduced?

Imagine JavaScript without hoisting.

Every function would have to appear before it is used.

Developers would constantly need to rearrange code.

Instead,

JavaScript allows you to organize related code more naturally.

Example

```javascript
startApplication();

function startApplication() {

    loadSettings();

    connectDatabase();

}
```

The "main" program appears first,

while implementation details appear later.

This often improves readability.

---

# JavaScript Execution Has Two Phases

Every execution context (global or function) goes through two phases.

```
JavaScript Execution

│

├───────────────┐
│               │
▼               ▼

Creation Phase

Execution Phase
```

Understanding these two phases explains hoisting completely.

---

# Phase 1 — Creation Phase

During this phase,

JavaScript **does not execute code.**

Instead, it prepares memory.

It performs tasks such as:

- Creating the execution context.
- Creating variable bindings.
- Creating function bindings.
- Setting up the scope chain.
- Determining the value of `this`.

Think of it as preparing a classroom before students arrive.

Nothing is being taught yet.

The room is simply being prepared.

---

# Phase 2 — Execution Phase

Only after preparation finishes does JavaScript begin executing code line by line.

```
Creation Complete

↓

Execute Line 1

↓

Execute Line 2

↓

Execute Line 3
```

---

# Example

```javascript
sayHello();

function sayHello() {

    console.log("Hello");

}
```

Let's walk through both phases.

---

## Creation Phase

JavaScript scans the entire scope.

It finds

```javascript
function sayHello() {

}
```

It creates a function object.

Conceptually

```text
Memory

↓

sayHello

↓

Function Object
```

Nothing has executed yet.

---

## Execution Phase

Execution begins at the first line.

```javascript
sayHello();
```

Since

```
sayHello
```

already exists in memory,

the function executes successfully.

Output

```text
Hello
```

---

# Visual Timeline

Source Code

```javascript
sayHello();

function sayHello() {

    console.log("Hello");

}
```

Timeline

```
Parser

↓

Creation Phase

↓

Function Created

↓

Execution Begins

↓

sayHello()

↓

Function Executes

↓

Hello
```

Notice:

The function existed **before** the first line executed.

# Hoisting Is Not Magic

Many beginners imagine this transformation:

```javascript
function greet() {

}

greet();
```

becoming

```javascript
function greet() {

}

greet();
```

or vice versa.

This never happens.

The file itself is unchanged.

Instead,

JavaScript's memory changes.

---

# Mental Model

Imagine a company preparing for a workday.

Before employees begin working,

Human Resources prepares:

- Employee records.
- Office assignments.
- Equipment.
- Access cards.

Only after preparation do employees start working.

JavaScript behaves similarly.

Creation Phase

```
Prepare Memory
```

Execution Phase

```
Run Code
```

---

# Function Declarations Are Fully Hoisted

This is the important rule.

Function declarations are hoisted **with their implementations**.

Example

```javascript
greet();

function greet() {

    console.log("Hello");

}
```

During the Creation Phase

```
greet

↓

Entire Function
```

is already available.

Not just the name.

The whole function object.

---

# Example 2 — Intermediate

```javascript
calculate();

    console.log(5 * 5);

```

Output


# Example 3 — Real-World Application

Imagine a Point of Sale application.

```javascript
startPOS();

function startPOS() {

    loadPrinter();

    authenticateUser();

    loadProducts();

}

function loadPrinter() {

    console.log("Printer Ready");

}

function authenticateUser() {

    console.log("User Authenticated");

}

function loadProducts() {

    console.log("Products Loaded");

}
```

Output

```text
Printer Ready

User Authenticated

Products Loaded
```

Notice something remarkable.

Every function is called before some of them appear in the source code.

This works because **all function declarations are created during the Creation Phase**.

# Common Mistakes

## Mistake 1 — Believing JavaScript Moves Code

Incorrect belief

```
JavaScript rewrites my file.
```

Reality

```
JavaScript prepares memory.

The source code remains unchanged.
```

---

## Mistake 2 — Confusing Hoisting with Execution

Creation

```
Create Function
```

Execution

```
Run Function
```

These are different steps.

---

## Mistake 3 — Assuming Everything Is Hoisted the Same Way

Functions,

`var`,

`let`,

`const`,

classes,

and imports all have different hoisting behaviors.

We'll study those differences shortly.

---

# Best Practices

Although function declarations can be called before they appear,

many teams still place important functions **before their first use** to improve readability.

Hoisting is a language feature—not a substitute for clear organization.

---

# Performance Considerations

The Creation Phase happens automatically before execution.

For ordinary applications, the cost is negligible.

Understanding hoisting is valuable primarily for:

- Debugging.
- Reading professional code.
- Understanding execution order.
- Technical interviews.

---

# Key Takeaways

- JavaScript execution has **two phases**:
  - Creation Phase
  - Execution Phase
- Function declarations are fully available during the Creation Phase.
- Hoisting does **not** move source code.
- Hoisting creates bindings in memory before execution starts.
- Function declarations are hoisted with their complete implementation.
