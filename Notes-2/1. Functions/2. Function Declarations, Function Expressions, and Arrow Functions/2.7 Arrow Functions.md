# Arrow Functions and `this`

> **This is one of the most important lessons in all of JavaScript.**

If there is one topic that has confused JavaScript developers for decades, it is:

```javascript
this
```

Entire books have been written about it.

Countless interview questions are based on it.

Many difficult bugs in JavaScript applications happen because developers misunderstand how `this` works.

Arrow functions were largely introduced to solve many of these problems.

Before we learn **how arrow functions handle `this`**, we must first understand **what `this` actually is.**

---

# What is `this`?

Most beginners think:

> "`this` refers to the current function."

This is **incorrect.**

Others think:

> "`this` always refers to the object."

Also incorrect.

The truth is more subtle.

### Correct Definition

> **`this` is a special keyword whose value is determined by *how a function is called*, not where it is written.**

That single sentence explains most of JavaScript's behavior.

---

# Why Was `this` Created?

Imagine JavaScript without `this`.

Suppose we have two users.

```javascript
const user1 = {
    name: "Alice"
};

const user2 = {
    name: "John"
};
```

We want both users to share the same greeting function.

Without `this`, we would have to write:

```javascript
function greet(user) {
    console.log(`Hello ${user.name}`);
}
```

Usage

```javascript
greet(user1);
greet(user2);
```

It works.

But every method would require passing its own object.

That quickly becomes repetitive.

JavaScript introduced `this` so the function could automatically know **which object invoked it**.

---

# Traditional Function Example

```javascript
const user = {

    name: "Alice",

    greet: function () {

        console.log(this.name);

    }

};

user.greet();
```

Output

```text
Alice
```

Why?

Because the function was called as:

```javascript
user.greet();
```

The object before the dot (`user`) becomes the value of `this`.

---

# Mental Model

Imagine the call:

```javascript
user.greet();
```

Conceptually, JavaScript sees:

```text
Object

↓

user

↓

Call Function

↓

Set

this = user

↓

Execute Function
```

Notice something important.

`this` is decided **when the function is called**, not when it is created.

---

# Internal Engine Behaviour

Suppose

```javascript
const user = {

    name: "Alice",

    greet() {

        console.log(this.name);

    }

};
```

Execution

```javascript
user.greet();
```

Conceptually

```text
Create Execution Context

↓

Determine Caller

↓

Caller = user

↓

this = user

↓

Execute Function

↓

this.name

↓

"Alice"
```

# Another Example

```javascript
const person = {

    name: "John",

    greet() {

        console.log(this.name);

    }

};

person.greet();
```

Output

```text
John
```

Nothing inside the function changed.

Only the caller changed.

Therefore

```javascript
this
```

changed.

# The Problem with Regular Functions

Now consider:

```javascript
const user = {

    name: "Alice",

    greet() {

        setTimeout(function () {

            console.log(this.name);

        }, 1000);

    }

};

user.greet();
```

Many beginners expect

```text
Alice
```

Instead,

depending on the environment, you may get:

```text
undefined
```

or

```text
Window
```

or another unexpected value.

Why?

---

# Understanding the Bug

Let's analyze carefully.

The outer function

```javascript
user.greet();
```

was called by

```javascript
user
```

So

```javascript
this === user
```

Correct.

However,

inside

```javascript
setTimeout(function () {

})
```

a **new regular function** is created.

Later,

`setTimeout()` executes that function independently.

Conceptually

```text
user.greet()

↓

this = user

----------------

Later

↓

setTimeout()

↓

Calls callback

↓

New this

↓

Not user
```

The callback has its own `this`.

It does **not** automatically inherit the outer one.

---

# The Old Solution

Before arrow functions,

developers often wrote:

```javascript
const user = {

    name: "Alice",

    greet() {

        const self = this;

        setTimeout(function () {

            console.log(self.name);

        }, 1000);

    }

};
```

Notice

```javascript
const self = this;
```

or sometimes

```javascript
const that = this;
```

Developers manually saved the outer `this`.

This pattern appeared in millions of JavaScript applications.

It worked,

but it wasn't elegant.

---

# Arrow Functions Solve This

Now consider:

```javascript
const user = {

    name: "Alice",

    greet() {

        setTimeout(() => {

            console.log(this.name);

        }, 1000);

    }

};

user.greet();
```

Output

```text
Alice
```

No `self`.

No `that`.

Why?

Because arrow functions **do not create their own `this`.**

---

# Lexical `this`

This is one of the defining characteristics of arrow functions.

> **Arrow functions inherit `this` from the surrounding scope.**

This is called **lexical `this`**.

The word **lexical** means:

> "Based on where the code is written."

Unlike regular functions,

arrow functions do **not** determine `this` when called.

Instead,

they look outward to the nearest enclosing non-arrow function.

---

# Visual Representation

Regular Function

```text
Call

↓

Create New this
```

Arrow Function

```text
Call

↓

No New this

↓

Look Outside

↓

Use Parent's this
```

---

# Internal Engine Behaviour

Consider

```javascript
const user = {

    name: "Alice",

    greet() {

        setTimeout(() => {

            console.log(this.name);

        });

    }

};
```

Execution

```text
Call greet()

↓

this = user

↓

Create Arrow Function

↓

Arrow Captures Parent this

↓

Later Executes

↓

Uses Captured this

↓

user.name

↓

Alice
```

Notice

the arrow function never creates another `this`.

# Example 3 — Real-World Application

Imagine a POS application.

```javascript
const printer = {

    deviceName: "POS Printer",

    printReceipt() {

        setTimeout(() => {
```

Output

```text
POS Printer Ready
```

Without arrow functions,

additional work would be required to preserve `this`.

# Common Mistakes

## Mistake 1 — Thinking Arrow Functions Have Their Own `this`

They don't.

They inherit it.

---

## Mistake 2 — Replacing Every Function with an Arrow Function

Arrow functions are **not** always the correct choice.

We'll soon see situations where regular functions are required.

---

## Mistake 3 — Forgetting That `this` Depends on Invocation

Never ask:

> "Where was this function written?"

Instead ask:

> "How was this function called?"

For regular functions, that determines `this`.

For arrow functions, ask:

> "What is the surrounding `this`?"

---

# Best Practices

- Use arrow functions for callbacks that should inherit the surrounding `this`.
- Use arrow functions with asynchronous APIs like:
  - `setTimeout()`
  - `setInterval()`
  - Promise callbacks
  - Event handlers (depending on the desired `this`)
- Avoid unnecessary `const self = this` patterns in modern JavaScript.
- Understand **why** arrow functions solve the problem before using them.

---

# Performance Considerations

Arrow functions avoid creating their own `this` binding.

The performance difference is generally negligible.

The real benefit is **correctness** and **readability**, not speed.

---

# Key Takeaways

- Regular functions determine `this` when they are **called**.
- Arrow functions **do not create** their own `this`.
- Arrow functions inherit `this` from the surrounding lexical scope.
- This behavior is called **lexical `this`**.
- Arrow functions greatly simplify callbacks and asynchronous code.
