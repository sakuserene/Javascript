# Function Expressions

In the previous lesson, we learned that **Function Declarations** are fully hoisted.

For example,

```javascript
greet();

function greet() {
    console.log("Hello");
}
```

works perfectly.

Now let's examine another way to create functions.

```javascript
const greet = function () {
    console.log("Hello");
};
```

At first glance, it seems almost identical.

It still creates a function.

It still prints `"Hello"`.

So why did JavaScript introduce another syntax?

The answer is one of the most important ideas in JavaScript:

> **Functions are values.**

---

# The Big Idea

Before understanding Function Expressions, you must first change the way you think about functions.

Most beginners think:

```
Functions are special pieces of code.
```

Professional JavaScript developers think:

```
Functions are objects.

Objects are values.

Therefore,

Functions are values.
```

This idea completely changes how you design programs.

---

# What Does "Functions Are Values" Mean?

Consider numbers.

```javascript
const age = 25;
```

The number

```javascript
25
```

is a value.

It can be

- stored,
- copied,
- assigned,
- returned,
- passed to another function.

Example

```javascript
const a = 10;

const b = a;
```

Now imagine functions behaving exactly the same way.

```javascript
const greet = function () {

    console.log("Hello");

};
```

The function is simply another value.

---

# Functions Are First-Class Citizens

JavaScript is known as a language with **first-class functions**.

This means functions can be:

- Stored inside variables
- Stored inside arrays
- Stored inside objects
- Passed as arguments
- Returned from other functions
- Assigned to new variables
- Created at runtime

Very few programming concepts are as important as this one.

Callbacks, Promises, async/await, event handlers, React components, Express middleware, and many other JavaScript patterns all rely on this capability.

---

# What Is a Function Expression?

A Function Expression is a function that is created **as part of an expression**.

General syntax:

```javascript
const variableName = function () {

};
```

Unlike a Function Declaration,

the function is assigned to a variable.

---

# Breaking Down the Syntax

Consider

```javascript
const greet = function () {

    console.log("Hello");

};
```

Let's examine every part.

---

## `const`

```javascript
const
```

Creates a constant variable.

That variable will store a function.

---

## Variable Name

```javascript
greet
```

This is **not** the function's name.

It is simply the variable holding the function object.

This distinction becomes important later.

---

## Assignment Operator

```javascript
=
```

The assignment operator stores the function object inside the variable.

Just like

```javascript
const age = 25;
```

stores a number,

```javascript
const greet = function () {

};
```

stores a function.

---

## `function`

```javascript
function
```

Creates the function object.

Notice something interesting.

There is **no function name** after the keyword.

This is called an **anonymous function**.

---

# Anonymous Function

Anonymous means

> **without a name**

Example

```javascript
function () {

}
```

This function has no name.

Normally,

anonymous functions are assigned immediately to variables.

```javascript
const greet = function () {

};
```

Although the function itself is anonymous,

the variable provides a way to access it.

---

# Internal JavaScript Engine Behaviour

Suppose we write

```javascript
const greet = function () {

    console.log("Hello");

};
```

Internally,

JavaScript performs several steps.

---

## Creation Phase

Unlike Function Declarations,

JavaScript does **not** create the function immediately.

Instead,

it creates the variable.

Conceptually

```text
Memory

↓

greet

↓

<uninitialized>
```

because

```javascript
const
```

uses the Temporal Dead Zone (TDZ), which we'll study in detail later.

No function object is assigned yet.

---

## Execution Phase

When JavaScript reaches

```javascript
const greet = function () {

};
```

it performs these steps:

1. Create Function Object.
2. Assign Function Object to `greet`.
3. Continue execution.

Memory becomes

```text
greet

↓

Function Object
```

Only **after** this assignment can the function be called.

---

# Function Expression Lifecycle

```text
Parser

↓

Creation Phase

↓

Variable Created

↓

Execution

↓

Create Function Object

↓

Assign to Variable

↓

Ready to Call
```

Notice how this differs from Function Declarations.

---

# Why Function Expressions Are Not Fully Hoisted

Consider

```javascript
greet();

const greet = function () {

    console.log("Hello");

};
```

Output

```text
ReferenceError
```

Why?

Because during the Creation Phase,

only the variable exists.

The function object does **not**.

The assignment happens later during execution.

Timeline

```text
Creation Phase

↓

Create Variable

↓

Execution Begins

↓

greet()

↓

Variable Not Initialized

↓

ReferenceError
```

---

# Visual Comparison

## Function Declaration

Creation Phase

```text
greet

↓

Entire Function
```

Execution

```
Ready
```

---

## Function Expression

Creation Phase

```text
greet

↓

Variable Only
```

Execution

```text
Create Function

↓

Assign

↓

Ready
```

This is one of the biggest behavioral differences between the two.

# Example 1 — Beginner

```javascript
const greet = function () {

    console.log("Hello");

};

greet();
```

Output

```text
Hello
```

Execution

```text
Variable Created

↓

Function Assigned

↓

greet()

↓

Hello
```

---

# Example 2 — Intermediate

```javascript
const multiply = function (a, b) {

    return a * b;

};

console.log(
    multiply(6, 8)
);
```

Output

```text
48
```

Notice that parameters and return values work exactly the same as Function Declarations.

The difference is **how the function is created**, not how it executes.

---

# Example 3 — Real-World Application

Suppose you're building a Point of Sale system.

You may want different calculation functions depending on configuration.

```javascript
const calculateTotal = function (
    amount,
    tax
) {

    return amount + tax;

};

console.log(
    calculateTotal(
        1000,
        160
    )
);
```

Output

```text
1160
```

Since the function is stored in a variable,

it can later be replaced, passed to another function, or selected dynamically based on user preferences.

This flexibility is one of the main reasons Function Expressions exist.

---

# Why Developers Use Function Expressions

Function Expressions allow us to treat functions like ordinary values.

For example,

a function can be stored inside an object.

```javascript
const calculator = {

    add: function (a, b) {

        return a + b;

    }

};
```

Or inside an array.

```javascript
const operations = [

    function () {

        console.log("Operation 1");

    },

    function () {

        console.log("Operation 2");

    }

];
```

Or passed to another function (callbacks—which we'll study in the next major topic).

---

# Common Mistakes

## Mistake 1 — Calling Before Assignment

Incorrect

```javascript
greet();

const greet = function () {

};
```

Produces

```text
ReferenceError
```

Always assign the function before calling it.

---

## Mistake 2 — Forgetting the Semicolon

Because a Function Expression is part of an assignment,

it should end with a semicolon.

Correct

```javascript
const greet = function () {

};
```

This follows standard JavaScript style conventions.

---

## Mistake 3 — Confusing Variables with Function Names

Remember:

```javascript
const greet = function () {

};
```

`greet` is the **variable**, not the function's internal name.

The function itself is anonymous.

---

# Best Practices

- Use Function Declarations for general-purpose reusable functions.
- Use Function Expressions when you want to store functions in variables or pass them around.
- Prefer `const` over `let` for function expressions because the function reference usually should not change.
- Choose descriptive variable names that explain what the function does.

---

# Performance Considerations

Modern JavaScript engines optimize both Function Declarations and Function Expressions very effectively.

Choosing one over the other is generally about readability, behavior, and design—not raw performance.
