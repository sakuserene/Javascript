# Named Function Expressions

In the previous lesson, we learned about **anonymous function expressions**.

Example:

```javascript
const greet = function () {

    console.log("Hello");

};
```

Notice something unusual.

The function itself has **no name**.

Only the variable does.

Many beginners think this is the only type of function expression.

It isn't.

JavaScript also allows **Named Function Expressions (NFEs).**

---

# What Is a Named Function Expression?

A Named Function Expression is a function expression that gives the function its own internal name.

General syntax:

```javascript
const variable = function functionName() {

};
```

Example:

```javascript
const greet = function sayHello() {

    console.log("Hello");

};
```

Notice there are now **two names**:

- Variable name: `greet`
- Function name: `sayHello`

At first this seems unnecessary.

Why have two names?

To answer that, we need to understand how JavaScript stores functions internally.

---

# Internal Memory Model

Consider this code:

```javascript
const greet = function sayHello() {

    console.log("Hello");

};
```

Many beginners imagine memory looks like this:

```text
greet

↓

Function
```

But internally it's closer to this conceptual model:

```text
Variable

greet

↓

Function Object

↓

Internal Name

sayHello
```

The variable points to the function object.

The function object also stores its own name.

---

# Why Was This Feature Introduced?

Anonymous functions work well most of the time.

However, they create two important problems.

## Problem 1 — Debugging

Imagine an application crashes.

With an anonymous function, the stack trace may look like:

```text
at <anonymous>

at <anonymous>

at <anonymous>
```

Not very helpful.

Now consider a named function expression.

```javascript
const calculate = function calculateInvoice() {

};
```

A stack trace can now include:

```text
calculateInvoice
```

which makes debugging much easier.

---

## Problem 2 — Self-Recursion

We'll study recursion in depth later.

For now, imagine a function that calls itself.

With an anonymous function,

this becomes difficult if the variable changes.

Named function expressions solve this problem.

---

# Anonymous Function vs Named Function

Anonymous

```javascript
const greet = function () {

};
```

Named

```javascript
const greet = function sayHello() {

};
```

The difference is only the presence of:

```javascript
sayHello
```

Everything else remains the same.

---

# Internal JavaScript Engine Behaviour

Suppose we write:

```javascript
const greet = function sayHello() {

    console.log("Hello");

};
```

Execution proceeds as follows:

### Creation Phase

JavaScript creates the variable.

```text
greet

↓

<uninitialized>
```

---

### Execution Phase

JavaScript:

1. Creates the function object.
2. Stores the internal name `sayHello`.
3. Assigns the function object to `greet`.

Conceptually:

```text
greet

↓

Function Object

↓

Internal Name

sayHello
```

---

# Calling the Function

You call the function using the variable:

```javascript
greet();
```

Output

```text
Hello
```

What about:

```javascript
sayHello();
```

Output

```text
ReferenceError
```

Why?

Because the internal function name is **not** available in the outer scope.

---

# Scope of the Internal Name

Consider:

```javascript
const greet = function sayHello() {

    console.log("Hello");

};

sayHello();
```

Output

```text
ReferenceError:
sayHello is not defined
```

The name

```javascript
sayHello
```

exists **only inside the function itself**.

Outside,

only

```javascript
greet
```

exists.

---

# Visual Representation

Global Scope

```text
greet

↓

Function Object
```

Inside the function

```text
Function Scope

↓

sayHello

↓

Function Object
```

Outside code cannot access

```
sayHello
```

---

# Example 1 — Beginner

```javascript
const greet = function sayHello() {

    console.log("Hello");

};

greet();
```

Output

```text
Hello
```

Attempt

```javascript
sayHello();
```

Output

```text
ReferenceError
```

# Example 2 — Self-Recursion

Later we'll learn recursion.

For now, consider:

```javascript
const countdown = function count(number) {

    if (number <= 0) {

        console.log("Done");

        return;

    }

    console.log(number);

    count(number - 1);

};

countdown(3);
```

Output

```text
3

2

1

Done
```

Notice something important.

Inside the function,

it uses

```javascript
count()
```

not

```javascript
countdown()
```

The internal name allows the function to call itself even if the outer variable changes.

---

# Why Is This Useful?

Suppose someone later changes

```javascript
countdown
```

to another value.

```javascript
countdown = null;
```

The internal function name

```javascript
count
```

still exists **inside the function**.

This makes recursion more reliable.

---

# Example 3 — Real-World Application

Imagine a retry mechanism.

```javascript
const retryRequest = function retry(attempt) {

    if (attempt === 0) {

        console.log("Giving up");

        return;

    }

    console.log(
        `Retry ${attempt}`
    );

    retry(attempt - 1);

};

retryRequest(3);
```

Output

```text
Retry 3

Retry 2

Retry 1

Giving up
```

Although recursion is the focus here, the important lesson is that the function uses its **internal name**, not the variable.

Open Chrome DevTools.


The debugger often displays:

```text

```


This makes stack traces far easier to understand.

Anonymous

```javascript

    throw new Error();

};

```

Possible stack trace

```text

```

Named

```javascript

    throw new Error();

```

Possible stack trace

```text

```

Much easier to debug.

# Common Mistakes

## Mistake 1 — Calling the Internal Name

Incorrect

```javascript
sayHello();
```

Correct

```javascript
greet();
```

The internal name exists only inside the function.

---

## Mistake 2 — Assuming Two Global Functions Exist

This

```javascript
const greet = function sayHello() {

};
```

does **not** create two global functions.

It creates:

- One global variable (`greet`)
- One internal function name (`sayHello`) scoped to the function itself.

---

## Mistake 3 — Using Anonymous Functions Everywhere

Anonymous functions are convenient,

but giving functions meaningful names improves debugging, stack traces, and maintainability.

---

# Best Practices

- Use anonymous function expressions for simple callbacks and short-lived functions.
- Use named function expressions when recursion or debugging benefits from an explicit name.
- Choose descriptive internal names if you provide one.
- Don't expose unnecessary names to the outer scope.

---

# Performance Considerations

Named function expressions and anonymous function expressions have nearly identical performance in modern JavaScript engines.

Choose between them based on readability, debugging, and maintainability rather than speed.

---

# Key Takeaways

- Function expressions may be **anonymous** or **named**.
- The variable name and the internal function name are different things.
- The internal name is available **only inside the function**.
- Named function expressions improve stack traces and recursive function design.
- Most modern code uses anonymous function expressions unless a specific reason exists for naming the function.
