# Introduction

In the previous lesson, we learned **how functions receive data**.

Now we will answer another important question:

> **How are functions actually created?**

At first, JavaScript appears to have only one way.

For example:

```javascript
function greet() {
    console.log("Hello");
}
```

But as you read more professional JavaScript code, you'll encounter functions written like this:

```javascript
const greet = function () {
    console.log("Hello");
};
```

or

```javascript
const greet = () => {
    console.log("Hello");
};
```

All three create functions.

Why?

Wouldn't one syntax be enough?

To answer that, we need to understand the history of JavaScript.

---

# Why Are There Multiple Ways to Create Functions?

Programming languages evolve over time.

As developers built larger applications, they discovered limitations in earlier language features.

Instead of replacing old syntax—which would break millions of existing websites—JavaScript added new, more expressive ways to write functions.

Today we have three primary forms:

1. **Function Declarations**
2. **Function Expressions**
3. **Arrow Functions**

Each solves different problems and has different behavior.

Understanding these differences is essential for reading professional codebases and passing JavaScript interviews.

---

# Historical Evolution

### JavaScript 1995

The original JavaScript language introduced **Function Declarations**.

Example

```javascript
function greet() {
    console.log("Hello");
}
```

This was the only practical way to define reusable functions.

---

### Later JavaScript

Developers realized that functions are values.

If numbers can be stored in variables,

```javascript
const age = 25;
```

why can't functions?

This led to **Function Expressions**.

```javascript
const greet = function () {
    console.log("Hello");
};
```

Now functions could be stored, passed around, and returned like any other value.

---

### ECMAScript 2015 (ES6)

JavaScript introduced **Arrow Functions**.

```javascript
const greet = () => {
    console.log("Hello");
};
```

Arrow functions were designed to provide:

- Shorter syntax
- Better handling of `this`
- Cleaner callback functions
- More functional programming patterns

We'll study all of those later.

---

# Function Declaration

A Function Declaration is the classic way of defining a function.

General syntax:

```javascript
function functionName(parameters) {

    // function body

}
```

Example:

```javascript
function greet() {

    console.log("Hello");

}
```

Calling it:

```javascript
greet();
```

Output

```text
Hello
```

---

# Breaking Down the Syntax

Consider:

```javascript
function greet() {

    console.log("Hello");

}
```

Let's examine each part.

---

## `function`

```javascript
function
```

This is a reserved keyword.

It tells the parser:

> "Create a function."

---

## Function Name

```javascript
greet
```

This is the identifier used to call the function.

Good names describe behavior.

Good

```javascript
calculateTax()

validateEmail()

printReceipt()
```

Poor

```javascript
abc()

run()

thing()

hello()
```

Professionals choose names that explain **what** the function does.

---

## Parentheses

```javascript
()
```

These hold parameters.

No parameters:

```javascript
function greet() {

}
```

One parameter:

```javascript
function greet(name) {

}
```

Multiple parameters:

```javascript
function greet(firstName, lastName) {

}
```

---

## Curly Braces

```javascript
{

}
```

Everything inside belongs to the function body.

Nothing executes until the function is called.

---

# Internal JavaScript Engine Behaviour

Consider:

```javascript
function greet() {

    console.log("Hello");

}
```

Many beginners imagine JavaScript simply stores this function somewhere.

Internally, more happens.

---

## Step 1 — Parsing

The parser reads:

```javascript
function greet() {

}
```

It recognizes a **Function Declaration**.

---

## Step 2 — Function Object Creation

JavaScript creates a function object.

Conceptually:

```text
Memory

↓

Function Object

↓

greet()
```

The function body is stored.

The function is **not executed**.

It is merely created.

---

## Step 3 — Store the Reference

The identifier

```javascript
greet
```

is associated with that function object.

Conceptually:

```text
greet

↓

Function Object
```

---

## Step 4 — Wait

The engine does nothing further until it encounters:

```javascript
greet();
```

Only then is a Function Execution Context created.

---

# Mental Model

Think of a function declaration as writing a recipe.

```text
Recipe Book

↓

Chocolate Cake Recipe
```

Writing the recipe does **not** bake the cake.

Only following the recipe does.

Similarly:

```javascript
function greet() {

}
```

creates the function.

Calling

```javascript
greet();
```

executes it.

---

# Function Declaration Lifecycle

```text
Source Code

↓

Parser

↓

Function Object Created

↓

Stored in Memory

↓

Wait

↓

Function Called

↓

Execution Context Created

↓

Execute Statements

↓

Return

↓

Execution Context Removed
```

This lifecycle repeats every time the function is called.

# Example 1 — Beginner

```javascript
function sayHello() {

    console.log("Hello!");

}

sayHello();
```

Output

```text
Hello!
```

### Explanation

- The declaration creates the function.
- `sayHello()` invokes it.
- A Function Execution Context is created.
- `"Hello!"` is printed.
- The execution context is destroyed.

---

# Example 2 — Intermediate

```javascript
function multiply(a, b) {

    return a * b;

}

const answer = multiply(6, 7);

console.log(answer);
```

Output

```text
42
```

Execution

```text
multiply()

↓

New Execution Context

↓

Bind Parameters

↓

Calculate

↓

Return 42

↓

Store in answer
```

# Example 3 — Real-World Application

Imagine a Point of Sale system.

```javascript
function printReceipt(receiptNumber) {

    console.log(
        `Printing receipt ${receiptNumber}`
    );

}
```

Usage

```javascript
printReceipt("RCT-001");

printReceipt("RCT-002");

printReceipt("RCT-003");
```

Output

```text
Printing receipt RCT-001

Printing receipt RCT-002

Printing receipt RCT-003
```

Notice how one function handles many different receipts.

# Common Mistakes

## Mistake 1 — Forgetting Parentheses

Incorrect

```javascript
printReceipt;
```

This refers to the function.

It does **not** call it.

Correct

```javascript
printReceipt();
```

---

## Mistake 2 — Poor Function Names

Avoid

```javascript
function test() {

}

function run() {

}

function work() {

}
```

Instead

```javascript
calculateInvoice()

validateLogin()

generateReport()
```

---

## Mistake 3 — Extremely Long Functions

Avoid writing functions that perform many unrelated tasks.

Instead of:

```javascript
processEverything();
```

Prefer:

```javascript
validateInput();

calculateTotal();

saveInvoice();

printReceipt();
```

Small functions are easier to test, debug, and reuse.

---

# Best Practices

- Use descriptive names.
- Keep each function focused on one responsibility.
- Prefer small, reusable functions.
- Place related functions together in your codebase.
- Write functions that are easy to read before optimizing them.

---

# Performance Considerations

Creating a function declaration is inexpensive.

Modern JavaScript engines optimize function creation and invocation heavily.

The real performance costs usually come from:

- Expensive computations.
- Large loops.
- DOM manipulation.
- Network requests.

Focus first on clean design and readability.
