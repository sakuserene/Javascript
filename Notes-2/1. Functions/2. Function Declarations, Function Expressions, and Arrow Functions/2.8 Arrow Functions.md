# Arrow Functions — Internal Characteristics

In the previous lesson we learned the **biggest difference** between regular functions and arrow functions:

> Arrow functions have **lexical `this`**.

Many developers stop there.

However, arrow functions differ from regular functions in several other important ways.

Understanding these differences will explain **why arrow functions were designed as a completely different kind of function rather than just shorter syntax.**

In this lesson we'll answer questions such as:

- Why can't arrow functions be constructors?
- Why don't arrow functions have a `prototype`?
- Why don't they have their own `arguments`?
- Why do they inherit `super`?
- Why do they inherit `new.target`?

These questions reveal how JavaScript internally represents arrow functions.

---

# JavaScript Has Different Kinds of Functions

One of the biggest misconceptions is believing that every function behaves identically.

They do not.

Conceptually, JavaScript has multiple function "types."

```
Functions

│

├───────────────┐
│               │

Regular Functions

Arrow Functions

Generator Functions

Async Functions

Async Generators

Class Methods
```

Arrow functions intentionally omit several capabilities that regular functions possess.

---

# Why Can't Arrow Functions Be Constructors?

Suppose we write a regular function.

```javascript
function User(name) {

    this.name = name;

}
```

Now

```javascript
const user = new User("Alice");

console.log(user.name);
```

Output

```text
Alice
```

This works because regular functions can act as **constructors**.

---

# What is a Constructor?

A constructor is a function used together with the keyword:

```javascript
new
```

Its purpose is to create new objects.

Example

```javascript
const person = new User("John");
```

Conceptually

```
new

↓

Create Empty Object

↓

Assign this

↓

Execute Function

↓

Return Object
```

---

# Trying the Same with an Arrow Function

```javascript
const User = (name) => {

    this.name = name;

};
```

Now

```javascript
const person = new User("Alice");
```

Output

```text
TypeError:

User is not a constructor
```

---

# Why?

Arrow functions were **never designed** to construct objects.

Remember:

Arrow functions do **not** have their own

```javascript
this
```

Constructors depend on creating a brand-new

```javascript
this
```

during object creation.

These two ideas conflict.

Therefore,

JavaScript simply forbids it.

---

# Internal Behaviour

Regular Function

```
new

↓

Create Object

↓

this

↓

Execute Function

↓

Return Object
```

Arrow Function

```
new

↓

Impossible

↓

No own this

↓

TypeError
```

---

# Arrow Functions Have No Prototype

Every regular function automatically receives a property called

```javascript
prototype
```

Example

```javascript
function Person() {

}

console.log(Person.prototype);
```

Output

```text
{}
```

Why?

Because constructors need prototypes.

---

Now

```javascript
const Person = () => {

};

console.log(Person.prototype);
```

Output

```text
undefined
```

Arrow functions have **no prototype**.

Why?

Because prototypes are used during object construction.

Arrow functions cannot construct objects.

Therefore,

they don't need one.

---

Regular Function


Arrow Function

# Arrow Functions Do Not Have Their Own `arguments`

Earlier in the course,

we studied

```javascript
arguments
```

Example

```javascript
function demo() {

    console.log(arguments);

}

demo(1, 2, 3);
```

Output

```text
Arguments(3)

[
1,
2,
3
]
```

Now try

```javascript
const demo = () => {

    console.log(arguments);

};

demo(1, 2, 3);
```

In most contexts,

this results in:

```text
ReferenceError
```

or it accesses the enclosing function's `arguments` if one exists.

Why?

Because arrow functions **do not create their own `arguments` object**.

Instead,

they inherit it from the surrounding function scope.

---

# Why Was This Decision Made?

Remember the design goal.

Arrow functions were intended to be

- lightweight
- concise
- lexical

Creating another

```javascript
arguments
```

object would contradict that design.

Instead,

JavaScript expects you to use

```javascript
...rest
```

Example

```javascript
const total = (...numbers) => {

    console.log(numbers);

};
```

Output

```text
[
1,
2,
3
]
```

This is the modern solution.

---

# Arrow Functions Inherit `super`

We'll study classes later.

For now,

understand this.

Regular methods create their own execution context.

Arrow functions do not.

Therefore,

they inherit

```javascript
super
```

from the surrounding method.

This allows arrow functions inside class methods to access parent class behavior naturally.

---

# Arrow Functions Inherit `new.target`

Regular constructor functions can inspect

```javascript
new.target
```

to determine whether they were called using `new`.

Arrow functions don't have their own `new.target`.

Like `this` and `arguments`,

they inherit it from the surrounding scope.

---

| Feature | Function Declaration | Function Expression | Arrow Function |
|----------|----------------------|---------------------|----------------|
| Hoisted | ✅ Yes | ❌ No | ❌ No |
| Own `this` | ✅ Yes | ✅ Yes | ❌ No |
| Own `arguments` | ✅ Yes | ✅ Yes | ❌ No |
| Constructor | ✅ Yes | ✅ Yes | ❌ No |
| `prototype` | ✅ Yes | ✅ Yes | ❌ No |
| Lexical `this` | ❌ No | ❌ No | ✅ Yes |
| Short Syntax | ❌ No | ❌ No | ✅ Yes |
| Best for Callbacks | Good | Good | Excellent |

# When Should You Use Function Declarations?

Use them when:

- Building reusable utility functions.
- Defining top-level application logic.
- You want hoisting.
- You want clear API-style code.

Example

```javascript
function calculateTax() {

}
```

---

# When Should You Use Function Expressions?

Use them when:

- Passing functions around.
- Assigning functions to variables.
- Creating configurable behavior.
- Storing functions inside objects.

Example

```javascript
const calculateTax = function () {

};
```

---

# When Should You Use Arrow Functions?

Arrow functions are excellent for:

- Callbacks
- Promise handlers
- `map()`
- `filter()`
- `reduce()`
- `setTimeout()`
- `setInterval()`
- Event callbacks that should inherit the surrounding `this`
- Functional programming

Example

```javascript
numbers.map(
    number => number * 2
);
```

---

# When Should You NOT Use Arrow Functions?

Avoid arrow functions when:

## 1. Creating Constructors

Incorrect

```javascript
const User = () => {

};

new User();
```

---

## 2. Methods That Need Their Own `this`

Suppose

```javascript
const user = {

    name: "Alice",

    greet: () => {

        console.log(this.name);

    }

};
```

This usually won't behave as expected because the arrow function inherits `this` from the surrounding scope rather than from the `user` object.

Instead, use a regular method:

```javascript
const user = {

    name: "Alice",

    greet() {

        console.log(this.name);

    }

};
```

---

## 3. When You Need `arguments`

Prefer

```javascript
(...args)
```

instead.

---

# Professional Guidelines

Modern JavaScript developers generally follow these patterns:

✅ Function Declaration

- Large reusable functions

✅ Arrow Function

- Small callbacks
- Functional programming
- Array methods
- Promise chains

✅ Method Syntax

- Object methods
- Class methods

Avoid blindly converting every function into an arrow function.

Choose the form whose behavior best matches the problem.

# Example 1 — Callback

```javascript
const numbers = [1, 2, 3];

const doubled = numbers.map(
    number => number * 2
);

console.log(doubled);
```

Output

```text
[2, 4, 6]
```

---

# Example 2 — Constructor

```javascript
function User(name) {

    this.name = name;

}

const user = new User("Alice");

console.log(user.name);
```

Output

```text
Alice
```

Attempting the same with an arrow function throws a `TypeError`.

---

# Example 3 — Object Method

```javascript
const printer = {

    device: "POS Printer",

    print() {

        console.log(this.device);

    }

};

printer.print();
```

Output

```text
POS Printer
```

Using a regular method allows `this` to refer to the object that invoked the method.

# Common Mistakes

### Mistake 1

Using arrow functions as constructors.

---

### Mistake 2

Using arrow functions for object methods that depend on their own `this`.

---

### Mistake 3

Trying to use `arguments` inside arrow functions instead of rest parameters.

---

# Best Practices

- Understand the behavioral differences before choosing a function type.
- Prefer arrow functions for concise callbacks and transformations.
- Prefer regular functions or method syntax when object-specific `this` is required.
- Use rest parameters instead of relying on `arguments` in new code.

---

# Key Takeaways

- Arrow functions **cannot** be used with `new`.
- Arrow functions have **no `prototype`**.
- Arrow functions do **not** create their own `this`.
- Arrow functions do **not** create their own `arguments`.
- Arrow functions inherit `super` and `new.target`.
- Use each function type according to its strengths rather than treating them as interchangeable.
