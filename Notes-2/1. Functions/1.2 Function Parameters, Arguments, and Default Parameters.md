# Introduction

In the previous lesson we learned that a function is a reusable block of code.

For example,

```javascript
function greet() {
    console.log("Hello");
}
```

works perfectly.

But imagine we wanted to greet **different people**.

Should we write

```javascript
function greetAlice() {
    console.log("Hello Alice");
}

function greetJohn() {
    console.log("Hello John");
}

function greetMary() {
    console.log("Hello Mary");
}
```

Of course not.

That would be terrible programming.

We are repeating the same logic.

Only one thing changes:

```
Alice
John
Mary
```

Everything else stays exactly the same.

So developers asked a question.

> "How can we make a function reusable for different data?"

That question led to one of the most important concepts in programming:

> **Parameters**

---

# Why Functions Need Inputs

Imagine a coffee vending machine.

```
          Coffee Machine

        +------------------+
Coins ->|                  |
Choice->|    Machine       |----> Coffee
        |                  |
        +------------------+
```

The machine does not always produce the same drink.

Instead, it waits for input.

Input determines output.

Likewise, functions work exactly the same way.

```
          Function

Input -------------> Processing -------------> Output
```

Without input...

every function would always do exactly one thing.

Example:

```javascript
function square() {
    return 25;
}
```

No matter how many times you call it,

```javascript
square();
square();
square();
```

Output

```
25
25
25
```

Not very useful.

Now compare that to

```javascript
function square(number) {
    return number * number;
}
```

Now,

```javascript
square(2);
square(5);
square(10);
```

Outputs

```
4
25
100
```

Same logic.

Different inputs.

Different outputs.

This is the true power of parameters.

---

# Real World Analogy

Imagine a blender.

```
Fruit
   ↓

+-----------+
| Blender   |
+-----------+
      ↓

Juice
```

The blender doesn't care whether you insert

- Mangoes
- Bananas
- Oranges
- Pineapples

It performs the same process.

Functions behave the same way.

```
Input

↓

Function

↓

Output
```

The function's algorithm remains identical.

Only the data changes.

# What is a Parameter?

## Definition

A **parameter** is a variable declared inside the function definition that receives data whenever the function is called.

Example:

```javascript
function greet(name) {

}
```

Here

```javascript
name
```

is **not** actual data.

It is simply a placeholder.

Think of it like an empty box.

```
+--------+
|  name  |
+--------+

Currently empty
```

When the function runs,

JavaScript places actual data inside that box.

---

## Another Analogy

Suppose someone tells you:

> "Dear _____"

Before sending the letter,

the blank space is empty.

```
Dear ______
```

Later you fill it.

```
Dear Alice
```

or

```
Dear John
```

or

```
Dear Sarah
```

The blank space is the parameter.

The inserted name is the argument.

---

# What is an Argument?

An **argument** is the actual value supplied to the function.

Example

```javascript
greet("Alice");
```

```
"Alice"
```

is the argument.

JavaScript performs this mapping internally.

```
Parameter

name

↓

Argument

"Alice"
```

Result

```
name = "Alice"
```

Then the function executes.

---

# Parameter vs Argument

Many beginners think they are the same.

They are not.

| Parameter | Argument |
|-----------|----------|
| Declared inside the function | Supplied when calling the function |
| Placeholder | Actual value |
| Exists before execution | Exists during the function call |
| Variable | Data |

Example

```javascript
function multiply(a, b) {

}
```

Parameters are

```
a
b
```

Calling

```javascript
multiply(10, 20);
```

Arguments become

```
10
20
```

Internally JavaScript performs

```
a = 10
b = 20
```

before executing the body.

---

# Why Were Parameters Invented?

Without parameters developers had to duplicate code.

Suppose you wanted to calculate VAT.

Without parameters

```javascript
const total1 = 100 * 1.16;

const total2 = 250 * 1.16;

const total3 = 400 * 1.16;

const total4 = 700 * 1.16;
```

Notice the repetition.

Now with parameters

```javascript
function calculateVAT(price) {
    return price * 1.16;
}
```

Usage

```javascript
calculateVAT(100);
calculateVAT(250);
calculateVAT(400);
calculateVAT(700);
```

One function.

Infinite possibilities.

---

# The Programming Principle

Parameters follow one of the oldest software engineering principles:

> **Separate logic from data.**

Logic

```javascript
price * 1.16
```

Data

```
100
250
700
1000
```

Never mix them unnecessarily.

Professionals write algorithms once.

Then feed them different data.

---

# Mental Model

Imagine functions as factories.

```
          Factory

Raw Materials

↓

Machine

↓

Finished Product
```

Parameters are the place where raw materials enter.

```
price

↓

Function

↓

VAT
```

The factory never changes.

Only the materials entering it change.

This is exactly why reusable code exists.

# Internal JavaScript Engine Behaviour

This is where JavaScript becomes fascinating.

Suppose we write

```javascript
function greet(name) {
    console.log(name);
}

greet("Alice");
```

You might think JavaScript immediately prints

```
Alice
```

That is **not** what happens.

Instead, many internal steps occur.

### Step 1

The JavaScript engine parses the source code.

```
Source Code

↓

Parser
```

The parser recognizes

```
function greet(name)
```

as a function declaration.

It creates a function object in memory.

At this stage, **the body has not executed**.

The function merely exists and is ready to be called.

# Internal JavaScript Engine Behaviour (Continued)

In the previous section, the JavaScript parser recognized the function declaration and created a **function object**.

Now let's see what actually happens when the function is called.

Consider this code:

```javascript
function greet(name) {
    console.log(name);
}

greet("Alice");
```

Many beginners imagine JavaScript simply jumps into the function.

Internally, much more happens.

---

## Step 2 — JavaScript Encounters the Function Call

Execution begins from top to bottom.

```text
Global Execution Context

↓

function greet(name) { ... }

↓

greet("Alice")
```

When JavaScript reaches

```javascript
greet("Alice");
```

it recognizes that a function must now execute.

But before executing the first line inside the function, JavaScript prepares a completely new environment.

---

# Step 3 — A New Execution Context Is Created

Whenever a normal JavaScript function is called, the engine creates a **Function Execution Context (FEC)**.

Think of an execution context as a temporary workspace.

```text
Global Workspace

↓

Function Workspace

↓

Execute Code

↓

Destroy Workspace
```

Every function call gets its own workspace.

Even if the same function is called many times.

Example:

```javascript
greet("Alice");

greet("John");

greet("Mary");
```

JavaScript does **not** reuse one workspace.

Instead it creates

```text
Workspace 1

↓

Destroyed

↓

Workspace 2

↓

Destroyed

↓

Workspace 3

↓

Destroyed
```

Each call is completely independent.

---

# What Exists Inside the Function Execution Context?

The Function Execution Context contains several things.

```text
Function Execution Context

├── Parameters
├── Arguments
├── Local Variables
├── this value
├── Scope Reference
└── Instruction Pointer
```

For now, we'll focus on the first three.

---

## Parameter Storage

Suppose we call

```javascript
greet("Alice");
```

Internally JavaScript allocates memory like this:

```text
Function Execution Context

name

↓

"Alice"
```

Notice something important.

The parameter

```javascript
name
```

did not already contain `"Alice"`.

It only becomes `"Alice"` **during the function call**.

---

# Step 4 — Parameter Binding

This process is called **parameter binding**.

Parameter binding means:

> Matching each argument to its corresponding parameter.

Example

```javascript
function add(a, b) {

}
```

Call

```javascript
add(10, 20);
```

JavaScript performs

```text
a ← 10

b ← 20
```

before executing any code inside the function.

Another example

```javascript
function introduce(firstName, lastName, age) {

}
```

Call

```javascript
introduce(
    "Alice",
    "Smith",
    24
);
```

Binding becomes

```text
firstName = "Alice"

lastName = "Smith"

age = 24
```

Only after binding does JavaScript execute the function body.

---

# Visualizing Parameter Binding

Imagine empty containers.

Before the function call

```text
+-----------+
| firstName |
|           |
+-----------+

+-----------+
| lastName  |
|           |
+-----------+

+-----------+
| age       |
|           |
+-----------+
```

Function call

```javascript
introduce(
    "Alice",
    "Smith",
    24
);
```

After binding

```text
+-----------+
| firstName |
| Alice     |
+-----------+

+-----------+
| lastName  |
| Smith     |
+-----------+

+-----------+
| age       |
| 24        |
+-----------+
```

The parameters now behave exactly like local variables.

# Parameters Become Local Variables

This surprises many beginners.

Inside the function

```javascript
function greet(name) {

}
```

the parameter

```javascript
name
```

is simply another local variable.

Example

```javascript
function greet(name) {

    console.log(name);

    name = "John";

    console.log(name);

}
```

Call

```javascript
greet("Alice");
```

Output

```text
Alice

John
```

Why?

Because inside the function

```javascript
name
```

is just a local variable.

Changing it does **not** affect the original argument when the argument is a primitive value (numbers, strings, booleans, etc.). Later in this lesson we'll explore how objects behave differently.

---

# Step 5 — Executing the Function Body

Once every parameter has been initialized,

JavaScript begins executing statements one by one.

```javascript
function greet(name) {

    console.log(name);

}
```

Execution flow

```text
Create Function Context

↓

Bind Parameters

↓

Execute Statement 1

↓

No More Statements

↓

Return undefined

↓

Destroy Function Context
```

Notice something important.

Every normal function returns **something**.

If you don't explicitly return a value,

JavaScript automatically returns

```text
undefined
```

We'll explore return values much more deeply in the next lesson.

---

# The Call Stack

To manage function execution, JavaScript uses a data structure called the **Call Stack**.

A stack follows the rule:

> Last In, First Out (LIFO)

Imagine a stack of plates.

```
Plate 3

Plate 2

Plate 1
```

You always remove the top plate first.

JavaScript manages functions the same way.

---

## Initial State

Before calling the function

```text
Call Stack

+----------------------+
| Global Execution     |
+----------------------+
```

---

## During Function Call

```javascript
greet("Alice");
```

JavaScript pushes the function onto the stack.

```text
Call Stack

+----------------------+
| greet()              |
+----------------------+
| Global Execution     |
+----------------------+
```

The top of the stack is always the function currently executing.

---

## After the Function Finishes

Once execution reaches the end of the function,

or encounters a `return` statement,

the function is removed.

```text
Call Stack

+----------------------+
| Global Execution     |
+----------------------+
```

This process happens thousands of times while programs run.

---

# Nested Function Calls

Consider

```javascript
function first() {

    second();

}

function second() {

    third();

}

function third() {

    console.log("Done");

}

first();
```

Call stack evolution

Initially

```text
Global
```

After calling `first()`

```text
first

Global
```

Inside `first`

```text
second

first

Global
```

Inside `second`

```text
third

second

first

Global
```

After `third` finishes

```text
second

first

Global
```

After `second`

```text
first

Global
```

After `first`

```text
Global
```

This constant pushing and popping is how JavaScript keeps track of where it is during execution.

---

# Memory Representation

Let's examine memory for a simple function call.

Code

```javascript
function multiply(a, b) {

    const answer = a * b;

    return answer;

}

multiply(6, 8);
```

Conceptually, memory looks like this while the function is executing:

```text
Function Execution Context

Parameter a
↓

6

Parameter b
↓

8

Local Variable answer
↓

48
```

When the function returns, this execution context becomes eligible for cleanup because nothing in the global code can directly access `a`, `b`, or `answer`.

---

# Key Ideas to Remember

- A parameter is a placeholder declared in the function definition.
- An argument is the actual value supplied when calling the function.
- During invocation, JavaScript binds arguments to parameters.
- Parameters behave like local variables.
- Every function call creates a new execution context.
- Execution contexts are managed using the call stack.
- When a function finishes, its execution context is removed from the call stack.
- If no value is explicitly returned, JavaScript returns `undefined`.

# Syntax Deep Dive

Now that we understand **what parameters and arguments are** and **how JavaScript binds them internally**, let's study the syntax in detail.

Consider the following function.

```javascript
function add(a, b) {
    return a + b;
}
```

Although it appears simple, every keyword and symbol has a specific purpose.

Let's examine each part.

---

## The `function` Keyword

```javascript
function
```

`function` is a reserved JavaScript keyword.

It tells the JavaScript parser:

> "I am about to declare a function."

Without this keyword, JavaScript would interpret the following text differently.

Example

```javascript
function greet() {

}
```

Here,

```javascript
function
```

is not a variable.

It is a language keyword.

---

## Function Name

```javascript
add
```

Every function should have a meaningful name.

Good examples

```javascript

calculateTotal()

validateEmail()

printReceipt()

createUser()
```

Poor examples

```javascript

```

Professional developers name functions according to **what they do**, not how they do it.

Think of the function name as the title of a book.

A good title immediately tells readers what to expect.

---

## Parentheses

```javascript
()
```

Parentheses indicate where parameters are declared.

No parameters

```javascript

}
```

One parameter

```javascript

}
```

Two parameters

```javascript

}
```

Three parameters

```javascript

    firstName,

}
```

Notice that commas separate parameters.

---

## Parameter Names

Inside

```javascript

```

the parameters are

```

b
```

JavaScript creates variables with those names whenever the function executes.

Example

```javascript

```

Calling

```javascript

```

internally becomes something conceptually like:

```javascript

```

This isn't the exact implementation used by JavaScript engines, but it is a useful mental model.

---

## Curly Braces

```javascript
{

}
```

Curly braces define the **function body**.

Everything inside executes when the function is called.

Example

```javascript

    console.log("Hello");

    console.log("Welcome!");

}
```

Nothing inside the braces runs until

```javascript
greet("Alice");
```

is executed.

---

## Return Statement

Consider

```javascript

    return number * number;

}
```

The keyword

```javascript

```

means

Example

```javascript

```

Execution

```

↓

↓

25

↓

Stored inside result
```

# Parameter Order Matters

JavaScript matches parameters **by position**, not by name.

Example

```javascript
function subtract(a, b) {

    return a - b;

}
```

Call

```javascript
subtract(10, 3);
```

Binding

```
a = 10

b = 3
```

Result

```
7
```

Now reverse the arguments.

```javascript
subtract(3, 10);
```

Binding

```
a = 3

b = 10
```

Result

```
-7
```

The parameter names did not change.

Only the order of the arguments changed.

# Missing Arguments

One of the first surprising behaviors in JavaScript is that **functions do not require you to pass every argument**.

In many programming languages, calling a function with fewer arguments than parameters produces an error.

JavaScript takes a different approach.

Consider this function:

```javascript
function greet(name) {
    console.log(`Hello ${name}`);
}
```

Calling it normally:

```javascript
greet("Alice");
```

Output

```text
Hello Alice
```

Now call it without an argument.

```javascript
greet();
```

Output

```text
Hello undefined
```

No error occurred.

Why?

Let's understand what happened internally.

---

# Internal Parameter Binding

Recall that JavaScript binds arguments to parameters before executing the function.

Normal call

```javascript
greet("Alice");
```

Binding

```
name

↓

Alice
```

Now consider

```javascript
greet();
```

No argument exists.

JavaScript still creates the parameter.

It simply assigns

```
undefined
```

Binding becomes

```
name

↓

undefined
```

Then execution continues normally.

---

# Why Does JavaScript Use `undefined`?

Imagine JavaScript threw an error every time an argument was missing.

This would make many useful programming techniques impossible.

Instead, the language designers chose a predictable rule:

> **Every declared parameter always receives a value.**

If no argument exists,

that value becomes

```javascript
undefined
```

This design makes functions more flexible and allows developers to decide how to handle missing information.

---

Notice that the parameter still exists.

Only its value changes.

We passed four arguments.

What happens?

# Internal Binding

JavaScript binds arguments in order.

```
Argument 1

↓

Parameter 1

----------------

Argument 2

↓

Parameter 2
```

After all declared parameters have been assigned,

remaining arguments are **not** assigned to named parameters.

Binding

```

↓

10

----------------

b

↓

20
```

Arguments

```

40
```

remain unused by the parameter list.

The function therefore returns

```

```

because

```

```

equals

```
30
```

This flexibility is one reason JavaScript is well suited to browser programming and event-driven applications.

# Default Parameters

Until now, we've seen that when an argument is missing, JavaScript assigns the corresponding parameter the value:

```javascript
undefined
```

For example:

```javascript
function greet(name) {
    console.log(`Hello ${name}`);
}

greet();
```

Output

```text
Hello undefined
```

Although this behavior is predictable, it often isn't what we want.

Suppose we want the function to greet **"Guest"** whenever no name is supplied.

One approach is to write extra code inside the function.

```javascript
function greet(name) {

    if (name === undefined) {
        name = "Guest";
    }

    console.log(`Hello ${name}`);
}
```

This works, but imagine repeating that pattern in hundreds of functions.

There had to be a better solution.

---

# Why Default Parameters Were Introduced

Before **ECMAScript 2015 (ES6)**, JavaScript had no built-in syntax for parameter defaults.

Developers typically wrote code like this:

```javascript
function greet(name) {

    name = name || "Guest";

    console.log(`Hello ${name}`);
}
```

This became extremely common.

---

## The Problem with the Old Technique

At first glance, this seems clever.

However, it has a hidden problem.

Consider this function:

```javascript
function printNumber(number) {

    number = number || 100;

    console.log(number);

}
```

Now call:

```javascript
printNumber(0);
```

What should happen?

Most people expect:

```text
0
```

Instead, the output is:

```text
100
```

Why?

Because the logical OR operator (`||`) treats several values as **falsy**, including:

- `0`
- `""` (empty string)
- `false`
- `NaN`
- `null`
- `undefined`

So when `0` is passed, the expression:

```javascript
number || 100
```

evaluates to:

```javascript
100
```

even though `0` was intentionally provided.

This made the old technique unreliable.

---

# ES6 Default Parameters

ES6 introduced a cleaner solution.

Instead of assigning defaults inside the function body, we can assign them directly in the parameter list.

Example:

```javascript
function greet(name = "Guest") {

    console.log(`Hello ${name}`);

}
```

Calling:

```javascript
greet();
```

Output

```text
Hello Guest
```

Calling:

```javascript
greet("Alice");
```

Output

```text
Hello Alice
```

The syntax is shorter, clearer, and avoids the `||` problem.

---

# Syntax

General syntax:

```javascript
function functionName(parameter = defaultValue) {

}
```

Example:

```javascript
function multiply(number = 1) {

    return number * 10;

}
```

The parameter receives the default value **only if** no argument (or `undefined`) is supplied.

---

# Internal JavaScript Engine Behaviour

Let's examine what happens internally.

Function:

```javascript
function greet(name = "Guest") {

    console.log(name);

}
```

Call:

```javascript
greet();
```

Execution sequence:

```text
1. Create Function Execution Context

↓

2. Bind arguments to parameters

↓

No argument exists

↓

Parameter initially becomes undefined

↓

JavaScript detects a default value

↓

Parameter becomes "Guest"

↓

Execute function body
```

Notice something important.

The default value is **not** assigned immediately when the function is declared.

Instead, it is evaluated **each time the function is called**.

---

# Visual Representation

Before calling:

```text
Parameter

name

↓

(no value yet)
```

Call:

```javascript
greet();
```

Temporary binding:

```text
name

↓

undefined
```

Default value applied:

```text
name

↓

"Guest"
```

Function body executes:

```javascript
console.log(name);
```

Output:

```text
Guest
```

---

# What Happens When an Argument Is Provided?

Suppose we call:

```javascript
greet("Mary");
```

Binding:

```text
name

↓

"Mary"
```

Since a value already exists, JavaScript **does not use** the default.

Output:

```text
Mary
```

Default values are only used when the parameter value is:

```javascript
undefined
```

---

# Passing `undefined` Explicitly

Consider:

```javascript
function greet(name = "Guest") {

    console.log(name);

}
```

Call:

```javascript
greet(undefined);
```

Output:

```text
Guest
```

Why?

Because the parameter receives:

```javascript
undefined
```

and JavaScript treats that exactly the same as a missing argument.

---

# Passing `null`

Now consider:

```javascript
greet(null);
```

Output:

```text
null
```

Why wasn't `"Guest"` used?

Because:

```javascript
null !== undefined
```

The default value is **only** applied when the parameter value is `undefined`.

This is one of the most important rules to remember.

---

| Argument Passed | Parameter Value |
|-----------------|----------------|
| No argument | Default value |
| `undefined` | Default value |
| `"Alice"` | `"Alice"` |
| `0` | `0` |
| `false` | `false` |
| `""` | `""` |
| `null` | `null` |

Unlike the old `||` technique, valid falsy values are preserved.

# Advanced Default Parameters

In the previous section, we learned that default parameters provide fallback values whenever a parameter receives:

```javascript
undefined
```

That solved many problems.

However, JavaScript's default parameters are far more powerful than simply assigning a string or number.

A default value can be:

- Another parameter
- An expression
- A function call
- An object
- An array
- A computed value

To understand this, we first need to understand **when** JavaScript evaluates default parameters.

---

# When Are Default Parameters Evaluated?

Consider this function.

```javascript
function greet(name = "Guest") {
    console.log(name);
}
```

Many beginners think:

> "The default value is stored permanently when the function is created."

That is **not** what happens.

The default value is evaluated **every time the function is called**.

Think of the process like this:

```text
Function Declared
        │
        ▼
Function Stored in Memory
        │
        ▼
Function Called
        │
        ▼
Arguments Bound
        │
        ▼
Default Values Evaluated
        │
        ▼
Function Body Executes
```

Notice that evaluation happens **after** parameter binding but **before** the first statement inside the function.

---

# Demonstration

```javascript
let counter = 0;

function nextNumber(number = ++counter) {
    console.log(number);
}
```

Calls

```javascript
nextNumber();
nextNumber();
nextNumber();
```

Output

```text
1
2
3
```

Why?

Because

```javascript
++counter
```

is evaluated every time the function is invoked.

If JavaScript evaluated defaults only once during declaration, every call would print the same number.

It does not.

---

# Default Expressions

A default value is not limited to a literal.

It can be **any valid JavaScript expression**.

Example

```javascript
function calculateArea(
    width,
    height = width
) {
    return width * height;
}
```

Call

```javascript
calculateArea(8);
```

Binding

```text
width

↓

8

height

↓

undefined

↓

width

↓

8
```

Result

```text
64
```

The default value uses another parameter.

---

# Parameter Evaluation Order

JavaScript evaluates parameters **from left to right**.

This rule is extremely important.

Example

```javascript
function createRectangle(
    width,
    height = width
) {

}
```

Evaluation order

```text
width

↓

Receives argument

↓

height

↓

Uses width
```

This works because `width` already has a value.

---

# Reversing the Order

Now look carefully.

```javascript
function createRectangle(
    width = height,
    height = 5
) {

}
```

This is a problem.

When JavaScript tries to evaluate

```javascript
width = height
```

the parameter

```javascript
height
```

has **not yet been initialized**.

Depending on the exact code, this results in a runtime error because parameters are initialized from left to right.

---

# Mental Model

Imagine filling containers.

```
Container 1

↓

width

Container 2

↓

height
```

JavaScript fills them in order.

```
1

↓

2

↓

3

↓

4
```

It cannot use the second container before the second container exists.

---

# Example

```javascript
function introduce(
    firstName,
    lastName = "Unknown"
) {

    console.log(
        `${firstName} ${lastName}`
    );

}
```

Call

```javascript
introduce("Alice");
```

Binding

```
firstName

↓

Alice

lastName

↓

Unknown
```

Output

```text
Alice Unknown
```

---

# Default Values Can Call Functions

Suppose we have

```javascript
function generateReceiptNumber() {

    return "RCT-001";

}
```

Now

```javascript
function printReceipt(
    receiptNumber = generateReceiptNumber()
) {

    console.log(receiptNumber);

}
```

Call

```javascript
printReceipt();
```

Execution

```text
Parameter receives undefined

↓

Default function executes

↓

generateReceiptNumber()

↓

Returns

RCT-001

↓

receiptNumber becomes

RCT-001
```

Output

```text
RCT-001
```

Notice something interesting.

The function

```javascript
generateReceiptNumber()
```

only runs **when necessary**.

If an argument is supplied,

the default function is **never called**.

---

# Lazy Evaluation

This behavior is called **lazy evaluation**.

Lazy evaluation means:

> Perform work only when it is actually needed.

Example

```javascript
function expensiveCalculation() {

    console.log("Calculating...");

    return 100;

}

function showTotal(
    total = expensiveCalculation()
) {

    console.log(total);

}
```

Call

```javascript
showTotal(50);
```

Output

```text
50
```

Notice

```
Calculating...
```

never appears.

Why?

Because JavaScript already has a value.

The default expression is skipped.

---

Now call

```javascript
showTotal();
```

Output

```text
Calculating...
100
```

This time the default expression was needed.

Imagine a report generator.

# Rest Parameters (`...args`)

So far, every function we've written has expected a **fixed number of arguments**.

For example:

```javascript
function add(a, b) {
    return a + b;
}
```

This function expects exactly **two** values.

```javascript
add(10, 20);
```

works perfectly.

But what if we want to add:

- 2 numbers
- 5 numbers
- 20 numbers
- 1,000 numbers

Should we write functions like these?

```javascript
add2(a, b);

add3(a, b, c);

add4(a, b, c, d);

add5(a, b, c, d, e);
```

Of course not.

This would become impossible to maintain.

We need a way for a function to accept **any number of arguments**.

That is the purpose of **Rest Parameters**.

---

# Why Were Rest Parameters Introduced?

Before ES6, JavaScript developers often relied on a special object called:

```javascript
arguments
```

Although useful, it had several limitations:

- It is **array-like**, not a real array.
- It cannot directly use array methods like `map()`, `filter()`, or `reduce()`.
- It behaves differently in arrow functions.
- It makes function signatures less explicit.

ES6 introduced **Rest Parameters** as a cleaner, more modern solution.

---

# What Is a Rest Parameter?

A **Rest Parameter** collects **all remaining arguments** into a single **real array**.

General syntax:

```javascript
function functionName(...parameterName) {

}
```

The three dots (`...`) are called the **rest operator** in this context.

Example:

```javascript
function showNumbers(...numbers) {
    console.log(numbers);
}
```

Calling:

```javascript
showNumbers(10, 20, 30, 40);
```

Output:

```text
[10, 20, 30, 40]
```

Notice something remarkable.

Instead of creating four separate parameters, JavaScript creates **one array**.

---

# Internal Engine Behaviour

Consider:

```javascript
function showNumbers(...numbers) {
    console.log(numbers);
}

showNumbers(1, 2, 3, 4);
```

Execution steps:

### Step 1

Create Function Execution Context.

---

### Step 2

Collect every supplied argument.

```text
1

2

3

4
```

---

### Step 3

Create a new array.

```text
numbers

↓

[1, 2, 3, 4]
```

---

### Step 4

Execute the function body.

```javascript
console.log(numbers);
```

Output:

```text
[1, 2, 3, 4]
```

---

# Mental Model

Imagine a basket.

Without rest parameters:

```text
Item 1

↓

Parameter 1

Item 2

↓

Parameter 2

Item 3

↓

Parameter 3
```

With rest parameters:

```text
Item 1

↓

Basket

Item 2

↓

Basket

Item 3

↓

Basket

Item 4

↓

Basket

↓

One Array
```

Every remaining argument is placed into the same basket.

---

# Why Is It Called "Rest"?

Because it collects **the rest** of the arguments.

Example:

```javascript
function example(first, second, ...others) {

    console.log(first);

    console.log(second);

    console.log(others);

}
```

Call:

```javascript
example(
    10,
    20,
    30,
    40,
    50
);
```

Binding:

```text
first

↓

10

----------------

second

↓

20

----------------

others

↓

[30, 40, 50]
```

Output:

```text
10

20

[30, 40, 50]
```

---

# Rest Parameter Must Be Last

This is one of the most important syntax rules.

Correct:

```javascript
function example(a, b, ...rest) {

}
```

Incorrect:

```javascript
function example(...rest, last) {

}
```

This produces a syntax error.

Why?

Imagine JavaScript trying to bind:

```javascript
example(1, 2, 3, 4);
```

Should `rest` collect:

```text
[1]
```

or

```text
[1, 2]
```

or

```text
[1, 2, 3]
```

There would be no way to know where to stop.

Therefore, JavaScript requires the rest parameter to be the **final parameter**.

# Example 1 — Beginner

Goal:

Display every supplied name.

```javascript

    console.log(names);

}

    "Alice",
    "John",
    "Mary"

```

Output:

```text

  "Alice",
  "John",
```

# The `arguments` Object

Before **ES6 (ECMAScript 2015)** introduced **Rest Parameters**, JavaScript already had a way to access all arguments passed to a function.

That mechanism is called the **`arguments` object**.

Many legacy JavaScript codebases still use it, so understanding it is important even if modern code often prefers rest parameters.

---

# What Is the `arguments` Object?

The `arguments` object is a special object that JavaScript automatically creates for **every regular function**.

It contains **all arguments passed** to that function, regardless of how many parameters were declared.

Example:

```javascript
function showArguments() {
    console.log(arguments);
}

showArguments("Alice", 25, true);
```

Output (browser-dependent formatting):

```text
Arguments(3) ["Alice", 25, true]
```

Unlike rest parameters, you do **not** declare `arguments`.

JavaScript provides it automatically.

---

# Internal JavaScript Engine Behaviour

Consider:

```javascript
function greet(name) {
    console.log(arguments);
}

greet("Alice");
```

Internally, JavaScript performs something conceptually like this:

```text
Create Function Execution Context

↓

Bind Parameters

↓

Create arguments object

↓

arguments[0] = "Alice"

↓

Execute Function Body
```

Inside the execution context, you now have:

```text
Execution Context

├── Parameter
│     name = "Alice"
│
├── arguments
│     {
│       0: "Alice",
│       length: 1
│     }
│
└── Local Variables
```

Notice that **both** the parameter and the `arguments` object exist at the same time.

---

# Visual Representation

Function

```javascript
function add(a, b) {

}
```

Call

```javascript
add(10, 20);
```

Memory conceptually becomes:

```text
Parameters

a

↓

10

b

↓

20

------------------------

arguments

↓

{
   0: 10,
   1: 20,
   length: 2
}
```

The same information is accessible in two different ways.

---

# Accessing Individual Arguments

Example

```javascript
function showValues() {

    console.log(arguments[0]);
    console.log(arguments[1]);
    console.log(arguments[2]);

}

showValues(
    "Alice",
    "John",
    "Mary"
);
```

Output

```text
Alice

John

Mary
```

The indexing starts at **0**, just like arrays.

---

# The `length` Property

The `arguments` object knows how many arguments were supplied.

Example

```javascript
function countArguments() {

    console.log(arguments.length);

}

countArguments(
    10,
    20,
    30,
    40
);
```

Output

```text
4
```

This was one of the primary ways developers handled variable-length argument lists before rest parameters existed.

---

# Iterating Over `arguments`

Although `arguments` is **not** a real array, you can still loop through it.

```javascript
function printArguments() {

    for (let i = 0; i < arguments.length; i++) {

        console.log(arguments[i]);

    }

}

printArguments(
    "Pen",
    "Book",
    "Bag"
);
```

Output

```text
Pen

Book

Bag
```

---

# Is `arguments` an Array?

This is one of the most common interview questions.

The answer is:

> **No.**

The `arguments` object is **array-like**, not an actual array.

It looks similar:

```text
{
    0: "Alice",
    1: "John",
    2: "Mary",
    length: 3
}
```

But it does **not** inherit from `Array.prototype`.

That means many array methods are unavailable.

Example:

```javascript
function demo() {
    arguments.map(value => value);
}
```

Output:

```text
TypeError: arguments.map is not a function
```

Why?

Because `map()` belongs to arrays, not to the `arguments` object.

---

# Converting `arguments` to a Real Array

Before ES6, developers often converted `arguments` into a real array.

One common approach was:

```javascript
function demo() {

    const values =
        Array.from(arguments);

    console.log(values);

}

demo(1, 2, 3);
```

Output

```text
[1, 2, 3]
```

Now `values` is a genuine array, so methods like `map()`, `filter()`, and `reduce()` become available.

```text
  "John"
]
```

### Explanation

# `arguments` vs Rest Parameters

Now that you've learned both, let's compare them.

| Feature | `arguments` | Rest Parameters (`...args`) |
|---------|-------------|-----------------------------|
| Available automatically | ✅ Yes | ❌ No (must be declared) |
| Real array | ❌ No | ✅ Yes |
| Supports `map()`, `filter()`, `reduce()` directly | ❌ No | ✅ Yes |
| Works in arrow functions | ❌ No | ✅ Yes |
| Modern recommended approach | ❌ Usually no | ✅ Yes |
| Available in legacy code | ✅ Yes | ❌ Only ES6+ |

---

# Why Modern JavaScript Prefers Rest Parameters

Suppose we write:

```javascript
function total(...numbers) {

    return numbers.reduce(
        (sum, value) => sum + value,
        0
    );

}
```

Because `numbers` is already an array, we can immediately use array methods.

Doing the same with `arguments` requires conversion first.

This makes rest parameters:

- More readable.
- More expressive.
- Easier to maintain.
- Better integrated with modern JavaScript features.

---

# `arguments` and Arrow Functions

One of the biggest differences is that **arrow functions do not have their own `arguments` object**.

Example:

```javascript
const demo = () => {
    console.log(arguments);
};
```

Running this in most contexts results in a reference error or accesses an outer function's `arguments`, because arrow functions do **not** create their own execution context in the same way regular functions do.

If you need all supplied values in an arrow function, use a rest parameter instead:

```javascript
const demo = (...values) => {
    console.log(values);
};
```

---

# Common Mistakes

## Mistake 1 — Treating `arguments` as an Array

Incorrect:

```javascript
function demo() {
    arguments.push(10);
}
```

Output

```text
TypeError
```

`push()` is an array method.

`arguments` is not an array.

---

## Mistake 2 — Using `arguments` Inside Arrow Functions

Incorrect:

```javascript
const greet = () => {
    console.log(arguments);
};
```

Use:

```javascript
const greet = (...args) => {
    console.log(args);
};
```

---

# Best Practices

- Prefer **rest parameters** in modern JavaScript.
- Learn `arguments` because you will encounter it in older codebases.
- Convert `arguments` to an array only when maintaining legacy code.
- Avoid mixing both approaches unnecessarily in the same function.
- When writing new code, choose the approach that makes your function signature clear and intentional.

> **Professional Tip:** If you're starting a new project today, reach for rest parameters first. Reserve `arguments` mainly for understanding or maintaining existing JavaScript code.

# Rest Parameters vs Spread Syntax (`...`)

One of the biggest sources of confusion for JavaScript developers is the `...` syntax.

You have already learned that:

```javascript
function total(...numbers) {

}
```

uses **Rest Parameters**.

Later, you'll also see code like this:

```javascript
const numbers = [10, 20, 30];

console.log(...numbers);
```

or

```javascript
const copy = [...numbers];
```

This looks identical.

The same three dots appear.

Yet they perform a completely different job.

Understanding this distinction is essential.

---

# One Symbol, Two Meanings

The three dots (`...`) represent **two different language features** depending on where they are used.

| Location | Meaning |
|----------|----------|
| Function parameter list | Rest Parameter |
| Function call, array literal, object literal | Spread Syntax |

The symbol is the same.

The behavior is different.

A useful way to remember this is:

> **Rest collects. Spread expands.**

---

# Mental Model

Imagine a basket filled with apples.

```text

```

### Rest Parameter

Rest **collects** apples into one basket.

```text

↓

↓

```

Result

```text

↓

🍎
]
```

---

### Spread Syntax

Spread does the opposite.

It takes the basket

```text
[
🍎
🍎
🍎
]
```

and pours the apples back out.

```text
Basket

↓

🍎 🍎 🍎
```

Rest gathers.

Spread separates.

Exactly the opposite.

Because both operations are conceptually related.

| Rest Parameter | Spread Syntax |
|---------------|---------------|
| Collects values | Expands values |
| Used in function parameters | Used in function calls, arrays, objects |
| Produces one array | Produces individual values |
| Appears once and must be the last parameter | Can appear in many positions where expansion is allowed |
| Makes variable-length parameter lists easy | Makes passing arrays and copying data easy |

## Mistake 1 — Confusing Rest with Spread

Incorrect assumption:

```javascript

```

# Performance Considerations

Rest parameters allocate a new array to hold the collected arguments.

Spread syntax iterates over an iterable (such as an array) and expands its elements.

In ordinary application code, both are highly optimized by modern JavaScript engines.

Choose the construct that makes your intent the clearest rather than trying to micro-optimize.

---

Understanding **collect vs expand** makes these patterns much easier to read.

- A simple memory aid is:


- Rest is used in **function parameter lists**.
- Spread is used in **function calls** and other contexts where expansion is needed.

# Function Parameter Destructuring

Until now, every function we've written has received data like this:

```javascript

```

This works well for small amounts of data.

However, modern JavaScript applications rarely pass many unrelated values individually.

Instead, they often pass **objects** and **arrays**.

For example, imagine a user object.

```javascript
const user = {

    firstName: "Alice",

    lastName: "Johnson",

    age: 25,

    role: "Admin"

};
```

Instead of passing four separate arguments,

we can pass one object.

```javascript
createUser(user);
```

But then another question arises:

> **How do we easily access the object's properties inside the function?**

The answer is **parameter destructuring**.

---

# Why Parameter Destructuring Exists

Suppose we write:

```javascript
function displayUser(user) {

    console.log(user.firstName);

    console.log(user.lastName);

    console.log(user.age);

    console.log(user.role);

}
```

This works.

However, notice the repetition.

Every property access begins with

```javascript
user.
```

As objects become larger,

the repetition becomes worse.

Developers wanted a cleaner way.

---

# What is Destructuring?

Destructuring is a JavaScript feature that extracts values from:

- Objects
- Arrays

and stores them into variables.

Instead of writing

```javascript
const firstName = user.firstName;

const lastName = user.lastName;

const age = user.age;
```

we can write

```javascript
const {

    firstName,

    lastName,

    age

} = user;
```

The result is identical.

But the code is shorter and easier to read.

---

# Parameter Destructuring

Now let's move the destructuring directly into the parameter list.

```javascript
function displayUser({

    firstName,

    lastName,

    age

    console.log(age);

}
```

```javascript
displayUser(user);
```

Output

```text

Johnson

25
```

Notice something amazing.

There is **no** variable called

```javascript
user
```

inside the function.

Instead,

JavaScript immediately extracts the required properties.

```text
↓

Read Property

↓

↓
```

```javascript
firstName
```

is the property.

```javascript
name
```

is the local variable.

Output

Since the object has no

property,

# Common Mistakes

## Mistake 1 — Passing the Wrong Type

Function

```javascript

    firstName

}) {

}
```

Incorrect

```javascript

```

A string doesn't have a `firstName` property.

The function expects an object.

---

## Mistake 2 — Forgetting Property Names

Incorrect

```javascript
function greet({

    name

}) {

}
```

when the object actually contains

```javascript

```

Property names must match unless you explicitly rename them.

---

## Mistake 3 — Destructuring Too Much

Avoid extracting dozens of properties that the function never uses.

Extract only what is necessary.

---

# Best Practices

- Pass objects when a function requires many related values.
- Destructure only the properties you need.
- Use default values for optional properties.
- Rename properties only when it improves readability.
- Prefer object parameters over long parameter lists in large applications.

---

# Performance Considerations

Object destructuring introduces a very small amount of work because JavaScript must read the requested properties before executing the function body.

In practice, the performance impact is negligible compared to the readability and maintainability benefits.

Choose destructuring because it clearly communicates the data your function depends on.

# Passing Values to Functions

One of the biggest misconceptions in JavaScript is:

> **"Objects are passed by reference."**

You'll hear this statement frequently.

It sounds correct.

It is repeated in blogs, videos, and interviews.

However...

**It is not technically correct.**

To understand why, we must first understand how JavaScript stores values in memory.

This lesson is one of the most important in all of JavaScript because it explains why changing some values inside a function affects the original variable while changing others does not.

If you truly understand this section, concepts like:

- Objects
- Arrays
- Closures
- React State
- Redux
- Vue Reactivity
- Immutable Programming

become much easier to understand.

---

# JavaScript Has Two Categories of Values

Every value in JavaScript belongs to one of two categories.

## Primitive Values

Primitives are simple values.

They include:

```text
String

Number

Boolean

Undefined

Null

BigInt

Symbol
```

Example

```javascript

const name = "Alice";

const active = true;
```

These values behave independently.

---

## Reference Values (Objects)

Objects include

```text
Objects

Arrays

Functions

Dates

Maps

Sets

Regular Expressions
```

Example

```javascript
const user = {

    name: "Alice"

};
```

An array

```javascript
const numbers = [

    10,

    20,

    30

];
```

A function

```javascript
function greet() {

}
```

These are all **objects** internally.

---

# Memory Model

Let's first understand primitives.

Example

```javascript
let age = 25;
```

Conceptually

```text
Memory

age

↓

25
```

The variable directly stores the value.

---

Now an object

```javascript
const user = {

    name: "Alice"

};
```

Conceptually

```text
Memory

user

↓

Address A

-----------------

Address A

↓

{
   name: "Alice"
}
```

Notice the difference.

The variable does **not** directly contain the object.

Instead,

it contains a reference (memory address) to where the object lives.

This distinction is crucial.

---

# Primitive Arguments

Suppose

```javascript

    number++;

    console.log(number);

}
```

Call

```javascript
let age = 20;

console.log(age);
```

What happens?

---

## Internal Behaviour

Initial memory

```text
age

↓

20
```

Function call

```javascript

```

JavaScript copies

```
20
```

into the parameter.

Execution Context

```text
number

↓

20
```

Now

```javascript
number++;
```

becomes

```text
21
```

The local parameter changes.

```
number

↓

21
```

But

```
age

↓

20
```

remains unchanged.

Output

```text
21

20
```

---

# Why?

Because primitives are copied.

Think of a photocopy.

Original

```
20
```

Photocopy

```
20
```

Writing on the photocopy

```

```

does not change the original paper.

---

# Mental Model

```
↓

↓

↓

Original

↓

```

# Example 1



The original value remains unchanged.

# Object Arguments

Objects behave differently.

Consider

```javascript
function updateUser(user) {

    user.name = "John";

}
```

Call

```javascript
const person = {

    name: "Alice"

};

updateUser(person);

console.log(person);
```

Output

```text
{
    name: "John"
}
```

Many beginners immediately conclude:

> "Objects are passed by reference."

But let's examine what actually happened.

---

# What Really Happened?

Initial memory

```text
person

↓

Address A

---------------

Address A

↓

{
   name: "Alice"
}
```

Function call

```javascript
updateUser(person);
```

JavaScript copies

```
Address A
```

not the object.

Execution Context

```text
user

↓

Address A
```

Notice carefully.

Both variables now point to the same object.

```
person

↓

Address A

user

↓

Address A
```

Changing

```javascript
user.name
```

changes the object located at Address A.

Therefore

```javascript
person.name
```

also appears changed.

---

# Important Conclusion

JavaScript did **not** copy the object.

It copied the reference.

Therefore,

JavaScript is still passing **by value**.

The value simply happens to be a reference.

This is why many experienced JavaScript developers prefer saying:

> **JavaScript passes object references by value.**

This wording is technically accurate.

---

# Visual Diagram

Before call

```text

↓

↓

   name: "Alice"
}
```

Function call

```javascript

```

```text

↓

Address A

user

↓

Address A
```

One object.

Two variables.

---

# Changing Properties

```javascript

```

The object changes.

```text
Address A

↓

{
   name: "John"
}
```

Since both variables point to the same object,

both observe the change.

---

# Reassigning the Parameter

Now consider

```javascript

    user = {

        name: "John"

    };

}
```

```javascript
    name: "Alice"

```

Output

```text

}
```

Why?

Because

```javascript

```

does **not** modify the original object.

Instead,

it changes which object the local parameter references.

Memory

Initially

```text
person

↓

Address A
```

Parameter

```text
user

↓

Address A
```

Reassignment

```javascript

    name: "John"

};
```

Now

```text

↓

Address B
```

while

```text
person

↓

Address A
```

The original variable still points to Address A.

---

# Key Rule

Changing

```javascript

```

changes the shared object.

Changing

```javascript

```

only changes the local parameter.

This distinction is one of the most frequently tested topics in JavaScript interviews.

# Object Mutation, Defensive Copying, and Immutable Programming

In the previous lesson, we learned an important fact:

> JavaScript **always passes values** to functions.

For objects and arrays, the value being copied is **the reference (memory address)**, not the object itself.

Because multiple variables can point to the same object, modifying that object through one variable affects what every other variable sees.

This behavior is called **mutation**.

Understanding mutation is essential because it influences how you design applications, avoid bugs, and work with modern frameworks like React, Vue, Angular, and state-management libraries.

---

# What Is Mutation?

A **mutation** is any operation that changes an existing object or array.

Example:

```javascript
const user = {
    name: "Alice",
    age: 25
};

user.age = 26;
```

The object itself has changed.

Before mutation:

```text
{
    name: "Alice",
    age: 25
}
```

After mutation:

```text
{
    name: "Alice",
    age: 26
}
```

Notice that we did **not** create a new object.

We modified the existing one.

---

# Internal Memory Representation

Initially

```text
user

↓

Address A

↓

{
    name: "Alice",
    age: 25
}
```

After

```javascript
user.age = 26;
```

Memory becomes

```text
user

↓

Address A

↓

{
    name: "Alice",
    age: 26
}
```

The address did not change.

Only the contents stored at that address changed.

---

# Mutation Inside Functions

Example:

```javascript
function celebrateBirthday(user) {

    user.age++;

}

const person = {
    name: "Alice",
    age: 25
};

celebrateBirthday(person);

console.log(person);
```

Output

```text
{
    name: "Alice",
    age: 26
}
```

### What happened?

Parameter binding

```text
person

↓

Address A

user

↓

Address A
```

Both variables refer to the same object.

When we execute

```javascript
user.age++;
```

JavaScript updates the object at Address A.

Therefore:

- `user.age` becomes `26`
- `person.age` also becomes `26`

because both references point to the same object.

---

# Arrays Can Also Be Mutated

Example

```javascript
function addProduct(products) {

    products.push("Notebook");

}

const cart = [
    "Pen",
    "Book"
];

addProduct(cart);

console.log(cart);
```

Output

```text
[
    "Pen",
    "Book",
    "Notebook"
]
```

The `push()` method changes the original array.

---

# Common Mutating Array Methods

These methods modify the existing array:

```javascript
push()

pop()

shift()

unshift()

splice()

sort()

reverse()

fill()

copyWithin()
```

Example

```javascript
const numbers = [3, 1, 2];

numbers.sort();

console.log(numbers);
```

Output

```text
[1, 2, 3]
```

The original array has changed.

---

# Non-Mutating Methods

Other methods return a new value instead of changing the original array.

Examples include:

```javascript
map()

filter()

reduce()

slice()

concat()

toSorted()

toReversed()

toSpliced()
```

Example

```javascript
const numbers = [3, 1, 2];

const sorted = numbers.toSorted();

console.log(numbers);
console.log(sorted);
```

Output

```text
[3, 1, 2]

[1, 2, 3]
```

The original array remains unchanged.

> **Note:** `toSorted()`, `toReversed()`, and `toSpliced()` are newer additions to JavaScript that return new arrays instead of mutating the original.

---

# Why Mutation Can Become Dangerous

Imagine two different parts of an application sharing the same object.

```javascript
const user = {
    name: "Alice",
    role: "Admin"
};
```

Function A:

```javascript
displayProfile(user);
```

Function B:

```javascript
changeRole(user);
```

If `changeRole()` mutates the object,

`displayProfile()` now sees different data.

Diagram

```text
Display Profile

↓

Shared Object

↑

Change Role
```

One function unexpectedly affects another.

This creates **hidden side effects**.

---

# Hidden Side Effects

A side effect is any observable change outside the function itself.

Example

```javascript
function changeName(user) {

    user.name = "John";

}
```

The function appears to change only its parameter.

In reality, it changes data owned by the caller.

This makes debugging harder because changes occur indirectly.

---

# Defensive Copying

One way to avoid accidental mutation is to make a copy.

Instead of changing the original object,

work on a duplicate.

This technique is called **defensive copying**.

---

# Copying with the Spread Operator

Example

```javascript
const original = {
    name: "Alice",
    age: 25
};

const copy = {
    ...original
};

copy.age = 30;

console.log(original);
console.log(copy);
```

Output

```text
{
    name: "Alice",
    age: 25
}

{
    name: "Alice",
    age: 30
}
```

The original object is untouched.

---

# Internal Representation

Initially

```text
original

↓

Address A

↓

{
    age: 25
}
```

Copy

```javascript
const copy = {
    ...original
};
```

Creates

```text
original

↓

Address A

↓

{
    age: 25
}

--------------------

copy

↓

Address B

↓

{
    age: 25
}
```

Two separate objects now exist.

Changing one does not affect the other.

---

# Shallow Copy

The spread operator performs a **shallow copy**.

This means it copies only the first level of properties.

Example

```javascript
const user = {
    name: "Alice",
    address: {
        city: "Nairobi"
    }
};

const copy = {
    ...user
};
```

Memory conceptually becomes:

```text
user

↓

Address A

↓

{
    name

    address

        ↓

        Address C
}

copy

↓

Address B

↓

{
    name

    address

        ↓

        Address C
}
```

Notice:

The outer objects are different.

The nested `address` object is still shared.

---

# Demonstration

```javascript
copy.address.city = "Mombasa";

console.log(user.address.city);
```

Output

```text
Mombasa
```

Even though the outer object was copied,

the nested object remained shared.

This is why it's called a **shallow** copy.

---

# Deep Copy

A **deep copy** duplicates:

- The object
- Nested objects
- Nested arrays

Every level is copied.

Conceptually

```text
Original

↓

Object A

↓

Nested Object A

---------------------

Copy

↓

Object B

↓

Nested Object B
```

Now changes cannot leak between them.

Modern JavaScript provides:

```javascript
structuredClone()
```

Example

```javascript
const copy = structuredClone(original);
```

This creates a deep copy for many common JavaScript data structures.

---

# Immutable Programming

**Immutable** means:

> Once created, a value is never changed.

Instead of mutating:

```javascript
user.age = 30;
```

create a new object:

```javascript
const updatedUser = {
    ...user,
    age: 30
};
```

Original

```text
{
    age: 25
}
```

New

```text
{
    age: 30
}
```

Both objects exist independently.

This style is heavily used in:

- React
- Redux
- Zustand
- NgRx
- MobX (depending on configuration)
- Many functional programming techniques

---

# Object.freeze()

JavaScript allows us to prevent accidental mutation.

Example

```javascript
const settings = Object.freeze({
    theme: "dark"
});
```

Attempt

```javascript
settings.theme = "light";
```

In strict mode, this throws an error.

Otherwise, the assignment fails silently.

The object remains unchanged.

Output

```text
{
    theme: "dark"
}
```

> **Important:** `Object.freeze()` is shallow. Nested objects can still be modified unless they are frozen separately.

---

# Real-World Example

Suppose your shopping cart is displayed on multiple pages.

Instead of mutating the original cart,

return a new version.

```javascript
function addItem(cart, item) {

    return [
        ...cart,
        item
    ];

}
```

Usage

```javascript
const newCart = addItem(cart, "Notebook");
```

Now:

- `cart` remains unchanged.
- `newCart` contains the additional item.

This approach avoids unexpected side effects.

---

# Professional Best Practices

- Avoid mutating function arguments unless the function is explicitly designed to do so.
- Prefer returning new objects or arrays when updating data.
- Use shallow copies for flat objects.
- Use `structuredClone()` or another deep-copy strategy when nested data must be independent.
- Clearly document functions that intentionally mutate their inputs.

# Common Interview Questions

### Q1. Are objects passed by reference?

**Answer:**

Not exactly.

JavaScript passes **the reference by value**.

The object itself is not copied, but neither is the original reference variable shared.

A copy of the reference is passed.

---

### Q2. What is mutation?

Mutation is changing an existing object or array instead of creating a new one.

---

### Q3. What is the difference between a shallow copy and a deep copy?

- **Shallow copy:** Copies only the first level.
- **Deep copy:** Copies every nested level.

---

### Q4. Why is immutable programming popular?

Because it:

- Reduces hidden side effects.
- Makes debugging easier.
- Simplifies state management.
- Makes changes easier to track.

```text

        ▼

        │
        ▼

        │
        ▼
Inspect Outputs
        │
        ▼

        │
        ▼
Find Root Cause
        │
        ▼

        │
```

Never guess.

Always investigate.

# Technique 7 — Inspecting the Call Stack


During debugging,

the Call Stack appears like:

```text
third()

↓

second()

↓

first()

↓

Global
```

This shows exactly how execution reached the current line.

Understanding the call stack is invaluable when debugging large applications.

# Technique 8 — Validating Parameters Early

Professional functions validate inputs immediately.

Example

```javascript
function divide(a, b) {

    if (typeof a !== "number") {

        throw new Error("a must be a number");

    }

    if (typeof b !== "number") {

        throw new Error("b must be a number");

    }

    if (b === 0) {

        throw new Error("Cannot divide by zero");

    }

    return a / b;

}
```

Benefits:

- Errors occur near their source.
- Bugs are easier to identify.
- Invalid data doesn't spread through the application.

# Beginner Interview Questions

### 1. What is the difference between a parameter and an argument?

**Answer**

A parameter is a variable declared in the function definition.

An argument is the actual value supplied when the function is called.

---

### 2. What happens if a function is called without enough arguments?

**Answer**

Missing parameters receive the value:

```javascript
undefined
```

unless a default parameter exists.

---

### 3. What is a default parameter?

**Answer**

A fallback value used when the parameter receives `undefined`.

---

### 4. What is a rest parameter?

**Answer**

A parameter that collects remaining arguments into a real array.

---

### 5. What is the `arguments` object?

**Answer**

A special array-like object available inside regular functions that contains every supplied argument.

---

# Intermediate Interview Questions

### 1. Why are rest parameters preferred over `arguments`?

**Answer**

Because rest parameters:

- Produce real arrays.
- Work with array methods.
- Work in arrow functions.
- Are easier to read.
- Clearly express intent.

---

### 2. What is parameter destructuring?

**Answer**

A technique for extracting object or array values directly into function parameters.

---

### 3. When are default parameters evaluated?

**Answer**

Each time the function is called—not when it is declared.

---

### 4. Can one default parameter depend on another?

**Answer**

Yes, provided the referenced parameter appears earlier in the parameter list.

Example

```javascript
function square(width, height = width) {

}
```

---

### 5. What happens when more arguments than parameters are supplied?

**Answer**

Named parameters receive values from left to right.

Extra arguments remain accessible through rest parameters or the `arguments` object.

---

# Advanced Interview Questions

### 1. Are objects passed by reference?

**Answer**

No.

JavaScript passes the **reference by value**.

The object itself is not copied, but the copied reference still points to the same object.

---

### 2. Explain shallow vs deep copying.

**Answer**

- Shallow copy duplicates only the first level.
- Nested objects remain shared.
- Deep copy duplicates every nested level.

---

### 3. Why is immutable programming important?

**Answer**

Because it:

- Prevents accidental mutations.
- Reduces hidden side effects.
- Simplifies debugging.
- Makes state updates predictable.

---

### 4. Why can't the rest parameter appear before another parameter?

**Answer**

Because JavaScript would not know where the "rest" should stop collecting arguments.

Therefore it must always be the final parameter.

---

### 5. Explain the difference between Rest and Spread.

**Answer**

Rest collects multiple values into one array.

Spread expands one iterable into multiple values.
