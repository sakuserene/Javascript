# 1. Introduction

A **function** is a reusable block of JavaScript code designed to perform a particular task.

Instead of writing the same instructions repeatedly, you can place those instructions inside a function and execute them whenever they are needed.

A function can:

* Receive information.
* Process information.
* Produce a result.
* Change application state.
* display information.
* Validate user input.
* communicate with an API.
* respond to an event.
* create HTML elements.
* perform calculations.
* coordinate other functions.

A simple function looks like this:

```javascript
function greet() {
  console.log("Hello!");
}
```

The function does not execute merely because it has been declared.

To execute it, you must **call** or **invoke** it:

```javascript
greet();
```

Output:

```text
Hello!
```

> [!IMPORTANT]
> Declaring a function defines what should happen. Calling the function tells JavaScript to perform those instructions.

# 2. Why Functions Exist

Programs are made of instructions.

As programs grow, the number of instructions increases. Without a way to organize and reuse instructions, code quickly becomes repetitive, difficult to understand, and difficult to maintain.

Consider a program that calculates the total cost of three customer orders.

Without functions:

```javascript
const firstPrice = 100;
const firstQuantity = 3;
const firstTotal = firstPrice * firstQuantity;
console.log(firstTotal);

const secondPrice = 250;
const secondQuantity = 2;
const secondTotal = secondPrice * secondQuantity;
console.log(secondTotal);

const thirdPrice = 80;
const thirdQuantity = 5;
const thirdTotal = thirdPrice * thirdQuantity;
console.log(thirdTotal);
```

Output:

```text
300
500
400
```

The multiplication operation is repeated three times.

A function allows us to define the operation once:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}

console.log(calculateTotal(100, 3));
console.log(calculateTotal(250, 2));
console.log(calculateTotal(80, 5));
```

Output:

```text
300
500
400
```

The function version is:

* Shorter.
* Easier to read.
* Easier to test.
* Easier to modify.
* Easier to reuse.
* Less likely to contain inconsistent logic.

# 3. The Problem Functions Solve

Functions solve several fundamental software-development problems.

## 3.1 Repetition

Without functions, developers repeat the same instructions.

```javascript
const subtotal1 = 1000;
const tax1 = subtotal1 * 0.16;
const total1 = subtotal1 + tax1;

const subtotal2 = 2500;
const tax2 = subtotal2 * 0.16;
const total2 = subtotal2 + tax2;
```

The tax calculation appears more than once.

With a function:

```javascript
function calculateTotalWithTax(subtotal) {
  const tax = subtotal * 0.16;
  return subtotal + tax;
}

const total1 = calculateTotalWithTax(1000);
const total2 = calculateTotalWithTax(2500);
```

## 3.2 Poor organization

A large program without functions becomes one long sequence of instructions.

```javascript
const username = "Saku";
const password = "secret123";

if (username.length < 3) {
  console.log("Invalid username");
}

if (password.length < 8) {
  console.log("Invalid password");
}

const subtotal = 5000;
const tax = subtotal * 0.16;
const total = subtotal + tax;

console.log(total);
```

Functions divide the program into meaningful units:

```javascript
function validateUsername(username) {
  return username.length >= 3;
}

function validatePassword(password) {
  return password.length >= 8;
}

function calculateTotalWithTax(subtotal) {
  return subtotal + subtotal * 0.16;
}
```

## 3.3 Difficult maintenance

Suppose tax changes from `16%` to `17%`.

If the formula was copied to twenty locations, all twenty locations must be updated.

If the formula exists in one function, only one place needs to change:

```javascript
function calculateTotalWithTax(subtotal) {
  return subtotal + subtotal * 0.17;
}
```

## 3.4 Difficult testing

It is difficult to test logic buried inside a large application.

Functions isolate logic:

```javascript
function isAdult(age) {
  return age >= 18;
}
```

The function can be tested with several values:

```javascript
console.log(isAdult(20)); // true
console.log(isAdult(18)); // true
console.log(isAdult(15)); // false
```

## 3.5 Lack of abstraction

An **abstraction** hides unnecessary implementation details behind a meaningful name.

Compare:

```javascript
const finalAmount = subtotal - subtotal * discountRate;
```

with:

```javascript
const finalAmount = calculateDiscountedPrice(subtotal, discountRate);
```

The second version explains the intention more clearly.

# 4. How Reusable Logic Was Handled Without Functions

Before understanding why functions are useful, consider the alternatives.

## 4.1 Repeating instructions manually

```javascript
console.log("Welcome, Alice");
console.log("Welcome, Bob");
console.log("Welcome, Carol");
```

This works for a tiny program but does not scale.

## 4.2 Copying and modifying code

```javascript
const aliceMessage = "Welcome, Alice";
console.log(aliceMessage);

const bobMessage = "Welcome, Bob";
console.log(bobMessage);
```

Copying code creates duplication. A mistake fixed in one copy may remain in another.

## 4.3 Using loops without functions

Loops solve repeated iteration, but they do not replace functions.

```javascript
const names = ["Alice", "Bob", "Carol"];

for (const name of names) {
  console.log(`Welcome, ${name}`);
}
```

This loop repeats an operation over a collection.

A function makes the operation independently reusable:

```javascript
function createWelcomeMessage(name) {
  return `Welcome, ${name}`;
}

const names = ["Alice", "Bob", "Carol"];

for (const name of names) {
  console.log(createWelcomeMessage(name));
}
```

> [!NOTE]
> Loops answer the question, “How many times should this operation run?” Functions answer the question, “What operation should be performed?”

# 5. Mental Model

Think of a function as a machine.

```text
                FUNCTION MACHINE

Input              Processing               Output
  │                    │                        │
  ▼                    ▼                        ▼
price = 100      price × quantity             300
quantity = 3
```

A function can be represented as:

```text
Input → Function → Output
```

For example:

```javascript
function multiply(price, quantity) {
  return price * quantity;
}
```

```text
100 and 3
    ↓
multiply()
    ↓
100 × 3
    ↓
300
```

Not every function must receive input.

Not every function must return useful output.

```javascript
function showWelcomeMessage() {
  console.log("Welcome!");
}
```

```text
No input
   ↓
showWelcomeMessage()
   ↓
Print message
   ↓
undefined
```

A more complete model is:

```text
┌──────────────────────────────────────────┐
│ Function                                 │
│                                          │
│  1. Receives zero or more inputs         │
│  2. Creates a local execution context    │
│  3. Executes instructions                │
│  4. May produce side effects             │
│  5. Returns exactly one value            │
└──────────────────────────────────────────┘
```

# 6. Function Terminology

Understanding the terminology is essential.

| Term              | Meaning                                                                    |
| ----------------- | -------------------------------------------------------------------------- |
| Function          | A reusable block of code designed to perform a task                        |
| Declaration       | The code that defines a function                                           |
| Function name     | The identifier used to refer to a function                                 |
| Parameter         | A named input listed in the function definition                            |
| Argument          | An actual value supplied when the function is called                       |
| Function body     | The statements inside the function’s curly braces                          |
| Call              | An instruction to execute a function                                       |
| Invocation        | Another word for calling a function                                        |
| Return value      | The value sent back by a function                                          |
| Caller            | The code that invoked the function                                         |
| Local variable    | A variable created inside a function                                       |
| Global variable   | A variable created outside functions and blocks at the top level           |
| Execution context | The internal environment created to execute JavaScript code                |
| Call stack        | The structure JavaScript uses to track currently executing functions       |
| Side effect       | A change outside the function, such as modifying the DOM or logging output |
| Abstraction       | A simplified interface that hides implementation details                   |

# 7. Function Syntax

A traditional function declaration uses the following syntax:

```javascript
function functionName(parameter1, parameter2) {
  // Function body
  return result;
}
```

Let us examine every part.

## 7.1 The `function` keyword

```javascript
function
```

The keyword `function` tells JavaScript that a function declaration is beginning.

A keyword is a reserved word with a special meaning in the language.

You cannot use `function` as a variable name:

```javascript
// SyntaxError
const function = "value";
```

## 7.2 The function name

```javascript
calculateTotal
```

The name allows the function to be referenced and called.

```javascript
function calculateTotal() {
  console.log("Calculating");
}

calculateTotal();
```

A function name should communicate what the function does.

Good names:

```javascript
calculateTotal();
validateEmail();
createUser();
renderTransactions();
fetchProducts();
```

Weak names:

```javascript
doIt();
thing();
run();
handle();
data();
```

## 7.3 Parentheses

```javascript
()
```

The parentheses contain the function’s parameters.

A function with no parameters still requires parentheses:

```javascript
function greet() {
  console.log("Hello");
}
```

A function with parameters lists them inside the parentheses:

```javascript
function greet(name, timeOfDay) {
  console.log(`Good ${timeOfDay}, ${name}`);
}
```

## 7.4 Curly braces

```javascript
{
}
```

Curly braces define the function body.

The function body contains the statements that execute when the function is called.

```javascript
function greet() {
  const message = "Hello";
  console.log(message);
}
```

## 7.5 The `return` keyword

```javascript
return
```

The `return` keyword ends the current function call and sends a value back to the caller.

```javascript
function add(a, b) {
  return a + b;
}
```

Calling the function:

```javascript
const result = add(10, 5);
```

The returned value `15` is assigned to `result`.

```javascript
console.log(result);
```

Output:

```text
15
```

## 7.6 Semicolons

A function declaration normally does not require a semicolon after its closing brace:

```javascript
function greet() {
  console.log("Hello");
}
```

Statements inside the function can end with semicolons:

```javascript
function greet() {
  const message = "Hello";
  console.log(message);
}
```

# 8. Declaring and Calling Functions

## 8.1 Declaring a function

Declaring a function creates the function but does not execute its body.

```javascript
function showMessage() {
  console.log("The function is running.");
}
```

At this point, no output appears.

JavaScript has created a function and associated it with the name `showMessage`.

## 8.2 Calling a function

To call the function, write its name followed by parentheses:

```javascript
showMessage();
```

Output:

```text
The function is running.
```

## 8.3 Calling a function multiple times

Functions are reusable.

```javascript
function showMessage() {
  console.log("The function is running.");
}

showMessage();
showMessage();
showMessage();
```

Output:

```text
The function is running.
The function is running.
The function is running.
```

## 8.4 Referencing versus calling

These two expressions are not the same:

```javascript
showMessage;
showMessage();
```

`showMessage` refers to the function object.

`showMessage()` executes the function.

```javascript
console.log(showMessage);
```

Possible output:

```text
ƒ showMessage() {
  console.log("The function is running.");
}
```

```javascript
console.log(showMessage());
```

Output:

```text
The function is running.
undefined
```

The first output comes from inside the function.

The second output, `undefined`, is the function’s return value because it does not explicitly return anything.

# 9. What Happens Internally

JavaScript does more than jump into a block of code.

When a function is called, the JavaScript engine creates an internal environment for that function call.

Consider:

```javascript
function add(a, b) {
  const result = a + b;
  return result;
}

const total = add(4, 6);
console.log(total);
```

## 9.1 Global code begins executing

The JavaScript engine creates a **global execution context**.

The global execution context manages top-level code.

```text
Call Stack

┌──────────────────────────┐
│ Global Execution Context │
└──────────────────────────┘
```

## 9.2 The function declaration is registered

JavaScript creates the `add` function and associates it with the identifier `add`.

Conceptually:

```text
Global Environment
┌───────────────┬──────────────────────────┐
│ Identifier    │ Value                    │
├───────────────┼──────────────────────────┤
│ add           │ Function object          │
│ total         │ Not initialized yet      │
└───────────────┴──────────────────────────┘
```

## 9.3 The function is called

JavaScript reaches:

```javascript
add(4, 6);
```

It creates a new **function execution context**.

```text
Call Stack

┌──────────────────────────┐
│ add Execution Context    │ ← currently running
├──────────────────────────┤
│ Global Execution Context │
└──────────────────────────┘
```

## 9.4 Arguments are assigned to parameters

The argument `4` is assigned to parameter `a`.

The argument `6` is assigned to parameter `b`.

```text
add Execution Context
┌────────────┬───────┐
│ Identifier │ Value │
├────────────┼───────┤
│ a          │ 4     │
│ b          │ 6     │
│ result     │ 10    │
└────────────┴───────┘
```

## 9.5 The function body executes

JavaScript evaluates:

```javascript
const result = a + b;
```

This becomes:

```javascript
const result = 4 + 6;
```

Therefore:

```text
result = 10
```

## 9.6 The function returns

JavaScript reaches:

```javascript
return result;
```

The function sends `10` back to the calling location.

```javascript
const total = 10;
```

## 9.7 The function execution context is removed

After returning, the `add` execution context is popped off the call stack.

```text
Call Stack

┌──────────────────────────┐
│ Global Execution Context │ ← execution continues here
└──────────────────────────┘
```

The local variables `a`, `b`, and `result` are no longer directly accessible.

## 9.8 Global execution continues

JavaScript executes:

```javascript
console.log(total);
```

Output:

```text
10
```

# 10. Parameters and Arguments

Parameters and arguments are related but different.

## 10.1 Parameter

A **parameter** is a named placeholder declared in the function definition.

```javascript
function greet(name) {
  console.log(`Hello, ${name}`);
}
```

Here, `name` is a parameter.

It represents a value that will be supplied later.

## 10.2 Argument

An **argument** is the actual value supplied when the function is called.

```javascript
greet("Amina");
```

Here, `"Amina"` is an argument.

During this function call:

```text
Parameter: name
Argument:  "Amina"

name receives "Amina"
```

## 10.3 Multiple parameters and arguments

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}

const total = calculateTotal(200, 3);
```

Mapping:

```text
price    ← 200
quantity ← 3
```

Result:

```text
600
```

## 10.4 Argument order matters

Arguments are normally assigned according to position.

```javascript
function describeUser(name, age) {
  return `${name} is ${age} years old.`;
}
```

Correct:

```javascript
console.log(describeUser("Ali", 25));
```

Output:

```text
Ali is 25 years old.
```

Incorrect order:

```javascript
console.log(describeUser(25, "Ali"));
```

Output:

```text
25 is Ali years old.
```

JavaScript does not know the intended meaning of each value. It assigns arguments by position.

## 10.5 Fewer arguments than parameters

```javascript
function greet(firstName, lastName) {
  return `Hello, ${firstName} ${lastName}`;
}

console.log(greet("Amina"));
```

Output:

```text
Hello, Amina undefined
```

The missing argument causes `lastName` to receive `undefined`.

## 10.6 More arguments than parameters

```javascript
function add(a, b) {
  return a + b;
}

console.log(add(10, 20, 30, 40));
```

Output:

```text
30
```

The function uses `10` and `20`.

The extra arguments are not assigned to named parameters.

They are not necessarily inaccessible, but accessing arbitrary arguments involves other features that will be taught later.

# 11. Return Values

A function can send a value back to the code that called it.

## 11.1 Basic return value

```javascript
function add(a, b) {
  return a + b;
}

const result = add(8, 2);

console.log(result);
```

Output:

```text
10
```

## 11.2 Returning versus logging

These functions behave differently:

```javascript
function logTotal(a, b) {
  console.log(a + b);
}

function returnTotal(a, b) {
  return a + b;
}
```

Calling the first function:

```javascript
const firstResult = logTotal(5, 4);
```

Output:

```text
9
```

But:

```javascript
console.log(firstResult);
```

Output:

```text
undefined
```

The function displayed `9`, but it did not return `9`.

Calling the second function:

```javascript
const secondResult = returnTotal(5, 4);

console.log(secondResult);
```

Output:

```text
9
```

> [!WARNING]
> `console.log()` displays a value for developers. `return` sends a value back to the caller. Logging and returning are not interchangeable.

## 11.3 Returned values can be stored

```javascript
function calculateArea(width, height) {
  return width * height;
}

const roomArea = calculateArea(8, 5);
```

## 11.4 Returned values can be passed to other functions

```javascript
function calculateArea(width, height) {
  return width * height;
}

console.log(calculateArea(8, 5));
```

The returned value `40` becomes the argument passed to `console.log()`.

## 11.5 Returned values can be used in expressions

```javascript
function calculateSubtotal(price, quantity) {
  return price * quantity;
}

const finalTotal = calculateSubtotal(200, 3) + 100;

console.log(finalTotal);
```

Output:

```text
700
```

## 11.6 `return` immediately ends the function

```javascript
function checkAge(age) {
  if (age < 18) {
    return "Access denied";
  }

  return "Access granted";
}
```

When `age` is less than `18`, the first return executes. The function ends immediately.

```javascript
console.log(checkAge(15));
```

Output:

```text
Access denied
```

## 11.7 Code after `return` is unreachable

```javascript
function getMessage() {
  return "Hello";
  console.log("This never runs");
}
```

The `console.log()` statement cannot execute because the function has already returned.

This is called **unreachable code**.

# 12. Functions Without Return Statements

Every JavaScript function returns a value.

When no explicit `return` statement is provided, the function returns `undefined`.

```javascript
function greet() {
  console.log("Hello");
}

const result = greet();

console.log(result);
```

Output:

```text
Hello
undefined
```

The function is effectively treated as though it contained:

```javascript
function greet() {
  console.log("Hello");
  return undefined;
}
```

A bare `return` also returns `undefined`:

```javascript
function stopEarly() {
  return;
}

console.log(stopEarly());
```

Output:

```text
undefined
```

# 13. Functions as Reusable Abstractions

Functions do more than reduce repeated code.

They allow developers to name operations.

Consider this expression:

```javascript
const isValid = value !== null && value !== undefined && value !== "";
```

It can be placed inside a reusable function:

```javascript
function hasValue(value) {
  return value !== null && value !== undefined && value !== "";
}
```

Now the calling code communicates intention:

```javascript
if (hasValue(username)) {
  console.log("Username received");
}
```

The caller does not need to know every internal comparison.

This is abstraction:

```text
Caller
   │
   │ asks: "Does this value exist?"
   ▼
hasValue(value)
   │
   │ hides comparison details
   ▼
true or false
```

Professional software relies heavily on small, meaningful abstractions.

Create a function that displays a welcome message.

### Code

```javascript
function showWelcomeMessage() {
  console.log("Welcome to JavaScript!");
}

```

```javascript
function showWelcomeMessage() {
```

* `showWelcomeMessage` is the function’s name.
* `()` means this function receives no parameters.
* `{` begins the function body.

```javascript
  console.log("Welcome to JavaScript!");
```

* `console.log()` displays information in the developer console.
* The string `"Welcome to JavaScript!"` is passed to `console.log()`.

```javascript
}
```

* `}` ends the function body.

```javascript
showWelcomeMessage();
```

* `showWelcomeMessage` refers to the function.
* `()` invokes the function.
* `;` ends the statement.

### Output

```text
Welcome to JavaScript!
```

### Execution flow

```text
Program starts
     ↓
Function declaration is registered
     ↓
showWelcomeMessage() is called
     ↓
Function body executes
     ↓
Message is logged
     ↓
Function returns undefined
     ↓
Program continues
```

## Example 2: Intermediate Example

### Goal

Calculate the total amount for a purchase.

### Code

```javascript
function calculatePurchaseTotal(price, quantity, taxRate) {
  const subtotal = price * quantity;
  const taxAmount = subtotal * taxRate;
  const finalTotal = subtotal + taxAmount;

  return finalTotal;
}

const amountToPay = calculatePurchaseTotal(500, 3, 0.16);

console.log(`Amount to pay: KES ${amountToPay}`);
```

### Line-by-line explanation

```javascript
function calculatePurchaseTotal(price, quantity, taxRate) {
```

The function receives three parameters:

* `price`: the price of one item.
* `quantity`: the number of items.
* `taxRate`: the tax expressed as a decimal.

```javascript
const subtotal = price * quantity;
```

The item price is multiplied by the quantity.

For the supplied arguments:

```text
500 × 3 = 1500
```

```javascript
const taxAmount = subtotal * taxRate;
```

Tax is calculated:

```text
1500 × 0.16 = 240
```

```javascript
const finalTotal = subtotal + taxAmount;
```

The subtotal and tax are added:

```text
1500 + 240 = 1740
```

```javascript
return finalTotal;
```

The function sends `1740` back to its caller.

```javascript
const amountToPay = calculatePurchaseTotal(500, 3, 0.16);
```

Argument mapping:

```text
price    ← 500
quantity ← 3
```

```javascript
console.log(`Amount to pay: KES ${amountToPay}`);
```

A template literal creates the final message.

## Example 3: Real-World Application

### Goal

Validate a login form submission using reusable functions.

### Code

```javascript
function isUsernameValid(username) {
  const trimmedUsername = username.trim();

  return trimmedUsername.length >= 3;
}

function isPasswordValid(password) {
  return password.length >= 8;
}

function validateLogin(username, password) {
  const usernameIsValid = isUsernameValid(username);
  const passwordIsValid = isPasswordValid(password);

  if (!usernameIsValid) {
    return {
      success: false,
      message: "Username must contain at least 3 characters.",
    };
  }

  if (!passwordIsValid) {
    return {
      success: false,
      message: "Password must contain at least 8 characters.",
    };
  }

  return {
    success: true,
    message: "Login data is valid.",
  };
}

const validationResult = validateLogin("saku", "secure123");

console.log(validationResult);
```

### Output

```text
{
  success: true,
  message: "Login data is valid."
}
```

### Explanation

#### Username validation function

```javascript
function isUsernameValid(username) {
```

This function has one responsibility: determine whether a username is valid.

```javascript
const trimmedUsername = username.trim();
```

The string method `trim()` removes whitespace from the beginning and end.

For example:

```text
"  saku  " → "saku"
```

```javascript
return trimmedUsername.length >= 3;
```

The comparison produces a Boolean:

```text
true  → username has at least 3 characters
false → username has fewer than 3 characters
```

#### Password validation function

```javascript
function isPasswordValid(password) {
  return password.length >= 8;
}
```

This function returns whether the password has at least eight characters.

#### Main validation function

```javascript
function validateLogin(username, password) {
```

This function coordinates the smaller validation functions.

```javascript
const usernameIsValid = isUsernameValid(username);
const passwordIsValid = isPasswordValid(password);
```

Instead of repeating validation rules, it calls the dedicated functions.

```javascript
if (!usernameIsValid) {
```

The `!` operator reverses a Boolean value.

```text
usernameIsValid = false
!usernameIsValid = true
```

If the username is invalid, the function returns an object:

```javascript
return {
  success: false,
  message: "Username must contain at least 3 characters.",
};
```

Returning immediately prevents unnecessary validation processing.

The password check follows the same pattern.

When both values are valid:

```javascript
return {
  success: true,
  message: "Login data is valid.",
};
```

### Why professionals write code this way

This design separates responsibilities:

```text
isUsernameValid()
        │
        ├── validates username only
        │
isPasswordValid()
        │
        ├── validates password only
        │
validateLogin()
        │
        └── coordinates the validation process
```

Benefits include:

* Each function has a clear purpose.
* Each rule can be tested independently.
* Validation rules can be reused elsewhere.
* The main function reads like a description of the process.
* A future rule change can be made in one location.
* The returned object contains both machine-readable status and a human-readable message.

# 15. Function Inputs, Processing, and Outputs

Many functions can be understood using three stages.

```text
Input → Processing → Output
```

Consider:

```javascript
function convertCelsiusToFahrenheit(celsius) {
  const fahrenheit = (celsius * 9) / 5 + 32;
  return fahrenheit;
}
```

## Input

```javascript
celsius
```

The function receives a Celsius temperature.

## Processing

```javascript
(celsius * 9) / 5 + 32
```

The function applies a conversion formula.

## Output

```javascript
return fahrenheit;
```

The function sends the Fahrenheit temperature back.

Example:

```javascript
const temperature = convertCelsiusToFahrenheit(25);

console.log(temperature);
```

Output:

```text
77
```

## Not all functions fit a pure input-output model

Some functions cause side effects:

```javascript
function displayNotification(message) {
  console.log(message);
}
```

Later, a browser-based version could change the DOM:

```javascript
function displayNotification(message) {
  document.querySelector("#notification").textContent = message;
}
```

The function changes something outside itself. That is a side effect.

# 16. Function Naming

Function names should usually describe an action.

## 16.1 Use verbs

Good:

```javascript
calculateTotal();
validateUser();
createTransaction();
deleteProduct();
renderDashboard();
sendRequest();
```

Functions perform actions, so verbs communicate their purpose.

## 16.2 Use Boolean-style prefixes for Boolean results

Functions that return `true` or `false` often use prefixes such as:

* `is`
* `has`
* `can`
* `should`

Examples:

```javascript
isAdult();
hasPermission();
canSubmitForm();
shouldRefreshToken();
```

## 16.3 Use specific names

Weak:

```javascript
function process(data) {
  // ...
}
```

Better:

```javascript
function calculateMonthlyRevenue(transactions) {
  // ...
}
```

## 16.4 Avoid misleading names

Bad:

```javascript
function getUser() {
  deleteUserFromDatabase();
}
```

The name suggests reading data, but the function deletes data.

This is dangerous because callers rely on names to understand behavior.

## 16.5 Follow camelCase

JavaScript function names conventionally use `camelCase`.

```javascript
calculateTotal();
validateEmailAddress();
createTransactionRecord();
```

Avoid:

```javascript
CalculateTotal();
calculate_total();
calculate-total();
```

`PascalCase` is typically reserved for classes and constructor-style functions.

# 17. Local Variables

A variable declared inside a function is normally local to that function.

```javascript
function calculateTotal(price, quantity) {
  const subtotal = price * quantity;

  return subtotal;
}

console.log(calculateTotal(100, 3));
```

Output:

```text
300
```

Attempting to access `subtotal` outside the function:

```javascript
console.log(subtotal);
```

Produces an error:

```text
ReferenceError: subtotal is not defined
```

## Why local variables exist

Local variables prevent unrelated parts of a program from accidentally accessing or modifying internal function data.

```text
Global Environment
│
├── calculateTotal function
│
└── total variable

calculateTotal Local Environment
│
├── price
├── quantity
└── subtotal
```

The function’s local environment acts as a protective boundary.

# 18. Global Variables

A variable declared at the top level is accessible to many parts of the program.

```javascript
const taxRate = 0.16;

function calculateTax(subtotal) {
  return subtotal * taxRate;
}

console.log(calculateTax(1000));
```

Output:

```text
160
```

The function can access `taxRate` from the outer environment.

However, excessive use of global variables creates problems.

## 18.1 Hidden dependencies

```javascript
let discountRate = 0.1;

function calculateDiscount(price) {
  return price * discountRate;
}
```

The function depends on `discountRate`, but that dependency is not visible in its parameter list.

A clearer design:

```javascript
function calculateDiscount(price, discountRate) {
  return price * discountRate;
}
```

## 18.2 Accidental modification

```javascript
let accountBalance = 1000;

function applyCharge() {
  accountBalance -= 200;
}
```

Any part of the program could modify `accountBalance`.

In large applications, this can make bugs difficult to trace.

## 18.3 Naming collisions

Two unrelated sections may use the same global name.

```javascript
let status = "loading";

// Much later:
let status = "active";
```

This creates a declaration error in the same scope.

> [!TIP]
> Prefer passing required data through parameters and returning results explicitly. This makes dependencies visible.

# 19. Common Mistakes

## Mistake 1: Declaring a function but never calling it

Incorrect:

```javascript
function greet() {
  console.log("Hello");
}
```

No output appears because the function was only declared.

Correct:

```javascript
function greet() {
  console.log("Hello");
}

greet();
```

Output:

```text
Hello
```

---

## Mistake 2: Forgetting parentheses when calling

Incorrect:

```javascript
function greet() {
  console.log("Hello");
}

greet;
```

`greet` refers to the function but does not execute it.

Correct:

```javascript
greet();
```

---

## Mistake 3: Confusing parameters and arguments

```javascript
function greet(name) {
  console.log(`Hello, ${name}`);
}

greet("Amina");
```

Correct terminology:

* `name` is a parameter.
* `"Amina"` is an argument.

---

## Mistake 4: Logging instead of returning

Incorrect:

```javascript
function add(a, b) {
  console.log(a + b);
}

const result = add(2, 3);

console.log(result * 2);
```

Output:

```text
5
NaN
```

Why?

The function logs `5` but returns `undefined`.

The final operation becomes:

```javascript
undefined * 2
```

That produces `NaN`.

Correct:

```javascript
function add(a, b) {
  return a + b;
}

const result = add(2, 3);

console.log(result * 2);
```

Output:

```text
10
```

---

## Mistake 5: Writing code after `return`

Incorrect:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;

  console.log("Calculation completed");
}
```

The log statement is unreachable.

Correct:

```javascript
function calculateTotal(price, quantity) {
  const total = price * quantity;

  console.log("Calculation completed");

  return total;
}
```

---

## Mistake 6: Using the wrong argument order

Incorrect:

```javascript
function createUserMessage(name, age) {
  return `${name} is ${age} years old.`;
}

console.log(createUserMessage(30, "John"));
```

Output:

```text
30 is John years old.
```

Correct:

```javascript
console.log(createUserMessage("John", 30));
```

---

## Mistake 7: Depending unnecessarily on global variables

Weak design:

```javascript
let price = 500;
let quantity = 3;

function calculateTotal() {
  return price * quantity;
}
```

Better:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}
```

---

## Mistake 8: Making a function perform unrelated tasks

Weak design:

```javascript
function processOrder(order) {
  const total = order.price * order.quantity;
  console.log(total);
  document.body.textContent = total;
  localStorage.setItem("lastOrder", JSON.stringify(order));
  sendOrderToServer(order);
  updateInventory(order);
}
```

This function calculates, renders, stores, sends, and updates.

A better design separates responsibilities:

```javascript
function calculateOrderTotal(order) {
  return order.price * order.quantity;
}

function saveOrderLocally(order) {
  localStorage.setItem("lastOrder", JSON.stringify(order));
}

function renderOrderTotal(total) {
  document.body.textContent = total;
}
```

The remaining operations can be coordinated by another function later.

---

## Mistake 9: Using unclear names

Weak:

```javascript
function x(a, b) {
  return a * b;
}
```

Better:

```javascript
function calculateRectangleArea(width, height) {
  return width * height;
}
```

---

## Mistake 10: Calling a function before understanding its return type

```javascript
function getUser() {
  return {
    name: "Amina",
    age: 24,
  };
}

const result = getUser();

console.log(result.toUpperCase());
```

Error:

```text
TypeError: result.toUpperCase is not a function
```

`result` is an object, not a string.

Correct:

```javascript
console.log(result.name.toUpperCase());
```

Output:

```text
AMINA
```

# 20. Best Practices

## 20.1 Give each function one clear responsibility

A function should usually do one conceptual job.

Good:

```javascript
function calculateSubtotal(price, quantity) {
  return price * quantity;
}

function calculateTax(subtotal, taxRate) {
  return subtotal * taxRate;
}

function calculateFinalTotal(subtotal, tax) {
  return subtotal + tax;
}
```

This principle is commonly called the **Single Responsibility Principle**.

It does not mean that every function must contain only one line. It means the function should have one coherent reason to change.

## 20.2 Prefer explicit inputs

Less clear:

```javascript
let currentPrice = 100;
let currentQuantity = 4;

function calculateTotal() {
  return currentPrice * currentQuantity;
}
```

More explicit:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}
```

## 20.3 Prefer returning values over logging inside calculation functions

Less reusable:

```javascript
function calculateTotal(price, quantity) {
  console.log(price * quantity);
}
```

More reusable:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}
```

The caller decides what to do with the result:

```javascript
const total = calculateTotal(100, 4);

console.log(total);
```

Later, the same result could be:

* Displayed in the DOM.
* Sent to a server.
* Saved in storage.
* Passed to another function.
* Used in another calculation.

## 20.4 Avoid unnecessary mutation

Mutation means changing an existing value or object.

Weak:

```javascript
let total = 0;

function calculateTotal(price, quantity) {
  total = price * quantity;
}
```

Better:

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}

const total = calculateTotal(100, 3);
```

## 20.5 Validate important inputs

```javascript
function calculateTotal(price, quantity) {
  if (typeof price !== "number" || typeof quantity !== "number") {
    throw new TypeError("Price and quantity must be numbers.");
  }

  return price * quantity;
}
```

Error handling will be covered deeply later in this module.

## 20.6 Keep functions reasonably small

A function hundreds of lines long is difficult to:

* Understand.
* Test.
* debug.
* reuse.
* modify safely.

Do not split functions merely to reduce line count. Split them when separate responsibilities or meaningful operations exist.

## 20.7 Use early returns to reduce nesting

Deeply nested:

```javascript
function getAccessMessage(user) {
  if (user) {
    if (user.isActive) {
      if (user.isAdmin) {
        return "Admin access granted";
      } else {
        return "Standard access granted";
      }
    } else {
      return "Account inactive";
    }
  } else {
    return "User not found";
  }
}
```

Clearer with early returns:

```javascript
function getAccessMessage(user) {
  if (!user) {
    return "User not found";
  }

  if (!user.isActive) {
    return "Account inactive";
  }

  if (user.isAdmin) {
    return "Admin access granted";
  }

  return "Standard access granted";
}
```

## 20.8 Do not hide unexpected side effects

A function named `getUser()` should not delete or modify the user.

Names should accurately communicate behavior.

# 21. Performance Considerations

Function calls have a small execution cost, but this cost is usually insignificant compared with clarity and maintainability.

Do not avoid functions because of premature performance concerns.

## 21.1 Time complexity

A simple arithmetic function performs a fixed number of operations.

```javascript
function calculateTotal(price, quantity) {
  return price * quantity;
}
```

Its time complexity is:

```text
O(1)
```

`O(1)` means the number of operations does not grow with the input size.

A function containing a loop may have different complexity:

```javascript
function calculateSum(numbers) {
  let total = 0;

  for (const number of numbers) {
    total += number;
  }

  return total;
}
```

For `n` numbers, the function performs approximately `n` iterations.

Time complexity:

```text
O(n)
```

## 21.2 Memory usage

Each function call creates an execution context containing:

* Parameter bindings.
* Local variables.
* Internal bookkeeping.
* A reference to its outer lexical environment.

When the function completes, its execution context is normally removed from the call stack.

```text
Function called
     ↓
Execution context created
     ↓
Parameters and local variables stored
     ↓
Function executes
     ↓
Function returns
     ↓
Execution context removed
```

## 21.3 Deep call stacks

Functions can call other functions.

```javascript
function first() {
  second();
}

function second() {
  third();
}

function third() {
  console.log("Done");
}

first();
```

Call stack:

```text
┌──────────────────────────┐
│ third Execution Context  │
├──────────────────────────┤
│ second Execution Context │
├──────────────────────────┤
│ first Execution Context  │
├──────────────────────────┤
│ Global Execution Context │
└──────────────────────────┘
```

Extremely deep chains can exhaust the call stack.

This is particularly important with recursion, which will be covered later.

## 21.4 Avoid repeated expensive calculations

Less efficient:

```javascript
function calculateLargeReport(data) {
  // Expensive calculation
}

console.log(calculateLargeReport(data));
saveReport(calculateLargeReport(data));
renderReport(calculateLargeReport(data));
```

The function executes three times.

Better:

```javascript
const report = calculateLargeReport(data);

console.log(report);
saveReport(report);
renderReport(report);
```

## 21.5 Browser performance

Calculation functions are usually inexpensive.

Functions that repeatedly manipulate the DOM can be expensive because they may trigger:

* Style recalculation.
* Layout.
* Painting.
* Compositing.

Later DOM lessons will show how to group updates and reduce unnecessary rendering work.

## 21.6 Optimization trade-offs

A highly condensed function is not automatically faster or better.

Difficult-to-read code often creates more maintenance cost than the tiny execution time it might save.

Prioritize:

1. Correctness.
2. Clarity.
3. Testability.
4. Measured performance improvements.

> [!WARNING]
> Do not optimize based only on guesses. Use browser performance tools to measure actual bottlenecks.

# 22. Debugging Functions

Debugging is the process of locating, understanding, and fixing defects.

## 22.1 Debugging with `console.log()`

Suppose this function returns the wrong total:

```javascript
function calculateTotal(price, quantity, discountRate) {
  const subtotal = price * quantity;
  const discount = subtotal + discountRate;
  const total = subtotal - discount;

  return total;
}
```

Add logs:

```javascript
function calculateTotal(price, quantity, discountRate) {
  console.log("price:", price);
  console.log("quantity:", quantity);
  console.log("discountRate:", discountRate);

  const subtotal = price * quantity;
  console.log("subtotal:", subtotal);

  const discount = subtotal + discountRate;
  console.log("discount:", discount);

  const total = subtotal - discount;
  console.log("total:", total);

  return total;
}

calculateTotal(1000, 2, 0.1);
```

Output:

```text
price: 1000
quantity: 2
discountRate: 0.1
subtotal: 2000
discount: 2000.1
total: -0.09999999999990905
```

The incorrect line becomes visible:

```javascript
const discount = subtotal + discountRate;
```

It should multiply:

```javascript
const discount = subtotal * discountRate;
```

Corrected function:

```javascript
function calculateTotal(price, quantity, discountRate) {
  const subtotal = price * quantity;
  const discount = subtotal * discountRate;
  const total = subtotal - discount;

  return total;
}
```

## 22.2 Use structured logging

Instead of several unrelated logs:

```javascript
console.log(price);
console.log(quantity);
console.log(discountRate);
```

Use labels:

```javascript
console.log("price:", price);
console.log("quantity:", quantity);
console.log("discountRate:", discountRate);
```

Or group related values:

```javascript
console.log({
  price,
  quantity,
  discountRate,
});
```

## 22.3 Debugging with breakpoints

A breakpoint pauses JavaScript before a selected line executes.

General browser procedure:

1. Open the page.
2. Open Developer Tools.
3. Select the **Sources** tab.
4. Locate the JavaScript file.
5. Click a line number.
6. Trigger the function.
7. Inspect local variables.
8. Use step controls to continue execution.

Common step controls:

| Control       | Purpose                                                    |
| ------------- | ---------------------------------------------------------- |
| Resume        | Continue until the next breakpoint                         |
| Step over     | Execute the current line without entering called functions |
| Step into     | Enter the function called on the current line              |
| Step out      | Finish the current function and return to its caller       |
| Restart frame | Restart the current function frame                         |

## 22.4 Inspecting scope

When paused inside:

```javascript
function calculateTotal(price, quantity) {
  const subtotal = price * quantity;
  return subtotal;
}
```

The DevTools scope panel may show:

```text
Local
├── price: 100
├── quantity: 3
└── subtotal: 300

Global
├── window
├── document
└── other global bindings
```

## 22.5 Inspecting the call stack

Consider:

```javascript
function createReceipt() {
  const total = calculateTotal(100, 3);
  printReceipt(total);
}

function calculateTotal(price, quantity) {
  return price * quantity;
}

```

When paused inside `printReceipt()`, the call stack may show:

```text

Global
```

This tells you:

* Which function is currently running.
* Which function called it.
* How execution reached the current line.

## 22.6 Using the `debugger` statement

You can place `debugger` inside code:

```javascript
function calculateTotal(price, quantity) {
  debugger;

  return price * quantity;
}
```

When DevTools is open, execution pauses at that statement.

Remove `debugger` statements before production deployment.

## 22.7 Debugging argument problems

```javascript
function calculateTotal(price, quantity) {
  console.log({
    price,
    quantity,
    priceType: typeof price,
```

```javascript
function calculateTax(subtotal) {
```

Output:


The function calculates the value but forgets to return it.

Correct:

```javascript
function calculateTax(subtotal) {
  const tax = subtotal * 0.16;

  return tax;
}
```

# 23. Interview Questions and Answers

## Beginner Questions

### 1. What is a JavaScript function?

A JavaScript function is a reusable block of code designed to perform a particular task. It can receive inputs through parameters, execute instructions, and return a value.

### 2. What is the difference between declaring and calling a function?

Declaring a function defines it. Calling a function executes its body.

```javascript
function greet() {
  console.log("Hello");
}

greet();
```

The first part declares the function. The final line calls it.

### 3. What is a parameter?

A parameter is a named placeholder listed in a function definition.

```javascript
function greet(name) {
  // name is a parameter
}
```

### 4. What is an argument?

An argument is an actual value supplied to a function call.

```javascript
greet("Amina");
```

`"Amina"` is an argument.

### 5. What does a function return when there is no `return` statement?

It returns `undefined`.

### 6. What is the difference between `return` and `console.log()`?

`return` sends a value back to the caller and ends the function. `console.log()` displays a value in the developer console and does not determine the function’s return value.

### 7. Can a function be called more than once?

Yes. Reusability is one of the main reasons functions exist.

### 8. Why are functions useful?

Functions reduce repetition, organize logic, create abstractions, improve testability, and make maintenance easier.

### 9. What happens to code after a `return` statement?

It is not executed during that function call because `return` immediately ends the function.

### 10. Can a function have no parameters?

Yes.

```javascript
function showMessage() {
  console.log("Hello");
}
```

---

## Intermediate Questions

### 1. What is a function execution context?

A function execution context is the internal environment JavaScript creates for a function call. It contains parameter bindings, local variables, access to the outer lexical environment, and execution bookkeeping.

### 2. What is the call stack?

The call stack is the data structure JavaScript uses to track active execution contexts. When a function is called, its execution context is pushed onto the stack. When it returns, that context is popped from the stack.

### 3. What is a local variable?

A local variable is declared inside a function or block and is only accessible within its permitted scope.

### 4. What is a global variable?

A global variable is declared in the global scope and may be accessible throughout the program. Excessive global state can create hidden dependencies and accidental modification.

### 5. Why should calculation functions usually return instead of log?

Returning makes the result reusable. The caller can log it, render it, store it, send it, test it, or use it in another expression.

### 6. What is an early return?

An early return exits a function as soon as a condition is met.

```javascript
function validateAge(age) {
  if (age < 0) {
    return "Invalid age";
  }

  return "Valid age";
}
```

Early returns often reduce unnecessary nesting.

### 7. What is a side effect?

A side effect is an observable change outside a function’s returned value, such as modifying a global variable, changing the DOM, writing to storage, logging, or sending a network request.

### 8. Why are explicit parameters preferable to hidden global dependencies?

Explicit parameters make a function’s requirements visible, improve testability, and reduce accidental coupling to external state.

### 9. Can a function return an object?

Yes. A function can return any JavaScript value, including objects, arrays, strings, numbers, Booleans, `null`, `undefined`, symbols, bigints, and other functions.

### 10. What is unreachable code?

Unreachable code is code that cannot execute because control flow exits before reaching it, often due to `return`, `throw`, `break`, or `continue`.

## Advanced Questions

### 1. Is a JavaScript function a value?

Yes. Functions are first-class values in JavaScript. They can be assigned to variables, stored in objects and arrays, passed as arguments, and returned from other functions.

This will be explored in the function-expression and callback lessons.

### 2. What is the relationship between a function call and the call stack?

Every normal synchronous function call creates an execution context that is pushed onto the call stack. Nested calls add additional frames. Returning removes the current frame and resumes execution in the caller.

### 3. Why can excessive recursion cause a stack overflow?

Each recursive call adds a new execution context to the call stack. If recursion does not terminate before the stack’s limit is reached, the engine throws an error such as:

```text
RangeError: Maximum call stack size exceeded
```

### 4. Do local variables always disappear immediately after a function returns?

Normally, the function’s execution context is removed from the call stack. However, values may remain reachable when referenced by another object or closure. Garbage collection depends on reachability, not merely on whether a function has returned.

Closures and garbage collection will be taught later.

### 5. What does it mean for a function to be an abstraction boundary?

It means callers interact with the function through its public interface—its name, parameters, and return value—without needing to know every internal implementation detail.

### 6. Why can many small functions become harmful?

Excessive fragmentation can make control flow difficult to follow. Functions should be extracted when they represent meaningful operations, reusable logic, testable units, or separate responsibilities—not merely to minimize line count.

### 7. What is referential transparency?

An expression is referentially transparent when it can be replaced by its resulting value without changing program behavior. Pure functions often support referential transparency.

This will be explored in the pure-functions lesson.

### 8. What is function-call overhead?

A function call requires the engine to create and manage execution state. Modern JavaScript engines can optimize many calls, including inlining suitable functions. In ordinary application code, clarity should be prioritized unless profiling identifies function calls as a real bottleneck.

### 9. Can JavaScript functions return multiple values?

A function directly returns one value. However, that value can be an array or object containing multiple pieces of information.

```javascript
function getUserDetails() {
  return {
    name: "Amina",
    age: 24,
  };
}
```

### 10. How do JavaScript engines optimize functions?

Modern engines may collect runtime type information, compile frequently executed code, inline small functions, specialize operations, and deoptimize when assumptions become invalid. Exact behavior differs among engines and is an implementation detail rather than a language guarantee.

# 27. Summary

In this lesson, you learned that a function is a reusable block of JavaScript code designed to perform a particular task.

A function can:

* Receive inputs through parameters.
* Process those inputs.
* Return an output.
* Cause side effects.
* Hide implementation details.
* organize application logic.
* reduce code duplication.
* improve testing and maintenance.

You learned the basic syntax:

```javascript
function functionName(parameter1, parameter2) {
  return result;
}
```

You learned the distinction between declaring and calling:

```javascript
function greet() {
  return "Hello";
}

greet();
```

You learned the distinction between parameters and arguments:

```javascript
function greet(name) {
  return `Hello, ${name}`;
}

greet("Amina");
```

* `name` is a parameter.
* `"Amina"` is an argument.

You learned that `return`:

* Sends a value to the caller.
* Ends the current function call.
* Makes results reusable.
* Is different from `console.log()`.

You also learned that a function without an explicit `return` produces:

```javascript
undefined
```

Internally, every normal function call creates a function execution context and pushes it onto the call stack.

```text
Function call
    ↓
Execution context created
    ↓
Context pushed onto call stack
    ↓
Parameters receive arguments
    ↓
Function body executes
    ↓
Return value produced
    ↓
Context removed from stack
```

Professional functions should generally:

* Have clear names.
* Have one coherent responsibility.
* Receive explicit inputs.
* Return reusable values.
* Avoid hidden global dependencies.
* Avoid unexpected side effects.
* Validate important inputs.
* Be tested independently.
