# 1. Introduction

In the previous lesson, you learned how to insert nodes at the beginning or end of a parent.

```javascript
parent.prepend(node);
parent.append(node);
```

These methods are excellent when you already know the parent and want to place content at one of its boundaries.

However, many real interfaces require placement relative to a specific existing element.

Examples:

* Display an error message after an input.
* Insert a heading before a list.
* Place a new table row after the selected row.
* Add a loading indicator before a button.
* Insert a separator between two menu items.
* Add a notification immediately before a card.
* Place a confirmation message after a form.
* Insert a new field after the currently focused field.

In these cases, you may know the reference element more naturally than its parent.

Example:

```html
<label>
  Email
  <input id="email">
</label>
```

You already have the input:

```javascript
const emailInput =
  document.querySelector(
    "#email"
  );
```

You want an error immediately after it:

```javascript
emailInput.after(
  errorMessage
);
```

Result:

```html
<label>
  Email
  <input id="email">
  <p class="error-message">
    Enter a valid email address.
  </p>
</label>
```

This is reference-based insertion.

---

# 2. What Is Relative DOM Insertion?

> Relative DOM insertion places a node or text at a position defined in relation to an existing reference element.

The reference element acts as an anchor.

```text
Before the reference
Reference element
After the reference
```

Relative insertion can also target positions inside the reference element:

```text
Beginning inside
Existing children
End inside
```

JavaScript provides several APIs:

```javascript
before()
after()
insertAdjacentElement()
insertAdjacentText()
insertAdjacentHTML()
```

---

# 3. Why Relative Insertion Exists

Without relative insertion, you may need to manually find the parent and calculate a position.

Example requirement:

> Insert an error immediately after the username input.

Longer structural approach:

```javascript
const parent =
  usernameInput.parentElement;

const next =
  usernameInput.nextSibling;

parent.insertBefore(
  errorMessage,
  next
);
```

Modern relative insertion:

```javascript
usernameInput.after(
  errorMessage
);
```

The newer code expresses the requirement directly.

```text
Input, place this message after yourself.
```

Relative insertion improves:

* Readability.
* Intent.
* Maintainability.
* Local component logic.
* Dynamic interface placement.

---

# 4. Parent-Based vs Reference-Based Insertion

## Parent-based insertion

```javascript
container.append(node);
container.prepend(node);
```

Question answered:

> Where should this node go inside the parent?

## Reference-based insertion

```javascript
reference.before(node);
reference.after(node);
```

Question answered:

> Where should this node go relative to this existing element?

## Position-string insertion

```javascript
reference.insertAdjacentElement(
  "beforebegin",
  node
);
```

Question answered:

> Which of the four positions around or inside this element should receive the node?

```javascript
bookB.before(newBook);
```

```javascript
bookB.after(newBook);
```

The method is called on the anchor element.

# 6. Insertion Positions Around an Element

Consider:

```html
<section id="card">
  <p>Existing content</p>
</section>
```

Four conceptual positions exist:

```text
beforebegin

<section>
    afterbegin

    Existing children

    beforeend
</section>

afterend
```

They mean:

| Position      | Location                                   |
| ------------- | ------------------------------------------ |
| `beforebegin` | Immediately before the element itself      |
| `afterbegin`  | Inside the element, before its first child |
| `beforeend`   | Inside the element, after its last child   |
| `afterend`    | Immediately after the element itself       |

These correspond closely to:

```javascript
element.before(node);
element.prepend(node);
element.append(node);
element.after(node);
```

---

# 7. `before()`

The `before()` method inserts nodes or strings immediately before the reference element.

Syntax:

```javascript
referenceElement.before(
  ...nodesOrStrings
);
```

Example:

```javascript
input.before(label);
```

The inserted node becomes a sibling of the reference.

---

# 8. `before()` Syntax

```javascript
element.before(
  value1,
  value2,
  value3
);
```

Each value may be:

* An element.
* A text node.
* Another node.
* A `DocumentFragment`.
* A string.

Strings become text nodes.

The inserted values appear before the reference in argument order.

---

# 9. Inserting One Node Before an Element

HTML:

```html
<input id="username">
```

JavaScript:

```javascript
const input =
  document.querySelector(
    "#username"
  );

const label =
  document.createElement(
    "label"
  );

label.htmlFor =
  "username";

label.textContent =
  "Username";

input.before(label);
```

Result:

```html
<label for="username">
  Username
</label>

<input id="username">
```

Relationships:

```javascript
label.nextElementSibling ===
  input;
```

Output:

```text
true
```

---

# 10. Inserting Multiple Nodes with `before()`

```javascript
input.before(
  label,
  helpText
);
```

Resulting order:

```text
label
helpText
input
```

Argument order is preserved.

Example:

```javascript
const heading =
  document.createElement(
    "h2"
  );

const description =
  document.createElement(
    "p"
  );

heading.textContent =
  "Payment Details";

description.textContent =
  "Enter the payment information.";

form.before(
  heading,
  description
);
```

Both nodes become siblings immediately before the form.

---

# 11. Inserting Strings with `before()`

```javascript
amount.before(
  "Amount: "
);
```

The string becomes a text node.

Before:

```html
<span id="amount">
  KES 500
</span>
```

After:

```html
Amount:
<span id="amount">
  KES 500
</span>
```

The string is not parsed as HTML.

---

# 12. `before()` Return Value

`before()` returns:

```text
undefined
```

Incorrect:

```javascript
const inserted =
  reference.before(node);
```

`inserted` is `undefined`.

Keep the original node reference:

```javascript
reference.before(node);

node.classList.add(
  "inserted"
);
```

---

# 13. `after()`

The `after()` method inserts nodes or strings immediately after the reference element.

Syntax:

```javascript
referenceElement.after(
  ...nodesOrStrings
);
```

Example:

```javascript
input.after(
  errorMessage
);
```

The message becomes the next sibling.

---

# 14. `after()` Syntax

```javascript
element.after(
  value1,
  value2
);
```

Values may be:

* Nodes.
* Elements.
* Fragments.
* Strings.

Multiple arguments are inserted after the reference while preserving argument order.

---

# 15. Inserting One Node After an Element

HTML:

```html
<input id="email">
```

JavaScript:

```javascript
const error =
  document.createElement(
    "p"
  );

error.className =
  "error-message";

error.textContent =
  "Enter a valid email address.";

emailInput.after(error);
```

Result:

```html
<input id="email">

<p class="error-message">
  Enter a valid email address.
</p>
```

Verification:

```javascript
emailInput
  .nextElementSibling ===
  error;
```

Output:

```text
true
```

---

# 16. Inserting Multiple Nodes with `after()`

```javascript
heading.after(
  description,
  divider
);
```

Result:

```text
heading
description
divider
previous next sibling
```

The browser inserts the arguments in the same order they were supplied.

---

# 17. Inserting Strings with `after()`

```javascript
label.after(
  ": "
);
```

This inserts a text node immediately after the label.

For complex content, explicit elements are usually clearer.

---

# 18. `after()` Return Value

Like `before()`:

```javascript
const result =
  reference.after(node);
```

`result` is:

```text
undefined
```

The inserted node remains accessible through its existing variable.

---

# 19. `before()` and `after()` vs `prepend()` and `append()`

Consider:

```html
<section id="container">
  <p id="reference">
    Reference
  </p>
</section>
```

## `reference.before(node)`

```html
<section>
  <new-node></new-node>
  <p id="reference"></p>
</section>
```

## `reference.after(node)`

```html
<section>
  <p id="reference"></p>
  <new-node></new-node>
</section>
```

## `reference.prepend(node)`

```html
<p id="reference">
  <new-node></new-node>
  Reference
</p>
```

## `reference.append(node)`

```html
<p id="reference">
  Reference
  <new-node></new-node>
</p>
```

Key distinction:

```text
before / after:
Insert siblings

prepend / append:
Insert children
```

---

# 20. Node Movement During Relative Insertion

Relative insertion moves existing nodes.

HTML:

```html
<ul>
  <li id="first">First</li>
  <li id="second">Second</li>
  <li id="third">Third</li>
</ul>
```

Code:

```javascript
third.before(first);
```

Result:

```text
Second
First
Third
```

`first` was moved.

It was not copied.

---

# 21. Reference Elements Must Have Parents

`before()` and `after()` insert siblings.

A sibling relationship requires a parent.

A newly created detached element has no parent:

```javascript
const reference =
  document.createElement(
    "p"
  );

console.log(
  reference.parentNode
);
```

Output:

```text
null
```

Calling:

```javascript
reference.before(node);
```

cannot insert `node` into a parent because no parent exists.

No useful document change occurs.

Before sibling insertion, verify:

```javascript
console.assert(
  reference.parentNode,
  "Reference element must have a parent."
);
```

# 22. `insertAdjacentElement()`

`insertAdjacentElement()` inserts one element at one of four named positions relative to a reference element.

Syntax:

```javascript
reference.insertAdjacentElement(
  position,
  element
);
```

Example:

```javascript
input.insertAdjacentElement(
  "afterend",
  errorMessage
);
```

This is similar to:

```javascript
input.after(
  errorMessage
);
```

---

# 23. `insertAdjacentElement()` Syntax

```javascript
const result =
  referenceElement
    .insertAdjacentElement(
      position,
      insertedElement
    );
```

Arguments:

## `position`

One of four strings:

```text
beforebegin
afterbegin
beforeend
afterend
```

## `insertedElement`

The element to insert.

Unlike `append()` and `before()`, this method accepts one element rather than multiple nodes or strings.

---

# 24. The Four Position Keywords

```text
beforebegin
afterbegin
beforeend
afterend
```

The names describe the insertion location relative to the opening and closing boundaries of the reference element.

Consider:

```html
<article>
  Existing content
</article>
```

The positions are:

```text
beforebegin
<article>
  afterbegin

  Existing content

  beforeend
</article>
afterend
```

---

# 25. `beforebegin`

Inserts immediately before the reference element.

```javascript
card.insertAdjacentElement(
  "beforebegin",
  notice
);
```

Equivalent idea:

```javascript
card.before(notice);
```

The inserted element becomes a previous sibling.

---

# 26. `afterbegin`

Inserts inside the reference as its first child.

```javascript
card.insertAdjacentElement(
  "afterbegin",
  heading
);
```

Equivalent idea:

```javascript
card.prepend(heading);
```

Verification:

```javascript
card.firstElementChild ===
  heading;
```

---

# 27. `beforeend`

Inserts inside the reference as its last child.

```javascript
card.insertAdjacentElement(
  "beforeend",
  footer
);
```

Equivalent idea:

```javascript
card.append(footer);
```

Verification:

```javascript
card.lastElementChild ===
  footer;
```

---

# 28. `afterend`

Inserts immediately after the reference element.

```javascript
card.insertAdjacentElement(
  "afterend",
  confirmation
);
```

Equivalent idea:

```javascript
card.after(confirmation);
```

The inserted element becomes a following sibling.

---

# 29. Position Diagram

```text
<!-- beforebegin -->

<section id="reference">

  <!-- afterbegin -->

  <p>Existing child</p>

  <!-- beforeend -->

</section>

<!-- afterend -->
```

Method mapping:

| Position      | Modern equivalent            |
| ------------- | ---------------------------- |
| `beforebegin` | `reference.before(element)`  |
| `afterbegin`  | `reference.prepend(element)` |
| `beforeend`   | `reference.append(element)`  |
| `afterend`    | `reference.after(element)`   |

---

# 30. `insertAdjacentElement()` Return Value

When successful, it returns the inserted element.

```javascript
const result =
  reference
    .insertAdjacentElement(
      "afterend",
      message
    );

console.log(
  result === message
);
```

Output:

```text
true
```

For outer positions such as `beforebegin` and `afterend`, insertion requires the reference element to have an eligible parent.

If insertion cannot occur because the required parent relationship is absent, the method may return `null`.

---

# 31. Invalid Positions

This is invalid:

```javascript
element.insertAdjacentElement(
  "middle",
  node
);
```

Only the four recognized keywords are allowed.

An invalid position causes an exception.

A defensive helper can validate positions:

```javascript
const validPositions =
  new Set([
    "beforebegin",
    "afterbegin",
    "beforeend",
    "afterend",
  ]);

function insertElementAt(
  reference,
  position,
  element
) {
  if (
    !validPositions.has(
      position
    )
  ) {
    throw new TypeError(
      `Invalid insertion position: ${position}`
    );
  }

  return reference
    .insertAdjacentElement(
      position,
      element
    );
}
```

---

# 32. `insertAdjacentElement()` vs Modern Convenience Methods

Prefer:

```javascript
reference.before(node);
reference.after(node);
reference.prepend(node);
reference.append(node);
```

when:

* The intended position is simple.
* Multiple nodes may be inserted.
* Strings may be inserted.
* Readability is the priority.

Use:

```javascript
insertAdjacentElement()
```

when:

* A position is represented dynamically.
* One shared helper handles four positions.
* You are working with APIs using the position-string vocabulary.
* You need its returned element.

Example dynamic placement:

```javascript
function placeElement(
  reference,
  position,
  element
) {
  return reference
    .insertAdjacentElement(
      position,
      element
    );
}
```

---

# 33. `insertAdjacentText()`

`insertAdjacentText()` inserts plain text at one of the four relative positions.

Syntax:

```javascript
element.insertAdjacentText(
  position,
  text
);
```

Example:

```javascript
amount.insertAdjacentText(
  "beforebegin",
  "Amount: "
);
```

The inserted string becomes text, not HTML.

---

# 34. `insertAdjacentText()` Syntax

```javascript
reference.insertAdjacentText(
  "beforebegin",
  "Plain text"
);
```

The position uses the same four keywords:

```text
beforebegin
afterbegin
beforeend
afterend
```

The second argument is converted to text.

---

# 35. Safe Text Insertion

Suppose user input is:

```javascript
const userValue =
  "<img src=x onerror=alert(1)>";
```

Using:

```javascript
container.insertAdjacentText(
  "beforeend",
  userValue
);
```

displays the characters as text.

It does not create an image element.

Safe alternatives include:

```javascript
element.textContent =
  userValue;
```

```javascript
element.append(
  userValue
);
```

```javascript
element.insertAdjacentText(
  "beforeend",
  userValue
);
```

# 36. `insertAdjacentHTML()`

`insertAdjacentHTML()` parses an HTML string and inserts the resulting nodes at one of the four positions.

Syntax:

```javascript
element.insertAdjacentHTML(
  position,
  htmlString
);
```

Example:

```javascript
list.insertAdjacentHTML(
  "beforeend",
  `
    <li class="service-item">
      Printing
    </li>
  `
);
```

The browser parses the string into DOM nodes.

---

# 37. `insertAdjacentHTML()` Syntax

```javascript
reference.insertAdjacentHTML(
  "beforeend",
  html
);
```

Arguments:

1. Position keyword.
2. HTML markup string.

Return value:

```text
undefined
```

Unlike `insertAdjacentElement()`, it does not return a direct reference to the inserted element.

You may need to query or capture a neighboring element afterward.

---

# 38. How HTML String Insertion Works

Code:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  `
    <p class="message">
      Saved
    </p>
  `
);
```

Conceptual browser work:

```text
Receive HTML string
    ↓
Parse markup
    ↓
Create p element
    ↓
Create text node
    ↓
Insert at beforeend
```

Result:

```html
<div id="container">
  <p class="message">
    Saved
  </p>
</div>
```

The markup is parsed, unlike strings supplied to `append()`.

---

# 39. `insertAdjacentHTML()` Security Risks

Never insert untrusted values directly into an HTML string.

Unsafe:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  `
    <p>
      ${userInput}
    </p>
  `
);
```

If `userInput` contains markup, the browser parses it.

This may create cross-site scripting vulnerabilities.

Safer:

```javascript
const paragraph =
  document.createElement(
    "p"
  );

paragraph.textContent =
  userInput;

container.append(paragraph);
```

Use `insertAdjacentHTML()` only when:

* The markup is trusted.
* The source is controlled.
* Values are correctly sanitized using a reliable strategy.
* Parsing HTML is genuinely necessary.

---

# 40. `insertAdjacentHTML()` vs `innerHTML`

## `innerHTML`

```javascript
container.innerHTML =
  html;
```

Replaces all current descendants.

Existing descendant references may become stale.

## `insertAdjacentHTML()`

```javascript
container.insertAdjacentHTML(
  "beforeend",
  html
);
```

Adds parsed content without replacing existing descendants.

Advantages over:

```javascript
container.innerHTML +=
  html;
```

The `+=` approach conceptually reads and reparses the complete current content, potentially replacing existing descendant objects.

`insertAdjacentHTML()` adds only at the chosen position.

However, both APIs parse HTML and require security care.

Use this decision guide.

## Insert a node as the last child



# 49. Dynamic Sorting and Reordering

Relative methods move existing nodes.


```javascript

  movingCard
```

```javascript

  item
) {
  const previous =
    item.previousElementSibling;

  if (!previous) {
    return;

  previous.before(item);
}
```

A clearer implementation is:

```javascript
previous.before(item);
```

```text

previous
item

After:
item
previous
```

```javascript
function moveItemDown(
  item

    item.nextElementSibling;

    return;
```

# 53. Performance Considerations

Relative insertion is generally efficient for ordinary interface operations.

Performance concerns arise when:

* Inserting thousands of nodes individually.
* Repeatedly inserting into large connected structures.
* Measuring layout after every insertion.
* Reordering large lists one node at a time.
* Parsing large HTML strings.
* Repeatedly querying reference elements.

Best practices:

1. Resolve the reference once.
2. Create and configure nodes while detached.
3. Insert batches where possible.
4. Avoid layout-read/write alternation.
5. Use fragments or complete grouped components for large sets.
6. Keep the DOM reasonably sized.
7. Measure realistic workflows.

---

# 54. Memory Considerations

Relative insertion moves existing nodes without cloning.

A node moved with:

```javascript
reference.after(node);
```

keeps:

* Its properties.
* Its descendants.
* Its listeners.
* JavaScript references to it.

Inserted HTML strings create new node objects.

Old references are not created unless your code stores them.

Removing or replacing content later requires lifecycle cleanup when external listeners, registries, or observers exist.

## Example 3: Real-World Application — Insert a Transaction Relative to Another Transaction

### State

```javascript
const state = {
  transactions: [
    {
      id: 101,
      receipt: "RCT-101",
      amount: 500,
    },
    {
      id: 102,
      receipt: "RCT-102",
      amount: 800,
    },
  ],
};
```

### Goal

Insert transaction `103` after transaction `101` in both:

* Application state.
* DOM order.

### Helper

```javascript
function findTransactionIndex(
  id
) {
  return state.transactions
    .findIndex(
      transaction =>
        transaction.id === id
    );
}
```

### Insertion function

```javascript
function insertTransactionAfterId(
  referenceId,
  transaction,
  container
) {
  const referenceIndex =
    findTransactionIndex(
      referenceId
    );

  if (
    referenceIndex === -1
  ) {
    throw new Error(
      `Transaction ${referenceId} was not found in state.`
    );
  }

  if (
    state.transactions.some(
      current =>
        current.id ===
        transaction.id
    )
  ) {
    throw new Error(
      `Transaction ${transaction.id} already exists.`
    );
  }

  const referenceCard =
    [
      ...container.children,
    ].find(
      child =>
        Number(
          child.dataset
            .transactionId
        ) === referenceId
    );

  if (!referenceCard) {
    throw new Error(
      `Transaction ${referenceId} was not found in the DOM.`
    );
  }

  const newCard =
    createTransactionCard(
      transaction
    );

  state.transactions.splice(
    referenceIndex + 1,
    0,
    transaction
  );

  referenceCard.after(
    newCard
  );

  return newCard;
}
```

### Resulting state

```javascript
[
  {
    id: 101,
  },
  {
    id: 103,
  },
  {
    id: 102,
  },
]
```

### Resulting DOM

```text
Card 101
Card 103
Card 102
```

### Why this is professional

* State remains authoritative.
* Duplicate identity is rejected.
* The DOM reference is locally resolved.
* State order and DOM order are updated together.
* The node is created through a reusable factory.
* The insertion function returns the new component root.

# 56. Common Mistakes

## Mistake 1: Confusing siblings with children

```javascript
input.after(error);
```

makes `error` a sibling.

It does not make it a child of the input.

Inputs cannot contain ordinary child content.

---

## Mistake 2: Calling `before()` or `after()` on a detached reference

A detached reference has no parent into which a sibling can be inserted.

---

## Mistake 3: Expecting `before()` to return the inserted node

It returns `undefined`.

---

## Mistake 4: Using an invalid `insertAdjacentElement()` position

Only four keywords are valid.

---

## Mistake 5: Using `afterbegin` when `afterend` was intended

```text
afterbegin:
Inside the reference

afterend:
Outside, after the reference
```

---

## Mistake 6: Assuming relative insertion copies a node

It moves the existing node.

---

## Mistake 7: Inserting invalid table or list structure

Use semantic parent-child relationships.

---

## Mistake 8: Updating DOM order but not state order

The next full render may restore the old state order.

---

## Mistake 9: Inserting duplicate validation messages

Check for an existing message before inserting another.

```javascript
const existingError =
  input.parentElement
    ?.querySelector(
      "[data-role='error']"
    );
```

---

## Mistake 10: Using `insertAdjacentHTML()` with untrusted input

This creates an injection risk.

---

## Mistake 11: Using `innerHTML +=` for repeated insertion

This may recreate existing descendants and invalidate references.

---

## Mistake 12: Focusing every inserted element

Focus changes should follow user intent and accessibility needs.

# 57. Professional Best Practices

1. Use `before()` and `after()` when the reference relationship is simple.
2. Use `prepend()` and `append()` for child insertion.
3. Use `insertAdjacentElement()` when a dynamic four-position API is helpful.
4. Use `insertAdjacentText()` for plain relative text.
5. Use `insertAdjacentHTML()` only with trusted or properly sanitized markup.
6. Create nodes while detached.
7. Validate reference elements before insertion.
8. Verify component boundaries.
9. Keep state order synchronized with DOM order.
10. Prevent duplicate status and validation messages.
11. Use semantic inserted structures.
12. Add accessibility relationships during insertion.
13. Manage focus only when the workflow requires it.
14. Remember that insertion moves existing nodes.
15. Store reusable references when needed.
16. Prefer readable insertion intent over manual parent calculations.
17. Avoid broad document queries for local references.
18. Return inserted roots from higher-level helper functions.
19. Use DOM breakpoints to diagnose unexpected placement.
20. Test first, last, and missing-reference edge cases.

# 59. Interview Questions and Answers

## Beginner Questions

### 1. What does `before()` do?

It inserts nodes or strings immediately before the reference element.

### 2. What does `after()` do?

It inserts nodes or strings immediately after the reference element.

### 3. Do `before()` and `after()` insert children?

No. They insert siblings.

### 4. What do `before()` and `after()` return?

`undefined`.

### 5. Can they accept multiple nodes?

Yes.

### 6. Can they accept strings?

Yes. Strings become text nodes.

### 7. What does `afterbegin` mean?

Inside the reference element before its existing children.

### 8. What does `beforeend` mean?

Inside the reference element after its existing children.

---

## Intermediate Questions

### 1. Compare `after()` with `append()`.

`after()` inserts a following sibling. `append()` inserts a last child.

### 2. Compare `before()` with `prepend()`.

`before()` inserts a previous sibling. `prepend()` inserts a first child.

### 3. What does `insertAdjacentElement()` return?

The inserted element when successful, or potentially `null` when an outer insertion cannot occur.

### 4. Why must a reference have a parent for `beforebegin` or `afterend`?

Those positions create siblings, which require a shared parent.

### 5. Does relative insertion clone nodes?

No. Existing nodes move.

### 6. Why is `insertAdjacentText()` safer for untrusted strings?

It inserts text instead of parsing HTML.

### 7. Why can `insertAdjacentHTML()` be dangerous?

Untrusted markup can be parsed into executable or malicious DOM content.

---

## Advanced Questions

### 1. Why may `insertAdjacentHTML()` be preferable to `innerHTML +=`?

It inserts at one position without reparsing and replacing all existing descendants.

### 2. How would you insert a new record after another while keeping state synchronized?

Find the reference index in state, insert the new record into the state array, resolve the reference DOM node, create the new component, and insert it with `after()`.

### 3. How would you avoid duplicate field-error elements?

Query the local field group for an existing error and update or replace it instead of inserting another.

### 4. What accessibility concerns apply to relative insertion?

DOM reading order, focus movement, live announcements, control-message relationships, and semantic hierarchy.

### 5. How do you move an item upward using relative insertion?

Insert the item before its previous sibling.

```javascript
previous.before(item);
```

### 6. What happens to listeners when a node is moved?

Listeners attached to the node remain attached because the same object is moved.

Relative insertion places content according to an existing reference element.

You learned:

## `before()`

* Inserts siblings before a reference.
* Accepts multiple nodes and strings.
* Returns `undefined`.

## `after()`

* Inserts siblings after a reference.
* Accepts multiple nodes and strings.
* Returns `undefined`.

## `insertAdjacentElement()`

Uses four positions:

```text

afterbegin
beforeend
afterend
```

It inserts one element and returns the inserted element when successful.

## `insertAdjacentText()`

Inserts plain text at one of the four positions.

Parses and inserts HTML markup.

It must not receive untrusted content without a reliable sanitization strategy.

* `before()` and `after()` create sibling relationships.
* `prepend()` and `append()` create child relationships.
* Existing nodes move rather than copy.
* Outer relative insertion requires a parent.
* DOM order and state order should stay synchronized.
* Relative insertion supports forms, tables, cards, lists, and menus.
* Accessibility relationships should be created during insertion.
* Focus should move only when appropriate.
* Semantic hierarchy must remain valid.
* `insertAdjacentHTML()` does not replace existing descendants like `innerHTML =`.
* Trusted HTML insertion and safe text insertion are different workflows.

```text

    ├── First child
    ├── Last child
    └── After sibling
    ↓

    ↓
Update state, focus, and accessibility
```
