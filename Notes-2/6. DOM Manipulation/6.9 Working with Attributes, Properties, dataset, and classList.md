# 1. Introduction

An HTML element has more than visible text and child elements.

It also has metadata and state.

Consider:

```html
<button
  id="save-button"
  class="button button-primary"
  type="button"
  disabled
  data-action="save"
  aria-label="Save transaction"
>
  Save
</button>
```

This one element contains several different concepts:

* `id` identifies the element.
* `class` connects it to CSS and visual state.
* `type` controls button behavior.
* `disabled` controls interactivity.
* `data-action` provides application metadata.
* `aria-label` gives it an accessible name.

JavaScript can interact with this information through two related systems:

```text
HTML attributes
DOM properties
```

It also provides specialized APIs:

```javascript
dataset
className
classList
```

A professional JavaScript developer must understand which API to use and why.

---

# 2. The Core Problem

Suppose the page contains:

```html
<input
  id="username"
  value="Amina"
>
```

The user changes the field to:

```text
Brian
```

What does this return?

```javascript
input.getAttribute(
  "value"
);
```

It may still return:

```text
Amina
```

But:

```javascript
input.value;
```

returns:

```text
Brian
```

Why?

Because the HTML `value` attribute often represents the initial or default value, while the DOM `.value` property represents the current live value.

This lesson explains such differences in depth.

---

# 3. What Is an HTML Attribute?

An attribute is metadata written in an element's opening HTML tag.

```html
<input
  id="email"
  type="email"
  required
>
```

Attributes here are:

```text
id
type
required
```

Attributes originate in HTML source or are added later using DOM attribute methods.

They are represented primarily as name-value pairs.

```text
name = value
```

Boolean attributes may appear without an explicit value:

```html
required
disabled
checked
```

Their presence is what matters.

---

# 4. What Is a DOM Property?

A DOM property is a JavaScript property on a browser-created object.

```javascript
input.id;
input.type;
input.required;
input.value;
input.disabled;
```

The browser converts HTML elements into JavaScript-accessible objects.

An `<input>` element commonly becomes an `HTMLInputElement`.

```javascript
input instanceof
  HTMLInputElement;
```

The object exposes:

* General node properties.
* Element properties.
* HTMLElement properties.
* Input-specific properties.
* Methods.
* Event-related features.
* Runtime state.

---

# 5. Attributes vs Properties

Consider:

```html
<input
  id="email"
  type="email"
  required
>
```

Attribute access:

```javascript
input.getAttribute(
  "type"
);
```

Property access:

```javascript
input.type;
```

Both may return:

```text
email
```

But they are not the same mechanism.

| Attributes                        | Properties                                 |
| --------------------------------- | ------------------------------------------ |
| Stored as HTML metadata           | Stored on DOM objects                      |
| Mostly string-based               | Can be strings, Booleans, numbers, objects |
| Accessed with attribute APIs      | Accessed with JavaScript property syntax   |
| Often represent initial markup    | May represent current live state           |
| Can include custom `data-*` names | Defined by DOM interfaces                  |

---

# 6. The Initial Parsing Process

Given:

```html
<button
  id="save"
  disabled
>
  Save
</button>
```

The browser conceptually:

```text
1. Parses the button element
2. Reads its attributes
3. Creates an HTMLButtonElement object
4. Initializes related DOM properties
5. Creates child text content
6. Inserts the object into the DOM tree
```

This is why:

```javascript
button.id;
```

reflects the `id` attribute.

And:

```javascript
button.disabled;
```

reflects the presence of the `disabled` attribute.

---

# 7. Attribute Reflection

Reflection means a DOM property and an HTML attribute represent related information.

Example:

```javascript
element.id =
  "summary";
```

often updates:

```html
id="summary"
```

And:

```javascript
element.setAttribute(
  "id",
  "report"
);
```

often updates:

```javascript
element.id;
```

to:

```text
report
```

Not every property reflects in the same way.

Some properties:

* Reflect directly.
* Normalize values.
* Represent current live state.
* Represent default state.
* Do not correspond to attributes at all.

---

# 8. When Attributes and Properties Stay Synchronized

Example:

```javascript
const link =
  document.createElement(
    "a"
  );

link.id =
  "reports-link";
```

Then:

```javascript
link.getAttribute(
  "id"
);
```

returns:

```text
reports-link
```

Set through attribute:

```javascript
link.setAttribute(
  "title",
  "View reports"
);
```

Then:

```javascript
link.title;
```

returns:

```text
View reports
```

These are reflected attributes.

---

# 9. When Attributes and Properties Diverge

Form controls are the most important examples.

```html
<input
  value="Original"
>
```

Initially:

```javascript
input.getAttribute(
  "value"
);
```

and:

```javascript
input.value;
```

may both return:

```text
Original
```

After the user types:

```text
Changed
```

the property changes:

```javascript
input.value;
```

returns:

```text
Changed
```

The attribute may remain:

```text
Original
```

The attribute describes the default.

The property describes the current value.

---

# 10. `getAttribute()`

`getAttribute()` reads the raw value of an attribute.

Syntax:

```javascript
const value =
  element.getAttribute(
    attributeName
  );
```

Example:

```javascript
const role =
  card.getAttribute(
    "data-role"
  );
```

If the attribute is absent:

```javascript
element.getAttribute(
  "missing"
);
```

returns:

```text
null
```

---

# 11. `setAttribute()`

`setAttribute()` creates or updates an attribute.

```javascript
element.setAttribute(
  name,
  value
);
```

Example:

```javascript
button.setAttribute(
  "aria-expanded",
  "false"
);
```

Values are converted to strings.

```javascript
element.setAttribute(
  "data-count",
  5
);
```

becomes conceptually:

```html
data-count="5"
```

---

# 12. `removeAttribute()`

Removes an attribute.

```javascript
element.removeAttribute(
  "hidden"
);
```

Example:

```javascript
input.removeAttribute(
  "aria-invalid"
);
```

The method returns:

```text
undefined
```

Removing an absent attribute does not normally cause an error.

---

# 13. `hasAttribute()`

Checks whether an attribute exists.

```javascript
element.hasAttribute(
  "disabled"
);
```

Returns:

```text
true
```

or:

```text
false
```

This is especially useful for Boolean attributes where presence is the important condition.

---

# 14. `toggleAttribute()`

Adds or removes an attribute.

```javascript
element.toggleAttribute(
  "hidden"
);
```

If absent, it is added.

If present, it is removed.

Use the optional force argument:

```javascript
element.toggleAttribute(
  "hidden",
  shouldHide
);
```

This is usually clearer because it makes the result deterministic.

---

# 15. `getAttributeNames()`

Returns an array of attribute names.

```javascript
const names =
  element.getAttributeNames();
```

Example output:

```javascript
[
  "id",
  "class",
  "data-action",
  "aria-label",
]
```

Useful for:

* Debugging.
* Generic DOM tools.
* Validation.
* Attribute auditing.

---

# 16. The `attributes` Collection

```javascript
element.attributes;
```

returns a `NamedNodeMap`.

You can inspect:

```javascript
for (
  const attribute
  of element.attributes
) {
  console.log(
    attribute.name,
    attribute.value
  );
}
```

For normal application code, specialized APIs are usually clearer than working directly with attribute nodes.

---

# 17. Standard Properties

Examples:

```javascript
element.id
element.title
element.hidden
element.lang
element.tabIndex
```

Form controls:

```javascript
input.value
input.checked
input.disabled
input.required
input.readOnly
```

Links:

```javascript
link.href
link.target
link.rel
```

Images:

```javascript
image.src
image.alt
image.width
image.height
```

Use standard properties when working with the current live state of standard DOM features.

---

# 18. Property Types

Properties can return different JavaScript types.

```javascript
input.disabled;
```

returns a Boolean.

```javascript
input.value;
```

returns a string.

```javascript
input.selectionStart;
```

returns a number or `null`.

```javascript
element.classList;
```

returns a `DOMTokenList`.

```javascript
element.style;
```

returns a style declaration object.

Attributes, by contrast, are generally read as strings or `null`.

---

# 19. String Attributes vs Typed Properties

HTML:

```html
<input
  maxlength="10"
>
```

Attribute:

```javascript
input.getAttribute(
  "maxlength"
);
```

returns:

```text
"10"
```

Property:

```javascript
input.maxLength;
```

returns:

```javascript
10
```

The property provides a number.

This typed behavior is one reason properties are often better for active JavaScript logic.

---

# 20. Boolean Attributes

Boolean attributes represent true through presence.

Examples:

```html
disabled
checked
required
readonly
multiple
autofocus
hidden
```

These do not behave like ordinary string-valued attributes.

```html
<button disabled="false">
```

is still disabled because the attribute is present.

---

# 21. `disabled`

Correct property control:

```javascript
button.disabled =
  true;
```

Enable:

```javascript
button.disabled =
  false;
```

Attribute control:

```javascript
button.setAttribute(
  "disabled",
  ""
);
```

Enable:

```javascript
button.removeAttribute(
  "disabled"
);
```

Property syntax is clearer for live Boolean state.

---

# 22. `checked`

```javascript
checkbox.checked =
  true;
```

represents the current checked state.

The `checked` attribute represents the default checked state in markup.

Use `.checked` for current user-interface state.

---

# 23. `required`

```javascript
input.required =
  true;
```

sets the field as required.

```javascript
input.required =
  false;
```

removes the requirement.

This is clearer than setting a string attribute value.

---

# 24. `readonly`

DOM property name:

```javascript
input.readOnly =
  true;
```

The HTML attribute is:

```html
readonly
```

The property uses camelCase because `readonly` conflicts with JavaScript naming conventions and DOM API design.

---

# 25. `hidden`

```javascript
element.hidden =
  true;
```

hides the element using the HTML hidden state.

Show:

```javascript
element.hidden =
  false;
```

This is semantic hiding, not just a visual CSS class.

---

# 26. Presence, Not String Value

Incorrect:

```javascript
button.setAttribute(
  "disabled",
  "false"
);
```

The button remains disabled.

Correct:

```javascript
button.disabled =
  false;
```

or:

```javascript
button.removeAttribute(
  "disabled"
);
```

Professional rule:

> Control Boolean DOM state through Boolean properties whenever possible.

---

# 27. `value` Attribute vs `.value` Property

Markup:

```html
<input
  value="Initial"
>
```

Attribute:

```javascript
input.getAttribute(
  "value"
);
```

represents the initial/default markup value.

Property:

```javascript
input.value;
```

represents current live value.

When the user types, `.value` changes.

The attribute usually does not automatically follow every keystroke.

---

# 28. `defaultValue`

```javascript
input.defaultValue;
```

represents the default value associated with the value attribute.

Example:

```javascript
input.defaultValue =
  "Amina";
```

A form reset may restore the control to its default value.

Understand three concepts:

```text
value attribute
defaultValue property
current value property
```

They are related but not identical.

---

# 29. `checked` Attribute vs `.checked` Property

Markup:

```html
<input
  type="checkbox"
  checked
>
```

Initial current state:

```javascript
checkbox.checked;
```

returns:

```text
true
```

After user interaction:

```javascript
checkbox.checked;
```

may become:

```text
false
```

The original attribute may still exist.

Use `.checked` for current state.

---

# 30. `defaultChecked`

```javascript
checkbox.defaultChecked;
```

represents the default checked state.

Form reset behavior may use it.

```text
checked attribute
    ↔
defaultChecked

current interaction
    ↔
checked property
```

---

# 31. `selected` Attribute vs `.selected` Property

An option may have:

```html
<option selected>
  Cash
</option>
```

The current selected state is:

```javascript
option.selected;
```

For the entire select control, usually use:

```javascript
select.value;
```

or:

```javascript
select.selectedIndex;
```

Do not rely only on the original `selected` attribute after user interaction.

---

# 32. URL Attributes vs URL Properties

HTML:

```html
<a
  href="/reports"
>
  Reports
</a>
```

Attribute:

```javascript
link.getAttribute(
  "href"
);
```

may return:

```text
/reports
```

Property:

```javascript
link.href;
```

may return an absolute resolved URL, such as:

```text
https://example.com/reports
```

The browser resolves the property relative to the document base URL.

---

# 33. `href`

Use property assignment:

```javascript
link.href =
  "/reports";
```

For external destinations, validate allowed protocols and destinations.

Do not assign untrusted URL strings without validation.

---

# 34. `src`

Image:

```javascript
image.src =
  "/images/receipt.png";
```

The property may return an absolute resolved URL.

Attribute:

```javascript
image.getAttribute(
  "src"
);
```

may return the original relative value.

---

# 35. Relative and Resolved URLs

```html
<img src="photo.png">
```

Attribute:

```text
photo.png
```

Property may become:

```text
https://example.com/current/path/photo.png
```

Use:

* Attribute when exact markup value matters.
* Property when resolved usable URL matters.

---

# 36. Property Names That Differ from Attribute Names

Some DOM property names differ because JavaScript naming conventions or language conflicts require different names.

Common examples:

| HTML Attribute | DOM Property |
| -------------- | ------------ |
| `class`        | `className`  |
| `for`          | `htmlFor`    |
| `readonly`     | `readOnly`   |
| `tabindex`     | `tabIndex`   |
| `maxlength`    | `maxLength`  |
| `colspan`      | `colSpan`    |

---

# 37. `class` vs `className`

Attribute:

```javascript
element.getAttribute(
  "class"
);
```

Property:

```javascript
element.className;
```

Set complete class string:

```javascript
element.className =
  "card active";
```

This replaces all current classes.

For individual class operations, use `classList`.

---

# 38. `for` vs `htmlFor`

HTML:

```html
<label for="email">
```

JavaScript:

```javascript
label.htmlFor =
  "email";
```

This associates the label with:

```html
<input id="email">
```

The relationship improves usability and accessibility.

---

# 39. `readonly` vs `readOnly`

```javascript
input.readOnly =
  true;
```

Use camelCase in JavaScript.

The HTML serialization becomes:

```html
readonly
```

---

# 40. `tabindex` vs `tabIndex`

```javascript
element.tabIndex =
  -1;
```

This allows programmatic focus without placing the element in the normal positive tab sequence.

Attribute:

```html
tabindex="-1"
```

---

# 41. Custom Attributes

Browsers allow unknown attributes in HTML, but application-specific metadata should normally use `data-*`.

Weak:

```html
<article transaction-id="42">
```

Better:

```html
<article data-transaction-id="42">
```

Benefits:

* Standards-compliant purpose.
* Accessible through `dataset`.
* Clear application metadata.
* Easy CSS and query selection.

---

# 42. Why `data-*` Exists

`data-*` attributes allow custom application data to be stored on elements.

Examples:

```html
data-transaction-id="42"
data-component="transaction-card"
data-action="delete"
data-status="pending"
```

They are useful for connecting:

```text
DOM structure
    ↔
Application behavior
```

They should not replace a proper state store.

---

# 43. The `dataset` API

Given:

```html
<article
  data-transaction-id="42"
  data-status="pending"
>
```

Read:

```javascript
card.dataset
  .transactionId;
```

```javascript
card.dataset.status;
```

`dataset` is a `DOMStringMap`.

---

# 44. `data-*` to camelCase Conversion

HTML:

```html
data-transaction-id
```

JavaScript:

```javascript
dataset.transactionId
```

HTML:

```html
data-payment-mode
```

JavaScript:

```javascript
dataset.paymentMode
```

Hyphen-separated names become camelCase.

---

# 45. Reading Dataset Values

```javascript
const id =
  card.dataset
    .transactionId;
```

The result is a string:

```text
"42"
```

Check for presence:

```javascript
if (
  card.dataset
    .transactionId !==
    undefined
) {
  // Present.
}
```

You can also use:

```javascript
card.hasAttribute(
  "data-transaction-id"
);
```

---

# 46. Writing Dataset Values

```javascript
card.dataset.status =
  "paid";
```

Result:

```html
data-status="paid"
```

Numbers should be converted intentionally:

```javascript
card.dataset.transactionId =
  String(transaction.id);
```

---

# 47. Deleting Dataset Values

Use the `delete` operator:

```javascript
delete card.dataset.status;
```

This removes:

```html
data-status
```

You can also use:

```javascript
card.removeAttribute(
  "data-status"
);
```

---

# 48. Dataset Values Are Strings

```javascript
card.dataset.quantity =
  3;
```

Reading:

```javascript
typeof card.dataset.quantity;
```

returns:

```text
string
```

The value is:

```text
"3"
```

Similarly:

```javascript
card.dataset.active =
  false;
```

becomes:

```text
"false"
```

which is still a truthy JavaScript string.

---

# 49. Parsing Numbers and Booleans from Dataset

Number:

```javascript
const transactionId =
  Number(
    card.dataset
      .transactionId
  );
```

Validate:

```javascript
if (
  !Number.isInteger(
    transactionId
  )
) {
  throw new Error(
    "Invalid transaction ID."
  );
}
```

Boolean:

```javascript
const isActive =
  card.dataset.active ===
  "true";
```

Do not use:

```javascript
Boolean(
  card.dataset.active
);
```

because:

```javascript
Boolean("false");
```

is:

```text
true
```

---

# 50. What Should Be Stored in `dataset`

Good uses:

* Record IDs.
* Action names.
* Component names.
* Semantic region names.
* Simple UI state.
* Sort keys.
* Filter categories.
* Small string metadata.

Examples:

```html
data-action="delete"
data-component="transaction-card"
data-transaction-id="42"
data-status="pending"
```

---

# 51. What Should Not Be Stored in `dataset`

Avoid storing:

* Passwords.
* Authentication tokens.
* Secret API keys.
* Large JSON objects.
* Complete database records.
* Sensitive personal information.
* Authoritative financial values.
* Functions.
* Complex application state.

Dataset is visible in the DOM and developer tools.

It is not secure storage.

---

# 52. Component Metadata Conventions

A consistent convention makes component logic easier.

Common pattern:

```html
data-component
data-role
data-action
data-id
data-state
```

These names have different responsibilities.

---

# 53. `data-component`

Marks a component root.

```html
<article
  data-component="transaction-card"
>
```

Use for:

* Ownership.
* Component discovery.
* Boundary checks.
* Registry initialization.
* Debugging.

---

# 54. `data-role`

Marks an internal semantic part.

```html
<p data-role="amount"></p>
```

Use for:

* Scoped queries.
* View construction.
* Component contracts.
* Updates.

It describes what the element represents inside the component.

---

# 55. `data-action`

Marks an interactive command.

```html
<button
  data-action="delete"
>
```

Use with event delegation:

```javascript
const action =
  event.target.closest(
    "[data-action]"
  );
```

It describes what should happen.

---

# 56. `data-id` and Record Identity

Prefer a descriptive name:

```html
data-transaction-id="42"
```

rather than generic:

```html
data-id="42"
```

Descriptive identity improves debugging and reduces ambiguity in nested components.

---

# 57. `className`

Read complete class string:

```javascript
element.className;
```

Set complete class string:

```javascript
element.className =
  "transaction-card is-paid";
```

This replaces every previous class.

Use when the entire class string is intentionally controlled.

---

# 58. The `classList` API

`classList` returns a token-based interface.

```javascript
element.classList;
```

It supports:

```javascript
add()
remove()
contains()
toggle()
replace()
```

It avoids manual string splitting and concatenation.

---

# 59. `classList.add()`

Add one class:

```javascript
card.classList.add(
  "is-selected"
);
```

Add several:

```javascript
card.classList.add(
  "transaction-card",
  "is-pending"
);
```

Existing identical classes are not duplicated.

---

# 60. `classList.remove()`

```javascript
card.classList.remove(
  "is-pending"
);
```

Remove several:

```javascript
card.classList.remove(
  "is-pending",
  "is-selected"
);
```

Removing a class that is absent does not normally cause an error.

---

# 61. `classList.contains()`

```javascript
card.classList.contains(
  "is-paid"
);
```

Returns a Boolean.

Use when visual class state genuinely needs inspection.

However, application state should usually remain authoritative.

---

# 62. `classList.toggle()`

```javascript
card.classList.toggle(
  "is-expanded"
);
```

Adds if absent and removes if present.

Useful for direct user toggles.

But deterministic state is often better.

---

# 63. The Second `toggle()` Argument

```javascript
card.classList.toggle(
  "is-paid",
  transaction.status ===
    "paid"
);
```

If the second argument is `true`, the class exists.

If `false`, it is removed.

This is ideal for rendering classes from state.

---

# 64. `classList.replace()`

```javascript
card.classList.replace(
  "is-pending",
  "is-paid"
);
```

Returns a Boolean indicating whether the old class was found and replaced.

Use when transitioning between mutually exclusive classes.

For complex state, deterministic removal and addition may be clearer.

---

# 65. Iterating over Classes

```javascript
for (
  const className
  of element.classList
) {
  console.log(className);
}
```

Convert to array:

```javascript
const classes =
  [
    ...element.classList,
  ];
```

Useful for diagnostics and testing.

---

# 66. `className` vs `classList`

| `className`                              | `classList`                     |
| ---------------------------------------- | ------------------------------- |
| Reads or replaces complete string        | Manages individual tokens       |
| Can accidentally remove existing classes | Preserves unrelated classes     |
| Useful for initial full assignment       | Best for state changes          |
| Requires manual string handling          | Provides class-specific methods |

---

# 67. Classes as Visual State

Examples:

```text
is-selected
is-loading
is-error
is-paid
is-collapsed
```

Classes should mainly drive styling and visual presentation.

```javascript
card.classList.toggle(
  "is-paid",
  transaction.status ===
    "paid"
);
```

Avoid using visual class names as the only business-state storage.

---

# 68. Attributes as Semantic State

Attributes can represent semantic and accessible state.

Examples:

```html
aria-expanded="true"
hidden
disabled
data-status="paid"
```

A menu toggle might update:

```javascript
button.setAttribute(
  "aria-expanded",
  String(isOpen)
);

menu.hidden =
  !isOpen;

menu.classList.toggle(
  "is-open",
  isOpen
);
```

Each mechanism serves a different purpose:

* `aria-expanded`: accessibility meaning.
* `hidden`: actual visibility state.
* `is-open`: CSS styling.
* Application state: source of truth.

---

# 69. Classes vs Data Attributes

Use classes for styling:

```html
class="transaction-card is-paid"
```

Use data attributes for application metadata:

```html
data-transaction-id="42"
data-status="paid"
```

Do not overload one system for every responsibility.

A CSS selector may still style data attributes:

```css
[data-status="paid"] {
  opacity: 0.8;
}
```

But clear conventions improve maintainability.

---

# 70. State Synchronization

Suppose state says:

```javascript
transaction.status =
  "paid";
```

The view may need to update:

```javascript
root.dataset.status =
  "paid";

root.classList.toggle(
  "is-paid",
  true
);

statusElement.textContent =
  "Paid";

markPaidButton.disabled =
  true;
```

One state change may affect several DOM representations.

Centralize this in a view method.

---

# 71. Application State as the Source of Truth

Weak:

```javascript
const isPaid =
  card.classList.contains(
    "is-paid"
  );
```

Better:

```javascript
const transaction =
  transactionStore.get(id);

const isPaid =
  transaction.status ===
  "paid";
```

The DOM reflects state.

It should not normally define business truth.

---

# 72. Rendering State into Attributes and Classes

```javascript
function renderTransactionStatus(
  view,
  transaction
) {
  const isPaid =
    transaction.status ===
    "paid";

  view.root.dataset.status =
    transaction.status;

  view.root.classList.toggle(
    "is-paid",
    isPaid
  );

  view.status.textContent =
    isPaid
      ? "Paid"
      : "Pending";

  view.markPaidButton.disabled =
    isPaid;
}
```

This creates deterministic UI rendering.

---

# 73. ARIA Attributes

ARIA attributes provide accessibility semantics when native HTML does not fully express the state.

Examples:

```html
aria-expanded
aria-controls
aria-invalid
aria-describedby
aria-live
aria-current
```

Use `setAttribute()`:

```javascript
button.setAttribute(
  "aria-expanded",
  "true"
);
```

ARIA reflection properties exist in some modern environments, but attribute syntax remains common and explicit.

---

# 74. `aria-expanded`

Used on a control that expands or collapses content.

```javascript
toggleButton.setAttribute(
  "aria-expanded",
  String(isOpen)
);
```

The value should reflect the actual state.

Do not update the visual class without updating accessible state.

---

# 75. `aria-hidden`

```javascript
icon.setAttribute(
  "aria-hidden",
  "true"
);
```

Common for decorative icons.

Be careful:

* `aria-hidden="false"` does not necessarily override hidden ancestors.
* Do not hide focusable active controls from assistive technology.
* Native `hidden` and CSS display state also matter.

---

# 76. `aria-invalid`

```javascript
input.setAttribute(
  "aria-invalid",
  "true"
);
```

Remove when valid:

```javascript
input.removeAttribute(
  "aria-invalid"
);
```

Connect the error using `aria-describedby`.

---

# 77. `aria-describedby`

```javascript
input.setAttribute(
  "aria-describedby",
  "email-error"
);
```

The referenced ID must exist and be unique.

If multiple descriptions are used, the attribute contains space-separated IDs.

Do not blindly remove all IDs when only one relationship should be removed.

---

# 78. ARIA Values Are Usually Strings

```javascript
button.setAttribute(
  "aria-expanded",
  false
);
```

is converted to:

```text
"false"
```

That is acceptable because ARIA attribute values are string tokens.

Do not confuse:

```html
aria-hidden="false"
```

with removing the attribute.

They can have different semantic meanings.

---

# 79. Native HTML vs ARIA

Prefer native elements and properties.

Use:

```html
<button disabled>
```

instead of:

```html
<div
  role="button"
  aria-disabled="true"
>
```

when a real button is appropriate.

ARIA supplements semantics.

It should not replace correct native HTML unnecessarily.

---

# 80. Attributes and CSS Selectors

CSS:

```css
[data-status="paid"] {
  border-color: green;
}
```

JavaScript query:

```javascript
container.querySelectorAll(
  '[data-status="paid"]'
);
```

Attribute selectors are useful for semantic state and component discovery.

Avoid building unsafe selectors from arbitrary values without escaping.

---

# 81. Attributes and Event Delegation

Markup:

```html
<button
  data-action="delete"
>
```

Handler:

```javascript
const actionElement =
  event.target.closest(
    "[data-action]"
  );

const action =
  actionElement
    ?.dataset.action;
```

Then route:

```javascript
switch (action) {
  case "delete":
    // Handle deletion.
    break;

  case "edit":
    // Handle editing.
    break;
}
```

Treat action values as a controlled allowlist.

---

# 82. Attributes and Templates

Templates should define stable structural metadata:

```html
data-component
data-role
data-action
```

Factories should assign instance-specific values:

```javascript
root.dataset.transactionId =
  String(transaction.id);
```

Avoid hard-coded repeated IDs inside templates.

# 83. Attributes and Cloning

Cloning copies attributes.

This includes:

* IDs.
* Classes.
* Dataset.
* ARIA relationships.
* Hidden state.
* Disabled state.

After cloning, normalize everything that must differ.

---

# 84. Attributes and Forms

Important current form properties include:

```javascript
input.value
input.checked
input.disabled
input.required
input.readOnly
select.value
textarea.value
```

Attribute methods are not the best way to read current user input.

Use form-control properties.

---

# 85. Security Considerations

Attributes can create security issues when they control:

* URLs.
* Resource loading.
* Navigation.
* Form submission.
* Script behavior.
* Embedded content.
* Event-handler attributes.

Avoid:

```javascript
element.setAttribute(
  "onclick",
  userInput
);
```

Avoid assigning untrusted URLs directly.

Validate protocols:

```text
https:
http:
```

according to application policy.

Do not store secrets in `data-*`.

Do not trust action names or record IDs merely because they came from the DOM.

Server-side authorization remains required.

---

# 86. Accessibility Considerations

Keep these synchronized:

* `disabled`.
* `hidden`.
* `aria-expanded`.
* `aria-invalid`.
* `aria-describedby`.
* Label `for`.
* Control `id`.
* Current visual state.
* Focusability.

A visual class such as:

```text
is-disabled
```

does not actually disable a button.

Use:

```javascript
button.disabled =
  true;
```

A collapsed class alone does not communicate state.

Update:

```javascript
aria-expanded
hidden
```

as appropriate.

---

# 87. Performance Considerations

Attribute and class changes are usually inexpensive.

However, they can trigger:

* Style recalculation.
* Layout.
* Paint.
* Resource loading.
* Accessibility-tree updates.

Repeated pattern:

```javascript
element.classList.add(...);
element.classList.remove(...);
element.setAttribute(...);
```

is normally fine for ordinary interfaces.

For large batches:

1. Update detached elements where possible.
2. Group state changes.
3. Avoid alternating layout reads and writes.
4. Do not repeatedly query the same element.
5. Measure realistic interactions.

---

# 88. Memory Considerations

Attributes and dataset strings belong to elements.

Large data stored in dataset increases DOM memory and HTML size.

Avoid:

```javascript
element.dataset.record =
  JSON.stringify(
    hugeRecord
  );
```

Keep full records in JavaScript state.

Store only identity or small metadata in the DOM.

## Example 2: Intermediate — Dynamic Form Field State

### HTML

```html
<div data-field="email">
  <label for="email">
    Email
  </label>

  <input
    id="email"
    name="email"
    type="email"
  >

  <p
    id="email-error"
    data-role="error"

  ></p>
</div>
```

### JavaScript

```javascript
function setFieldError(
  input,
  errorElement,
  message
) {
  const hasError =
    typeof message ===
      "string" &&
    message.trim() !== "";

  input.setAttribute(
    "aria-invalid",
    String(hasError)
  );

  input.classList.toggle(
    "is-invalid",
    hasError
  );

  if (hasError) {
    errorElement.textContent =
      message;

    errorElement.hidden =
      false;

    input.setAttribute(
      "aria-describedby",
      errorElement.id
    );

    return;
  }

  errorElement.textContent =
    "";

  errorElement.hidden =
    true;

  input.removeAttribute(
    "aria-describedby"
  );

  input.removeAttribute(
    "aria-invalid"
  );
}
```

### Use

```javascript
setFieldError(
  emailInput,
  emailError,
  "Enter a valid email address."
);
```

### Professional lesson

The function synchronizes content, classes, native hidden state, and ARIA relationships.

## Example 3: Real-World Application — Transaction Card Status

### State

```javascript
const transaction = {
  id: 42,
  receipt: "RCT-42",
  status: "pending",
  isActive: true,
};
```

### View

```javascript
const view = {
  root:
    card,

  status:
    card.querySelector(
      "[data-role='status']"
    ),

  markPaidButton:
    card.querySelector(
      "[data-action='mark-paid']"
    ),

  deleteButton:
    card.querySelector(
      "[data-action='delete']"
    ),

```

### Renderer

```javascript
function renderTransactionState(
  view,
  transaction
) {
  const isPaid =
    transaction.status ===
    "paid";

  const isActive =
    transaction.isActive;

  view.root.dataset
    .transactionId =
    String(transaction.id);

  view.root.dataset.status =
    transaction.status;

  view.root.dataset.active =
    String(isActive);

  view.root.classList.toggle(
    "is-paid",
    isPaid
  );

  view.root.classList.toggle(
    "is-pending",
    !isPaid

      : `Mark ${transaction.receipt} as paid`
  );

  view.deleteButton.disabled =
    !isActive;
}
```

* State remains authoritative.
* Dataset provides inspectable metadata.
* Classes control styling.
* Native properties control interaction.
* ARIA labels reflect current meaning.
* One renderer creates deterministic state.

## Mistake 1: Using `disabled="false"`

The attribute is still present, so the control is disabled.

---

## Mistake 2: Reading current input with `getAttribute("value")`

Use:

```javascript
input.value;
```

---

## Mistake 3: Treating dataset numbers as numbers

They are strings.

---

## Mistake 4: Treating `"false"` as a false Boolean

```javascript
Boolean("false")
```

is true.

---

## Mistake 5: Storing complete objects in dataset

They become strings and duplicate application state.

---

## Mistake 6: Replacing every class with `className`

This may remove unrelated structural classes.

---

## Mistake 7: Using classes as the only source of business state

State should exist in JavaScript data.

---

## Mistake 8: Updating visual state but not ARIA state

A menu appears open but still has:

```html
aria-expanded="false"
```

---

## Mistake 9: Hard-coding IDs in repeated templates

This breaks labels and ARIA references.

---

## Mistake 10: Using `setAttribute()` for every standard live property

## Mistake 11: Trusting a DOM record ID for authorization

The server must validate permissions.

## Mistake 12: Assigning unsafe URLs

Validate destinations and protocols.

# 91. Professional Best Practices

1. Use properties for current standard element state.
2. Use attribute methods for raw attributes, ARIA, and custom metadata.
3. Use Boolean properties for Boolean state.
4. Use `.value`, `.checked`, and `.selected` for current form state.
5. Use `dataset` for small non-sensitive metadata.
6. Convert dataset values explicitly.
7. Keep full business records in application state.
8. Use `data-component`, `data-role`, and `data-action` consistently.
9. Use descriptive identity attributes.
10. Use `classList` for individual class changes.
11. Use the second `toggle()` argument for deterministic rendering.
12. Keep classes focused on styling.
13. Keep semantic and accessibility state synchronized.
14. Prefer native HTML features over ARIA substitutes.
15. Generate unique IDs for repeated components.
16. Validate URLs before assignment.
17. Treat action names as a controlled allowlist.
18. Centralize state-to-view rendering.
19. Inspect attributes and properties separately during debugging.
20. Do not store secrets in the DOM.

# 93. Interview Questions and Answers

## Beginner Questions

### 1. What is an HTML attribute?

Metadata written on an element, usually as a name-value pair.

### 2. What is a DOM property?

A JavaScript property on a browser-created DOM object.

### 3. What does `getAttribute()` return when an attribute is absent?

`null`.

### 4. What does `setAttribute()` do?

Creates or updates an attribute.

### 5. What does `removeAttribute()` do?

Removes an attribute.

### 6. What does `hasAttribute()` return?

A Boolean.

### 7. What is `dataset`?

An interface for reading and writing `data-*` attributes.

### 8. What does `classList` manage?

Individual CSS class tokens.

---

## Intermediate Questions

### 1. Why can `.value` differ from `getAttribute("value")`?

The property represents current live input state, while the attribute often represents the initial/default value.

### 2. Why does `disabled="false"` still disable a button?

Boolean attributes are true through presence, regardless of the string value.

### 3. What type are dataset values?

Strings.

### 4. How does `data-transaction-id` map to JavaScript?

```javascript
dataset.transactionId
```

### 5. What is the benefit of `classList.toggle(name, condition)`?

It creates deterministic class state from a Boolean condition.

### 6. Compare `className` and `classList`.

`className` reads or replaces the complete class string. `classList` manages individual class tokens.

### 7. Why should IDs be unique in template instances?

Selectors, labels, ARIA references, and browser behavior rely on ID uniqueness.

---

## Advanced Questions

### 1. Explain attribute reflection.

It is the relationship where a DOM property represents and may synchronize with a corresponding HTML attribute, sometimes with normalization or type conversion.

### 2. Why are form-control properties called live state?

They reflect the control's current interactive state, which may diverge from the initial markup attributes.

### 3. How would you design a component metadata convention?

Use component roots marked by `data-component`, internal parts by `data-role`, commands by `data-action`, and descriptive stable record IDs for identity.

### 4. How should application state relate to DOM attributes and classes?

Application state should remain authoritative, while rendering code derives attributes, properties, content, and classes from it.

### 5. What security issues apply to attribute assignment?

Unsafe URLs, event-handler attributes, embedded content, form destinations, and DOM-stored sensitive data can create vulnerabilities.

### 6. How do native state and ARIA state differ?

Native properties control actual browser behavior, while ARIA communicates semantics to accessibility APIs. ARIA does not automatically implement native behavior.

# 97. Summary

HTML attributes and DOM properties are related but distinct systems.

You learned attribute methods:

```javascript
getAttribute()
setAttribute()
removeAttribute()
hasAttribute()
toggleAttribute()
getAttributeNames()
```

You learned that properties:

* Live on DOM objects.
* Can have typed values.
* Often represent current interactive state.
* May reflect attributes.
* May normalize values.

You learned critical form distinctions:

```text
value attribute
defaultValue property
value property
```

```text
checked attribute
defaultChecked property
checked property
```

You learned Boolean attributes:

* Depend on presence.
* Should normally be controlled through Boolean properties.
* Do not become false through values such as `"false"`.

You learned `dataset`:

* Maps `data-*` names to camelCase properties.
* Stores strings.
* Is useful for small component metadata.
* Should not store secrets or complete records.

You learned `classList`:

```javascript
add()
remove()
contains()
toggle()
replace()
```

You learned professional conventions:

```html
data-component
data-role
data-action
data-transaction-id
```

You learned that application state should drive:

* Properties.
* Attributes.
* Dataset.
* Classes.
* Content.
* Accessibility state.

The professional rendering model is:

```text
Application state
    ↓
Native properties for behavior
    ↓
ARIA attributes for accessibility
    ↓
Dataset for simple metadata
    ↓
Classes for visual state
    ↓
Text regions for displayed content
```
