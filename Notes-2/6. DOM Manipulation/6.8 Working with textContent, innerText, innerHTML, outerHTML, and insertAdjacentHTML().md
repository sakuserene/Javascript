# 1. Introduction

A DOM element can contain two broad kinds of content:

```text
Text
HTML structure
```

Consider:

```html
<article id="message">
  <strong>Success:</strong>
  Transaction saved.
</article>
```

The element contains:

* A `<strong>` element.
* A text node containing `Transaction saved.`

JavaScript may need to:

* Read all textual content.
* Read only text visible to the user.
* Replace content with plain text.
* Read the current HTML markup.
* Replace descendants with parsed HTML.
* Replace the element itself with new markup.
* Insert trusted markup without replacing existing descendants.

These tasks look similar, but they have very different behavior.

The main APIs are:

```javascript
textContent
innerText
innerHTML
outerHTML
insertAdjacentHTML()
```

Choosing the wrong one may cause:

* Lost child elements.
* Lost event listeners.
* Stale references.
* Invalid component state.
* Layout performance problems.
* Cross-site scripting vulnerabilities.
* Broken accessibility behavior.

---

# 2. The Core Problem

Suppose the application receives this string:

```javascript
const message =
  "<strong>Payment complete</strong>";
```

Should the user see:

```text
<strong>Payment complete</strong>
```

or:

**Payment complete**

Those are different requirements.

To display the string literally:

```javascript
element.textContent =
  message;
```

To parse it as HTML:

```javascript
element.innerHTML =
  message;
```

The second operation treats the string as markup.

That distinction is central to safe DOM programming.

---

# 3. Text vs Markup

## Text

Text is displayed as characters.

```javascript
element.textContent =
  "<em>Hello</em>";
```

Output:

```text
<em>Hello</em>
```

The browser creates a text node.

## Markup

Markup is parsed into DOM structure.

```javascript
element.innerHTML =
  "<em>Hello</em>";
```

Output:

```html
<em>Hello</em>
```

The browser creates an `<em>` element containing a text node.

## Core distinction

```text
textContent:
Treat value as text.

innerHTML:
Treat value as HTML source code.
```

---

# 4. The Five Main APIs

| API                    | Reads                    | Writes                                       | Parses HTML? | Main Use                                |
| ---------------------- | ------------------------ | -------------------------------------------- | -----------: | --------------------------------------- |
| `textContent`          | All descendant text      | Plain text                                   |           No | Safe text handling                      |
| `innerText`            | Rendered visible text    | Rendered-style text                          |           No | Human-visible text                      |
| `innerHTML`            | Descendant HTML          | Replaces descendants with parsed HTML        |          Yes | Trusted markup                          |
| `outerHTML`            | Element plus descendants | Replaces the element itself with parsed HTML |          Yes | Whole-root serialization or replacement |
| `insertAdjacentHTML()` | Not a reading API        | Inserts parsed HTML at a selected position   |          Yes | Trusted incremental markup insertion    |

---

# 5. Mental Model

Imagine an element as a container.

```text
Element
├── Child element
│   └── Text
└── Text
```

Different APIs look at the container differently.

## `textContent`

```text
Give me all text-node content.
```

## `innerText`

```text
Give me the text as a user would approximately see it after layout and CSS.
```

## `innerHTML`

```text
Give me the serialized markup inside the container.
```

## `outerHTML`

```text
Give me the serialized markup including the container itself.
```

## `insertAdjacentHTML()`

```text
Parse and insert markup at one of four locations.
```

---

# 6. Text Nodes

HTML:

```html
<p>Hello <strong>World</strong></p>
```

Conceptual DOM:

```text
p
├── Text node: "Hello "
└── strong
    └── Text node: "World"
```

The paragraph does not store one simple JavaScript string internally.

It contains nodes.

When reading:

```javascript
paragraph.textContent;
```

the browser combines descendant text:

```text
Hello World
```

When assigning:

```javascript
paragraph.textContent =
  "New message";
```

the browser removes existing children and creates one new text node.

---

# 7. `textContent`

`textContent` reads or replaces the textual content of a node and its descendants.

Syntax:

```javascript
const text =
  element.textContent;
```

Write:

```javascript
element.textContent =
  "New text";
```

It does not parse HTML.

For most dynamic user or server-provided text, this is the preferred API.

---

# 8. Reading with `textContent`

HTML:

```html
<div id="message">
  Hello
  <strong>World</strong>
  <span hidden>Secret</span>
</div>
```

JavaScript:

```javascript
const message =
  document.querySelector(
    "#message"
  );

console.log(
  message.textContent
);
```

Conceptual result:

```text
Hello World Secret
```

Whitespace may reflect formatting in the HTML source.

`textContent` includes text from hidden descendants.

It works from the DOM tree, not primarily from rendered layout.

---

# 9. Writing with `textContent`

```javascript
message.textContent =
  "Transaction saved.";
```

Before:

```html
<div id="message">
  <strong>Saving</strong>
  <span>Please wait</span>
</div>
```

After:

```html
<div id="message">
  Transaction saved.
</div>
```

The `<strong>` and `<span>` nodes are removed.

A text node replaces them.

---

# 10. What Writing `textContent` Does to Existing Children

Consider:

```javascript
const button =
  document.querySelector(
    "#save-button"
  );

const icon =
  button.querySelector(
    "svg"
  );
```

Then:

```javascript
button.textContent =
  "Saved";
```

The SVG icon is removed.

The `icon` variable now refers to a detached node.

```javascript
console.log(
  icon.isConnected
);
```

Output:

```text
false
```

Do not assign `textContent` to a component root if you intend to preserve its child structure.

Instead, target a dedicated text region:

```html
<button>
  <svg></svg>
  <span data-role="label">
    Save
  </span>
</button>
```

Update:

```javascript
label.textContent =
  "Saved";
```

---

# 11. `textContent` and Hidden Content

HTML:

```html
<section id="report">
  <h2>Revenue Report</h2>
  <p hidden>Internal draft</p>
</section>
```

```javascript
report.textContent;
```

includes:

```text
Internal draft
```

`textContent` is useful when:

* Processing complete textual DOM content.
* Building search indexes.
* Reading hidden template content.
* Extracting raw document text.
* Avoiding layout-dependent behavior.

It is not necessarily a representation of what the user sees.

---

# 12. `textContent` and Script or Style Elements

`textContent` can expose text stored inside elements such as:

```html
<script>
  console.log("Hello");
</script>
```

or:

```html
<style>
  .card { display: none; }
</style>
```

For ordinary page-content containers, this rarely matters.

For generic document-processing tools, understand that `textContent` works through descendant text nodes regardless of visual rendering.

---

# 13. `textContent` Return Values

Reading returns a string or, for some node types, potentially `null`.

For normal elements:

```javascript
typeof element.textContent;
```

is usually:

```text
string
```

Writing is an assignment expression:

```javascript
const result =
  (
    element.textContent =
      "Saved"
  );
```

`result` is the assigned string:

```text
Saved
```

This differs from method calls such as `append()`, which return `undefined`.

---

# 14. `innerText`

`innerText` reads or writes text while considering rendered layout and CSS visibility.

Syntax:

```javascript
const visibleText =
  element.innerText;
```

Write:

```javascript
element.innerText =
  "Updated text";
```

It is designed to approximate the text a user could select or copy from the rendered page.

---

# 15. Reading Visible Text with `innerText`

HTML:

```html
<div id="message">
  Hello
  <span hidden>Secret</span>
  <span>World</span>
</div>
```

```javascript
message.innerText;
```

usually excludes the hidden span.

Conceptual result:

```text
Hello World
```

Exact formatting can depend on CSS and rendering.

---

# 16. Writing with `innerText`

```javascript
message.innerText =
  "Line one\nLine two";
```

Browsers represent the line break as rendered line separation, commonly through generated `<br>` behavior or equivalent DOM representation.

Result may serialize similarly to:

```html
Line one<br>Line two
```

The exact DOM result should be inspected rather than assumed in advanced code.

For simple plain-text replacement, `textContent` is usually more predictable.

---

# 17. `innerText` and CSS Layout

Reading `innerText` may require the browser to account for:

* `display`.
* Visibility.
* Line breaks.
* Block-level structure.
* CSS-generated visual layout.
* Whether content is rendered.

This may require layout calculation.

Therefore, repeatedly reading `innerText` in large loops can be more expensive than reading `textContent`.

Example:

```javascript
for (
  const item
  of items
) {
  console.log(
    item.innerText
  );
}
```

Use it only when rendered visible text is genuinely required.

---

# 18. `innerText` and Line Breaks

HTML:

```html
<div>
  <p>First paragraph</p>
  <p>Second paragraph</p>
</div>
```

`innerText` may return:

```text
First paragraph

Second paragraph
```

because layout semantics influence text formatting.

`textContent` may return text with source whitespace but without the same rendered block separation.

This makes `innerText` useful for:

* Copy-like text extraction.
* Testing visible labels.
* Reading rendered content for user-facing tools.

It is not ideal for storing application data.

---

# 19. `textContent` vs `innerText`

| Feature                                     | `textContent` | `innerText` |
| ------------------------------------------- | ------------- | ----------- |
| Includes hidden text                        | Yes           | Usually no  |
| Uses rendered layout                        | No            | Yes         |
| May trigger layout work                     | Usually no    | Can         |
| Preserves source-style text-node content    | More directly | No          |
| Reflects visible line breaks                | Not primarily | Yes         |
| Best for setting plain text                 | Yes           | Sometimes   |
| Best for reading user-visible rendered text | Not always    | Yes         |

## Prefer `textContent` when:

* Setting dynamic plain text.
* Handling user data safely.
* Reading full textual DOM content.
* Performance and predictability matter.
* Hidden content should be included.

## Use `innerText` when:

* You specifically need rendered visible text.
* CSS visibility should affect the result.
* Line breaks should reflect layout.
* Implementing copy-like or visual-text behavior.

---

# 20. `innerHTML`

`innerHTML` reads or replaces the serialized HTML inside an element.

Read:

```javascript
const markup =
  element.innerHTML;
```

Write:

```javascript
element.innerHTML =
  "<strong>Saved</strong>";
```

The browser parses the supplied string as HTML.

---

# 21. Reading Markup with `innerHTML`

HTML:

```html
<div id="message">
  Hello
  <strong>World</strong>
</div>
```

```javascript
console.log(
  message.innerHTML
);
```

Result resembles:

```html
Hello
<strong>World</strong>
```

It does not include the outer `<div>`.

The browser serializes the current descendant DOM.

The result may not exactly match the original source formatting.

---

# 22. Writing Markup with `innerHTML`

```javascript
message.innerHTML =
  `
    <strong>Success:</strong>
    Transaction saved.
  `;
```

The browser:

1. Parses the string.
2. Creates new DOM nodes.
3. Removes current descendants.
4. Inserts the newly parsed nodes.

Result:

```html
<div id="message">
  <strong>Success:</strong>
  Transaction saved.
</div>
```

---

# 23. How HTML Parsing Works

Given:

```javascript
container.innerHTML =
  `
    <article class="card">
      <h2>Printing</h2>
    </article>
  `;
```

Conceptual process:

```text
Receive string
    ↓
Parse according to HTML rules and element context
    ↓
Create article element
    ↓
Create h2 element
    ↓
Create text node
    ↓
Replace current descendants
```

The parsing context matters.

For example, table-related markup may be interpreted according to table parsing rules.

Browsers may correct invalid markup automatically.

Always inspect the resulting DOM rather than assuming the source string was preserved exactly.

---

# 24. What `innerHTML` Replaces

Before:

```html
<div id="container">
  <button id="save">
    Save
  </button>
</div>
```

Store reference:

```javascript
const oldButton =
  document.querySelector(
    "#save"
  );
```

Replace:

```javascript
container.innerHTML =
  `
    <button id="save">
      Save Again
    </button>
  `;
```

The visible button may look similar.

But it is a new node.

```javascript
const newButton =
  document.querySelector(
    "#save"
  );

console.log(
  oldButton === newButton
);
```

Output:

```text
false
```

---

# 25. Stale References After `innerHTML`

After descendant replacement:

```javascript
console.log(
  oldButton.isConnected
);
```

Output:

```text
false
```

If code later runs:

```javascript
oldButton.disabled =
  true;
```

the visible button is not affected.

The old reference points to a detached element.

This is one of the most common `innerHTML` bugs.

---

# 26. Direct Event Listeners and `innerHTML`

Suppose:

```javascript
oldButton.addEventListener(
  "click",
  handleSave
);
```

Then:

```javascript
container.innerHTML =
  `
    <button id="save">
      Save Again
    </button>
  `;
```

The new button does not have the old direct listener.

The listener remains associated with the detached old button while it is referenced.

Solutions:

* Reinitialize direct listeners.
* Use event delegation on a stable ancestor.
* Avoid replacing the button when a targeted text update is enough.
* Use component lifecycle methods.

---

# 27. Form State and `innerHTML`

Replacing form descendants can lose:

* Current input values.
* Selection ranges.
* Checked states.
* Browser autofill.
* Focus.
* Validation messages.
* File selections.
* Undo history.

Example:

```javascript
form.innerHTML =
  createFormMarkup();
```

This recreates controls.

For active forms, targeted property updates are usually safer.

---

# 28. `innerHTML +=` Problems

This code appears to append:

```javascript
container.innerHTML +=
  `
    <p>New item</p>
  `;
```

Conceptually, it:

1. Reads all current serialized HTML.
2. Concatenates the new string.
3. Parses the entire combined string.
4. Replaces all descendants.

Consequences:

* Existing nodes may be recreated.
* Direct event listeners may be lost.
* Cached references become stale.
* Form state may reset.
* More content is reparsed than necessary.

Use:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  trustedMarkup
);
```

or create nodes:

```javascript
container.append(
  newElement
);
```

---

# 29. `outerHTML`

`outerHTML` reads or replaces the serialized markup of an element including the element itself.

Read:

```javascript
const markup =
  element.outerHTML;
```

Example result:

```html
<article class="card">
  <h2>Printing</h2>
</article>
```

Unlike `innerHTML`, the root `<article>` is included.

---

# 30. Reading with `outerHTML`

HTML:

```html
<p id="message">
  Saved
</p>
```

```javascript
console.log(
  message.outerHTML
);
```

Output:

```html
<p id="message">Saved</p>
```

Uses:

* Debugging.
* Serialization.
* Snapshot comparison.
* Inspecting generated components.
* Testing markup output.

Remember that serialization may normalize quoting, casing, or formatting.

---

# 31. Writing with `outerHTML`

```javascript
message.outerHTML =
  `
    <div class="success">
      Saved
    </div>
  `;
```

The `<p>` element itself is replaced.

Its parent receives the newly parsed node.

The original `message` variable remains a reference to the detached old `<p>`.

---

# 32. Stale Root References After `outerHTML`

```javascript
const oldMessage =
  message;

message.outerHTML =
  `
    <div class="success">
      Saved
    </div>
  `;
```

Now:

```javascript
oldMessage.isConnected;
```

returns:

```text
false
```

The variable does not automatically update to point to the new `<div>`.

This makes `outerHTML` assignment error-prone in component architecture.

Prefer:

```javascript
oldMessage.replaceWith(
  newElement
);
```

when you need a direct reference to the new root.

---

# 33. `insertAdjacentHTML()`

`insertAdjacentHTML()` parses and inserts an HTML string at one of four positions without replacing all existing descendants.

Syntax:

```javascript
element.insertAdjacentHTML(
  position,
  htmlString
);
```

Example:

```javascript
list.insertAdjacentHTML(
  "beforeend",
  `
    <li>Printing</li>
  `
);
```

---

# 34. The Four Insertion Positions

```text
beforebegin
afterbegin
beforeend
afterend
```

Diagram:

```text
beforebegin

<element>
  afterbegin

  Existing content

  beforeend
</element>

afterend
```

Examples:

```javascript
card.insertAdjacentHTML(
  "beforebegin",
  trustedNoticeMarkup
);
```

```javascript
card.insertAdjacentHTML(
  "afterbegin",
  trustedBadgeMarkup
);
```

```javascript
card.insertAdjacentHTML(
  "beforeend",
  trustedActionsMarkup
);
```

```javascript
card.insertAdjacentHTML(
  "afterend",
  trustedDetailsMarkup
);
```

---

# 35. `insertAdjacentHTML()` vs `innerHTML`

## `innerHTML =`

Replaces all descendants.

```javascript
container.innerHTML =
  markup;
```

## `insertAdjacentHTML()`

Adds parsed content at one location.

```javascript
container.insertAdjacentHTML(
  "beforeend",
  markup
);
```

Benefits of `insertAdjacentHTML()`:

* Existing descendants remain.
* Existing direct listeners on those descendants remain.
* Existing references remain current.
* Only new markup is parsed and inserted.
* Placement can be inside or outside the reference.

Both APIs remain security-sensitive because they parse HTML.

---

# 36. `insertAdjacentHTML()` Return Value

The method returns:

```text
undefined
```

It does not return the inserted node.

To capture the inserted element, you may:

* Create it using DOM APIs instead.
* Insert a unique ID and query afterward.
* Record a neighboring element before insertion.
* Use `insertAdjacentElement()` with an already created node.

For component code, creating an element or cloning a template often provides better references.

---

# 37. Safe Text vs Parsed HTML

Given:

```javascript
const value =
  "<img src=x onerror=alert(1)>";
```

## Safe text

```javascript
element.textContent =
  value;
```

The user sees the characters.

## Parsed HTML

```javascript
element.innerHTML =
  value;
```

The browser creates an image element with an event-handler attribute.

This is dangerous.

Professional default:

> Treat dynamic data as text unless rich markup is explicitly required and safely controlled.

---

# 38. Cross-Site Scripting

Cross-site scripting, commonly called XSS, occurs when untrusted data is interpreted as executable browser content.

Dangerous pattern:

```javascript
results.innerHTML =
  searchTerm;
```

An attacker may provide markup containing:

* Event-handler attributes.
* Dangerous URLs.
* Embedded browser-active content.
* Markup that changes the application's interface.
* Forms intended to steal data.

The exact exploit depends on browser behavior, security headers, and context.

The fundamental problem is:

```text
Untrusted data
    ↓
Parsed as code or markup
    ↓
Attacker-controlled browser behavior
```

# 39. Common Injection Examples

Unsafe interpolation:

```javascript
container.innerHTML =
  `
    <p>${userComment}</p>
  `;
```

Unsafe adjacent insertion:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  userComment
);
```

Unsafe attribute interpolation:

```javascript
container.innerHTML =
  `
    <a href="${userUrl}">
      Open
    </a>
  `;
```

Unsafe image source interpolation:

```javascript
container.innerHTML =
  `
    <img src="${imageUrl}">
  `;
```

Safer node construction:

```javascript
const paragraph =
  document.createElement(
    "p"
  );

paragraph.textContent =
  userComment;

container.append(
  paragraph
);
```

# 40. Why Escaping Is Context-Sensitive

A value inserted into text content differs from a value inserted into:

* HTML attributes.
* URLs.
* JavaScript strings.
* CSS.
* HTML markup.
* SVG.

There is no single naive replacement function that safely handles every context.

Example:

```javascript
value.replace(
  "<",
  "&lt;"
);
```

is not a complete general security solution.

Prefer avoiding HTML parsing for dynamic data.

Use trusted, established sanitization or encoding strategies when rich content is required.

---

# 41. Sanitization

Sanitization removes or neutralizes dangerous markup while preserving allowed content.

Use cases:

* User-authored rich text.
* Controlled CMS content.
* Markdown converted to HTML.
* Trusted editor output that still requires policy enforcement.

A sanitizer should define allowed:

* Tags.
* Attributes.
* URL protocols.
* Embedded content.
* CSS-related behavior.

Do not attempt to build a production sanitizer with a few regular expressions.

HTML parsing and browser security are too complex for simplistic filtering.

---

# 42. Trusted Markup

Markup may be treated as trusted when:

* It is a fixed application constant.
* It comes from a reviewed local template.
* It is produced entirely by trusted application code.
* It has passed a robust sanitization policy.

Example:

```javascript
const spinnerMarkup =
  `
    <span
      class="spinner"
      aria-hidden="true"
    ></span>
  `;
```

Because the string is fixed application code, inserting it may be acceptable.

Even then, template cloning or `createElement()` may provide better maintainability and references.

---

# 43. Trusted Types Concept

Trusted Types is a browser security mechanism designed to reduce DOM-based XSS by restricting assignments to dangerous HTML injection sinks.

Conceptually:

```text
Raw string
    ✕
innerHTML

Approved policy output
    ✓
innerHTML
```

It encourages applications to centralize and audit markup creation.

Not every project uses Trusted Types, but the architectural principle is valuable:

> Do not allow arbitrary code paths to send raw strings into HTML-parsing APIs.

Centralize trusted markup creation.

# 44. Choosing the Correct API

Use this decision tree.

## Need plain text?

```javascript
element.textContent =
  value;
```

## Need text as rendered visibly?

Read:

```javascript
element.innerText;
```

## Need to replace descendants with trusted markup?

```javascript
element.innerHTML =
  trustedMarkup;
```

## Need to inspect descendant markup?

```javascript
element.innerHTML;
```

## Need to inspect the root plus descendants?

```javascript
element.outerHTML;
```

## Need to insert trusted markup without replacing current descendants?

```javascript
element.insertAdjacentHTML(
  position,
  trustedMarkup
);
```

## Need structured dynamic content from untrusted data?

Create elements or clone a template, then use `textContent`.

# 45. Reading Text from Form Controls

Do not use `textContent` to read input values.

HTML:

```html
<input
  id="username"
  value="Amina"
>
```

Incorrect:

```javascript
input.textContent;
```

Inputs do not store user-entered value as child text.

Use:

```javascript
input.value;
```

For textareas:

```javascript
textarea.value;
```

For select elements:

```javascript
select.value;
```

DOM text APIs and form-control value properties solve different problems.

# 46. Reading Business Data from Visible Text

Weak:

const amount =

* Formatting may change.

* Localized separators may appear.

const transaction =

# 51. Replacing Component Internals

Suppose a region genuinely needs new structured content.

Prefer node-based replacement:

```javascript
detailsRegion.replaceChildren(
  createTransactionDetails(
    transaction
  )
);
```

Instead of:

```javascript
detailsRegion.innerHTML =
  buildTransactionDetailsMarkup(
    transaction
  );
```

Node-based rendering provides:

* Safe text assignment.
* Direct references.
* Easier lifecycle management.
* Better structural validation.

HTML strings may still be appropriate for trusted static markup.

# 52. Content APIs and Event Delegation

Replacing descendants with `innerHTML` removes old descendant nodes.

A delegated listener on a stable ancestor continues to work for new matching descendants.

```javascript
container.addEventListener(
  "click",
  handleClick
);
```

After:

```javascript
container.innerHTML =
  trustedMarkup;
```

the container listener remains because the container was not replaced.

However:

* Direct child listeners are lost.
* Cached child references become stale.
* Component registries may need rebuilding.
* Focus may be lost.

Event delegation solves only the event-listener part, not the complete lifecycle.

# 53. Accessibility Considerations

## Status updates

Use:

```javascript
statusRegion.textContent =
  "Saved";
```

inside a stable live region.

## Hidden content

Remember that `textContent` includes hidden text.

Do not use it to claim exactly what screen-reader or visual users perceive.

## `innerText`

Rendered visible text does not equal the accessibility tree in every situation.

Accessibility APIs, roles, names, labels, and hidden states are more complex than visual text alone.

## Markup replacement

Replacing headings, labels, or form controls may break:

* Heading order.
* Label relationships.
* ARIA references.
* Focus.
* Live-region behavior.
* Keyboard state.

Use targeted updates whenever possible.

---

# 54. Performance Considerations

## Reading

`textContent` is generally cheaper than `innerText` because it does not need rendered-layout interpretation.

## Writing

Both `textContent` and `innerHTML` replace descendants.

The cost depends on:

* Number of removed nodes.
* Size of new content.
* HTML parsing.
* Layout and paint afterward.
* Event and view reinitialization.

## `innerHTML +=`

May reparse all existing descendant markup and should be avoided for incremental updates.

## Large content

For large structured interfaces:

* Use templates.
* Batch node creation.
* Use fragments.
* Update only changed regions.
* Consider pagination or virtualization.
* Profile realistic interactions.

---

# 55. Memory Considerations

Assigning:

```javascript
container.textContent =
  "";
```

or:

```javascript
container.innerHTML =
  "";
```

detaches old descendants.

They remain in memory if referenced by:

* Arrays.
* Maps.
* View objects.
* Closures.
* Listeners.
* Timers.
* Observers.

Before replacing complex component content:

* Destroy component resources.
* Clear registries.
* Abort pending work.
* Release stale references.

DOM replacement and JavaScript lifecycle cleanup are separate responsibilities.

## Example 3: Real-World Application — Safe Transaction Rendering

### Unsafe version

```javascript
function renderTransactionUnsafe(
  transaction,
  container
) {
  container.insertAdjacentHTML(
    "beforeend",
    `
      <article
        data-transaction-id="${transaction.id}"
      >
        <h3>
          ${transaction.receipt}
        </h3>

        <p>
          ${transaction.service}
        </p>
      </article>
    `
  );
}
```

Problems:

* Values are parsed as markup.
* Attribute interpolation is unsafe.
* No direct root reference is returned.
* Business identity is mixed into a string.
* Validation is difficult.

### Safer template version

```html
<template id="transaction-template">
  <article
    data-component="transaction-card"
  >
    <h3 data-role="receipt"></h3>
    <p data-role="service"></p>
    <p data-role="amount"></p>

    <button
      type="button"
      data-action="delete"
    >
      Delete
    </button>
  </article>
</template>
```

```javascript
function createTransactionCard(
  transaction
) {
  const fragment =
    transactionTemplate
      .content
      .cloneNode(true);

  const root =
    fragment.firstElementChild;

  const receipt =
    root.querySelector(
      "[data-role='receipt']"
    );

  const service =
    root.querySelector(
      "[data-role='service']"
    );

  const amount =
    root.querySelector(
      "[data-role='amount']"
    );

  const deleteButton =
    root.querySelector(
      "[data-action='delete']"
    );

  root.dataset.transactionId =
    String(transaction.id);

  receipt.textContent =
    transaction.receipt;

  service.textContent =
    transaction.service;

  amount.textContent =
    `KES ${transaction.amount}`;

  deleteButton.setAttribute(
    "aria-label",
    `Delete ${transaction.receipt}`
  );

  return root;
}
```

### Why this is professional

* Structure comes from a controlled template.
* Dynamic values are assigned as text.
* Identity is assigned through a property API.
* A root reference is returned.
* Event delegation can handle actions.
* No raw dynamic HTML string is parsed.

Use application state.

# 60. Interview Questions and Answers

## Beginner Questions

### 1. What does `textContent` do?

It reads or replaces textual content without parsing HTML.

### 2. What does `innerText` do?

It reads or writes text while considering rendered layout and visibility.

### 3. What does `innerHTML` do?

It reads descendant markup or replaces descendants with parsed HTML.

### 4. What does `outerHTML` include?

The element itself and all descendant markup.

### 5. Does `textContent` parse HTML?

No.

### 6. Does `innerHTML` parse HTML?

Yes when assigned.

### 7. Which API is safest for user text?

`textContent`.

### 8. Does `innerHTML` include the element itself when read?

No.

---

## Intermediate Questions

### 1. Why can `innerText` be slower than `textContent`?

It may need rendered-layout and CSS visibility information.

### 2. What happens to children when `textContent` is assigned?

They are removed and replaced by text content.

### 3. What happens to descendants when `innerHTML` is assigned?

They are removed and replaced by newly parsed nodes.

### 4. Why do cached child references become stale after `innerHTML`?

They point to detached old nodes rather than newly created descendants.

### 5. Why is `innerHTML +=` problematic?

It can reserialize and reparse all existing content, recreating descendants and losing runtime state.

### 6. What is the difference between `innerHTML` and `insertAdjacentHTML()`?

`innerHTML` replaces descendants. `insertAdjacentHTML()` inserts parsed markup at a selected position without replacing existing descendants.

### 7. Why does `outerHTML` assignment make the current variable stale?

It replaces the element itself, while the variable still references the removed old node.

---

## Advanced Questions

### 1. How would you safely render user-authored rich text?

Use a well-maintained sanitizer with a strict allowlist, safe URL policy, security headers where appropriate, and centralized trusted-markup handling.

### 2. Why is escaping context-sensitive?

Text, attributes, URLs, JavaScript, CSS, and HTML require different safety rules.

### 3. How would you audit a codebase for DOM XSS risks?

Search for HTML-parsing sinks such as `innerHTML`, `outerHTML`, `insertAdjacentHTML()`, document-writing APIs, and unsafe URL assignments; trace their data sources and sanitization policies.

### 4. How do markup replacement and event delegation interact?

A listener on a stable ancestor survives descendant replacement and can handle new descendants, but cached views, direct listeners, focus, and other lifecycle state still require management.

### 5. When should `innerHTML` be chosen over template cloning?

When the markup is trusted, relatively simple, naturally represented as a string, and direct node references or complex lifecycle behavior are unnecessary.

### 6. Why should business logic not parse amounts from `textContent`?

Presentation formatting is unstable and may include labels, localization, or hidden content. Application state should remain authoritative.

# 64. Summary

DOM content can be treated as plain text or parsed markup.

You learned five major APIs:

```javascript
textContent
innerText
innerHTML
outerHTML
insertAdjacentHTML()
```

## `textContent`

* Reads all descendant text.
* Includes hidden descendant text.
* Does not depend primarily on layout.
* Replaces children with plain text when assigned.
* Is the preferred default for dynamic text.

## `innerText`

* Reflects rendered visible text.
* Considers CSS and layout.
* May include rendered line breaks.
* Can require layout-related work.

## `innerHTML`

* Reads descendant markup.
* Parses assigned strings as HTML.
* Replaces current descendants.
* Invalidates cached descendant references.
* Can create XSS vulnerabilities with untrusted input.

## `outerHTML`

* Includes the element itself.
* Replaces the complete element when assigned.
* Leaves the old variable pointing to a detached node.

## `insertAdjacentHTML()`

* Parses trusted markup.
* Inserts at one of four relative positions.
* Does not replace existing descendants.
* Returns `undefined`.

You also learned:

* Text nodes form the basis of DOM textual content.
* Updating a root's `textContent` removes child structure.
* `innerHTML +=` can recreate all descendants.
* Direct listeners do not transfer to newly parsed nodes.
* Delegation survives descendant replacement when the ancestor remains.
* Form values and focus can be lost during replacement.
* Untrusted data should normally be assigned through `textContent`.
* Rich HTML requires robust sanitization and centralized control.
* Escaping rules depend on context.
* Templates and DOM construction are safer defaults for structured dynamic content.
* Business data should remain in application state rather than visible text.

The professional content workflow is:

```text
Identify content source
    ↓
Is it plain data?
    ├── Yes → textContent
    └── No
         ↓
Is rich markup genuinely required?
    ├── No → structured DOM creation
    └── Yes
         ↓
Is markup trusted or robustly sanitized?
    ├── No → reject or render as text
    └── Yes → controlled HTML insertion
```
