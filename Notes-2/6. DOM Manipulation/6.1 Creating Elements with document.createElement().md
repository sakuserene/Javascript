# 1. Introduction

Until now, most of your DOM work has focused on finding and understanding elements that already exist.

You learned how to:

* Select elements.
* Traverse parents.
* Traverse children.
* Find descendants.
* Navigate siblings.
* Discover component roots.
* Read element properties.
* Inspect nodes.

Now you will begin creating the interface itself.

Suppose the page contains:

```html
<ul id="services"></ul>
```

JavaScript receives this data:

```javascript
const services = [
  {
    id: 1,
    name: "Printing",
    price: 20,
  },
  {
    id: 2,
    name: "Binding",
    price: 100,
  },
];
```

The HTML list begins empty.

JavaScript must create:

```html
<li>
  <h3>Printing</h3>
  <p>KES 20</p>
</li>
```

and:

```html
<li>
  <h3>Binding</h3>
  <p>KES 100</p>
</li>
```

This requires two different operations:

```text
Create DOM elements
    ↓
Insert them into the document
```

This lesson focuses on creation.

Insertion will be taught in the next lesson.

---

# 2. What Is DOM Manipulation?

> DOM manipulation is the process of creating, modifying, moving, replacing, or removing nodes in the Document Object Model.

DOM manipulation includes:

* Creating elements.
* Creating text nodes.
* Changing text.
* Changing attributes.
* Changing classes.
* Changing styles.
* Adding elements.
* Moving elements.
* Replacing elements.
* Removing elements.
* Building complete interfaces from JavaScript data.

Traversal answers:

```text
Where is the node?
```

Manipulation answers:

```text
What should happen to the node?
```

---

# 3. Why DOM Manipulation Exists

HTML defines the initial page structure.

But modern applications change after loading.

Examples:

* A user adds a task.
* New transactions arrive.
* Search results appear.
* A modal opens.
* Validation errors display.
* Notifications are created.
* API data becomes available.
* A shopping cart updates.
* A table receives new rows.
* A dashboard refreshes.

The server cannot know every future interaction when the initial HTML is generated.

JavaScript needs the ability to modify the live page.

DOM manipulation provides that ability.

---

# 4. What Problem Dynamic Element Creation Solves

Suppose a user submits this form:

```html
<form id="task-form">
  <input name="title">
  <button type="submit">
    Add Task
  </button>
</form>
```

The page also contains:

```html
<ul id="task-list"></ul>
```

After submission, the application needs to create:

```html
<li>
  <span>Prepare report</span>
  <button>Delete</button>
</li>
```

The new task did not exist in the original HTML.

The page should not reload just to display it.

JavaScript can create the structure immediately.

```text
User input
    ↓
JavaScript state
    ↓
Create DOM elements
    ↓
Insert into page
    ↓
User sees new task
```

---

# 5. How Interfaces Were Built Before Dynamic DOM APIs

Before rich browser applications became common, many interactions required:

1. Submitting a form.
2. Sending data to the server.
3. Server generating a complete new HTML page.
4. Browser replacing the old page.
5. User waiting for navigation.

This model is still useful for many websites.

However, it is inefficient for highly interactive interfaces.

Dynamic DOM manipulation enables smaller page updates.

Instead of replacing the entire page:

```text
Update one list
Update one card
Update one message
Update one table row
```

The browser preserves the rest of the interface.

# 6. Mental Model: Create, Configure, Insert

The basic creation workflow has three stages.

```text
1. Create
2. Configure
3. Insert
```

Example:

```javascript
const button =
  document.createElement(
    "button"
  );

button.textContent =
  "Save";

button.type =
  "button";

container.append(button);
```

Breakdown:

```text
Create:
document.createElement("button")

Configure:
textContent
type

Insert:
container.append(button)
```

During the first two stages, the button may exist only in memory.

It does not become visible until inserted into a connected DOM tree.

---

# 7. What `document.createElement()` Does

`document.createElement()` creates a new element object.

Example:

```javascript
const paragraph =
  document.createElement(
    "p"
  );
```

This creates a new `<p>` element.

Conceptually:

```html
<p></p>
```

But it has not yet been inserted into the page.

The variable stores a reference to the newly created browser-managed element object.

```text
JavaScript variable
    ↓
New DOM element object
    ↓
Not connected yet
```

---

# 8. Syntax

```javascript
document.createElement(
  tagName
);
```

Example:

```javascript
const heading =
  document.createElement(
    "h2"
  );
```

Example:

```javascript
const card =
  document.createElement(
    "article"
  );
```

Example:

```javascript
const input =
  document.createElement(
    "input"
  );
```

The method returns the created element.

---

# 9. Understanding Every Part of the Syntax

Consider:

```javascript
const button =
  document.createElement(
    "button"
  );
```

## `const`

Creates a variable binding that cannot be reassigned.

```javascript
const button = ...;
```

The element itself can still be modified.

```javascript
button.textContent =
  "Save";
```

## `document`

The current page's `Document` object.

It provides element-creation methods.

## `.`

Property-access operator.

It accesses the `createElement` method on `document`.

## `createElement`

The DOM method that creates an element object.

## `(` and `)`

Function-call parentheses.

They contain the argument passed to the method.

## `"button"`

A string containing the HTML tag name.

## `=`

Assignment operator.

It stores the returned element reference in `button`.

## `;`

Statement terminator.

It is optional in many JavaScript cases but recommended for consistency.

# 11. Created Does Not Mean Visible

This code:

```javascript
const paragraph =
  document.createElement(
    "p"
  );

paragraph.textContent =
  "Hello";
```

does not display anything.

The element exists, but it is detached.

Check:

```javascript
console.log(
  paragraph.isConnected
);
```

Output:

```text
false
```

Insert it:

```javascript
document.body.append(
  paragraph
);
```

Now:

```javascript
console.log(
  paragraph.isConnected
);
```

Output:

```text
true
```

Creation and insertion are separate operations.

---

# 12. Detached Elements

A newly created element begins detached.

```javascript
const card =
  document.createElement(
    "article"
  );
```

Detached means:

* It exists as a DOM object.
* JavaScript can modify it.
* It may have descendants.
* It may have attributes.
* It may have event listeners.
* It is not visible in the connected page.
* Its `isConnected` property is `false`.

This is useful.

You can construct a complete component before inserting it.

```text
Create card
    ↓
Create heading
    ↓
Create amount
    ↓
Create buttons
    ↓
Connect children
    ↓
Insert completed card
```

---

# 13. How the Browser Creates an Element Internally

When JavaScript runs:

```javascript
const button =
  document.createElement(
    "button"
  );
```

The browser conceptually performs these steps:

```text
1. Receive tag-name string
2. Normalize it for the HTML document
3. Determine matching DOM interface
4. Allocate a new browser-managed element object
5. Initialize default properties
6. Return a reference to JavaScript
```

The resulting object is not merely a string containing HTML.

It is a real DOM object with methods and properties.

```javascript
console.log(
  button instanceof
    HTMLButtonElement
);
```

Output:

```text
true
```

It also inherits from broader DOM interfaces.

```text
Node
    ↓
Element
    ↓
HTMLElement
    ↓
HTMLButtonElement
```

# 15. Creating Text Content

An empty element is rarely useful.

```javascript
const heading =
  document.createElement(
    "h2"
  );
```

You can give it text using:

```javascript
heading.textContent =
  "Daily Collections";
```

Resulting structure:

```html
<h2>
  Daily Collections
</h2>
```

Internally:

```text
h2 element
└── Text node:
    "Daily Collections"
```

The browser creates or updates text-node content for you.

---

# 16. `textContent`

Use `textContent` for plain text.

```javascript
const message =
  document.createElement(
    "p"
  );

message.textContent =
  "Transaction saved successfully.";
```

Advantages:

* Simple.
* Clear.
* Treats content as text.
* Does not parse HTML.
* Appropriate for untrusted display strings.

Example:

```javascript
message.textContent =
  "<strong>Saved</strong>";
```

Visible output:

```text
<strong>Saved</strong>
```

The string is not converted into a `<strong>` element.

---

# 17. `document.createTextNode()`

You can explicitly create a text node.

```javascript
const text =
  document.createTextNode(
    "Save"
  );
```

Then create a button:

```javascript
const button =
  document.createElement(
    "button"
  );
```

Attach the text:

```javascript
button.append(text);
```

Result:

```html
<button>Save</button>
```

`createTextNode()` is useful when:

* You want explicit node-level control.
* You are combining text and child elements.
* You are writing generic DOM tools.
* Multiple text nodes are needed in one element.

---

# 18. `textContent` vs `createTextNode()`

## `textContent`

```javascript
button.textContent =
  "Save";
```

Best when replacing all text content.

## `createTextNode()`

```javascript
const text =
  document.createTextNode(
    "Save "
  );

button.append(text);
```

Useful for mixed content.

Example:

```html
<button>
  Save
  <span>Ctrl + S</span>
</button>
```

Creation:

```javascript
const button =
  document.createElement(
    "button"
  );

const label =
  document.createTextNode(
    "Save "
  );

const shortcut =
  document.createElement(
    "span"
  );

shortcut.textContent =
  "Ctrl + S";
```

The text node and element can coexist.

# 19. Setting IDs

Use the `id` property.

```javascript
const section =
  document.createElement(
    "section"
  );

section.id =
  "transactions";
```

Result:

```html
<section id="transactions"></section>
```

IDs should be unique within the document.

Avoid generating repeated components with duplicate IDs.

For repeated components, prefer:

```javascript
card.dataset.transactionId =
  String(transaction.id);
```


Add several:

```javascript
card.classList.add(
```

Result:

```html
<article
  class="transaction-card is-pending"
></article>
```

---

# 21. Using `className`

You can assign the complete class string.

```javascript
card.className =
  "transaction-card is-pending";
```

This replaces existing classes.

Use when:

* Setting the initial complete class list.
* The entire class value is known.

Be careful:

```javascript
card.className =
  "selected";
```

removes all previous classes.

---

# 22. Using `classList`

`classList` provides class-specific operations.

```javascript
card.classList.add(
  "transaction-card"
);
```

```javascript
card.classList.remove(
  "is-pending"
);
```

```javascript
card.classList.toggle(
  "is-selected"
);
```

```javascript
card.classList.contains(
  "transaction-card"
);
```

For creation, this is especially useful when classes depend on data.

```javascript
card.classList.add(
  "transaction-card",
  `status-${transaction.status}`
);
```

Validate dynamic class values before using them.

# 25. Properties vs Attributes During Creation

Use properties for current interactive state.

```javascript
button.disabled =
  true;

input.checked =
  false;

input.value =
  "Amina";
```

Use attributes for metadata and ARIA.

```javascript
button.setAttribute(
  "aria-expanded",
  "false"
);

input.setAttribute(
  "autocomplete",
  "email"
);
```

Some values can be set through either interface.

Choose the one that most clearly expresses the intended behavior.

---

# 26. Creating Boolean States

Boolean properties include:

```javascript
button.disabled =
  true;

input.required =
  true;

checkbox.checked =
  true;

section.hidden =
  false;
```

Do not write:

```javascript
button.setAttribute(
  "disabled",
  "false"
);
```

The attribute remains present, which means disabled.

To enable:

```javascript
button.disabled =
  false;
```

or:

```javascript
button.removeAttribute(
  "disabled"
);
```

---

# 27. Creating Data Attributes

Use `dataset`.

```javascript
const card =
  document.createElement(
    "article"
  );

card.dataset.transactionId =
  "42";

card.dataset.status =
  "pending";
```

Result:

```html
<article
  data-transaction-id="42"
  data-status="pending"
></article>
```

Dataset values become strings.

```javascript
card.dataset.transactionId =
  String(transaction.id);
```

Use data attributes for:

* Record IDs.
* Action names.
* Component names.
* UI state.
* Semantic roles.

Do not store large application objects in `dataset`.

---

# 28. Creating ARIA Attributes

ARIA attributes improve accessibility when native HTML alone is insufficient.

Example:

```javascript
const button =
  document.createElement(
    "button"
  );

button.setAttribute(
  "aria-expanded",
  "false"
);

button.setAttribute(
  "aria-controls",
  "transaction-menu"
);
```

Do not use ARIA to replace native semantics unnecessarily.

Prefer:

```javascript
const button =
  document.createElement(
    "button"
  );
```

over:

```javascript
const div =
  document.createElement(
    "div"
  );

div.setAttribute(
  "role",
  "button"
);
```

Native elements include built-in keyboard and accessibility behavior.

# 38. Element Factories

An element factory is a function that creates and configures an element.

Example:

```javascript
function createButton(
  label,
  action
) {
  const button =
    document.createElement(
      "button"
    );

  button.type =
    "button";

  button.textContent =
    label;

  button.dataset.action =
    action;

  return button;
}
```

Usage:

```javascript
const deleteButton =
  createButton(
    "Delete",
    "delete"
  );
```

Factories reduce repeated creation code.

# 39. Factory Parameters

Parameters should represent the data needed to create the element.

```javascript
function createServiceItem(
  service
) {
  const item =
    document.createElement(
      "li"
    );

  item.dataset.serviceId =
    String(service.id);

  item.textContent =
    `${service.name}: KES ${service.price}`;

  return item;
}
```

Avoid factories with too many positional arguments.

Less clear:

```javascript
createCard(
  42,
  "RCT-42",
  500,
  "pending",
  true,
  false
);
```

Clearer:

```javascript
createCard({
  id: 42,
  receipt: "RCT-42",
  amount: 500,
  status: "pending",
  canDelete: true,
  isSelected: false,
});
```

# 40. Factory Return Values

Most factories should return the component root.

```javascript
function createCard(
  transaction
) {
  const card =
    document.createElement(
      "article"
    );

  return card;
}
```

Sometimes a factory may return a view object.

```javascript
function createCardView(
  transaction
) {
  const root =
    document.createElement(
      "article"
    );

  const status =
    document.createElement(
      "p"
    );

  return {
    root,
    status,

    setStatus(value) {
      status.textContent =
        value;
    },
  };
}
```

Choose based on architecture.

Return only the root for simple rendering.

Return a view object when repeated updates need internal references.

# 41. Factories vs Direct Creation

Use direct creation when:

* Only one simple element is needed.
* The code is local and obvious.
* Reuse is unlikely.

```javascript
const message =
  document.createElement(
    "p"
  );
```

Use a factory when:

* The same structure appears repeatedly.
* Validation is needed.
* Many properties must be configured.
* Component consistency matters.
* Creation logic should be tested.

Do not create a factory for every trivial one-line element.

# 45. Validation During Creation

Factories should validate critical inputs.

```javascript
function createTransactionCard(
  transaction
) {
  if (
    !transaction ||
    typeof transaction !==
      "object"
  ) {
    throw new TypeError(
      "A transaction object is required."
    );
  }

  if (
    !Number.isFinite(
      transaction.id
    )
  ) {
    throw new TypeError(
      "Transaction id must be a number."
    );
  }

  if (
    typeof transaction.receipt !==
      "string"
  ) {
    throw new TypeError(
      "Transaction receipt must be a string."
    );
  }

  const card =
    document.createElement(
      "article"
    );

  return card;
}
```

Validation prevents malformed components from entering the DOM.

# 46. Defensive Element Creation

Defensive creation includes:

* Input validation.
* Meaningful defaults.
* Semantic element choice.
* Safe text insertion.
* Stable data attributes.
* Required accessibility attributes.
* Explicit button types.
* Consistent factory return values.

Example:

```javascript
function createActionButton({
  label,
  action,
  disabled = false,
  ariaLabel = label,
}) {
  if (
    typeof label !==
      "string" ||
    label.trim() === ""
  ) {
    throw new TypeError(

    );
  }

  if (
    typeof action !==
      "string" ||
    action.trim() === ""
  ) {
    throw new TypeError(
      "Button action is required."
    );
  }

  const button =
    document.createElement(
      "button"
    );

  button.type =
    "button";

  button.textContent =
    label;

  button.dataset.action =
    action;

  button.disabled =
    disabled;

  button.setAttribute(
    "aria-label",
    ariaLabel
  );

  return button;
}
```

# 47. Security Considerations

## Prefer `textContent`

```javascript
element.textContent =
  userProvidedValue;
```

This treats the value as text.

## Avoid unsafe HTML parsing

Do not do this with untrusted data:

```javascript
element.innerHTML =
  userProvidedValue;
```

This can create markup and potentially dangerous content.

## Validate URLs

Before setting link destinations from untrusted data, validate the protocol and destination.

## Validate dynamic classes and attributes

Avoid directly inserting arbitrary data into selector-sensitive, class-sensitive, or URL-sensitive positions.

## Use native elements

Native buttons, links, and inputs reduce the need for custom event and accessibility behavior.

# 48. Performance Considerations

Creating one element is inexpensive.

Performance concerns appear when creating:

* Thousands of elements.
* Deep nested structures.
* Large tables.
* Repeated complete rerenders.
* Many individual insertions into the connected DOM.

Important principle:

```text
Create detached
    ↓
Configure detached
    ↓
Connect descendants
    ↓
Insert completed structure
```

This limits partial live-DOM updates.

Later you will learn:

* `DocumentFragment`.
* Batch insertion.
* `replaceChildren()`.
* Efficient rendering.
* Avoiding layout thrashing.
* Incremental updates.
* Pagination and virtualization.

---

# 49. Memory Considerations

Created elements remain in memory while reachable.

Example:

```javascript
const card =
  document.createElement(
    "article"
  );
```

If `card` is never inserted and all references disappear, it becomes eligible for garbage collection.

If stored in an array:

```javascript
createdCards.push(card);
```

it remains reachable.

Detached elements are not automatically leaks.

They become a problem when unintentionally retained.

Manage component lifecycle carefully.

# 51. Common Mistakes

## Mistake 1: Expecting a created element to appear automatically

```javascript
document.createElement(
  "p"
);
```

Creation does not insert.

---

## Mistake 2: Creating invalid tag names accidentally

```javascript
document.createElement(
  ".card"
);
```

The argument is a tag name, not a selector.

Correct:

```javascript
const card =
  document.createElement(
    "article"
  );

card.className =
  "card";
```

---

## Mistake 3: Using `innerHTML` for plain text

```javascript
message.innerHTML =
  userMessage;
```

Use:

```javascript
message.textContent =
  userMessage;
```

---

## Mistake 4: Forgetting button type

A created button inside a form may submit unexpectedly.

```javascript
button.type =
  "button";
```

---

## Mistake 5: Creating duplicate IDs

Repeated components should not share IDs.

Use data attributes and scoped roles.

---

## Mistake 6: Using visual classes as business data

Avoid:

```javascript
card.classList.add(
  `transaction-${id}`
);
```

for identity lookup.

Prefer:

```javascript
card.dataset.transactionId =
  String(id);
```

---

## Mistake 7: Storing whole objects in dataset

```javascript
card.dataset.transaction =
  transaction;
```

This becomes:

```text
"[object Object]"
```

Store only simple string metadata.

Keep full objects in application state.

---

## Mistake 8: Creating a `<div>` for every purpose

Choose semantic elements.

* Button for actions.
* Link for navigation.
* List for collections.
* Article for standalone records.
* Form controls for input.

---

## Mistake 9: Returning inconsistent factory values

One factory call should not sometimes return an element and sometimes an object.

Define a stable contract.

---

## Mistake 10: Failing to validate required data

Malformed data creates malformed interface elements.

---

# 52. Professional Best Practices

1. Create elements with semantic tag names.
2. Use `textContent` for plain and untrusted text.
3. Use properties for live standard state.
4. Use `setAttribute()` for ARIA and raw attributes.
5. Give form buttons explicit types.
6. Add required accessibility information during creation.
7. Use `data-component`, `data-role`, and `data-action` consistently.
8. Keep complete business objects outside `dataset`.
9. Validate factory inputs.
10. Return a predictable factory result.
11. Build complex structures while detached.
12. Separate creation from insertion.
13. Separate application state from DOM representation.
14. Avoid duplicate IDs.
15. Create reusable factories only when repetition justifies them.
16. Prefer readable code over compact creation tricks.
17. Document important component contracts.
18. Test created elements before insertion.

# 54. Interview Questions and Answers

## Beginner Questions

### 1. What does `document.createElement()` do?

It creates a new DOM element object with the specified tag name.

### 2. Does a created element appear immediately?

No. It must be inserted into a connected DOM tree.

### 3. What is a detached element?

An element that exists but is not connected to the current document.

### 4. How do you add plain text safely?

Use `textContent` or `createTextNode()`.

### 5. How do you add classes to a created element?

Use `className` or `classList`.

---

## Intermediate Questions

### 1. What is the difference between `textContent` and `createTextNode()`?

`textContent` sets or replaces text content. `createTextNode()` creates an explicit text-node object.

### 2. Why should buttons have an explicit `type`?

Inside forms, the default behavior may be submission.

### 3. Why use `dataset`?

To store simple string metadata such as component identity, record IDs, roles, actions, or UI state.

### 4. Why should full objects not be stored in `dataset`?

Dataset values are strings and are not suitable as authoritative structured state.

### 5. What is an element factory?

A function that creates and configures an element or component.

---

## Advanced Questions

### 1. Why build complex components while detached?

It allows configuration before insertion and avoids exposing incomplete structures in the live page.

### 2. When should a factory return a view object instead of only a root element?

When later updates need stable references to internal component parts.

### 3. How do you separate state from DOM creation?

Pass state data into a factory, create the DOM representation, and keep the original state in a separate store.

### 4. What security benefit does `textContent` provide?

It treats supplied strings as text instead of parsing them as HTML.

### 5. What memory issue can occur with created but unused elements?

They remain in memory if retained by variables, arrays, maps, closures, or registries.

### 6. How would you create thousands of records efficiently?

Create structured elements while detached, batch insertion, reduce DOM size through pagination or virtualization, and avoid unnecessary full rerenders.

# 58. Summary

`document.createElement()` creates a new DOM element object.

Syntax:

```javascript
const element =
  document.createElement(
    "tag-name"
  );
```

Created elements begin detached.

```javascript
element.isConnected;
```

returns:

```text
false
```

You learned to configure created elements using:

```javascript
textContent
id
className
classList
dataset
setAttribute()
standard element properties
Boolean properties
ARIA attributes
```

You learned how to create:

* Paragraphs.
* Headings.
* Buttons.
* Inputs.
* Labels.
* Links.
* Images.
* Lists.
* Table rows.
* Semantic component roots.

You learned that professional creation code:

* Uses semantic elements.
* Validates input.
* Uses safe text insertion.
* Keeps state separate.
* Sets accessibility attributes.
* Uses stable component metadata.
* Creates reusable factories.
* Returns predictable results.
* Builds complex components while detached.
* Inserts only after configuration is complete.

The core mental model is:

```text
Application data
    ↓
Create elements
    ↓
Configure properties
    ↓
Connect child structure
    ↓
Insert into live DOM
```
