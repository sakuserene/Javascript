# 1. Introduction

Modern browser applications frequently need to know:

* How wide an element is.
* How tall an element is.
* Where an element is located.
* Whether an element overflows.
* How far a container has been scrolled.
* Whether a user has reached the bottom of a list.
* Where a tooltip should appear.
* Whether a dropdown would leave the viewport.
* How much content exists outside a scrollable area.
* Whether an element's dimensions have changed.

Consider a transaction list:

```html
<section
  id="transaction-list"
  class="transaction-list"
>
  <!-- Transaction cards -->
</section>
```

JavaScript may need to determine whether the user has reached the bottom:

```javascript
const reachedBottom =
  transactionList.scrollTop +
    transactionList.clientHeight >=
  transactionList.scrollHeight;
```

A tooltip may need the position of a button:

```javascript
const rect =
  button.getBoundingClientRect();
```

A collapsible section may need its full content height:

```javascript
const height =
  details.scrollHeight;
```

These APIs are related, but each measures a different box, coordinate system, or scrolling concept.

---

# 2. What Is Element Geometry?

> Element geometry is the collection of sizes, positions, boundaries, and scrolling measurements that describe how an element exists in the rendered page.

Geometry includes:

```text
Width
Height
Top position
Left position
Right edge
Bottom edge
Visible area
Scrollable area
Current scroll offset
Viewport position
Document position
```

The DOM tree tells us how elements are related.

Geometry APIs tell us where those elements are rendered and how large they are.

---

# 3. Why Geometry APIs Exist

HTML and CSS describe layout declaratively.

However, JavaScript sometimes needs the browser's calculated result.

Examples:

```text
CSS says:
width: 50%

JavaScript needs:
Actual rendered width in pixels
```

```text
CSS says:
position: absolute

JavaScript needs:
Exact viewport coordinates
```

```text
Container has:
overflow: auto

JavaScript needs:
Current scroll position and full content height
```

Geometry APIs expose the browser's layout calculations.

---

# 4. The Browser Rendering Pipeline

A simplified rendering pipeline is:

```text
HTML parsing
    ↓
DOM tree

CSS parsing
    ↓
CSSOM

DOM + CSSOM
    ↓
Render tree

Style calculation
    ↓
Layout

Layout
    ↓
Paint

Paint
    ↓
Compositing
```

## Style calculation

The browser determines which CSS rules apply.

## Layout

The browser calculates:

* Element dimensions.
* Positions.
* Line wrapping.
* Box relationships.
* Scrollable extents.

## Paint

The browser draws visual pixels.

Geometry APIs commonly depend on layout results.

---

# 5. The CSS Box Model

Every ordinary element can be understood through layers:

```text
┌─────────────────────────────┐
│           Margin            │
│  ┌───────────────────────┐  │
│  │        Border         │  │
│  │  ┌─────────────────┐  │  │
│  │  │     Padding     │  │  │
│  │  │  ┌───────────┐  │  │  │
│  │  │  │  Content  │  │  │  │
│  │  │  └───────────┘  │  │  │
│  │  └─────────────────┘  │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
```

Different geometry properties measure different layers.

For example:

* `clientWidth` includes content and padding.
* `offsetWidth` includes content, padding, and borders.
* Margins are generally not included in either.
* `scrollWidth` represents the complete scrollable content width.

---

# 6. The Main Measurement Families

| Family                    | Main Purpose                                             |
| ------------------------- | -------------------------------------------------------- |
| `offset*`                 | Border-box dimensions and offset-parent position         |
| `client*`                 | Inner visible area including padding                     |
| `scroll*`                 | Scrollable content dimensions and current scroll offsets |
| `getBoundingClientRect()` | Viewport-relative border-box geometry                    |
| Scrolling methods         | Programmatically move viewport or containers             |
| Observers                 | React to geometry or visibility changes                  |

---

# 7. Coordinate Systems

Several coordinate systems appear in browser programming.

## Viewport coordinates

Measured relative to the visible browser viewport.

```text
Viewport top-left = 0, 0
```

Used by:

```javascript
getBoundingClientRect()
clientX
clientY
```

## Document coordinates

Measured relative to the complete page document.

Used conceptually when combining viewport coordinates with page scrolling.

```javascript
documentX =
  rect.left +
  window.scrollX;
```

```javascript
documentY =
  rect.top +
  window.scrollY;
```

## Element-local coordinates

Measured relative to an element's own box.

Useful for:

* Canvas drawing.
* Drag interactions.
* Charts.
* Custom sliders.

## Screen coordinates

Measured relative to the physical screen.

Pointer events expose:

```javascript
event.screenX
event.screenY
```

Screen coordinates are less commonly used for normal page layout.

---

# 8. `offsetWidth` and `offsetHeight`

`offsetWidth` returns the element's layout width as an integer.

```javascript
const width =
  element.offsetWidth;
```

`offsetHeight` returns the layout height.

```javascript
const height =
  element.offsetHeight;
```

These values generally represent the border box.

---

# 9. What Offset Dimensions Include

`offsetWidth` generally includes:

```text
Content width
+
Left and right padding
+
Left and right borders
+
Vertical scrollbar width, when present
```

`offsetHeight` generally includes:

```text
Content height
+
Top and bottom padding
+
Top and bottom borders
+
Horizontal scrollbar height, when present
```

Margins are not included.

Example:

```css
.box {
  width: 200px;
  padding: 20px;
  border: 5px solid;
}
```

With default `box-sizing: content-box`:

```text
Content width: 200
Padding: 40
Borders: 10
offsetWidth: 250
```

---

# 10. Offset Dimensions and Hidden Elements

If an element or an ancestor uses:

```css
display: none;
```

then:

```javascript
element.offsetWidth;
element.offsetHeight;
```

normally return:

```text
0
```

This happens because the element has no active layout box.

By contrast, an element with:

```css
visibility: hidden;
```

still participates in layout and can have dimensions.

---

# 11. `offsetParent`

`offsetParent` identifies the ancestor used as the reference for `offsetTop` and `offsetLeft`.

```javascript
const parent =
  element.offsetParent;
```

It is commonly:

* The nearest positioned ancestor.
* A table-related ancestor.
* The body in some layouts.
* `null` for certain hidden or specially positioned elements.

Do not assume it is always `parentElement`.

---

# 12. `offsetTop` and `offsetLeft`

```javascript
const top =
  element.offsetTop;

const left =
  element.offsetLeft;
```

These values represent the element's position relative to its `offsetParent`.

They are not necessarily viewport coordinates.

Example:

```text
positioned container
└── element
```

`element.offsetTop` is relative to the offset container's padding edge, not necessarily the document.

---

# 13. Limitations of Offset Position

`offsetTop` and `offsetLeft` can be confusing because:

* The reference is `offsetParent`.
* Nested offset parents require accumulation.
* CSS transforms are not represented in the same way as visual geometry.
* Inline elements can have complex box behavior.
* Scrolling affects viewport appearance but not raw offset values in the same way.
* Writing modes and layout systems may complicate interpretation.

For viewport positioning, prefer:

```javascript
getBoundingClientRect()
```

---

# 14. `clientWidth` and `clientHeight`

`clientWidth` measures the inner visible width.

```javascript
const width =
  element.clientWidth;
```

It generally includes:

```text
Content
+
Padding
```

It excludes:

```text
Borders
Margins
Vertical scrollbar width
```

`clientHeight` follows the same pattern vertically.

---

# 15. `clientTop` and `clientLeft`

These properties commonly represent border thicknesses:

```javascript
element.clientTop;
element.clientLeft;
```

For a typical left-to-right layout:

```text
clientTop ≈ top border width
clientLeft ≈ left border width
```

They may also reflect scrollbar placement in particular writing modes or direction settings.

---

# 16. Offset vs Client Dimensions

Suppose:

```css
.box {
  width: 200px;
  padding: 20px;
  border: 5px solid;
}
```

Conceptually:

```text
clientWidth:
200 + 40 = 240

offsetWidth:
200 + 40 + 10 = 250
```

| Property      | Content | Padding | Border |                   Scrollbar |
| ------------- | ------: | ------: | -----: | --------------------------: |
| `clientWidth` |     Yes |     Yes |     No | Excludes vertical scrollbar |
| `offsetWidth` |     Yes |     Yes |    Yes | Commonly includes scrollbar |

---

# 17. `scrollWidth` and `scrollHeight`

`scrollWidth` returns the total width required to contain the element's content without horizontal clipping.

```javascript
const fullWidth =
  element.scrollWidth;
```

`scrollHeight` returns the complete content height.

```javascript
const fullHeight =
  element.scrollHeight;
```

These can exceed `clientWidth` and `clientHeight`.

Example:

```text
Visible height:
300px

Full content height:
900px
```

Then:

```javascript
element.clientHeight;
```

returns approximately:

```text
300
```

and:

```javascript
element.scrollHeight;
```

returns approximately:

```text
900
```

---

# 18. `scrollTop` and `scrollLeft`

`scrollTop` is the number of pixels the content has been scrolled vertically.

```javascript
const verticalPosition =
  element.scrollTop;
```

`scrollLeft` is the horizontal scroll offset.

```javascript
const horizontalPosition =
  element.scrollLeft;
```

These properties can also be assigned:

```javascript
element.scrollTop =
  0;
```

Scroll to the left:

```javascript
element.scrollLeft =
  0;
```

---

# 19. Detecting Overflow

Vertical overflow:

```javascript
const hasVerticalOverflow =
  element.scrollHeight >
  element.clientHeight;
```

Horizontal overflow:

```javascript
const hasHorizontalOverflow =
  element.scrollWidth >
  element.clientWidth;
```

Example helper:

```javascript
function getOverflowState(
  element
) {
  return {
    horizontal:
      element.scrollWidth >
      element.clientWidth,

    vertical:
      element.scrollHeight >
      element.clientHeight,
  };
}
```

---

# 20. Detecting the Bottom of a Scroll Container

A common calculation is:

```javascript
const reachedBottom =
  element.scrollTop +
    element.clientHeight >=
  element.scrollHeight;
```

Because measurements may involve fractional values or rounding, use a tolerance:

```javascript
function isNearBottom(
  element,
  tolerance = 2
) {
  return (
    element.scrollTop +
      element.clientHeight >=
    element.scrollHeight -
      tolerance
  );
}
```

This is useful for:

* Infinite scrolling.
* Loading more messages.
* Enabling a Continue button after reading.
* Chat interfaces.

---

# 21. `getBoundingClientRect()`

This method returns the element's visual border-box rectangle relative to the viewport.

```javascript
const rect =
  element.getBoundingClientRect();
```

It is one of the most important geometry APIs.

Use it for:

* Tooltips.
* Dropdown positioning.
* Visibility checks.
* Collision detection.
* Dragging.
* Viewport-relative layout.
* Sticky behavior.

---

# 22. The `DOMRect` Object

A rectangle commonly contains:

```javascript
rect.x
rect.y
rect.top
rect.right
rect.bottom
rect.left
rect.width
rect.height
```

Relationships:

```javascript
rect.right ===
  rect.left +
  rect.width;
```

```javascript
rect.bottom ===
  rect.top +
  rect.height;
```

Values may be fractional.

---

# 23. Viewport Coordinates

`rect.top` measures from the viewport's top edge.

```text
Viewport top
    ↓
rect.top
    ↓
Element top
```

If the page scrolls, `rect.top` changes.

Example:

```javascript
const before =
  element
    .getBoundingClientRect()
    .top;

window.scrollBy(
  0,
  100
);

const after =
  element
    .getBoundingClientRect()
    .top;
```

`after` is typically about 100 pixels smaller.

---

# 24. Document Coordinates

Convert viewport coordinates to document coordinates:

```javascript
const rect =
  element.getBoundingClientRect();

const documentLeft =
  rect.left +
  window.scrollX;

const documentTop =
  rect.top +
  window.scrollY;
```

This describes the position relative to the page document.

Use document coordinates carefully because:

* Nested scrolling containers may matter.
* Fixed positioning behaves differently.
* Transforms affect visual positions.
* Zoom and fractional pixels may appear.

---

# 25. Transformed Elements

CSS transforms affect visual geometry.

```css
.card {
  transform:
    scale(1.2);
}
```

`getBoundingClientRect()` generally reflects the transformed visual rectangle.

`offsetWidth` generally reflects layout dimensions before transforms.

Therefore:

```javascript
element.offsetWidth;
```

and:

```javascript
element
  .getBoundingClientRect()
  .width;
```

may differ.

---

# 26. Fractional Measurements

`getBoundingClientRect()` may return values such as:

```javascript
{
  left: 120.375,
  width: 249.625,
}
```

Do not assume integer pixels.

Fractional values arise from:

* Zoom.
* Percentage layout.
* Transforms.
* High-density displays.
* Flexible layouts.
* Font metrics.

Round only when the application genuinely requires it.

---

# 27. Measuring Hidden Elements

## `display: none`

Dimensions are usually zero because no layout box exists.

## `visibility: hidden`

The box exists and can usually be measured.

## Detached element

A detached element may not have meaningful layout because it has not been inserted into a rendering context.

## Measurement strategy

A common approach is:

1. Insert into a suitable container.
2. Make it non-visible without removing layout participation.
3. Measure.
4. Restore or remove.

Example:

```javascript
function measureHiddenElement(
  element,
  container
) {
  const previous = {
    position:
      element.style.position,

    visibility:
      element.style.visibility,

    display:
      element.style.display,
  };

  element.style.position =
    "absolute";

  element.style.visibility =
    "hidden";

  element.style.display =
    "block";

  container.append(element);

  const rect =
    element.getBoundingClientRect();

  element.remove();

  element.style.position =
    previous.position;

  element.style.visibility =
    previous.visibility;

  element.style.display =
    previous.display;

  return rect;
}
```

This pattern should be used carefully because styles and layout context affect measurements.

---

# 28. Measuring Newly Inserted Elements

After insertion:

```javascript
container.append(element);
```

You can often measure immediately:

```javascript
const rect =
  element.getBoundingClientRect();
```

The browser may synchronously complete pending layout calculations to provide the result.

For coordinated visual transitions, use:

```javascript
requestAnimationFrame(
  () => {
    const rect =
      element
        .getBoundingClientRect();

    // Use measurement.
  }
);
```

  0,

# 31. `window.scrollBy()`

Scroll relative to the current position.

```javascript

  0,
  200
);
```

Options:

```javascript

  top: 200,
  behavior: "smooth",
});
```

Difference:

```text
scrollTo:
Go to this absolute position.

scrollBy:
Move this distance from the current position.
```

element.scrollLeft

```css
.panel {
  max-height: 300px;
  overflow-y: auto;
}

panel.scrollBy({

element.scrollIntoView();

center
end
nearest
```

```text

center
end
nearest
```

# 36. Smooth Scrolling

CSS:

```css
html {
  scroll-behavior:
    smooth;
}
```

JavaScript:

```javascript
element.scrollIntoView({
  behavior: "smooth",
});
```

Respect reduced-motion preferences.

```javascript
const reduceMotion =
  matchMedia(
    "(prefers-reduced-motion: reduce)"
  ).matches;

element.scrollIntoView({
  behavior:
    reduceMotion
      ? "auto"
      : "smooth",
});
```

# 37. Scroll Margins and Fixed Headers

A fixed header can cover content scrolled to the top.

CSS solution:

```css
section {
  scroll-margin-top:
    80px;
}
```

Then:

```javascript
section.scrollIntoView({
  block: "start",
});
```

The browser leaves the configured margin.

This is better than manually subtracting a fixed header height in many cases.

# 38. Reading Mouse and Pointer Coordinates

Pointer-related events expose several coordinate values.

```javascript
event.clientX
event.clientY
```

Viewport-relative coordinates.

```javascript
event.pageX
event.pageY
```

Document-relative coordinates.

```javascript
event.screenX
event.screenY
```

Screen-relative coordinates.

For normal DOM positioning, `clientX` and `clientY` often pair naturally with `getBoundingClientRect()`.

---

# 39. Converting Pointer Coordinates to Local Coordinates

Suppose a user clicks inside a chart.

```javascript
const rect =
  chart.getBoundingClientRect();

const localX =
  event.clientX -
  rect.left;

const localY =
  event.clientY -
  rect.top;
```

Now:

```text
localX = distance from chart's left edge
localY = distance from chart's top edge
```

This is useful for:

* Canvas drawing.
* Sliders.
* Heat maps.
* Dragging.
* Custom graph interactions.

---

# 40. Tooltip Positioning

Basic pattern:

```javascript
function positionTooltip(
  tooltip,
  target
) {
  const targetRect =
    target.getBoundingClientRect();

  const tooltipRect =
    tooltip.getBoundingClientRect();

  const left =
    targetRect.left +
    (
      targetRect.width -
      tooltipRect.width
    ) /
      2;

  const top =
    targetRect.top -
    tooltipRect.height -
    8;

  tooltip.style.left =
    `${left}px`;

  tooltip.style.top =
    `${top}px`;
}
```

This assumes the tooltip uses:

```css
position: fixed;
```

because the coordinates are viewport-relative.

# 48. Collision Detection Basics

Two rectangles overlap when:

```javascript
function rectanglesOverlap(
  first,
  second
) {
  return !(
    first.right <=
      second.left ||
    first.left >=
      second.right ||
    first.bottom <=
      second.top ||
    first.top >=
      second.bottom
  );
}
```

Use cases:

* Drag-and-drop.
* Games.
* Floating interfaces.
* Selection boxes.
* Drop targets.

This is axis-aligned rectangle collision detection.

# 53. Layout Thrashing

Problem:

```javascript
for (
  const card
  of cards
) {
  const width =
    card.offsetWidth;

  card.style.height =
    `${width}px`;
}
```

Each iteration reads layout and writes style.

A future read may force another layout.

This can create repeated expensive browser work.

---

# 54. Batching Geometry Reads and Writes

Better:

```javascript
const measurements =
  cards.map(
    card => ({
      card,
      width:
        card.offsetWidth,
    })
  );

for (
  const {
    card,
    width,
  } of measurements
) {
  card.style.height =
    `${width}px`;
}
```

Workflow:

```text
Read all
    ↓
Calculate all
    ↓
Write all
```

const article =
  document.querySelector(
    "#article"

  document.querySelector(
    ".reading-progress"

  progress.querySelector(

  const rect =
    article

    rect.top +

        ratio * 100

## Example 3: Real-World Application — Transaction List Infinite Loading

### HTML

```html
<section
  id="transaction-list"
  class="transaction-list"
  aria-busy="false"
>
  <div
    data-role="items"
  ></div>

  <div
    data-role="sentinel"
    aria-hidden="true"
  ></div>

  <p
    data-role="status"
    aria-live="polite"
  ></p>
</section>
```

### JavaScript

```javascript
function createInfiniteLoader({
  root,
  loadNextPage,
}) {
  const items =
    root.querySelector(
      "[data-role='items']"
    );

  const sentinel =
    root.querySelector(
      "[data-role='sentinel']"
    );

  const status =
    root.querySelector(
      "[data-role='status']"
    );

  if (
    !items ||
    !sentinel ||
    !status
  ) {
    throw new Error(
      "Invalid infinite-list structure."
    );
  }

  let loading = false;
  let complete = false;

  const observer =
    new IntersectionObserver(
      async entries => {
        const entry =
          entries[0];

        if (
          !entry?.isIntersecting ||
          loading ||
          complete
        ) {
          return;
        }

        loading = true;

        root.setAttribute(
          "aria-busy",
          "true"
        );

        status.textContent =
          "Loading more transactions...";

        try {
          const result =
            await loadNextPage();

          for (
            const transaction
            of result.transactions
          ) {
            items.append(
              createTransactionCard(
                transaction
              )
            );
          }

          complete =
            result.hasMore ===
            false;

          status.textContent =
            complete
              ? "All transactions loaded."
              : `${result.transactions.length} more transactions loaded.`;

          if (complete) {
            observer.disconnect();
          }
        } catch (error) {
          status.textContent =
            "Unable to load more transactions.";
        } finally {
          loading = false;

          root.setAttribute(
            "aria-busy",
            "false"
          );
        }
      },
      {
        root:
          root,

        rootMargin:
          "0px 0px 200px 0px",
      }
    );

  observer.observe(
    sentinel
  );

  function destroy() {
    observer.disconnect();
  }

  return {
    destroy,
  };
}
```

### Why this is professional

* A sentinel avoids repeated manual scroll calculations.
* A loading guard prevents duplicate requests.
* `aria-busy` communicates state.
* A live region announces progress.
* Observer cleanup is exposed.
* Completion disconnects observation.
* Errors preserve the component.

# 62. Common Mistakes

## Mistake 1: Assuming `offsetWidth` is content width

It generally includes padding and borders.

---

## Mistake 2: Assuming `clientWidth` includes borders

It does not.

---

## Mistake 3: Comparing `scrollHeight` with `offsetHeight` blindly

For scrolling logic, `clientHeight` is usually the better visible-area comparison.

---

## Mistake 4: Expecting hidden `display: none` elements to have dimensions

Their layout dimensions are generally zero.

---

## Mistake 5: Treating `offsetTop` as viewport position

It is relative to `offsetParent`.

---

## Mistake 6: Forgetting scroll offsets when converting to document coordinates

Use:

```javascript
rect.top +
window.scrollY
```

---

## Mistake 7: Mixing viewport coordinates with absolute document positioning

Coordinate systems must match the CSS positioning strategy.

---

## Mistake 8: Using exact equality at the bottom of a scroll container

Rounding may prevent equality.

Use a tolerance.

---

## Mistake 9: Measuring a tooltip while it is `display: none`

It has no measurable layout box.

---

## Mistake 10: Repeatedly reading and writing layout in one loop

This can cause layout thrashing.

---

## Mistake 11: Forgetting to reposition floating UI after resize or scroll

The reference geometry may have changed.

---

## Mistake 12: Forgetting observer cleanup

Removed components may remain observed.

---

# 63. Professional Best Practices

1. Choose the measurement API based on the box you need.
2. Use `offsetWidth` for border-box layout dimensions.
3. Use `clientWidth` for inner visible dimensions.
4. Use `scrollWidth` and `scrollHeight` for complete content size.
5. Use `scrollTop` and `scrollLeft` for current scroll offsets.
6. Use `getBoundingClientRect()` for viewport-relative visual geometry.
7. Convert coordinates explicitly when changing coordinate systems.
8. Expect fractional rectangle values.
9. Do not measure `display: none` content directly.
10. Match positioning coordinates with `fixed` or `absolute` positioning correctly.
11. Clamp floating interfaces to viewport boundaries.
12. Use tolerances in scroll-boundary calculations.
13. Prefer `IntersectionObserver` for visibility and sentinel workflows.
14. Prefer `ResizeObserver` for element-size changes.
15. Batch geometry reads before DOM writes.
16. Avoid forced synchronous layout in loops.
17. Cache measurements only while valid.
18. Respect reduced-motion preferences during scrolling.
19. Keep scrolling and focus behavior separate and intentional.
20. Disconnect observers and remove listeners during teardown.

# 65. Interview Questions and Answers

## Beginner Questions

### 1. What does `offsetWidth` measure?

The element's layout border-box width, commonly including content, padding, borders, and scrollbar width.

### 2. Does `offsetWidth` include margins?

No.

### 3. What does `clientWidth` measure?

The inner visible width, generally including content and padding but excluding borders and the vertical scrollbar.

### 4. What does `scrollHeight` represent?

The complete height of scrollable content.

### 5. What does `scrollTop` represent?

The number of pixels the element's content has been scrolled vertically.

### 6. What does `getBoundingClientRect()` return?

A viewport-relative rectangle describing the element's visual border box.

### 7. What type does `getBoundingClientRect()` return?

A `DOMRect`-like object.

### 8. Are rectangle coordinates always integers?

No.

---

## Intermediate Questions

### 1. Compare `offsetWidth` and `clientWidth`.

`offsetWidth` includes borders and commonly scrollbars, while `clientWidth` excludes borders and the vertical scrollbar.

### 2. How do you detect vertical overflow?

```javascript
element.scrollHeight >
element.clientHeight
```

### 3. How do you detect whether a user is near the bottom?

Compare `scrollTop + clientHeight` against `scrollHeight` with a tolerance.

### 4. Why does `getBoundingClientRect().top` change during scrolling?

It is relative to the viewport.

### 5. How do you convert a rectangle top to document coordinates?

```javascript
rect.top +
window.scrollY
```

### 6. Why can a transformed element's rectangle width differ from `offsetWidth`?

The rectangle reflects visual transforms, while offset dimensions represent layout dimensions.

### 7. Why do `display: none` elements return zero dimensions?

They do not have active layout boxes.

---

## Advanced Questions

### 1. What is a forced synchronous layout?

It occurs when JavaScript requests a layout-dependent value after pending style changes, forcing the browser to calculate layout immediately.

### 2. What is layout thrashing?

Repeatedly alternating layout reads and writes, causing frequent forced layout recalculations.

### 3. Why is `ResizeObserver` better than only listening for window resize?

Elements can resize without the window changing size.

### 4. Why is `IntersectionObserver` useful for infinite scrolling?

It efficiently reports when a sentinel approaches or enters the visible root area without continuous manual geometry calculations.

### 5. How would you position a dropdown safely?

Measure the trigger and dropdown, choose a matching coordinate system, clamp horizontal position, flip vertically when space is insufficient, and reposition on relevant resize or scroll changes.

### 6. How should geometry caches be invalidated?

On resize, content changes, font or image loading, style changes, scrolling, transforms, and relevant observer notifications.

### 7. How do geometry APIs relate to accessibility?

They support visual placement, but focus, keyboard operation, semantic roles, zoom, reflow, and reduced-motion behavior must be implemented separately.

# 69. Summary

Element geometry APIs reveal how the browser has laid out and positioned content.

You learned four major measurement families.

## Offset measurements

```javascript
offsetWidth
offsetHeight
offsetTop
offsetLeft
offsetParent
```

They provide border-box dimensions and offset-parent-relative positioning.

## Client measurements

```javascript
clientWidth
clientHeight
clientTop
clientLeft
```

They describe the inner visible area and borders.

## Scroll measurements

```javascript
scrollWidth
scrollHeight
scrollTop
scrollLeft
```

They describe complete scrollable content and current scroll offsets.

## Viewport rectangles

```javascript
getBoundingClientRect()
```

returns:

```javascript
top
right
bottom
left
width
height
x
y
```

You also learned:

* Geometry APIs measure different box-model layers.
* `offsetWidth` includes borders.
* `clientWidth` excludes borders.
* `scrollHeight` may exceed visible height.
* Viewport coordinates change during scrolling.
* Document coordinates require adding window scroll offsets.
* Transforms affect visual rectangles.
* Measurements may be fractional.
* `display: none` elements do not have measurable layout boxes.
* `scrollIntoView()` brings elements into view.
* CSS `scroll-margin` helps with fixed headers.
* Pointer coordinates can be converted to element-local coordinates.
* Floating UI must be clamped or flipped.
* `IntersectionObserver` helps with visibility and infinite scrolling.
* `ResizeObserver` tracks element-size changes.
* Geometry reads can force layout.
* Layout reads and writes should be batched.
* Observers and listeners require lifecycle cleanup.

The professional geometry workflow is:

```text
Choose coordinate system
    ↓
Choose correct measurement API
    ↓
Read required geometry
    ↓
Calculate positions
    ↓
Clamp or adjust
    ↓
Write styles
    ↓
Observe future changes
    ↓
Clean resources during teardown
```
