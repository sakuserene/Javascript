# 1. Introduction

You have already learned how to:

* Create DOM elements.
* Append and prepend children.
* Insert siblings before and after reference elements.
* Insert elements at precise relative positions.

The next major operation is replacement.

Suppose the page displays a loading message:

```html
<section id="transaction-area">
  <p id="loading-message">
    Loading transactions...
  </p>
</section>
```

After data arrives, the loading message should disappear and a transaction list should take its place.

You could manually:

1. Insert the new list.
2. Remove the loading message.

But replacement expresses the intention more directly:

```javascript
loadingMessage.replaceWith(
  transactionList
);
```

Updated DOM:

```html
<section id="transaction-area">
  <ul id="transaction-list"></ul>
</section>
```

Replacement is useful when one DOM structure becomes outdated and another structure should occupy its position.

---

# 2. What Is DOM Replacement?

> DOM replacement is the process of removing an existing node or child collection and placing new nodes or text in its position.

Replacement combines two conceptual operations:

```text
Remove old structure
    +
Insert new structure
```

But it performs them as one intentional DOM operation.

Examples:

* Replace a loading indicator with content.
* Replace an old card with an updated card.
* Replace an error panel with a success panel.
* Replace a table row after editing.
* Replace all list children after filtering.
* Replace form steps in a multi-step form.
* Replace modal content.
* Clear and rebuild a dashboard region.

---

# 3. Why Replacement APIs Exist

Without replacement APIs, developers may write:

```javascript
oldElement.before(
  newElement
);

oldElement.remove();
```

This can work.

However, replacement APIs communicate the relationship more clearly:

```javascript
oldElement.replaceWith(
  newElement
);
```

The code says:

> Replace this old element with the new one.

Replacement APIs improve:

* Readability.
* Intent.
* Component rerendering.
* Loading-state transitions.
* Empty-state transitions.
* Dynamic form workflows.
* Error recovery.
* List and table updates.

---

# 4. Insertion vs Replacement vs Removal

## Insertion

Adds content while keeping the existing content.

```javascript
reference.after(
  newElement
);
```

## Replacement

Exchanges old content for new content.

```javascript
oldElement.replaceWith(
  newElement
);
```

## Removal

Deletes old content without adding a replacement.

```javascript
oldElement.remove();
```

Decision:

```text
Keep old and add new?
Use insertion.

Discard old and show new?
Use replacement.

Discard old completely?
Use removal.
```

---

# 5. Mental Model

Imagine a row of objects:

```text
A
B
C
```

Replace `B` with `X`:

```text
A
X
C
```

The new node occupies the old node's position among siblings.

Replacing all children:

```text
Parent
├── A
├── B
└── C
```

Replace children with `X` and `Y`:

```text
Parent
├── X
└── Y
```

The parent remains.

Its old child collection is replaced.

---

# 6. How Replacement Changes the DOM Tree

Before:

```text
section
├── heading
├── loading-message
└── footer
```

Code:

```javascript
loadingMessage.replaceWith(
  transactionList
);
```

After:

```text
section
├── heading
├── transaction-list
└── footer
```

The old loading message becomes detached.

```javascript
loadingMessage.isConnected;
```

Output:

```text
false
```

The new list becomes connected.

```javascript
transactionList.isConnected;
```

Output:

```text
true
```

---

# 7. The Three Main Replacement APIs

| Method              | Called On           | Replaces     | Accepts Multiple Values? |  Accepts Strings? | Return Value      |
| ------------------- | ------------------- | ------------ | -----------------------: | ----------------: | ----------------- |
| `replaceWith()`     | Node being replaced | That node    |                      Yes |               Yes | `undefined`       |
| `replaceChild()`    | Parent              | One child    |                       No | No, requires Node | Removed old child |
| `replaceChildren()` | Parent              | All children |                      Yes |               Yes | `undefined`       |

Each API answers a different structural question.

---

# 8. `replaceWith()` Introduction

`replaceWith()` replaces the reference node with one or more nodes or strings.

Syntax:

```javascript
oldNode.replaceWith(
  ...nodesOrStrings
);
```

Example:

```javascript
loading.replaceWith(
  content
);
```

The new content takes the loading node's position.

---

# 9. `replaceWith()` Syntax

```javascript
referenceNode.replaceWith(
  value1,
  value2,
  value3
);
```

Each value may be:

* An element.
* A text node.
* A comment node.
* A document fragment.
* A string.

Strings become text nodes.

Multiple values are inserted in argument order.

The original reference node is removed.

---

# 10. Replacing One Element with Another

HTML:

```html
<div id="status-area">
  <p id="loading">
    Loading...
  </p>
</div>
```

JavaScript:

```javascript
const loading =
  document.querySelector(
    "#loading"
  );

const success =
  document.createElement(
    "p"
  );

success.className =
  "success-message";

success.textContent =
  "Transactions loaded.";

loading.replaceWith(
  success
);
```

Result:

```html
<div id="status-area">
  <p class="success-message">
    Transactions loaded.
  </p>
</div>
```

---

# 11. Replacing with Multiple Nodes

`replaceWith()` can replace one node with several nodes.

Before:

```html
<section>
  <p id="placeholder">
    No content
  </p>
</section>
```

JavaScript:

```javascript
placeholder.replaceWith(
  heading,
  paragraph,
  button
);
```

After:

```html
<section>
  <h2></h2>
  <p></p>
  <button></button>
</section>
```

The old placeholder occupied one sibling position.

The three replacement nodes now occupy that position in argument order.

---

# 12. Replacing with Strings

```javascript
message.replaceWith(
  "Operation completed."
);
```

The string becomes a text node.

The result is not an element.

If semantic structure matters, create an element instead:

```javascript
const status =
  document.createElement(
    "p"
  );

status.textContent =
  "Operation completed.";

message.replaceWith(
  status
);
```

---

# 13. `replaceWith()` Return Value

`replaceWith()` returns:

```text
undefined
```

Example:

```javascript
const result =
  oldElement.replaceWith(
    newElement
  );

console.log(result);
```

Output:

```text
undefined
```

Do not expect the replacement element from the return value.

Keep its original reference:

```javascript
oldElement.replaceWith(
  newElement
);

newElement.focus?.();
```

---

# 14. What Happens to the Replaced Node?

The old node becomes detached.

```javascript
oldElement.isConnected;
```

Output:

```text
false
```

Its parent becomes:

```javascript
oldElement.parentElement;
```

Output:

```text
null
```

The JavaScript reference still exists while stored in a variable.

You can inspect it.

```javascript
console.log(
  oldElement
);
```

Detached does not mean destroyed.

It becomes eligible for garbage collection only when no reachable references remain.

---

# 15. `replaceChild()` Introduction

`replaceChild()` is called on the parent.

Syntax:

```javascript
parent.replaceChild(
  newChild,
  oldChild
);
```

This replaces one direct child with another node.

Example:

```javascript
container.replaceChild(
  successMessage,
  loadingMessage
);
```

---

# 16. `replaceChild()` Syntax

```javascript
const removedNode =
  parent.replaceChild(
    newChild,
    oldChild
  );
```

Requirements:

* `oldChild` must be a direct child of `parent`.
* `newChild` must be a Node.
* One replacement node is accepted.
* Plain strings are not accepted.

---

# 17. Argument Order in `replaceChild()`

The argument order is:

```javascript
newChild,
oldChild
```

This is easy to confuse.

Read:

```javascript
parent.replaceChild(
  newChild,
  oldChild
);
```

as:

> Parent, replace old child with new child.

A common memory aid:

```text
replaceChild(new, old)
```

Incorrect order:

```javascript
parent.replaceChild(
  oldChild,
  newChild
);
```

usually fails because `newChild` is not yet a child of the parent.

---

# 18. `replaceChild()` Return Value

`replaceChild()` returns the removed old child.

```javascript
const removed =
  parent.replaceChild(
    newChild,
    oldChild
  );

console.log(
  removed === oldChild
);
```

Output:

```text
true
```

This can be useful when:

* Building undo functionality.
* Logging removed nodes.
* Temporarily preserving the replaced node.
* Inspecting the old component.

Be careful not to retain old nodes indefinitely without reason.

---

# 19. `replaceWith()` vs `replaceChild()`

| Feature                    | `replaceWith()` | `replaceChild()`     |
| -------------------------- | --------------- | -------------------- |
| Called on                  | Old node        | Parent               |
| Requires parent variable   | No              | Yes                  |
| Multiple replacements      | Yes             | No                   |
| Accepts strings            | Yes             | No                   |
| Return value               | `undefined`     | Removed old node     |
| Readability                | Very direct     | More parent-oriented |
| Requires old node be child | Implicitly      | Explicitly           |

Prefer `replaceWith()` for modern, direct replacement.

Use `replaceChild()` when:

* The parent is already central to the workflow.
* You need the removed node returned.
* Maintaining older code.
* Working with generic parent-child APIs.

---

# 20. `replaceChildren()` Introduction

`replaceChildren()` removes all current children from a parent and inserts the provided nodes or strings.

Syntax:

```javascript
parent.replaceChildren(
  ...nodesOrStrings
);
```

It keeps the parent itself.

It replaces only the parent's child collection.

Example:

```javascript
list.replaceChildren(
  firstItem,
  secondItem
);
```

All old list children are removed.

The two new items become the complete child set.

---

# 21. `replaceChildren()` Syntax

```javascript
parent.replaceChildren(
  value1,
  value2,
  value3
);
```

Accepted values:

* Elements.
* Nodes.
* Fragments.
* Strings.
* Multiple arguments.
* No arguments.

No arguments means:

```text
Remove all children.
```

---

# 22. Replacing All Children with One Node

Before:

```html
<section id="content">
  <p>Old paragraph</p>
  <button>Old button</button>
</section>
```

JavaScript:

```javascript
const loading =
  document.createElement(
    "p"
  );

loading.textContent =
  "Loading...";

content.replaceChildren(
  loading
);
```

After:

```html
<section id="content">
  <p>Loading...</p>
</section>
```

The section remains connected.

Its old descendants become detached.

---

# 23. Replacing All Children with Multiple Nodes

```javascript
container.replaceChildren(
  heading,
  description,
  actions
);
```

The parent now contains exactly:

```text
heading
description
actions
```

All previous children are removed.

Argument order becomes DOM order.

---

# 24. Replacing Children with Text

```javascript
container.replaceChildren(
  "No transactions found."
);
```

The parent receives one text node.

For semantic empty states, an element is usually better:

```javascript
const message =
  document.createElement(
    "p"
  );

message.textContent =
  "No transactions found.";

container.replaceChildren(
  message
);
```

---

# 25. Clearing All Children

Call `replaceChildren()` with no arguments:

```javascript
container.replaceChildren();
```

The parent becomes empty.

Equivalent intention:

```text
Keep parent
Remove every child
```

This is a clear modern alternative to:

```javascript
container.innerHTML =
  "";
```

It avoids HTML-string parsing.

---

# 26. `replaceChildren()` Return Value

`replaceChildren()` returns:

```text
undefined
```

Example:

```javascript
const result =
  container.replaceChildren(
    message
  );

console.log(result);
```

Output:

```text
undefined
```

Keep references to inserted nodes separately when needed.

---

# 27. Replacement and Node Movement

Replacement methods move existing nodes.

Suppose:

```html
<div id="first">
  <p id="message">
    Existing message
  </p>
</div>

<div id="second">
  <p id="placeholder">
    Placeholder
  </p>
</div>
```

Code:

```javascript
placeholder.replaceWith(
  message
);
```

Result:

```html
<div id="first"></div>

<div id="second">
  <p id="message">
    Existing message
  </p>
</div>
```

`message` moved from the first container to the second.

---

# 28. A Reused Node Is Moved, Not Cloned

A DOM node can occupy one position.

```javascript
firstPlaceholder.replaceWith(
  sharedElement
);

secondPlaceholder.replaceWith(
  sharedElement
);
```

After the second replacement, `sharedElement` exists only at the second location.

To create copies, use:

* A factory.
* `cloneNode()`.
* Repeated element creation.

Cloning is covered in a later lesson.

---

# 29. Replacing with a DocumentFragment

Create a fragment:

```javascript
const fragment =
  document.createDocumentFragment();

fragment.append(
  heading,
  paragraph,
  button
);
```

Replace one placeholder:

```javascript
placeholder.replaceWith(
  fragment
);
```

The fragment's children take the placeholder's position.

The fragment becomes empty.

```javascript
fragment.childNodes.length;
```

Output:

```text
0
```

# 30. Replacing Component Roots

Suppose a transaction card must be rebuilt.

Existing root:

```html
<article
  data-component="transaction-card"
  data-transaction-id="42"
>
</article>
```

Create new root:

```javascript
const updatedCard =
  createTransactionCard(
    updatedTransaction
  );
```

Replace:

```javascript
oldCard.replaceWith(
  updatedCard
);
```

Important consequences:

* `oldCard` becomes detached.
* Old descendant references become detached.
* Old direct listeners remain on the old node, not the new one.
* Delegated parent listeners continue to work for the new card.
* Registries must be updated.
* Focus may need restoration.

---

# 31. Replacing Component Internals

Instead of replacing the complete root, replace a region inside it.

Example:

```html
<article class="transaction-card">
  <header></header>
  <section data-region="body"></section>
  <footer></footer>
</article>
```

Create new body:

```javascript
const newBody =
  createTransactionBody(
    transaction
  );
```

Replace existing region:

```javascript
oldBody.replaceWith(
  newBody
);
```

The root remains stable.

This can preserve:

* Root identity.
* Root-level event references.
* Registry references.
* Some component state.
* Focus outside the replaced region.

---

# 32. Whole-Component vs Partial Replacement

## Whole-component replacement

```javascript
oldCard.replaceWith(
  newCard
);
```

Advantages:

* Simple mental model.
* Complete consistency.
* Fresh structure.
* Useful when layout changes significantly.

Trade-offs:

* All descendant references become stale.
* Local listeners may be lost.
* Focus may be lost.
* More DOM work.

## Partial replacement

```javascript
oldStatus.replaceWith(
  newStatus
);
```

Advantages:

* Smaller update.
* Fewer stale references.
* Better focus preservation.
* Less DOM work.

Trade-offs:

* More detailed update logic.
* Risk of inconsistent partial state if poorly designed.

Professional rule:

> Replace the smallest meaningful region that fully represents the changed state.

---

# 33. Stale References After Replacement

Suppose:

```javascript
const status =
  card.querySelector(
    "[data-role='status']"
  );

status.replaceWith(
  newStatus
);
```

The `status` variable still points to the old node.

```javascript
status.isConnected;
```

Output:

```text
false
```

If code later runs:

```javascript
status.textContent =
  "Paid";
```

it updates the detached old node.

The visible new status remains unchanged.

---

# 34. Detecting Stale References

Compare cached and current references:

```javascript
const currentStatus =
  card.querySelector(
    "[data-role='status']"
  );

console.log({
  cached:
    status,

  current:
    currentStatus,

  same:
    status ===
      currentStatus,

  cachedConnected:
    status.isConnected,

  currentConnected:
    currentStatus
      ?.isConnected,
});
```

A stale cache often shows:

```javascript
{
  same: false,
  cachedConnected: false,
  currentConnected: true,
}
```

# 35. Refreshing Component Views

Refreshable view:

```javascript
function createTransactionView(
  root
) {
  let parts;

  function refresh() {
    parts = {
      receipt:
        root.querySelector(
          "[data-role='receipt']"
        ),

      amount:
        root.querySelector(
          "[data-role='amount']"
        ),

      status:
        root.querySelector(
          "[data-role='status']"
        ),
    };
  }

  refresh();

  return {
    root,

    refresh,

    get parts() {
      return parts;
    },
  };
}
```

After internal replacement:

```javascript
view.refresh();
```

After root replacement, recreate the view or update the registry with a new view.

# 36. Replacing Loading States

Initial state:

```html
<section id="transactions">
  <p data-state="loading">
    Loading transactions...
  </p>
</section>
```

Create list:

```javascript
const list =
  createTransactionList(
    transactions
  );
```

Replace:

```javascript
loadingMessage.replaceWith(
  list
);
```

Alternative parent-state rendering:

```javascript
transactionArea.replaceChildren(
  list
);
```

Use `replaceWith()` when replacing a known placeholder.

Use `replaceChildren()` when the entire region should represent one current state.

---

# 37. Replacing Empty States

Current:

```html
<section id="transaction-area">
  <div data-state="empty">
    No transactions found.
  </div>
</section>
```

When the first transaction is added:

```javascript
emptyState.replaceWith(
  transactionList
);
```

Or:

```javascript
transactionArea.replaceChildren(
  transactionList
);
```

When the final transaction is deleted, reverse the state:

```javascript
transactionList.replaceWith(
  createEmptyState()
);
```

State transitions should be explicit.

---

# 38. Replacing Error and Success States

Loading fails:

```javascript
loadingMessage.replaceWith(
  createErrorPanel(
    "Unable to load transactions."
  )
);
```

Retry succeeds:

```javascript
errorPanel.replaceWith(
  transactionList
);
```

Form status replacement:

```javascript
statusRegion.replaceChildren(
  createSuccessMessage(
    "Transaction saved."
  )
);
```

A stable status-region parent can be easier to manage than repeatedly replacing the outer region.

# 39. Replacing Form Sections

Multi-step form:

```text
Form
└── Current step region
```

Replace step:

```javascript
stepRegion.replaceChildren(
  createPaymentStep()
);
```

Or replace the complete step root:

```javascript
currentStep.replaceWith(
  nextStep
);
```

Consider preserving:

* Entered values.
* Focus.
* Validation state.
* Step identity.
* Accessible step heading.

---

# 40. Replacing List Items

Suppose an item is edited.

Old item:

```javascript
const oldItem =
  findItemElementById(
    id
  );
```

```javascript
  newItem
);
```

If the updated item remains structurally identical, targeted text updates may be more efficient than complete replacement.

# 46. Identity During Replacement

When replacing one record view, preserve the same application identity.

Old:

```html
  data-transaction-id="42"
></article>
```

New:

```html

  data-transaction-id="42"
></article>
```

The DOM object changes.

The business identity remains.

This distinction matters:

```text

Old element !== new element

Application identity:
Both represent transaction 42
```

Registries should be keyed by application identity, then updated to point to the new view.

# 48. Preserving Selection and Input State

Replacing an input element loses its live state unless copied.

Potentially lost:

* Current value.
* Selection range.
* Checked state.
* Browser autofill state.
* Focus.
* Validation UI.
* Undo history.

Before replacement:

```javascript
const value =
  input.value;

const start =
  input.selectionStart;

const end =
  input.selectionEnd;
```

After replacement:

```javascript
newInput.value =
  value;

  start,
  end
);
```

# 51. Security Considerations

Replacement methods that accept strings treat them as text.

Safe:

```javascript
message.replaceWith(
  userInput
);
```

This creates a text node.

Safe semantic version:

```javascript
const paragraph =
  document.createElement(
    "p"
  );

paragraph.textContent =
  userInput;

message.replaceWith(
  paragraph
);
```

Security risks arise when replacement content is created through unsafe HTML parsing:

```javascript
container.innerHTML =
  userInput;
```

or:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  userInput
);
```

Prefer node creation and `textContent` for untrusted values.

# 52. Performance Considerations

Replacement may be more expensive than targeted updates.

Compare:

## Targeted update

```javascript
status.textContent =
  "Paid";
```

## Whole-card replacement

```javascript
oldCard.replaceWith(
  createTransactionCard(
    transaction
  )
);
```

Whole replacement:

* Creates more nodes.
* Removes more nodes.
* Invalidates more references.
* May disrupt focus.
* May cause more rendering work.

Use full replacement when:

* Structure changes significantly.
* Simplicity outweighs small cost.
* The component is small.
* Updates are infrequent.
* A clean rebuild is safer.

Use targeted updates when:

* Only text, attributes, or classes change.
* Inputs should retain state.
* Updates are frequent.
* Focus should remain stable.

# 53. Memory Considerations

Replaced nodes remain in memory while referenced.

Potential retainers:

* Component registries.
* Arrays.
* Undo history.
* Closures.
* Timers.
* Global listeners.
* Observer callbacks.

After root replacement:

```javascript
registry.set(
  id,
  newView
);
```

Ensure the old view is no longer retained.

If an old component registered global listeners, destroy it before replacement.

```javascript
oldController.destroy({
  removeRoot: false,
});

oldRoot.replaceWith(
  newRoot
);
```

Lifecycle order depends on the architecture.

## Example 3: Real-World Application — Replace a Transaction Card Safely

### View factory

```javascript
function createTransactionCardView(
  transaction
) {
  const root =
    createTransactionCard(
      transaction
    );

  return {
    root,

    get id() {
      return Number(
        root.dataset
          .transactionId
      );
    },

    getAction(
      action
    ) {
      return root.querySelector(
        `[data-action="${CSS.escape(
          action
        )}"]`
      );
    },
  };
}
```

### Replacement function

```javascript
function replaceTransactionCard({
  transaction,
  oldView,
  registry,
}) {
  if (
    oldView.id !==
    transaction.id
  ) {
    throw new Error(
      "Replacement identity mismatch."
    );
  }

  const activeElement =
    document.activeElement;

  let focusAction =
    null;

  if (
    activeElement instanceof
      Element &&
    oldView.root.contains(
      activeElement
    )
  ) {
    focusAction =
      activeElement.closest(
        "[data-action]"
      )?.dataset.action ??
      null;
  }

  const newView =
    createTransactionCardView(
      transaction
    );

  oldView.root.replaceWith(
    newView.root
  );

  registry.set(
    transaction.id,
    newView
  );

  if (focusAction) {
    newView
      .getAction(
        focusAction
      )
      ?.focus();
  }

  return newView;
}
```

### Why this is professional

* Application identity is validated.
* Focus context is captured before replacement.
* A fresh view is built.
* The old root is replaced in place.
* The registry is updated.
* Focus is restored to the equivalent action.
* The returned view represents the current DOM.

# 55. Common Mistakes

## Mistake 1: Expecting `replaceWith()` to return the new node

It returns `undefined`.

---

## Mistake 2: Reversing `replaceChild()` arguments

Correct:

```javascript
parent.replaceChild(
  newChild,
  oldChild
);
```

---

## Mistake 3: Replacing a node that is not a child of the parent

`replaceChild()` requires a direct child relationship.

---

## Mistake 4: Continuing to use the old reference

```javascript
oldElement.replaceWith(
  newElement
);

oldElement.textContent =
  "Updated";
```

This updates the detached old node.

---

## Mistake 5: Forgetting to update a registry

The registry may still point to the detached old view.

---

## Mistake 6: Replacing a focused element without restoring focus

Keyboard users lose context.

---

## Mistake 7: Replacing input elements unnecessarily

This can destroy current values, selection, and browser state.

---

## Mistake 8: Using full replacement for a tiny text change

Prefer:

```javascript
status.textContent =
  "Paid";
```

when structure is unchanged.

---

## Mistake 9: Replacing children and expecting old descendant listeners to remain usable

The old descendants were removed.

Listeners attached to them remain on detached objects, not new descendants.

---

## Mistake 10: Reusing one replacement node in several places

It moves each time.

---

## Mistake 11: Clearing with replacement but retaining all old nodes in an array

They remain reachable.

---

## Mistake 12: Replacing semantic structure incorrectly

Do not replace a table row with an invalid element or a list item with a structurally invalid child.

1. Use `replaceWith()` for direct modern node replacement.
2. Use `replaceChild()` when parent-oriented logic or the removed-node return value is useful.
3. Use `replaceChildren()` for complete region rendering or clearing.
4. Replace the smallest meaningful region.
5. Update state before or together with the view.
6. Preserve application identity.
7. Refresh cached descendant references.
8. Update component registries.
9. Destroy old component resources when necessary.
10. Capture focus context before replacement.
11. Restore focus to an equivalent or meaningful target.
12. Avoid replacing active inputs unless necessary.
13. Preserve selection and scroll when the workflow requires it.
14. Use semantic factories for replacement content.
15. Prefer `textContent` for untrusted text.
16. Avoid unnecessary HTML-string replacement.
17. Validate parent-child relationships.
18. Return new roots or views from high-level replacement helpers.
19. Test empty, loading, error, and success transitions.
20. Use DevTools DOM breakpoints to diagnose unexpected replacement.

# 58. Interview Questions and Answers

## Beginner Questions

### 1. What does `replaceWith()` do?

It replaces the reference node with one or more nodes or strings.

### 2. What does `replaceChild()` do?

It replaces one direct child of a parent with another node.

### 3. What does `replaceChildren()` do?

It removes all current children and inserts the provided nodes or strings.

### 4. What does `replaceWith()` return?

`undefined`.

### 5. What does `replaceChild()` return?

The removed old child.

### 6. What happens when `replaceChildren()` is called without arguments?

All children are removed.

### 7. Can `replaceWith()` accept strings?

Yes. Strings become text nodes.

### 8. Can `replaceChild()` accept strings?

No. It requires a Node.

---

## Intermediate Questions

### 1. What is the argument order for `replaceChild()`?

```javascript
replaceChild(
  newChild,
  oldChild
)
```

### 2. What happens to the old node after replacement?

It becomes detached but remains in memory while referenced.

### 3. Does replacement clone an existing node?

No. Existing nodes are moved.

### 4. Why can replacement create stale references?

Stored variables may still point to detached old descendants.

### 5. When is `replaceChildren()` useful?

For clearing, complete-region rendering, loading states, empty states, and replacing all child content.

### 6. Why can full-component replacement disrupt focus?

The focused node may be removed from the document.

### 7. Why should state be updated with replacement?

Otherwise DOM and application data may become inconsistent.

---

## Advanced Questions

### 1. When should partial replacement be preferred over whole-component replacement?

When only a small region changes, references and focus should remain stable, or updates are frequent.

### 2. How would you preserve focus during component-root replacement?

Capture a semantic focus target before replacement, replace the root, then find and focus the equivalent target in the new component.

### 3. How should a component registry handle replacement?

Replace the old view entry with a new view that points to the connected new root.

### 4. Why might replacing an input be harmful?

It can lose current value, selection, undo history, focus, autofill, and browser-managed state.

### 5. How do you detect stale component references?

Check `isConnected` and compare cached descendants against fresh queries.

### 6. What memory problem can replacement cause?

Old detached views may remain retained by registries, arrays, closures, listeners, or observers.

### 7. How would you choose between targeted updates and replacement?

Consider structural change, update frequency, component size, focus requirements, reference stability, and performance measurements.

# 62. Summary

DOM replacement exchanges old structure for new structure.

You learned three main APIs:

```javascript
replaceWith()
replaceChild()
replaceChildren()
```

## `replaceWith()`

* Called on the node being replaced.
* Accepts multiple nodes and strings.
* Returns `undefined`.
* Detaches the original node.

## `replaceChild()`

* Called on the parent.
* Uses `newChild, oldChild` argument order.
* Accepts one Node.
* Returns the removed old child.

## `replaceChildren()`

* Called on the parent.
* Replaces the entire child collection.
* Accepts multiple nodes and strings.
* Clears the parent when called without arguments.
* Returns `undefined`.

You also learned:

* Replacement moves existing nodes.
* Replaced nodes become detached.
* Stored old references can become stale.
* Registries must be updated.
* Whole-component replacement invalidates descendant references.
* Partial replacement preserves more structure.
* State and DOM must remain synchronized.
* Application identity can remain stable while DOM-node identity changes.
* Focus, selection, input state, and scroll may need preservation.
* Delegated listeners survive descendant replacement when the stable parent remains.
* Direct listeners on removed descendants do not transfer to new nodes.
* Replacing the smallest meaningful region is usually best.
* Old nodes should not be retained unnecessarily.

Core model:

```text
Old view
    ↓
Capture required context
    ↓
Create new view
    ↓
Update state
    ↓
Replace old structure
    ↓
Refresh references and registry
    ↓
Restore focus and accessibility context
```
