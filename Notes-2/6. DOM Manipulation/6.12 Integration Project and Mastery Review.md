Knowing these APIs individually is necessary, but professional development requires something more important:

> You must understand how the APIs cooperate inside a complete application.

A real interface rarely performs only one DOM operation.

For example, creating a transaction may require you to:

1. Validate form values.
2. Build a transaction object.
3. Add it to application state.
4. Clone an HTML template.
5. Populate content safely.
6. Assign a unique identity.
7. Insert the new card.
8. Update summary values.
9. update filters.
10. Update accessibility status.
11. Scroll the new card into view.
12. Move keyboard focus.
13. Store the view in a registry.
14. Keep DOM and state synchronized.

This integration lesson builds that complete workflow.

# 3. Why an Integration Project Matters

Isolated examples can hide architectural problems.

This looks simple:

```javascript
card.remove();
```

But in a complete application, deletion also affects:

* State.
* Registries.
* Summary calculations.
* Empty-state rendering.
* Focus.
* Status announcements.
* Undo data.
* Filters.
* Server synchronization.

Similarly:

```javascript
container.innerHTML =
  markup;
```

may appear convenient, but it may:

* Destroy cached references.
* Remove direct listeners.
* Lose focus.
* Reset form values.
* Create injection risks.
* Break view registries.

The integration project forces you to reason about the complete lifecycle.

# 5. Features and Requirements

## Core CRUD requirements

### Create

The user can add a transaction containing:

* Service name.
* Reference number.
* Quantity.
* Unit charge.
* Payment mode.

The application calculates the total amount.

### Read

Transactions appear as reusable cards.

Each card displays:

* Receipt number.
* Service.
* Reference.
* Quantity.
* Unit charge.
* Total amount.
* Payment mode.
* Status.
* Creation date.

### Update

The user can:

* Mark a transaction as paid.
* Mark it as pending.
* Edit its service.
* Edit its reference.
* Edit quantity and unit charge.

### Delete

The user can:

* Soft-delete a transaction.
* Restore a deleted transaction.
* Permanently remove a deleted transaction.

## Supporting requirements

* The DOM must be rendered from JavaScript state.
* Dynamic text must use `textContent`.
* Repeated cards must use `<template>`.
* Every transaction must have a stable ID.
* Every view must use `data-transaction-id`.
* Filter and sorting state must be separate from the DOM.
* Empty state must display when no visible records remain.
* Status changes must be announced.
* Focus must remain predictable.
* No business value should be parsed from displayed text.
* Deleted DOM views must not remain in the registry.

# 13. Template Validation

```javascript
function requireTemplate(
  selector
) {
  const template =
    document.querySelector(
      selector
    );

  if (
    !(template instanceof
      HTMLTemplateElement)
  ) {
    throw new Error(
      `Required template not found: ${selector}`
    );
  }

  return template;
}

const transactionTemplate =
  requireTemplate(
    "#transaction-card-template"
  );
```

Validate required parts.

```javascript
function queryRequired(
  root,
  selector,
  description = selector
) {
  const element =
    root.querySelector(
      selector
    );

  if (!element) {
    throw new Error(
      `Missing required component part: ${description}`
    );
  }

  return element;
}
```

A view object stores:

```javascript
function createTransactionView(
  transaction
) {
  const fragment =
    transactionTemplate

      .cloneNode(true);

      queryRequired(
```

# 15. State-to-View Rendering

```javascript
function renderTransactionIntoView(
  view,
  transaction
) {
  const isPaid =
    transaction.status ===
    "paid";

  const isDeleted =
    transaction.isDeleted;

  view.root.dataset
    .transactionId =
    String(transaction.id);

  view.root.dataset.status =
    transaction.status;

  view.root.dataset.deleted =
    String(isDeleted);

  view.root.classList.toggle(
    "is-paid",
    isPaid
  );

  view.root.classList.toggle(
    "is-pending",
    !isPaid
  );

  view.root.classList.toggle(
    "is-deleted",
    isDeleted
  );

  view.parts.receipt.textContent =
    transaction.receiptNumber;

  view.parts.date.textContent =
    formatDate(
      transaction.createdAt
    );

  view.parts.status.textContent =
    isDeleted
      ? "Deleted"
      : isPaid
        ? "Paid"
        : "Pending";

  view.parts.service.textContent =
    transaction.service;

  view.parts.reference.textContent =
    transaction.reference;

  view.parts.quantity.textContent =
    String(
      transaction.quantity
    );

  view.parts.unitCharge.textContent =
    formatCurrency(
      transaction.unitCharge
    );

  view.parts.amount.textContent =
    formatCurrency(
      transaction.amount
    );

  view.parts.paymentMode.textContent =
    transaction.paymentMode;

  view.parts.togglePaidButton
    .textContent =
    isPaid
      ? "Mark Pending"
      : "Mark Paid";

  view.parts.togglePaidButton
    .disabled =
    isDeleted;

  view.parts.editButton.hidden =
    isDeleted;

  view.parts.softDeleteButton.hidden =
    isDeleted;

  view.parts.restoreButton.hidden =
    !isDeleted;

  view.parts.permanentDeleteButton
    .hidden =
    !isDeleted;

  view.parts.togglePaidButton
    .setAttribute(
      "aria-label",
      isPaid
        ? `Mark ${transaction.receiptNumber} as pending`
        : `Mark ${transaction.receiptNumber} as paid`
    );

  view.parts.editButton
    .setAttribute(
      "aria-label",
      `Edit ${transaction.receiptNumber}`
    );

  view.parts.softDeleteButton
    .setAttribute(
      "aria-label",
      `Delete ${transaction.receiptNumber}`
    );

  view.parts.restoreButton
    .setAttribute(
      "aria-label",
      `Restore ${transaction.receiptNumber}`
    );

  view.parts.permanentDeleteButton
    .setAttribute(
      "aria-label",
      `Permanently delete ${transaction.receiptNumber}`
    );
}
```

# 16. Rendering the Transaction Collection

Create a registry:

```javascript
const transactionViews =
  new Map();
```

Full renderer:

```javascript
function renderTransactions() {
  const visibleTransactions =
    selectVisibleTransactions();

  transactionViews.clear();

  const fragment =
    document
      .createDocumentFragment();

  for (
    const transaction
    of visibleTransactions
  ) {
    const view =
      createTransactionView(
        transaction
      );

    transactionViews.set(
      transaction.id,
      view
    );

    fragment.append(
      view.root
    );
  }

  elements.transactionList
    .replaceChildren(
      fragment
    );

  renderEmptyState(
    visibleTransactions.length ===
      0
  );

  renderSummary();
}
```

# 19. Reading and Displaying Transactions

Selectors derive the list to display.

```javascript
function selectVisibleTransactions() {
  const search =
    state.filters.search
      .trim()
      .toLowerCase();

  const filtered =
    state.transactions
      .filter(
        transaction => {
          if (
            state.filters.status ===
              "active" &&
            transaction.isDeleted
          ) {
            return false;
          }

          if (
            state.filters.status ===
              "deleted" &&
            !transaction.isDeleted
          ) {
            return false;
          }

          if (
            state.filters.status ===
              "paid" &&
            (
              transaction.status !==
                "paid" ||
              transaction.isDeleted
            )
          ) {
            return false;
          }

          if (
            state.filters.status ===
              "pending" &&
            (
              transaction.status !==
                "pending" ||
              transaction.isDeleted
            )
          ) {
            return false;
          }

          if (!search) {
            return true;
          }

          const searchableText =
            [
              transaction.receiptNumber,
              transaction.service,
              transaction.reference,
              transaction.paymentMode,
            ]
              .join(" ")
              .toLowerCase();

          return searchableText
            .includes(search);
        }
      );

  return sortTransactions(
    filtered
  );
}
```

# 20. Updating Transactions

```javascript
function getTransactionById(
  id
) {
  return state.transactions
    .find(
      transaction =>
        transaction.id === id

}
```

Toggle paid state:

```javascript
function toggleTransactionPaid(
  id
) {
  const transaction =
    getTransactionById(id);

  if (
    !transaction ||
    transaction.isDeleted
```

```javascript
function reconcileTransactionAfterChange(
  transaction,
  action
) {
  const remainsVisible =
    selectVisibleTransactions()
      .some(
        visible =>
          visible.id ===
          transaction.id
      );

  if (remainsVisible) {
    const existingView =
      transactionViews.get(
        transaction.id
      );

    if (existingView) {
      existingView.update(
        transaction
      );
    } else {
      renderTransactions();
    }

      .get(transaction.id)
      ?.root
      .remove();

    transactionViews.delete(
      transaction.id

    renderTransactions();
  }

  renderSummary();

  announce(
    `${transaction.receiptNumber} ${action}.`
  );
}
```

# 24. Sorting Transactions

```javascript
function sortTransactions(
  transactions
) {
  const copy =
    [...transactions];

  const multiplier =
    state.sorting.direction ===
      "ascending"
      ? 1
      : -1;

  copy.sort(
    (
      first,
      second
    ) => {
      switch (
        state.sorting.field
      ) {
        case "amount":
          return (
            first.amount -
            second.amount
          ) * multiplier;

        case "service":
          return first.service
            .localeCompare(
              second.service
            ) * multiplier;

        case "receipt":
          return first
            .receiptNumber
            .localeCompare(
              second.receiptNumber
            ) * multiplier;

        case "createdAt":
        default:
          return (
            first.createdAt -
            second.createdAt
          ) * multiplier;
      }
    }
  );

  return copy;
}
```

Never mutate the state array merely to obtain a sorted view unless that order change is intentional business state.

# 26. Derived Summary Values

```javascript
function calculateSummary() {
  const active =
    state.transactions.filter(
      transaction =>
        !transaction.isDeleted
    );

  return active.reduce(
    (
      summary,
      transaction
    ) => {
      summary.count += 1;

      summary.total +=
        transaction.amount;

      if (
        transaction.status ===
        "paid"

          transaction.amount;

          transaction.amount;

      return summary;

      pending: 0,
    }
  );
}
```

# 29. Targeted Updates

Use a targeted update when:

* One transaction changed.
* The record remains visible.
* Its sort position does not change.
* Its filter eligibility does not change.
* Structure remains stable.

```javascript
function updateTransactionView(
  transaction
) {
  const view =
    transactionViews.get(
      transaction.id
    );

  if (!view) {
    renderTransactions();
    return;
  }

  view.update(transaction);
}
```

# 30. Full Rerendering

Use full rerendering when:

* Filter changes.
* Search changes.
* Sort order changes.
* Many records change.
* A record enters or leaves the visible set.
* Simplicity is more important than preserving local state.

```javascript
renderTransactions();
```

Full rerendering recreates views and clears old references.

---

# 31. Choosing Targeted Updates or Rerendering

```text
Did one visible record change?
    ↓
Does its position and visibility stay the same?
    ├── Yes → targeted update
    └── No → rerender collection
```

Targeted updates preserve:

* Focus.
* Local DOM identity.
* Input state.
* Scroll stability.

Full rerendering provides:

* Simpler correctness.
* Correct filter and sorting order.
* Fresh view structure.

# 38. Validation Architecture

```javascript
function validateTransactionInput(
  input
) {
  const errors = {};

  if (!input.service.trim()) {
    errors.service =
      "Service is required.";
  }

  if (!input.reference.trim()) {
    errors.reference =
      "Reference is required.";
  }

  const quantity =
    Number(input.quantity);

  if (
    !Number.isInteger(quantity) ||
    quantity <= 0
  ) {
    errors.quantity =
      "Quantity must be a positive whole number.";
  }

  const unitCharge =
    Number(input.unitCharge);

  if (
    !Number.isFinite(
      unitCharge
    ) ||
    unitCharge < 0
  ) {
    errors.unitCharge =

  }

  return {
    valid:
      Object.keys(errors)
        .length === 0,

    errors,
  };
}
```

Display errors in dedicated regions.

Do not use alerts as the primary validation architecture.

# 39. Safe Content Rendering

Always use:

```javascript
element.textContent =
  dynamicValue;
```

Avoid:

```javascript
element.innerHTML =
  dynamicValue;
```

A service name such as:

```text
<img src=x onerror=alert(1)>
```

must appear as text, not markup.

Templates provide trusted structure.

State values provide text.

# 40. Security Review

The project should follow these rules:

1. Dynamic values use `textContent`.
2. Record IDs are validated after reading from dataset.
3. Action names are handled through an allowlist.
4. Business authorization is not based on hidden buttons.
5. No authentication data is stored in dataset.
6. Form values are validated.
7. User-provided CSS is not assigned.
8. User-provided HTML is not parsed.
9. Destructive actions require appropriate confirmation in production.
10. Server-side validation remains mandatory when connected to an API.

---

# 41. Performance Review

The project uses:

* Cached stable DOM references.
* Templates for repeated structures.
* `DocumentFragment` for collection rendering.
* Targeted updates when possible.
* Full rerendering when order or visibility changes.
* No `innerHTML +=`.
* No repeated business-data parsing from the DOM.
* Controlled scroll and geometry operations.
* A view registry for direct updates.

Potential future improvements:

* Pagination.
* Virtualization.
* Debounced search.
* Batched server synchronization.
* Incremental sorting.
* Observer-based loading.

# 42. Memory and Lifecycle Review

Potential retention sources:

* The view registry.
* Timers used for highlights.
* Observers.
* Pending API requests.
* Global listeners.
* Removed cards stored for undo.

Cleanup principles:

```text
Remove state record
    ↓
Delete registry entry
    ↓
Destroy external resources
    ↓
Remove DOM root
    ↓
Release remaining references
```

Do not retain removed views unnecessarily.

```javascript
  transactions

    [...transactions];

  const multiplier =
    state.sorting.direction ===
      "ascending"
      ? 1
      : -1;

  copy.sort(
    (
      first,
      second

      switch (
        state.sorting.field

        case "amount":
          return (
            first.amount -
            second.amount

        case "service":
          return first.service
            .localeCompare(
              second.service

        case "receipt":
          return first
            .receiptNumber

              second.receiptNumber
            ) * multiplier;

        case "createdAt":

            first.createdAt -

              break;

            case "pending":
              if (

            case "deleted":
              if (

            case "all":
              break;

            default:
              throw new Error(

          if (!search) {
            return true;

          const searchableText =
            [
              transaction
                .receiptNumber,

              transaction.service,

              transaction.reference,

              transaction
                .paymentMode,

              .join(" ")
              .toLowerCase();

          return searchableText
            .includes(search);

          break;

        default:
          throw new Error(
            "Unsupported transaction action."
          );
      }
    }
  );

renderTransactions();
```

# 53. Interview Questions and Answers

## 1. Why should application state be separate from the DOM?

The DOM is a rendered view that can be replaced, filtered, or formatted. State preserves authoritative business data independently of presentation.

## 2. Why use stable IDs rather than array indexes?

Indexes change during sorting, filtering, insertion, and deletion. Stable IDs preserve record identity.

## 3. Why use an HTML template?

It provides clean inert reusable structure without copying runtime state from a live component.

## 4. What is a view registry?

A map from application identity to the current connected DOM view or view object.

## 5. When should you use a targeted update?

When one visible component changes without changing its position or filter eligibility.

## 6. When should you rerender the collection?

When searching, filtering, sorting, or structural changes affect multiple records or order.

## 7. Why is `textContent` important?

It renders dynamic values as text rather than parsing them as HTML, reducing injection risk.

## 8. Why use event delegation?

One stable ancestor can handle actions from existing and dynamically generated children.

## 9. How do you prevent stale view references?

Update or clear registries when nodes are replaced or removed, and verify `isConnected`.

## 10. How do you preserve focus after deletion?

Capture a meaningful next, previous, or container target before removing the focused component, then focus that target afterward.

## 11. Why should classes not be the source of business state?

Classes represent presentation and can change independently. JavaScript state should remain authoritative.

## 12. How would you improve performance for thousands of records?

Use pagination or virtualization, reduce full rerenders, batch changes, and profile actual bottlenecks.

# 55. Topic Summary

DOM Manipulation is the process of constructing, changing, measuring, and removing browser document structures.

You learned how to:

```text
Create nodes
    ↓
Populate safe content
    ↓
Insert nodes
    ↓
Assign identity and state
    ↓
Update classes and properties
    ↓
Replace outdated structures
    ↓
Remove obsolete structures
    ↓
Measure rendered geometry
    ↓
Scroll and position components
    ↓
Clean resources
```

The most important architectural principle is:

> Application state should remain authoritative, while the DOM is a synchronized view of that state.

The professional state-to-view cycle is:

```text
User or system change
    ↓
Validate input
    ↓
Update application state
    ↓
Derive visible state
    ↓
Create, update, replace, or remove views
    ↓
Synchronize properties, attributes, classes, and text
    ↓
Restore focus and announce changes
    ↓
Verify registry and lifecycle cleanup
```
