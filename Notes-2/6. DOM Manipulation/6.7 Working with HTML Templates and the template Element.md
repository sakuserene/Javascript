# 1. Introduction

In the previous lesson, you learned how to clone DOM nodes.

```javascript
const clone =
  source.cloneNode(true);
```

Cloning is useful, but cloning an arbitrary live element can copy unwanted state:

* Existing record IDs.
* Validation errors.
* Hidden states.
* Disabled controls.
* User-entered values.
* Expanded menus.
* Duplicate IDs.
* Old ARIA relationships.
* Temporary visual classes.

A better source for repeated components is a clean blueprint.

HTML provides a native element designed for exactly this purpose:

```html
<template>
```

A template stores HTML structure that:

* Is parsed by the browser.
* Is not displayed.
* Does not participate in the active page layout.
* Can be cloned repeatedly.
* Can be populated from JavaScript data.
* Can produce clean component instances.

Example:

```html
<template id="service-card-template">
  <article class="service-card">
    <h3 data-role="name"></h3>
    <p data-role="price"></p>

    <button
      type="button"
      data-action="select-service"
    >
      Select
    </button>
  </article>
</template>
```

JavaScript can clone the blueprint:

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

Then populate the clone:

```javascript
fragment.querySelector(
  "[data-role='name']"
).textContent =
  service.name;
```

Finally, insert the instance:

```javascript
container.append(fragment);
```

---

# 2. What Is the `<template>` Element?

> The `<template>` element stores HTML markup as an inert, reusable blueprint that JavaScript can clone and insert later.

Example:

```html
<template id="message-template">
  <article class="message">
    <h3 data-role="title"></h3>
    <p data-role="body"></p>
  </article>
</template>
```

The `<article>` does not appear on the page merely because it is inside the template.

The browser stores it separately in:

```javascript
template.content
```

The content can be cloned to create actual rendered instances.

---

# 3. Why HTML Templates Exist

Modern interfaces frequently repeat complex markup.

Examples:

* Transaction cards.
* Product cards.
* Table rows.
* Notifications.
* Form fields.
* Comments.
* Modal panels.
* Search results.
* Dashboard widgets.
* Empty and loading states.

Without templates, developers may:

* Rebuild every node using `createElement()`.
* Clone a live component with stale state.
* Build large HTML strings.
* Duplicate markup across JavaScript files.
* Mix view structure and business logic.

Templates provide a clean separation:

```text
HTML:
Structure

JavaScript:
Data and behavior

CSS:
Presentation
```

---

# 4. The Problem Templates Solve

Consider a transaction card with:

* Header.
* Receipt.
* Service.
* Amount.
* Status.
* Actions.
* Accessibility attributes.
* Nested dropdown.

Building every element manually can become verbose.

```javascript
const article =
  document.createElement(
    "article"
  );

const header =
  document.createElement(
    "header"
  );

const heading =
  document.createElement(
    "h3"
  );

// Many more elements...
```

Using an HTML string may be shorter, but dynamic values must be handled safely.

Using a template allows the structure to remain readable in HTML:

```html
<template id="transaction-template">
  <article
    class="transaction-card"
    data-component="transaction-card"
  >
    <header>
      <h3 data-role="receipt"></h3>
    </header>

    <section>
      <p data-role="service"></p>
      <p data-role="amount"></p>
      <p data-role="status"></p>
    </section>

    <footer>
      <button
        type="button"
        data-action="mark-paid"
      >
        Mark Paid
      </button>

      <button
        type="button"
        data-action="delete"
      >
        Delete
      </button>
    </footer>
  </article>
</template>
```

JavaScript only supplies data and state.

# 5. Mental Model: Blueprint vs Instance

Think of the template as an architectural drawing.

```text
Template
    ↓ clone
Component instance 1

Template
    ↓ clone
Component instance 2

Template
    ↓ clone
Component instance 3
```

The template remains unchanged.

Each clone is independent.

```javascript
const first =
  template.content
    .cloneNode(true);

const second =
  template.content
    .cloneNode(true);
```

The cloned descendant nodes are different objects.

# 6. Basic Template Syntax

```html
    <p data-role="position"></p>
```

Important parts:

## `<template>`


# 7. The `HTMLTemplateElement` Interface

Selecting a template:

```javascript
const template =
  document.querySelector(
    "#user-card-template"
  );
```

The object is generally an:

```text
HTMLTemplateElement
```

Verify:

```javascript
console.log(
  template instanceof
    HTMLTemplateElement
);
```

Output:

```text
true
```

It inherits from:

```text
Node
    ↓
Element
    ↓
HTMLElement
    ↓
HTMLTemplateElement
```

Its most important property is:

```javascript
template.content
```

# 8. `template.content`

`template.content` returns a `DocumentFragment`.

```javascript
const content =
  template.content;
```

Verify:

```javascript
console.log(
  content instanceof
    DocumentFragment
);
```

Output:

```text
true
```

The fragment contains the template's parsed child nodes.

```javascript
console.log(
  content.childNodes
);
```

The content is separate from the template element itself.

---

# 9. Why Template Content Is Inert

Template content is inert.

Inert means it does not behave as active page content while stored in the template.

For example:

* Images are not displayed as active page images.
* Forms are not submitted from template contents.
* Buttons are not user-interactive.
* The content does not affect layout.
* The content is not part of the rendered accessibility tree.
* Scripts inside templates do not execute merely because the template is parsed.

The content becomes active after it is cloned and inserted into the live DOM.

---

# 10. Template Content Is Not Rendered

HTML:

```html
<template id="notice-template">
  <p>
    Important notice
  </p>
</template>
```

The paragraph is not displayed.

Selecting:

```javascript
document.querySelector(
  "#notice-template p"
);
```

may not behave like querying ordinary rendered descendants because the actual template nodes are held inside the template's `content` fragment.

Use:

```javascript
template.content
  .querySelector("p");
```

The template element and its content are separate structural contexts.

---

# 11. Templates and `DocumentFragment`

A `DocumentFragment` is a lightweight container for detached nodes.

The template content is stored as a fragment:

```javascript
template.content;
```

When cloned deeply:

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

the result is another fragment containing cloned template nodes.

When appended:

```javascript
container.append(fragment);
```

the fragment's children move into the container.

The fragment itself does not become visible markup.

---

# 12. Cloning Template Content

Typical syntax:

```javascript
const clone =
  template.content
    .cloneNode(true);
```

Use `true` because you usually want the complete template subtree.

The clone begins detached:

```javascript
console.log(
  clone.isConnected
);
```

Output:

```text
false
```

The cloned fragment can be:

* Queried.
* Populated.
* Validated.
* Appended.
* Returned from a factory.

---

# 13. Shallow vs Deep Template Cloning

Shallow:

```javascript
const clone =
  template.content
    .cloneNode(false);
```

This creates an empty `DocumentFragment`.

The descendants are not copied.

Deep:

```javascript
const clone =
  template.content
    .cloneNode(true);
```

This copies the entire template structure.

For normal template instantiation, deep cloning is almost always required.

---

# 14. Finding the Cloned Root

Suppose the template contains one root element.

```html
<template id="card-template">
  <article class="card"></article>
</template>
```

After cloning:

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

Find the root:

```javascript
const root =
  fragment.firstElementChild;
```

Or:

```javascript
const root =
  fragment.querySelector(
    ".card"
  );
```

`firstElementChild` is appropriate when a single-root contract is enforced.

Querying by component selector can be clearer:

```javascript
const root =
  fragment.querySelector(
    "[data-component='transaction-card']"
  );
```

# 16. Template Instance Factories

A factory should contain the complete instantiation workflow.

```javascript
function createServiceCard(
  service
) {
  const fragment =
    serviceTemplate.content
      .cloneNode(true);

  const card =
    fragment.querySelector(
      ".service-card"
    );

  card.dataset.serviceId =
    String(service.id);

  card.querySelector(
    "[data-role='name']"
  ).textContent =
    service.name;

  card.querySelector(
    "[data-role='price']"
  ).textContent =
    `KES ${service.price}`;

  return card;
}
```

Important detail:

The factory returns `card`, not the fragment.

When an element is removed from a fragment and later appended, it can be inserted normally.

Alternatively, return the complete fragment when multiple roots are intentional.

---

# 17. Returning a Root Element

Single-root component factory:

```javascript
function instantiateTemplate(
  template
) {
  const fragment =
    template.content
      .cloneNode(true);

  const root =
    fragment
      .firstElementChild;

  if (!root) {
    throw new Error(
      "Template did not contain a root element."
    );
  }

  return root;
}
```

Advantages:

* Clear component root.
* Easy registry integration.
* Easy insertion.
* Easy testing.
* Predictable ownership.

---

# 18. Returning a View Object

For repeated updates:

```javascript
function createServiceCardView(
  service
) {
  const fragment =
    serviceTemplate.content
      .cloneNode(true);

  const root =
    fragment.querySelector(
      "[data-component='service-card']"
    );

  const name =
    root.querySelector(
      "[data-role='name']"
    );

  const price =
    root.querySelector(
      "[data-role='price']"
    );

  root.dataset.serviceId =
    String(service.id);

  name.textContent =
    service.name;

  price.textContent =
    `KES ${service.price}`;

  return {
    root,
    name,
    price,

    setPrice(value) {
      price.textContent =
        `KES ${value}`;
    },
  };
}
```

A view object exposes stable internal references and meaningful operations.

# 21. Rendering with `DocumentFragment`

```javascript
function renderServices(
  services,
  container
) {
  const batch =
    document
      .createDocumentFragment();

  for (
    const service
    of services
  ) {
    batch.append(
      createServiceCard(
        service
      )
    );
  }

  container.append(batch);
}
```

Benefits:

* Builds the group while detached.
* Makes batching explicit.
* Inserts a completed group.
* Keeps rendering logic organized.

# 22. Rendering from Application State

State:

```javascript
const state = {
  services: [
    {
      id: 1,
      name: "Printing",
      price: 20,
    },
    {
      id: 2,
      name: "Binding",
      price: 100,
    },
  ],
};
```

Render:

```javascript
function render() {
  const cards =
    state.services.map(
      createServiceCard
    );

  serviceList.replaceChildren(
    ...cards
  );
}
```

State remains authoritative.

Templates define structure.

Factories connect state to structure.

---

# 23. Append-Only Template Rendering

When one record is added:

```javascript
function addService(
  service
) {
  state.services.push(
    service
  );

  serviceList.append(
    createServiceCard(
      service
    )
  );
}
```

This avoids recreating existing components.

Use append-only rendering when:

* One record is added.
* Existing records do not need updates.
* New records belong at the end.
* State order and DOM order match.

---

# 24. Full Template Rerendering

```javascript
function renderServices() {
  const batch =
    document
      .createDocumentFragment();

  for (
    const service
    of state.services
  ) {
    batch.append(
      createServiceCard(
        service
      )
    );
  }

  serviceList.replaceChildren(
    batch
  );
}
```

Use full rerendering when:

* The list is small.
* Filters or sorting changed many items.
* Simplicity is preferred.
* Focus and live state are not endangered.
* Old component resources are cleaned.

Do not full-rerender active complex forms unnecessarily.

---

# 25. Identity-Based Rendering

Every repeated instance should carry stable application identity.

```javascript
card.dataset.serviceId =
  String(service.id);
```

Identity supports:

* Event handling.
* State lookup.
* Updating.
* Deleting.
* Registry mapping.
* Debugging.

Do not use the visible index as permanent identity.

```javascript
data-service-id="42"
```

is more stable than:

```javascript
data-index="3"
```

after sorting or filtering.

# 31. Templates for Modals

```html
<template id="modal-template">
  <div

    data-component="modal"

      class="modal"

```

Each instance may require:

* Unique heading ID.
* `aria-labelledby`.
* Focus initialization.
* Focus trapping.
* Opener restoration.
* Teardown.

Template cloning provides structure, not lifecycle management.

---

# 32. Nested Templates

A page can use separate templates for nested component types.

Example:

```text
Transaction list template
    ↓ contains card container

Transaction card template
    ↓ contains action region

Dropdown template
    ↓ inserted into card
```

This encourages component boundaries.

Avoid putting every possible nested structure into one enormous template.

Smaller templates can be composed.

---

# 33. Templates Inside Templates

A `<template>` may contain another `<template>`.

Example:

```html
<template id="form-template">
  <form>
    <div data-role="fields"></div>

    <template data-template="field">
      <div data-field></div>
    </template>
  </form>
</template>
```

Nested templates remain inert.

However, this can complicate discovery and cloning.

A separate top-level template with a stable ID is often easier to maintain and reuse.

---

# 34. Multiple Root Elements

A template may contain multiple root elements.

```html
<template id="message-template">
  <h2>Notice</h2>
  <p data-role="message"></p>
</template>
```

Clone:

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

The fragment has multiple children.

This is valid.

But a component factory returning one root becomes more difficult.

Use multiple roots when the output is intentionally a fragment.

---

# 35. Single-Root Template Contracts

For component templates, a single root is often preferable.

```html
<template id="message-template">
  <section data-component="message">
    <h2>Notice</h2>
    <p data-role="message"></p>
  </section>
</template>
```

Benefits:

* Clear ownership.
* Easy insertion and removal.
* Registry compatibility.
* Easier focus handling.
* Easier replacement.
* Easier event delegation.
* Predictable styling.

Validate the contract:

```javascript
if (
  fragment.children.length !==
    1
) {
  throw new Error(
    "Component template must contain exactly one root element."
  );
}
```

# 36. Template Validation

Templates are structural contracts.

Validate them during development.

```javascript
function requireTemplate(
  selector
) {
  const template =
    document.querySelector(
      selector
    );

  if (
    !(template instanceof
      HTMLTemplateElement)
  ) {
    throw new Error(
      `Template was not found: ${selector}`
    );
  }

  return template;
}
```

Use:

```javascript
const transactionTemplate =
  requireTemplate(
    "#transaction-template"
  );
```

# 37. Required Template Parts

```javascript
function queryRequired(
  root,
  selector,
  description = selector
) {
  const element =
    root.querySelector(
      selector
    );

  if (!element) {
    throw new Error(
      `Missing required template part: ${description}`
    );
  }

  return element;
}
```

Use during instantiation:

```javascript
const receipt =
  queryRequired(
    root,
    "[data-role='receipt']",
    "transaction receipt"
  );
```

If the template changes unexpectedly, the failure occurs immediately.

---

# 38. Optional Template Parts

Optional parts may be absent.

```javascript
const description =
  root.querySelector(
    "[data-role='description']"
  );
```

Use:

```javascript
if (description) {
  description.textContent =
    service.description;
}
```

Distinguish:

```text
Required structure:
Missing means invalid component.

Optional structure:
Missing means feature is absent.
```

# 39. Unique IDs in Template Instances

Avoid hard-coded IDs inside repeated templates.

Weak:

```html
<input id="phone-input">
```

Every clone receives the same ID.

Better blueprint:

```html
<input data-role="input">
```

Factory:

```javascript
const inputId =
  `phone-${fieldId}`;

input.id =
  inputId;

label.htmlFor =
  inputId;
```

IDs should be assigned per instance.

# 40. Labels and ARIA Relationships

Template:

```html
<label data-role="label"></label>
<input data-role="input">
<p data-role="error" hidden></p>
```

Factory:

```javascript
const inputId =
  `email-${user.id}`;

const errorId =
  `${inputId}-error`;

label.htmlFor =
  inputId;

input.id =
  inputId;

error.id =
  errorId;
```

When invalid:

```javascript
input.setAttribute(
  "aria-invalid",
  "true"
);

input.setAttribute(
  "aria-describedby",
  errorId
);
```

Every cloned relationship must be instance-specific.

# 41. Resetting Runtime State

A clean template should generally have neutral state.

Example:

```html
<input
  data-role="input"
  value=""
>
```

```html
<p
  data-role="error"
  hidden
></p>
```

```html
<button
  type="button"
  data-action="submit"
>
  Submit
</button>
```

Factories should still set current state explicitly:

```javascript
input.value =
  data.value ?? "";

error.hidden =
  true;

button.disabled =
  false;
```

Explicit state is safer than relying on template defaults alone.

# 42. Templates and Event Delegation

Template-generated controls commonly use:

```html
data-action="delete"
```

A stable container listener:

```javascript
container.addEventListener(
  "click",
  handleClick
);
```

can process every generated instance.

```javascript
function handleClick(
  event
) {
  if (
    !(event.target instanceof
      Element)
  ) {
    return;
  }

  const action =
    event.target.closest(
      "[data-action]"
    );

  if (
    !action ||
    !container.contains(action)
  ) {
    return;
  }

  // Resolve component and route action.
}
```

This avoids attaching direct listeners to every clone.

# 43. Templates and Direct Event Listeners

Templates do not contain listeners registered through JavaScript.

After cloning, initialize behavior:

```javascript
function initializeModal(
  root
) {
  const closeButton =
    root.querySelector(
      "[data-action='close-modal']"
    );

  closeButton.addEventListener(
    "click",
    () => {
      root.remove();
    }
  );
}
```

Call:

```javascript
const modal =
  createModal();

initializeModal(modal);
```

For repeated simple controls, delegation is usually easier.

For isolated components such as modals, direct lifecycle-controlled listeners may be appropriate.

# 44. Templates and Component Registries

Each template instance may become a view:

```javascript
const view =
  createTransactionView(
    transaction
  );
```

Register:

```javascript
registry.set(
  transaction.id,
  view
);
```

The registry maps application identity to the current connected view.

When removing or replacing an instance:

```javascript
registry.delete(
  transaction.id
);
```

Do not register the template blueprint itself.

Register each created instance.

# 45. Templates vs `createElement()`

## Templates

Best when:

* Structure is large.
* Designers work mainly in HTML.
* Markup is stable.
* Semantic layout should remain readable.
* Repeated components share the same skeleton.

## `createElement()`

Best when:

* Structure is small.
* Elements are generated algorithmically.
* The tag type is dynamic.
* Fine-grained construction logic is important.
* A reusable template would add complexity.

These approaches can be combined.

A template can provide the shell while JavaScript creates dynamic substructures.

---

# 46. Templates vs Cloning Live Elements

## Cloning a live element

May copy:

* Current values.
* Validation state.
* Expanded state.
* Loading state.
* Record ID.
* Temporary classes.

## Cloning a template

Starts from a clean, controlled blueprint.

Professional preference:

> Clone templates for new instances. Clone live elements only when copying current state is intentional.

# 47. Templates vs HTML Strings

HTML string:

```javascript
container.insertAdjacentHTML(
  "beforeend",
  `
    <article>
      ${userInput}
    </article>
  `
);
```

This can create security risks when interpolating untrusted values.

Template approach:

```javascript
const root =
  createArticleFromTemplate();

root.querySelector(
  "[data-role='message']"
).textContent =
  userInput;
```

Benefits:

* Safe text assignment.
* Direct references.
* Structural validation.
* No repeated string parsing logic in application code.
* Better component contracts.

# 48. Templates vs Frontend Framework Components

Templates provide:

* Static markup blueprints.
* Cloning.
* Manual population.
* Manual insertion.
* Manual updates.
* Manual cleanup.

Frameworks add:

* Declarative state rendering.
* Component lifecycle systems.
* Automatic update scheduling.
* Reconciliation or compilation.
* Reactive dependencies.
* Built-in composition models.

Understanding templates prepares you to understand framework component rendering.

# 49. Security Considerations

Templates do not automatically sanitize data.

Safe:

```javascript
name.textContent =
  service.name;
```

Unsafe:

```javascript
name.innerHTML =
  service.name;
```

unless the source is trusted and properly sanitized.

Additional concerns:

* Unsafe `href` values.
* Unsafe image URLs.
* Copied inline event attributes.
* Dangerous template markup.
* Overly powerful `data-action` values.
* Server-rendered untrusted template contents.

Treat template structure as trusted application code.

Treat dynamic data as untrusted unless proven otherwise.

---

# 50. Accessibility Considerations

Template instances must have valid:

* Heading order.
* Button labels.
* Form labels.
* Unique IDs.
* ARIA relationships.
* Focus behavior.
* Keyboard order.
* Modal roles.
* Live-region semantics.

A correct template can improve consistency because accessibility structure is defined once.

A broken template repeats the same accessibility defect across every instance.

Test the template thoroughly.

---

# 51. Performance Considerations

Template cloning is generally efficient for reusable structures.

Main costs include:

* Deep cloning.
* Querying cloned descendants.
* Populating values.
* Creating thousands of instances.
* Inserting large batches.
* Full rerendering.
* Registry creation.

Optimization strategies:

1. Cache the template element.
2. Clone while detached.
3. Query only required parts.
4. Use a fragment for batches.
5. Append one item for single additions.
6. Avoid full rerenders when targeted updates are sufficient.
7. Reduce DOM size with pagination or virtualization.
8. Measure real application workflows.

---

# 52. Memory Considerations

Each clone creates new DOM objects.

Memory can grow when:

* Detached instances remain in arrays.
* Registries retain removed views.
* Direct listeners retain controllers.
* Modal instances are not destroyed.
* Large batches are created but never inserted.
* Undo history stores complete DOM trees.

Keep templates themselves stable.

Release unused instances and controllers.

Prefer storing application data for long-lived history.

# 53. Three Progressive Examples

## Example 1: Beginner — Render a Notification

### HTML

```html
<section id="notifications"></section>

<template id="notification-template">
  <article
    class="notification"
    data-component="notification"
  >
    <h2 data-role="title"></h2>
    <p data-role="message"></p>

    <button
      type="button"
      data-action="dismiss"
    >
      Dismiss
    </button>
  </article>
</template>
```

### JavaScript

```javascript
const notificationTemplate =
  document.querySelector(
    "#notification-template"
  );

const notifications =
  document.querySelector(
    "#notifications"
  );

function createNotification(
  notification
) {
  const fragment =
    notificationTemplate
      .content
      .cloneNode(true);

  const root =
    fragment.firstElementChild;

  root.dataset.notificationId =
    String(notification.id);

  root.querySelector(
    "[data-role='title']"
  ).textContent =
    notification.title;

  root.querySelector(
    "[data-role='message']"
  ).textContent =
    notification.message;

  return root;
}
```

### Insert

```javascript
const notification =
  createNotification({
    id: 1,
    title: "Report Ready",
    message:
      "Your monthly report is ready.",
  });

notifications.append(
  notification
);
```

The template defines structure once, while the factory supplies instance-specific identity and content.

# 54. Common Mistakes

## Mistake 1: Querying template descendants from the template element

Use:

```javascript
template.content
```

rather than treating template content like ordinary rendered children.

---

## Mistake 2: Forgetting deep cloning

```javascript
template.content
  .cloneNode(false);
```

creates an empty fragment.

---

## Mistake 3: Appending `template.content` directly

```javascript
container.append(
  template.content
);
```

This moves the blueprint nodes out of the template.

Future clones become empty.

Always clone:

```javascript
container.append(
  template.content
    .cloneNode(true)
);
```

---

## Mistake 4: Mutating the blueprint instead of the clone

```javascript
template.content
  .querySelector(
    "[data-role='name']"
  )
  .textContent =
  service.name;
```

This changes the reusable blueprint.

Populate the cloned fragment instead.

---

## Mistake 5: Hard-coding duplicate IDs

Every template instance receives the same ID.

Assign IDs in the factory.

---

## Mistake 6: Assuming templates add event listeners

Template cloning copies markup, not listeners registered through JavaScript.

---

## Mistake 7: Returning a node still owned by a fragment without understanding insertion

Returning the root is valid, but ensure the component contract is clear and unused sibling roots are not accidentally discarded.

---

## Mistake 8: Multiple roots when one component root is expected

This complicates ownership, removal, registries, and focus management.

---

## Mistake 9: Using `innerHTML` for untrusted instance values

Use `textContent`.

---

## Mistake 10: Full rerendering complex active forms

This can lose focus, values, selection, and browser-managed state.

---

## Mistake 11: Forgetting registry cleanup before rerender

Old view objects may retain detached roots.

---

## Mistake 12: Treating template instances as application state

The DOM represents state; it does not replace the state store.

---

# 55. Professional Best Practices

1. Cache template references during initialization.
2. Verify selected nodes are `HTMLTemplateElement` instances.
3. Clone `template.content`, never move it directly.
4. Use deep cloning for complete instances.
5. Prefer a single semantic root for component templates.
6. Use `data-component`, `data-role`, and `data-action`.
7. Populate dynamic values with `textContent`.
8. Assign unique IDs during instantiation.
9. Build label and ARIA relationships per instance.
10. Validate required template parts.
11. Distinguish required and optional regions.
12. Keep templates in a neutral state.
13. Populate from authoritative application state.
14. Return a stable root or view object.
15. Use event delegation for repeated template-generated actions.
16. Initialize direct behavior through a consistent lifecycle function.
17. Register every component instance independently.
18. Clean old views before complete rerendering.
19. Batch large groups with `DocumentFragment`.
20. Prefer targeted updates when only a small value changes.
21. Avoid enormous all-purpose templates.
22. Compose smaller templates for nested components.
23. Test accessibility once at the blueprint level and again with multiple instances.
24. Never treat templates as automatic sanitization.
25. Profile large rendering workflows.

```javascript
console.assert(
```

```javascript
console.log(
```

```javascript
console.log(
```

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

```javascript
console.assert(
  fragment.children.length ===
```

```javascript
console.log(
  template.content
    .querySelector(
```

Blueprint content should normally remain empty or contain defaults.

If:

```javascript
template.content
```

after rendering, you may have appended `template.content` directly instead of cloning it.

## Detect duplicate IDs

```javascript
const ids =

      element.id
  );

const duplicates =
  ids.filter(
    (
      id,
      index
    ) =>
      ids.indexOf(id) !==
      index
  );

console.table(
  [
    ...new Set(
      duplicates
    ),
  ]
);
```

# 57. Interview Questions and Answers

## Beginner Questions

### 1. What is the `<template>` element?

It stores inert HTML markup that can be cloned and inserted later.

### 2. Is template content displayed automatically?

No.

### 3. Where is a template's markup stored?

In:

```javascript
template.content
```

### 4. What type is `template.content`?

A `DocumentFragment`.

### 5. How do you create an instance?

```javascript
template.content
  .cloneNode(true);
```

### 6. Why use `true`?

To clone the template's descendants.

### 7. Does the clone begin connected?

No.

### 8. Does appending a fragment insert the fragment element?

No. Its children are inserted.

---

## Intermediate Questions

### 1. Why should `template.content` not be appended directly?

Its blueprint nodes would move out of the template, leaving it empty.

### 2. Why are templates safer than cloning live components?

They provide a clean neutral blueprint without temporary runtime state.

### 3. Why assign IDs during instantiation?

Hard-coded template IDs are duplicated across instances.

### 4. How do templates work with event delegation?

Generated controls carry semantic action attributes, and one stable ancestor listener handles their events.

### 5. When should a factory return a view object?

When later updates need direct internal references and component operations.

### 6. Why validate required template parts?

Template changes should fail early rather than causing distant null-reference bugs.

### 7. What is the benefit of a single root element?

It provides clear ownership, insertion, removal, registry, and focus boundaries.

---

## Advanced Questions

### 1. Compare templates and `createElement()`.

Templates keep large stable markup readable in HTML; `createElement()` provides explicit algorithmic construction and is ideal for small or highly dynamic structures.

### 2. Compare templates with HTML strings.

Templates are parsed once as structured markup and can be safely populated through DOM properties. HTML strings require parsing and become security-sensitive when interpolating untrusted values.

### 3. How would you render thousands of template instances?

Cache the template, clone and populate while detached, batch through fragments, limit the visible DOM through pagination or virtualization, and profile realistic workflows.

### 4. How should complete rerendering clean old component views?

Destroy external resources, clear registries, replace container children, and register the new connected views.

### 5. How do templates relate to framework components?

They provide reusable structural blueprints, but state updates, lifecycle, reactivity, and reconciliation remain manual.

### 6. How would you validate a template contract?

Verify the template type, root count, root selector, required roles, action controls, duplicate IDs, semantic hierarchy, and accessibility relationships.

# 61. Summary

The `<template>` element stores reusable inert HTML.

Example:

```html
<template id="card-template">
  <article class="card"></article>
</template>
```

Its content is accessed through:

```javascript
template.content
```

The content is a:

```text
DocumentFragment
```

Create an instance with:

```javascript
const fragment =
  template.content
    .cloneNode(true);
```

You learned that:

* Template content is not rendered automatically.
* Template content is inert until inserted.
* The blueprint should remain unchanged.
* `template.content` must be cloned rather than moved.
* Deep cloning is normally required.
* Component templates benefit from a single semantic root.
* Dynamic values should be assigned to the clone.
* `textContent` should be used for untrusted text.
* IDs, labels, and ARIA references must be unique per instance.
* Templates support cards, rows, forms, modals, lists, and UI states.
* Event delegation works well with template-generated controls.
* Direct listeners require explicit initialization.
* Every component instance needs its own view and registry entry.
* Application state remains authoritative.
* Fragments support batch rendering.
* Full rerendering can disrupt focus and live state.
* Templates provide structure but not automatic reactivity or lifecycle management.

The professional template pipeline is:

```text
Select and validate template
    ↓
Clone template.content deeply
    ↓
Validate root and required parts
    ↓
Assign unique application identity
    ↓
Populate safe content from state
    ↓
Create accessibility relationships
    ↓
Return root or view
    ↓
Initialize behavior
    ↓
Register
    ↓
Insert
```
