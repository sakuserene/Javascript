# 1. Introduction

JavaScript can change an element's:

* Content.
* Attributes.
* Properties.
* Classes.
* Position in the DOM.
* Presentation.

Consider:

```html
<div
  id="progress-bar"
  class="progress-bar"
></div>
```

JavaScript may need to display progress dynamically:

```javascript
progressBar.style.width =
  "75%";
```

The resulting markup contains an inline declaration:

```html
<div
  id="progress-bar"
  class="progress-bar"
  style="width: 75%;"
></div>
```

JavaScript can also inspect the final style after CSS rules, inheritance, and the cascade have been applied:

```javascript
const styles =
  getComputedStyle(
    progressBar
  );

console.log(
  styles.width
);
```

This lesson distinguishes two important systems:

```text
element.style
    ↓
Inline declarations only

getComputedStyle(element)
    ↓
Final resolved styles
```

It also explains when direct style manipulation is appropriate and when classes or CSS custom properties are the better solution.

---

# 2. What Is Programmatic Styling?

> Programmatic styling is reading or changing an element's visual presentation through JavaScript and DOM style APIs.

Examples:

```javascript
element.style.width =
  "300px";

element.style.backgroundColor =
  "teal";

element.classList.add(
  "is-active"
);

element.style.setProperty(
  "--progress",
  "75%"
);
```

JavaScript should not replace CSS as the primary styling system.

Its role is normally to:

* Express dynamic values.
* Apply state classes.
* Set calculated dimensions.
* Set custom-property values.
* Coordinate transitions.
* Read styles for behavior or measurements.

---

# 3. Why JavaScript Sometimes Changes Styles

CSS handles most static and responsive presentation.

JavaScript becomes useful when styles depend on runtime information.

Examples:

* Progress percentage from application state.
* Tooltip coordinates.
* Drag position.
* Element dimensions.
* Chart bar heights.
* Theme values selected by the user.
* Dynamically calculated animation delays.
* Resizable panels.
* Scroll-based progress.
* Element geometry returned by browser APIs.

Example:

```javascript
const percentage =
  completed / total * 100;

progressBar.style.width =
  `${percentage}%`;
```

The value cannot always be known when the stylesheet is written.

---

# 4. CSS Rules vs Inline Styles

## Stylesheet rule

```css
.transaction-card {
  border-radius: 8px;
}
```

## Inline style

```html
<article
  style="border-radius: 8px;"
>
```

JavaScript:

```javascript
card.style.borderRadius =
  "8px";
```

Stylesheet rules are usually better for:

* Reusable design.
* Themes.
* Responsive layouts.
* Hover and focus states.
* Media queries.
* Maintainability.

Inline styles are useful for:

* Runtime-calculated values.
* Instance-specific dimensions.
* Dynamic coordinates.
* CSS custom-property values.

---

# 5. Mental Model

An element may receive styles from several sources.

```text
Browser defaults
    ↓
External stylesheets
    ↓
Internal <style> rules
    ↓
Classes and selectors
    ↓
Inline styles
    ↓
CSS priorities and cascade
    ↓
Computed style
```

`element.style` represents only inline declarations.

`getComputedStyle()` represents the resulting resolved presentation.

---

# 6. The `style` Property

Every `HTMLElement` generally exposes:

```javascript
element.style;
```

It allows JavaScript to read and modify inline declarations.

Example:

```javascript
card.style.backgroundColor =
  "#ffffff";
```

Example:

```javascript
card.style.padding =
  "16px";
```

Example:

```javascript
card.style.display =
  "none";
```

---

# 7. What `element.style` Returns

```javascript
const styles =
  element.style;
```

The result is a:

```text
CSSStyleDeclaration
```

It is not a plain object.

It contains methods and property accessors for CSS declarations.

```javascript
console.log(
  styles instanceof
    CSSStyleDeclaration
);
```

Common methods include:

```javascript
setProperty()
getPropertyValue()
removeProperty()
getPropertyPriority()
```

---

# 8. Reading Inline Styles

HTML:

```html
<div
  id="box"
  style="
    width: 200px;
    background-color: red;
  "
></div>
```

JavaScript:

```javascript
box.style.width;
```

returns:

```text
200px
```

```javascript
box.style.backgroundColor;
```

returns:

```text
red
```

These values come from the inline `style` attribute.

---

# 9. Writing Inline Styles

```javascript
box.style.width =
  "300px";

box.style.height =
  "150px";

box.style.backgroundColor =
  "teal";
```

The browser updates the inline style attribute.

```html
<div
  style="
    width: 300px;
    height: 150px;
    background-color: teal;
  "
></div>
```

---

# 10. CSS Property Naming in JavaScript

CSS uses hyphenated names:

```css
background-color
font-size
border-radius
```

JavaScript property syntax uses camelCase:

```javascript
element.style
  .backgroundColor;

element.style.fontSize;

element.style.borderRadius;
```

Conversion examples:

| CSS                      | JavaScript            |
| ------------------------ | --------------------- |
| `background-color`       | `backgroundColor`     |
| `font-size`              | `fontSize`            |
| `border-top-left-radius` | `borderTopLeftRadius` |
| `margin-inline-start`    | `marginInlineStart`   |
| `z-index`                | `zIndex`              |

---

# 11. Hyphenated CSS Properties

Bracket notation may use the exact CSS name:

```javascript
element.style[
  "background-color"
] =
  "blue";
```

But camelCase is more common:

```javascript
element.style
  .backgroundColor =
  "blue";
```

For CSS custom properties, use `setProperty()` because their names contain leading hyphens.

---

# 12. Numeric Values Require Units

Incorrect:

```javascript
element.style.width =
  200;
```

This commonly fails or produces an invalid declaration because width requires a unit.

Correct:

```javascript
element.style.width =
  "200px";
```

Dynamic:

```javascript
const width = 200;

element.style.width =
  `${width}px`;
```

Other valid units include:

```text
%
rem
em
vh
vw
ch
fr
```

depending on the property and context.

---

# 13. Unitless CSS Properties

Some CSS properties accept unitless numbers.

Examples:

```javascript
element.style.opacity =
  "0.5";

element.style.zIndex =
  "10";

element.style.flexGrow =
  "1";

element.style.lineHeight =
  "1.5";
```

Although the property may be numeric in meaning, style values are represented as strings.

```javascript
typeof element.style.opacity;
```

returns:

```text
string
```

---

# 14. Setting Multiple Inline Styles

Direct assignments:

```javascript
element.style.width =
  "300px";

element.style.height =
  "150px";

element.style.backgroundColor =
  "teal";
```

Reusable helper:

```javascript
function applyStyles(
  element,
  styles
) {
  for (
    const [
      property,
      value,
    ] of Object.entries(
      styles
    )
  ) {
    element.style[property] =
      value;
  }
}
```

Use:

```javascript
applyStyles(
  card,
  {
    width: "300px",
    padding: "16px",
    backgroundColor:
      "white",
  }
);
```

Only use trusted property names and values.

---

# 15. The `cssText` Property

`cssText` reads or replaces the complete inline declaration string.

Read:

```javascript
element.style.cssText;
```

Write:

```javascript
element.style.cssText =
  `
    width: 300px;
    height: 150px;
    background-color: teal;
  `;
```

Important:

> Assigning `cssText` replaces existing inline declarations.

Existing styles may be lost.

Prefer individual properties unless complete replacement is intentional.

---

# 16. `style.setProperty()`

Syntax:

```javascript
element.style
  .setProperty(
    propertyName,
    value,
    priority
  );
```

Example:

```javascript
element.style
  .setProperty(
    "background-color",
    "teal"
  );
```

Custom property:

```javascript
element.style
  .setProperty(
    "--progress",
    "75%"
  );
```

Optional priority:

```javascript
element.style
  .setProperty(
    "display",
    "block",
    "important"
  );
```

Use `"important"`, not `"!important"`.

---

# 17. `style.getPropertyValue()`

Read an inline declaration using its CSS name:

```javascript
const value =
  element.style
    .getPropertyValue(
      "background-color"
    );
```

Custom property:

```javascript
const progress =
  element.style
    .getPropertyValue(
      "--progress"
    );
```

If the property is not inline, the result may be an empty string.

For final computed values, use `getComputedStyle()`.

---

# 18. `style.removeProperty()`

Remove an inline declaration:

```javascript
element.style
  .removeProperty(
    "background-color"
  );
```

It returns the previous inline value.

```javascript
const oldValue =
  element.style
    .removeProperty(
      "width"
    );
```

Removing the inline declaration allows stylesheet rules and inherited values to take effect again.

---

# 19. CSS Priorities and `!important`

Set priority:

```javascript
element.style
  .setProperty(
    "color",
    "red",
    "important"
  );
```

Inspect priority:

```javascript
element.style
  .getPropertyPriority(
    "color"
  );
```

Result:

```text
important
```

Avoid frequent `!important` usage.

It makes the cascade harder to reason about.

Prefer improving stylesheet structure and selector design.

---

# 20. CSS Custom Properties

CSS custom properties are variables defined using names beginning with `--`.

CSS:

```css
.progress-bar {
  width: var(--progress);
}
```

JavaScript:

```javascript
progressBar.style
  .setProperty(
    "--progress",
    "75%"
  );
```

They are useful for exposing dynamic values to CSS without placing every property directly inline.

---

# 21. Setting CSS Variables

Set on one element:

```javascript
card.style.setProperty(
  "--accent-color",
  "#078f82"
);
```

Set globally:

```javascript
document
  .documentElement
  .style
  .setProperty(
    "--app-accent",
    "#078f82"
  );
```

CSS:

```css
.button-primary {
  background:
    var(--app-accent);
}
```

---

# 22. Reading CSS Variables

Inline value:

```javascript
element.style
  .getPropertyValue(
    "--accent-color"
  );
```

Computed value:

```javascript
getComputedStyle(
  element
).getPropertyValue(
  "--accent-color"
);
```

Trim whitespace when appropriate:

```javascript
const value =
  getComputedStyle(element)
    .getPropertyValue(
      "--accent-color"
    )
    .trim();
```

---

# 23. Removing CSS Variables

```javascript
element.style
  .removeProperty(
    "--accent-color"
  );
```

The element may then inherit the variable or use a stylesheet fallback.

```css
color:
  var(
    --accent-color,
    black
  );
```

---

# 24. Inline Style Precedence

Inline styles normally override ordinary stylesheet declarations.

CSS:

```css
.card {
  color: blue;
}
```

Inline:

```html
<article
  class="card"
  style="color: red;"
>
```

The rendered color is normally red.

An author stylesheet rule with `!important` can affect priority comparisons.

Avoid building applications that depend on constant specificity battles.

---

# 25. The CSS Cascade

The cascade decides which declaration wins.

Factors include:

* Origin.
* Importance.
* Cascade layers.
* Specificity.
* Scope.
* Source order.
* Inline declarations.
* Inheritance.

JavaScript does not bypass the cascade.

It adds or removes declarations that participate in it.

---

# 26. Inheritance

Some properties inherit from ancestors.

Examples:

```text
color
font-family
font-size
line-height
```

Others normally do not:

```text
width
margin
padding
border
```

Example:

```css
main {
  color: darkgreen;
}
```

A child paragraph may have:

```javascript
paragraph.style.color;
```

as:

```text
""
```

but:

```javascript
getComputedStyle(
  paragraph
).color;
```

shows the inherited resolved color.

---

# 27. Why `element.style` May Return an Empty String

CSS:

```css
.card {
  width: 300px;
  background-color: white;
}
```

JavaScript:

```javascript
card.style.width;
```

returns:

```text
""
```

because no inline width is set.

But:

```javascript
getComputedStyle(
  card
).width;
```

may return:

```text
300px
```

`element.style` is not the complete style of the element.

It is the inline style declaration object.

---

# 28. `getComputedStyle()`

`getComputedStyle()` returns the final resolved CSS values for an element.

```javascript
const styles =
  getComputedStyle(
    element
  );
```

It considers:

* Stylesheets.
* Inline styles.
* Cascade.
* Inheritance.
* Browser defaults.
* Current state.
* Some layout calculations.

---

# 29. `getComputedStyle()` Syntax

```javascript
const styles =
  window.getComputedStyle(
    element,
    pseudoElement
  );
```

`window` is usually omitted:

```javascript
const styles =
  getComputedStyle(
    element
  );
```

Optional pseudo-element:

```javascript
getComputedStyle(
  element,
  "::before"
);
```

---

# 30. Computed vs Inline Values

CSS:

```css
.card {
  padding: 16px;
}
```

JavaScript:

```javascript
card.style.padding;
```

Output:

```text
""
```

Computed:

```javascript
getComputedStyle(
  card
).padding;
```

Output may be:

```text
16px
```

---

# 31. Reading Computed Properties

```javascript
const styles =
  getComputedStyle(
    card
  );

console.log(
  styles.display
);

console.log(
  styles.backgroundColor
);

console.log(
  styles.paddingTop
);
```

Exact CSS-name access:

```javascript
styles.getPropertyValue(
  "background-color"
);
```

---

# 32. Computed Values Are Read-Only

This does not update the element:

```javascript
const styles =
  getComputedStyle(
    element
  );

styles.width =
  "500px";
```

Computed style objects are for inspection.

To change style:

```javascript
element.style.width =
  "500px";
```

or update classes and CSS rules.

---

# 33. Resolved and Used Values

Computed styles often return resolved values.

Example:

```css
width: 50%;
```

Depending on the property and layout state:

```javascript
getComputedStyle(
  element
).width;
```

may return a pixel value such as:

```text
400px
```

Do not assume computed output uses the same units written in the stylesheet.

---

# 34. Colors in Computed Styles

CSS:

```css
color: red;
```

Computed output may be:

```text
rgb(255, 0, 0)
```

CSS:

```css
color: #078f82;
```

Computed output may become an RGB serialization.

Do not compare computed colors to source strings without normalization.

---

# 35. Dimensions in Computed Styles

```javascript
const width =
  getComputedStyle(
    element
  ).width;
```

Returns a string:

```text
300px
```

Convert if needed:

```javascript
const numericWidth =
  Number.parseFloat(
    width
  );
```

For actual geometry and viewport-relative dimensions, APIs such as `getBoundingClientRect()` may be more appropriate.

Those are covered in the next lesson.

---

# 36. Pseudo-Element Styles

Read `::before`:

```javascript
const beforeStyles =
  getComputedStyle(
    element,
    "::before"
  );
```

Example:

```javascript
console.log(
  beforeStyles.content
);
```

Pseudo-elements are not normal DOM element nodes.

You cannot select them with `querySelector()`.

You can inspect their computed styles.

---

# 37. Display and Visibility

Several different mechanisms can make content unseen.

These mechanisms do not behave identically.

```text
display: none
visibility: hidden
opacity: 0
hidden attribute
off-screen positioning
clipping
```

Understanding the differences matters for:

* Layout.
* Interaction.
* Focus.
* Accessibility.
* Measurements.

---

# 38. `display: none`

```javascript
element.style.display =
  "none";
```

Effects:

* Removed from normal layout.
* Usually not visually rendered.
* Descendants are also not displayed.
* Generally unavailable for ordinary interaction.
* Geometry may become zero or unavailable.

Restore:

```javascript
element.style.removeProperty(
  "display"
);
```

Avoid restoring with a guessed value such as `"block"` when the original display type may differ.

---

# 39. `visibility: hidden`

```javascript
element.style.visibility =
  "hidden";
```

Effects:

* Element occupies layout space.
* It is not visible.
* Interaction behavior differs from visible content.
* Descendant visibility can be nuanced.

Restore:

```javascript
element.style.visibility =
  "";
```

or:

```javascript
element.style
  .removeProperty(
    "visibility"
  );
```

---

# 40. `opacity: 0`

```javascript
element.style.opacity =
  "0";
```

Effects:

* Element still occupies layout space.
* It may still receive pointer or keyboard interaction.
* It remains structurally present.
* It is visually transparent.

Opacity alone should not be used to make interactive content unavailable.

---

# 41. The `hidden` Property vs CSS Hiding

Semantic HTML hiding:

```javascript
element.hidden =
  true;
```

CSS class:

```javascript
element.classList.add(
  "is-hidden"
);
```

Inline:

```javascript
element.style.display =
  "none";
```

Prefer `hidden` when the content should semantically not be presented.

Use CSS classes for design-specific visibility behavior such as animated collapse.

Keep visual and accessibility states synchronized.

---

# 42. Classes vs Inline Styles

Weak:

```javascript
card.style.backgroundColor =
  "green";

card.style.borderColor =
  "darkgreen";

card.style.color =
```

Use inline styles when the value is:

* Calculated at runtime.

# 61. Style Reads and Writes

Style write:

```javascript
element.style.width =
  "200px";
```


# 62. Layout Recalculation

When styles affecting geometry change, the browser may need to recalculate layout.

Examples:

```text
width
height
padding
margin
display
position
font-size
```

A later geometry read may force pending calculations to complete.

This is sometimes called a forced synchronous layout.

---

# 63. Layout Thrashing

Problem:

```javascript
for (
  const item
  of items
) {
  item.style.width =
    `${item.offsetWidth + 10}px`;
}
```

Each iteration:

1. Reads layout.
2. Writes style.
3. Forces future layout work.
4. Repeats.

Better:

```javascript
const widths =
  items.map(
    item =>
      item.offsetWidth
  );

items.forEach(
  (
    item,
    index
  ) => {
    item.style.width =
      `${widths[index] + 10}px`;
  }
);
```

Read first, then write.

---

# 64. Batching Reads and Writes

Good structure:

```text
1. Read all needed geometry
2. Calculate values
3. Write all style changes
```

Example:

```javascript
const measurements =
  cards.map(
    card => ({
      card,
      width:
        card.offsetWidth,
    })
  );

for (
  const {
    card,
    width,
  } of measurements
) {
  card.style.width =
    `${width + 20}px`;
}
```

# 67. Security Considerations

Style assignment can be security-sensitive when untrusted data controls:

* URLs.
* Positioning.
* Visibility.
* Click interception.
* Overlay dimensions.
* Resource loading.
* CSS custom-property content used in sensitive contexts.

Avoid accepting arbitrary property names and values from users.

Weak:

```javascript
element.style.cssText =
  userStyle;
```

Better:

```javascript
const allowedColors =
  new Set([
    "red",
    "blue",
    "green",
  ]);

if (
  allowedColors.has(
    userColor
  )
) {
  element.style.color =
    userColor;
}
```

Use strict allowlists and validation.

# 68. Performance Considerations

1. Prefer classes for known visual states.
2. Use inline styles for truly dynamic values.
3. Avoid repeated `cssText` replacement.
4. Avoid unnecessary computed-style reads.
5. Batch layout reads and writes.
6. Minimize forced layout.
7. Use transforms and opacity for smoother animation where appropriate.
8. Avoid rendering huge DOM collections.
9. Measure real interactions.
10. Respect reduced-motion preferences.

# 69. Memory Considerations

Styles themselves rarely create major memory leaks.

However, style-related systems can retain components through:

* Animation callbacks.
* Media-query listeners.
* Resize observers.
* Timers.
* `requestAnimationFrame()` loops.
* Transition handlers.
* Controller registries.

Clean up:

```javascript
cancelAnimationFrame(
  frameId
);

mediaQuery.removeEventListener(
  "change",
  handler
);
```

Component lifecycle remains important.

## Example 3: Real-World Application — Expandable Transaction Details

### HTML

```html
<article
  class="transaction-card"
  data-component="transaction-card"
>
  <button
    type="button"
    data-action="toggle-details"
    aria-expanded="false"
  >
    View Details
  </button>

  <section
    data-role="details"
    hidden
  >
    <p data-role="service"></p>
    <p data-role="amount"></p>
  </section>
</article>
```

### CSS

```css
.transaction-card
[data-role="details"] {
  max-height: 0;
  overflow: hidden;
  opacity: 0;
  transition:
    max-height 250ms ease,
    opacity 250ms ease;
}

.transaction-card.is-expanded
[data-role="details"] {
  max-height:
    var(
      --details-height
    );

  opacity: 1;
}
```

### JavaScript

```javascript
function setExpanded(
  card,
  isExpanded
) {
  const button =
    card.querySelector(
      "[data-action='toggle-details']"
    );

  const details =
    card.querySelector(
      "[data-role='details']"
    );

  if (
    !button ||
    !details
  ) {
    throw new Error(
      "Invalid transaction card structure."
    );
  }

  button.setAttribute(
    "aria-expanded",
    String(isExpanded)
  );

  card.classList.toggle(
    "is-expanded",
    isExpanded
  );

  if (isExpanded) {
    details.hidden =
      false;

    const requiredHeight =
      `${details.scrollHeight}px`;

    card.style.setProperty(
      "--details-height",
      requiredHeight
    );

    return;
  }

  card.style.setProperty(
    "--details-height",
    "0px"
  );

  const handleTransitionEnd =
    event => {
      if (
        event.propertyName !==
          "max-height"
      ) {
        return;
      }

      details.hidden =
        true;

      details.removeEventListener(
        "transitionend",
        handleTransitionEnd
      );
    };

  details.addEventListener(
    "transitionend",
    handleTransitionEnd
  );
}
```

### Why this is professional

* JavaScript controls semantic state.
* ARIA state is synchronized.
* A class represents named visual state.
* A custom property carries a calculated height.
* CSS owns transition presentation.
* The section becomes semantically hidden after collapse.

# 71. Common Mistakes

## Mistake 1: Forgetting units

```javascript
element.style.width =
  200;
```

Use:

```javascript
element.style.width =
  "200px";
```

---

## Mistake 2: Using CSS property names with dot notation

Incorrect:

```javascript
element.style
  .background-color =
  "red";
```

Correct:

```javascript
element.style
  .backgroundColor =
  "red";
```

---

## Mistake 3: Expecting `element.style` to show stylesheet values

Use:

```javascript
getComputedStyle(
  element
);
```

---

## Mistake 4: Trying to modify computed styles

Computed declarations are read-only.

---

## Mistake 5: Replacing all inline styles with `cssText`

This can remove unrelated declarations.

---

## Mistake 6: Setting many known state styles inline

Use a class.

---

## Mistake 7: Restoring `display` with a guessed value

```javascript
element.style.display =
  "block";
```

The original may have been `flex`, `grid`, or `inline`.

Remove the inline declaration instead.

---

## Mistake 8: Using opacity alone to disable content

Transparent controls may remain interactive.

---

## Mistake 9: Alternating layout reads and writes in a loop

This can cause layout thrashing.

---

## Mistake 10: Ignoring reduced-motion preferences

Animations may harm users with motion sensitivity.

---

## Mistake 11: Comparing computed colors to source strings

Computed color serialization may differ.

---

## Mistake 12: Assigning arbitrary untrusted CSS text

This creates security and interface-integrity risks.

# 72. Professional Best Practices

1. Keep reusable presentation in CSS.
2. Use classes for named states.
3. Use inline styles for calculated instance-specific values.
4. Use CSS custom properties to connect runtime values with stylesheet rules.
5. Include units where required.
6. Validate dynamic numeric values.
7. Use `setProperty()` for custom properties.
8. Use `removeProperty()` to restore stylesheet control.
9. Read final style using `getComputedStyle()`.
10. Treat computed values as read-only.
11. Avoid `cssText` unless complete replacement is intentional.
12. Do not guess original display values.
13. Keep semantic state synchronized with visual state.
14. Respect reduced-motion and contrast preferences.
15. Batch layout reads and writes.
16. Avoid forced synchronous layout where possible.
17. Use `requestAnimationFrame()` for coordinated visual writes.
18. Clean media-query and animation listeners.
19. Validate untrusted style inputs with strict allowlists.
20. Keep application state separate from DOM style values.

# 74. Interview Questions and Answers

## Beginner Questions

### 1. What does `element.style` represent?

The element's inline style declarations.

### 2. What type does `element.style` return?

A `CSSStyleDeclaration`.

### 3. How do you set `background-color` through dot notation?

```javascript
element.style
  .backgroundColor =
  "red";
```

### 4. Why does width usually need a unit?

CSS dimensions require valid units such as `px`, `%`, or `rem`.

### 5. How do you remove an inline declaration?

```javascript
element.style
  .removeProperty(
    "width"
  );
```

### 6. What does `getComputedStyle()` return?

A read-only style declaration containing resolved styles.

### 7. Does `element.style` include class-based stylesheet values?

No.

### 8. Can computed styles be assigned to?

No.

---

## Intermediate Questions

### 1. Why might `element.style.color` be empty while the element is visibly colored?

The color may come from a stylesheet or inheritance rather than an inline declaration.

### 2. When should a class be used instead of inline styles?

When the style represents a reusable named state or changes several properties together.

### 3. Why are CSS custom properties useful with JavaScript?

They allow JavaScript to provide dynamic values while CSS remains responsible for presentation.

### 4. What happens when `cssText` is assigned?

The complete inline declaration set is replaced.

### 5. How do you set `!important` through JavaScript?

```javascript
style.setProperty(
  property,
  value,
  "important"
);
```

### 6. Why should `display: block` not always be used to restore a hidden element?

Its natural display value may be `flex`, `grid`, `inline`, or another type.

### 7. What is the difference between `display: none`, `visibility: hidden`, and `opacity: 0`?

They differ in layout participation, visibility, and interactivity.

---

## Advanced Questions

### 1. What is layout thrashing?

Repeated interleaving of layout-dependent reads and style writes that forces frequent synchronous layout recalculation.

### 2. How can layout thrashing be reduced?

Batch reads first, calculate, then batch writes.

### 3. When might `getComputedStyle()` trigger expensive browser work?

When the browser must resolve pending style or layout changes before returning current values.

### 4. Why should application state not be derived from CSS classes or inline values?

Presentation may change independently, and CSS values are not reliable authoritative business state.

### 5. How would you implement a dynamic theme?

Set validated CSS custom properties on a suitable root and let stylesheet rules consume them.

### 6. How should reduced-motion preferences influence JavaScript animation?

Detect the preference or rely on CSS media queries, then reduce or disable non-essential motion.

### 7. What security risks exist in dynamic styling?

Untrusted CSS values may alter visibility, positioning, resource URLs, overlays, or interaction integrity.

# 78. Summary

JavaScript can inspect and modify element presentation.

You learned:

```javascript
element.style
```

which represents inline declarations.

You learned:

```javascript
getComputedStyle(
  element
)
```

which returns resolved styles after the cascade and inheritance are applied.

Important inline APIs include:

```javascript
style.propertyName
style.setProperty()
style.getPropertyValue()
style.removeProperty()
style.cssText
```

You learned that:

* CSS property names usually become camelCase in JavaScript.
* Numeric dimensions commonly require units.
* Some properties are unitless.
* `cssText` replaces all inline declarations.
* Custom properties connect runtime values to CSS.
* Inline styles participate in the cascade.
* `element.style` does not include class or stylesheet values.
* Computed values are read-only.
* Computed values may use normalized units and color formats.
* Classes are better for named visual states.
* Inline styles are appropriate for calculated instance-specific values.
* CSS should remain responsible for reusable presentation.
* `display`, `visibility`, `opacity`, and `hidden` have different behavior.
* Visual and semantic state must remain synchronized.
* CSS transitions and animations should respect reduced-motion preferences.
* Alternating style writes and layout reads can cause layout thrashing.
* Reads and writes should be batched.
* Untrusted style values require validation.

The professional styling flow is:

```text
Application state
    ↓
Named visual state?
    ├── Yes → CSS class
    └── No
         ↓
Calculated runtime value?
    ├── Yes → inline style or CSS variable
    └── No → stylesheet rule
         ↓
Need final rendered value?
    └── getComputedStyle()
```
