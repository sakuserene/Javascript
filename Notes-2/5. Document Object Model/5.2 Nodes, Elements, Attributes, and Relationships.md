# 1. Introduction

In the previous lesson, you learned that the browser converts HTML into a tree-shaped object model.

Example HTML:

```html
<section>
  <h2>Transactions</h2>

  <p>Today: KES 5,000</p>
</section>
```

Simplified DOM tree:

```text
section
├── h2
│   └── "Transactions"
└── p
    └── "Today: KES 5,000"
```

At first glance, this tree appears to contain only HTML elements.

However, the DOM contains more than elements.

It can contain:

* The document itself.
* The doctype.
* HTML elements.
* Text.
* Comments.
* Document fragments.
* Other specialized node types.

Every item in the tree is a **node**.

An element is therefore one type of node, but not every node is an element.

This distinction is essential because DOM properties often come in two versions:

```javascript
element.children;
```

and:

```javascript
element.childNodes;
```

These do not return the same thing.

Similarly:

```javascript
element.firstChild;
```

and:

```javascript
element.firstElementChild;
```

may produce different results.

To navigate and manipulate the DOM confidently, you must understand exactly what kinds of objects exist inside the tree.

---

# 2. Why DOM Node Types Exist

Consider this HTML:

```html
<div>
  Hello
  <!-- Status message -->
  <span>World</span>
</div>
```

The `<div>` contains three meaningful pieces:

1. The text `"Hello"`.
2. A comment.
3. A `<span>` element.

The browser cannot represent all three as the same kind of object because they have different capabilities.

An element can have:

* Attributes.
* Classes.
* Styles.
* Child elements.
* Event listeners.

A text node contains text but has no:

* Tag name.
* Classes.
* HTML attributes.
* Element-specific style interface.

A comment node stores comment text but is not displayed as page content.

The DOM therefore uses different node types.

```text
Node
├── Document
├── DocumentType
├── Element
├── Text
├── Comment
└── DocumentFragment
```

These specialized types share common node behavior while adding type-specific capabilities.

---

# 3. What Is a Node?

A **node** is a single object in the DOM tree.

A node may represent:

* The entire document.
* An HTML element.
* Text inside an element.
* An HTML comment.
* A document fragment.
* The document type declaration.

Example:

```html
<p>Hello</p>
```

This produces at least two nodes:

```text
Element node: p
└── Text node: "Hello"
```

The paragraph element and its text are separate DOM objects.

## Precise definition

> A DOM node is an object representing one structural unit in a document tree.

All common DOM node types inherit shared behavior from the `Node` interface.

---

# 4. The Node Interface

The `Node` interface defines properties and methods shared by DOM nodes.

Common properties include:

```javascript
node.nodeType;
node.nodeName;
node.nodeValue;
node.textContent;
node.parentNode;
node.childNodes;
node.firstChild;
node.lastChild;
node.previousSibling;
node.nextSibling;
```

Common methods include:

```javascript
node.appendChild();
node.removeChild();
node.replaceChild();
node.cloneNode();
node.contains();
```

Different node types interpret some properties differently.

For example:

```javascript
element.nodeValue;
```

normally returns:

```text
null
```

while:

```javascript
textNode.nodeValue;
```

returns the text stored in that node.

---

# 5. Common DOM Node Types

The DOM standard defines several node types.

The most important types for everyday frontend development are:

| Node Type             | Constant                      | Numeric Value |
| --------------------- | ----------------------------- | ------------: |
| Element node          | `Node.ELEMENT_NODE`           |           `1` |
| Text node             | `Node.TEXT_NODE`              |           `3` |
| Comment node          | `Node.COMMENT_NODE`           |           `8` |
| Document node         | `Node.DOCUMENT_NODE`          |           `9` |
| DocumentType node     | `Node.DOCUMENT_TYPE_NODE`     |          `10` |
| DocumentFragment node | `Node.DOCUMENT_FRAGMENT_NODE` |          `11` |

Example:

```javascript
console.log(
  Node.ELEMENT_NODE
);
```

Output:

```text
1
```

You should normally compare against named constants rather than memorizing numeric values.

Prefer:

```javascript
if (
  node.nodeType ===
  Node.TEXT_NODE
) {
  // Handle text.
}
```

instead of:

```javascript
if (node.nodeType === 3) {
}
```

Named constants communicate intent.

---

# 6. Element Nodes

An element node represents an HTML or XML element.

Examples:

```html
<div></div>
```

```html
<button></button>
```

```html
<input>
```

```html
<section></section>
```

JavaScript can select an element:

```javascript
const button =
  document.querySelector(
    "button"
  );
```

The selected object is an element node.

Check:

```javascript
console.log(
  button.nodeType ===
    Node.ELEMENT_NODE
);
```

Output:

```text
true
```

Element nodes can have:

* Tag names.
* Attributes.
* Classes.
* Child nodes.
* Child elements.
* Styles.
* Event listeners.
* Element-specific properties.

---

# 7. HTMLElement Objects

`Element` is a general DOM interface.

HTML elements usually implement more specialized interfaces.

For example:

```html
<button>Save</button>
```

may be represented by an object whose interface chain conceptually includes:

```text
Node
    ↓
Element
    ↓
HTMLElement
    ↓
HTMLButtonElement
```

This inheritance gives the object increasing capabilities.

## Node level

```javascript
button.parentNode;
button.childNodes;
button.textContent;
```

## Element level

```javascript
button.id;
button.classList;
button.getAttribute();
button.querySelector();
```

## HTMLElement level

```javascript
button.hidden;
button.title;
button.style;
button.dataset;
```

## HTMLButtonElement level

```javascript
button.disabled;
button.type;
button.form;
```

Different HTML elements can have specialized interfaces.

Examples:

| HTML Element | Common DOM Interface |
| ------------ | -------------------- |
| `<button>`   | `HTMLButtonElement`  |
| `<input>`    | `HTMLInputElement`   |
| `<form>`     | `HTMLFormElement`    |
| `<img>`      | `HTMLImageElement`   |
| `<a>`        | `HTMLAnchorElement`  |
| `<video>`    | `HTMLVideoElement`   |

This explains why an input has:

```javascript
input.value;
```

while an ordinary `<div>` does not use `value` in the same standard way.

---

# 8. Text Nodes

Text inside an element is represented by a text node.

HTML:

```html
<h1>Daily Collections</h1>
```

DOM:

```text
h1 element
└── Text node: "Daily Collections"
```

Access the text node:

```javascript
const heading =
  document.querySelector(
    "h1"
  );

const textNode =
  heading.firstChild;
```

Check its type:

```javascript
console.log(
  textNode.nodeType ===
    Node.TEXT_NODE
);
```

Output:

```text
true
```

Read its value:

```javascript
console.log(
  textNode.nodeValue
);
```

Output:

```text
Daily Collections
```

A text node is not an element.

It does not have:

```javascript
textNode.classList;
```

or:

```javascript
textNode.getAttribute();
```

Those belong to elements.

---

# 9. Comment Nodes

HTML comments become comment nodes.

HTML:

```html
<!-- Main dashboard -->
<section></section>
```

The comment is part of the DOM tree.

Conceptually:

```text
Comment node:
" Main dashboard "
```

A comment node has:

```javascript
nodeType ===
  Node.COMMENT_NODE
```

Comments are normally not displayed visually, but JavaScript can inspect them.

Example traversal:

```javascript
for (
  const node
  of document.body.childNodes
) {
  if (
    node.nodeType ===
    Node.COMMENT_NODE
  ) {
    console.log(
      node.nodeValue
    );
  }
}
```

Comments can help developers, but relying on comments as application data is usually not ideal.

---

# 10. Document Nodes

The `document` object itself is a node.

Check:

```javascript
console.log(
  document.nodeType ===
    Node.DOCUMENT_NODE
);
```

Output:

```text
true
```

The document node sits above the `<html>` element in the DOM tree.

```text
Document
└── html
```

Its parent is normally:

```javascript
document.parentNode;
```

Output:

```text
null
```

The document is the top-level DOM node for the loaded page.

---

# 11. DocumentType Nodes

The HTML doctype declaration:

```html
<!DOCTYPE html>
```

becomes a `DocumentType` node.

Access it:

```javascript
console.log(
  document.doctype
);
```

Its node type is:

```javascript
Node.DOCUMENT_TYPE_NODE
```

Conceptual DOM:

```text
Document
├── DocumentType: html
└── html
```

The doctype helps the browser choose standards-compliant rendering behavior.

It is not an ordinary HTML element.

---

# 12. DocumentFragment Nodes

A `DocumentFragment` is a lightweight container for DOM nodes.

Create one:

```javascript
const fragment =
  document.createDocumentFragment();
```

Check its type:

```javascript
console.log(
  fragment.nodeType ===
    Node.DOCUMENT_FRAGMENT_NODE
);
```

Output:

```text
true
```

A fragment can temporarily hold nodes before they are inserted into the document.

```javascript
const fragment =
  document.createDocumentFragment();

const itemOne =
  document.createElement("li");

const itemTwo =
  document.createElement("li");

fragment.append(
  itemOne,
  itemTwo
);
```

When the fragment is appended to the document, its children move into the target.

The fragment itself is not inserted as a visible wrapper.

```javascript
list.append(fragment);
```

Result:

```html
<ul>
  <li></li>
  <li></li>
</ul>
```

Not:

```html
<ul>
  <document-fragment>
    <li></li>
    <li></li>
  </document-fragment>
</ul>
```

Document fragments will become more important in the DOM Manipulation module.

# 13. `nodeType`

The `nodeType` property identifies a node's type.

Example HTML:

```html
<p>Hello</p>
```

JavaScript:

```javascript
const paragraph =
  document.querySelector(
    "p"
  );

const text =
  paragraph.firstChild;

console.log(
  paragraph.nodeType
);

console.log(
  text.nodeType
);
```

Output:

```text
1
3
```

Prefer constants:

```javascript
console.log(
  paragraph.nodeType ===
    Node.ELEMENT_NODE
);

console.log(
  text.nodeType ===
    Node.TEXT_NODE
);
```

## Why `nodeType` is useful

Use it when code must process mixed node collections.

```javascript
for (
  const node
  of container.childNodes
) {
  if (
    node.nodeType ===
    Node.ELEMENT_NODE
  ) {
    console.log(
      "Element:",
      node.nodeName
    );
  }

  if (
    node.nodeType ===
    Node.TEXT_NODE
  ) {
    console.log(
      "Text:",
      node.nodeValue
    );
  }
}
```

# 14. `nodeName`

`nodeName` returns a node-specific name.

## Element node

```javascript
const paragraph =
  document.querySelector(
    "p"
  );

console.log(
  paragraph.nodeName
);
```

Output:

```text
P
```

For HTML documents, element node names are commonly uppercase.

## Text node

```javascript
console.log(
  paragraph.firstChild.nodeName
);
```

Output:

```text
#text
```

## Comment node

```text
#comment
```

## Document node

```text
#document
```

`nodeName` is useful for generic node inspection.

For element-specific tag inspection, developers often use:

```javascript
element.tagName;
```

---

# 15. `nodeValue`

`nodeValue` behaves differently across node types.

## Text node

```javascript
textNode.nodeValue;
```

returns the node's text.

## Comment node

```javascript
commentNode.nodeValue;
```

returns the comment text.

## Element node

```javascript
element.nodeValue;
```

normally returns:

```text
null
```

Why?

An element's content is stored in child nodes rather than directly in `nodeValue`.

Example:

```html
<p>Hello</p>
```

Structure:

```text
p element
└── text node: "Hello"
```

Therefore:

```javascript
paragraph.nodeValue;
```

returns:

```text
null
```

while:

```javascript
paragraph.firstChild.nodeValue;
```

returns:

```text
Hello
```

---

# 16. `textContent`

`textContent` returns or replaces the textual content of a node and its descendants.

HTML:

```html
<p>
  Hello
  <strong>World</strong>
</p>
```

JavaScript:

```javascript
const paragraph =
  document.querySelector(
    "p"
  );

console.log(
  paragraph.textContent
);
```

Conceptual output:

```text
Hello World
```

It combines descendant text.

Set it:

```javascript
paragraph.textContent =
  "Updated";
```

The existing child content is replaced by a text node.

Before:

```text
p
├── Text: "Hello"
└── strong
    └── Text: "World"
```

After:

```text
p
└── Text: "Updated"
```

> [!WARNING]
> Assigning `textContent` removes existing child elements inside the target.

## Security advantage

`textContent` treats content as text rather than HTML markup.

```javascript
element.textContent =
  "<strong>Hello</strong>";
```

Displays:

```text
<strong>Hello</strong>
```

It does not create a `<strong>` element.

This makes `textContent` safer for untrusted text than parsing HTML strings.

# 17. Attributes

Attributes appear in HTML start tags.

```html
<button
  id="save-button"
  class="primary"
  disabled
  aria-label="Save transaction"
>
  Save
</button>
```

Attributes include:

```text
id
class
disabled
aria-label
```

The DOM exposes attribute APIs through element objects.

Common methods:

```javascript
element.getAttribute();
element.setAttribute();
element.hasAttribute();
element.removeAttribute();
element.toggleAttribute();
```

Attributes provide metadata and configuration for elements.

---

# 18. HTML Attributes vs DOM Properties

Attributes and DOM properties are related but not always identical.

HTML:

```html
<input
  id="username"
  value="Amina"
>
```

Attribute:

```javascript
input.getAttribute(
  "value"
);
```

Property:

```javascript
input.value;
```

Initially, both may be:

```text
Amina
```

Now the user types:

```text
Brian
```

The current property may become:

```javascript
input.value;
```

```text
Brian
```

while the original attribute may remain:

```javascript
input.getAttribute(
  "value"
);
```

```text
Amina
```

## Mental model

```text
HTML attribute
    ↓
Initial configuration

DOM property
    ↓
Current live state
```

This difference is especially important for:

* Form values.
* Checked checkboxes.
* Selected options.
* Disabled state.
* URLs.
* Boolean attributes.

---

# 19. Reading Attributes

Use:

```javascript
element.getAttribute(
  "attribute-name"
);
```

Example:

```html
<button
  id="save"
  aria-label="Save transaction"
>
  Save
</button>
```

JavaScript:

```javascript
const button =
  document.querySelector(
    "#save"
  );

console.log(
  button.getAttribute(
    "aria-label"
  )
);
```

Output:

```text
Save transaction
```

Check existence:

```javascript
button.hasAttribute(
  "disabled"
);
```

Returns a Boolean.

List attributes:

```javascript
console.log(
  button.attributes
);
```

`attributes` provides a named collection of attribute objects.

In everyday code, direct attribute methods are usually simpler than manually iterating this collection.

---

# 20. Writing Attributes

Use:

```javascript
element.setAttribute(
  name,
  value
);
```

Example:

```javascript
button.setAttribute(
  "aria-label",
  "Save current transaction"
);
```

Set a custom title:

```javascript
button.setAttribute(
  "title",
  "Save"
);
```

For common reflected properties, property assignment is often clearer.

```javascript
button.id =
  "save-button";
```

```javascript
button.title =
  "Save";
```

Choose based on the specific attribute and whether you need raw markup-level access or live property behavior.

---

# 21. Removing Attributes

Use:

```javascript
element.removeAttribute(
  "attribute-name"
);
```

Example:

```javascript
button.removeAttribute(
  "disabled"
);
```

Toggle an attribute:

```javascript
button.toggleAttribute(
  "hidden"
);
```

Explicit force:

```javascript
button.toggleAttribute(
  "hidden",
  true
);
```

This ensures the attribute exists.

```javascript
button.toggleAttribute(
  "hidden",
  false
);
```

This ensures it is removed.

---

# 22. Boolean Attributes

Boolean attributes represent true when present and false when absent.

Examples:

```html
<input disabled>
```

```html
<input required>
```

```html
<input checked>
```

```html
<section hidden></section>
```

The value text does not work like a normal Boolean.

These all mean the attribute is present:

```html
<input disabled>
```

```html
<input disabled="">
```

```html
<input disabled="false">
```

Even:

```html
disabled="false"
```

still means disabled because the attribute is present.

Correct ways to enable:

```javascript
input.disabled =
  false;
```

or:

```javascript
input.removeAttribute(
  "disabled"
);
```

## Best practice

For standard Boolean states, prefer DOM properties:

```javascript
button.disabled = true;
button.hidden = false;
checkbox.checked = true;
```

These communicate the current state clearly.

---

# 23. Custom Data Attributes

HTML supports custom data attributes beginning with:

```text
data-
```

Example:

```html
<button
  data-transaction-id="42"
  data-action="delete"
>
  Delete
</button>
```

Read using `getAttribute()`:

```javascript
button.getAttribute(
  "data-transaction-id"
);
```

Or using `dataset`:

```javascript
button.dataset.transactionId;
```

Output:

```text
42
```

Hyphenated names convert to camel case.

```html
data-user-role="admin"
```

becomes:

```javascript
element.dataset.userRole;
```

Write:

```javascript
element.dataset.status =
  "pending";
```

This creates:

```html
data-status="pending"
```

## Important detail

Dataset values are strings.

```javascript
element.dataset.count =
  5;
```

Reading it gives:

```javascript
"5"
```

Convert when necessary:

```javascript
const count =
  Number(
    element.dataset.count
  );
```

# 24. Parent and Child Relationships

The DOM tree creates relationships.

HTML:

```html
<section id="summary">
  <h2>Summary</h2>
  <p>Total: KES 5,000</p>
</section>
```

Tree:

```text
section
├── h2
│   └── text
└── p
    └── text
```

Relationships:

* `section` is parent of `h2` and `p`.
* `h2` and `p` are children of `section`.
* `h2` and `p` are siblings.
* The text inside `h2` is a child node of `h2`.
* `section` is an ancestor of that text node.
* The text node is a descendant of `section`.

---

# 25. `parentNode` vs `parentElement`

## `parentNode`

Returns the parent node, regardless of its type.

```javascript
node.parentNode;
```

## `parentElement`

Returns the parent only if it is an element.

```javascript
node.parentElement;
```

Otherwise it returns:

```text
null
```

For ordinary elements inside the body, these often return the same object.

```javascript
paragraph.parentNode ===
  paragraph.parentElement;
```

may be:

```text
true
```

But for the `<html>` element:

```javascript
document.documentElement
  .parentNode;
```

returns the document node.

```javascript
document.documentElement
  .parentElement;
```

returns:

```text
null
```

because the document is not an element.

---

# 26. `childNodes` vs `children`

This is one of the most important DOM distinctions.

## `childNodes`

Returns all direct child nodes.

This may include:

* Elements.
* Text nodes.
* Comments.

```javascript
container.childNodes;
```

## `children`

Returns only direct child elements.

```javascript
container.children;
```

Example HTML:

```html
<div id="box">
  Hello
```

`childNodes` returns a:


`children` returns an:

Both are array-like, but they are not ordinary arrays.

# 27. `firstChild` vs `firstElementChild`

## `firstChild`

Returns the first child node of any type.

```javascript
container.firstChild;
```

It may be a whitespace text node.

## `firstElementChild`

Returns the first child element.

```javascript
container.firstElementChild;
```

Example:

```html

  <li>Printing</li>
  <li>Binding</li>
</ul>
```

Because of the newline and indentation, this may happen:


Output:

```text
3
```

The first child is a text node containing whitespace.

But:

```javascript
list.firstElementChild
  .textContent;
```

Output:

```text
Printing
```

returns the last node of any type.

returns the last child element.

# 29. Sibling Relationships

Sibling nodes share the same parent.

HTML:

```html
<section>
  <h2>Summary</h2>
  <p>Total</p>
  <button>Refresh</button>
</section>
```

Element siblings:

```text
h2
p
button
```

However, whitespace text nodes may also exist between them.

Node-based sibling APIs include those text nodes.

Element-based sibling APIs skip them.

```javascript

```

This is usually what you want for element navigation.

# 32. Ancestors and Descendants

A parent is the immediate containing node.

An ancestor is any node higher in the tree.

HTML:

```html
<body>
  <main>
    <section>
      <button>Save</button>
    </section>
  </main>
</body>
```

For the button:

* Parent: `section`
* Ancestors:

  * `section`
  * `main`
  * `body`
  * `html`
  * `document`

A child is an immediate contained node.

A descendant is any node lower in the tree.

For `main`, descendants include:

* `section`
* `button`
* Text nodes inside the button

Useful method:

```javascript
ancestor.contains(
  possibleDescendant
);
```

Example:

```javascript
main.contains(button);
```

returns:

```text
true
```

# 33. Whitespace Text Nodes

Formatted HTML contains whitespace.

Example:

```html
<ul>
  <li>Printing</li>
  <li>Binding</li>
</ul>
```

Between `<ul>` and the first `<li>` is a newline plus spaces.

The browser may represent this as a text node.

Simplified tree:

```text
ul
├── Text: "\n  "
├── li
│   └── Text: "Printing"
├── Text: "\n  "
├── li
│   └── Text: "Binding"
└── Text: "\n"
```

This explains why:

```javascript
list.childNodes.length;
```

may be greater than:

```javascript
list.children.length;
```

## Do whitespace text nodes matter?

Sometimes.

They matter when using:

* `childNodes`.
* `firstChild`.
* `nextSibling`.
* `previousSibling`.
* Generic node-processing code.

They usually do not matter when using:

* `children`.
* `firstElementChild`.
* `nextElementSibling`.
* CSS selectors.

# 34. Live Collections vs Static Collections

Some DOM collections update automatically when the DOM changes.

These are called **live collections**.

## Live `HTMLCollection`

```javascript
const items =
  list.children;
```

Add another child:

```javascript
list.append(
  document.createElement("li")
);
```

Now:

```javascript
items.length;
```

automatically reflects the new child.

## Live `NodeList`

`childNodes` is also live.

```javascript
const nodes =
  list.childNodes;
```

Changes to direct children are reflected automatically.

## Static `NodeList`

`querySelectorAll()` returns a static `NodeList`.

```javascript
const selectedItems =
  document.querySelectorAll(
    "li"
  );
```

If a new `<li>` is added later, the original `selectedItems` does not automatically include it.

You would need to query again.

## Why this matters

Changing a live collection while iterating over it can create surprising behavior.

Example:

```javascript
for (
  let index = 0;
  index < list.children.length;
  index += 1
) {
  list.children[index]
    .remove();
}
```

Removing an item changes the collection indexes immediately.

Some elements may be skipped.

Safer strategies include:

```javascript
while (
  list.firstElementChild
) {
  list.firstElementChild
    .remove();
}
```

or converting to an array:

```javascript
const children = [
  ...list.children,
];

for (const child of children) {
  child.remove();
}
```

# 35. Internal Browser Representation

Consider:

```html
<div id="card">
  <h2>Receipt</h2>
  <p>KES 500</p>
</div>
```

A simplified object relationship might look like:

```text
HTMLDivElement
├── id: "card"
├── parentNode → body
├── childNodes → NodeList
│   ├── Text
│   ├── HTMLHeadingElement
│   ├── Text
│   ├── HTMLParagraphElement
│   └── Text
└── children → HTMLCollection
    ├── HTMLHeadingElement
    └── HTMLParagraphElement
```

Each element holds references to related nodes.

The browser maintains the tree so properties such as:

```javascript

node.nextSibling;
```

can be resolved.

JavaScript variables usually store references to these browser-managed objects.

```javascript
const card =
  document.querySelector(
    "#card"
  );
```

Conceptual memory:

```text
JavaScript variable: card
    ↓ reference
Browser-managed HTMLDivElement
    ↓
DOM tree connections
```

# 37. Common Mistakes

## Mistake 1: Treating every node as an element

Incorrect:

```javascript
for (
  const node
  of container.childNodes
) {
  node.classList.add(
    "active"
  );
}
```

A text node has no `classList`.

Correct:

```javascript
for (
  const child
  of container.children
) {
  child.classList.add(
    "active"
  );
}
```

or check:

```javascript
if (
  node.nodeType ===
  Node.ELEMENT_NODE
) {
}
```

---

## Mistake 2: Using `firstChild` when the first element is required

```javascript
const firstItem =
  list.firstChild;
```

This may return whitespace.

Use:

```javascript
const firstItem =
  list.firstElementChild;
```

---

## Mistake 3: Expecting `element.nodeValue` to return element text

```javascript
paragraph.nodeValue;
```

returns `null`.

Use:

```javascript
paragraph.textContent;
```

---

## Mistake 4: Confusing attributes with current live properties

The `value` attribute may differ from the current `input.value`.

---

## Mistake 5: Writing `disabled="false"`

The element remains disabled because the Boolean attribute is present.

Use:

```javascript
button.disabled = false;
```

---

## Mistake 6: Forgetting dataset values are strings

```javascript
const id =
  card.dataset.transactionId;
```

returns a string.

Convert when numeric behavior is required.

---

## Mistake 7: Modifying a live collection while iterating forward

Indexes shift as nodes are removed.

Convert to an array or use a safer removal strategy.

---

## Mistake 8: Assuming `children` includes text nodes

It includes only elements.

Use `childNodes` when every node type matters.

---

## Mistake 9: Replacing child elements unintentionally with `textContent`

```javascript
container.textContent =
  "Updated";
```

removes all existing descendants.

# 38. Best Practices

## 38.1 Use element APIs for element-focused work

Prefer:

```javascript
children
firstElementChild
nextElementSibling
parentElement
```

when text and comment nodes do not matter.

## 38.2 Use node APIs for generic document processing

Use:

```javascript
childNodes
firstChild
nodeType
nodeValue
```

when processing mixed node types.

## 38.3 Prefer named node constants

```javascript
Node.ELEMENT_NODE
```

is clearer than:

```javascript
1
```

## 38.4 Use `textContent` for plain text

It is clear and avoids parsing untrusted text as HTML.

## 38.5 Prefer DOM properties for live Boolean states

```javascript
button.disabled = true;
section.hidden = false;
checkbox.checked = true;
```

## 38.6 Use `data-*` for application metadata

Do not misuse visual classes as data storage.

## 38.7 Convert dataset values to the required type

```javascript
const amount =
  Number(
    element.dataset.amount
  );
```

## 38.8 Understand whether a collection is live

This prevents iteration bugs after DOM changes.

# 39. Performance Considerations

## Node collections

Repeatedly reading live collections can reflect DOM changes dynamically.

This is useful but can make code harder to reason about.

For stable processing, copy values:

```javascript
const children = [
  ...container.children,
];
```

## DOM traversal cost

Direct relationship properties are usually efficient:

```javascript
element.parentElement;
element.nextElementSibling;
```

They avoid searching the full document.

## Repeated text replacement

Setting `textContent` may remove and recreate child text structure.

This is efficient for replacing all content, but inappropriate when only one nested part should change.

## Attribute reads and writes

Attribute operations cross the JavaScript-to-DOM boundary.

They are normally inexpensive individually, but avoid unnecessary repeated work inside extremely large loops.

## Large DOM trees

Maintaining many nodes increases:

* Memory usage.
* Style calculation work.
* Layout complexity.
* Traversal costs.
* Garbage-collection pressure after removal.

# 41. Interview Questions and Answers

## Beginner Questions

### 1. What is a DOM node?

An object representing one structural unit in the document tree.

### 2. Is every node an element?

No. Text, comments, the document, and document fragments are also nodes.

### 3. What is an element node?

A node representing an HTML or XML element.

### 4. What is a text node?

A node representing textual content inside the document.

### 5. What does `nodeType` identify?

The type of a DOM node.

### 6. What does `textContent` do?

It reads or replaces the combined text content of a node and its descendants.

---

## Intermediate Questions

### 1. What is the difference between `children` and `childNodes`?

`children` contains only child elements. `childNodes` contains all child node types, including text and comments.

### 2. What is the difference between `firstChild` and `firstElementChild`?

`firstChild` returns any node type. `firstElementChild` returns only the first child element.

### 3. What is the difference between `parentNode` and `parentElement`?

`parentNode` can return any parent node type. `parentElement` returns the parent only if it is an element.

### 4. Why can `nextSibling` return a text node?

Whitespace between HTML elements is often represented as a text node.

### 5. What is the difference between an HTML attribute and a DOM property?

An attribute usually represents markup-level initial configuration, while a property often represents current live state.

### 6. What are Boolean attributes?

Attributes whose presence means true and absence means false.

---

## Advanced Questions

### 1. Why does `element.nodeValue` normally return `null`?

Because an element's content is stored in child nodes rather than directly in the element node's value.

### 2. Why may `input.value` differ from `input.getAttribute("value")`?

The property reflects the current live input value, while the attribute may preserve the initial markup value.

### 3. What is a live DOM collection?

A collection that automatically updates as the underlying DOM changes.

### 4. Which common collections are live?

`HTMLCollection` values such as `children`, and the `NodeList` returned by `childNodes`, are live.

### 5. Is the `NodeList` returned by `querySelectorAll()` live?

No. It is static.

### 6. Why can modifying a live collection during iteration be dangerous?

The collection changes immediately, shifting indexes and potentially causing items to be skipped or processed repeatedly.
