# 1. Introduction

Until now, most of the JavaScript code in this course has worked with values stored inside the JavaScript program.

Examples include:

```javascript
const user = {
  name: "Amina",
  role: "Admin",
};
```

```javascript
const services = [
  "Printing",
  "Binding",
  "Scanning",
];
```

```javascript
function calculateTotal(
  price,
  quantity
) {
  return price * quantity;
}
```

This JavaScript can:

* Calculate values.
* Store data.
* Transform arrays.
* Call functions.
* Handle asynchronous work.
* Communicate with APIs.

However, users do not normally interact directly with JavaScript variables.

Users interact with a **web page**.

They see:

* Headings.
* Buttons.
* Input fields.
* Tables.
* Images.
* Forms.
* Menus.
* Transaction cards.
* Loading indicators.
* Error messages.

JavaScript needs a way to access and control these visible page structures.

That bridge is the **Document Object Model**, commonly called the **DOM**.

The DOM allows JavaScript to:

```text
Read the page
    ↓
Find page elements
    ↓
Change text
    ↓
Create new content
    ↓
Remove content
    ↓
Respond to user actions
    ↓
Build interactive applications
```

Without the DOM, JavaScript could perform calculations, but it could not directly transform the web page users see.

---

# 2. What Is the DOM?

DOM stands for:

```text
Document Object Model
```

Each word is important.

## Document

The word **document** refers to the loaded web page.

For example:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>POS Dashboard</title>
  </head>

  <body>
    <h1>Daily Collections</h1>
  </body>
</html>
```

The browser loads this HTML document.

## Object

The browser converts parts of the document into JavaScript-accessible objects.

For example, the heading:

```html
<h1>Daily Collections</h1>
```

becomes an element object that JavaScript can access.

Conceptually:

```javascript
{
  tagName: "H1",
  textContent: "Daily Collections",
  children: [],
}
```

The real browser object contains many more properties and methods.

## Model

A **model** is a structured representation of something.

The DOM is not the HTML text itself.

It is the browser's structured, object-based representation of the document.

## Precise definition

> The DOM is a tree-structured object representation of a document that allows programming languages such as JavaScript to read, traverse, modify, create, and remove document content.

---

# 3. Why Was the DOM Created?

HTML was originally designed to describe document structure.

```html
<h1>My Website</h1>

<p>Welcome to the website.</p>
```

This is useful for creating static documents.

However, modern applications need pages that can change after loading.

For example:

* Add a product to a cart.
* Display newly fetched transactions.
* Show a validation error.
* Open a modal.
* Hide a loading spinner.
* Update a total.
* Change a button label.
* Remove a deleted record.
* Display search results.
* Add form fields dynamically.

JavaScript needed a standard way to communicate with document structure.

The DOM was created to provide that interface.

```text
HTML document
    ↓
Browser converts it to objects
    ↓
JavaScript accesses those objects
    ↓
Page changes
```

---

# 4. What Problem Does the DOM Solve?

Suppose this HTML exists:

```html
<h1>Daily Collections</h1>
```

The application receives a selected date from the user.

The heading should become:

```text
Collections for 15/07/2026
```

Without the DOM, JavaScript has no standard connection to the `<h1>`.

With the DOM:

```javascript
const heading =
  document.querySelector("h1");

heading.textContent =
  "Collections for 15/07/2026";
```

The DOM solves several major problems.

## Problem 1: Finding document content

```javascript
document.querySelector("#total");
```

## Problem 2: Reading content

```javascript
element.textContent;
```

## Problem 3: Changing content

```javascript
element.textContent = "KES 5,000";
```

## Problem 4: Creating content

```javascript
document.createElement("li");
```

## Problem 5: Removing content

```javascript
element.remove();
```

## Problem 6: Responding to interaction

```javascript
button.addEventListener(
  "click",
  handleClick
);
```

## Problem 7: Updating styles and classes

```javascript
element.classList.add("active");
```

The DOM provides a standard programming interface for all of these operations.

---

# 5. How Web Pages Worked Without Dynamic DOM Interaction

Before advanced client-side scripting became common, most web pages were primarily static.

A typical workflow looked like this:

```text
User opens page
    ↓
Server generates complete HTML
    ↓
Browser displays HTML
    ↓
User clicks a link or submits a form
    ↓
Browser requests another complete page
    ↓
Server generates new HTML
    ↓
Browser replaces the old page
```

To update one small section, the browser often loaded an entirely new document.

Modern DOM-based applications can instead update only the affected part.

```text
User clicks Add Item
    ↓
JavaScript updates state
    ↓
JavaScript creates one DOM element
    ↓
Browser displays the new item
```

This enables:

* Faster interactions.
* Rich user interfaces.
* Single-page applications.
* Live validation.
* Dynamic dashboards.
* Real-time data updates.
* Interactive forms.

> [!NOTE]
> Server-rendered pages remain important. Modern applications frequently combine server rendering with DOM-based client-side interaction.

# 6. HTML Is Not the DOM

This distinction is fundamental.

## HTML

HTML is text written using markup syntax.

```html
<p>Hello</p>
```

It may exist:

* In a source file.
* In a server response.
* In the browser's original page source.
* As a string inside JavaScript.

## DOM

The DOM is the browser's in-memory object representation after parsing the HTML.

```text
HTML text
    ↓ parser
DOM objects
```

Conceptually:

```javascript
HTMLParagraphElement {
  textContent: "Hello"
}
```

## Example

Source HTML:

```html
<body>
  <h1>Welcome</h1>
</body>
```

DOM tree:

```text
Document
└── html
    ├── head
    └── body
        └── h1
            └── "Welcome"
```

The HTML is input.

The DOM is the resulting model.

---

# 7. The DOM as an Object Model

The DOM represents document pieces as objects.

Consider:

```html
<button id="save-button">
  Save
</button>
```

The browser creates a button element object.

JavaScript can access properties:

```javascript
button.id;
button.textContent;
button.disabled;
```

JavaScript can call methods:

```javascript
button.click();
button.remove();
button.setAttribute(
  "aria-label",
  "Save transaction"
);
```

JavaScript can register event listeners:

```javascript
button.addEventListener(
  "click",
  saveTransaction
);
```

The DOM is called an object model because page components are represented as related objects with:

* Properties.
* Methods.
* Parent-child relationships.
* Events.
* State.

# 8. The Browser as a Runtime Environment

JavaScript alone does not provide the DOM.

The browser provides it.

The JavaScript language defines features such as:

* Variables.
* Functions.
* Objects.
* Arrays.
* Promises.
* Classes.
* Loops.

The browser environment provides features such as:

* `window`
* `document`
* DOM APIs
* Timers
* Events
* `fetch`
* `localStorage`
* Navigation
* History
* Rendering

Conceptual architecture:

```text
Browser
├── JavaScript Engine
├── HTML Parser
├── CSS Parser
├── DOM
├── CSSOM
├── Rendering Engine
├── Networking
├── Event System
└── Browser APIs
```

This explains why DOM code normally works in a browser but not automatically in a plain JavaScript runtime with no document.

For example:

```javascript
document.querySelector("h1");
```

depends on the browser-provided `document` object.

# 9. Browser Rendering Overview

A browser does much more than simply "show HTML."

It performs a multi-stage process.

A simplified pipeline is:

```text
HTML
    ↓
Parse HTML
    ↓
Build DOM

CSS
    ↓
Parse CSS
    ↓
Build CSSOM

DOM + CSSOM
    ↓
Render Tree
    ↓
Layout
    ↓
Paint
    ↓
Composite
    ↓
Pixels on screen
```

JavaScript can influence many stages by changing:

* DOM structure.
* Text content.
* CSS classes.
* Inline styles.
* Attributes.
* Element dimensions.

Understanding the rendering pipeline helps explain why some DOM operations are expensive.

---

# 10. Step 1: Receiving HTML

The browser may receive HTML from:

* A server response.
* A local file.
* Generated application content.
* A browser extension.
* An embedded document.

Example response:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>POS</title>
  </head>

  <body>
    <h1>Daily Collections</h1>
  </body>
</html>
```

The browser receives this as a stream of bytes.

Simplified transformation:

```text
Network bytes
    ↓
Decode characters
    ↓
HTML text
```

The browser must then understand the structure represented by that text.

---

# 11. Step 2: HTML Tokenization

The HTML tokenizer reads the text and identifies meaningful tokens.

Example input:

```html
<h1>Daily Collections</h1>
```

Possible conceptual tokens:

```text
StartTag: h1
Text: Daily Collections
EndTag: h1
```

For:

```html
<button id="save">Save</button>
```

Conceptual tokens:

```text
StartTag:
  name: button
  attribute:
    id = save

Text:
  Save

EndTag:
  button
```

Tokenization converts raw characters into structured pieces the parser can understand.

---

# 12. Step 3: HTML Parsing

The parser processes tokens according to HTML parsing rules.

It determines:

* Which elements are created.
* Which element becomes the parent.
* How malformed markup is corrected.
* Where text nodes belong.
* When parsing enters special modes.
* How scripts affect parsing.

Example:

```html
<body>
  <h1>Title</h1>
  <p>Description</p>
</body>
```

The parser understands that:

* `<h1>` belongs inside `<body>`.
* Its text belongs inside `<h1>`.
* `<p>` is a sibling of `<h1>`.

---

# 13. Step 4: Building the DOM Tree

As tokens are parsed, the browser creates DOM nodes.

HTML:

```html
<body>
  <h1>Daily Collections</h1>
  <p>Welcome</p>
</body>
```

DOM tree:

```text
body
├── h1
│   └── "Daily Collections"
└── p
    └── "Welcome"
```

Each item is a node.

This includes:

* The body element.
* The heading element.
* The paragraph element.
* The text inside each element.

The DOM tree is built incrementally while HTML is parsed.

It is not always necessary to wait for the entire document before some DOM objects exist.

---

# 14. Step 5: Parsing CSS and Building the CSSOM

CSS is also parsed into an object structure.

CSS:

```css
h1 {
  color: teal;
}

p {
  font-size: 16px;
}
```

The browser creates a CSS Object Model, commonly called the **CSSOM**.

Conceptually:

```text
CSSOM
├── Rule: h1
│   └── color: teal
└── Rule: p
    └── font-size: 16px
```

The DOM describes document structure.

The CSSOM describes style rules.

```text
DOM → What exists
CSSOM → How it should look
```

---

# 15. Step 6: Creating the Render Tree

The browser combines relevant DOM and CSSOM information to create a render tree.

```text
DOM + CSSOM
    ↓
Render Tree
```

The render tree contains visual objects that need to be displayed.

Not every DOM node necessarily appears in the render tree.

For example:

```css
.hidden {
  display: none;
}
```

An element with `display: none` remains in the DOM but does not generate a visible render-tree box.

```text
DOM:
Element exists

Render Tree:
Element excluded
```

This distinction is important.

The DOM represents document structure, not only currently visible pixels.

---

# 16. Step 7: Layout

Layout determines geometry.

The browser calculates:

* Width.
* Height.
* Position.
* Margins.
* Padding.
* Borders.
* Line wrapping.
* Element coordinates.
* Relationships between boxes.

Example:

```text
Header:
x = 0
y = 0
width = 1280
height = 70
```

```text
Button:
x = 1050
y = 15
width = 120
height = 40
```

Layout is sometimes called **reflow**.

Changes that affect geometry may force layout recalculation.

Examples:

* Changing width.
* Changing height.
* Adding elements.
* Removing elements.
* Changing font size.
* Changing display mode.
* Reading certain layout measurements after writes.

---

# 17. Step 8: Paint

Painting converts visual instructions into drawing operations.

The browser paints things such as:

* Text.
* Backgrounds.
* Borders.
* Shadows.
* Images.
* Icons.
* Decorations.

Conceptually:

```text
Render object
    ↓
Paint instructions
    ↓
Draw background
    ↓
Draw border
    ↓
Draw text
```

Changing visual properties may cause repainting.

Examples:

* Text color.
* Background color.
* Border color.
* Box shadow.

A repaint may not require a full layout if geometry did not change.

---

# 18. Step 9: Compositing

Modern browsers may paint parts of the page into separate layers.

The compositor combines those layers into the final image.

Examples of elements that may receive separate layers include:

* Transformed elements.
* Animated elements.
* Fixed-position elements.
* Video.
* Canvas.
* Certain opacity effects.

Conceptual process:

```text
Layer 1: Page background
Layer 2: Header
Layer 3: Modal
Layer 4: Animation
    ↓
Composite layers
    ↓
Final screen image
```

Compositing can make certain animations smoother because the browser can move a layer without repainting every pixel.

# 19. Complete Browser Rendering Pipeline

A simplified complete diagram:

```text
Server or local source
        ↓
HTML bytes
        ↓
Decode
        ↓
HTML tokens
        ↓
HTML parser
        ↓
DOM Tree

CSS files and style blocks
        ↓
CSS parser
        ↓
CSSOM

DOM + CSSOM
        ↓
Render Tree
        ↓
Layout
        ↓
Paint
        ↓
Composite
        ↓
Pixels on screen
```

JavaScript interaction:

```text
JavaScript
    ↓
Read or modify DOM
    ↓
Read or modify styles
    ↓
Possible style recalculation
    ↓
Possible layout
    ↓
Possible paint
    ↓
Possible compositing
```

> [!IMPORTANT]
> Not every DOM change triggers every rendering stage. The exact work depends on what changed.

# 20. What Is a Tree Data Structure?

A tree is a hierarchical data structure.

It contains nodes connected through parent-child relationships.

Example:

```text
Root
├── Child A
│   ├── Grandchild A1
│   └── Grandchild A2
└── Child B
```

Important terminology:

* **Root**: the topmost node.
* **Parent**: a node containing another node.
* **Child**: a node directly inside another node.
* **Sibling**: nodes sharing the same parent.
* **Ancestor**: any node higher in the hierarchy.
* **Descendant**: any node lower in the hierarchy.
* **Leaf**: a node with no children.

The DOM uses this structure because HTML is hierarchical.

```html
<body>
  <section>
    <h1>Title</h1>
  </section>
</body>
```

Relationships:

```text
body
└── section
    └── h1
        └── text
```

# 21. Understanding the DOM Tree

Consider:

```html
<!DOCTYPE html>
<html>
  <head>
    <title>Dashboard</title>
  </head>

  <body>
    <main>
      <h1>Daily Collections</h1>

      <button>
        Refresh
      </button>
    </main>
  </body>
</html>
```

Simplified DOM tree:

```text
Document
├── DocumentType: html
└── html
    ├── head
    │   └── title
    │       └── "Dashboard"
    └── body
        └── main
            ├── h1
            │   └── "Daily Collections"
            └── button
                └── "Refresh"
```

Notice that text becomes separate text nodes.

The heading element is not the same node as the text inside it.

```text
h1 element
└── text node
```

This distinction becomes important when navigating nodes.

# 22. The `document` Object

The browser exposes the current DOM through the global `document` object.

```javascript
console.log(document);
```

The `document` object represents the loaded document.

It provides access to:

* The document element.
* Head.
* Body.
* Elements.
* Node creation.
* Selection methods.
* Event registration.
* Metadata.
* Document state.

Examples:

```javascript
document.documentElement;
```

Returns the `<html>` element.

```javascript
document.head;
```

Returns the `<head>` element.

```javascript
document.body;
```

Returns the `<body>` element.

```javascript
document.title;
```

Reads or changes the page title.

```javascript
document.createElement("div");
```

Creates a new element.

```javascript
document.querySelector("h1");
```

Finds an element.

# 23. The `window` Object

In browsers, `window` represents the browser window or tab's global environment.

```javascript
console.log(window);
```

It contains many browser APIs.

Examples:

```javascript
window.document;
window.location;
window.history;
window.localStorage;
window.setTimeout;
window.fetch;
```

In classic browser scripts, global variables declared with certain syntax may become `window` properties, although modern module behavior differs.

The important relationship is:

```javascript
window.document === document;
```

Output:

```text
true
```

The `document` object belongs to the `window`.

---

# 24. `window` vs `document`

| `window`                                  | `document`                         |
| ----------------------------------------- | ---------------------------------- |
| Represents the browser global environment | Represents the loaded web document |
| Contains browser APIs                     | Contains DOM APIs                  |
| Has `location`, `history`, timers         | Has `body`, `head`, elements       |
| Top-level browser object                  | Property of `window`               |
| Represents tab/window context             | Represents page structure          |

Conceptual relationship:

```text
window
├── document
│   └── DOM tree
├── location
├── history
├── localStorage
├── fetch
└── timers
```

Example:

```javascript
window.document.body;
```

is normally equivalent to:

```javascript
document.body;
```

# 25. HTML Document to DOM Tree Example

## HTML

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8">

    <title>
      Revenue Collection
    </title>
  </head>

  <body>
    <header>
      <h1>
        Daily Collections
      </h1>
    </header>

    <main>
      <section id="summary">
        <p>
          Total: KES 5,000
        </p>
      </section>
    </main>
  </body>
</html>
```

## DOM tree

```text
Document
├── DocumentType: html
└── html
    ├── head
    │   ├── meta
    │   └── title
    │       └── "Revenue Collection"
    └── body
        ├── header
        │   └── h1
        │       └── "Daily Collections"
        └── main
            └── section
                └── p
                    └── "Total: KES 5,000"
```

## Attribute representation

The section contains:

```html
id="summary"
```

The `id` is associated with the element.

JavaScript can read it:

```javascript
const summary =
  document.querySelector(
    "#summary"
  );

console.log(summary.id);
```

Output:

```text
summary
```

# 26. The DOM Is Live and Mutable

The DOM exists in memory while the page is open.

JavaScript can modify it.

Initial HTML:

```html
<h1 id="title">
  Daily Collections
</h1>
```

JavaScript:

```javascript
const title =
  document.querySelector(
    "#title"
  );

title.textContent =
  "Monthly Collections";
```

The DOM changes:

```text
Before:
h1
└── "Daily Collections"

After:
h1
└── "Monthly Collections"
```

The browser then updates what the user sees.

The original HTML file on the server does not automatically change.

Only the current document in the browser is modified.

# 27. Source HTML vs Current DOM

These are not always identical.

## Original source

```html
<p id="status">
  Loading
</p>
```

## JavaScript modification

```javascript
const status =
  document.querySelector(
    "#status"
  );

status.textContent =
  "Complete";
```

## Current DOM

```html
<p id="status">
```

# 28. Automatic Browser Corrections

Browsers may correct incomplete or invalid HTML while building the DOM.

Example source:

```html
<table>
  <tr>
    <td>Item</td>
  </tr>
</table>
```

The browser may insert a `<tbody>` element in the DOM.

Resulting DOM:

```html
<table>
  <tbody>
    <tr>
      <td>Item</td>
    </tr>
  </tbody>
</table>
```

Another example:

```html
<body>
  <p>Hello
  <div>World</div>
</body>
```

Because a `<div>` cannot remain inside a paragraph in that way, the parser may close the paragraph automatically.

The DOM reflects browser parsing rules, not necessarily the exact text formatting of the source.

> [!WARNING]
> Never assume malformed HTML produces the exact tree you intended. Use valid semantic HTML and inspect the actual DOM.

# 29. DOM Lifecycle

The document passes through several loading stages.

Simplified lifecycle:

```text
Browser begins loading
    ↓
HTML parsing begins
    ↓
DOM constructed progressively
    ↓
DOM parsing completes
    ↓
DOMContentLoaded
    ↓
Images and other resources finish
    ↓
load event
```

JavaScript must sometimes wait until required DOM elements exist.

If code runs too early:

```javascript
const button =
  document.querySelector(
    "#save-button"
  );
```

it may return:

```text
null
```

because the browser has not parsed that button yet.

# 30. `document.readyState`

The property:

```javascript
document.readyState
```

describes document loading status.

Common values:

| Value           | Meaning                                     |
| --------------- | ------------------------------------------- |
| `"loading"`     | HTML is still being parsed                  |
| `"interactive"` | DOM parsing is complete                     |
| `"complete"`    | Document and dependent resources are loaded |

Example:

```javascript
console.log(
  document.readyState
);
```

Possible output:

```text
loading
```

or:

```text
interactive
```

or:

```text
complete
```

# 31. `DOMContentLoaded`

The `DOMContentLoaded` event fires after the HTML document has been completely parsed and the DOM has been constructed.

```javascript
document.addEventListener(
  "DOMContentLoaded",
  () => {
    console.log(
      "DOM is ready"
    );
  }
);
```

It does not necessarily wait for every image or other external resource to finish loading.

Use it when code needs the DOM structure to exist.

Example:

```javascript
document.addEventListener(
  "DOMContentLoaded",
  () => {
    const button =
      document.querySelector(
        "#save-button"
      );

    console.log(button);
  }
);
```

# 32. `load`

The `load` event usually fires later than `DOMContentLoaded`.

```javascript
window.addEventListener(
  "load",
  () => {
    console.log(
      "Page resources loaded"
    );
  }
);
```

It waits for resources such as:

* Images.
* Stylesheets.
* Certain external resources.
* The complete page load process.

Comparison:

```text
DOMContentLoaded
    ↓
DOM structure ready

load
    ↓
DOM and page resources ready
```

Use `DOMContentLoaded` for most DOM initialization.

Use `load` when the code specifically depends on resource completion or dimensions that require loaded assets.

# 33. Script Placement and Parsing

Consider:

```html
<head>
  <script src="app.js"></script>
</head>

<body>
  <button id="save-button">
    Save
  </button>
</body>
```

A normal script encountered during HTML parsing may pause parsing.

```text
Parse HTML
    ↓
Encounter script
    ↓
Fetch script if necessary
    ↓
Execute script
    ↓
Continue parsing
```

If `app.js` tries to find the button before the parser reaches it:

```javascript
const button =
  document.querySelector(
    "#save-button"
  );

console.log(button);
```

Output may be:

```text
null
```

One older solution was placing scripts near the end of `<body>`:

```html
<body>
  <button id="save-button">
    Save
  </button>

  <script src="app.js"></script>
</body>
```

Now the button is parsed first.

Modern HTML also provides `defer`.

# 34. `defer` and `async`

## Normal script

```html
<script src="app.js"></script>
```

May block HTML parsing while fetched and executed.

## Deferred script

```html
<script
  src="app.js"
  defer
></script>
```

Behavior:

* Downloads while HTML parsing continues.
* Executes after HTML parsing completes.
* Preserves document order among deferred scripts.
* Usually ideal for application scripts that depend on the DOM.

Timeline:

```text
Parse HTML ───────────────┐
Download deferred script ─┤
                         ↓
HTML parsing complete
                         ↓
Execute deferred script
                         ↓
DOMContentLoaded
```

## Async script

```html
<script
  src="analytics.js"
  async
></script>
```

Behavior:

* Downloads while parsing continues.
* Executes as soon as download finishes.
* May interrupt parsing.
* Execution order between async scripts is not guaranteed.

Timeline:

```text
Parse HTML
    ↓
Async script finishes downloading
    ↓
Pause parsing
    ↓
Execute script
    ↓
Continue parsing
```

Use cases:

* Analytics.
* Independent third-party scripts.
* Scripts that do not depend on DOM readiness or other scripts.

## Comparison

| Script type |               Downloads during parsing | Blocks parsing during execution | Ordered | DOM ready before execution |
| ----------- | -------------------------------------: | ------------------------------: | ------: | -------------------------: |
| Normal      | No or browser-dependent fetch behavior |                             Yes |     Yes |             Not guaranteed |
| `defer`     |                                    Yes |      No until parsing completes |     Yes |                        Yes |
| `async`     |                                    Yes |                  Yes when ready |      No |             Not guaranteed |

# 35. Internal JavaScript Engine and DOM Interaction

The JavaScript engine and rendering engine cooperate.

Consider:

```javascript
const heading =
  document.querySelector(
    "#title"
  );

heading.textContent =
  "Updated Title";
```

Conceptual steps:

```text
JavaScript calls querySelector
    ↓
Browser DOM API searches DOM tree
    ↓
Returns element object reference
    ↓
JavaScript assigns textContent
    ↓
DOM text node changes
    ↓
Browser marks visual state dirty
    ↓
Rendering work scheduled
    ↓
Updated pixels displayed
```

The DOM element is a host object supplied by the browser environment.

JavaScript holds a reference to that object.

```text
JavaScript variable
    ↓
Reference
    ↓
DOM element object
    ↓
Browser-managed document node
```

Changing a property may affect browser rendering.

# 37. Common Mistakes

## Mistake 1: Thinking HTML and DOM are identical

HTML is source markup.

The DOM is the parsed, live object model.

---

## Mistake 2: Accessing elements before they exist

```javascript
const button =
  document.querySelector(
    "#save-button"
  );
```

may return `null` when the script executes before the button is parsed.

Solutions include:

* Use `defer`.
* Place the script after the required HTML.
* Wait for `DOMContentLoaded`.

---

## Mistake 3: Using `window` and `document` interchangeably

`window` represents the browser environment.

`document` represents the loaded page.

---

## Mistake 4: Assuming every DOM node is visible

Nodes may exist in the DOM but not appear in the render tree.

Examples:

* `display: none`.
* Metadata nodes.
* Script nodes.
* Elements outside the viewport.
* Hidden elements.

---

## Mistake 5: Assuming browser parsing preserves malformed HTML exactly

Browsers may insert, close, or rearrange elements according to parsing rules.

---

## Mistake 6: Waiting for `load` when only the DOM is required

`DOMContentLoaded` or `defer` is often sufficient and earlier.

---

## Mistake 7: Using `async` for scripts that depend on document order

Async scripts execute whenever they finish downloading.

Use `defer` when order and DOM readiness matter.

---

## Mistake 8: Confusing the DOM tree with the render tree

The DOM contains document nodes.

The render tree contains visual render objects.

---

## Mistake 9: Expecting DOM changes to modify the server file

Changing:

```javascript
element.textContent = "Updated";
```

changes only the current browser document unless the application sends data to a server.

# 39. Performance Considerations

DOM access is generally more expensive than manipulating ordinary JavaScript values.

Compare:

```javascript
let total = 0;

total += 500;
```

with:

```javascript
document.querySelector(
  "#total"
).textContent = "500";
```

The second operation crosses from JavaScript into browser-managed DOM and may trigger rendering work.

## DOM tree size

A very large DOM may increase:

* Selector costs.
* Style calculation time.
* Layout time.
* Memory usage.
* Rendering complexity.

## Frequent updates

Repeatedly updating the DOM may produce unnecessary rendering work.

Less efficient:

```javascript
for (const item of items) {
  list.innerHTML +=
    `<li>${item}</li>`;
}
```

This may repeatedly parse and rebuild content.

Better strategies will be taught in the DOM Manipulation module.

## Layout-related reads and writes

Mixing layout writes and reads repeatedly may force the browser to perform synchronous layout calculations.

Example pattern to avoid:

```text
Write size
Read size
Write size
Read size
```

Prefer grouping writes and reads where practical.

## Render pipeline awareness

Different changes may trigger different work:

```text
Text change
    ↓
Possible layout + paint

Color change
    ↓
Usually paint

Transform change
    ↓
May use compositing
```

Exact browser behavior can vary.

# 41. Interview Questions and Answers

## Beginner Questions

### 1. What does DOM stand for?

Document Object Model.

### 2. What is the DOM?

A tree-structured object representation of a document that programming languages can read and modify.

### 3. Is HTML the same as the DOM?

No. HTML is source markup. The DOM is the browser's parsed object representation.

### 4. What does the `document` object represent?

The currently loaded web document.

### 5. What is the root element of an HTML document?

The `<html>` element, accessible through:

```javascript
document.documentElement
```

### 6. What is a DOM tree?

A hierarchy of nodes representing document structure.

---

## Intermediate Questions

### 1. What is the difference between `window` and `document`?

`window` represents the browser global environment. `document` represents the page loaded inside it.

### 2. What is `DOMContentLoaded`?

An event fired after the HTML has been parsed and the DOM has been constructed.

### 3. What is the difference between `DOMContentLoaded` and `load`?

`DOMContentLoaded` waits for DOM parsing. `load` waits for the page and dependent resources to finish loading.

### 4. Why can `querySelector()` return `null`?

No matching element exists at the time of selection, often because of an incorrect selector or script timing.

### 5. What is the CSSOM?

The browser's object representation of parsed CSS rules.

### 6. What is the render tree?

A visual tree built from relevant DOM nodes and computed CSS information.

---

## Advanced Questions

### 1. Describe the browser rendering pipeline.

HTML is parsed into the DOM, CSS into the CSSOM, both contribute to the render tree, then layout calculates geometry, paint draws visual content, and compositing combines layers.

### 2. Why might the DOM differ from the original HTML source?

JavaScript may modify it, and the HTML parser may correct or insert elements according to browser parsing rules.

### 3. Does every DOM node appear in the render tree?

No. Some nodes are non-visual, and elements such as those with `display: none` are excluded from the visual render tree.

### 4. Why is `defer` often preferred for application scripts?

It allows parallel downloading, preserves script order, avoids blocking HTML parsing, and executes after the document has been parsed.

### 5. How can DOM modifications affect browser performance?

They may trigger style recalculation, layout, paint, or compositing work.

### 6. Why are browser DOM objects called host objects?

They are supplied by the browser environment rather than defined solely by the core JavaScript language.

# 45. Summary

The Document Object Model is the bridge between JavaScript and the web page.

HTML provides source markup:

```html

```

The browser parses it into a DOM tree:

```text
h1
└── "Hello"
```

JavaScript accesses the DOM through:

```javascript
document
```

The browser rendering process can be summarized as:

```text
HTML
    ↓
DOM

CSS
    ↓
CSSOM

DOM + CSSOM
    ↓
Render Tree
    ↓
Layout
    ↓
Paint
    ↓
Composite
    ↓
Screen
```

You learned that:

* HTML and the DOM are different.
* The DOM is a live object model.
* The DOM uses a tree structure.
* The browser may correct malformed HTML.
* `window` represents the browser environment.
* `document` represents the loaded page.
* DOM construction happens during parsing.
* `DOMContentLoaded` means DOM parsing is complete.
* `load` waits for additional resources.
* `defer` is usually appropriate for application scripts.
* `async` is appropriate for independent scripts.
* DOM changes may trigger browser rendering work.
* The Elements panel displays the current live DOM.
