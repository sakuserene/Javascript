# 1. Introduction

You have learned how to traverse the DOM correctly.

You can now:

* Move upward with `closest()`.
* Move downward with `querySelector()`.
* inspect direct children with `children`.
* Move sideways with sibling properties.
* Enforce component boundaries.
* Resolve action context.
* Manage focus after removal.
* Handle nested components.

These skills are sufficient for small scripts.

However, production applications usually contain:

* Many repeated components.
* Many selectors.
* Dynamic insertion and removal.
* Multiple forms.
* Multiple tables.
* Shared traversal rules.
* Event delegation.
* Cached references.
* Business logic.
* Application state.
* Error handling.
* Testing requirements.

Without architecture, DOM traversal becomes scattered.

Example:

```javascript
document
  .querySelector(
    ".transaction-card .amount"
  )
  .textContent = "KES 500";
```

Elsewhere:

```javascript
event.target
  .parentElement
  .parentElement
  .querySelector(
    ".status"
  )
  .textContent = "Paid";
```

Elsewhere:

```javascript
const card =
  document.querySelector(
    `[data-id="${id}"]`
  );
```

Each piece may work individually, but the application becomes fragile.

Production architecture organizes traversal into:

```text
Stable component roots
    ↓
Centralized selectors
    ↓
Reusable query helpers
    ↓
Component view objects
    ↓
Structured action context
    ↓
Business logic
```

---

# 2. Why Traversal Architecture Matters

Traversal architecture matters because DOM structure changes.

A designer may:

* Add a wrapper.
* Rename a visual class.
* Move an action button.
* Add a nested component.
* Replace component content.
* Reorder table columns.
* Add accessibility markup.

If traversal code depends on exact presentation structure, every change breaks multiple files.

Fragile:

```javascript
const amount =
  card.children[1]
    .children[0];
```

More stable:

```javascript
const amount =
  queryRequired(
    card,
    "[data-role='amount']"
  );
```

Architecture reduces the number of places that know about raw markup structure.

---

# 3. The Problem with Scattered DOM Queries

Consider a transaction application.

```javascript
const amount =
  document.querySelector(
    ".amount"
  );
```

```javascript
const receipt =
  document.querySelector(
    ".receipt"
  );
```

```javascript
const status =
  document.querySelector(
    ".status"
  );
```

Problems:

1. Queries may select the wrong component.
2. Selectors are duplicated.
3. Missing elements fail inconsistently.
4. Structural assumptions are undocumented.
5. Tests become difficult.
6. Dynamic replacements make references stale.
7. Business logic becomes tightly coupled to HTML.
8. Refactoring selectors requires many edits.

Architecture centralizes these assumptions.

---

# 4. Mental Model: The DOM as a View Layer

Application architecture can be simplified into three layers.

```text
Application State
    ↓
Business Logic
    ↓
DOM View Layer
```

## Application state

```javascript
{
  id: 42,
  amount: 500,
  status: "pending",
}
```

## Business logic

```javascript
markTransactionPaid(
  transaction
);
```

## DOM view layer

```javascript
transactionView
  .setStatus("paid");
```

The DOM view layer knows:

* Where elements are.
* How to find component parts.
* How to read visible values.
* How to expose component references.

Business logic should not repeatedly know raw selectors.

---

# 5. Traversal Architecture Goals

A strong architecture should provide:

## Locality

Queries remain inside the correct component.

## Predictability

Required elements either exist or fail clearly.

## Reusability

Shared traversal behavior is implemented once.

## Maintainability

Selectors and structural contracts are centralized.

## Dynamic support

Inserted and removed components can be handled.

## Testability

Helpers and component contracts can be tested independently.

## Separation of concerns

Traversal finds DOM context.

Business logic applies application rules.

# 6. Component Roots as Architectural Boundaries

A component root should be clearly identifiable.

```html
<article
  data-component="transaction-card"
  data-transaction-id="42"
>
```

The component root defines:

* Where queries begin.
* Which metadata belongs to the component.
* Which descendants are owned.
* Where event context resolves.
* Where nested components begin.

Example:

```javascript
function getTransactionCard(
  element
) {
  return element.closest(
    "[data-component='transaction-card']"
  );
}
```

Component boundaries reduce global DOM dependence.

---

# 7. Component Contracts

A component contract describes required structure.

Example transaction-card contract:

```text
Root:
[data-component="transaction-card"]

Required:
[data-role="receipt"]
[data-role="amount"]
[data-role="status"]
[data-role="actions"]

Optional:
[data-role="error"]

Metadata:
data-transaction-id
data-status
```

Represent it in JavaScript:

```javascript
const transactionCardContract = {
  root:
    "[data-component='transaction-card']",

  required: {
    receipt:
      "[data-role='receipt']",

    amount:
      "[data-role='amount']",

    status:
      "[data-role='status']",

    actions:
      "[data-role='actions']",
  },

  optional: {
    error:
      "[data-role='error']",
  },
};
```

The contract becomes the single source of truth for traversal.

---

# 8. Semantic Selectors

Semantic selectors describe purpose.

Prefer:

```html
data-role="amount"
```

over:

```html
class="text-large text-bold mt-2"
```

Visual classes may change during redesign.

Behavior roles should remain stable.

Common semantic attributes:

```html
data-component
data-role
data-action
data-field
data-column
data-status
```

Example:

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

This communicates meaning directly.

---

# 9. Centralizing Selectors

Instead of repeating strings:

```javascript
const selectors = {
  transactionCard: {
    root:
      "[data-component='transaction-card']",

    receipt:
      "[data-role='receipt']",

    amount:
      "[data-role='amount']",

    status:
      "[data-role='status']",

    action:
      "[data-action]",
  },
};
```

Usage:

```javascript
const amount =
  card.querySelector(
    selectors
      .transactionCard
      .amount
  );
```

Benefits:

* Easier refactoring.
* Fewer typing inconsistencies.
* Clear component vocabulary.
* Improved testing.
* Better code search.

Do not create an unnecessarily huge global selector object.

Group selectors by component or module.

# 10. Required Query Utilities

```javascript
function queryRequired(
  root,
  selector,
  description = selector
) {
  if (
    !root ||
    typeof root.querySelector !==
      "function"
  ) {
    throw new TypeError(
      "A valid query root is required."
    );
  }

  const element =
    root.querySelector(
      selector
    );

  if (!element) {
    throw new Error(
      `Required DOM element was not found: ${description}`
    );
  }

  return element;
}
```

Use:

```javascript
const amount =
  queryRequired(
    card,
    selectors
      .transactionCard
      .amount,
    "transaction amount"
  );
```

Required helpers turn hidden assumptions into explicit contracts.

---

# 11. Optional Query Utilities

```javascript
function queryOptional(
  root,
  selector
) {
  if (
    !root ||
    typeof root.querySelector !==
      "function"
  ) {
    throw new TypeError(
      "A valid query root is required."
    );
  }

  return root.querySelector(
    selector
  );
}
```

Use:

```javascript
const errorMessage =
  queryOptional(
    form,
    "[data-role='error-summary']"
  );
```

Optional helpers still validate the root while allowing absence.

# 12. Collection Query Utilities

```javascript
function queryAll(
  root,
  selector
) {
  if (
    !root ||
    typeof root.querySelectorAll !==
      "function"
  ) {
    throw new TypeError(
      "A valid query root is required."
    );
  }

  return [
    ...root.querySelectorAll(
      selector
    ),
  ];
}
```

This returns a stable Array.

Use:

```javascript
const actions =
  queryAll(
    card,
    "[data-action]"
  );
```

Advantages:

* Array methods.
* Static snapshot.
* Predictable mutation.
* Easier testing.

---

# 13. Direct-Child Query Utilities

```javascript
function queryDirectChildren(
  root,
  selector = "*"
) {
  return [
    ...root.children,
  ].filter(
    child =>
      child.matches(
        selector
      )
  );
}
```

Or using `:scope`:

```javascript
function queryDirectChildren(
  root,
  selector
) {
  return [
    ...root.querySelectorAll(
      `:scope > ${selector}`
    ),
  ];
}
```

The first version avoids building selector strings.

Use direct-child utilities when component ownership depends on one-level structure.

# 14. Component View Objects

A component view object wraps DOM references and exposes meaningful operations.

Instead of:

```javascript
card.querySelector(
  "[data-role='status']"
).textContent = "Paid";
```

Use:

```javascript
transactionView
  .setStatus("Paid");
```

Example:

```javascript
function createTransactionCardView(
  root
) {
  const receipt =
    queryRequired(
      root,
      "[data-role='receipt']"
    );

  const amount =
    queryRequired(
      root,
      "[data-role='amount']"
    );

  const status =
    queryRequired(
      root,
      "[data-role='status']"
    );

  return {
    root,
    receipt,
    amount,
    status,

    getId() {
      return Number(
        root.dataset
          .transactionId
      );
    },

    setStatus(value) {
      status.textContent =
        value;

      root.dataset.status =
        value.toLowerCase();
    },
  };
}
```

The view object hides traversal details from callers.

---

# 15. Creating a View Factory

A view factory validates and constructs a component interface.

```javascript
function createViewFactory(
  contract
) {
  return function createView(
    root
  ) {
    if (
      !root.matches(
        contract.root
      )
    ) {
      throw new Error(
        "Invalid component root."
      );
    }

    const required = {};

    for (
      const [
        name,
        selector,
      ]
      of Object.entries(
        contract.required
      )
    ) {
      required[name] =
        queryRequired(
          root,
          selector,
          name
        );
    }

    const optional = {};

    for (
      const [
        name,
        selector,
      ]
      of Object.entries(
        contract.optional ??
          {}
      )
    ) {
      optional[name] =
        root.querySelector(
          selector
        );
    }

    return {
      root,
      required,
      optional,
    };
  };
}
```

This turns component contracts into reusable view creators.

---

# 16. Cached DOM References

Caching means querying once and storing the result.

```javascript
const amount =
  card.querySelector(
    "[data-role='amount']"
  );
```

Later:

```javascript
amount.textContent =
  "KES 1,000";
```

Benefits:

* Avoids repeated queries.
* Makes code clearer.
* Centralizes validation.
* Improves component API design.

However, caching is safe only while the referenced node remains valid.

---

# 17. When Caching Is Safe

Caching is generally safe when:

* The component structure is stable.
* The node is not replaced.
* The root remains connected.
* Only text, attributes, or classes change.
* The same descendant object remains in the DOM.

Example:

```javascript
status.textContent =
  "Paid";
```

The same status element remains valid.

---

# 18. When Cached References Become Stale

A reference becomes stale when it no longer represents the current component structure.

Example:

```javascript
card.innerHTML = `
  <p data-role="status">
    Updated
  </p>
`;
```

The old cached `status` element was removed.

It may still exist as a detached object:

```javascript
status.isConnected;
```

returns:

```text
false
```

The current status element must be queried again.

Other stale-reference causes:

* `replaceChildren()`.
* `innerHTML`.
* Component rerendering.
* Parent removal.
* Moving the component.
* Replacing a subtree.
* Template reconstruction.

---

# 19. Refreshing Cached Views

A refreshable view can re-query its parts.

```javascript
function createRefreshableCardView(
  root
) {
  let parts;

  function refresh() {
    parts = {
      receipt:
        queryRequired(
          root,
          "[data-role='receipt']"
        ),

      amount:
        queryRequired(
          root,
          "[data-role='amount']"
        ),

      status:
        queryRequired(
          root,
          "[data-role='status']"
        ),
    };

    return parts;
  }

  refresh();

  return {
    root,

    refresh,

    get parts() {
      return parts;
    },
  };
}
```

Call:

```javascript
view.refresh();
```

after structural replacement.

# 20. Lazy DOM Queries

Lazy querying delays a query until the property is needed.

```javascript
function createLazyCardView(
  root
) {
  return {
    root,

    get amount() {
      return queryRequired(
        root,
        "[data-role='amount']"
      );
    },

    get status() {
      return queryRequired(
        root,
        "[data-role='status']"
      );
    },
  };
}
```

Each property access performs a fresh query.

Benefits:

* Always reflects current structure.
* Avoids querying unused elements.
* Handles replaced descendants.

Trade-offs:

* Repeated access repeats the query.
* Missing structure errors occur later.
* Harder to distinguish initialization failures.

---

# 21. Eager vs Lazy Querying

| Eager Querying                  | Lazy Querying                |
| ------------------------------- | ---------------------------- |
| Queries during view creation    | Queries when accessed        |
| Fails early                     | Fails later                  |
| Good for stable required parts  | Good for replaceable content |
| Avoids repeated queries         | May query repeatedly         |
| Validates component immediately | Avoids unused queries        |

Hybrid approach:

```javascript
function createCardView(
  root
) {
  const requiredStaticParts = {
    actions:
      queryRequired(
        root,
        "[data-role='actions']"
      ),
  };

  return {
    root,
    ...requiredStaticParts,

    get status() {
      return queryRequired(
        root,
        "[data-role='status']"
      );
    },
  };
}
```

Use eager caching for stable structure and lazy access for replaceable regions.

# 22. Query Factories

A query factory creates root-scoped query helpers.

```javascript
function createQueryContext(
  root
) {
  return {
    one(selector) {
      return root.querySelector(
        selector
      );
    },

    required(
      selector,
      description = selector
    ) {
      return queryRequired(
        root,
        selector,
        description
      );
    },

    all(selector) {
      return [
        ...root.querySelectorAll(
          selector
        ),
      ];
    },

    direct(selector) {
      return [
        ...root.children,
      ].filter(
        child =>
          child.matches(
            selector
          )
      );
    },
  };
}
```

Usage:

```javascript
const query =
  createQueryContext(
    card
  );

const amount =
  query.required(
    "[data-role='amount']"
  );

const actions =
  query.all(
    "[data-action]"
  );
```

This makes scope explicit.

# 23. Component Registries

A component registry maps application identities to component views.

```javascript
class ComponentRegistry {
  #items = new Map();

  register(
    id,
    view
  ) {
    if (
      this.#items.has(id)
    ) {
      throw new Error(
        `Component already registered: ${id}`
      );
    }

    this.#items.set(
      id,
      view
    );
  }

  get(id) {
    return (
      this.#items.get(id) ??
      null
    );
  }

  unregister(id) {
    return this.#items.delete(
      id
    );
  }

  has(id) {
    return this.#items.has(
      id
    );
  }
}
```

A registry avoids repeatedly searching the document by ID or dataset value.

---

# 24. Registering Repeated Components

```javascript
function registerTransactionCards(
  container,
  registry
) {
  const cards =
    container.querySelectorAll(
      ":scope > [data-component='transaction-card']"
    );

  for (const card of cards) {
    const id =
      Number(
        card.dataset
          .transactionId
      );

    registry.register(
      id,
      createTransactionCardView(
        card
      )
    );
  }
}
```

Each repeated card gets its own scoped view.

---

# 25. Looking Up Components by Identity

Without a registry:

```javascript
const card =
  document.querySelector(
    `[data-transaction-id="${id}"]`
  );
```

With a registry:

```javascript
const cardView =
  registry.get(id);
```

Benefits:

* Avoids repeated selector construction.
* Avoids selector escaping issues.
* Gives direct access to component methods.
* Supports cleanup when removed.
* Separates identity lookup from DOM searching.

The registry must remain synchronized with the DOM.

---

# 26. Avoiding Global DOM Lookups

Global queries are sometimes necessary during startup.

```javascript
const applicationRoot =
  document.querySelector(
    "#application"
  );
```

After locating the application root, prefer scoped discovery.

```text
Document
    ↓
Application root
    ↓
Feature root
    ↓
Component root
    ↓
Component parts
```

Avoid repeatedly returning to `document` for component internals.

---

# 27. Traversal Service Layers

A traversal service centralizes shared DOM discovery.

```javascript
const traversalService = {
  getActionElement(
    target,
    boundary
  ) {
    if (
      !(target instanceof Element)
    ) {
      return null;
    }

    const action =
      target.closest(
        "[data-action]"
      );

    if (
      !action ||
      !boundary.contains(
        action
      )
    ) {
      return null;
    }

    return action;
  },

  getComponentRoot(
    element,
    name
  ) {
    return element.closest(
      `[data-component="${CSS.escape(
        name
      )}"]`
    );
  },
};
```

Use a service when behavior is genuinely shared.

Do not turn every simple query into a large abstraction.

---

# 28. Separating Traversal from State

Traversal reads the view.

Application state stores authoritative data.

Avoid treating DOM text as the primary data source.

Weak architecture:

```javascript
const amount =
  Number(
    amountElement
      .textContent
      .replace("KES", "")
  );
```

Better:

```javascript
const transaction =
  state.transactions
    .get(id);
```

The DOM displays the amount.

State owns the amount.

Traversal should discover identity and relevant elements, not replace state management.

---

# 29. Separating Traversal from Business Logic

Traversal function:

```javascript
function resolveTransactionContext(
  target,
  boundary
) {
  // Find action and card.
}
```

Business function:

```javascript
function markTransactionPaid(
  transactionId
) {
  // Update state and persistence.
}
```

View function:

```javascript
function renderPaidState(
  cardView
) {
  cardView.setStatus(
    "Paid"
  );
}
```

Clear separation:

```text
Traversal:
What was interacted with?

Business logic:
What should happen?

View:
How should the DOM reflect it?
```

# 30. DOM View Models

A DOM view model exposes meaningful UI operations.

```javascript
function createTransactionView(
  root
) {
  const parts = {
    status:
      queryRequired(
        root,
        "[data-role='status']"
      ),

    markPaidButton:
      queryRequired(
        root,
        "[data-action='mark-paid']"
      ),
  };

  return {
    root,
```

# 31. Component Context Objects

A context object combines:

* Event target.
* Action element.
* Component root.
* Component view.
* Application identity.
* Action name.

```javascript
function createActionContext(
  event,
  boundary,
  registry
) {
  const target =
    event.target;

  if (
    !(target instanceof Element)
  ) {
    return null;
  }

  const actionElement =
    target.closest(
      "[data-action]"
    );

  if (
    !actionElement ||
    !boundary.contains(
      actionElement
    )
  ) {
    return null;
  }

  const card =
    actionElement.closest(
      "[data-component='transaction-card']"
    );

  if (!card) {
    return null;
  }

  const id =
    Number(
      card.dataset
        .transactionId
    );

  return {
    target,
    actionElement,
    card,
    id,
    action:
      actionElement.dataset
        .action,
    view:
      registry.get(id),
  };
}
```

Event handler:

```javascript
function handleClick(

) {

      container,
      registry
```

# 36. Handling Removed Components

Before removal:

1. Read identity.
2. Store focus fallback.
3. Unregister view.
4. Remove root.
5. Restore focus.

```javascript
function removeCard(
  view,
  registry,
  fallback
) {
  const root =
    view.root;

  const focusTarget =
    root.nextElementSibling ??
    root.previousElementSibling ??
    fallback;

  registry.unregister(
    view.getId()
  );

  root.remove();

  focusTarget.focus?.();
}
```

The registry must not retain removed views.

# 37. Disconnected DOM References

A removed node may remain in memory if referenced.

```javascript
const removedView =
  registry.get(id);
```

If the registry was not cleaned, the root remains reachable.

```javascript

  .isConnected;
```

returns:

```text
false
```

This may contribute to unnecessary memory retention.

Cleanup architecture should release references to removed components.

## Example 3: Real-World Application — Transaction List Architecture

### JavaScript

```javascript
const selectors = {
  list:
    "[data-component='transaction-list']",

  card:
    "[data-component='transaction-card']",

  receipt:
    "[data-role='receipt']",

  amount:
    "[data-role='amount']",

  status:
    "[data-role='status']",

  action:
    "[data-action]",
};

class TransactionCardView {
  constructor(root) {
    if (
      !root.matches(
        selectors.card
      )
    ) {
      throw new Error(
        "Invalid transaction card root."
      );
    }

    this.root = root;

    this.receipt =
      queryRequired(
        root,
        selectors.receipt
      );

    this.amount =
      queryRequired(
        root,
        selectors.amount
      );

    this.status =
      queryRequired(
        root,
        selectors.status
      );
  }

  get id() {
    return Number(
      this.root.dataset
        .transactionId
    );
  }

  setPaid() {
    this.root.dataset.status =
      "paid";

    this.status.textContent =
      "Paid";
  }

  remove() {
    this.root.remove();
  }
}

class TransactionListView {
  constructor(root) {
    this.root = root;
    this.registry =
      new ComponentRegistry();

    this.initializeCards();

    this.root.addEventListener(
      "click",
      event =>
        this.handleClick(
          event
        )
    );
  }

  initializeCards() {
    const cards =
      queryDirectChildren(
        this.root,
        selectors.card
      );

    for (const card of cards) {
      const view =
        new TransactionCardView(
          card
        );

      this.registry.register(
        view.id,
        view
      );
    }
  }

  resolveContext(event) {
    const target =
      event.target;

    if (
      !(target instanceof Element)
    ) {
      return null;
    }

    const actionElement =
      target.closest(
        selectors.action
      );

    if (
      !actionElement ||
      !this.root.contains(
        actionElement
      )
    ) {
      return null;
    }

    const cardRoot =
      actionElement.closest(
        selectors.card
      );

    if (
      !cardRoot ||
      cardRoot.parentElement !==
        this.root
    ) {
      return null;
    }

    const id =
      Number(
        cardRoot.dataset
          .transactionId
      );

    return {
      action:
        actionElement.dataset
          .action,

      actionElement,

      id,

      view:
        this.registry.get(id),
    };
  }

  handleClick(event) {
    const context =
      this.resolveContext(
        event
      );

    if (
      !context ||
      !context.view
    ) {
      return;
    }

    if (
      context.action ===
      "mark-paid"
    ) {
      context.view.setPaid();

      context.actionElement
        .disabled = true;

      context.actionElement
        .textContent = "Paid";

      return;
    }

    if (
      context.action ===
      "delete"
    ) {
      const {
        root,
      } = context.view;

      const focusTarget =
        root.nextElementSibling ??
        root.previousElementSibling ??
        this.root;

      this.registry.unregister(
        context.id
      );

      context.view.remove();

      focusTarget.focus?.();
    }
  }
}
```

Initialize:

```javascript
const listRoot =
  queryRequired(
    document,
    selectors.list
  );

const transactionListView =
  new TransactionListView(
    listRoot
  );
```

### Why this architecture is professional

* Selectors are centralized.
* Component roots are validated.
* Each card has an isolated view.
* A registry maps IDs to views.
* One delegated listener handles actions.
* Context resolution is separated.
* Direct-child ownership is enforced.
* Removed views are unregistered.
* Focus recovery is included.
* Business behavior no longer repeats traversal logic.

# 45. Common Architectural Mistakes

## Mistake 1: Overengineering tiny pages

A two-button page may not need classes, registries, and service layers.

Architecture should match application size.

## Mistake 2: Hiding every query behind unnecessary abstractions

Simple local queries can remain simple.

## Mistake 3: Treating the DOM as application state

The DOM should usually reflect state, not replace it.

## Mistake 4: Caching references after subtree replacement

Detached references no longer represent current content.

## Mistake 5: Forgetting registry cleanup

Removed components may remain referenced.

## Mistake 6: Mixing traversal and network logic

A DOM view should not automatically own every API request.

## Mistake 7: Using global selectors inside component methods

Component methods should remain root-scoped.

## Mistake 8: Ignoring nested component ownership

Broad descendant searches can cross logical boundaries.

## Mistake 9: Creating selector strings from arbitrary data

Prefer registries, iteration, or escaped values.

## Mistake 10: Making component contracts implicit

Required structure should be documented and validated.

# 46. Best Practices

1. Start with simple scoped queries, then introduce architecture as complexity grows.
2. Mark stable component roots semantically.
3. Centralize component-specific selectors.
4. Use required-query helpers for structural contracts.
5. Keep component views focused on DOM responsibilities.
6. Store authoritative data outside the DOM.
7. Refresh caches after structural replacement.
8. Unregister removed component views.
9. Use event delegation for dynamic repeated components.
10. Test component contracts with small fixtures.
11. Keep ownership rules explicit.
12. Avoid global queries after application initialization.

# 47. Performance Considerations

## Query caching

Caching saves repeated searches but risks stale references.

## Registries

Map lookup is generally faster and clearer than repeated DOM searching by application identity.

## Initialization cost

Creating views for thousands of components upfront may be expensive.

Possible alternatives:

* Lazy view creation.
* Event-time context resolution.
* Virtualized rendering.
* Register only visible components.

## Delegated events

One listener reduces listener count but performs traversal per event.

This is usually a good trade-off for dynamic lists.

## Memory

Registries and cached views hold DOM references.

Clean them when components are removed.

## Architecture overhead

Abstraction itself has cognitive and runtime cost.

Use the simplest architecture that remains maintainable.

# 49. Interview Questions and Answers

## Beginner

### 1. Why centralize selectors?

To reduce duplication and make structural changes easier to maintain.

### 2. What is a component view?

An object that wraps a component root and exposes its DOM parts and operations.

### 3. What is a stale DOM reference?

A reference to a node that no longer represents the current component structure.

### 4. Why use scoped queries inside components?

To prevent cross-component matches.

### 5. What is a component contract?

A definition of required, optional, and structural DOM parts for a component.

## Intermediate

### 1. When should DOM references be cached?

When component structure is stable and the nodes will not be replaced.

### 2. When should cached references be refreshed?

After replacing or rebuilding the relevant subtree.

### 3. What is the purpose of a component registry?

To map application identities to component views without repeated DOM searches.

### 4. Why clean registries after removal?

To avoid stale lookups and unnecessary memory retention.

### 5. What is lazy querying?

Delaying a DOM query until the value is requested.

## Advanced

### 1. How would you architect traversal for thousands of repeated components?

Use a stable container, delegated events, direct-child ownership, lazy view creation, and a registry only where identity lookup is needed.

### 2. What are the trade-offs between eager and lazy component views?

Eager views validate early and avoid repeated queries; lazy views reduce startup work and better handle replaced structure.

### 3. Why should DOM views not become the application's state store?

DOM text is presentation-oriented, difficult to normalize, and can diverge from authoritative business data.

### 4. How do frameworks reduce traversal architecture work?

They provide component identity, state-driven rendering, references, event binding, and lifecycle systems.

### 5. How can stale-reference bugs be detected?

Check `isConnected`, compare cached references with fresh queries, and refresh views after structural replacement.
