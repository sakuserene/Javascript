# 1. Introduction

In the previous lesson, you learned how to inspect direct children.

Given:

```html
<section id="dashboard">
  <article class="card">
    <header>
      <h2>Summary</h2>
    </header>

    <div class="content">
      <p class="amount">
        KES 5,000
      </p>
    </div>
  </article>
</section>
```

This code:

```javascript
dashboard.children;
```

returns only:

```text
article.card
```

It does not return:

* `header`
* `h2`
* `div.content`
* `p.amount`

Those elements are descendants, but they are not direct children of `#dashboard`.

To find nested descendants at any depth, JavaScript provides selector-based APIs:

```javascript
querySelector()
querySelectorAll()
```

These methods allow you to search a document or subtree using CSS selectors.

Example:

```javascript
const amount =
  dashboard.querySelector(
    ".amount"
  );
```

The browser searches downward through the dashboard subtree until it finds the first matching descendant.

This lesson teaches how to perform that search accurately, safely, and efficiently.

---

# 2. What Is Descendant Traversal?

> Descendant traversal is the process of moving or searching downward through any number of nested levels beneath a starting node.

Consider:

```text
section.dashboard
└── article.card
    ├── header
    │   └── h2
    └── div.content
        └── p.amount
```

For `section.dashboard`:

* `article.card` is a child and descendant.
* `header` is a descendant.
* `h2` is a descendant.
* `div.content` is a descendant.
* `p.amount` is a descendant.

Descendant traversal may inspect:

```text
One level
Two levels
Three levels
Any depth below the root
```

Selector methods are the most common way to find meaningful descendants.

---

# 3. Why Descendant Searching Exists

Interface markup often contains wrapper elements.

Example:

```html
<article class="transaction-card">
  <header class="card-header">
    <div class="receipt-wrapper">
      <h3 class="receipt-number">
        RCT-101
      </h3>
    </div>
  </header>

  <section class="card-body">
    <div class="amount-wrapper">
      <p class="amount">
        KES 500
      </p>
    </div>
  </section>
</article>
```

To find the amount through direct child properties, code would need to know every wrapper level.

Fragile:

```javascript
const amount =
  card
    .children[1]
    .children[0]
    .children[0];
```

If designers add or remove a wrapper, the path breaks.

Selector-based descendant search expresses meaning:

```javascript
const amount =
  card.querySelector(
    ".amount"
  );
```

This means:

> Find the first element representing the amount anywhere inside this card.

The code depends on semantic identity rather than exact nesting depth.

---

# 4. Child vs Descendant

Consider:

```html
<nav id="navigation">
  <ul>
    <li>
      <a href="/">Home</a>
    </li>

    <li>
      <a href="/reports">
        Reports
      </a>
    </li>
  </ul>
</nav>
```

For the `<nav>`:

| Element | Direct Child? | Descendant? |
| ------- | ------------: | ----------: |
| `<ul>`  |           Yes |         Yes |
| `<li>`  |            No |         Yes |
| `<a>`   |            No |         Yes |

Use direct-child traversal:

```javascript
navigation.children;
```

to get the `<ul>`.

Use descendant searching:

```javascript
navigation.querySelectorAll(
  "a"
);
```

to get the links.

# 5. Mental Model

Think of a building.

```text
Building
├── Floor 1
│   ├── Room 1
│   └── Room 2
└── Floor 2
    └── Room 3
```

Direct children of the building:

```text
Floor 1
Floor 2
```

Descendants of the building:

```text
Floor 1
Room 1
Room 2
Floor 2
Room 3
```

A child query looks only one level down.

A descendant query can search the entire structure below the starting point.

# 6. Relationship Traversal vs Selector Searching

There are two major downward-navigation approaches.

## Relationship traversal

```javascript
element.children;
element.firstElementChild;
```

Advantages:

* Direct structural meaning.
* One-level navigation.
* No selector needed.
* Useful when position matters.

## Selector searching

```javascript
element.querySelector(
  ".amount"
);
```

Advantages:

* Searches any depth.
* Uses semantic identifiers.
* Tolerates wrapper changes.
* Supports complex conditions.

## Example decision

Use:

```javascript
list.firstElementChild;
```

when the requirement is:

> Get the first direct list item.

Use:

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

when the requirement is:

> Find the amount display somewhere inside this card.

---

# 7. `querySelector()`

The `querySelector()` method returns the first element matching a CSS selector within the chosen search root.

Syntax:

```javascript
root.querySelector(
  selector
);
```

The root may be:

* `document`
* An element
* A document fragment
* Other selector-capable DOM roots

Example:

```javascript
const heading =
  document.querySelector(
    "h1"
  );
```

Element-scoped example:

```javascript
const amount =
  card.querySelector(
    ".amount"
  );
```

If no match exists:

```text
null
```

---

# 8. How `querySelector()` Searches

Consider:

```html
<section id="dashboard">
  <article>
    <p class="amount">
      KES 500
    </p>
  </article>

  <article>
    <p class="amount">
      KES 900
    </p>
  </article>
</section>
```

Code:

```javascript
const amount =
  dashboard.querySelector(
    ".amount"
  );
```

Returns the first matching amount in document-tree order:

```text
KES 500
```

Conceptual search:

```text
Start inside dashboard
    ↓
Inspect descendants in tree order
    ↓
First matching .amount found
    ↓
Return element
    ↓
Stop searching
```

`querySelector()` does not return every match.

---

# 9. `querySelectorAll()`

The `querySelectorAll()` method returns all matching descendants.

Syntax:

```javascript
root.querySelectorAll(
  selector
);
```

Example:

```javascript
const amounts =
  dashboard.querySelectorAll(
    ".amount"
  );
```

Returns a static `NodeList`.

Iterate:

```javascript
for (
  const amount
  of amounts
) {
  console.log(
    amount.textContent
      .trim()
  );
}
```

Possible output:

```text
KES 500
KES 900
```

If no elements match, it returns an empty `NodeList`, not `null`.

```javascript
results.length;
```

returns:

```text
0
```

---

# 10. Static NodeLists

The `NodeList` returned by `querySelectorAll()` is static.

Example:

```javascript
const cards =
  container.querySelectorAll(
    ".transaction-card"
  );

console.log(
  cards.length
);
```

Suppose:

```text
2
```

Add another card:

```javascript
container.append(
  createTransactionCard()
);
```

The original collection still reports:

```text
2
```

Query again:

```javascript
const updatedCards =
  container.querySelectorAll(
    ".transaction-card"
  );
```

Now:

```text
3
```

A static query result is a snapshot.

This is often useful because mutation does not alter the collection during iteration.

---

# 11. Document-Scoped Queries

Searching from `document` searches the entire current document.

```javascript
document.querySelector(
  ".transaction-card"
);
```

This is appropriate when:

* The element is globally unique.
* Initialization needs a page-level root.
* The entire page is the intended scope.
* There is no closer known component root.

Example:

```javascript
const application =
  document.querySelector(
    "#application"
  );
```

Avoid global searching when a local root is already available.

---

# 12. Element-Scoped Queries

Elements can search inside their own descendant subtrees.

```javascript
const amount =
  card.querySelector(
    ".amount"
  );
```

This searches only inside `card`.

It ignores matching `.amount` elements in other cards.

Example:

```html
<article class="card" id="card-one">
  <p class="amount">KES 500</p>
</article>

<article class="card" id="card-two">
  <p class="amount">KES 900</p>
</article>
```

```javascript
const cardTwo =
  document.querySelector(
    "#card-two"
  );

const amount =
  cardTwo.querySelector(
    ".amount"
  );
```

Returns:

```text
KES 900
```

Scoped searches improve correctness by reducing ambiguity.

---

# 13. Why Scoped Queries Matter

Consider:

```javascript
const amount =
  document.querySelector(
    ".amount"
  );
```

If many cards exist, this returns only the first amount on the page.

Correct component-scoped pattern:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );

const amount =
  card.querySelector(
    ".amount"
  );
```

Flow:

```text
Nested control
    ↑
Find owning card
    ↓
Search only inside card
    ↓
Find correct amount
```

Benefits:

* Prevents cross-component matches.
* Reduces dependence on globally unique selectors.
* Improves maintainability.
* Makes component logic reusable.
* Better represents application structure.

---

# 14. Does `querySelector()` Test the Root Element?

No.

Suppose:

```html
<article
  class="transaction-card"
  id="card"
>
  <p>Content</p>
</article>
```

```javascript
const card =
  document.querySelector(
    "#card"
  );
```

This:

```javascript
card.querySelector(
  ".transaction-card"
);
```

returns:

```text
null
```

because the method searches descendants, not the root itself.

Compare:

```javascript
card.matches(
  ".transaction-card"
);
```

returns:

```text
true
```

And:

```javascript
card.closest(
  ".transaction-card"
);
```

returns the card itself.

Key comparison:

| Method            | Tests Starting Element? | Search Direction     |
| ----------------- | ----------------------: | -------------------- |
| `matches()`       |                     Yes | Current element only |
| `closest()`       |                     Yes | Upward               |
| `querySelector()` |                      No | Downward             |

---

# 15. `:scope`

The `:scope` pseudo-class represents the element on which the selector method was called.

Example:

```javascript
container.querySelectorAll(
  ":scope > .transaction-card"
);
```

Here:

```text
:scope
```

means:

```text
container
```

The selector means:

> Select `.transaction-card` elements that are direct children of this container.

Without `:scope`, a child combinator at the beginning is invalid:

```javascript
container.querySelectorAll(
  "> .transaction-card"
);
```

Use:

```javascript
":scope > .transaction-card"
```

for explicit root-relative selectors.

---

# 16. Direct-Child Selectors

Suppose:

```html
<section id="transactions">
  <article class="transaction-card">
    <div>
      <article class="transaction-card">
        Nested card
      </article>
    </div>
  </article>
</section>
```

This query:

```javascript
transactions.querySelectorAll(
  ".transaction-card"
);
```

returns both outer and nested cards.

To get only direct card children:

```javascript
transactions.querySelectorAll(
  ":scope > .transaction-card"
);
```

This returns only the outer card.

Alternative:

```javascript
[
  ...transactions.children,
].filter(
  child =>
    child.matches(
      ".transaction-card"
    )
);
```

Both approaches enforce one-level structure.

---

# 17. Descendant Selectors

A space in a CSS selector means descendant.

```css
.transaction-card .amount
```

JavaScript:

```javascript
card.querySelector(
  ".transaction-card .amount"
);
```

However, when searching from `card`, including `.transaction-card` may create a subtle issue because `querySelector()` does not test the root.

If the only `.transaction-card` is the root itself, this may not match as expected.

Prefer:

```javascript
card.querySelector(
  ".amount"
);
```

Or, when the root relationship must be explicit:

```javascript
card.querySelector(
  ":scope .amount"
);
```

A descendant selector allows any number of intermediate levels.

```text
ancestor descendant
```

means:

> Find descendants matching the second selector inside an ancestor matching the first.

---

# 18. Child Combinators

The `>` combinator means direct child.

CSS:

```css
.card > .amount
```

JavaScript:

```javascript
card.querySelector(
  ":scope > .amount"
);
```

This matches only if `.amount` is directly inside the card.

It does not match:

```html
<article class="card">
  <div>
    <p class="amount"></p>
  </div>
</article>
```

because the amount is a grandchild.

Use the child combinator when direct structure is a requirement.

---

# 19. Adjacent-Sibling Selectors

The `+` combinator matches the immediately following sibling.

HTML:

```html
<label for="amount">
  Amount
</label>

<input id="amount">

<p class="help-text">
  Enter a positive value.
</p>
```

From a form root:

```javascript
const helpText =
  form.querySelector(
    "#amount + .help-text"
  );
```

This matches `.help-text` only when it immediately follows `#amount`.

Although sibling properties often express this relationship more directly, selector combinators are useful for complex structural queries.

---

# 20. General-Sibling Selectors

The `~` combinator matches later siblings sharing the same parent.

HTML:

```html
<h2>Summary</h2>
<p class="value">KES 500</p>
<p class="note">Pending clearance</p>
```

Query:

```javascript
section.querySelectorAll(
  "h2 ~ p"
);
```

Returns both paragraphs after the heading.

Use this when matching a structural group of later siblings.

---

# 21. Class Selectors

Class selector:

```css
.transaction-card
```

Query:

```javascript
container.querySelectorAll(
  ".transaction-card"
);
```

Multiple classes:

```javascript
container.querySelectorAll(
  ".transaction-card.selected"
);
```

This means elements having both classes.

Descendant relation:

```javascript
container.querySelectorAll(
  ".transaction-card .delete-button"
);
```

Classes are useful for styling and broad element categories.

For application behavior, data attributes can sometimes communicate roles more explicitly.

---

# 22. ID Selectors

ID selector:

```css
#payment-form
```

Query:

```javascript
document.querySelector(
  "#payment-form"
);
```

IDs should be unique within a document.

Inside a component, relying on repeated IDs is invalid and unsafe.

Avoid patterns such as:

```html
<article class="card">
  <p id="amount"></p>
</article>

<article class="card">
  <p id="amount"></p>
</article>
```

Use classes or data roles:

```html
<p data-role="amount"></p>
```

Scoped query:

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

---

# 23. Attribute Selectors

Attribute-existence selector:

```css
[disabled]
```

Query:

```javascript
form.querySelectorAll(
  "[disabled]"
);
```

Exact value:

```css
[type="submit"]
```

```javascript
form.querySelector(
  '[type="submit"]'
);
```

Prefix:

```css
[href^="https://"]
```

Suffix:

```css
[href$=".pdf"]
```

Substring:

```css
[class*="card"]
```

Space-separated token:

```css
[aria-describedby~="amount-help"]
```

Attribute selectors are powerful for semantic and state-based searches.

---

# 24. Data-Attribute Selectors

Application roles:

```html
<button
  data-action="delete"
>
  Delete
</button>
```

Query:

```javascript
card.querySelector(
  "[data-action='delete']"
);
```

Component role:

```html
<p data-role="amount"></p>
```

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

State:

```html
<article data-status="pending">
```

```javascript
container.querySelectorAll(
  "[data-status='pending']"
);
```

Data attributes can separate behavior identity from visual class names.

# 25. State and Structural Selectors

## Disabled elements

```javascript
form.querySelectorAll(
  ":disabled"
);
```

## Checked controls

```javascript
form.querySelectorAll(
  ":checked"
);
```

## Required inputs

```javascript
form.querySelectorAll(
  ":required"
);
```

## First child

```javascript
list.querySelector(
  ":scope > :first-child"
);
```

## Last child

```javascript
list.querySelector(
  ":scope > :last-child"
);
```

## Empty elements

```javascript
container.querySelectorAll(
  ":empty"
);
```

Important:

`:empty` does not match when the element contains whitespace text.

## `nth-child()`

```javascript
list.querySelector(
  ":scope > :nth-child(2)"
);
```

Selects the second child element according to CSS structural rules.

# 26. Selector Groups

A comma separates selector alternatives.

```javascript
container.querySelectorAll(
  "button, a, input"
);
```

This selects buttons, links, and inputs.

Focusable-control example:

```javascript
component.querySelectorAll(
  `
    button:not([disabled]),
    a[href],
    input:not([disabled]),
    select:not([disabled]),
    textarea:not([disabled])
  `
);
```

Selector groups reduce repeated queries when several selector types are logically equivalent.

---

# 27. Selector Escaping

Dynamic values cannot always be inserted directly into CSS selectors.

Suppose an element ID is:

```text
transaction:42
```

Unsafe:

```javascript
document.querySelector(
  "#transaction:42"
);
```

The colon has CSS selector meaning.

Similarly, user-provided or server-provided identifiers may contain:

* Spaces.
* Colons.
* Periods.
* Brackets.
* Quotes.
* Other special characters.

Dynamic selector values should be escaped.

---

# 28. `CSS.escape()`

Use:

```javascript
CSS.escape(
  value
);
```

Example:

```javascript
const id =
  "transaction:42";

const element =
  document.querySelector(
    `#${CSS.escape(id)}`
  );
```

Attribute value example:

```javascript
const transactionId =
  '42"] [data-admin="true';

const selector =
  `[data-transaction-id="${CSS.escape(
    transactionId
  )}"]`;
```

However, when building quoted attribute selectors, escaping and quoting rules deserve care.

A simpler approach may be to avoid selector interpolation and compare dataset values in JavaScript when values are untrusted or complex.

> [!WARNING]
> Never build selectors from arbitrary input without considering CSS syntax and selector injection.

---

# 29. Required-Element Helpers

`querySelector()` returns `null` when no match exists.

In many components, some descendants are required.

Helper:

```javascript
function queryRequired(
  root,
  selector,
  description = selector
) {
  const element =
    root.querySelector(
      selector
    );

  if (!element) {
    throw new Error(
      `Required element not found: ${description}`
    );
  }

  return element;
}
```

Usage:

```javascript
const amount =
  queryRequired(
    card,
    "[data-role='amount']",
    "transaction amount"
  );
```

Benefits:

* Fails early.
* Produces meaningful errors.
* Avoids later null-reference failures.
* Documents structural requirements.

---

# 30. Required-Collection Helpers

Sometimes at least one matching element is required.

```javascript
function queryAllRequired(
  root,
  selector,
  description = selector
) {
  const elements =
    root.querySelectorAll(
      selector
    );

  if (
    elements.length === 0
  ) {
    throw new Error(
      `No required elements found: ${description}`
    );
  }

  return elements;
}
```

For stable array behavior:

```javascript
function queryAllArray(
  root,
  selector
) {
  return [
    ...root.querySelectorAll(
      selector
    ),
  ];
}
```

Use strict helpers only when absence indicates broken markup or invalid application state.

# 31. Optional Descendant Lookups

Some descendants are optional.

# 33. Avoiding Component Leakage

Suppose a card should process only actions owned directly by that card, not actions belonging to nested components.

One approach:

```javascript
function belongsToComponent(
  element,
  componentRoot
) {
  const nearestComponent =
    element.closest(
      "[data-component]"
    );

  return (
    nearestComponent ===
    componentRoot
  );
}
```

Filter actions:

```javascript
const ownedActions =
  [
    ...card.querySelectorAll(
      "[data-action]"
    ),
  ].filter(
    action =>
      belongsToComponent(
        action,
        card
      )
  );
```

This prevents nested components from leaking their controls into the parent component's logic.

# 35. Recursive Descendant Traversal

Selector APIs are ideal when CSS selectors express the requirement.

Some tasks require examining every node manually.

Recursive traversal:

```javascript
function visitDescendants(
  node,
  visitor
) {
  for (
    const child
    of node.childNodes
  ) {
    visitor(child);

    visitDescendants(
      child,
      visitor
    );
  }
}
```

Usage:

```javascript
visitDescendants(
  document.body,
  node => {
    console.log(
      node.nodeName
    );
  }
);
```

This performs depth-first traversal.

> [!WARNING]
> Recursive traversal over extremely deep trees can risk call-stack overflow. Iterative approaches are safer for untrusted or unusually deep structures.

---

# 36. Depth-First Traversal

Depth-first traversal explores one branch completely before moving to the next.

Tree:

```text
A
├── B
│   ├── D
│   └── E
└── C
```

Depth-first order:

```text
B
D
E
C
```

Recursive implementation:

```javascript
function* descendantsDepthFirst(
  root
) {
  for (
    const child
    of root.childNodes
  ) {
    yield child;

    yield*
      descendantsDepthFirst(
        child
      );
  }
}
```

Usage:

```javascript
for (
  const node
  of descendantsDepthFirst(
    container
  )
) {
  console.log(node);
}
```

Useful for:

* Tree inspection.
* Serialization.
* Searching deeply.
* Structural validation.
* Text extraction.

---

# 37. Breadth-First Traversal

Breadth-first traversal explores nodes level by level.

Tree:

```text
A
├── B
│   ├── D
│   └── E
└── C
```

Breadth-first order:

```text
B
C
D
E
```

Implementation:

```javascript
function* descendantsBreadthFirst(
  root
) {
  const queue = [
    ...root.childNodes,
  ];

  while (
    queue.length > 0
  ) {
    const node =
      queue.shift();

    yield node;

    queue.push(
      ...node.childNodes
    );
  }
}
```

For large queues, repeatedly using `shift()` can be inefficient.

Improved:

```javascript
function* descendantsBreadthFirst(
  root
) {
  const queue = [
    ...root.childNodes,
  ];

  let index = 0;

  while (
    index < queue.length
  ) {
    const node =
      queue[index];

    index += 1;

    yield node;

    queue.push(
      ...node.childNodes
    );
  }
}
```

Breadth-first traversal is useful when the nearest shallow match should be preferred.

# 38. Searching by Predicate

Custom predicate search:

```javascript
function findDescendant(
  root,
  predicate
) {
  for (
    const node
    of descendantsDepthFirst(
      root
    )
  ) {
    if (predicate(node)) {
      return node;
    }
  }

  return null;
}
```

Example:

```javascript
const node =
  findDescendant(
    container,
    candidate =>
      candidate.nodeType ===
        Node.ELEMENT_NODE &&
      candidate.children.length >
        5
  );
```

Use predicate traversal when CSS selectors cannot express conditions such as:

* Computed style.
* Text length.
* Runtime properties.
* Child counts.
* Application state stored outside attributes.
* Complex custom rules.

# 39. Searching Text and Comment Nodes

CSS selectors match elements, not text or comment nodes.

To find text nodes:

```javascript
const textNodes = [];

for (
  const node
  of descendantsDepthFirst(
    container
  )
) {
  if (
    node.nodeType ===
      Node.TEXT_NODE &&
    node.nodeValue
      .trim()
      .length > 0
  ) {
    textNodes.push(node);
  }
}
```

Find comments:

```javascript
const comments = [];

for (
  const node
  of descendantsDepthFirst(
    container
  )
) {
  if (
    node.nodeType ===
      Node.COMMENT_NODE
  ) {
    comments.push(node);
  }
}
```

This is useful for DOM analysis tools and content processors.

# 40. Querying Dynamic Content

Queries inspect the DOM at the moment they run.

```javascript
const cards =
  container.querySelectorAll(
    ".transaction-card"
  );
```

Add another card later:

```javascript
container.append(
  createCard()
);
```

The old static collection does not update.

To include current content:

```javascript
const currentCards =
  container.querySelectorAll(
    ".transaction-card"
  );
```

Alternatively, use event delegation so new elements can be handled without repeatedly attaching listeners.

Dynamic applications must decide whether to:

* Re-query.
* Maintain JavaScript state.
* Use live direct-child collections.
* Delegate events.
* Observe mutations.

# 41. Mutation and Static Query Results

Static collections can be safely iterated while matching nodes are removed.

```javascript
const cards =
  container.querySelectorAll(
    ".transaction-card"
  );

for (const card of cards) {
  card.remove();
}
```

The `cards` snapshot retains references to all originally matched elements.

Even as they are removed from the document, iteration continues predictably.

Check:

```javascript
card.isConnected;
```

after removal:

```text
false
```

Static snapshots simplify mutation workflows.

# 42. Browser Internal Search Model

When JavaScript calls:

```javascript
root.querySelector(
  selector
);
```

Conceptually, the browser:

```text
1. Parses selector text
2. Validates selector syntax
3. Starts from the root's descendant scope
4. Examines candidate elements in tree order
5. Tests selector conditions
6. Returns first match
7. Returns null when no match exists
```

For:

```javascript
root.querySelectorAll(
  selector
);
```

the browser:

```text
1. Parses selector
2. Searches descendants
3. Collects every matching element
4. Creates a static NodeList
5. Returns the snapshot
```

Invalid selector syntax throws a `DOMException`.

Example:

```javascript
document.querySelector(
  "["
);
```

throws instead of returning `null`.

# 44. Common Mistakes

## Mistake 1: Searching globally for local content

Incorrect:

```javascript
document.querySelector(
  ".amount"
);
```

Correct:

```javascript
card.querySelector(
  ".amount"
);
```

---

## Mistake 2: Assuming `querySelector()` includes the root

```javascript
card.querySelector(
  ".card"
);
```

does not return `card`.

Use:

```javascript
card.matches(
  ".card"
);
```

when testing the root.

---

## Mistake 3: Confusing direct children with all descendants

```javascript
container.querySelectorAll(
  ".card"
);
```

may include deeply nested cards.

Use:

```javascript
container.querySelectorAll(
  ":scope > .card"
);
```

for direct children.

---

## Mistake 4: Assuming `querySelectorAll()` is live

It is static.

Re-query after dynamic changes when current matches are needed.

---

## Mistake 5: Dereferencing a missing optional result

Unsafe:

```javascript
form.querySelector(
  ".error"
).textContent = "";
```

The query may return `null`.

---

## Mistake 6: Silently ignoring a required missing result

If a component cannot function without an amount element, throw a meaningful initialization error.

---

## Mistake 7: Using duplicate IDs inside repeated components

IDs must be unique.

Use classes or data roles with component-scoped searching.

---

## Mistake 8: Building unsafe dynamic selectors

Use `CSS.escape()` or avoid selector interpolation for arbitrary values.

---

## Mistake 9: Allowing nested component leakage

A parent's broad descendant query may include controls owned by nested child components.

---

## Mistake 10: Using selectors when text nodes are required

CSS selector APIs return elements only.

Use node traversal for text and comments.

---

## Mistake 11: Ignoring invalid selector exceptions

An invalid selector throws.

Validate or control selector input.

---

## Mistake 12: Overly complex selectors

Extremely long selectors become difficult to maintain and tightly couple JavaScript to markup structure.

Prefer stable semantic roles and scoped roots.

# 45. Best Practices

## 45.1 Query from the smallest meaningful root

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

## 45.2 Use semantic roles

```html
data-role="amount"
data-action="delete"
data-component="transaction-card"
```

## 45.3 Distinguish required and optional descendants

Required:

```javascript
queryRequired();
```

Optional:

```javascript
querySelector();
```

with explicit conditional handling.

## 45.4 Use `:scope` for root-relative structure

```javascript
container.querySelectorAll(
  ":scope > .item"
);
```

## 45.5 Avoid duplicate IDs

Repeated components should use scoped classes or data attributes.

## 45.6 Re-query when current dynamic results are required

Static NodeLists do not update.

## 45.7 Protect nested component boundaries

Determine which component owns each descendant.

## 45.8 Escape dynamic selector values

Use:

```javascript
CSS.escape()
```

where appropriate.

## 45.9 Prefer selectors that express meaning, not wrapper depth

Good:

```javascript
"[data-role='amount']"
```

Fragile:

```javascript
"section > div:nth-child(2) > p"
```

## 45.10 Store frequently reused results

```javascript
const amount =
  card.querySelector(
    ".amount"
  );
```

Do not repeatedly run the same query when the structure has not changed.

# 46. Performance Considerations

## Search scope

Document-wide search may inspect more of the tree than a component-scoped search.

Prefer:

```javascript
card.querySelector(
  ".amount"
);
```

over:

```javascript
document.querySelector(
  `#${card.id} .amount`
);
```

when the card reference already exists.

## Selector complexity

Simple, meaningful selectors are easier to maintain and usually efficient enough.

Performance problems more often come from:

* Repeated querying inside large loops.
* Very large DOM trees.
* Excessive DOM creation.
* Layout-triggering operations.
* Unnecessary mutation.

## Query caching

Store stable descendant references:

```javascript
const parts = {
  amount:
    card.querySelector(
      ".amount"
    ),

  status:
    card.querySelector(
      ".status"
    ),
};
```

Do not cache references if the descendants will be replaced and the cache will become stale.

## Static snapshots

`querySelectorAll()` creates a static result.

This may require memory proportional to the number of matches, but it gives predictable iteration during mutation.

## Recursive traversal

Manual traversal visits nodes one at a time.

Approximate time complexity:

```text
O(n)
```

where `n` is the number of examined nodes.

Breadth-first traversal also needs queue memory related to tree width.

# 48. Interview Questions and Answers

## Beginner Questions

### 1. What does `querySelector()` return?

The first descendant element matching a CSS selector, or `null`.

### 2. What does `querySelectorAll()` return?

A static `NodeList` containing every matching descendant element.

### 3. Does `querySelector()` search the root element itself?

No. It searches descendants.

### 4. What does `:scope` represent?

The element on which the query method is being called.

### 5. How do you select direct child cards?

```javascript
container.querySelectorAll(
  ":scope > .card"
);
```

---

## Intermediate Questions

### 1. What is the difference between `children` and `querySelectorAll()`?

`children` returns all direct child elements as a live HTMLCollection. `querySelectorAll()` returns matching descendants at any depth as a static NodeList.

### 2. Why are scoped queries safer?

They prevent matching unrelated elements outside the intended component.

### 3. Why can duplicate IDs cause problems?

IDs should be unique, and global ID-based searches may return the wrong repeated component element.

### 4. What happens when a selector is invalid?

The selector method throws a DOM exception.

### 5. Why use a required-element helper?

To fail early with a meaningful structural error instead of crashing later on `null`.

### 6. What does a child combinator mean?

The right-side element must be a direct child of the left-side element.

---

## Advanced Questions

### 1. How do you prevent a parent component from owning descendants inside nested components?

Compare each descendant's nearest component root with the intended owner.

### 2. Why is `querySelectorAll()` useful during removal?

Its static snapshot remains stable while matched elements are removed.

### 3. When is manual recursive traversal necessary?

When processing non-element nodes or applying predicates that CSS selectors cannot express.

### 4. Compare depth-first and breadth-first traversal.

Depth-first explores a branch before siblings. Breadth-first explores level by level.

### 5. How should arbitrary dynamic IDs be used in selectors?

Escape them using `CSS.escape()` or avoid interpolation and compare values programmatically.

### 6. What is the approximate complexity of scanning every descendant?

Linear in the number of examined nodes, commonly described as `O(n)`.

# 52. Summary

Descendant traversal searches downward through any depth beneath a chosen root.

Core APIs:

```javascript
querySelector()
querySelectorAll()
```

`querySelector()`:

```text
Returns the first match
Returns null when no match exists
```

`querySelectorAll()`:

```text
Returns every match
Returns a static NodeList
Returns an empty collection when no match exists
```

You learned that:

* Element queries search descendants, not the root itself.
* `matches()` tests the current element.
* `closest()` searches upward.
* Scoped queries prevent cross-component matches.
* `:scope` represents the current query root.
* `:scope > selector` matches direct children.
* Attribute and data-role selectors can express semantic behavior.
* Dynamic selector values may require `CSS.escape()`.
* Required and optional descendants should be handled differently.
* Nested component ownership must be enforced.
* Static query results do not update after mutation.
* CSS selectors match elements only.
* Manual traversal is needed for text and comment nodes.
* Depth-first and breadth-first traversal provide different visit orders.

A common professional workflow is:

```javascript
const card =
  button.closest(
    "[data-component='transaction-card']"
  );

const amount =
  queryRequired(
    card,
    "[data-role='amount']",
    "transaction amount"
  );
```

Move upward to context, then search downward within that context.
