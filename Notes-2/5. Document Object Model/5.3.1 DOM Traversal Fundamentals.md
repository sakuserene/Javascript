# 1. Introduction

In the previous lesson, you learned that the DOM is a connected tree of nodes.

Example:

```html
<main>
  <section class="summary">
    <h2>Daily Summary</h2>

    <p class="amount">
      KES 5,000
    </p>
  </section>

  <section class="transactions">
    <article class="transaction-card">
      <h3>Receipt RCT-001</h3>

      <button>
        Print
      </button>
    </article>
  </section>
</main>
```

Simplified DOM tree:

```text
main
├── section.summary
│   ├── h2
│   └── p.amount
└── section.transactions
    └── article.transaction-card
        ├── h3
        └── button
```

Every node is connected to related nodes through references such as:

* Parent.
* Children.
* Previous sibling.
* Next sibling.

Once JavaScript obtains one element, it can use those relationships to move through the DOM.

For example, starting from the button:

```text
button
    ↑ parent
article.transaction-card
    ↑ parent
section.transactions
    ↑ parent
main
```

Or starting from the transaction card:

```text
article.transaction-card
├── h3
└── button
```

The process of moving through these relationships is called **DOM traversal**.

---

# 2. What Is DOM Traversal?

> DOM traversal is the process of moving from one DOM node to related nodes through parent, child, sibling, ancestor, or descendant relationships.

Suppose JavaScript already has this button:

```javascript
const button =
  document.querySelector(
    ".transaction-card button"
  );
```

From that starting point, JavaScript can move to:

* Its parent transaction card.
* The heading inside the same card.
* The next transaction card.
* The transaction list containing it.
* The main page container.
* Other descendants inside the same card.

Traversal means navigating the existing tree rather than always searching the entire document from the beginning.

---

# 3. Why DOM Traversal Exists

Real interfaces are hierarchical.

A button usually belongs to:

* A form.
* A card.
* A modal.
* A table row.
* A navigation menu.
* A list item.
* A component section.

When a user interacts with a nested element, the application often needs information from the surrounding structure.

Example:

```html
<article
  class="transaction-card"
  data-transaction-id="42"
>
  <p>KES 500</p>

  <button class="delete-button">
    Delete
  </button>
</article>
```

When the button is clicked, JavaScript needs the transaction ID stored on the containing card.

The button itself does not contain the ID.

Traversal solves this:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );

const transactionId =
  card.dataset.transactionId;
```

Without traversal, developers might add duplicate IDs to every nested element or perform repeated global searches.

Traversal preserves meaningful structure.

---

# 4. The Problem DOM Traversal Solves

Consider a list of transaction cards.

```html
<section id="transactions">
  <article
    class="transaction-card"
    data-id="1"
  >
    <h3>Receipt 1</h3>
    <button>Delete</button>
  </article>

  <article
    class="transaction-card"
    data-id="2"
  >
    <h3>Receipt 2</h3>
    <button>Delete</button>
  </article>
</section>
```

A delete button is clicked.

The application must determine:

1. Which button was clicked?
2. Which card contains the button?
3. Which transaction ID belongs to that card?
4. Which list contains the card?
5. Which neighboring card should receive focus after deletion?

These are relationship questions.

```text
Clicked button
    ↓
Containing card
    ↓
Transaction ID
    ↓
Parent list
    ↓
Next or previous card
```

DOM traversal provides APIs for answering these questions.

# 5. Mental Model: The DOM as a Connected Tree

Think of every DOM node as a location connected by paths.

```text
                main
                 │
        ┌────────┴────────┐
        │                 │
     summary         transactions
        │                 │
     amount             card
                          │
                    ┌─────┴─────┐
                    │           │
                 heading      button
```

From `button`, possible movements include:

```text
Up:
button → card → transactions → main

Sideways:
button → previous sibling heading

Down:
button → text node
```

Traversal is similar to navigating folders.

```text
Project
├── src
│   ├── app.js
│   └── utils.js
└── public
    └── index.html
```

You can move:

* Into a folder.
* Back to a parent folder.
* Across to a sibling.
* Down through descendants.

# 6. Starting Point and Destination

Every traversal operation has:

1. A starting node.
2. A direction.
3. A destination condition.

Example:

```javascript
const button =
  document.querySelector(
    ".delete-button"
  );
```

Starting point:

```text
button.delete-button
```

Direction:

```text
upward
```

Destination:

```text
nearest .transaction-card
```

Code:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );
```

Another example:

Starting point:

```text
transaction list
```

Direction:

```text
downward
```

Destination:

```text
first child card
```

Code:

```javascript
const firstCard =
  list.firstElementChild;
```

Good traversal starts by clearly identifying these three parts.

---

# 7. The Three Main Traversal Directions

DOM traversal has three fundamental directions.

```text
Upward
Downward
Sideways
```

## Upward

Move toward parents and ancestors.

```text
button
    ↑
card
    ↑
section
    ↑
main
```

## Downward

Move toward children and descendants.

```text
card
├── heading
├── amount
└── button
```

## Sideways

Move between siblings sharing the same parent.

```text
heading ↔ amount ↔ button
```

These three directions cover most ordinary DOM-navigation tasks.

---

# 8. Upward Traversal

Upward traversal starts with a node and moves toward the root.

Common APIs:

```javascript
node.parentNode;
element.parentElement;
element.closest(selector);
```

Example:

```html
<main>
  <section>
    <article>
      <button id="save">
        Save
      </button>
    </article>
  </section>
</main>
```

From the button:

```javascript
button.parentElement;
```

returns:

```text
article
```

Another step:

```javascript
button
  .parentElement
  .parentElement;
```

returns:

```text
section
```

A selector-based ancestor search is often more robust:

```javascript
button.closest(
  "section"
);
```

Parent and ancestor traversal will be studied deeply in the next lesson.

---

# 9. Downward Traversal

Downward traversal moves into children and descendants.

Common direct-child APIs:

```javascript
element.children;
node.childNodes;
element.firstElementChild;
element.lastElementChild;
node.firstChild;
node.lastChild;
```

Selector-based descendant APIs:

```javascript
element.querySelector();
element.querySelectorAll();
```

Example:

```html
<article class="card">
  <h3>Receipt</h3>
  <p>KES 500</p>
  <button>Print</button>
</article>
```

From the card:

```javascript
card.children;
```

returns the three direct child elements.

```javascript
card.querySelector(
  "button"
);
```

searches descendants and returns the first matching button.

Direct child traversal and descendant searching are related but different.

---

# 10. Sideways Traversal

Sideways traversal moves between nodes that share the same parent.

Common APIs:

```javascript
node.previousSibling;
node.nextSibling;
element.previousElementSibling;
element.nextElementSibling;
```

Example:

```html
<section>
  <h2>Summary</h2>
  <p>KES 5,000</p>
  <button>Refresh</button>
</section>
```

Starting from the paragraph:

```javascript
paragraph.previousElementSibling;
```

returns the heading.

```javascript
paragraph.nextElementSibling;
```

returns the button.

Sibling traversal is useful when order matters.

Examples:

* Moving focus to the next item.
* Comparing adjacent table rows.
* Creating previous/next navigation.
* Finding an associated label or message.
* Removing an item and selecting its neighbor.

---

# 11. Node Traversal vs Element Traversal

This distinction is essential.

## Node traversal

Node-based properties include every node type.

```javascript
parentNode
childNodes
firstChild
lastChild
previousSibling
nextSibling
```

These may return:

* Elements.
* Text nodes.
* Comments.
* The document.
* Other node types.

## Element traversal

Element-based properties return only elements.

```javascript
parentElement
children
firstElementChild
lastElementChild
previousElementSibling
nextElementSibling
```

## Example

```html
<ul>
  <li>Printing</li>
  <li>Binding</li>
</ul>
```

Because indentation creates whitespace text nodes:

```javascript
list.firstChild;
```

may return:

```text
Text node
```

while:

```javascript
list.firstElementChild;
```

returns:

```text
<li>Printing</li>
```

## Decision rule

Use node traversal when:

* Text nodes matter.
* Comments matter.
* You are building generic DOM tooling.
* You need exact document structure.

Use element traversal when:

* Building ordinary user interfaces.
* Working with cards, rows, buttons, inputs, or sections.
* Whitespace text nodes are irrelevant.

---

# 12. Direct Relationships vs Searching

There are two broad ways to reach another DOM element.

## Relationship traversal

```javascript
element.parentElement;
element.firstElementChild;
element.nextElementSibling;
```

This follows an existing direct relationship.

## Selector searching

```javascript
element.querySelector(
  ".amount"
);
```

This searches a subtree for a matching selector.

## Example

HTML:

```html
<article class="card">
  <header>
    <h3>Receipt</h3>
  </header>

  <p class="amount">
    KES 500
  </p>
</article>
```

To get the parent:

```javascript
heading.parentElement;
```

To find the amount from the card:

```javascript
card.querySelector(
  ".amount"
);
```

There is no rule that one approach is always better.

Use the method that best expresses the structural requirement.

---

# 13. Subtree Traversal

A **subtree** is a node and all of its descendants.

```text
article.card
├── h3
├── p.amount
└── button
```

The entire card is a subtree.

Calling:

```javascript
card.querySelector(
  "button"
);
```

searches only within the card subtree.

It does not search unrelated parts of the document.

This is often safer than:

```javascript
document.querySelector(
  "button"
);
```

when multiple components contain similar elements.

## Component-scoped selection

```javascript
const card =
  document.querySelector(
    ".transaction-card"
  );

const amount =
  card.querySelector(
    ".amount"
  );

const button =
  card.querySelector(
    ".print-button"
  );
```

This keeps element selection local to the component.

---

# 14. Traversal Boundaries

Traversal should respect logical boundaries.

Suppose a button belongs to one transaction card.

```html
<article class="transaction-card">
  <button class="delete-button">
    Delete
  </button>
</article>
```

It is usually appropriate to search upward only as far as the card or transaction list.

```javascript
const card =
  button.closest(
    ".transaction-card"
  );
```

It may be dangerous to assume an arbitrary number of parent steps:

```javascript
button
  .parentElement
  .parentElement
  .parentElement;
```

If markup changes, the code may reach the wrong node.

Logical boundaries include:

* A component root.
* A form.
* A table.
* A list.
* A modal.
* A navigation region.
* A card.
* The document body.

A robust traversal strategy knows where it should stop.

# 15. Core Traversal APIs Overview

| Direction         | Node-Based API                          | Element-Based API                                   | Selector-Based API                      |
| ----------------- | --------------------------------------- | --------------------------------------------------- | --------------------------------------- |
| Upward            | `parentNode`                            | `parentElement`                                     | `closest()`                             |
| Downward          | `childNodes`, `firstChild`, `lastChild` | `children`, `firstElementChild`, `lastElementChild` | `querySelector()`, `querySelectorAll()` |
| Sideways          | `previousSibling`, `nextSibling`        | `previousElementSibling`, `nextElementSibling`      | Usually not needed                      |
| Relationship test | `contains()`                            | `matches()`                                         | CSS selector test                       |

Additional advanced traversal tools include:

```javascript
TreeWalker
NodeIterator
```

These will be covered in a later part.

---

# 16. Upward APIs

## `parentNode`

Returns the immediate parent node.

```javascript
node.parentNode;
```

## `parentElement`

Returns the immediate parent if it is an element.

```javascript
element.parentElement;
```

## `closest()`

Searches the current element and then its ancestors for the nearest element matching a selector.

```javascript
element.closest(
  ".transaction-card"
);
```

Important detail:

`closest()` tests the starting element itself first.

```javascript
card.closest(
  ".transaction-card"
);
```

returns the card itself.

---

# 17. Downward APIs

## `childNodes`

All direct child nodes.

```javascript
container.childNodes;
```

## `children`

Direct child elements only.

```javascript
container.children;
```

## `firstChild`

First direct node.

```javascript
container.firstChild;
```

## `firstElementChild`

First direct element.

```javascript
container.firstElementChild;
```

## `lastChild`

Last direct node.

```javascript
container.lastChild;
```

## `lastElementChild`

Last direct element.

```javascript
container.lastElementChild;
```

## `querySelector()`

First matching descendant, or the element itself is not tested.

```javascript
container.querySelector(
  ".amount"
);
```

## `querySelectorAll()`

All matching descendants in a static `NodeList`.

```javascript
container.querySelectorAll(
  ".transaction-card"
);
```

---

# 18. Sideways APIs

## `previousSibling`

Previous node of any type.

```javascript
node.previousSibling;
```

## `previousElementSibling`

Previous sibling element.

```javascript
element.previousElementSibling;
```

## `nextSibling`

Next node of any type.

```javascript
node.nextSibling;
```

## `nextElementSibling`

Next sibling element.

```javascript
element.nextElementSibling;
```

When building ordinary interfaces, the element-based versions are usually easier to use.

---

# 19. Relationship-Testing APIs

Traversal is not only about moving.

Sometimes code needs to test relationships.

## `contains()`

Checks whether a node contains another node.

```javascript
container.contains(
  element
);
```

A node contains itself.

```javascript
element.contains(
  element
);
```

returns:

```text
true
```

## `matches()`

Checks whether an element matches a CSS selector.

```javascript
element.matches(
  ".transaction-card"
);
```

Returns a Boolean.

## Combined use

```javascript
if (
  transactionsContainer
    .contains(target) &&
  target.matches(
    ".delete-button"
  )
) {
  // Handle delete.
}
```

These APIs become especially important in event delegation.

# 20. Selector-Based Traversal

Selectors allow code to move based on meaning rather than fixed tree depth.

## Descendant search

```javascript
card.querySelector(
  ".amount"
);
```

## Ancestor search

```javascript
button.closest(
  ".transaction-card"
);
```

## Selector test

```javascript
element.matches(
  "[data-status='pending']"
);
```

This creates a powerful pattern:

```text
Start element
    ↓
Find meaningful ancestor
    ↓
Search meaningful descendant
```

Example:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );

const amount =
  card.querySelector(
    ".amount"
  );
```

This is often more robust than relying on exact parent and sibling positions.

# 21. How Traversal Works Internally

The browser maintains structural references between DOM nodes.

Conceptually, a node may contain internal links such as:

```text
Node
├── parent reference
├── first child reference
├── last child reference
├── previous sibling reference
└── next sibling reference
```

When JavaScript reads:

```javascript
element.parentElement;
```

the browser follows the stored parent relationship.

When JavaScript reads:

```javascript
element.nextElementSibling;
```

the browser moves through sibling relationships until it finds an element or reaches the end.

When JavaScript calls:

```javascript
element.querySelector(
  ".amount"
);
```

the browser performs selector matching within the relevant subtree.

Simplified flow:

```text
JavaScript traversal request
    ↓
Browser DOM implementation
    ↓
Follow relationship or search subtree
    ↓
Return browser-managed node reference
    ↓
JavaScript stores reference
```

# 22. Traversal Strategy

A strong traversal strategy follows these steps.

## Step 1: Identify the starting node

```javascript
const button =
  event.target;
```

## Step 2: Identify the required relationship

Examples:

* Nearest card.
* First child row.
* Next sibling item.
* Descendant input.
* Parent form.

## Step 3: Choose node or element traversal

Ask whether text and comment nodes matter.

## Step 4: Define the boundary

Examples:

* Stop at the card.
* Stay inside the form.
* Search only inside the table.
* Do not escape the modal.

## Step 5: Handle missing results

Many traversal properties can return:

```text
null
```

Example:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );

if (!card) {
  return;
}
```

## Step 6: Verify assumptions

Use:

```javascript
matches()
contains()
```

when needed.

# 23. When to Traverse and When to Query

## Prefer traversal when:

* The target has a direct structural relationship.
* You already have a nearby element.
* The markup relationship is meaningful.
* You need the next or previous item.
* You need the containing component.
* You need the immediate parent or child.

Example:

```javascript
button.closest(
  ".transaction-card"
);
```

## Prefer querying when:

* You need a semantically identified descendant.
* Tree depth may change.
* Multiple descendant levels may exist.
* The selector clearly communicates intent.

Example:

```javascript
card.querySelector(
  "[data-role='amount']"
);
```

## Avoid unnecessary global querying

Less scoped:

```javascript
document.querySelector(
  ".amount"
);
```

More scoped:

```javascript
card.querySelector(
  ".amount"
);
```

The scoped version avoids selecting an amount from another card.

# 24. Traversal and Component Structure

A component is a self-contained section of interface.

Example:

```html
<article class="transaction-card">
  <h3 data-role="receipt"></h3>
  <p data-role="amount"></p>
  <button data-action="delete"></button>
</article>
```

A good component root acts as a traversal boundary.

```javascript
function getCardParts(card) {
  return {
    receipt:
      card.querySelector(
        "[data-role='receipt']"
      ),

    amount:
      card.querySelector(
        "[data-role='amount']"
      ),

    deleteButton:
      card.querySelector(
        "[data-action='delete']"
      ),
  };
}
```

Starting from a nested button:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );
```

This pattern is common in framework-free applications.

# 25. Traversal and Event Handling

Events often begin on deeply nested nodes.

Example:

```html
<button class="delete-button">
  <span class="icon">×</span>
  <span>Delete</span>
</button>
```

A user may click the icon `<span>` rather than the button itself.

The event target may therefore be:

```text
span.icon
```

Traversal can normalize the target:

```javascript
const button =
  event.target.closest(
    ".delete-button"
  );
```

Then locate the component:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );
```

This is a foundation of event delegation, which will be taught in the Events module.

# 26. Traversal and Forms

Forms contain related controls.

```html
<form id="login-form">
  <label>
    Username
    <input name="username">
  </label>

  <button type="submit">
    Log In
  </button>
</form>
```

Starting from the button:

```javascript
const form =
  button.closest(
    "form"
  );
```

Starting from an input:

```javascript
const label =
  input.closest(
    "label"
  );
```

Starting from the form:

```javascript
const usernameInput =
  form.querySelector(
    "[name='username']"
  );
```

Traversal keeps form logic scoped to the correct form.

---

# 27. Traversal and Tables

Tables have structured relationships.

```html
<table>
  <tbody>
    <tr data-id="42">
      <td>RCT-42</td>
      <td>KES 500</td>
      <td>
        <button>Delete</button>
      </td>
    </tr>
  </tbody>
</table>
```

From the button:

```javascript
const row =
  button.closest(
    "tr"
  );
```

Read the row ID:

```javascript
const id =
  row.dataset.id;
```

Get cells:

```javascript
const cells =
  row.children;
```

Move to another row:

```javascript
const nextRow =
  row.nextElementSibling;
```

These patterns are common in dashboards and administrative interfaces.

# 28. Traversal and Dynamic Content

DOM content may be added or removed after page load.

Traversal operates on the current live tree.

Example:

```javascript
const newCard =
  document.createElement(
    "article"
  );

transactionsContainer.append(
  newCard
);
```

Now:

```javascript
transactionsContainer
  .lastElementChild;
```

returns the new card.

Traversal references can become detached after removal.

```javascript
const card =
  button.closest(
    ".transaction-card"
  );

card.remove();
```

The variable `card` still references the element object, but it is no longer connected to the document.

Check:

```javascript
card.isConnected;
```

Output:

```text
false
```

Dynamic interfaces require awareness of current connection state.

## Example 3: Real-World Application — Removing a Card and Managing Focus

### HTML

```html
<section
  id="transactions"
  aria-label="Transactions"
>
  <article
    class="transaction-card"
    data-id="1"
    tabindex="-1"
  >
    <h3>Receipt 1</h3>

    <button
      class="delete-button"
      type="button"
    >
      Delete
    </button>
  </article>

  <article
    class="transaction-card"
    data-id="2"
    tabindex="-1"
  >
    <h3>Receipt 2</h3>

    <button
      class="delete-button"
      type="button"
    >
      Delete
    </button>
  </article>

  <article
    class="transaction-card"
    data-id="3"
    tabindex="-1"
  >
    <h3>Receipt 3</h3>

    <button
      class="delete-button"
      type="button"
    >
      Delete
    </button>
  </article>
</section>
```

### JavaScript

```javascript
const container =
  document.querySelector(
    "#transactions"
  );

container.addEventListener(
  "click",
  event => {
    const deleteButton =
      event.target.closest(
        ".delete-button"
      );

    if (!deleteButton) {
      return;
    }

    if (
      !container.contains(
        deleteButton
      )
    ) {
      return;
    }

    const card =
      deleteButton.closest(
        ".transaction-card"
      );

    if (!card) {
      return;
    }

    const nextCard =
      card.nextElementSibling;

    const previousCard =
      card.previousElementSibling;

    const transactionId =
      Number(
        card.dataset.id
      );

    card.remove();

    const focusTarget =
      nextCard ??
      previousCard ??
      container;

    focusTarget.focus?.();

    console.log(
      `Deleted transaction ${transactionId}`
    );
  }
);
```

### Traversal flow

```text
Clicked nested node
    ↓ closest(".delete-button")
Delete button
    ↓ closest(".transaction-card")
Current card
    ↓ nextElementSibling / previousElementSibling
Potential focus target
    ↓
Remove card
    ↓
Focus remaining neighbor
```

### Why professionals write it this way

* `closest()` handles clicks on nested button content.
* `contains()` verifies that the matched button belongs to the intended container.
* The card is found structurally rather than globally.
* Neighbor references are saved before removal.
* Focus is moved to a meaningful surviving element.
* Missing relationships are handled safely.
* The logic supports dynamically added cards.

# 30. Common Mistakes

## Mistake 1: Assuming a parent always exists

```javascript
element.parentElement
  .classList.add(
    "active"
  );
```

`parentElement` may be `null`.

Safer:

```javascript
element.parentElement
  ?.classList.add(
    "active"
  );
```

Or explicitly validate.

---

## Mistake 2: Chaining a fixed number of parents

Fragile:

```javascript
button
  .parentElement
  .parentElement
  .parentElement;
```

Markup changes can break this.

Prefer:

```javascript
button.closest(
  ".transaction-card"
);
```

---

## Mistake 3: Using node sibling APIs when elements are required

```javascript
item.nextSibling;
```

may return whitespace.

Use:

```javascript
item.nextElementSibling;
```

---

## Mistake 4: Searching the entire document unnecessarily

```javascript
document.querySelector(
  ".amount"
);
```

may return the wrong card's amount.

Use:

```javascript
card.querySelector(
  ".amount"
);
```

---

## Mistake 5: Forgetting that `closest()` includes the starting element

This can be useful, but it may surprise developers who expect only ancestors.

---

## Mistake 6: Traversing outside the intended boundary

A broad `closest()` selector may find an unrelated outer component.

Use specific selectors and verify containment when needed.

---

## Mistake 7: Using child position as business identity

Fragile:

```javascript
container.children[2];
```

If order changes, the wrong item is selected.

Prefer semantic selectors or data attributes.

---

## Mistake 8: Accessing sibling references after removal

Store required neighbors before calling:

```javascript
element.remove();
```

---

## Mistake 9: Assuming a selected node remains connected

Dynamic code may remove it.

Check:

```javascript
element.isConnected;
```

when necessary.

# 31. Best Practices

## 31.1 Traverse from the nearest known element

Avoid starting from `document` when a component root is already available.

## 31.2 Prefer semantic boundaries

Use selectors such as:

```javascript
closest(
  ".transaction-card"
)
```

instead of fixed parent counts.

## 31.3 Use element APIs for interface work

Prefer:

```javascript
parentElement
children
firstElementChild
nextElementSibling
```

unless non-element nodes matter.

## 31.4 Scope descendant queries

```javascript
component.querySelector(
  "[data-role='amount']"
);
```

## 31.5 Handle `null`

Traversal reaches boundaries.

```javascript
const next =
  item.nextElementSibling;

if (!next) {
  // Item is last.
}
```

## 31.6 Store references before mutation

```javascript
const next =
  card.nextElementSibling;

card.remove();
```

## 31.7 Verify delegated event targets

Use:

```javascript
container.contains(
  target
);
```

where appropriate.

## 31.8 Use data attributes for identity

```html
data-transaction-id="42"
```

is more robust than relying on child index.

# 32. Performance Considerations

## Direct relationship traversal

Properties such as:

```javascript
parentElement
firstElementChild
nextElementSibling
```

follow structural references and are generally efficient.

## Scoped queries

This:

```javascript
card.querySelector(
  ".amount"
);
```

searches a smaller subtree than:

```javascript
document.querySelector(
  ".amount"
);
```

Scoped queries can improve correctness and may reduce search work.

## Repeated traversal

Avoid repeatedly calculating the same path inside a loop.

Less efficient:

```javascript
for (
  let index = 0;
  index < 1000;
  index += 1
) {
  button.closest(
    ".transaction-card"
  );
}
```

Store the result:

```javascript
const card =
  button.closest(
    ".transaction-card"
  );
```

## Deep DOM structures

Excessively deep trees can make traversal logic harder to understand and may increase style and layout complexity.

## Mutation during traversal

Removing or inserting elements may change live relationships and collections.

Plan the mutation order carefully.

# 34. Interview Questions and Answers

## Beginner Questions

### 1. What is DOM traversal?

Moving through related DOM nodes using parent, child, sibling, ancestor, or descendant relationships.

### 2. What are the three main traversal directions?

Upward, downward, and sideways.

### 3. How do you get an element's parent element?

```javascript
element.parentElement
```

### 4. How do you get the first child element?

```javascript
element.firstElementChild
```

### 5. How do you get the next sibling element?

```javascript
element.nextElementSibling
```

---

## Intermediate Questions

### 1. What is the difference between `parentNode` and `parentElement`?

`parentNode` can return any node type. `parentElement` returns only an element parent.

### 2. What is the difference between `nextSibling` and `nextElementSibling`?

`nextSibling` may return text or comment nodes. `nextElementSibling` returns only an element.

### 3. What does `closest()` do?

It tests the current element and then its ancestors, returning the nearest element matching a CSS selector.

### 4. What does `contains()` do?

It checks whether one node contains another node, including itself.

### 5. Why should queries often be scoped to a component root?

To avoid matching unrelated elements and to keep code aligned with component structure.

---

## Advanced Questions

### 1. When is direct traversal preferable to selector searching?

When the target has a clear immediate structural relationship, such as a parent, direct child, or adjacent sibling.

### 2. Why is `closest()` usually more robust than repeated `parentElement` access?

It expresses the semantic destination and tolerates changes in intermediate wrapper depth.

### 3. Why should neighbor references be stored before removing a node?

After removal, sibling relationships in the live tree change and the removed element is detached.

### 4. How does traversal support event delegation?

An event's target can be normalized to a matching control using `closest()`, then traversed to its component or row.

### 5. What is a traversal boundary?

A logical node beyond which traversal should not continue, such as a component root, form, table, modal, or list.

### 6. What is the performance benefit of scoped querying?

It limits selector matching to a smaller subtree and reduces the chance of selecting unrelated elements.
