# Why Promise Combinators Exist

Suppose a POS dashboard needs three pieces of data:

- Today's transactions
- Current cashier
- Service prices

A beginner may write:

```javascript
getTransactions()

.then(() => {

    return getCashier();

})

.then(() => {

    return getServices();

});
```

This works.

But notice what happens.

```
Transactions

↓

Wait

↓

Cashier

↓

Wait

↓

Services
```

Every request waits for the previous one.

If each request takes

```
2 seconds
```

Total time

```
6 seconds
```

---

# Parallel Execution

Now imagine all requests start immediately.

```
Transactions

───────────┐

Cashier

───────────┤

Services

───────────┘

↓

Finished Together
```

Instead of

```
6 seconds
```

the total time becomes roughly

```
2 seconds
```

This is much faster.

Promise combinators make this possible.

---

# Sequential vs Parallel

Sequential

```text
Task 1

↓

Task 2

↓

Task 3
```

Parallel

```text
Task 1 ────┐

Task 2 ────┤

Task 3 ────┘
```

Notice

Parallel does **not** mean JavaScript suddenly becomes multi-threaded.

JavaScript still has one main thread.

The asynchronous operations are handled by external systems (browser APIs, operating system, servers, etc.) while JavaScript coordinates their results.

---

# Promise.all()

This is the most commonly used Promise combinator.

General syntax

```javascript
Promise.all([

    promise1,

    promise2,

    promise3

]);
```

It accepts an array of Promises.

---

# Mental Model

Imagine ordering:

- Pizza
- Juice
- Dessert

You tell the waiter:

> "Bring everything together."

You don't want the pizza immediately if the rest isn't ready.

You wait until **everything** is ready.

That is exactly how

```javascript
Promise.all()
```

works.

---

# Success Behaviour

Suppose

```javascript
Promise.all([

    Promise.resolve("A"),

    Promise.resolve("B"),

    Promise.resolve("C")

]);
```

Result

```text
[
"A",
"B",
"C"
]
```

Notice

The order matches the array,

not the finishing order.

---

# Failure Behaviour

Suppose

```javascript
Promise.all([

    Promise.resolve("User"),

    Promise.reject("Server Error"),

    Promise.resolve("Orders")

]);
```

Immediately

```text
Rejected

↓

Server Error
```

As soon as one Promise rejects,

the entire `Promise.all()` rejects.

This is called **fail-fast behavior**.

---

# Internal Behaviour

Conceptually

```text
Promise.all

↓

Wait

↓

Promise 1

Promise 2

Promise 3

↓

All Fulfilled?

↓

Yes

↓

Return Array
```

If one rejects

```text
Reject Immediately
```

---

# Example 1 — Beginner

```javascript
Promise.all([

    Promise.resolve(10),

    Promise.resolve(20),

    Promise.resolve(30)

])

.then(values => {

    console.log(values);

});
```

Output

```text
[10, 20, 30]
```

---

# Real-World Example

```javascript
Promise.all([

    fetchUser(),

    fetchOrders(),

    fetchSettings()

])

.then(([user, orders, settings]) => {

    console.log(user);

    console.log(orders);

    console.log(settings);

});
```

All requests begin together.

# Promise.race()

General syntax

```javascript
Promise.race([...]);
```

Race means:

> "Whichever Promise settles first wins."

Notice

It does **not** mean

"first fulfilled."

It means

"first settled."

Settled includes:

- fulfilled
- rejected

---

# Mental Model

Imagine three taxis.

You take whichever arrives first.

You don't wait for the others.

---

Example

```javascript
Promise.race([

    fastPromise,

    slowPromise

]);
```

If

```
fastPromise
```

finishes first,

its result becomes the final result.

---

# Real-World Use

Suppose a network request must finish within five seconds.

```javascript
Promise.race([

    fetchData(),

    timeoutPromise()

]);
```

If the timeout finishes first,

you can display an error.

---

# Promise.allSettled()

Sometimes failure of one Promise shouldn't cancel everything.

Imagine loading

- User Profile
- Notifications
- Weather

Even if Weather fails,

the page should still load.

`Promise.allSettled()` waits for **every Promise** to settle.

General syntax

```javascript
Promise.allSettled([...]);
```

---

Example

```javascript
Promise.allSettled([

    Promise.resolve("A"),

    Promise.reject("B"),

    Promise.resolve("C")

]);
```

Result

```javascript
[
  {
    status: "fulfilled",
    value: "A"
  },
  {
    status: "rejected",
    reason: "B"
  },
  {
    status: "fulfilled",
    value: "C"
  }
]
```

Notice

Nothing is discarded.

You receive every outcome.

---

# Promise.any()

General syntax

```javascript
Promise.any([...]);
```

Promise.any waits for:

> **The first fulfilled Promise.**

Rejected Promises are ignored unless **every Promise rejects**.

---

Example

```javascript
Promise.any([

    Promise.reject(),

    Promise.resolve("Winner"),

    Promise.resolve("Another")

]);
```

Output

```text
Winner
```

---

If every Promise rejects,

JavaScript throws an

```text
AggregateError
```

because multiple failures occurred.

---

# Example 3 — Real-World POS Application

A POS system starts several background operations.

```javascript
Promise.allSettled([

    uploadTransactions(),

    syncInventory(),

    backupReceipts()

])

.then(results => {

    console.log(results);

});
```

Even if one synchronization fails,

the others continue.

This is often preferable for background jobs.

# Which Combinator Should You Use?

| Situation | Best Choice |
|-----------|-------------|
| Every task must succeed | `Promise.all()` |
| Need the fastest response | `Promise.race()` |
| Want the first successful result | `Promise.any()` |
| Need every result regardless of failure | `Promise.allSettled()` |

# Common Mistakes

## Mistake 1

Using `Promise.all()` when partial success is acceptable.

If one Promise fails,

everything fails.

---

## Mistake 2

Expecting `Promise.race()` to wait for success.

It doesn't.

The first rejection also wins.

---

## Mistake 3

Confusing `Promise.any()` with `Promise.race()`.

Remember

```
race()

↓

First Settled

--------------

any()

↓

First Fulfilled
```

---

## Mistake 4

Thinking Promises inside `Promise.all()` execute one after another.

They don't.

They begin immediately (assuming they are already created or started by the functions that produced them).

---

# Best Practices

- Use `Promise.all()` for independent tasks that all need to succeed.
- Use `Promise.allSettled()` when every result matters.
- Use `Promise.race()` for timeouts and competing operations.
- Use `Promise.any()` when any successful result is sufficient.
- Destructure arrays returned by `Promise.all()` for clearer code.

---

# Performance Considerations

Running independent asynchronous tasks in parallel often improves performance significantly.

However:

- Don't run dependent tasks in parallel.
- Avoid creating unnecessary requests.
- Consider server limits and API rate limits when launching many requests simultaneously.

# Interview Questions

### Beginner

**Q:** What does `Promise.all()` return?

**Answer:**

A Promise that fulfills with an array of results if all input Promises fulfill, or rejects immediately if any input Promise rejects.

---

### Intermediate

**Q:** What's the difference between `Promise.race()` and `Promise.any()`?

**Answer:**

- `Promise.race()` resolves or rejects with the **first settled** Promise.
- `Promise.any()` resolves with the **first fulfilled** Promise and ignores rejections until all Promises reject.

---

### Advanced

**Q:** When would you use `Promise.allSettled()` instead of `Promise.all()`?

**Answer:**

When you need the outcome of every asynchronous operation, even if some fail—for example, background synchronization, analytics, or loading independent dashboard widgets.
