# Why Promise Consumers Exist

Suppose we create a Promise.

```javascript
const promise = new Promise((resolve, reject) => {

    resolve("Transaction Uploaded");

});
```

Question:

Where does the value

```text
"Transaction Uploaded"
```

go?

It is stored inside the Promise object.

But simply storing it isn't enough.

Applications need to **use** that value.

JavaScript therefore provides **consumer methods**.

These methods register functions that will execute when the Promise settles.

---

# The Three Main Consumer Methods

Every Promise provides three important methods.

| Method | Purpose |
|----------|----------|
| `.then()` | Handles successful completion |
| `.catch()` | Handles failure |
| `.finally()` | Runs regardless of success or failure |

These methods allow us to react to the Promise's outcome.

---

# Understanding `.then()`

The most common Promise method is:

```javascript
.then()
```

General syntax

```javascript
promise.then(successHandler);
```

or

```javascript
promise.then(value => {

});
```

Notice something important.

`.then()` does **not** immediately execute the function.

Instead,

it **registers** the function.

---

# Mental Model

Imagine ordering food.

You tell the waiter:

> "When my food arrives,
> call me."

You don't receive the food immediately.

You simply register interest.

`.then()` works the same way.

```
Promise

↓

Still Pending

↓

Register Success Handler

↓

Promise Settles

↓

Execute Handler
```

---

# What Is a Success Handler?

The function passed into `.then()` is often called:

- Success callback
- Fulfillment handler
- Success handler

All three terms refer to the same concept.

Example

```javascript
promise.then(function(value) {

    console.log(value);

});
```

If the Promise resolves,

the function receives the resolved value.

---

# Example

```javascript
const promise = new Promise((resolve) => {

    resolve("Hello");

});

promise.then(function(value) {

    console.log(value);

});
```

Output

```text
Hello
```

---

# Internal JavaScript Behaviour

Suppose we write

```javascript
promise.then(handler);
```

Internally,

JavaScript conceptually performs:

```text
Promise Object

↓

Success Handlers

↓

[
handler
]
```

The handler is stored.

It is **not executed yet**.

Later,

when the Promise becomes fulfilled,

JavaScript schedules that handler as a **microtask**.

---

# Promise Timeline

```text
Create Promise

↓

Pending

↓

Register .then()

↓

Still Pending

↓

resolve()

↓

Fulfilled

↓

Microtask Queue

↓

Execute Handler
```

Notice

The handler executes **after** the current synchronous code finishes.

---

# Understanding `.catch()`

Promises may fail.

If no error handler exists,

the application may report an unhandled Promise rejection.

`.catch()` exists to handle failures.

General syntax

```javascript
promise.catch(errorHandler);
```

---

Example

```javascript
const promise = new Promise((resolve, reject) => {

    reject("Network Error");

});

promise.catch(function(error) {

    console.log(error);

});
```

Output

```text
Network Error
```

---

# Internal Behaviour

When JavaScript encounters

```javascript
.catch(handler)
```

it conceptually stores

```text
Failure Handlers

↓

[
handler
]
```

When the Promise becomes rejected,

the handler enters the **Microtask Queue**.

---

# Understanding `.finally()`

Sometimes we need code that should execute whether the Promise succeeds or fails.

Examples

- Hide loading spinner
- Close database connection
- Stop animation
- Release resources

Instead of duplicating code,

we use

```javascript
.finally()
```

Example

```javascript
promise.finally(function() {

    console.log("Finished");

});
```

This executes:

- after fulfillment
- after rejection

---

# Important Difference

`.finally()` does **not** receive:

- success value
- rejection reason

Its purpose is cleanup,

not processing results.

---

# Visual Comparison

```text
Promise

↓

Fulfilled

↓

.then()

-----------------

Promise

↓

Rejected

↓

.catch()

-----------------

Either Outcome

↓

.finally()
```

# Example 3 — Real-World Application

Suppose a POS application uploads transactions.

```javascript
function uploadTransaction() {

    return new Promise((resolve) => {

        resolve("Upload Successful");

    });

}
```

Consume it.

```javascript
uploadTransaction()

.then(function(message) {

    console.log(message);

})

.finally(function() {

    console.log("Hide Loading Spinner");

});
```

Output

```text
Upload Successful

Hide Loading Spinner
```

Notice

Cleanup always happens.

# Promise Reactions Are Microtasks

This is an extremely important concept.

Recall the Event Loop.

Promise reactions are **Microtasks**.

Example

```javascript
console.log("A");

Promise.resolve("Done")

.then(console.log);

console.log("B");
```

Output

```text
A

B

Done
```

Why?

Execution

```text
A

↓

Register .then()

↓

B

↓

Stack Empty

↓

Execute Microtask

↓

Done
```

This is why Promise handlers execute before timers.

---

# Common Mistakes

## Mistake 1

Expecting `.then()` to execute immediately.

It doesn't.

It schedules a microtask.

---

## Mistake 2

Using `.catch()` for successful results.

Incorrect

```javascript
promise.catch(result => {

});
```

`.catch()` handles failures only.

---

## Mistake 3

Expecting `.finally()` to receive the resolved value.

Incorrect

```javascript
promise.finally(value => {

});
```

`value` will not contain the fulfilled result.

Use `.then()` or `.catch()` for that.

---

## Mistake 4

Forgetting to handle Promise failures.

Unhandled rejections can terminate Node.js processes (depending on configuration) or produce browser console warnings.

Always consider the error path.

---

# Best Practices

- Use `.then()` for successful outcomes.
- Use `.catch()` for failures.
- Use `.finally()` only for cleanup.
- Keep handlers short and focused.
- Prefer meaningful parameter names such as:

```javascript
result

response

user

error

message
```

instead of

```javascript
x

y

data1
```

---

# Performance Considerations

Promise handlers execute as **Microtasks**.

Microtasks have higher priority than Macrotasks.

Creating excessive Promise chains is usually inexpensive,

but creating endless Microtasks can delay rendering and timer callbacks.

---

# Interview Questions

## Beginner

**Q:** What does `.then()` do?

**Answer:**

Registers a function that executes when a Promise is fulfilled.

---

## Intermediate

**Q:** What is the purpose of `.finally()`?

**Answer:**

To execute cleanup code regardless of whether the Promise succeeds or fails.

---

## Advanced

**Q:** Why does a `.then()` callback execute before a `setTimeout(..., 0)` callback?

**Answer:**

Because Promise reactions are placed in the **Microtask Queue**, which the Event Loop completely drains before processing the next **Macrotask**.
