# Why Do We Need to Create Promises?

Suppose you're building a POS application.

You need to upload transactions.

Uploading takes time.

Without Promises, you might write:

```javascript
uploadTransaction(function () {

    console.log("Upload Complete");

});
```

This works.

But callbacks become difficult to compose as workflows grow.

Instead,

your upload function can return a Promise.

```javascript
uploadTransaction()

    .then(() => {

        console.log("Upload Complete");

    });
```

This is much easier to extend.

# The Promise Constructor

Every Promise begins with the `Promise` constructor.

General syntax

```javascript
const promise = new Promise(executor);
```

Notice two important things:

1. `Promise` is a constructor.
2. We use the `new` keyword.

This creates a brand-new Promise object.

---

# Breaking Down the Syntax

```javascript
const promise = new Promise(executor);
```

### `new`

Creates a new object.

Just like

```javascript
const person = new Person();
```

creates a new object,

```javascript
new Promise(...)
```

creates a new Promise object.

---

### `Promise`

The built-in Promise constructor provided by JavaScript.

---

### `executor`

A function that JavaScript executes immediately.

This function tells JavaScript:

- how to complete successfully
- how to fail

---

# The Executor Function

General form

```javascript
new Promise(function(resolve, reject) {

});
```

or

```javascript
new Promise((resolve, reject) => {

});
```

The executor always receives **two functions** supplied by JavaScript itself:

```text
resolve

reject
```

You do **not** create them.

JavaScript provides them automatically.

---

# Mental Model

Think of the executor as the person responsible for completing a task.

```
Promise Created

↓

Executor Starts Working

↓

Task Succeeds?

↓

Yes → resolve()

↓

No → reject()
```

---

# What Does `resolve()` Do?

`resolve()` tells JavaScript:

> **"The asynchronous operation completed successfully."**

Example

```javascript
const promise = new Promise((resolve, reject) => {

    resolve("Success");

});
```

The Promise changes from

```text
Pending
```

to

```text
Fulfilled
```

and stores:

```text
"Success"
```

---

# What Does `reject()` Do?

`reject()` tells JavaScript:

> **"The operation failed."**

Example

```javascript
const promise = new Promise((resolve, reject) => {

    reject("Network Error");

});
```

The Promise changes from

```text
Pending
```

to

```text
Rejected
```

and stores:

```text
"Network Error"
```

---

# Promise State Transition

Initially

```text
Pending
```

Calling

```javascript
resolve(...)
```

becomes

```text
Pending

↓

Fulfilled
```

Calling

```javascript
reject(...)
```

becomes

```text
Pending

↓

Rejected
```

There are no other transitions.

---

# Important Rule

A Promise may settle **only once**.

Example

```javascript
new Promise((resolve, reject) => {

    resolve("Success");

    reject("Failure");

});
```

Only

```javascript
resolve()
```

takes effect.

The later `reject()` call is ignored.

Likewise,

```javascript
reject();

resolve();
```

leaves the Promise rejected.

Once settled,

its state never changes.

---

# Internal JavaScript Behaviour

Suppose we write

```javascript
const promise = new Promise((resolve, reject) => {

    resolve("Done");

});
```

Conceptually,

JavaScript creates:

```text
Promise Object

↓

State

Pending

↓

Result

undefined
```

Then it immediately executes the executor.

---

The executor calls

```javascript
resolve("Done");
```

JavaScript updates the Promise internally:

```text
Promise Object

↓

State

Fulfilled

↓

Result

"Done"
```

The Promise object itself remains the same object.

Only its internal fields change.

---

# Does the Executor Wait?

A very common beginner misconception is:

> "The executor runs later."

This is **false**.

The executor runs **immediately** when the Promise is created.

Example

```javascript
console.log("Before");

new Promise((resolve) => {

    console.log("Inside Executor");

    resolve();

});

console.log("After");
```

Output

```text
Before

Inside Executor

After
```

Notice

The executor executes synchronously.

Only the Promise's reactions (`.then()`, `.catch()`, etc.) are scheduled asynchronously.

We'll study that in the next lesson.

        reject("Minor");

```

Execution

```
Check Age

↓

↓

Yes

↓

Fulfilled

↓

# Example 3 — Real-World Application

Suppose a POS system uploads transactions.

```javascript
function uploadTransaction() {

    return new Promise((resolve, reject) => {

        const networkAvailable = true;

        if (networkAvailable) {

            resolve("Transaction Uploaded");

        } else {

            reject("Network Unavailable");

        }

    });

}
```

This function returns a Promise that represents the future outcome of the upload.

Notice that callers don't need to know how the upload works internally.

They only need to know whether it eventually succeeds or fails.

# Common Mistakes

## Mistake 1 — Calling Both `resolve()` and `reject()`

Incorrect

```javascript
resolve("Success");

reject("Failure");
```

Only the first call has any effect.

---

## Mistake 2 — Forgetting to Settle the Promise

```javascript
new Promise((resolve, reject) => {

    console.log("Working...");

});
```

Neither `resolve()` nor `reject()` is called.

The Promise remains:

```text
Pending
```

forever.

---

## Mistake 3 — Assuming the Executor Is Asynchronous

Remember:

The executor itself executes immediately.

Only Promise reactions execute asynchronously.

---

# Best Practices

- Settle every Promise exactly once.
- Call either `resolve()` or `reject()`.
- Use meaningful success values and informative error messages.
- Keep executor functions small and focused.
- Avoid performing unnecessary heavy synchronous work inside the executor.

---

# Performance Considerations

Creating a Promise is lightweight.

However, avoid creating unnecessary Promises in tight loops or when synchronous code is sufficient.

Use Promises to model **genuinely asynchronous** operations.

# Interview Questions

### Beginner

**Q:** What does the Promise constructor return?

**Answer:**

A new Promise object.

---

### Intermediate

**Q:** When does the executor function execute?

**Answer:**

Immediately when the Promise is created.

---

### Advanced

**Q:** What happens if both `resolve()` and `reject()` are called?

**Answer:**

Only the first call changes the Promise state. All later attempts to settle the Promise are ignored.
