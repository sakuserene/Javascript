# Promises (Part 1)

# Introduction to Promises

> **This module is one of the most important modules in modern JavaScript.**

Promises changed how JavaScript developers write asynchronous code.

Before Promises, large JavaScript applications relied heavily on nested callbacks.

As applications became larger, Callback Hell became increasingly difficult to manage.

Promises were introduced to solve this problem.

Understanding Promises deeply will make learning:

- `fetch()`
- `async/await`
- React
- Vue
- Angular
- Node.js
- Express
- Database libraries

significantly easier.

# What Is a Promise?

Let's begin with a precise definition.

> **A Promise is an object that represents the eventual completion or failure of an asynchronous operation.**

Notice the important words:

- **object**
- **eventual**
- **completion**
- **failure**

A Promise does **not** necessarily contain a value immediately.

Instead, it represents a value that may become available in the future.

---

# Why Is It Called a Promise?

Imagine your friend says:

> "I'll send you the report tomorrow."

Right now, you don't have the report.

What you have is a **promise**.

Two things can happen:

1. Your friend sends the report.
2. Your friend fails to send it.

Until tomorrow arrives, the outcome is unknown.

JavaScript Promises behave in exactly the same way.

---

# Real-Life Analogy

Suppose you order a custom laptop online.

Timeline

```text
Place Order

↓

Order Being Processed

↓

Laptop Shipped

↓

Delivered
```

When you place the order:

- You do **not** immediately receive the laptop.
- You receive confirmation that the process has started.

That confirmation is similar to a JavaScript Promise.

The Promise represents future work.

---

# Why Were Promises Invented?

Recall the callback example:

```javascript
authenticate(function () {

    upload(function () {

        print(function () {

            notify(function () {

                finish();

            });

        });

    });

});
```

Problems:

- Deep nesting
- Difficult error handling
- Hard to read
- Hard to maintain

Promises flatten this structure.

Instead of nesting,

we chain operations.

```javascript
authenticate()

    .then(upload)

    .then(print)

    .then(notify)

    .then(finish);
```

Notice the improvement.

The code now flows from top to bottom instead of moving further to the right.

---

# A Promise Is an Object

Many beginners think a Promise is simply "asynchronous code."

Not quite.

A Promise is an **object**.

Example

```javascript
const promise = new Promise(() => {

});
```

Conceptually

```text
promise

↓

Promise Object
```

This object stores information about an asynchronous operation.

---

# What Information Does a Promise Store?

Conceptually, a Promise stores:

```text
Promise

│

├── State

├── Result

├── Success Callbacks

└── Failure Callbacks
```

The JavaScript engine manages these details internally.

---

# Promise States

A Promise can exist in one of **three states**.

```text
Pending

↓

Fulfilled

OR

Rejected
```

Let's study each one carefully.

---

## 1. Pending

This is the initial state.

The asynchronous operation has started,

but it has not finished.

Example

```text
Downloading File...

```

The Promise is waiting.

---

## 2. Fulfilled

The operation completed successfully.

Example

```text
Download Complete
```

The Promise now has a successful result.

---

## 3. Rejected

Something went wrong.

Examples

- Network failure
- File not found
- Authentication error
- Server unavailable

The Promise stores the failure reason.

---

# Promise State Diagram

```text
            Pending

           /       \

          /         \

         ▼           ▼

 Fulfilled       Rejected
```

Notice something important.

A Promise can only move **once**.

It never goes:

```text
Pending

↓

Fulfilled

↓

Rejected
```

or

```text
Pending

↓

Rejected

↓

Fulfilled
```

Once settled,

its state is permanent.

---

# Promise Lifecycle

Imagine downloading user information.

```text
Create Promise

↓

Pending

↓

Waiting

↓

Server Responds

↓

Fulfilled

↓

OR

↓

Rejected
```

After settlement,

the Promise never changes again.

---

# Internal JavaScript Behaviour

Suppose we create

```javascript
const promise = new Promise(() => {

});
```

Conceptually,

JavaScript creates something similar to:

```text
Promise Object

↓

State

Pending

↓

Result

undefined

↓

Success Callbacks

[]

↓

Failure Callbacks

[]
```

As asynchronous work progresses,

JavaScript updates the Promise internally.

---

# What Happens When the Promise Finishes?

If successful

```text
State

↓

Fulfilled

↓

Result

User Data
```

If unsuccessful

```text
State

↓

Rejected

↓

Result

Error Object
```

The Promise object itself remains the same.

Only its internal state changes.

---

# Why Promises Are Better Than Callbacks

Callbacks answer the question:

> **"What function should execute later?"**

Promises answer a richer question:

> **"What object represents the future result of this asynchronous operation?"**

This difference allows JavaScript to:

- Chain operations
- Handle errors consistently
- Compose asynchronous workflows
- Avoid deep nesting

# Promise vs Callback

| Callback | Promise |
|----------|----------|
| Function | Object |
| Executes later | Represents future result |
| Can create deep nesting | Supports chaining |
| Error handling is manual | Built-in error propagation |
| Harder to compose | Easy to compose |

# Common Mistakes

## Mistake 1

Thinking a Promise contains the final value immediately.

Incorrect.

Initially,

the Promise only represents future work.

---

## Mistake 2

Thinking a Promise can change state multiple times.

It cannot.

A Promise settles only once.

---

## Mistake 3

Confusing the Promise object with its result.

These are different.

Example

```javascript
const promise = fetch(...);
```

`promise` is **not** the downloaded data.

It is a Promise that may later produce that data.

---

# Best Practices

- Think of Promises as **future values**, not immediate values.
- Design asynchronous APIs to return Promises rather than relying on nested callbacks.
- Handle both success and failure paths.
- Remember that Promise objects are immutable after settlement.

---

# Performance Considerations

Promises introduce a small amount of runtime overhead compared to simple callbacks.

However, they dramatically improve readability, maintainability, composability, and error handling.

In almost all real-world applications, these benefits outweigh the minimal overhead.

**Q:** What are the three Promise states?

**Answer:**

### Advanced

**Q:** Can a fulfilled Promise become rejected later?

**Answer:**

No.

Once a Promise settles (fulfilled or rejected), its state is permanent.
