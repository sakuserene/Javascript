# Promise Chaining (Deep Dive)

> **Promise Chaining is the feature that made Promises revolutionary.**

Before Promises, JavaScript developers wrote deeply nested callbacks.

After Promises, asynchronous code became linear, readable, and maintainable.

Understanding Promise chaining is **far more important than memorizing Promise syntax.**

By the end of this lesson, you'll understand:

- Why every `.then()` returns another Promise.
- How values move through a chain.
- How errors travel through a chain.
- Why Promise chains are "flat" instead of deeply nested.
- How professional applications organize asynchronous workflows.

# Why Promise Chaining Exists

Imagine a POS system.

Workflow:

```
Authenticate User

↓

Download Profile

↓

Download Today's Transactions

↓

Calculate Totals

↓

Update Dashboard

↓

Print Welcome Message
```

Each task depends on the previous one.

---

## Before Promises

Developers wrote

```javascript
authenticate(function () {

    getProfile(function () {

        getTransactions(function () {

            calculateTotals(function () {

                updateDashboard(function () {

                    printWelcome();

                });

            });

        });

    });

});
```

This is Callback Hell.

---

## With Promises

The same workflow becomes

```javascript
authenticate()

.then(getProfile)

.then(getTransactions)

.then(calculateTotals)

.then(updateDashboard)

.then(printWelcome);
```

Instead of moving right,

the code flows downward.

Professional developers strongly prefer this style.

---

# The Biggest Secret of `.then()`

Most beginners think:

> ".then() executes a callback."

That is true.

But it is **not** the most important thing.

The most important fact is:

> **Every call to `.then()` returns a brand-new Promise.**

This is the reason chaining is possible.

---

# Visual Representation

```text
Promise A

↓

.then()

↓

Promise B

↓

.then()

↓

Promise C

↓

.then()

↓

Promise D
```

Notice

Each `.then()` creates another Promise.

---

# Internal Mental Model

Suppose we write

```javascript
fetchUser()

.then(getOrders)

.then(calculateTotal)

.then(showDashboard);
```

Conceptually JavaScript sees

```text
Promise

↓

.then()

↓

New Promise

↓

.then()

↓

Another Promise

↓

.then()

↓

Another Promise
```

The result of one Promise becomes the input of the next.

---

# Returning Values from `.then()`

Let's examine a simple example.

```javascript
Promise.resolve(10)

.then(function(value) {

    return value * 2;

})

.then(function(value) {

    console.log(value);

});
```

Output

```text
20
```

Question:

How did the second `.then()` receive `20`?

Answer:

Because the first `.then()` returned it.

---

# Execution Diagram

```text
Promise

↓

10

↓

First .then()

↓

return 20

↓

New Promise

↓

20

↓

Second .then()

↓

Print
```

The returned value automatically becomes the fulfillment value of the next Promise in the chain.

---

# Returning Normal Values

Suppose

```javascript
Promise.resolve(5)

.then(value => {

    return value + 10;

})

.then(value => {

    console.log(value);

});
```

Output

```text
15
```

The returned number is automatically wrapped inside a fulfilled Promise.

Conceptually JavaScript behaves like:

```javascript
return Promise.resolve(15);
```

You don't need to write that yourself.

JavaScript does it automatically.

---

# Returning Another Promise

Now suppose

```javascript
Promise.resolve(5)

.then(value => {

    return Promise.resolve(value * 3);

})

.then(value => {

    console.log(value);

});
```

Output

```text
15
```

Notice something.

The next `.then()` **waited**.

It did not receive the Promise object.

It received the resolved value.

---

# Why?

JavaScript automatically unwraps returned Promises.

This behavior is called:

> **Promise Resolution**

---

# Promise Resolution

Suppose

```javascript
.then(() => {

    return somePromise;

});
```

JavaScript internally performs something conceptually similar to:

```text
Returned Promise

↓

Wait

↓

Fulfilled?

↓

Use Value

↓

Continue Chain
```

This automatic waiting is one of the most powerful features of Promises.

---

# Internal Promise Resolution Algorithm

Conceptually,

JavaScript follows these rules.

If a `.then()` callback returns:

### A normal value

```javascript
return 100;
```

JavaScript creates

```text
Fulfilled Promise

↓

100
```

---

### Another Promise

```javascript
return fetch(...);
```

JavaScript waits.

```text
Promise

↓

Pending

↓

Wait

↓

Fulfilled

↓

Continue Chain
```

---

### An Error

```javascript
throw new Error();
```

JavaScript immediately creates

```text
Rejected Promise
```

We'll explore this more in the next section.

---

# Value Propagation

Values naturally move down the chain.

Example

```javascript
Promise.resolve(2)

.then(value => {

    return value + 3;

})

.then(value => {

    return value * 10;

})

.then(value => {

    console.log(value);

});
```

Execution

```text
2

↓

5

↓

50

↓

Print
```

Output

```text
50
```

Each step transforms the previous value.

---

# Error Propagation

Errors also move through the chain.

Suppose

```javascript
Promise.resolve()

.then(() => {

    throw new Error("Server Error");

})

.catch(error => {

    console.log(error.message);

});
```

Output

```text
Server Error
```

Notice

The error skipped every remaining success handler until a `.catch()` handled it.

---

# Visual Diagram

```text
Promise

↓

.then()

↓

Error

↓

Skip

↓

Skip

↓

.catch()

↓

Recovered
```

This automatic error propagation is one of the biggest improvements over callback-based code.

```javascript
Promise.resolve("Alice")

.then(name => {

Output

```

Each `.then()` performs one transformation.

# Example 3 — Real-World Application

Imagine a POS application.

```javascript
authenticate()

.then(getProfile)

.then(getTransactions)

.then(calculateTotals)

.then(displayDashboard)

.then(() => {

    console.log("Dashboard Ready");

})

.catch(error => {

    console.error(error);

});
```

Workflow

```text
Authenticate

↓

Profile

↓

Transactions

↓

Totals

↓

Dashboard

↓

Ready
```

Any failure automatically jumps to `.catch()`.

This keeps the code clean and easy to maintain.

# Common Mistakes

## Mistake 1 — Forgetting to Return

Incorrect

```javascript
.then(value => {

    value * 2;

});
```

Nothing is returned.

Correct

```javascript
.then(value => {

    return value * 2;

});
```

Or

```javascript
.then(value => value * 2);
```

---

## Mistake 2 — Creating Nested Promise Chains

Incorrect

```javascript
authenticate()

.then(() => {

    getProfile()

    .then(() => {

        // ...

    });

});
```

Correct

```javascript
authenticate()

.then(() => {

    return getProfile();

});
```

Returning the Promise keeps the chain flat.

---

## Mistake 3 — Expecting Returned Promises to Execute Immediately

Returned Promises must first settle.

The chain waits automatically.

---

# Best Practices

- Return values from `.then()` when transforming data.
- Return Promises from `.then()` when performing more asynchronous work.
- Keep each `.then()` focused on a single responsibility.
- Avoid nested Promise chains.
- Let Promise chaining flatten your asynchronous workflow naturally.

---

# Performance Considerations

Promise chaining itself has very little overhead.

The primary benefit is improved readability, maintainability, and centralized error handling.

Deeply nested Promises are harder to understand than flat chains, so always prefer returning Promises instead of nesting them.

# Interview Questions

### Beginner

**Q:** What does `.then()` return?

**Answer:**

A brand-new Promise.

---

### Intermediate

**Q:** What happens if a `.then()` callback returns a normal value?

**Answer:**

JavaScript automatically wraps it in a fulfilled Promise and passes it to the next `.then()`.

---

### Advanced

**Q:** What happens if a `.then()` callback returns another Promise?

**Answer:**

JavaScript waits for that Promise to settle, then continues the chain using its fulfilled value or rejected reason.
