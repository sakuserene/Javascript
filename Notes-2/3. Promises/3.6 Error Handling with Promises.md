# JavaScript Promises (Part 6)

# Error Handling with Promises

> **Professional JavaScript isn't about writing code that never fails.**
>
> It is about writing code that **fails gracefully**, **recovers correctly**, and **provides meaningful information when something goes wrong**.

Error handling is one of the biggest advantages Promises have over callbacks.

In callback-based code, every asynchronous operation often needed its own error handling.

Promises centralized this into a clean, predictable model.

# Why Promise Error Handling Exists

Suppose we download user information.

Many things can go wrong.

```
Request

↓

Internet Lost

↓

Server Offline

↓

Authentication Failed

↓

Invalid JSON

↓

Database Error

↓

Permission Denied
```

Every one of these is a possible failure.

If failures are ignored,

applications become:

- unreliable
- unpredictable
- difficult to debug

Promises solve this by making failures first-class citizens.

---

# Promise Rejections

Recall the Promise lifecycle.

```
Pending

↓

Fulfilled

OR

Rejected
```

Rejected Promises contain:

```
Failure Reason
```

That reason is often:

```javascript
Error
```

or

```javascript
new Error(...)
```

---

# Internal Promise Representation

Conceptually

```
Promise

↓

State

Rejected

↓

Reason

↓

Error Object
```

The Promise now remembers:

- what failed
- why it failed

---

# What Is Error Propagation?

One of the greatest features of Promises is automatic error propagation.

Suppose

```javascript
Promise.resolve()

.then(() => {

    throw new Error("Server Offline");

})

.then(() => {

    console.log("Never Runs");

})

.catch(error => {

    console.log(error.message);

});
```

Output

```text
Server Offline
```

Notice

The second `.then()` never executes.

Instead,

JavaScript automatically skips forward until it finds a `.catch()`.

---

# Error Flow Diagram

```
Promise

↓

.then()

↓

throw Error

↓

Rejected Promise

↓

Skip Success Handlers

↓

.catch()

↓

Handle Error
```

This automatic propagation makes Promise chains much easier to reason about.

---

# Why `.catch()` Works

Recall

```javascript
promise.catch(handler);
```

Conceptually,

JavaScript treats it like

```javascript
promise.then(

    undefined,

    handler

);
```

The first argument handles success.

The second handles failure.

`.catch()` is simply a cleaner way to register only the failure handler.

---

# Throwing Errors Inside `.then()`

Many beginners think:

> "If I throw an error inside `.then()`, the application crashes."

It doesn't.

Example

```javascript
Promise.resolve(100)

.then(value => {

    throw new Error("Calculation Failed");

})

.catch(error => {

    console.log(error.message);

});
```

Output

```text
Calculation Failed
```

Throwing an error inside a `.then()` callback automatically rejects the Promise returned by that `.then()`.

---

# Internal Behaviour

```
.then()

↓

throw Error

↓

JavaScript Creates

↓

Rejected Promise

↓

Next .catch()
```

You don't need to call `reject()` yourself here.

---

# Recovering from Errors

Sometimes we don't want the program to stop.

We want to recover.

Example

```javascript
Promise.reject(

    new Error("Network Error")

)

.catch(error => {

    console.log(error.message);

    return "Offline Mode";

})

.then(mode => {

    console.log(mode);

});
```

Output

```text
Network Error

Offline Mode
```

Notice

The chain continues.

Why?

Because `.catch()` returned a normal value.

Returning a value from `.catch()` converts the chain back into a fulfilled Promise.

---

# Recovery Diagram

```
Rejected Promise

↓

.catch()

↓

Return Value

↓

Fulfilled Promise

↓

Continue Chain
```

---

# Rethrowing Errors

Sometimes we can't recover.

Instead,

we pass the error to another handler.

Example

```javascript
Promise.reject(

    new Error("Database Error")

)

.catch(error => {

    console.log("Logging...");

    throw error;

})

.catch(error => {

    console.log(error.message);

});
```

Output

```text
Logging...

Database Error
```

Notice

The error continued down the chain.

---

# `finally()` Revisited

Recall

```javascript
.finally()
```

Its purpose is cleanup.

Examples

- Stop loading spinner
- Close file
- Release lock
- Hide modal

Example

```javascript
Promise.resolve("Done")

.finally(() => {

    console.log("Cleanup");

});
```

Output

```text
Cleanup
```

If the Promise rejects,

`finally()` still executes.

---

# Important Rule

`finally()` does **not** receive

- result
- error

Its job is cleanup,

not decision making.

# Custom Error Objects

Instead of throwing plain strings

```javascript
throw "Error";
```

Professionals throw Error objects.

```javascript
throw new Error("Invalid Credentials");
```

Why?

Error objects contain:

- message
- stack trace
- name

These make debugging much easier.

---

# Creating Custom Errors

JavaScript allows custom error classes.

Example

```javascript
class AuthenticationError extends Error {

    constructor(message) {

        super(message);

        this.name = "AuthenticationError";

    }

}
```

Usage

```javascript
throw new AuthenticationError(

    "Access Denied"

);
```

Output

```text
AuthenticationError

Access Denied
```

Custom errors make it easier to distinguish between different failure types.

# Global Promise Rejection Handling

Sometimes a rejected Promise has no `.catch()`.

Browsers report:

```text
Uncaught (in promise)
```

Example

```javascript
Promise.reject(

    new Error("Server Error")

);
```

Browser Console

```text
Uncaught (in promise)

Error:

Server Error
```

In production applications,

every Promise rejection should eventually be handled.

---

# Example 1 — Beginner

```javascript
Promise.reject(

    new Error("Invalid Login")

)

.catch(error => {

    console.log(error.message);

});
```

Output

```text
Invalid Login
```

---

# Example 2 — Intermediate

```javascript
Promise.resolve(50)

.then(value => {

    if (value < 100) {

        throw new Error(

            "Amount Too Small"

        );

    }

    return value;

})

.catch(error => {

    console.log(error.message);

    return 100;

})

.then(value => {

    console.log(value);

});
```

Output

```text
Amount Too Small

100
```

Notice

The chain recovered.

# Example 3 — Real-World Application

Suppose a POS application uploads transactions.

```javascript
uploadTransaction()

.then(saveReceipt)

.then(syncCloud)

.catch(error => {

    console.log(

        "Switching to Offline Mode"

    );

    return cacheTransaction();

})

.then(() => {

    console.log(

        "Application Continues"

    );

})

.finally(() => {

    hideLoadingSpinner();

});
```

Workflow

```
Upload

↓

Success?

↓

No

↓

Offline Cache

↓

Continue

↓

Cleanup
```

This is how professional offline-first applications are built.

# Common Mistakes

## Mistake 1

Throwing strings.

Incorrect

```javascript
throw "Server Error";
```

Correct

```javascript
throw new Error(

    "Server Error"

);
```

---

## Mistake 2

Forgetting to return from `.catch()` when recovery is intended.

Incorrect

```javascript
.catch(() => {

    "Offline";

});
```

Correct

```javascript
.catch(() => {

    return "Offline";

});
```

---

## Mistake 3

Ignoring rejected Promises.

Unhandled Promise rejections make debugging much more difficult.

---

## Mistake 4

Using `finally()` for business logic.

Keep `finally()` for cleanup.

Use `.then()` and `.catch()` for decision making.

---

# Best Practices

- Always handle Promise rejections.
- Throw `Error` objects instead of strings.
- Recover only when recovery makes sense.
- Rethrow errors that cannot be handled locally.
- Keep cleanup inside `finally()`.
- Create custom error classes for complex applications.

---

# Performance Considerations

Error handling introduces very little overhead.

The cost of proper error handling is insignificant compared to the cost of debugging production failures.

Readable, maintainable error handling should always be preferred.

# Interview Questions

### Beginner

**Q:** What does `.catch()` do?

**Answer:**

It registers a handler that executes if the Promise is rejected or if an earlier `.then()` throws an error.

---

### Intermediate

**Q:** What happens if a `.then()` callback throws an error?

**Answer:**

The Promise returned by that `.then()` becomes rejected, and the error propagates to the next available `.catch()`.

---

### Advanced

**Q:** What is the difference between recovering from an error and rethrowing it?

**Answer:**

Recovering returns a value (or fulfilled Promise), allowing the chain to continue successfully. Rethrowing creates another rejected Promise, allowing the error to continue propagating down the chain.
