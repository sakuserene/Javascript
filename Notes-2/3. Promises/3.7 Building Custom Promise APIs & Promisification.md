# Introduction

Suppose you're building a reusable library.

Should users write

```javascript
readFile(function(result) {

});
```

or

```javascript
readFile()

.then(result => {

});
```

Today,

most JavaScript developers expect Promise APIs.

Modern JavaScript has largely standardized on Promises because they are easier to compose and integrate with `async`/`await`.

---

# Why Build Promise APIs?

Imagine you're creating a POS SDK.

You expose this function:

```javascript
uploadTransaction();
```

You don't know how other developers will use it.

Some may want

```javascript
.then()
```

Others

```javascript
await
```

If your function returns a Promise,

both become possible.

That is why Promise-returning APIs are so flexible.

---

# Designing Promise-Based Functions

A professional Promise API usually follows this pattern.

```javascript
function doSomething() {

    return new Promise(

        (resolve, reject) => {

            // asynchronous work

        }

    );

}
```

Notice

The function itself does not return data immediately.

It returns a Promise representing future work.

---

# Example

```javascript
function greetUser() {

    return new Promise(

        resolve => {

            resolve("Hello Alice");

        }

    );

}
```

Usage

```javascript
greetUser()

.then(message => {

    console.log(message);

});
```

Output

```text
Hello Alice
```

---

# Internal Behaviour

Execution

```text
Call greetUser()

↓

Create Promise

↓

Return Promise Immediately

↓

Caller Registers .then()

↓

Promise Resolves

↓

Microtask Queue

↓

Execute Handler
```

Notice

The caller never waits synchronously.

---

# Returning Promises from Functions

Suppose

```javascript
function getProfile() {

    return fetch("/profile");

}
```

The function doesn't return profile data.

It returns a Promise.

The caller decides when and how to consume it.

This separation of responsibilities is a hallmark of good API design.

---

# Promisification

## What Is Promisification?

Promisification means:

> **Converting a callback-based API into a Promise-based API.**

This was especially important when Promises were introduced because many existing APIs used callbacks.

---

# Before Promisification

```javascript
function login(callback) {

    const success = true;

    if (success) {

        callback(null, "Logged In");

    } else {

        callback(

            new Error("Login Failed")

        );

    }

}
```

Usage

```javascript
login(function(error, result) {

    if (error) {

        console.log(error);

        return;

    }

    console.log(result);

});
```

This works,

but it relies on callback conventions.

---

# After Promisification

We wrap the callback API in a Promise.

```javascript
function loginPromise() {

    return new Promise(

        (resolve, reject) => {

            const success = true;

            if (success) {

                resolve("Logged In");

            } else {

                reject(

                    new Error("Login Failed")

                );

            }

        }

    );

}
```

Usage

```javascript
loginPromise()

.then(console.log)

.catch(console.error);
```

Much cleaner.

---

# Wrapping Legacy Callback APIs

Suppose we cannot modify an old API.

```javascript
function oldAPI(callback) {

    callback(

        null,

        "Legacy Data"

    );

}
```

We can create a Promise wrapper.

```javascript
function newAPI() {

    return new Promise(

        (resolve, reject) => {

            oldAPI(

                (error, result) => {

                    if (error) {

                        reject(error);

                        return;

                    }

                    resolve(result);

                }

            );

        }

    );

}
```

This allows legacy systems to integrate with modern Promise-based code.

# Reusable Promise Utilities

Professional developers often create reusable Promise helpers.

Example

```javascript
function delay(milliseconds) {

    return new Promise(

        resolve => {

            setTimeout(

                resolve,

                milliseconds

            );

        }

    );

}
```

Usage

```javascript
delay(2000)

.then(() => {

    console.log(

        "Two Seconds Passed"

    );

});
```

Output

```text
Two Seconds Passed
```

This utility becomes even more powerful when used with `await`, which we'll learn in the next module.

---

# Another Utility

Suppose we want to retry an operation.

```javascript
function retry(operation) {

    return operation()

    .catch(() => {

        return operation();

    });

}
```

Usage

```javascript
retry(uploadTransaction);
```

This pattern is common in production systems where temporary failures may occur.

---

# Promise Design Patterns

Professional Promise APIs generally follow these principles:

### 1. Return a Promise

Never mix callbacks and Promises in the same API unless absolutely necessary.

---

### 2. Reject with `Error` Objects

Prefer

```javascript
reject(

    new Error("Network Error")

);
```

instead of

```javascript
reject("Network Error");
```

---

### 3. Keep APIs Focused

Each Promise-returning function should perform one primary asynchronous responsibility.

---

### 4. Compose Small Functions

Instead of one huge Promise chain,

create many reusable Promise-returning functions.

---

# Anti-Patterns

## Anti-Pattern 1

Creating unnecessary Promises.

Incorrect

```javascript
function add(a, b) {

    return new Promise(

        resolve => {

            resolve(a + b);

        }

    );

}
```

Addition is synchronous.

Use

```javascript
function add(a, b) {

    return a + b;

}
```

Only wrap genuinely asynchronous work in Promises.

---

## Anti-Pattern 2

Mixing callbacks and Promises

```javascript
function upload(callback) {

    return new Promise(...);

}
```

Choose one style.

Mixing both creates confusing APIs.

---

## Anti-Pattern 3

Ignoring returned Promises

Incorrect

```javascript
uploadTransaction();
```

If the Promise can reject,

you should eventually handle it.

# Example 3 — Real-World POS Application

Suppose a POS application uploads transactions.

```javascript
function uploadTransaction(transaction) {

    return new Promise(

        (resolve, reject) => {

            const networkAvailable = true;

            if (networkAvailable) {

                resolve({

                    id: 101,

                    status: "Uploaded",

                    transaction

                });

            } else {

                reject(

                    new Error(

                        "No Internet Connection"

                    )

                );

            }

        }

    );

}
```

Usage

```javascript
uploadTransaction({

    receipt: "RCT-1001",

    amount: 850

})

.then(result => {

    console.log(result);

})

.catch(error => {

    console.error(error.message);

});
```

This API can now be used with:

- `.then()`
- `.catch()`
- `Promise.all()`
- `Promise.race()`
- `async/await`

without modification.

# Common Mistakes

## Mistake 1

Wrapping synchronous work in Promises unnecessarily.

---

## Mistake 2

Returning nothing instead of returning the Promise.

Incorrect

```javascript
function login() {

    new Promise(...);

}
```

Correct

```javascript
function login() {

    return new Promise(...);

}
```

---

## Mistake 3

Rejecting with plain strings.

Prefer `Error` objects for richer debugging information.

---

# Best Practices

- Return Promises from asynchronous APIs.
- Keep Promise-returning functions small and reusable.
- Reject with `Error` objects.
- Avoid mixing callback and Promise styles.
- Build utility functions (such as `delay`) that compose well with other Promise-based code.

---

# Performance Considerations

Promise wrappers introduce very little overhead.

The larger concern is avoiding unnecessary Promises around synchronous operations.

Design Promise APIs for tasks that naturally involve waiting, such as network requests, timers, file operations, or database queries.

# Interview Questions

### Beginner

**Q:** What is promisification?

**Answer:**

The process of converting a callback-based API into a Promise-based API.

---

### Intermediate

**Q:** Why should modern asynchronous APIs return Promises?

**Answer:**

Because they integrate naturally with Promise chaining, combinators, and `async`/`await`, making them easier to compose and maintain.

---

### Advanced

**Q:** Why is wrapping synchronous work in a Promise considered an anti-pattern?

**Answer:**

Because it adds unnecessary complexity and scheduling overhead without providing any benefit. Promises should represent genuinely asynchronous operations.
